# TCR-260413 — 계약 등록 페이지 개선 테스트 케이스

**작성일**: 2026-04-13

**관련 작업계획**: PLN-260413-계약등록페이지개선  

---

## 1. R1 — 금액 입력 천단위 콤마 포맷

### TC-1.1: CurrencyInput 기본 동작
- **절차**: `/billing/contracts/new` 페이지 → 계약 금액(amount) 필드에 `1234567` 입력
- **기대 결과**: 입력 시 `1,234,567`로 실시간 포맷, 실제 form 값은 `1234567` (숫자)
- **검증 항목**: 화면 표시, `handleChange` 전달 값

### TC-1.2: 소수점 입력
- **절차**: `1234567.89` 입력
- **기대 결과**: `1,234,567.89` 표시, form 값 `1234567.89`
- **검증 항목**: 소수점 이하 유지

### TC-1.3: 0 및 음수
- **절차**: `0` 입력 → 값 확인 / `-1000` 입력 시도
- **기대 결과**: `0`은 정상 표시. 음수 입력 불가 (inputMode="decimal")

### TC-1.4: 기존 계약 편집 시 포맷 표시
- **절차**: 기존 계약(amount=5000000) 편집 페이지 진입
- **기대 결과**: amount 필드에 `5,000,000` 표시

## 2. R2 — 결제 유형 및 스케줄

### TC-2.1: 결제 유형 라디오 표시
- **절차**: `/billing/contracts/new` → Billing Options 섹션 확인
- **기대 결과**: 미설정 / 정기 / 비정기 라디오 3개 표시, 기본값 "미설정"

### TC-2.2: 정기(REGULAR) 선택 시 회차 청구 금액 표시
- **절차**: "정기" 라디오 선택
- **기대 결과**: "회차 청구 금액" CurrencyInput 필드 표시 (천단위 포맷)

### TC-2.3: 비정기(IRREGULAR) 선택 시 스케줄 에디터 표시
- **절차**: "비정기" 라디오 선택
- **기대 결과**: PaymentScheduleEditor 컴포넌트 노출 (결제 스케줄 헤더 + 회차 추가 버튼)

### TC-2.4: 회차 추가/삭제
- **절차**: "비정기" 선택 → "회차 추가" 3회 클릭 → 2번째 삭제
- **기대 결과**: 
  - 회차 3개 (#1, #2, #3) 생성
  - #2 삭제 후 → #1, #2 (기존 #3이 #2로 재번호)
  - seq 자동 재정렬

### TC-2.5: 스케줄 금액 합계 검증
- **절차**: 계약 금액 100,000 입력 → "비정기" → 회차 추가 2개 (50,000 / 50,000)
- **기대 결과**: 합계 100,000 (초록색), 불일치 경고 없음

### TC-2.6: 스케줄 금액 불일치 경고
- **절차**: 계약 금액 100,000 입력 → "비정기" → 회차 추가 2개 (50,000 / 30,000)
- **기대 결과**: 합계 80,000 (빨간색), "합계가 계약 금액(100,000)과 다릅니다" 경고 표시

### TC-2.7: 결제 유형 저장 (REGULAR)
- **절차**: 신규 계약 → "정기" 선택 + billing_amount=5,000,000 + 필수 정보 입력 → 저장
- **기대 결과**: API payload에 `payment_type: "REGULAR"`, `billing_amount: 5000000` 포함
- **검증**: DB `amb_bil_contracts.ctr_payment_type = 'REGULAR'`, `ctr_billing_amount = 5000000`

### TC-2.8: 결제 유형 저장 (IRREGULAR)
- **절차**: 신규 계약 → "비정기" 선택 + 스케줄 3회 추가 → 저장
- **기대 결과**: 
  - `payment_type: "IRREGULAR"`, `payment_schedules` 배열 포함
  - DB `amb_bil_payment_schedules` 에 3건 레코드 생성
  - `pms_seq` 1, 2, 3 순서

### TC-2.9: 기존 계약 편집 시 스케줄 로드
- **절차**: TC-2.8에서 생성한 계약 편집 페이지 진입
- **기대 결과**: "비정기" 라디오 선택됨, PaymentScheduleEditor에 3회차 데이터 표시

### TC-2.10: 스케줄 수정 후 저장
- **절차**: TC-2.9의 계약 → 1회차 삭제 + 새 회차 추가 → 저장
- **기대 결과**: 기존 스케줄 전체 삭제 → 새 스케줄 2건으로 재생성 (delete+recreate 패턴)

### TC-2.11: 갱신(renew) 시 결제 유형 복사
- **절차**: REGULAR 결제 유형 + billing_amount 설정된 계약 → 갱신
- **기대 결과**: 새 계약에 `ctr_payment_type`, `ctr_billing_amount` 동일 값 복사

## 3. R3 — 신규 등록 시 파일 첨부

### TC-3.1: 신규 등록 시 첨부 UI 표시
- **절차**: `/billing/contracts/new` → 관련 문서 섹션 확인
- **기대 결과**: "관련 문서" 헤더 + "업로드" 버튼 표시 (DocumentManager 대신 간소화 UI)

### TC-3.2: 파일 선택
- **절차**: "업로드" 클릭 → 파일 2개 선택
- **기대 결과**: 선택한 파일 2개가 목록에 표시 (파일명 + 삭제 버튼)

### TC-3.3: 파일 제거
- **절차**: TC-3.2 → 첫 번째 파일 삭제 버튼 클릭
- **기대 결과**: 해당 파일 목록에서 제거, 1개만 남음

### TC-3.4: 저장 후 파일 업로드
- **절차**: 필수 정보 + 파일 2개 → 저장 클릭
- **기대 결과**: 
  1. 계약 먼저 생성 (createMutation)
  2. 반환된 `contractId`로 파일 2건 순차 업로드 (uploadMutation)
  3. 목록 페이지로 이동

### TC-3.5: 기존 계약 편집 시 DocumentManager
- **절차**: 기존 계약 편집 페이지 진입
- **기대 결과**: 기본 DocumentManager 컴포넌트 표시 (기존 동작 유지)

## 4. 통합 테스트

### TC-4.1: 전체 흐름 — 신규 등록
- **절차**: 
  1. 계약 제목/거래처/시작일/금액 입력
  2. "비정기" 선택 → 스케줄 3회 추가
  3. 파일 2개 첨부
  4. 저장
- **기대 결과**: 계약 생성 → 스케줄 3건 생성 → 파일 2건 업로드 → 목록 이동

### TC-4.2: 전체 흐름 — 편집
- **절차**: TC-4.1의 계약 편집 → 금액 변경 + 스케줄 1회 추가 → 저장
- **기대 결과**: 계약 수정 + 스케줄 4건으로 교체 + 변경 이력 기록

### TC-4.3: 결제 유형 없음(미설정)에서 정기로 전환
- **절차**: 기존 payment_type=NULL 계약 → "정기" 선택 + billing_amount 입력 → 저장
- **기대 결과**: `ctr_payment_type='REGULAR'`, `ctr_billing_amount` 설정됨, 변경 이력에 기록

## 5. 엣지 케이스

### TC-5.1: 스케줄 0건 상태에서 저장
- **절차**: "비정기" 선택 → 회차 미추가 → 저장
- **기대 결과**: `payment_type: "IRREGULAR"`, `payment_schedules: []`로 저장 (빈 배열)

### TC-5.2: 결제 유형 전환 시 데이터 독립성
- **절차**: "비정기" → 스케줄 3건 입력 → "정기"로 전환 → "비정기"로 복원
- **기대 결과**: PaymentScheduleEditor 숨겨짐→다시 표시 시 기존 state 유지 (React state가 언마운트 시에도 부모에 보관)

### TC-5.3: 파일 첨부 없이 신규 저장
- **절차**: 필수 정보만 입력 + 파일 첨부 없이 → 저장
- **기대 결과**: 계약만 생성, 파일 업로드 건너뜀, 정상 리다이렉트
