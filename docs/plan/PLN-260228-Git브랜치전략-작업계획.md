# Git 브랜치 전략 - AMB Management

> 작성일: 2026-02-28
> 목적: 3인 이상 공동개발 + 스테이징 테스트 + 프로덕션 안정 운영

---

## 1. 현황 분석 (AS-IS)

| 항목 | 현재 상태 | 문제점 |
|------|----------|--------|
| 브랜치 | `main` 단일 브랜치 | 개발·테스트·운영 구분 없음 |
| 스테이징 배포 | main에서 직접 배포 | 미검증 코드가 스테이징에 배포될 수 있음 |
| 프로덕션 배포 | main에서 직접 배포 | 스테이징 검증 없이 프로덕션 배포 가능 |
| 코드 리뷰 | 없음 | 품질 관리 부재 |
| 충돌 관리 | 없음 | 다수 개발자 동시작업 시 충돌 위험 |

## 2. 브랜치 전략 (TO-BE)

### 2.1 브랜치 구조

```
production ← 프로덕션 배포 전용 (보호 브랜치)
  ↑ PR (merge)
main ← 스테이징 배포 + 통합 브랜치 (보호 브랜치)
  ↑ PR (merge)
feature/* ← 기능 개발 브랜치
hotfix/*  ← 긴급 수정 (production → hotfix → production + main)
```

### 2.2 브랜치별 역할

| 브랜치 | 용도 | 배포 대상 | 보호 수준 | 병합 방식 |
|--------|------|----------|----------|----------|
| `production` | 프로덕션 운영 | 프로덕션 서버 | 보호 (PR 필수, 1명 이상 승인) | main → PR → production |
| `main` | 통합·스테이징 | 스테이징 서버 | 보호 (PR 필수) | feature → PR → main |
| `feature/*` | 기능 개발 | 로컬/개발 | 없음 | 개발자 자유 |
| `hotfix/*` | 긴급 수정 | - | 없음 | production에서 분기, 양쪽 병합 |

### 2.3 브랜치 네이밍 규칙

```
feature/{이슈번호}-{간략설명}    예: feature/12-user-level-rename
feature/{개발자}-{간략설명}      예: feature/gray-sidebar-menu
hotfix/{이슈번호}-{간략설명}     예: hotfix/45-login-500-error
```

## 3. 개발 플로우

### 3.1 일반 기능 개발

```
1. main에서 feature 브랜치 생성
   $ git checkout main && git pull
   $ git checkout -b feature/12-user-level-rename

2. 기능 개발 및 커밋
   $ git add . && git commit -m "feat: 사용자 레벨 리네이밍"

3. main에 Pull Request 생성
   $ git push origin feature/12-user-level-rename
   → GitHub에서 PR 생성 (main ← feature/12-user-level-rename)

4. 코드 리뷰 + 승인

5. main에 병합 → 스테이징 자동/수동 배포

6. 스테이징 테스트 완료 후 production에 PR
   → GitHub에서 PR 생성 (production ← main)

7. production 승인 + 병합 → 프로덕션 배포
```

### 3.2 긴급 수정 (Hotfix)

```
1. production에서 hotfix 브랜치 생성
   $ git checkout production && git pull
   $ git checkout -b hotfix/45-login-500-error

2. 수정 및 커밋
   $ git commit -m "fix: 로그인 500 에러 수정"

3. production에 PR → 승인 → 병합 → 프로덕션 배포

4. main에도 병합 (역방향)
   $ git checkout main && git merge hotfix/45-login-500-error
   $ git push origin main
```

## 4. 배포 플로우

```
개발자 PC (feature)
    │
    ▼ PR + Code Review
main 브랜치
    │
    ▼ 배포 (수동 또는 자동)
스테이징 서버 (mng.amoeba.site)
    │
    ▼ QA 테스트 완료 확인
    │
    ▼ PR + 승인
production 브랜치
    │
    ▼ 배포 (수동)
프로덕션 서버 (mng.amoeba.com)
```

## 5. 환경-브랜치 매핑

| 환경 | 브랜치 | 서버 | 배포 방식 |
|------|--------|------|----------|
| 개발 | `feature/*` | 로컬 (localhost) | `npm run dev` |
| 스테이징 | `main` | amb-staging | `deploy-staging.sh` |
| 프로덕션 | `production` | amb-production | `deploy-production.sh` |

## 6. 커밋 메시지 규칙

```
<type>: <subject>

type:
  feat     새 기능
  fix      버그 수정
  refactor 리팩토링
  docs     문서
  style    포맷팅 (코드 변경 없음)
  test     테스트
  chore    빌드/설정 변경

예시:
  feat: 사용자 레벨 코드 리네이밍
  fix: 로그인 시 500 에러 수정
  refactor: 에이전트 서비스 모듈 분리
```

## 7. PR 규칙

### 7.1 PR 템플릿
- 변경 사항 요약
- 관련 이슈 번호
- 테스트 방법
- 스크린샷 (UI 변경 시)

### 7.2 리뷰 정책

| 대상 브랜치 | 최소 승인 수 | 리뷰어 |
|------------|-------------|--------|
| `main` | 1명 | 팀원 중 아무나 |
| `production` | 1명 | 리드 개발자 필수 |

### 7.3 병합 방식
- `main` ← feature: **Squash and Merge** (커밋 정리)
- `production` ← main: **Merge Commit** (이력 보존)
