# 작업 계획서: 로그인 프로세스 변경 (법인코드 기반 로그인)

- **작성일**: 2026-03-09
- **상태**: 작업 대기

---

## 1. 요구사항 요약

### AS-IS (현재)
```
LoginPage: [이메일] + [비밀번호] → 로그인 버튼
  ↓
ADMIN_LEVEL → step:complete → JWT 발급 → /select-entity (API 법인 목록)
USER_LEVEL  → step:select_entity → /select-entity (selectToken + 법인 목록)
  ↓
법인 선택 → JWT 발급 → 대시보드
```

**문제점:**
- USER_LEVEL 사용자가 이메일만으로 로그인 시도 → 소속된 **모든 법인 목록이 노출됨** (fremd@naver.com → btbz, AMOEBA 둘 다 보임)
- 법인 코드를 모르는 사용자가 어떤 법인에 로그인해야 하는지 알기 어려움
- 2단계 프로세스(로그인 → 법인 선택)로 UX 복잡

### TO-BE (변경 후)
```
LoginPage: [법인코드] + [이메일] + [비밀번호] → 로그인 버튼
           [소속법인 찾기] 버튼 → 검색 모달
  ↓
법인코드+이메일+비밀번호로 직접 JWT 발급 → 대시보드
(ADMIN_LEVEL은 법인코드 없이 기존 방식 유지)
```

**핵심 변경:**
1. 로그인 페이지에 **법인코드(Entity Code)** 입력 필드 추가
2. **"소속법인 찾기"** 버튼 → 회사명/코드 검색 모달 → 선택 시 법인코드 자동 입력
3. 법인코드+이메일+비밀번호 → 특정 법인 계정으로 **직접 로그인** (법인 선택 페이지 불필요)
4. USER_LEVEL 사용자에게 **타법인 정보가 절대 노출되지 않음**

## 2. 상세 설계

### 2.1 로그인 플로우

#### USER_LEVEL (일반 사용자)
```
1. 로그인 페이지 접속
2. [소속법인 찾기] 클릭 → 검색 모달 열림
3. "amoeba" 검색 → 법인 목록 표시 (AMOEBA CO., LTD / VN01)
4. 법인 선택 → 법인코드 "VN01" 자동 입력, 모달 닫힘
5. 이메일 + 비밀번호 입력
6. [로그인] 클릭 → POST /auth/login { entity_code: "VN01", email, password }
7. 백엔드: entity_code로 법인 조회 → email+법인으로 사용자 조회 → 비밀번호 검증 → JWT
8. step: 'complete' → 대시보드로 이동
```

#### ADMIN_LEVEL (관리자)
```
1. 로그인 페이지 접속
2. 법인코드 필드 비워둠 (또는 입력해도 무시)
3. 이메일 + 비밀번호 입력
4. [로그인] 클릭 → POST /auth/login { email, password }
5. 백엔드: 기존 로직 유지 → step: 'complete' → JWT
6. /select-entity → API 법인 목록 → 법인 선택 → 대시보드
```

### 2.2 소속법인 찾기 모달

```
┌─────────────────────────────────────┐
│  소속법인 찾기                    ✕  │
│                                     │
│  [🔍 회사명 또는 법인코드 검색___]  │
│                                     │
│  ┌─────────────────────────────┐    │
│  │ 🇻🇳 AMOEBA CO., LTD         │    │
│  │    VN01 · Vietnam           │    │
│  └─────────────────────────────┘    │
│  ┌─────────────────────────────┐    │
│  │ 🇰🇷 btbz                    │    │
│  │    BTBZ951 · Korea          │    │
│  └─────────────────────────────┘    │
│                                     │
│  2자 이상 입력하면 검색됩니다       │
└─────────────────────────────────────┘
```

- 최소 2자 이상 입력 시 검색 시작 (debounce 300ms)
- 검색 대상: `ent_name`, `ent_name_en`, `ent_code` (LIKE %keyword%)
- 결과: 법인명, 코드, 국가 표시
- 클릭 시 법인코드 자동 입력 + 법인명 표시 + 모달 닫힘

## 3. 사이드 임팩트 분석

### 영향 있음

| 대상 | 영향 | 변경 내용 |
|------|------|----------|
| **LoginPage** | 높음 | 법인코드 필드 + 검색 모달 추가, 로그인 로직 변경 |
| **LoginRequest DTO** | 중간 | `entity_code` 필드 추가 (optional) |
| **AuthService.login()** | 높음 | entity_code 제공 시 해당 법인 사용자만 조회하도록 변경 |
| **AuthController** | 낮음 | 새 Public 엔드포인트 `/auth/entities/search` 추가 |
| **EntitySelectPage** | 중간 | selectToken 플로우 제거 (USER_LEVEL 더이상 안 옴), ADMIN 전용으로 변경 |
| **ForgotPasswordPage** | 중간 | 법인코드 입력 추가, 다법인 분기 로직 제거 가능 |
| **auth.service.ts (FE)** | 중간 | login()에 entity_code 파라미터 추가, 새 searchEntities() 메서드 |
| **i18n (3개 언어)** | 낮음 | 새 번역 키 추가 |

### 영향 없음

| 대상 | 이유 |
|------|------|
| RegisterPage | 초대 토큰 기반 가입이므로 법인코드 불필요 |
| ResetPasswordPage | 토큰 기반이므로 법인 정보 불필요 |
| EntitySelector (헤더) | 이미 USER_LEVEL은 return null 처리됨 |
| JWT/쿠키 시스템 | 변경 없음 |
| Mobile 앱 | 동일 API 사용, entity_code optional이므로 하위 호환 |
| refresh/logout | 변경 없음 |

### 보안 고려

| 항목 | 설명 | 대응 |
|------|------|------|
| 법인 검색 API 노출 | Public 엔드포인트로 법인 목록 검색 가능 | Rate limit 적용, 최소 2자 검색, 민감 정보(주소/대표자) 미포함 |
| 법인코드 브루트포스 | 법인코드를 바꿔가며 로그인 시도 | 기존 Rate limit (5회/60초) 유지 |
| 타법인 정보 차단 | USER_LEVEL이 다른 법인 계정 존재 여부 파악 | 에러 메시지 통일 ("잘못된 인증 정보") |

## 4. 구현 단계

### Phase 1: 백엔드 — 법인 검색 Public API

| 파일 | 작업 |
|------|------|
| `apps/api/src/domain/auth/controller/auth.controller.ts` | `GET /auth/entities/search` 엔드포인트 추가 |
| `apps/api/src/domain/auth/service/auth.service.ts` | `searchEntities(query)` 메서드 추가 |

**상세:**
- `GET /auth/entities/search?q=keyword` (Public, Rate limit 10회/60초)
- 검색: `ent_name ILIKE '%keyword%' OR ent_name_en ILIKE '%keyword%' OR ent_code ILIKE '%keyword%'`
- 응답: `{ entityId, code, name, nameEn, country }` (최소 정보만)
- 최소 2자 이상, 최대 10건 반환, ACTIVE 법인만

### Phase 2: 백엔드 — 로그인 로직 변경

| 파일 | 작업 |
|------|------|
| `apps/api/src/domain/auth/dto/request/login.request.ts` | `entity_code` optional 필드 추가 |
| `apps/api/src/domain/auth/service/auth.service.ts` | login() 메서드 변경: entity_code 기반 조회 |

**상세:**
```typescript
// LoginRequest에 추가
@IsOptional()
@IsString()
entity_code?: string;
```

**login() 로직 변경:**
```
if (entity_code 제공) {
  1. entity_code로 법인 조회 → entId 획득
  2. (email + usrCompanyId=entId)로 사용자 1건 조회
  3. 비밀번호 검증
  4. step: 'complete' → JWT 발급
}
else {
  // 기존 로직 유지 (ADMIN_LEVEL 호환)
  1. email로 사용자 다건 조회
  2. ADMIN_LEVEL → step: 'complete'
  3. USER_LEVEL 다건 → 에러 (법인코드 필수 안내)
     USER_LEVEL 1건 → step: 'complete'
}
```

## 8. 제거 대상 (클린업)

이번 변경으로 불필요해지는 코드:

1. **`POST /auth/select-entity` 엔드포인트** — selectToken 플로우 전체 제거
2. **`SelectEntityRequest` DTO** — 더 이상 사용하지 않음
3. **`ForgotPasswordEntityRequest` DTO** — entity_code 기반으로 통합
4. **`POST /auth/forgot-password/entity` 엔드포인트** — forgotPassword에 entity_code 통합
