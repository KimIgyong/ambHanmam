## 1. 현황 분석 (AS-IS)

### 사이트 관리 현재 구조
- **경로**: `/admin/site/`
- **레이아웃**: `SiteLayout.tsx` — 사이드바 + Outlet
- **현재 메뉴**: menus, pages, posts(Coming Soon), subscribers(Coming Soon)
- **통계**: 법인별 CMS 통계 (메뉴/페이지/게시글/구독자 수)만 존재
- **방문자 분석**: 전무 — GA 연동, 자체 트래킹 코드 없음

### 관련 시스템 현황
| 사이트 | URL | 용도 |
|--------|-----|------|
| 포털 사이트 | stg-www.amoeba.site | 고객 대상 CMS 사이트 (랜딩, 회원가입, 로그인, 구독) |
| AMB 앱 | stg-ama.amoeba.site | 내부 직원 SaaS 앱 (Entity별 업무) |

### 3.2 기능 상세

| 항목 | 설명 |
|------|------|
| GA Measurement ID | GA4 속성의 `G-XXXXXXXXXX` 형식 ID 입력 |
| 연동 상태 확인 | 입력된 ID로 GA API 접근 가능 여부 표시 |
| 사이트별 분리 | 포털 사이트 / AMB 앱 각각 별도 ID 설정 |
| 저장 위치 | `amb_system_settings` 테이블 (키: `ga_portal_measurement_id`, `ga_app_measurement_id`) |

### 3.3 데이터 수집 방식

**방안 A: GA4 gtag.js 삽입 (권장)**
- 포털 웹(`portal-web`)의 `index.html`에 gtag.js 스크립트 동적 삽입
- AMB 앱(`web`)의 `index.html`에 gtag.js 스크립트 동적 삽입
- GA Admin API로 통계 데이터 조회
- **장점**: 구글 인프라 활용, 정확한 방문자/세션/리퍼러 분석
- **단점**: GA API 서비스 계정 설정 필요, 데이터 지연(~24시간)

**방안 B: 자체 이벤트 로그 수집 + GA 병행**
- 로그인, 회원가입, 페이지뷰 등 핵심 이벤트를 자체 DB에 기록
- GA는 방문자 분석용, 자체 DB는 비즈니스 지표용
- **장점**: 실시간 데이터, GA 의존도 낮음
- **단점**: 추가 테이블/API 개발 필요

**권장**: **방안 B** — GA + 자체 이벤트 로그 병행  
(GA 데이터 지연이 있으므로, 로그인·회원가입 등 비즈니스 지표는 자체 수집)

### 5.2 신규 DB 테이블

#### `amb_event_logs` — 자체 이벤트 로그

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `evl_id` | UUID (PK) | 이벤트 ID |
| `evl_site` | VARCHAR(20) | 사이트 구분: `portal` / `app` |
| `evl_event_type` | VARCHAR(50) | 이벤트 타입: `login`, `register_page_view`, `subscription` |
| `evl_entity_id` | UUID (nullable) | Entity ID (앱 로그인 시) |
| `evl_user_id` | UUID (nullable) | 사용자 ID |
| `evl_ip_address` | VARCHAR(45) | 클라이언트 IP |
| `evl_user_agent` | TEXT | User Agent |
| `evl_referrer` | VARCHAR(500) | 리퍼러 URL |
| `evl_metadata` | JSONB | 추가 메타데이터 |
| `evl_created_at` | TIMESTAMP | 이벤트 발생 시각 |

#### `amb_system_settings` 확장 (기존 테이블 활용)

| 키 | 값 예시 | 설명 |
|----|--------|------|
| `ga_portal_measurement_id` | `G-XXXXXXXXXX` | 포털 GA Measurement ID |
| `ga_app_measurement_id` | `G-YYYYYYYYYY` | 앱 GA Measurement ID |
| `ga_service_account_json` | `{...}` | GA Data API 서비스 계정 JSON (암호화 저장) |

## 7. 구현 단계 제안

### Phase 1: 기반 구축 (GA 설정 + 자체 이벤트 로그)
1. `amb_event_logs` 테이블 생성 (마이그레이션)
2. `amb_system_settings`에 GA 설정 키 추가
3. 백엔드 `analytics` 모듈 생성 (controller, service, entity)
4. GA 설정 화면 (`SiteGaSettingsPage`) 구현
5. SiteLayout 사이드바에 메뉴 추가 + 라우트 등록

### Phase 2: 자체 이벤트 수집
1. 이벤트 로깅 API (`POST /analytics/events`) 구현
2. AMB 앱 로그인 시 이벤트 자동 기록 (기존 auth 모듈 hook)
3. 포털 로그인/회원가입 페이지 방문 이벤트 기록
4. 이벤트 조회 API 구현 (집계 쿼리)

### Phase 3: 통계 대시보드 UI
1. `SiteAnalyticsPage` 메인 화면 (탭, 기간 필터)
2. AMB 앱 탭 구현 (자체 DB 데이터 기반)
3. 포털 사이트 탭 구현 (자체 DB 데이터 기반)

### Phase 4: GA4 연동 (선택)
1. GA4 Data API 서비스 계정 설정
2. `ga-data.service.ts` — GA4 Data API 연동
3. 방문자 수, 리퍼러, 페이지뷰 데이터 GA에서 가져오기
4. GA 데이터 + 자체 DB 데이터 통합 표시

## 8. 고려사항

### 보안
- GA 서비스 계정 JSON은 암호화하여 DB 저장
- 이벤트 로그 API는 rate limiting 적용
- GA 설정은 `ADMIN` 권한 필요

### 성능
- 이벤트 로그 테이블에 `(evl_site, evl_event_type, evl_created_at)` 복합 인덱스
- 통계 조회 시 집계 쿼리 최적화 (날짜 범위 인덱스 활용)
- 대량 데이터 시 일별 집계 테이블(`amb_analytics_daily`) 추가 검토

### i18n
- 모든 UI 텍스트 번역 파일 사용 (`site` 네임스페이스 확장)
- 한국어, 영어, 베트남어 3개 언어 지원
