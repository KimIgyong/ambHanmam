## 2. 사전 고려 사항

### 2.1 핵심 리스크

| # | 리스크 | 심각도 | 대응 방안 |
|---|--------|--------|-----------|
| **R1** | JWT payload 필드 변경 → 기존 토큰 무효화 → 전 사용자 강제 로그아웃 | 🔴 HIGH | `tokenVersion` 증가로 제어 + 배포 공지 |
| **R2** | DB 컬럼 리네임 시 다운타임 | 🟡 MED | ALTER TABLE RENAME COLUMN은 메타데이터만 변경 → 즉시 완료 (PostgreSQL) |
| **R3** | 코드-DB 간 불일치 가능성 | 🔴 HIGH | DB 변경 → 코드 배포를 원자적으로 처리. 하위 호환 과도기 설정 |
| **R4** | 타입 변경 전파 → import 깨짐 | 🟡 MED | packages → api → web 순서로 빌드, 단계별 에러 확인 |
| **R5** | 외부 API 연동이 group 필드에 의존 | 🟡 MED | 현재 외부 연동 없음 확인. 향후 추가 시 API 버전 분리 |

### 2.2 배포 순서 (Critical Path)

```
① DB 마이그레이션 (컬럼 리네임 + 값 변경)
    ↓ (이 시점에서 기존 코드와 불일치 발생 → 서비스 중단 최소화)
② 코드 배포 (패키지 → API → Web 순서로 빌드)
    ↓
③ JWT 토큰 무효화 (tokenVersion 증가)
    ↓
④ 전 사용자 재로그인 필요
```

> ⚠️ ①~② 사이 간격을 최소화해야 합니다. 또는 하위 호환 코드를 먼저 배포 후 DB 변경하는 방식도 가능합니다 ([하위 호환 전략](#하위-호환-전략) 참고).

- [ ] 스테이징 환경에서 전체 테스트 완료
- [ ] 배포 시간대 공지 (업무 외 시간 권장)

## Phase 5: DB 마이그레이션

### 5.1 마이그레이션 스크립트

```sql
-- =============================================
-- 사용자 레벨 코드 리네이밍 마이그레이션
-- 실행 환경: PostgreSQL 15
-- 대상 DB: db_amb
-- =============================================

-- 트랜잭션으로 원자적 실행
BEGIN;

-- ── 1. amb_users 테이블 ──

-- 1-1. 값 변경 (컬럼 리네임 전에 실행)
UPDATE amb_users SET usr_group_code = 'ADMIN_LEVEL' WHERE usr_group_code = 'ADMIN_GROUP';
UPDATE amb_users SET usr_group_code = 'USER_LEVEL'  WHERE usr_group_code = 'USER_GROUP';

-- 1-2. 기본값 변경
ALTER TABLE amb_users ALTER COLUMN usr_group_code SET DEFAULT 'USER_LEVEL';

-- 1-3. 컬럼 리네임 (메타데이터 변경만, 즉시 완료)
ALTER TABLE amb_users RENAME COLUMN usr_group_code TO usr_level_code;

-- ── 2. amb_invitations 테이블 ──

-- 2-1. 값 변경
UPDATE amb_invitations SET inv_group_code = 'ADMIN_LEVEL' WHERE inv_group_code = 'ADMIN_GROUP';
UPDATE amb_invitations SET inv_group_code = 'USER_LEVEL'  WHERE inv_group_code = 'USER_GROUP';

-- 2-2. 컬럼 리네임
ALTER TABLE amb_invitations RENAME COLUMN inv_group_code TO inv_level_code;

-- ── 3. 검증 ──
-- 변경 결과 확인:
-- SELECT usr_level_code, COUNT(*) FROM amb_users GROUP BY usr_level_code;
-- SELECT inv_level_code, COUNT(*) FROM amb_invitations GROUP BY inv_level_code;

COMMIT;
```

### 5.2 PostgreSQL RENAME COLUMN 특성

- `ALTER TABLE ... RENAME COLUMN`은 **시스템 카탈로그만 수정** → 테이블 리빌드 없음
- 실행 시간: 밀리초 단위 (데이터 양 무관)
- **인덱스**: 컬럼명 참조 인덱스가 있으면 자동 반영됨
- **VIEW**: 해당 컬럼을 참조하는 VIEW가 있으면 **깨짐** → 확인 필요
- **TypeORM**: `@Column({ name: ... })` 데코레이터의 name만 맞으면 OK

### 5.3 확인해야 할 의존성

```sql
-- VIEW 의존성 확인
SELECT table_name, column_name
FROM information_schema.columns
WHERE column_name IN ('usr_group_code', 'inv_group_code')
  AND table_schema = 'public';

-- 인덱스 확인
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename IN ('amb_users', 'amb_invitations')
  AND indexdef LIKE '%group_code%';
```

## Phase 6: JWT 토큰 무효화

### 6.1 문제

JWT payload에 `group` 필드가 포함되어 있음:

```json
{
  "sub": "user-uuid",
  "email": "user@example.com",
  "group": "ADMIN_GROUP",   ← 변경 후: "level": "ADMIN_LEVEL"
  "role": "SUPER_ADMIN",
  ...
}
```

변경 후 기존 토큰으로 로그인 시:
- 서버는 `payload.level`을 읽지만, 기존 토큰에는 `payload.group`만 존재
- `payload.level === undefined` → `ADMIN_LEVEL`도 `USER_LEVEL`도 아님 → **권한 판정 오류**

### 6.2 대응 방안

**방안 A: tokenVersion 증가 (권장)**

```sql
-- 배포 직후 실행
UPDATE amb_users SET usr_token_version = usr_token_version + 1;
```

→ 기존 토큰의 `tokenVersion`과 DB 불일치 → 자동 무효화 → 재로그인 필요

**방안 B: 하위 호환 로직 (2주 과도기)**

```typescript
// jwt-payload.interface.ts (과도기)
export interface JwtPayload {
  level: string;
  /** @deprecated Use level instead */
  group?: string;
}

// guard/interceptor에서
const userLevel = user.level || user.group; // 기존 토큰도 인식
```

→ 재로그인 없이 전환 가능하지만, 하위 호환 코드 2주 후 정리 필요

**권장**: **방안 A + 배포 공지**  
(업무 외 시간 배포 + "보안 업데이트로 인해 재로그인이 필요합니다" 공지)

## 하위 호환 전략

### 옵션 1: Big-bang 배포 (권장 — 소규모 사용자)

```
DB 마이그레이션 → 코드 배포 → JWT 무효화  (동시에, 점검 시간 5분)
```

- **장점**: 깔끔한 전환, 하위 호환 코드 불필요
- **단점**: 전 사용자 강제 로그아웃
- **적합**: 현재 사용자 12명 → 영향 최소

### 옵션 2: 점진적 전환 (대규모 사용자일 경우)

```
Phase A: 하위 호환 코드 배포 (group/level 양쪽 모두 인식)
    ↓ (1주일 운영)
Phase B: DB 마이그레이션 (값 변경 + 컬럼 리네임)
    ↓ (1주일 운영)
Phase C: 하위 호환 코드 제거 (level만 인식)
```

- **장점**: 중단 없음, 재로그인 불필요
- **단점**: 3단계 배포, 하위 호환 코드 관리 필요

### 권장

현재 사용자 12명이므로 **옵션 1 (Big-bang)** 권장.

### 롤백 시나리오: 배포 후 심각한 문제 발견

```bash
# 1. 코드 롤백
ssh amb-staging "cd ~/ambManagement && git revert HEAD && bash docker/staging/deploy-staging.sh"

# 2. DB 롤백
ssh amb-staging "docker exec amb-postgres-staging psql -U amb_user -d db_amb <<'SQL'
BEGIN;
ALTER TABLE amb_users RENAME COLUMN usr_level_code TO usr_group_code;
UPDATE amb_users SET usr_group_code = 'ADMIN_GROUP' WHERE usr_group_code = 'ADMIN_LEVEL';
UPDATE amb_users SET usr_group_code = 'USER_GROUP'  WHERE usr_group_code = 'USER_LEVEL';
ALTER TABLE amb_users ALTER COLUMN usr_group_code SET DEFAULT 'USER_GROUP';

ALTER TABLE amb_invitations RENAME COLUMN inv_level_code TO inv_group_code;
UPDATE amb_invitations SET inv_group_code = 'ADMIN_GROUP' WHERE inv_group_code = 'ADMIN_LEVEL';
UPDATE amb_invitations SET inv_group_code = 'USER_LEVEL'  WHERE inv_group_code = 'USER_LEVEL';
COMMIT;
SQL"

# 3. JWT 무효화 (다시 한 번)
ssh amb-staging "docker exec amb-postgres-staging psql -U amb_user -d db_amb -c \"UPDATE amb_users SET usr_token_version = usr_token_version + 1;\""
```
