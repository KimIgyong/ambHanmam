## 1. 시스템 현황 분석 요약

### 현재 시스템 상태
- `amb_attendances` 테이블: Note/메모 컬럼 없음, 변경 이력 없음
- `validateWeekday()`: 토/일 등록 차단 (E12006)
- 수정 권한: 본인만 수정/삭제 가능 (`entity.usrId !== userId` 체크)
- 역할 체계: USER_ROLE에 'HR' 없음 → **MASTER + MANAGER** 를 HR 역할로 사용

### 역할 결정
- `USER_ROLE`에 `HR`이 정의되어 있지 않음
- MASTER(법인 관리자) + MANAGER(부서 관리자)를 Amendment 권한 보유자로 설정

## 2. 단계별 구현 계획

### Phase 1: DB 스키마 + 엔티티 (백엔드)
- **`amb_attendance_amendments`** 테이블 + 엔티티 생성
- SQL 마이그레이션 파일 작성

### Phase 2: 주말 등록 허용
- **백엔드**: `validateWeekday()` 제거
- **프론트엔드**: `isDateEditable()` 수정 불필요 (주말은 이미 날짜 범위 내이면 통과)
- `handleCellClick`에서 주말 제한 없으므로 별도 수정 불필요

### Phase 3: Backend Amendment 구현
- DTO: `AmendAttendanceRequest` 생성
- Service: `amendAttendance()`, `getAmendments()` 메서드
- Controller: `POST /:id/amend`, `GET /:id/amendments` 엔드포인트
- Mapper: Amendment 매핑
- 에러코드: E12007~E12009 추가

### Phase 4: 공유 타입 확장
- `AttendanceAmendmentResponse` 인터페이스 추가
- `AttendanceResponse`에 `amendments` optional 필드 추가
- Mapper에서 amendments join + 매핑

### Phase 5: Frontend Amendment 구현
- Service: `amendAttendance()`, `getAmendments()` API 메서드
- Hook: `useAmendAttendance` mutation
- AttendancePage: MASTER/MANAGER 셀 클릭 시 Amendment 모달 표시
- **Amendment 표시**: 셀 클릭 시 원본 + 수정이력 표시
- i18n: 3개 언어 번역 키 추가

### Phase 6: 빌드 검증

## 3. 사이드 임팩트
- 기존 개인 수정/삭제 플로우 영향 없음 (Amendment는 별도 경로)
