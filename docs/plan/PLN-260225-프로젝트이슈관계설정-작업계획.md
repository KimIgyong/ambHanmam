## 1. 작업 개요

### 1.1 목적
요구사항 분석서(R1~R8)에 기반하여 현행 AMB Management 시스템의 프로젝트-이슈 관계 설정, 권한 체계 개선, 이슈 타입 확장, 레드마인 마이그레이션 호환 스키마를 구현한다.

### 1.2 작업 범위 요약

| Phase | 범위 | 예상 소요 | 요구사항 |
|-------|------|----------|---------|
| Phase 1 | DB 스키마 + 백엔드 핵심 변경 | 2일 | R3, R4, R5, R7 |
| Phase 2 | 백엔드 권한·API 확장 | 1일 | R2, R4, R6 |
| Phase 3 | 프론트엔드 UI 개선 | 2일 | R1, R4, R5, R6, R7 |
| Phase 4 | 레드마인 마이그레이션 | 2일 | R8 |

> R1(모든 사용자 프로젝트 제안)은 현행 시스템에서 이미 충족되어 코드 변경 불필요.

### 2.1 작업 1-1: SQL 마이그레이션 스크립트 작성

**파일**: `sql/migration_project_issue_relation.sql`

```sql
-- ============================================
-- Migration: 프로젝트-이슈 관계 설정 및 이슈 타입 확장
-- Date: 2026-02-25
-- Req: R3, R4, R5, R7, R8
-- ============================================

BEGIN;

-- [R5] 이슈 테이블에 프로젝트 FK 추가
ALTER TABLE amb_issues
  ADD COLUMN IF NOT EXISTS pjt_id UUID NULL;

ALTER TABLE amb_issues
  ADD CONSTRAINT fk_issues_project
  FOREIGN KEY (pjt_id) REFERENCES kms_projects(pjt_id)
  ON DELETE SET NULL;

CREATE INDEX IF NOT EXISTS idx_amb_issues_pjt_id ON amb_issues(pjt_id);

-- [R8] 레드마인 마이그레이션 호환 컬럼 (이슈)
ALTER TABLE amb_issues
  ADD COLUMN IF NOT EXISTS iss_redmine_id INT NULL,
  ADD COLUMN IF NOT EXISTS iss_start_date DATE NULL,
  ADD COLUMN IF NOT EXISTS iss_due_date DATE NULL,
  ADD COLUMN IF NOT EXISTS iss_done_ratio INT DEFAULT 0;

ALTER TABLE amb_issues
  ADD CONSTRAINT chk_done_ratio CHECK (iss_done_ratio BETWEEN 0 AND 100);

CREATE UNIQUE INDEX IF NOT EXISTS idx_issues_redmine_id
  ON amb_issues(iss_redmine_id) WHERE iss_redmine_id IS NOT NULL;

CREATE INDEX IF NOT EXISTS idx_issues_dates
  ON amb_issues(iss_start_date, iss_due_date);

-- [R7] 이슈 타입 마이그레이션: FEATURE/ENHANCEMENT → FEATURE_REQUEST
UPDATE amb_issues SET iss_type = 'FEATURE_REQUEST' WHERE iss_type IN ('FEATURE', 'ENHANCEMENT');

-- [R8] 프로젝트 테이블 확장 (레드마인 호환 + 계층)
ALTER TABLE kms_projects
  ADD COLUMN IF NOT EXISTS pjt_redmine_id INT NULL,
  ADD COLUMN IF NOT EXISTS pjt_parent_id UUID NULL;

ALTER TABLE kms_projects
  ADD CONSTRAINT fk_projects_parent
  FOREIGN KEY (pjt_parent_id) REFERENCES kms_projects(pjt_id)
  ON DELETE SET NULL;

CREATE UNIQUE INDEX IF NOT EXISTS idx_projects_redmine_id
  ON kms_projects(pjt_redmine_id) WHERE pjt_redmine_id IS NOT NULL;

CREATE INDEX IF NOT EXISTS idx_projects_parent
  ON kms_projects(pjt_parent_id);

COMMIT;
```

**롤백 스크립트**: `sql/rollback_project_issue_relation.sql`

```sql
BEGIN;

-- 이슈 타입 롤백 (FEATURE_REQUEST → FEATURE)
UPDATE amb_issues SET iss_type = 'FEATURE' WHERE iss_type = 'FEATURE_REQUEST';

-- 이슈 테이블 컬럼 제거
ALTER TABLE amb_issues
  DROP CONSTRAINT IF EXISTS chk_done_ratio,
  DROP CONSTRAINT IF EXISTS fk_issues_project;

DROP INDEX IF EXISTS idx_amb_issues_pjt_id;
DROP INDEX IF EXISTS idx_issues_redmine_id;
DROP INDEX IF EXISTS idx_issues_dates;

ALTER TABLE amb_issues
  DROP COLUMN IF EXISTS pjt_id,
  DROP COLUMN IF EXISTS iss_redmine_id,
  DROP COLUMN IF EXISTS iss_start_date,
  DROP COLUMN IF EXISTS iss_due_date,
  DROP COLUMN IF EXISTS iss_done_ratio;

-- 프로젝트 테이블 컬럼 제거
ALTER TABLE kms_projects
  DROP CONSTRAINT IF EXISTS fk_projects_parent;

DROP INDEX IF EXISTS idx_projects_redmine_id;
DROP INDEX IF EXISTS idx_projects_parent;

ALTER TABLE kms_projects
  DROP COLUMN IF EXISTS pjt_redmine_id,
  DROP COLUMN IF EXISTS pjt_parent_id;

COMMIT;
```

```typescript
async approveProject(id: string, entityId: string, reviewerId: string, comment?: string): Promise<ProjectResponse> {
  const project = await this.projectRepo.findOne({ where: { pjtId: id, entId: entityId } });
  if (!project) throw new NotFoundException(ERROR_CODE.PROJECT_NOT_FOUND.message);

  // SUBMITTED → REVIEW → APPROVED 또는 REVIEW → APPROVED
  const currentStatus = project.pjtStatus;
  let targetStatus = 'APPROVED';

  if (currentStatus === 'SUBMITTED') {
    targetStatus = 'REVIEW';
  } else if (currentStatus !== 'REVIEW') {
    throw new BadRequestException(ERROR_CODE.PROJECT_INVALID_TRANSITION.message);
  }

  project.pjtStatus = targetStatus;
  if (targetStatus === 'APPROVED') project.pjtApprovedAt = new Date();
  await this.projectRepo.save(project);

  // REVIEW 단계를 거친 경우, 바로 APPROVED까지 진행 가능한지는 비즈니스 결정
  // 여기서는 SUBMITTED → APPROVED (REVIEW 스킵) 허용
  if (currentStatus === 'SUBMITTED') {
    project.pjtStatus = 'APPROVED';
    project.pjtApprovedAt = new Date();
    await this.projectRepo.save(project);
  }

  return this.findById(id, entityId);
}

async rejectProject(id: string, entityId: string, reviewerId: string, comment?: string): Promise<ProjectResponse> {
  const project = await this.projectRepo.findOne({ where: { pjtId: id, entId: entityId } });
  if (!project) throw new NotFoundException(ERROR_CODE.PROJECT_NOT_FOUND.message);

  const allowed = VALID_TRANSITIONS[project.pjtStatus] || [];
  if (!allowed.includes('REJECTED')) {
    throw new BadRequestException(ERROR_CODE.PROJECT_INVALID_TRANSITION.message);
  }

  project.pjtStatus = 'REJECTED';
  project.pjtRejectedAt = new Date();
  await this.projectRepo.save(project);

  return this.findById(id, entityId);
}
```

```sql
  redmine_user_id INT PRIMARY KEY,
  amb_user_id UUID NOT NULL REFERENCES amb_users(usr_id),
  redmine_login VARCHAR(100),
  redmine_email VARCHAR(200),
  mapped_at TIMESTAMPTZ DEFAULT NOW()
);

-- 마이그레이션 로그
CREATE TABLE IF NOT EXISTS amb_migration_logs (
  mgl_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  mgl_batch_id UUID NOT NULL,
  mgl_source VARCHAR(20) NOT NULL DEFAULT 'REDMINE',
  mgl_entity_type VARCHAR(20) NOT NULL,
  mgl_source_id INT NOT NULL,
  mgl_target_id UUID NULL,
  mgl_status VARCHAR(20) NOT NULL DEFAULT 'PENDING',
  mgl_error_message TEXT NULL,
  mgl_created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_migration_logs_source
  ON amb_migration_logs(mgl_source, mgl_entity_type, mgl_source_id);

CREATE INDEX IF NOT EXISTS idx_migration_logs_batch
  ON amb_migration_logs(mgl_batch_id);

COMMIT;
```

## 7. 사이드 임팩트 및 리스크

### 7.1 하위 호환 주의사항

| 항목 | 리스크 | 대응 |
|------|--------|------|
| **IssueType 변경** | 프론트엔드에서 `FEATURE`, `ENHANCEMENT`을 하드코딩한 곳이 있으면 에러 | P1에서 DB 마이그레이션과 타입 변경을 동시 배포 |
| **ProjectResponse 타입 확장** | 신규 필드가 TypeScript 컴파일 에러를 유발할 수 있음 | 매퍼를 먼저 수정하여 신규 필드 반환 보장 |
| **법인 필터 제거** | 프로젝트 수가 많아지면 목록 조회 성능 저하 | 페이지네이션 강화 (기본 50건 유지), 인덱스 활용 |
| **순환 참조** | ProjectEntity ↔ IssueEntity 순환 import | TypeORM 팩토리 패턴 `() => Entity` 사용 (기존과 동일 방식) |

### 7.2 배포 순서 주의

```
1. DB 마이그레이션 실행 (sql/migration_project_issue_relation.sql)
2. API 배포 (백엔드 코드 변경)
3. Web 배포 (프론트엔드 코드 변경)
```

DB 마이그레이션은 **nullable 컬럼 추가**와 **데이터 UPDATE**이므로, 기존 API가 동작하는 상태에서도 안전하게 실행 가능. API 배포 실패 시에도 기존 기능에 영향 없음.
