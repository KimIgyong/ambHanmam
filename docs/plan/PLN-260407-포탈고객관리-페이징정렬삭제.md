## 2. 단계별 구현 계획

### Phase 1: 백엔드 — 정렬/국가필터/삭제 API 추가

#### 2.1.1 DTO 확장 (`portal-customer-query.request.ts`)

| 추가 필드 | 타입 | 설명 |
|-----------|------|------|
| `sort_by` | `'name' \| 'email' \| 'company' \| 'country' \| 'created_at'` | 정렬 컬럼 (기본: `created_at`) |
| `sort_order` | `'ASC' \| 'DESC'` | 정렬 방향 (기본: `DESC`) |
| `country` | `string` | 국가 코드 필터 |

#### 2.1.2 서비스 수정 (`portal-bridge.service.ts`)

- `findPortalCustomers()`: 
  - `country` 필터: `c.pctCountry = :country`
  - `sort_by` → 컬럼 매핑 후 `orderBy` 동적 적용
  - **sort_by 안전 검증**: 허용된 컬럼명 화이트리스트로 SQL Injection 방지

- 신규 메서드 `getDistinctCountries()`:
  - `SELECT DISTINCT pct_country FROM amb_svc_portal_customers WHERE pct_deleted_at IS NULL AND pct_country IS NOT NULL`

- 신규 메서드 `deletePortalCustomer(pctId: string, adminUserId: string)`:
  1. 고객 존재 확인
  2. 활성 매핑 조회 → 있으면 전부 REVOKE (pumStatus='REVOKED', pumRevokedBy, pumRevokedAt)
  3. `pct_deleted_at = NOW()` 소프트 삭제 (createQueryBuilder UPDATE 사용 — 읽기전용 엔티티 우회)
  4. 삭제 결과 반환

#### 2.1.3 컨트롤러 확장 (`portal-bridge.controller.ts`)

| 신규 엔드포인트 | 메서드 | 설명 |
|----------------|--------|------|
| `GET /portal-bridge/customers/countries` | `getCountries()` | 국가 코드 목록 조회 |
| `DELETE /portal-bridge/customers/:pctId` | `deleteCustomer()` | 포탈 고객 소프트 삭제 |

두 엔드포인트 모두 `@AdminOnly()` 적용.

#### 사이드 임팩트
- `findPortalCustomers`: 기존 정렬 기본값 유지 (`created_at DESC`) → **기존 동작 변경 없음**
- 새 엔드포인트 추가만이므로 기존 API 호환성 유지
- 읽기전용 엔티티에 직접 UPDATE → `createQueryBuilder` 사용으로 TypeORM soft delete 제한 우회

## 4. 사이드 임팩트 분석

### 4.1 백엔드

| 변경 | 영향 | 위험도 | 대응 |
|------|------|--------|------|
| DTO sort_by/sort_order/country 추가 | 기존 API는 필드 미전송 시 기본값 사용 → **호환성 유지** | 낮음 | `@IsOptional()` + 기본값 |
| 삭제 API 추가 | 새 엔드포인트 → 기존 영향 없음 | 낮음 | `@AdminOnly()` 보호 |
| 매핑 자동 REVOKE | 삭제 시 활성 매핑 해제 → 내부 사용자 계정은 유지 | 중간 | 트랜잭션 처리, 로그 기록 |
| 읽기전용 엔티티에 UPDATE 수행 | TypeORM 패턴 우회 | 낮음 | `createQueryBuilder().update()` 직접 사용 |

### 4.2 프론트엔드

| 변경 | 영향 | 위험도 | 대응 |
|------|------|--------|------|
| PortalCustomersTab UI 변경 | 필터/정렬 추가로 UI 변경 | 낮음 | 기존 레이아웃 유지, 필터만 추가 |
| 삭제 버튼 + confirm | 새 UI 요소 | 낮음 | InternalUsersTab 패턴 참고 |
| i18n 키 추가 | 3개 언어 파일 수정 | 낮음 | 기존 JSON 구조에 추가만 |

### 4.3 portal-api 영향

| 변경 | 영향 |
|------|------|
| `pct_deleted_at` 설정 | portal-api의 `PortalCustomerEntity`는 `@DeleteDateColumn`으로 pctDeletedAt 관리 → **TypeORM이 자동으로 deleted 레코드 필터링** → 삭제된 고객은 포탈 로그인 시 `findOne`에서 자동 제외 |
| 포탈 회원가입 | 삭제된 이메일로 재가입 시 unique 제약 확인 필요 → `pct_email` unique index가 `WHERE pct_deleted_at IS NULL` 조건부인지 확인 |

## 5. DB 마이그레이션

**필요 없음** — 기존 `pct_deleted_at`, `pct_country`, `pum_status` 컬럼이 이미 존재. 새 테이블/컬럼 추가 불필요.

단, `pct_email` unique 인덱스가 soft delete를 고려하지 않은 경우 확인 필요:
```sql
-- 현재 인덱스 확인 (스테이징에서 검증)
SELECT indexname, indexdef 
FROM pg_indexes 
WHERE tablename = 'amb_svc_portal_customers' 
  AND indexdef LIKE '%pct_email%';
```
