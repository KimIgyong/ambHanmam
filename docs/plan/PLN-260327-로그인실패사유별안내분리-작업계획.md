## 1. 시스템 개발 현황 분석

### 1.1 백엔드 인증 아키텍처 (현재)

| 구분 | 서비스 | 예외 처리 방식 | 에러 코드 | 로그인 실패 횟수 추적 |
|------|--------|-------------|----------|------------------|
| **메인 Auth** | `AuthService.login()` | `BusinessException` | E1002 일괄 | ✅ (5회 → 15분 잠금) |
| **클라이언트 Auth** | `ClientAuthService.login()` | `UnauthorizedException` | 없음 | ❌ |
| **파트너 Auth** | `PartnerAuthService.login()` | `UnauthorizedException` | 없음 | ❌ |

#### 메인 Auth 로그인 흐름 (auth.service.ts)
```
login(entity_code?, email, password)
├─ entity_code 있음 (법인코드 지정 로그인)
│   ├─ entity 조회 → 없으면 E1002
│   ├─ user 조회 (email + companyId) → 없으면 adminUser fallback
│   ├─ !user || bcrypt 실패 → E1002 (⚠️ user/password 구분 안 됨)
│   ├─ checkAccountLocked → E1020
│   ├─ WITHDRAWN/SUSPENDED/INACTIVE → E1002
│   └─ 성공 → generateTokens
└─ entity_code 없음 (ADMIN/단일법인 로그인)
    ├─ 모든 email 매칭 users 조회 → 없으면 E1002
    ├─ 각 user bcrypt 검증 → 매칭 0명이면 모든 user 실패 기록 + E1002
    ├─ ADMIN_LEVEL → 바로 JWT
    ├─ 단일 USER_LEVEL → 바로 JWT
    └─ 다중 USER_LEVEL → E1010 (법인 선택 필요)
```

**핵심 문제점**: `if (!user || !(await bcrypt.compare(...)))` 에서 user 미존재와 비밀번호 불일치가 동일 분기로 처리됨.

#### 클라이언트 Auth 로그인 흐름 (client-auth.service.ts)
```
login(entityCode, email, password)
├─ entity 조회 → 없으면 UnauthorizedException('Invalid credentials')
├─ user 조회 (email + CLIENT_LEVEL) → client.cliEntId ≠ entity.entId → UnauthorizedException
├─ bcrypt 검증 → 실패 → UnauthorizedException('Invalid credentials')
├─ 비활성 계정 → UnauthorizedException('Account is not active')
└─ 성공 → generateTokens
```

#### 파트너 Auth 로그인 흐름 (partner-auth.service.ts)
```
login(partnerCode, email, password)
├─ partner 조회 → 없으면 UnauthorizedException('Invalid credentials')
├─ user 조회 (email + PARTNER_LEVEL + partnerId) → 없으면 UnauthorizedException
├─ bcrypt 검증 → 실패 → UnauthorizedException('Invalid credentials')
├─ 비활성 계정 → UnauthorizedException('Account is not active')
└─ 성공 → generateTokens
```

### 1.2 에러 코드 체계 현황

```typescript
// E1xxx: Authentication/Authorization (현재 13개)
E1001 UNAUTHORIZED       - 인증 필요
E1002 INVALID_CREDENTIALS - ⚠️ 모든 로그인 실패 통합
E1003 TOKEN_EXPIRED       - 토큰 만료
E1004 INVALID_TOKEN       - 유효하지 않은 토큰
E1005 FORBIDDEN           - 접근 거부
E1006 INSUFFICIENT_ROLE   - 역할 부족
E1007 RESET_TOKEN_EXPIRED - 리셋 토큰 만료
E1008 RESET_TOKEN_INVALID - 리셋 토큰 무효
E1009 CURRENT_PASSWORD_WRONG - 현재 비밀번호 틀림
E1010 ACCOUNT_PENDING     - 계정 승인 대기 (다중 법인)
E1011 MUST_CHANGE_PASSWORD - 비밀번호 변경 필요
E1012 INSUFFICIENT_LEVEL  - 레벨 부족
E1013 INSUFFICIENT_ROLE_FOR_ENTITY - 법인 접근 역할 부족
// E1014~E1019 미사용 → 신규 할당 가능
// E1020 계정 잠금 (코드에 하드코딩, ERROR_CODE 상수에 미등록)
```

### 1.3 프론트엔드 에러 처리 현황

| 페이지 | API 서비스 | 에러 접근 경로 | 현재 동작 |
|--------|-----------|-------------|---------|
| LoginPage | `authService.login()` | `axiosErr.response?.data?.error?.message` | E1010이면 entityCodeRequired, 그외 loginFailed |
| AdminLoginPage | `authService.login()` | 없음 (catch all) | `t('adminLogin.errorInvalidCredentials')` 고정 |
| ClientLoginPage | `clientPortalApiService.login()` | `error.response?.data?.error?.message` | 현재 `UnauthorizedException` 응답은 `{statusCode,message,error}` 형식이라 `.error?.message`가 undefined → fallback `'Login failed'` |
| PartnerLoginPage | `partnerPortalApiService.login()` | `error.response?.data?.error?.message` | 동일: undefined → `'Login failed'` |

### 1.4 응답 포맷 불일치 (핵심 사이드 임팩트)

**NestJS 기본 UnauthorizedException 응답:**
```json
{ "statusCode": 401, "message": "Invalid credentials", "error": "Unauthorized" }
```

**BusinessException (via BusinessExceptionFilter) 응답:**
```json
{ "success": false, "data": null, "error": { "code": "E1002", "message": "Invalid email or password." }, "timestamp": "..." }
```

⚠️ Client/Partner 로그인의 `UnauthorizedException` → `BusinessException` 전환 시:
- `error.response.data.error`가 문자열 `"Unauthorized"` → 객체 `{ code, message }`로 변경
- 프론트엔드의 `error.response?.data?.error?.message` 접근이 **기존에는 undefined**였으나 **변경 후에는 실제 메시지를 반환** → **오히려 개선됨**
- Client/Partner 포탈의 다른 API(`changePassword`, `register` 등)에서 `UnauthorizedException`/`BadRequestException` 사용하는 곳도 있으나, 이들은 **이번 변경 대상이 아님** (login 메서드만 변경)

## 2. 사이드 임팩트 분석

### 2.1 🔴 높음 - Client/Partner UnauthorizedException → BusinessException 전환

| 항목 | 상세 |
|------|------|
| **변경** | `client-auth.service.ts`, `partner-auth.service.ts`의 login() 메서드에서 `UnauthorizedException` → `BusinessException` 변경 |
| **영향 범위** | Client/Partner 로그인 API 응답 포맷 변경 |
| **기존 응답** | `{ statusCode: 401, message: "...", error: "Unauthorized" }` |
| **변경 응답** | `{ success: false, data: null, error: { code: "E1014", message: "..." }, timestamp: "..." }` |
| **프론트 영향** | `error.response?.data?.error?.message` — 기존: undefined (error는 문자열 "Unauthorized"), 변경 후: 실제 메시지 반환 → **개선 방향** |
| **완화 전략** | 프론트엔드 에러 파싱과 동시 배포. 단, 기존 fallback `'Login failed'`가 있으므로 프론트 미배포 시에도 치명적 오류는 없음 |
| **⚠️ 주의** | login() **이외** 메서드의 `UnauthorizedException`/`BadRequestException`은 **변경하지 않음** → 응답 포맷 혼재 발생. 하지만 login 외 엔드포인트는 인증 후 호출이므로 에러 UI가 다르고 영향 미미 |

### 2.2 🟡 중간 - 보안: 이메일 존재 여부 노출

| 항목 | 상세 |
|------|------|
| **변경** | 로그인 실패 시 E1014(이메일 미존재) vs E1015(비밀번호 불일치) 분리 |

  ```typescript
  import type { TFunction } from 'i18next';
  
  export function getLoginErrorMessage(error: unknown, t: TFunction, ns?: string): string {
    const code = (error as any)?.response?.data?.error?.code;
    const prefix = ns ? `${ns}:` : '';
    switch (code) {
      case 'E1014': return t(`${prefix}loginErrorEmailNotFound`);
      case 'E1015': return t(`${prefix}loginErrorPasswordIncorrect`);
      case 'E1020': return t(`${prefix}loginErrorAccountLocked`);
      default:      return t(`${prefix}loginFailed`);
    }
  }
  ```

### Phase 3: Reset Password — Entity Name 표시

#### Step 3.1: verify-reset-token API 추가 (백엔드)
- **파일 1**: `apps/api/src/domain/auth/service/auth.service.ts`
  ```typescript
  async verifyResetToken(token: string): Promise<{
    email: string;
    entityName: string | null;
  }> {
    const resetEntity = await this.passwordResetRepository.findOne({
      where: { prsToken: token },
      relations: ['user', 'user.company'],
    });
    
    if (!resetEntity || resetEntity.prsUsedAt) {
      throw new BusinessException(ERROR_CODE.RESET_TOKEN_INVALID...);
    }
    if (new Date() > resetEntity.prsExpiresAt) {
      throw new BusinessException(ERROR_CODE.RESET_TOKEN_EXPIRED...);
    }
    
    const maskedEmail = this.maskEmail(resetEntity.user.usrEmail);
    return {
      email: maskedEmail,
      entityName: resetEntity.user.company?.entName || null,
    };
  }
  
  private maskEmail(email: string): string {
    const [local, domain] = email.split('@');
    const masked = local.length <= 2 
      ? local[0] + '***' 
      : local.slice(0, 2) + '***' + local.slice(-1);
    return `${masked}@${domain}`;
  }
  ```

- **파일 2**: `apps/api/src/domain/auth/controller/auth.controller.ts`
  ```typescript

    return { success: true, data, timestamp: new Date().toISOString() };
  }
  ```

## 5. 배포 전략

### 5.1 배포 순서 (중요)

**반드시 백엔드 먼저 배포 후 프론트엔드 배포**:
1. 백엔드에 E1014, E1015 에러 코드가 추가되어도, 프론트엔드 미배포 시 기존 `default` fallback으로 `loginFailed` 표시 → 안전
2. 프론트엔드만 먼저 배포되면 E1014/E1015 코드를 받지 못해 항상 fallback 메시지 표시 → 의미 없지만 오류도 아님
3. **권장**: 동시 배포 (`deploy-staging.sh`가 API + Web 함께 빌드/배포하므로 문제 없음)

### 5.2 롤백 계획

- 백엔드: `E1002` fallback이 기존 코드에 유지되므로, 프론트엔드만 롤백해도 안전
- 프론트엔드: 에러 유틸의 `default` 케이스가 항상 기존 메시지로 fallback

## 6. 주의사항 및 체크리스트

- [ ] Client/Partner login()에서 `UnauthorizedException` → `BusinessException` 변경 시 **login() 메서드만** 변경. changePassword, register 등 다른 메서드의 Exception은 변경하지 않음
- [ ] 비활성 계정(WITHDRAWN/SUSPENDED/INACTIVE)은 E1002 유지 — 계정 상태 노출 방지
- [ ] Entity/Partner Code 미존재도 E1002 유지 — 코드 유효성 노출 방지
- [ ] forgot-password는 기존대로 항상 성공 응답 (이메일 존재 여부 비노출)
- [ ] verify-reset-token은 마스킹된 이메일 반환 (전체 이메일 비노출)
- [ ] Rate Limiting 설정 유지/추가: 신규 엔드포인트에 `@Throttle` 적용
- [ ] admin.json에 `forgotPassword` 링크 관련 번역 키 추가 시 기존 키와 충돌 없는지 확인
- [ ] VITE 환경변수 관련 없음 (순수 API/UI 로직 변경) → deploy-staging.sh 정상 사용
