## 1. 구현 범위 요약

| 영역 | 작업 항목 | 비고 |
|------|---------|------|
| DB | 신규 테이블 **7개** | TypeORM 마이그레이션 |
| Backend | NestJS 모듈 신규 (`expense-request`) | Controller, Service, DTO, Entity |
| Backend | Cron Job (정기 지출 자동 생성) | `@nestjs/schedule` |
| Backend | 집행 정보 입력 API | APPROVED → EXECUTED 상태 전이 |
| Backend | 월별 실적 리포트 API + Excel | 집행일 기준 집계 |
| Backend | 익월 예정 리포트 API + Excel | 정기지출 기반 자동 구성 |
| Backend | 이메일 발송 연동 | 기존 SmtpModule 재사용 |
| Frontend | 도메인 신규 (`expense-request`) | 목록/작성/상세 + 집행정보 폼 + 리포트 2종 |
| Frontend | i18n 3개 언어 번역 파일 | ko/en/vi |
| Frontend | 라우터 등록 | `/expense-request`, `/reports/monthly`, `/reports/forecast` |
| 공통 | `packages/types` 타입 추가 | 집행·예정리포트 타입 포함 |
| 공통 | 메뉴 코드 + ACL 권한 추가 | `EXPENSE_REQUEST` |

## 2. 시스템 개발 현황 분석

### 2.1 재사용 가능한 기존 구성요소

| 구성요소 | 위치 | 재사용 방법 |
|---------|------|----------|
| 역할 가드 (`RolesGuard`) | `apps/api/src/global/guard/roles.guard.ts` | `@Roles('MEMBER')` 데코레이터 적용 |
| 인증 데코레이터 (CurrentUser) | `apps/api/src/global/decorator/current-user.decorator.ts` | 작성자 ID 추출에 활용 |
| HrEntityEntity | `apps/api/src/domain/hr/entity/hr-entity.entity.ts` | 법인(ent_id) 관계 |
| SmtpModule / EmailTemplate | `apps/api/src/domain/settings/` | 이메일 발송 재사용 |
| DriveModule | `apps/api/src/domain/drive/` | 파일 업로드 저장소로 활용 가능 |
| Notification 모듈 | `apps/api/src/domain/notification/` | 앱 내 알림 발송 |
| 인보이스 승인 패턴 | `apps/web/src/domain/billing/components/invoice/InvoiceApprovalActions.tsx` | 승인 액션 UI 패턴 참조 |
| MenuGuard (프론트) | `apps/web/src/router/` | 메뉴 권한 가드 적용 |
| useEntityStore | `apps/web/src/domain/hr/store/entity.store.ts` | 현재 법인 정보 조회 |
| useAuthStore | `apps/web/src/domain/auth/store/auth.store.ts` | 로그인 사용자 역할 조회 |

### 2.2 참조할 구현 패턴

- **목록 페이지**: `InvoiceListPage.tsx` — 헤더(검색+필터+버튼), 테이블, 페이지네이션 구조
- **상세/작성 페이지**: `InvoiceDetailPage.tsx` — 폼 + 하위 컴포넌트 조합 구조  
- **승인 액션 컴포넌트**: `InvoiceApprovalActions.tsx` — 역할별 버튼 노출 + 반려 모달 패턴
- **Line Item 편집기**: `InvoiceItemsEditor.tsx` — 동적 행 추가·삭제 편집 테이블 패턴
- **라우팅**: `apps/web/src/router/index.tsx` — `MenuGuard` 래핑 패턴

### Phase 2 — 자동화 & 통합

#### 2-1. 결의서 번호 채번 서비스

```
EXP-{YYYYMM}-{NNNN}
- YYYYMM: 제출 시점 기준 연월
- NNNN: 법인(ent_id) + 연월 기준 4자리 순번 (0001부터)
- DB: amb_expense_requests.exr_number (UNIQUE)
- 제출(submit) 시 트랜잭션 내에서 원자적으로 채번
```

#### 2-2. 이메일 발송 연동

재사용: `apps/api/src/domain/settings/email-template/` + `SmtpModule`

| 이벤트 | 수신자 | 템플릿 |
|--------|--------|--------|
| 제출(PENDING 전환) | 1차 승인자 | `expense_request_approval` |
| 1차 승인 완료(APPROVED_L1) | 2차 승인자 | `expense_request_approval` |
| 최종 승인(APPROVED) | 작성자 | `expense_request_approved` |
| 반려(REJECTED) | 작성자 | `expense_request_rejected` |
| Cron 자동 생성 | 작성자 | `expense_request_auto_generated` |
| 집행 완료(EXECUTED) | 1차·2차 승인자 (참고용) | `expense_request_executed` |

#### 2-3. 정기 지출 Cron Job

## 6. 사이드 임팩트 분석

| 영역 | 임팩트 | 조치 사항 |
|------|--------|---------|
| `app.module.ts` | `ExpenseRequestModule` 추가로 DB 연결 시 엔티티 자동 등록 | AppModule imports 배열에 추가 필요 |
| `menu-code.constant.ts` | `DEFAULT_PERMISSIONS`에 신규 코드 추가 시 기존 사용자 권한 영향 없음 (신규 키 추가) | — |
| TypeORM Synchronize | `synchronize: true`(개발환경) 시 테이블 자동 생성. Staging/Production은 마이그레이션 스크립트 별도 실행 필요 | SQL 마이그레이션 스크립트 준비 |
| `SmtpModule` | 이메일 발송 의존 — SMTP 미설정 법인은 이메일 발송 생략 처리 (graceful fail) | try-catch로 이메일 실패가 승인 트랜잭션에 영향 주지 않도록 |
| `@nestjs/schedule` | 이미 사용 중인지 확인 필요 (기존 Cron 있으면 모듈 공유, 없으면 새로 등록) | `ScheduleModule.forRoot()` app.module 등록 여부 확인 |
| Notification 모듈 | 기존 알림 모듈 연동 | 지출결의서 알림 타입 추가 필요 여부 확인 |
| 파일 업로드 | 첨부파일 + 증빙파일 업로드 — 용량 제한 및 허용 확장자 정책 필요 | API 단에서 `Multer` 설정, 권장: 최대 10MB, pdf/jpg/png/xlsx/docx 허용 |
| ACL 퍼미션 초기화 | 신규 메뉴 코드는 새로 role을 부여받은 사용자에게만 적용. 기존 사용자는 ACL 재설정 필요할 수 있음 | 관리자 페이지에서 수동 권한 부여 또는 마이그레이션 스크립트 |
| `exceljs` 의존성 | 월별 리포트 + 예정 리포트 Excel 다운로드에 필요 | `exceljs` 이미 회계 모듈에서 사용 중이면 재사용, 아니면 `npm install exceljs` 필요 |
| 리포트 집계 성능 | 월별 집계 쿼리는 대용량 데이터 시 느릴 수 있음 | `idx_exd_executed_at` 인덱스 필수, 필요 시 집계 결과 캐싱 고려 |
