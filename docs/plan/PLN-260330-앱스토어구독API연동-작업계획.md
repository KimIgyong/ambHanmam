# PLAN-앱스토어구독API연동-작업계획-20260330

## 1. 시스템 개발 현황 분석

### 1.1 현행 아키텍처 (AS-IS)

현재 "앱스토어에서 가져오기" 기능은 **내부 DB 테이블을 직접 쿼리**하여 구독 앱 목록을 조회한다.

```
AMA Backend ──▶ amb_partner_app_installs (Internal DB)
              ──▶ amb_partner_apps (Internal DB)
```

- **`getImportableApps()`**: `PartnerAppInstallEntity` WHERE `paiIsActive=true` + `PartnerAppEntity` WHERE `papStatus='PUBLISHED'`
- **`importFromStore()`**: `installRepo.findOne()` → `install.app` 으로 파트너앱 정보 획득 → Custom App 생성
- **Module**: `PartnerAppEntity`, `PartnerAppInstallEntity`가 `EntitySettingsModule`의 `TypeOrmModule.forFeature()`에 등록됨
- **Service**: `partnerAppRepo`, `installRepo` 2개 Repository 주입됨

### 1.2 문제점

1. **데이터 정합성**: `amb_partner_app_installs`/`amb_partner_apps` 테이블은 앱스토어(`apps.amoeba.site`)의 데이터 복제본. 앱스토어 측 상태 변경(구독 해지, 앱 비활성화 등)이 AMA DB에 반영되지 않을 수 있음.
2. **결합도**: AMA가 앱스토어의 내부 테이블 스키마에 직접 의존. 앱스토어 스키마 변경 시 AMA도 변경 필요.
3. **중복 데이터**: 동일 데이터가 두 시스템에 존재.

### 1.3 목표 아키텍처 (TO-BE)

앱스토어 Public API를 호출하여 **실시간 구독 현황**을 조회한다.

```
AMA Backend ──▶ GET /api/v1/platform/subscriptions/entity/{ent_id}
               (External: stg-apps.amoeba.site / apps.amoeba.site)
```

- **API 스펙**: `docs/AppStore-Subscription-API-for-AMA.md` 참조
- **인증 불필요** (Public API, UUID 기반)
- **CORS**: AMA 도메인 허용 (stg-ama.amoeba.site, ama.amoeba.site)
- **응답 데이터**: `appSlug`, `appName`, `appNameEn`, `appStatus`, `appIconUrl`, `subscription` (null/object)

### 1.4 데이터 매핑 (AppStore API → 현행 ImportableApp)

| AppStore API 필드 | 현행 ImportableApp 필드 | 매핑 규칙 |
|-------------------|------------------------|-----------|
| `appSlug` | `papCode` → **`appSlug`** | 필드명 변경 |
| `appName` | `papName` → **`appName`** | 필드명 변경 |
| `appNameEn` | (신규) **`appNameEn`** | 추가 |
| `appIconUrl` | `papIcon` → **`appIconUrl`** | 필드명 변경 |
| `subscription` | (신규) **`subscription`** | 추가 (status, requestedAt 등) |
| `subscription !== null` | `isRegistered`와 별도 | 구독 여부 |
| `appStatus` | (신규) **`appStatus`** | ACTIVE / COMING_SOON |
| - | `papUrl` | AppStore base URL + appSlug로 생성 |
| - | `papAuthMode` | 고정 'jwt' (AppStore 앱은 SSO JWT) |
| - | `papOpenMode` | 고정 'iframe' (기본값) |
| - | `isRegistered` | AMA DB 조회로 유지 (ecaCode 또는 ecaSourceAppSlug 매칭) |

### 1.5 Import 식별자 변경

| 항목 | AS-IS | TO-BE |
|------|-------|-------|
| 가져오기 URL 파라미터 | `POST /import/:papId` (UUID) | `POST /import/:appSlug` (string) |
| 원본 추적 컬럼 | `eca_source_pap_id` (UUID) | `eca_source_app_slug` (VARCHAR) |
| 중복 체크 기준 | `ecaSourcePapId` + `ecaCode` | `ecaSourceAppSlug` + `ecaCode` |

#### 2.1.4 Service 리팩토링 — `getImportableApps()`

**변경 전**: `installRepo.find()` + `partnerAppRepo` 쿼리
**변경 후**: `fetch()` → AppStore API 호출 → 응답 매핑

```typescript
async getImportableApps(entityId: string) {
  // 1. AppStore API 호출
  const storeApps = await this.fetchStoreSubscriptions(entityId);
  
  // 2. 기존 Custom Apps에서 등록 여부 확인
  const existingApps = await this.customAppRepo.find({
    where: { entId: entityId, ecaDeletedAt: IsNull() },
    select: ['ecaSourceAppSlug', 'ecaSourcePapId', 'ecaCode'],
  });
  const registeredSlugs = new Set(existingApps.filter(a => a.ecaSourceAppSlug).map(a => a.ecaSourceAppSlug));
  const registeredCodes = new Set(existingApps.map(a => a.ecaCode));

  // 3. ACTIVE 구독만 필터 + 매핑
  return storeApps
    .filter(app => app.subscription?.status === 'ACTIVE')
    .map(app => ({
      appSlug: app.appSlug,
      appName: app.appName,
      appNameEn: app.appNameEn,
      appStatus: app.appStatus,
      appIconUrl: app.appIconUrl,
      subscription: app.subscription,
      isRegistered: registeredSlugs.has(app.appSlug) || registeredCodes.has(app.appSlug),
    }));
}
```

#### 2.1.5 Service 리팩토링 — `importFromStore()`

**변경 전**: `installRepo.findOne()` → `install.app` 사용
**변경 후**: `fetchStoreSubscriptions()` → `appSlug` 기반 조회

```typescript
async importFromStore(entityId: string, appSlug: string, overrides: ImportCustomAppRequest, userId: string) {
  // 1. AppStore API에서 해당 앱 구독 상태 확인
  const storeApps = await this.fetchStoreSubscriptions(entityId);
  const storeApp = storeApps.find(a => a.appSlug === appSlug);
  
  if (!storeApp || !storeApp.subscription) {
    throw new NotFoundException('App not subscribed or not found');
  }
  if (storeApp.subscription.status !== 'ACTIVE') {
    throw new BadRequestException('Subscription is not active');
  }

  // 2. 코드 중복 / source 중복 체크
  const code = appSlug.toLowerCase().replace(/[^a-z0-9-]/g, '-');
  // ... (기존 로직 유지)
  
  // 3. Custom App 생성
  const appStoreBaseUrl = this.configService.get('APP_STORE_API_URL');
  const customApp = this.customAppRepo.create({
    entId: entityId,
    ecaCode: code,
    ecaName: overrides.name || storeApp.appName,
    ecaDescription: overrides.description ?? null,
    ecaIcon: overrides.icon || storeApp.appIconUrl || 'AppWindow',
    ecaUrl: overrides.url || `${appStoreBaseUrl}/${appSlug}/`,
    ecaAuthMode: 'jwt', // AppStore 앱은 항상 JWT
    ecaOpenMode: overrides.open_mode || 'iframe',
    ecaAllowedRoles: overrides.allowed_roles || null,
    ecaSortOrder: overrides.sort_order ?? 0,
    ecaRegisteredBy: userId,
    ecaSourceAppSlug: appSlug,
  });

  const saved = await this.customAppRepo.save(customApp);
  return this.toResponse(saved);
}
```

## 4. 사이드 임팩트 분석

### 4.1 하위 호환성

| 항목 | 영향 | 대응 |
|------|------|------|
| `eca_source_pap_id` 기존 데이터 | 기존에 import된 앱의 `eca_source_pap_id` 값 유지 필요 | 컬럼 삭제하지 않고 유지. 새 import는 `eca_source_app_slug` 사용 |
| `POST /import/:papId` → `/:appSlug` | 클라이언트 호출 변경 필요 | 프론트엔드 동시 배포로 해결 |
| `isRegistered` 판별 로직 | 기존 `ecaSourcePapId` 매칭 + 새 `ecaSourceAppSlug` 매칭 필요 | 두 필드 모두 체크 |

### 4.2 외부 API 의존성

| 리스크 | 영향 | 대응 |
|--------|------|------|
| AppStore API 다운 | importable 목록 조회 실패 | try-catch + 사용자 안내 메시지 표시 |
| 네트워크 지연 | UX 저하 | 15초 timeout + Loader UI |
| AppStore API 응답 구조 변경 | 매핑 오류 | 응답 검증 + 방어적 코딩 |

### 4.3 PartnerApp 엔티티 의존성

- `PartnerAppEntity`, `PartnerAppInstallEntity`가 `EntitySettingsModule` 외에서도 사용되는지 확인 필요
- 사용처가 없으면 Module에서 제거. 사용처가 있으면 유지

### 4.4 기존 Import 데이터 정합성

- 기존에 `eca_source_pap_id`로 import된 앱: `toResponse()`에서 `sourcePapId` 필드 계속 반환
- 새로 import된 앱: `sourceAppSlug` 필드 추가 반환
