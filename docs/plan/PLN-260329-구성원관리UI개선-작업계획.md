## 2. 단계별 구현 계획

### Phase 1: 백엔드 수정 (API 변경)

#### 1-1. findMembers 응답에 CLIENT 정보 추가
- `entity-member.service.ts`의 `findMembers()` 수정
- CLIENT_LEVEL 멤버의 `usrCliId`로 `SvcClientEntity` 조인
- 응답에 `clientInfo: { clientId, clientCode, clientName, clientStatus }` 추가

#### 1-2. 비밀번호 초기화 API 수정
- `resetMemberPassword()` 메서드에 `mode` 파라미터 추가
  - `mode: 'email'` (기본값): 기존 방식 — 이메일 발송
  - `mode: 'generate'`: 관리자 직접 확인 — 임시 비밀번호를 응답에 포함
- 컨트롤러 Body에 `mode` 필드 추가
- 응답 형태: `{ userId, emailSent, tempPassword? }`

#### 1-3. 프론트엔드 타입 업데이트
- `EntityMember` 인터페이스에 `clientInfo` 필드 추가
- `entity-settings.service.ts`에 `resetMemberPassword` 메서드 시그니처 변경

### Phase 2: 프론트엔드 UI 개편

#### 2-1. 메인 탭 구조 변경
```
기존: [구성원] [초대]
변경: [구성원 (N)] [초대 대기 (N)] [승인 기록 (N)]
```
- 초대 탭을 → 초대 대기(PENDING) + 승인 기록(ACCEPTED/CANCELLED/EXPIRED) 2개로 분리

#### 2-2. 구성원 탭 내부 서브탭
```
[마스터] [매니저] [멤버] [뷰어]  |  [클라이언트-매니저] [클라이언트-멤버] [연결된 고객사]
```
- 전체(ALL) 기본 선택 없이, 마스터 탭이 기본
- 데이터는 이미 로드된 members에서 프론트엔드 필터링 (추가 API 호출 불필요)
- "연결된 고객사" 탭: 고유 clientInfo를 기준으로 고객사 목록 표시

#### 2-3. MemberDetailModal 개선
- CLIENT_LEVEL 멤버: 고객사 정보 섹션 추가
- 상세/수정 정보 일치화: Level, JoinMethod, Role, Unit, Status 모두 표시

#### 2-4. 비밀번호 초기화 UI 변경
- 기존 단일 버튼 → 2개 버튼:
  - `[이메일로 발송]` — 기존 방식
  - `[임시 비밀번호 생성]` — 신규, 결과를 화면에 표시

### Phase 3: i18n 키 추가

- `ko/entitySettings.json`, `en/entitySettings.json`, `vi/entitySettings.json`에 새 키 추가
