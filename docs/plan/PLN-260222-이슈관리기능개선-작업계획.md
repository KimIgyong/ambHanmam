## 단계별 구현 계획

### Phase 1: PATCH 400 에러 수정 + 담당자 스키마 변경 (Backend)

> **목표**: 이슈 수정 시 400 에러 해결 + assignee를 UUID FK 기반으로 전환

#### 1-1. SQL 마이그레이션 작성
- **파일**: `sql/migration_issue_assignee.sql`
- 기존 `iss_assignee` VARCHAR(50) 컬럼 유지 (하위 호환, 추후 삭제)
- 새 컬럼 `iss_assignee_id` UUID nullable 추가
- FK 제약: `REFERENCES amb_users(usr_id) ON DELETE SET NULL`
- 기존 데이터 마이그레이션: `iss_assignee` 이름으로 `amb_users.usr_name` 매칭하여 `iss_assignee_id` 업데이트
- 매칭 안 되는 값은 NULL 유지

```sql
-- 1) 컬럼 추가
ALTER TABLE amb_issues ADD COLUMN IF NOT EXISTS iss_assignee_id UUID;
ALTER TABLE amb_issues ADD CONSTRAINT fk_issues_assignee 
  FOREIGN KEY (iss_assignee_id) REFERENCES amb_users(usr_id) ON DELETE SET NULL;

-- 2) 기존 데이터 마이그레이션 (이름 → UUID)
UPDATE amb_issues i SET iss_assignee_id = u.usr_id 
FROM amb_users u WHERE i.iss_assignee = u.usr_name AND i.iss_assignee IS NOT NULL;

-- 3) 인덱스
CREATE INDEX IF NOT EXISTS idx_issues_assignee_id ON amb_issues(iss_assignee_id);
```

#### 1-2. Entity 수정
- **파일**: `apps/api/src/domain/issues/entity/issue.entity.ts`
- 새 프로퍼티 `issAssigneeId` (Column, uuid, nullable)
- `@ManyToOne(() => UserEntity)` + `@JoinColumn({ name: 'iss_assignee_id' })` → `assignee: UserEntity`
- 기존 `issAssignee` VARCHAR(50) 프로퍼티는 **유지** (하위 호환, deprecated 표시)

#### 1-3. UpdateIssueRequest 수정
- **파일**: `apps/api/src/domain/issues/dto/request/update-issue.request.ts`
- **`type` 필드 추가**: `@IsOptional() @IsIn(['BUG', 'FEATURE', 'ENHANCEMENT'])` → PATCH 400 해결
- **`assignee_id` 필드 추가**: `@IsOptional() @IsUUID()` (nullable)
- 기존 `assignee` 텍스트 필드는 `@IsOptional()` 유지 (하위 호환)

#### 1-4. CreateIssueRequest 수정
- **파일**: `apps/api/src/domain/issues/dto/request/create-issue.request.ts`
- **`assignee_id` 필드 추가**: `@IsOptional() @IsUUID()`

#### 1-5. IssueService 수정
- **파일**: `apps/api/src/domain/issues/service/issue.service.ts`
- `createIssue`: `dto.assignee_id`가 있으면 `issue.issAssigneeId = dto.assignee_id` 설정
- `updateIssue`: `dto.type` 매핑 (`entity.issType = dto.type`), `dto.assignee_id` 매핑 (`entity.issAssigneeId = dto.assignee_id`)
- `getIssueById`, `getIssues`, `getMyIssues` 등: `relations: ['reporter', 'assignee']` 추가
- **`getMyIssues` 수정**: `iss.issAssignee = :userName` → `iss.issAssigneeId = :userId`로 변경 (userName 조회 불필요)
- `updateIssue`에서 `dto.type` 처리: `if (dto.type) entity.issType = dto.type`

#### 1-6. IssueMapper 수정
- **파일**: `apps/api/src/domain/issues/mapper/issue.mapper.ts`
- `toResponse`에 추가 매핑:
  - `assigneeId: entity.assignee?.usrId || entity.issAssigneeId || null`
  - `assigneeName: entity.assignee?.usrName || entity.issAssignee || null`
  - `visibility: entity.issVisibility || 'ENTITY'`
  - `groupId: entity.issGroupId || null`
  - `originalLang: entity.issOriginalLang || 'ko'`
- 기존 `assignee` 필드: `entity.assignee?.usrName || entity.issAssignee || null` (하위 호환)

#### 1-7. 공유 타입 수정
- **파일**: `packages/types/src/domain.types.ts`
- `IssueResponse` 인터페이스에 추가:
  ```typescript
  assigneeId: string | null;
  assigneeName: string | null;
  visibility: 'PRIVATE' | 'GROUP' | 'ENTITY';
  groupId: string | null;
  originalLang: string;
  ```

- 클릭: 상세 모달 열기
- 드래그 중: 반투명 스타일 + 그림자

#### 4-3. IssueKanbanColumn 컴포넌트
- **파일**: `apps/web/src/domain/issues/components/IssueKanbanColumn.tsx` (신규)
- `useDroppable` 훅 사용
- Props: `status`, `issues`, `isValidTarget`
- 컬럼 헤더: 상태명 + 이슈 수 뱃지
- 유효한 드롭 대상: 밝은 하이라이트, 유효하지 않으면 회색 비활성
- 상태별 컬럼 색상:
  - OPEN: 회색, APPROVED: 파란색, IN_PROGRESS: 주황색
  - RESOLVED: 초록색, CLOSED: 보라색, REJECTED: 빨간색

#### 4-4. IssueKanbanView 컴포넌트
- **파일**: `apps/web/src/domain/issues/components/IssueKanbanView.tsx` (신규)
- `DndContext`, `DragOverlay` 사용
- Props: `issues` (전체 이슈 배열)
- 상태별 그룹핑: `issues.reduce((acc, issue) => { acc[issue.status].push(issue) }, ...)`
- 컬럼 순서: `OPEN → APPROVED → IN_PROGRESS → RESOLVED → CLOSED → REJECTED`
- 드래그 시작: `onDragStart` → 현재 드래그 중인 이슈 저장
- 드롭 시: `onDragEnd` → `ISSUE_VALID_TRANSITIONS` 검증 후 `updateIssueStatus` API 호출
- MANAGER 권한 체크: APPROVED/REJECTED/CLOSED 전환은 MANAGER 이상만 허용
- 드롭 실패 시 토스트 알림

#### 4-5. 칸반 데이터 로드 전략
- 칸반 뷰 활성 시: `size=999` 파라미터로 전체 이슈 로드 (또는 별도 endpoint)
- `staleTime: 30000` 캐싱으로 불필요한 재요청 방지
- 상태 변경 후: `queryClient.invalidateQueries(['issues'])` 로 갱신

### Phase 5: 간트차트 보기 구현

> **목표**: 이슈별 일정 타임라인 SVG 기반 커스텀 구현

#### 5-1. IssueGanttView 컴포넌트
- **파일**: `apps/web/src/domain/issues/components/IssueGanttView.tsx` (신규)
- 커스텀 SVG 기반 (외부 라이브러리 미사용)
- X축: 날짜 타임라인
  - 범위: 전체 이슈 중 가장 이른 `createdAt` ~ 가장 늦은 `resolvedAt` (또는 오늘)
  - 간격: 주 단위 (축소/확대 가능)
  - 오늘 날짜: 빨간 수직 점선
- Y축: 이슈 목록 (우선순위순 → 상태순 정렬)
- 바: `createdAt` ~ `resolvedAt`(완료된 이슈) 또는 `createdAt` ~ 오늘(진행 중 이슈)
- 바 색상: 상태별
  - OPEN: `#9CA3AF`, APPROVED: `#3B82F6`, IN_PROGRESS: `#F59E0B`
  - RESOLVED: `#10B981`, CLOSED: `#8B5CF6`, REJECTED: `#EF4444`
- 왼쪽 고정 영역: 이슈 제목 + 담당자 (200px)

#### 5-2. 간트차트 인터랙션
- 바 마우스 오버: 툴팁 (제목, 상태, 담당자, 기간)
- 바 클릭: 상세 모달 표시 (`onIssueClick` 콜백)
- 스크롤: X축 수평 스크롤 (마우스 휠)
- 줌: Ctrl + 마우스 휠로 시간 단위 확대/축소
- 왼쪽 이슈 목록 영역은 고정 (position: sticky)

#### 5-3. 성능 최적화
- 가시 영역만 렌더링 (IntersectionObserver 또는 가상 스크롤)
- 50개 초과 이슈: 스크롤 + 가시 영역 SVG 렌더링
- `useMemo`로 바 위치 계산 캐싱

| 이슈 엔티티 | **높음** | `iss_assignee_id` UUID FK 추가, `@ManyToOne` 관계 추가 |
| 이슈 API 응답 | **높음** | `assigneeId`, `assigneeName`, `visibility`, `groupId`, `originalLang` 5필드 추가 |
| 이슈 생성/수정 DTO | **중간** | `type` 추가(update), `assignee_id` 추가(create/update) |
| 공유 타입 | **중간** | `IssueResponse` 인터페이스 5필드 추가 |
| 프론트엔드 이슈 폼 | **중간** | 담당자 텍스트 → AssigneeSelector |
| IssuesPage 구조 | **높음** | 탭 UI 추가, 리스트 분리, 2개 뷰 신규 |
| `getMyIssues` | **중간** | userName 매칭 → userId 매칭 전환 |
| 기존 데이터 | **낮음** | SQL 마이그레이션으로 이름→UUID 자동 매핑, 매칭 안 되면 NULL |
| KMS 이벤트 | **낮음** | 영향 없음 (이슈 이벤트 데이터 구조 변경 없음) |
| 번들 사이즈 | **낮음** | `@dnd-kit` ~22KB gzipped 추가 |

### 위험 요소 및 대응
| 위험 | 영향 | 대응 |
|------|------|------|
| 기존 assignee 문자열 → UUID 매핑 실패 | 중간 | 매칭 안 되면 NULL 유지, 기존 `iss_assignee` 컬럼 보존 |

| 하위 호환성 (기존 assignee 텍스트) | 낮음 | 기존 `assignee` 텍스트 필드 응답에 유지, deprecated 처리 |

## 작업 순서 의존성

```
Phase 1 (Backend 스키마/DTO/서비스)
  ↓
Phase 2 (Frontend 담당자 UI) ← Phase 1 완료 후
  ↓
Phase 3 (탭 UI + 리스트 분리) ← Phase 2와 병렬 가능하나 순차 권장
  ↓
Phase 4 (칸반) ← Phase 3 완료 후 (탭 UI 필요)
  ↓
Phase 5 (간트차트) ← Phase 3 완료 후 (탭 UI 필요, Phase 4와 병렬 가능)
  ↓
Phase 6 (i18n + 테스트) ← Phase 4, 5 완료 후
```

예상 소요: Phase별 각 1~2시간, 전체 약 6~10시간
