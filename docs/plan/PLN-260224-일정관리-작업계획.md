## 1) 시스템 개발 현황 기반 요약

- 현재 `work-schedule`은 출퇴근 기록 목적이며, 신규 일정관리와 목적/정책이 다름.
- 일정관리 전용 백엔드 도메인, 테이블, API, 프론트 화면은 아직 없음.
- 재사용 가능한 기반은 존재함:
  - 권한/메뉴: `settings` 도메인
  - 검색 연계: `work_items` + `MODULE_DATA_EVENTS`
  - 번역 연계: `translation` `sourceType` 확장 구조
  - 배치/알림: `@nestjs/schedule` + `Cron`
- 따라서 **분리 구현 + 공통 패턴 재사용** 전략으로 진행.

## 2) 구현 목표

1. `work-schedule`과 완전히 분리된 신규 `schedule` 도메인 구축
2. 일정 CRUD + 반복 + 참여자 + 알림 + 공개범위 + 검색 연계를 MVP로 제공
3. Google Calendar 단방향 동기화와 동기화 상태 관리 구현
4. AI 일정 보조(초안/충돌/최적화) API 뼈대 및 1차 연동 제공

## 4) 사이드 임팩트 분석

- 설정/권한: 메뉴 코드 `SCHEDULE` 추가 시 권한 기본값/그룹권한 동시 반영 필요
- 검색: `work_items` 이벤트 누락 시 검색 불일치 발생 가능
- 알림: Talk/Email 채널 실패 시 재시도 및 장애 가시화 필요
- 기존 기능: `work-schedule` 미영향 보장(데이터/메뉴/API 완전 분리)

## 5) 리스크 및 대응

- 반복 예외 로직 복잡도 증가
  - 대응: MVP는 단건 취소/단건 변경 우선, 이후 split 확장
- Google API 의존성
  - 대응: 3회 재시도 + `SYNC_FAILED` 상태 고정 + 사용자 안내
- 동시 편집 충돌
  - 대응: `sch_updated_at` 낙관적 락 강제 + 409 처리 UX
- 성능 저하(월 뷰 대량 일정)
  - 대응: 기간/건수 제한 + 인덱스 + 페이지네이션
