# 작업 계획서 (PLAN)

- 문서명: 웹메일 설정 관리 기능 추가
- 작성일: 2026-02-23
- 기준 분석서: `docs/analysis/REQ-웹메일설정관리-20260223.md`

## 1) 시스템 개발 현황 분석 기반 요약

- 메일 계정 관리 화면/CRUD는 이미 구현되어 있음.
- 계정 엔티티에 SMTP/IMAP 필드가 존재하여 기본 데이터 모델 확장은 최소화 가능.
- 발송 실패가 runtime 시점에 드러나며, 사전 점검(테스트) UX가 부재.
- 운영 안정성을 위해 "설정 조회/수정 + 연결 테스트"가 필요한 상태.

## 2) 구현 목표

1. 메일 계정 관리 페이지에 `웹메일 설정` 탭 추가
2. 계정별 SMTP/IMAP 설정 조회/수정 API 구현
3. SMTP 연결 테스트 API 및 UI 추가
4. 오류 메시지 가시성 개선(운영자가 즉시 원인 파악 가능)

## 3) 단계별 구현 계획

### Phase 1. 백엔드 API 확장

- 대상 파일
  - `apps/api/src/domain/webmail/controller/mail-admin.controller.ts`
  - `apps/api/src/domain/webmail/service/mail-account.service.ts`
  - `apps/api/src/domain/webmail/service/mail-send.service.ts` (테스트 로직 재사용 검토)
  - `apps/api/src/domain/webmail/dto/request/*` (신규 DTO)

- 작업 내용
  1. 설정 조회 API 추가
     - `GET /api/v1/mail/admin/accounts/:id/settings`
  2. 설정 저장 API 추가
     - `PATCH /api/v1/mail/admin/accounts/:id/settings`
     - 입력: `display_name`, `imap_host`, `imap_port`, `smtp_host`, `smtp_port`, `password?`
  3. SMTP 테스트 API 추가
     - `POST /api/v1/mail/admin/accounts/:id/test-smtp`
     - timeout 명시
  4. 예외 처리 정비
     - timeout/auth/connect 에러를 사용자 친화 메시지로 변환

- 완료 조건
  - Swagger 또는 API 호출로 3개 API 정상 동작 확인

### Phase 2. 프론트엔드 탭/폼 구현

- 대상 파일
  - `apps/web/src/domain/settings/pages/MailAccountManagementPage.tsx`
  - `apps/web/src/domain/settings/hooks/useMailAccounts.ts`
  - `apps/web/src/domain/settings/service/settings.service.ts` (또는 mail-account API 파일)
  - `apps/web/src/domain/settings/components/*` (웹메일 설정 탭 컴포넌트 신규)

- 작업 내용
  1. `계정 목록` / `웹메일 설정` 탭 UI 구성
  2. 계정 선택 + 현재 설정 로드
  3. 설정 수정 폼(호스트/포트/표시명/선택적 비밀번호)
  4. `연결 테스트` 버튼 + 결과 상태 표시
  5. 저장 성공/실패 피드백

- 완료 조건
  - 관리자 UI에서 설정 조회/수정/테스트 가능

### Phase 3. i18n/유효성/보안 보강

- 대상 파일
  - `apps/web/src/locales/ko/settings.json`
  - `apps/web/src/locales/en/settings.json`
  - `apps/web/src/locales/vi/settings.json`

- 작업 내용
  1. 신규 탭/폼/테스트 메시지 번역 키 추가
  2. 클라이언트 유효성 체크
     - host 필수, port 범위(1~65535), email 형식 등
  3. 비밀번호 처리 정책 반영
     - 미입력 시 유지, 입력 시만 업데이트

- 완료 조건
  - 3개 언어에서 깨짐 없이 렌더링

### Phase 4. 검증 및 배포

- 검증
  1. 린트/빌드 통과
  2. 스테이징에서 실제 계정 대상으로
     - 설정 조회
     - 잘못된 호스트 테스트 실패 확인
     - 정상 호스트 테스트 성공 확인
     - 저장 후 실제 `POST /mail/messages` 발송 확인

- 배포
  - `main` 머지 후 스테이징 배포 스크립트 실행

## 4) 사이드 임팩트 분석

- 메일 발송 도메인: 설정값 변경으로 실제 발송 경로가 바뀌므로 영향도 높음
- 사용자 관리 도메인: 계정-사용자 매핑 로직은 유지 (영향도 낮음)
- 인프라/보안: SMTP 서버 접근 정책 및 방화벽 상태에 따라 테스트 결과 변동 가능
- 운영 절차: 장애 대응이 DB 수동 변경에서 UI 중심으로 전환
