### 1.3 수정된 아키텍처 전략

요구사항 분석서에서 제안한 **방안 A (모노레포 확장)**을 유지하되, `billing-core` 패키지 추출은 불필요하다. 기존 서비스 관리 모듈이 이미 독립된 도메인으로 구성되어 있으므로, 이를 확장하는 방식으로 진행한다.

```
ambManagement/ (기존 모노레포)
├── apps/
│   ├── api/                ← 기존 내부 API + 서비스 관리 모듈 확장
│   ├── web/                ← 기존 내부 프론트엔드 (변경 최소)
│   ├── portal-api/         ← 🆕 서비스 포탈 전용 API (고객 인증, Stripe 연동)
│   └── portal-web/         ← 🆕 서비스 포탈 프론트엔드 (공개 사이트 + 고객 포탈)
├── packages/
│   ├── types/              ← 공유 타입 확장 (포탈 관련 타입 추가)
│   ├── common/             ← 공유 유틸리티
│   └── portal-shared/      ← 🆕 포탈 API ↔ 내부 API 공유 코드 (엔티티, CryptoService 등)
```

**핵심 변경점**:
- `portal-api`는 기존 `api`와 **동일 PostgreSQL DB를 공유** → 기존 `amb_svc_*` 테이블 직접 참조
- 기존 서비스 관리 엔티티를 `portal-shared` 패키지로 추출하여 양쪽에서 import
- 고객 셀프서비스 인증은 별도 `amb_svc_portal_customers` 테이블로 내부 사용자와 분리
- Stripe 연동 컬럼은 기존 엔티티에 추가 (마이그레이션)

```
  ├── pct_id (PK)
  ├── pct_email (UNIQUE)
  ├── pct_password (bcrypt)
  ├── pct_cli_id (FK → amb_svc_clients.cli_id)  ← 1:1 매핑
  └── pct_stripe_customer_id

amb_svc_clients (기존 — 비즈니스 데이터)
  ├── cli_id (PK)
  ├── cli_company_name, cli_country, cli_industry ...
  ├── cli_bil_partner_id (FK → amb_bil_partners.ptn_id)
  └── cli_portal_source: 'MANUAL' | 'PORTAL'

  포탈 회원가입 → portal_customers 생성 → svc_clients 자동 생성 (cli_portal_source='PORTAL')
  내부 관리자가 수동 생성한 클라이언트 → portal_customers 없음 (내부 관리만)
```

**검증**:
