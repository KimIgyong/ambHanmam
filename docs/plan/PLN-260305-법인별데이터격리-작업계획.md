## 작업 전략

**핵심 원칙**: Controller에 `@UseGuards(EntityGuard)` 추가 → Service에 `entityId` 파라미터 추가 → QueryBuilder/Repository WHERE 조건 추가

**FE 변경 없음**: 프론트엔드는 이미 `X-Entity-Id` 헤더를 모든 API 요청에 자동 주입하고 있으므로 백엔드만 수정

## 단계별 구현 계획

### 1단계: HR — 인사 모듈 격리 (가장 광범위)

#### 1-1: employee.controller.ts + employee.service.ts

**Controller 변경**:
- `@UseGuards(EntityGuard)` 클래스 레벨 추가
- `EntityGuard` import 추가
- 모든 엔드포인트에 `@Req() req: any` 추가 (없는 곳)
- `req.entityId`를 서비스 호출에 전달

**Service 변경** — `getEmployees()`:
```typescript
// AS-IS (line ~26)
async getEmployees(status?: string, department?: string)
  → qb (no entId filter)

// TO-BE
async getEmployees(entityId: string, status?: string, department?: string)
  → qb.andWhere('emp.ent_id = :entityId', { entityId });
```

**영향 메서드**:
| 메서드 | 변경 내용 |
|--------|---------|
| `getEmployees()` | entityId 파라미터 + WHERE 추가 |
| `getEmployeeById()` | entityId 검증 추가 (entId 일치 확인) |
| `createEmployee()` | 이미 entId 저장 여부 확인, 없으면 추가 |
| `updateEmployee()` | entityId 검증 추가 |
| `deleteEmployee()` | entityId 검증 추가 |

#### 1-2: timesheet.controller.ts + timesheet.service.ts

**Controller**: `@UseGuards(EntityGuard)` 추가

**Service 변경** — `getMonthlyTimesheet()`:
```typescript
// AS-IS
async getMonthlyTimesheet(year: number, month: number)
  → this.employeeRepo.find({ where: { empStatus: 'OFFICIAL' } })

// TO-BE
async getMonthlyTimesheet(entityId: string, year: number, month: number)
  → this.employeeRepo.find({ where: { entId: entityId, empStatus: 'OFFICIAL' } })
```

#### 1-3: leave.controller.ts + leave.service.ts

**Controller**: `@UseGuards(EntityGuard)` 추가

**Service 변경** — `getLeaveBalances()`:
```typescript
// AS-IS
async getLeaveBalances(year: number)
  → this.employeeRepo.find({ where: [{ empStatus: 'OFFICIAL' }, { empStatus: 'PROBATION' }] })

// TO-BE
async getLeaveBalances(entityId: string, year: number)
  → this.employeeRepo.find({ where: [
      { entId: entityId, empStatus: 'OFFICIAL' },
      { entId: entityId, empStatus: 'PROBATION' }
    ] })
```

#### 1-4: ot-record.controller.ts + ot-record.service.ts

**Controller**: `@UseGuards(EntityGuard)` 추가

**Service 변경** — `getMonthlyOtRecords()`:
```typescript
// TO-BE: 직원 조회 시 entityId 필터 추가
this.employeeRepo.find({ where: { entId: entityId, empStatus: In(['OFFICIAL', 'PROBATION']) } })
```

#### 1-5: payroll.controller.ts + payroll.service.ts

**Controller**: `@UseGuards(EntityGuard)` 추가 (AdminGuard/ManagerGuard와 병행)

**Service 변경** — `getPeriods()`:
```typescript
// AS-IS
async getPeriods()
  → this.periodRepo.find({ relations: ['details'], order: {...} })

// TO-BE
async getPeriods(entityId: string)
  → this.periodRepo.find({ where: { entId: entityId }, relations: ['details'], order: {...} })
```

#### 1-6: severance.controller.ts + severance.service.ts

**Controller**: `@UseGuards(EntityGuard)` 추가

**Service**: 모든 퇴직금 조회/계산 메서드에 entityId 필터 추가

#### 1-7: hr-report.controller.ts + hr-report.service.ts

**Controller**: `@UseGuards(EntityGuard)` 추가

**Service**: 리포트 생성 메서드에 entityId 전달

**사이드 임팩트**: HR 모듈은 가장 많은 파일 변경, 각 서비스의 파라미터 시그니처가 변경되므로 해당 서비스를 호출하는 다른 곳도 확인 필요

### 7단계: 빌드 및 검증

1. `npm run build` — 전체 빌드 확인
2. 기능 검증 체크리스트:
   - [ ] 법인 A 로그인 → HR 직원 목록에서 법인 B 직원이 보이지 않는지 확인
   - [ ] 법인 A 로그인 → 회계 거래내역에서 법인 B 데이터가 보이지 않는지 확인
   - [ ] 법인 A 로그인 → 거래처 목록에서 법인 B 거래처가 보이지 않는지 확인
   - [ ] 법인 A 로그인 → 프로젝트 목록/상세에서 법인 B 프로젝트가 보이지 않는지 확인
   - [ ] SUPER_ADMIN 로그인 → 모든 법인 데이터 접근 가능한지 확인
   - [ ] 기존 기능 정상 동작 (CRUD, 필터, 검색 등)

## 작업 순서 및 의존성

```
1단계 (HR 격리) ← 가장 광범위, 최우선
    ↓
2단계 (회계 격리)
    ↓
3단계 (거래처 격리) ← 엔티티 컬럼 확인 필요
    ↓
4단계 (웹메일 격리) ← 엔티티 컬럼 존재 여부에 따라 범위 결정
    ↓
5단계 (KMS 보완)
    ↓
6단계 (멤버/Cell/프로젝트 보완)
    ↓
7단계 (빌드/검증)
```

- 1~2단계: 독립적, 핵심 우선순위
- 3~4단계: DB 스키마 확인 후 진행
- 5~6단계: 보완 작업 (부분 격리 → 완전 격리)

## 사이드 임팩트 분석

| 영역 | 영향도 | 설명 |
|------|:---:|------|
| **HR API** | 높음 | 대부분의 서비스 메서드 시그니처 변경 |
| **회계 API** | 중간 | 거래 관련 메서드 시그니처 변경 |
| **거래처** | 중간 | 전체 CRUD에 entityId 추가 |
| **프로젝트** | 중간 | Guard 적용됨, 7개 엔드포인트 entityId 전달 누락 수정 |
| **프론트엔드** | 없음 | X-Entity-Id 헤더 이미 자동 주입 |
| **SUPER_ADMIN** | 없음 | EntityGuard가 자동 바이패스 |
| **기존 데이터** | 낮음 | 단일 법인이면 영향 없음, 다중 법인이면 데이터 정리 필요 |
| **성능** | 없음 | WHERE 조건 추가는 인덱스 활용으로 성능 영향 미미 |
