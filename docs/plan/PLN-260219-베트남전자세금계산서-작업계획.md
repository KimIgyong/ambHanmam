### 1.1 기존 빌링 모듈 활용 가능 요소

- **TypeORM 마이그레이션**: `synchronize: true`인 경우 엔티티 수정만으로 반영 (확인 필요)
- **pdfkit**: 이미 사용 중, 추가 설치 불필요
- **XML 생성**: `fast-xml-parser` 또는 직접 템플릿 방식 (추가 패키지 최소화)

### STEP 4: GDT 표준 XML 생성 서비스

**새 파일**: `apps/api/src/domain/billing/service/einvoice/einvoice-xml.service.ts`

핵심 로직:
1. InvoiceEntity + InvoiceItemEntity[] + PartnerEntity + HrEntityEntity 데이터 수집
2. GDT XML 스키마(`<HDon>`) 구조로 매핑
3. 필수 항목 검증:
   - 판매자 MST(`entity.entTaxId`) 존재 여부
   - 구매자 MST(`partner.ptnTaxId`) 존재 여부
   - 최소 1개 이상의 invoice item 존재
4. XML 문자열 생성 (UTF-8)

XML 매핑:

| AMB 데이터 | GDT XML 필드 |
|-----------|-------------|
| `entity.entName` | `NBan > Ten` |
| `entity.entTaxId` | `NBan > MST` |
| `entity.entAddress` | `NBan > DChi` |
| `partner.ptnCompanyName` | `NMua > Ten` |
| `partner.ptnTaxId` | `NMua > MST` |
| `partner.ptnAddress` | `NMua > DChi` |
| `item.itmDescription` | `HHDVu > THHDVu` |
| `item.itmQuantity` | `HHDVu > SLuong` |
| `item.itmUnitPrice` | `HHDVu > DGia` |
| `item.itmAmount` | `HHDVu > ThTien` |
| `invoice.invTaxRate` | `HHDVu > TSuat` |
| `invoice.invSubtotal` | `TToan > TgTCThue` |
| `invoice.invTaxAmount` | `TToan > TgTThue` |
| `invoice.invTotal` | `TToan > TgTTTBSo` |

### STEP 5: E-Invoice 오케스트레이션 서비스

**새 파일**: `apps/api/src/domain/billing/service/einvoice/einvoice.service.ts`

#### 메서드: `issueEinvoice(invoiceId: string, userId: string)`

```
1. 인보이스 조회 (+ items, partner, entity relations)
2. 검증:
   - invApprovalStatus === 'APPROVED_ADMIN' 필수
   - invEinvStatus === 'NONE' 또는 'FAILED' 만 허용
   - entity.entCountry === 'VN' 확인
   - partner.ptnTaxId 존재 확인
3. invEinvStatus = 'PENDING' 업데이트
4. EinvoiceXmlService로 XML 생성
5. TvanApiService.issueInvoice() 호출
6. 성공 시:
   - invEinvStatus = 'ISSUED'
   - invEinvNumber, invEinvGdtCode, invEinvIssuedAt, invEinvLookupUrl 저장
   - 서명된 XML/PDF 파일 저장 → 경로 저장
7. 실패 시:
   - invEinvStatus = 'FAILED'
   - invEinvError = 에러 메시지
```

#### 메서드: `cancelEinvoice(invoiceId: string, reason: string, userId: string)`

```
1. invEinvStatus === 'ISSUED' 검증
2. TvanApiService.cancelInvoice() 호출
3. invEinvStatus = 'CANCELLED' 업데이트
```

#### 메서드: `getEinvoiceInfo(invoiceId: string)`

```
1. 인보이스 조회
2. 전자세금계산서 필드 반환
```

## 3. 사이드 임팩트 분석

### 3.1 기존 기능 영향

| 기능 | 영향 | 대응 |
|------|------|------|
| InvoiceEntity | 컬럼 추가 (기존 데이터는 `invEinvStatus=NONE` 기본값) | 마이그레이션 안전 |
| InvoiceMapper | 응답 필드 추가 | 기존 프론트엔드는 추가 필드 무시 |
| InvoiceService | 변경 없음 (CRUD 그대로) | 전자세금계산서는 별도 서비스에서 처리 |
| InvoiceApprovalService | 변경 없음 | 발행은 승인 후 별도 액션 |
| KR 법인 인보이스 | 영향 없음 | VN 엔티티일 때만 전자세금계산서 UI 노출 |
| 자동 인보이스 생성 | 영향 없음 | 자동 생성은 DRAFT 상태까지만 |

### 3.2 리스크

| 리스크 | 확률 | 영향도 | 대응 |
|--------|------|--------|------|
| TVAN 업체 미선정 | 높음 | 중 | Mock 서비스로 Phase 1 완료, 실연동은 Phase 2 |
| XML 스키마 불일치 | 중 | 높음 | TVAN 업체 API 문서 확보 후 검증 필요 |
| 디지털 서명 인증서 미확보 | 중 | 높음 | TVAN 업체가 대행 서명하므로 Phase 1에서는 무관 |
| DB 마이그레이션 실패 | 낮음 | 높음 | nullable 컬럼만 추가, default 값 설정 |
