## 2. 상세 작업 계획

### 2.1 백엔드 Entity (단계 1)

| 파일 | 테이블 | 주요 컬럼 |
|------|--------|----------|
| `meeting-note-project.entity.ts` | `amb_meeting_note_projects` | mnp_id, mtn_id, pjt_id |
| `meeting-note-participant.entity.ts` | `amb_meeting_note_participants` | mnpt_id, mtn_id, usr_id |
| `meeting-note-issue.entity.ts` | `amb_meeting_note_issues` | mni_id, mtn_id, iss_id |
| `meeting-note-comment.entity.ts` | `amb_meeting_note_comments` | mnc_id, mtn_id, mnc_author_id, mnc_content |
| `meeting-note-rating.entity.ts` | `amb_meeting_note_ratings` | mnr_id, mtn_id, usr_id, mnr_rating |

#### 2.1.2 기존 Entity 수정

**`meeting-note.entity.ts`:**
- mtn_assignee_id (UUID FK → amb_users, nullable) 컬럼 추가
- OneToMany 관계 5개 추가 (projects, participants, issues, comments, ratings)
- ManyToOne assignee 관계 추가

### 2.2 공유 타입 (단계 2)

**`packages/types/src/domain.types.ts`:**
```typescript
// MeetingNoteResponse 확장
+ assigneeId: string | null
+ assigneeName: string | null  
+ participants: MeetingNoteParticipantResponse[]
+ projects: MeetingNoteProjectResponse[]
+ issues: MeetingNoteIssueResponse[]
+ commentCount: number
+ avgRating: number | null
+ ratingCount: number
+ myRating: number | null

// 신규 타입
+ MeetingNoteParticipantResponse { userId, name }
+ MeetingNoteProjectResponse { projectId, name, code }
+ MeetingNoteIssueResponse { issueId, title, status }
+ MeetingNoteCommentResponse { commentId, meetingNoteId, authorId, authorName, content, createdAt }
+ MeetingNoteRatingResponse { ratingId, meetingNoteId, userId, rating, createdAt }
```

## 3. 사이드 임팩트 분석

| 영역 | 임팩트 | 대응 |
|------|--------|------|
| KMS 이벤트 | 기존 MeetingNote CREATED/UPDATED 이벤트에 새 데이터 포함 필요 | Mapper에서 자동 반영 |
| 번역 시스템 | MeetingNoteResponse 구조 변경 → 기존 TranslationPanel 호환 | sourceFields 변경 없음 (title, content만) |
| Entity 등록 | 5개 Entity를 app.module.ts에 등록 안 하면 에러 | 체크리스트에 포함 |
| 기존 API 호환 | 응답에 새 필드 추가 (추가만이므로 하위 호환) | optional 필드로 처리 |
