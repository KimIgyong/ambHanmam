## 2. 시스템 개발 현황 분석

### 2.1 영향 분석

| 구성요소 | 현재 상태 | 변경 필요 | 사이드 임팩트 |
|---------|----------|----------|-------------|
| `amb_users.usr_group_code` | `ADMIN_GROUP` / `USER_GROUP` | **변경 없음** | 없음 |
| `EntityGuard` | `user.role === 'ADMIN'` 체크 | 이미 수정 완료 (SUPER_ADMIN 포함) | 없음 |
| `DataScopeInterceptor` | `user.group === 'ADMIN_GROUP'` | **변경 없음** | 없음 |
| `auth.store.ts` | `user?.group === 'ADMIN_GROUP'` | **변경 없음** | 없음 |
| i18n `settings.json` (3개 언어) | 기존 명칭 | **표시 명칭 변경** | UI 텍스트만 변경 |
| `RolePermissionTab.tsx` | 주석에 `ADMIN_GROUP`/`USER_GROUP` | **주석 업데이트** | 없음 |
| `MemberDetailPage.tsx` | department 라벨 사용 | **Function 라벨 추가** (선택) | 없음 |

### 2.2 변경하지 않는 항목

- DB 컬럼값 (`ADMIN_GROUP`, `USER_GROUP`) — 다수 백엔드 로직에서 사용
- 라우트 경로 (`/settings/entities`, `/settings/members`, `/settings/permissions`)
- 부서/그룹 CRUD 로직 — 기 구현 완료
- `menu-code.constant.ts`의 ROLES 배열, DEFAULT_PERMISSIONS — 유지

## 3. 단계별 구현 계획

### Phase 1: i18n 명칭 변경 (핵심)

#### 1-1. `locales/ko/settings.json` 수정

```json
// AS-IS
"groups": {
  "ADMIN_GROUP": "관리자 그룹",
  "USER_GROUP": "일반 사용자 그룹"

"adminGroupNote": "관리자 그룹(ADMIN_GROUP)은 모든 메뉴에 항상 접근 가능하며 변경할 수 없습니다."

// TO-BE
"groups": {
  "ADMIN_GROUP": "관리자 등급 그룹 (Admin Level Group)",
  "USER_GROUP": "사용자 등급 그룹 (User Level Group)"

"adminGroupNote": "관리자 등급 그룹(Admin Level Group)은 모든 메뉴에 항상 접근 가능하며 변경할 수 없습니다."
```

### Phase 2: 프론트엔드 코드 정리

#### 2-1. `RolePermissionTab.tsx` 주석 정리

```tsx
// AS-IS
type GroupTab = 'ADMIN_GROUP' | 'USER_GROUP';
// ADMIN_GROUP: 잠금 (전체 접근, 수정 불가)
// USER_GROUP: 수정 가능

// TO-BE
type GroupTab = 'ADMIN_GROUP' | 'USER_GROUP';
// Admin Level Group: 잠금 (전체 접근, 수정 불가)
// User Level Group: 수정 가능
```

#### 2-2. `MemberDetailPage.tsx` 기본 정보 라벨 정리 (선택)

현재 기본 정보 표시 순서: Name, Email, Role, Department, Job Title, Company Email, Joined At

요구사항 순서: Name, Email, Role, Function, Job Title

- `Department` 라벨을 `Function`으로 표현할 경우 i18n 키 변경 필요
- 현재 `usr_department`가 레거시 텍스트 필드이므로, 의미상 "Function(직능)"으로 재해석 가능
- **결정 필요**: department 라벨을 Function으로 변경할지, 아니면 별도 `usr_function` 컬럼 추가할지

### Phase 3: 권한관리 그룹 표시 이슈 점검

#### 3-1. `useGroupList` 훅 동작 확인

`RolePermissionTab.tsx`에서 `useGroupList()` 호출 → `sortedGroups`가 빈 배열이면 그룹 컬럼이 표시되지 않음

- 원인 가능성: `useGroupList`가 `membersService.getGroups()` 호출 시 403 (EntityGuard) 발생
- `group.controller.ts`의 Guard 설정 확인 필요
- 그룹 API가 `X-Entity-Id` 헤더 없이도 동작하는지 확인

#### 3-2. 그룹 관련 API 정상 동작 검증

```bash
# 스테이징 서버에서 그룹 API 테스트
curl -H "Authorization: Bearer <token>" \
     -H "X-Entity-Id: <entity_id>" \
     https://mng.amoeba.site/api/v1/groups
```
