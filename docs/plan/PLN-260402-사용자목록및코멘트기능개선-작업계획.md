## 2. 단계별 구현 계획

### Phase 1: 사용자 리스트 개선 (Active 필터 + USER_LEVEL 우선 정렬)

#### 2.1.1 백엔드 — Active 사용자만 반환
**파일**: `apps/api/src/domain/members/service/member.service.ts`

**변경 내용**:
- `findAll()` 최종 조회 시 `usrStatus = 'ACTIVE'` 조건 추가
- 또는 최종 filter 단계에서 `usrStatus !== 'ACTIVE'` 제외

**구현**:
```typescript
// 기존: 전체 users 조회 후 entityId로 필터
const users = await this.userRepository.find({
  order: { usrCreatedAt: 'DESC' },
});

// 변경: ACTIVE 조건 추가
const users = await this.userRepository.find({
  where: { usrStatus: 'ACTIVE' },
  order: { usrCreatedAt: 'DESC' },
});
```

**사이드 임팩트**: 
- `useMemberList()` 전체 호출처에 영향 → INACTIVE 사용자가 모든 드롭다운에서 사라짐
- 이미 할당된 Assignee/Participant가 INACTIVE라면 목록에는 안 보이지만 기존 데이터는 유지됨
- Entity 설정의 멤버 관리 페이지는 별도 API (`entity-settings/members`) 사용 → 영향 없음

#### 2.1.2 프론트엔드 — USER_LEVEL 우선 정렬 유틸 추가
**파일**: `apps/web/src/domain/members/hooks/useMembers.ts`

**변경 내용**:
- `useMemberList()` 반환 시 `select` 옵션으로 정렬 적용
- 정렬 기준: USER_LEVEL > CLIENT_LEVEL > 기타 → 같은 레벨 내 이름 오름차순

**구현**:
```typescript
export const useMemberList = () => {
  const entityId = useEntityStore((s) => s.currentEntity?.entityId);
  return useQuery({
    queryKey: memberKeys.list(entityId),
    queryFn: () => membersService.getMembers(),
    enabled: !!entityId,
    select: (data) => sortMembersByLevel(data),
  });
};

function sortMembersByLevel(members: MemberResponse[]): MemberResponse[] {
  const levelOrder: Record<string, number> = {
    ADMIN_LEVEL: 0,
    USER_LEVEL: 0,
    CLIENT_LEVEL: 1,
    PARTNER_LEVEL: 2,
  };
  return [...members].sort((a, b) => {
    const la = levelOrder[a.levelCode || 'USER_LEVEL'] ?? 9;
    const lb = levelOrder[b.levelCode || 'USER_LEVEL'] ?? 9;
    if (la !== lb) return la - lb;
    return (a.name || '').localeCompare(b.name || '');
  });
}
```

**사이드 임팩트**:
- 모든 `useMemberList()` 호출처에 자동 적용 (AssigneeSelector, ParticipantSelector, MentionInput 등)
- TalkMessageInput은 채널 멤버 사용 → 영향 없음

## 4. 사이드 임팩트 분석

### Low Risk
- **Phase 1 (Active 필터)**: 이미 비활성 사용자는 EntityUserRole `eurStatus='ACTIVE'`로 일부 필터링됨. `usrStatus` 추가 필터는 더 엄격한 보완
- **Phase 2 (멘션 방향)**: CSS 위치 변경만 → 기능 로직 무관
- **Phase 3 (스타일)**: 순수 CSS 변경 → 기능 무관

### Medium Risk
- **Phase 1 (정렬)**: `useMemberList()` 전체 호출처 영향 → 다만 정렬만 변경이므로 기능 장애 없음
- **Phase 4 (Reply)**: DB 스키마 변경 → 수동 SQL 필요, 기존 코멘트는 `parent_id=NULL`로 정상 유지

### HIGH Risk
없음
