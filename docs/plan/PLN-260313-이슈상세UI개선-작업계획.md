### Phase 2: 상태 이력 표시 방식 변경 (R2)

**작업 범위**: 프론트엔드 컴포넌트 + 번역 키 추가

| # | 작업 | 파일 |
|---|------|------|
| 2-1 | IssueStatusHistory 컴포넌트 레이아웃 재구성 | `apps/web/src/domain/issues/components/IssueStatusHistory.tsx` |
| 2-2 | 번역 키 추가 | `apps/web/src/locales/{ko,en,vi}/issues.json` |

**컴포넌트 변경 상세** (`IssueStatusHistory.tsx`):

AS-IS 레이아웃:
```
[이전상태] → [변경상태]  유저이름 - 메모  날짜
```

TO-BE 레이아웃:
```
유저이름 이(가) 상태를 [이전상태]에서 [변경상태](으)로 변경  날짜
 └ 메모 (있을 경우)
```

**추가 번역 키**:
```json
{
  "statusHistory": "상태 변경 이력",
  "statusChangedFromTo": "{{user}} 이(가) 상태를 [{{fromStatus}}]에서 [{{toStatus}}](으)로 변경",
  "statusChangedNote": "메모"
}
```

**사이드 임팩트**: 없음 (표시 방식만 변경)

### Phase 3: 코멘트 저장 시 이슈 상태 표기 (R3)

**작업 범위**: DB + 백엔드 + 프론트엔드

| # | 작업 | 파일 |
|---|------|------|
| 3-1 | DB 컬럼 추가 (isc_issue_status) | SQL 마이그레이션 |
| 3-2 | IssueCommentEntity에 필드 추가 | `apps/api/src/domain/issues/entity/issue-comment.entity.ts` |
| 3-3 | IssueMapper.toCommentResponse에 상태 매핑 | `apps/api/src/domain/issues/mapper/issue.mapper.ts` |
| 3-4 | IssueService.addIssueComment에서 현재 상태 저장 | `apps/api/src/domain/issues/service/issue.service.ts` |
| 3-5 | IssueCommentResponse 타입에 필드 추가 | `packages/types/src/domain.types.ts` |
| 3-6 | 코멘트 UI에 상태 뱃지 표시 | `apps/web/src/domain/issues/components/IssueDetailModal.tsx` |

**DB 변경**:
```sql
ALTER TABLE amb_issue_comments 
ADD COLUMN isc_issue_status VARCHAR(20) DEFAULT NULL;
```

**엔티티 변경** (`issue-comment.entity.ts`):
```typescript
@Column({ name: 'isc_issue_status', length: 20, nullable: true })
iscIssueStatus: string | null;
```

**타입 변경** (`domain.types.ts`):
```typescript
export interface IssueCommentResponse {
  // ... 기존 필드
  issueStatus?: string;  // 코멘트 작성 시점의 이슈 상태
}
```

**사이드 임팩트**:
- 기존 코멘트의 `issueStatus`는 null → UI에서 null 처리 필요
- TypeORM `synchronize: true` 환경(dev/staging)에서는 자동 적용, production은 수동 SQL 필요

### Phase 4: 코멘트 작성 시 이미지/파일 첨부 (R4)

**작업 범위**: 프론트엔드 (코멘트 입력 영역 교체)

| # | 작업 | 파일 |
|---|------|------|
| 4-1 | 코멘트 입력을 RichTextEditor로 교체 | `apps/web/src/domain/issues/components/IssueDetailModal.tsx` |
| 4-2 | 코멘트 표시를 HTML 렌더링으로 변경 | `apps/web/src/domain/issues/components/IssueDetailModal.tsx` |
| 4-3 | RichTextEditor minHeight 조정 (코멘트용 컴팩트) | props 전달 |

**구현 방식**:
- `MentionInput` 제거 → `RichTextEditor` 사용
- RichTextEditor에 이미 드래그앤드롭, 이미지 업로드, URL 삽입 기능 내장
- content는 HTML 형식으로 저장
- 기존 plain text 코멘트와 호환: `dangerouslySetInnerHTML + DOMPurify` 적용

**멘션 기능 대안**:
- MentionInput의 @멘션 기능은 별도 구현 과제로 분리 (TipTap mention extension 추가 필요)
- 1차 구현에서는 RichTextEditor로 교체하여 이미지/파일 첨부 우선 적용
- 멘션 기능은 후속 개선

**코멘트 HTML 렌더링 변경**:
```tsx
// AS-IS
<p className="mt-1 text-sm text-gray-600 whitespace-pre-wrap">{comment.content}</p>

// TO-BE
<div
  className="mt-1 prose prose-sm max-w-none text-gray-600 [&_img]:max-w-full [&_img]:rounded"
  dangerouslySetInnerHTML={{ __html: comment.content }}
/>
```

**사이드 임팩트**:
- 기존 plain text 코멘트: HTML 태그 없으므로 `dangerouslySetInnerHTML`로 렌더링해도 동일 표시
- 멘션(`@사용자`) 기능 일시적 미지원 → 후속 개선 필요
- XSS 방지: content sanitize 처리 필요

## 3. 구현 순서 (권장)

```
Phase 1 (R1) → Phase 2 (R2) → Phase 3 (R3) → Phase 4 (R4)
```

- Phase 1, 2: 프론트엔드만 변경, 독립적
- Phase 3: 백엔드 + DB 변경 필요, Phase 4에 의존하지 않음
- Phase 4: 프론트엔드 변경, Phase 3 이후 적용 권장
