## Step 1: 백엔드 — 개인 Drive 루트 폴더 조회 추가

### 1-1. `GoogleDriveService` 수정

**파일**: `apps/api/src/infrastructure/external/google-drive/google-drive.service.ts`

**변경 내용**: 개인 드라이브(My Drive) 루트 폴더 목록을 조회하는 기능은 이미 `listFiles(folderId)` 메서드로 구현되어 있음. Google Drive API에서 `folderId = 'root'`를 전달하면 내 드라이브의 최상위 폴더를 조회할 수 있음.

**추가 작업**: 없음 — 기존 `listFiles('root')` 호출로 충분.

### 1-2. `DriveSettingsController` 수정

**파일**: `apps/api/src/domain/settings/controller/drive-settings.controller.ts`

**변경 내용**: 개인 드라이브 루트 폴더 목록 조회 엔드포인트 추가.

```typescript
// 기존 GET /shared-drives 와 동일 패턴
@Get('my-drive-folders')
async listMyDriveFolders() {
  const data = await this.driveSettingsService.listFolders('root');
  return { success: true, data, timestamp: new Date().toISOString() };
}
```

> **대안 검토**: 기존 `GET /folders?parentId=root`로도 가능하지만, 프론트에서 "내 드라이브" 클릭 시의 명시적 엔트리 포인트가 되므로 별도 엔드포인트가 더 명확.
> → **결론**: 기존 `/folders?parentId=root` 활용으로 충분. 별도 엔드포인트 불필요. 프론트에서 `parentId = 'root'`로 호출.

**최종**: 백엔드 컨트롤러 변경 없음. 기존 `GET /settings/drive/folders?parentId=root` 활용.

### 1-3. `DriveSettingsService` 수정 — 자동 폴더 등록

**파일**: `apps/api/src/domain/settings/service/drive-settings.service.ts`

**변경 내용**: `updateSettings()` 메서드에서 회사 공용 폴더가 저장될 때 `amb_drive_folders`에 자동 등록.

```typescript
// updateSettings() 메서드 끝에 추가
if (saved.drsBillingRootFolderId && saved.drsBillingRootFolderName) {
  await this.autoRegisterCompanyFolder(
    saved.drsBillingRootFolderId,
    saved.drsBillingRootFolderName,
    userId,
  );
}
```

**신규 private 메서드**:
```typescript
private async autoRegisterCompanyFolder(
  folderId: string,
  folderName: string,
  userId: string,
): Promise<void> {
  // amb_drive_folders에 이미 등록되어 있는지 확인
  const existing = await this.driveFolderRepo.findOne({
    where: { drfFolderId: folderId, drfDeletedAt: IsNull() },
  });
  if (existing) return; // 이미 등록됨

  // soft-deleted 된 동일 폴더가 있으면 복구
  const deleted = await this.driveFolderRepo.findOne({
    where: { drfFolderId: folderId },
    withDeleted: true,
  });
  if (deleted) {
    deleted.drfDeletedAt = null;
    deleted.drfFolderName = folderName;
    await this.driveFolderRepo.save(deleted);
    return;
  }

  // 신규 등록
  await this.driveFolderRepo.save(
    this.driveFolderRepo.create({
      drfFolderId: folderId,
      drfFolderName: folderName,
      drfDriveType: 'shared', // 기본값, 실제로는 선택된 드라이브 타입
      drfCreatedBy: userId,
    }),
  );
}
```

**의존성 추가**: `DriveSettingsService`에 `DriveFolderEntity` Repository 주입 필요.

```typescript
// constructor에 추가
@InjectRepository(DriveFolderEntity)
private readonly driveFolderRepo: Repository<DriveFolderEntity>,
```

**모듈 변경**: `SettingsModule`에서 `DriveFolderEntity`를 `TypeOrmModule.forFeature()`에 추가.

## Step 2: 프론트엔드 — 폴더 브라우저 확장

### 2-1. `DriveSettingsPage` 수정

**파일**: `apps/web/src/domain/settings/pages/DriveSettingsPage.tsx`

**변경 내용**: 폴더 브라우저에 "내 드라이브" 탭/항목 추가.

#### 현재 로직
```
handleBrowseDrives() → listSharedDrives() → setSharedDrives([...]) → 공유 드라이브만 표시
```

#### 변경 후 로직
```
handleBrowseDrives() → listSharedDrives() → 결과에 "내 드라이브" 가상 항목 prepend
  → [{ id: 'root', name: '내 드라이브' }, ...sharedDrives]
```

**구체적 변경**:

```typescript
// handleBrowseDrives() 수정
const handleBrowseDrives = async () => {
  // ... (기존 이메일 저장 로직)
  try {
    const drives = await listDrivesMutation.mutateAsync();
    // "내 드라이브" 가상 항목을 맨 앞에 추가
    const allDrives: SharedDriveItem[] = [
      { id: 'root', name: t('settings:drive.myDrive') },
      ...drives,
    ];
    setSharedDrives(allDrives);
    setShowDriveBrowser(true);
    // ...
  } catch (error) { /* ... */ }
};
```

**UI 변경**:
- "내 드라이브" 항목에 별도 아이콘 (`User` 또는 `HardDrive`) 적용
- 공유 드라이브 항목은 기존 `HardDrive` 아이콘 유지
- breadcrumb에서 "내 드라이브" / "공유 드라이브 이름" 구분 표시

## 사이드 임팩트 분석

### 1. Billing 문서 관리 (`BillingDocumentService`)

| 항목 | 영향 | 설명 |
|------|------|------|
| `getBillingRootFolderId()` | ❌ 영향 없음 | 동일 필드(`drsBillingRootFolderId`)를 읽기만 함 |
| `ensurePartnerFolder()` | ❌ 영향 없음 | 루트 폴더 하위에 별도 계층 구조 생성 |
| `uploadDocument()` | ❌ 영향 없음 | 폴더 ID만 사용, 폴더 자체는 건드리지 않음 |

**결론**: Billing 모듈은 `drsBillingRootFolderId`를 읽기만 하므로 이 필드의 의미가 "Billing 루트"에서 "회사 공용 폴더"로 확장되어도 기존 동작에 영향 없음. Billing은 이 폴더 하위에 `{country}/{partnerType}/{partnerName}/` 계층을 독립적으로 관리함.

### 2. Drive 모듈 (`DriveService`)

| 항목 | 영향 | 설명 |
|------|------|------|
| `registerFolder()` | ⚠️ 주의 필요 | `drfFolderId` unique 제약 → 중복 등록 시 에러 발생 가능 |
| `getRegisteredFolders()` | ❌ 영향 없음 | 자동 등록된 폴더도 정상 조회됨 |
| `unregisterFolder()` | ⚠️ 주의 필요 | 사용자가 수동으로 등록 해제 시 설정과 불일치 가능 |

**대응**:
- 자동 등록 시 `findOne({ drfFolderId })` 확인 후 skip
- soft-deleted 레코드 존재 시 복구 (`drfDeletedAt = null`)
- 문서함에서 등록 해제 시 `amb_drive_settings`의 폴더 ID는 유지 (설정은 별도 관리)

### 3. GoogleDriveService

| 항목 | 영향 | 설명 |
|------|------|------|
| `listFiles('root')` | ⚠️ 동작 확인 필요 | `'root'`를 부모로 사용 시 위임 사용자의 내 드라이브 루트 접근 |
| `supportsAllDrives` | ❌ 영향 없음 | 공유 Drive + 개인 Drive 모두 지원 |

**대응**:
- `listFiles('root')` 동작은 Google Drive API v3에서 공식 지원됨
- 위임 사용자가 없으면 서비스 계정 자체의 Drive(빈 드라이브) 반환 → 적절한 에러 메시지 필요
