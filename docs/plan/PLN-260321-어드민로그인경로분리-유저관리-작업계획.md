# PLAN-어드민로그인경로분리-유저관리-작업계획-20260321

## 1. 시스템 개발 현황 분석

### 1.1 백엔드 Admin 모듈

| 항목 | 현황 |
|------|------|
| **모듈** | `AdminModule` — `AdminController` + `AdminService` 단일 구조 |
| **가드** | `AdminLevelGuard` (`level === 'ADMIN_LEVEL'`). SuperAdminGuard **미존재** |
| **엔드포인트** | 14개 — 사용자 목록, 법인 CRUD, 역할 관리, 드라이브/AI 설정 |
| **DTO** | 전용 DTO 없음. 타 도메인(`hr`, `settings`) DTO 재사용 |
| **데코레이터** | `@AdminOnly()` 존재하나 AdminController에서 미사용 (직접 `@UseGuards(AdminLevelGuard)`) |

### 1.2 백엔드 로그인 API

- `POST /api/v1/auth/login` — `entity_code` 미전달 시 ADMIN_LEVEL 자동 매칭 로직 이미 구현
- 프론트에서만 `entity_code`를 필수로 요구하고 있어 어드민 로그인이 불편

### 1.3 프론트엔드 Admin

| 항목 | 현황 |
|------|------|
| **라우터** | `/admin` 하위 28개 라우트 — `AdminGuard` > `AdminLayout` |
| **AdminGuard** | `isAdmin()` = `user.level === 'ADMIN_LEVEL'` 체크 |
| **AdminLayout** | 사이드바 3섹션 (Dashboard, Services, Settings 22항목) |
| **AdminDashboardPage** | 4 스탯 카드(3개 미연동) + Quick Actions (SETTINGS_CARDS 재활용) |
| **서비스** | `admin.service.ts`, `partner-admin.service.ts`, `partner-app-admin.service.ts` |
| **i18n** | `ko/admin.json` — `sidebar`, `dashboard`, `partner`, `partnerApps` 등 7 섹션 |

### 1.4 핵심 갭

1. **AdminLoginPage**: 없음 → 신규 구현 필요
2. **SuperAdminGuard**: 없음 → 어드민 유저 생성/삭제에 필수
3. **Admin 사용자 CRUD API**: 없음 → 전체 신규
4. **Partner 사용자 전용 API**: 없음 → `findAllUsers`에 level 필터 추가 또는 전용 엔드포인트
5. **Partner 초대 전용 API**: 부분 존재 (invitations 조회는 있으나 partner 필터 없음)

## 2. 아키텍처 결정

### 2.1 로그인 경로 전략

**결정: AdminLoginPage 신규 + 기존 `/login` 하위 호환 유지**

- `/admin/login` → 어드민 전용 (법인 선택 없음, 이메일+비밀번호만)
- 기존 `/login` → 일반/클라이언트 사용자 (어드민도 사용 가능 — 하위 호환)
- 백엔드 API 변경 없음 — 기존 `entity_code` 미전달 로직 활용

### 2.2 SuperAdminGuard 전략

**결정: 백엔드 `SuperAdminGuard` 신규 + 프론트 `isSuperAdmin()` 활용**

- `SUPER_ADMIN` 역할만 어드민 계정 생성/역할변경/삭제 가능
- `ADMIN` 역할은 어드민 목록 조회 + 본인 비밀번호 초기화만 가능

### 2.3 API 확장 전략

**결정: 기존 `AdminController`에 엔드포인트 추가**

- 별도 컨트롤러 분리 대신, 기존 `AdminController`에 admin-users, partner-users, partner-invitations 엔드포인트 그룹 추가
- `AdminService`에 메서드 추가
- 이유: 모두 `AdminLevelGuard` 하위이고, 기존 `UserEntity` 리포지토리를 공유

### 2.4 프론트엔드 서비스 전략

**결정: 기존 `admin.service.ts`에 메서드 추가 + 필요 시 서비스 파일 분리**

- `AdminUsersPage`, `PartnerUsersPage`: `admin.service.ts` 확장
- `PartnerInvitationsPage`: 초대 조회/관리는 `admin.service.ts` 또는 기존 `entitySettingsService` 재사용

### Phase 3: 어드민 유저 관리 (REQ-03)

#### Step 3.1: SuperAdminGuard 백엔드 신규

**신규 파일**: `apps/api/src/global/guard/super-admin.guard.ts`

```typescript
@Injectable()
export class SuperAdminGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    const user = context.switchToHttp().getRequest().user;
    if (!user || user.role !== 'SUPER_ADMIN') {
      throw new ForbiddenException({ errorCode: 'E1003', message: 'SUPER_ADMIN role required' });
    }
    return true;
  }
}
```

## 7. 리스크 및 대응

| 리스크 | 영향 | 대응 |
|--------|------|------|
| `/admin/login`이 AdminGuard 범위에 포함되면 접근 불가 | 어드민이 로그인 페이지에 접근 못 함 | 라우터에서 `/admin/login`을 AdminGuard 블록 위에 독립 배치 |
| 마지막 SUPER_ADMIN 삭제/역할 변경 | SUPER_ADMIN 0명이 되면 복구 불가 | 백엔드에서 SUPER_ADMIN 수 count 체크 → 1명이면 차단 |
| AdminController 코드 비대화 | 단일 컨트롤러에 28개+ 엔드포인트 | 필요 시 Phase 후반에 AdminUserController, AdminPartnerController로 분리 |
| 파트너 초대 발송 시 기존 inviteMember API 의존 | API 시그니처 변경 시 초대 깨짐 | 기존 API 계약 유지, 통합 테스트 검증 |
| 대시보드 통계 API 호출이 성능에 영향 | 4개 API 동시 호출 시 느릴 수 있음 | 각 API에서 `limit=1`로 count만 가져오기, SWR 캐싱 활용 |
