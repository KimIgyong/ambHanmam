### 구현 상세
```typescript
async getUserDeletionPreview(userId: string) {
  // 1. 사용자 조회 (withDeleted: true)
  const user = await userRepo.findOne({
    where: { usrId: userId },
    withDeleted: true,
  });
  
  // 2. 각 연관 테이블 COUNT 쿼리
  const [entityRoles, unitRoles, conversations, talkChannels, ...] = await Promise.all([
    entityUserRoleRepo.count({ where: { usrId: userId } }),
    userUnitRoleRepo.count({ where: { usrId: userId } }),
    conversationRepo.count({ where: { usrId: userId }, withDeleted: true }),
    talkChannelMemberRepo.count({ where: { usrId: userId } }),
    // ... 기타
  ]);
  
  return { user, counts, warnings };
}
```

## 3. Phase 2: 소프트 삭제 개선 + 완전 삭제 API

### 3.1 소프트 삭제 개선

**기존**: `userRepository.softRemove(user)` 만 수행  
**개선**: 추가 정리 작업 수행

```typescript
async improvedSoftDelete(userId: string, adminUserId: string) {
  return this.userRepo.manager.transaction(async (manager) => {
    // 1. 사용자 소프트 삭제
    const user = await manager.findOne(UserEntity, { where: { usrId: userId } });
    await manager.softRemove(user);
    
    // 2. EntityUserRole → INACTIVE 전환
    await manager.update(EntityUserRoleEntity,
      { usrId: userId, eurStatus: 'ACTIVE' },
      { eurStatus: 'INACTIVE' }
    );
    
    // 3. TalkChannelMember → LEFT 상태 전환
    await manager.update(TalkChannelMemberEntity,
      { usrId: userId, tcmStatus: 'ACTIVE' },
      { tcmStatus: 'LEFT' }
    );
    
    // 4. 인증 데이터 정리
    await manager.delete(PushSubscriptionEntity, { usrId: userId });
    await manager.delete(OAuthTokenEntity, { usrId: userId });
    await manager.delete(OAuthAuthorizationCodeEntity, { usrId: userId });
    await manager.delete(PasswordResetEntity, { usrId: userId });
    
    // 5. 감사 로그
    this.logger.warn(`[SOFT-DELETE] Admin ${adminUserId} soft-deleted user ${userId} (${user.usrEmail})`);
  });
}
```

### 3.2 완전 삭제 API

**새 엔드포인트**:
```
DELETE /api/v1/admin/users/:userId/permanent
@AdminOnly()
Body: { level: 1 | 2, confirm_email: string }
```

**삭제 순서 (FK 의존성 역순)**:

#### Level 1: 사용자 계정 + 부수 데이터 삭제

#### Level 2: 전체 데이터 삭제 (Level 1 대체)
```
── Step 1~5: Level 1과 동일 ────────────

── Step 6: 소유 데이터 CASCADE 삭제 ─────
  (추가) 토크 메시지의 리액션/숨김 삭제 → 토크 메시지 삭제
  (추가) 대화의 메시지 삭제 → 대화 삭제
  (추가) 투두의 댓글/참여자/평가 삭제 → 투두 삭제
  (추가) 워크아이템의 댓글 삭제 → 워크아이템 삭제
  (추가) 캘린더의 참여자 삭제 → 캘린더 삭제
  (추가) 회의록의 댓글/참여자/평가 삭제 → 회의록 삭제
  (추가) 회의록 폴더 삭제
  (추가) 업무 보고서 삭제
  (추가) 데일리 미션 삭제
  (추가) 출퇴근의 수정이력 삭제 → 출퇴근 삭제
  (추가) 공지사항의 읽음기록 삭제 → 공지사항 삭제

── Step 7: 참조 NULL 처리 (타인 데이터) ─
  amb_att_attendances            (SET att_approved_by = NULL where approver)
  amb_admin_site_error_logs      (SET sel_usr_id = NULL)
  amb_oauth_open_api_logs        (SET opl_usr_id = NULL)

── Step 8: 사용자 레코드 물리 삭제 ──────
  amb_users                      (DELETE)
```

### 검증 규칙

- `confirm_email`이 대상 사용자 이메일과 불일치 → `400 Bad Request`

## 5. 사이드 임팩트 분석

### 5.1 영향 없음 (예상)
| 영역 | 이유 |
|------|------|
| 포털 고객 삭제 | 별도 모듈 (`portal-bridge`), 간섭 없음 |
| 로그인/인증 | 소프트 삭제된 사용자는 이미 로그인 차단됨 |
| API 권한 체계 | `AdminOnly` 데코레이터 사용, 기존 가드 패턴 동일 |

### 5.2 주의 필요
| 영역 | 리스크 | 대응 |
|------|--------|------|
| **이름 클릭 변경** | 기존 이름 클릭 → 상세 페이지 네비게이션이 삭제 옵션 모달로 변경됨 | 상세 보기 링크를 별도 제공 (이메일 클릭 또는 Actions 열에 상세 아이콘) |
| **Level 2 토크 메시지 삭제** | 채널의 다른 참여자 대화 맥락 손상 | 모달에서 강력한 경고 표시 |
| **Level 1 작성자 NULL** | "알 수 없는 사용자" 표시 필요 | 프론트 전체에서 `user?.name \|\| t('common:deletedUser')` 패턴 이미 사용 확인 필요 |
| **대량 데이터 삭제** | Level 2에서 트랜잭션 크기 검토 | 30개+ 테이블 DELETE → 타임아웃 가능성 → 적절한 인덱스 활용 |

### 5.3 기존 소프트 삭제 호출자
| 호출 위치 | 영향 |
|-----------|------|
| `InternalUsersTab.tsx` → `useDeleteMember()` | 이 위치에서만 사용 → Phase 3에서 모달 방식으로 교체 |
| `MemberDetailPage.tsx` (있을 경우) | 별도 확인 필요 |

## 6. DB 마이그레이션

이 기능은 **기존 테이블/컬럼 변경 없음** (신규 테이블/컬럼 없음).
- 삭제 프리뷰: 기존 테이블 COUNT 쿼리
- 소프트 삭제 개선: 기존 컬럼 UPDATE
- 완전 삭제: 기존 레코드 DELETE/UPDATE

→ **수동 SQL 마이그레이션 불필요**
