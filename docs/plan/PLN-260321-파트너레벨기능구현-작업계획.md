## 2. 구현 전략 — 아키텍처 결정

### 2.1 파트너 소속 조직 설계

**결정**: `amb_partners` 신규 테이블 생성 (빌링 파트너와 분리)

**이유**:
- `amb_bil_partners`는 법인(Entity)에 종속된 거래처이며, `ent_id` FK가 필수
- PARTNER_LEVEL 파트너사는 법인과 독립적인 조직 (HQ와 직접 계약)
- 파트너 앱 등록/심사 등 고유 비즈니스 로직 보유

### 2.2 인증 분리 전략

**결정**: 별도 `PartnerAuthModule` 생성 (`/partner-auth/*` 엔드포인트)

**이유**:
- CLIENT_LEVEL이 `/client-auth/*` 별도 API를 사용하는 패턴과 일치
- 파트너 전용 검증 로직(파트너사 활성 상태, 앱 심사 권한 등) 분리
- `entity_code` 불필요 (파트너는 법인 단위가 아님)

### 2.3 데이터 스코프 전략

**결정**: `PARTNER` 스코프 추가 (partnerId 기반 필터링)

```
ADMIN_LEVEL  → scope: ALL      (전체 데이터)
USER_LEVEL   → scope: OWN_ORG  (소속 법인 데이터)
PARTNER_LEVEL→ scope: PARTNER  (자사 파트너 데이터만)  ← 신규
CLIENT_LEVEL → scope: CLIENT   (배정 프로젝트만)
```

### 2.4 역할 계층 전략

**결정**: PARTNER 역할은 CLIENT처럼 **계층 외 격리** 처리

```typescript
ROLE_HIERARCHY = {
  CLIENT_MEMBER: 0,
  CLIENT_ADMIN: 0,
  PARTNER_MEMBER: 0,   // ← 신규: 계층 외
  PARTNER_ADMIN: 0,    // ← 신규: 계층 외
  VIEWER: 1,
  MEMBER: 2,
  MANAGER: 3,
  MASTER: 4,
  ADMIN: 5,
  SUPER_ADMIN: 6,
};
```

**이유**: PARTNER는 USER_LEVEL과 비교 불가능한 별도 레벨. `@Roles('MANAGER')` 같은 계층 비교가 의미 없음. 파트너 전용 API는 `@RequireLevels('PARTNER_LEVEL')` 데코레이터로 보호.

## 7. 리스크 및 대응

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 기존 `ROLE_HIERARCHY` 비교 로직에서 PARTNER 역할이 0으로 취급 | `@Roles('MEMBER')` 가드에서 PARTNER 사용자 차단 | PARTNER 전용 API는 `@RequireLevels` 사용, `@Roles` 미사용 |
| `DataScopeInterceptor`의 else 분기가 PARTNER를 OWN_ORG로 처리 | PARTNER 사용자가 법인 데이터 접근 시도 | Phase 1에서 PARTNER_LEVEL 분기를 CLIENT 아래에 배치 (else if 순서 주의) |
| `OwnEntityGuard`에서 PARTNER 차단 | 파트너가 Entity Settings 접근 불가 | PARTNER는 OwnEntityGuard 적용 API 미사용 (파트너 전용 API만 사용) |
| User Entity unique 제약 (email + companyId) | PARTNER는 companyId=NULL → email 전역 unique 필요 | PARTNER 초대 시 email+NULL 조합 체크, 또는 별도 unique 인덱스 |
| JWT 쿠키 충돌 (main app과 partner가 같은 도메인) | 동시 로그인 시 토큰 덮어쓰기 | 쿠키명 분리: `access_token` (main) vs `partner_access_token` (partner) |
