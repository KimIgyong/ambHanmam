## 2. 단계별 구현 계획

### STEP 1: EntitySelector를 전역 헤더로 이동

| 작업 | 파일 | 설명 |

| 헤더에 배치 | `apps/web/src/layouts/MainLayout.tsx` | EntitySelector + useEntities 추가 |

### STEP 2: 백엔드 엔티티 ent_id 컬럼 추가

대상 6개 엔티티에 다음 패턴 적용:

```typescript
@Column({ name: 'ent_id', type: 'uuid', nullable: true })
entId: string | null;

@ManyToOne(() => HrEntityEntity, { nullable: true })
@JoinColumn({ name: 'ent_id' })
hrEntity: HrEntityEntity;
```

| 엔티티 | 파일 |
|--------|------|
| NoticeEntity | `apps/api/src/domain/notices/entity/notice.entity.ts` |
| TodoEntity | `apps/api/src/domain/todo/entity/todo.entity.ts` |
| MeetingNoteEntity | `apps/api/src/domain/meeting-notes/entity/meeting-note.entity.ts` |
| WorkScheduleEntity | `apps/api/src/domain/work-schedule/entity/work-schedule.entity.ts` |
| ConversationEntity | `apps/api/src/domain/chat/entity/conversation.entity.ts` |
| BankAccountEntity | `apps/api/src/domain/accounting/entity/bank-account.entity.ts` |

### STEP 4: 백엔드 Service에 법인 필터링 로직 추가

| MeetingNoteService | getMeetingNotes에 법인 필터, createMeetingNote에 entId 설정 |

### STEP 6: 프론트엔드 Hook에 entityId 의존성 추가

12개 훅 파일에 다음 패턴 적용:

```typescript
import { useEntityStore } from '@/domain/hr/store/entity.store';
const entityId = useEntityStore((s) => s.currentEntity?.entityId);
// queryKey에 entityId 포함
// enabled: !!entityId
```

| 훅 파일 | 도메인 |
|---------|--------|

| `useTodos.ts` | 할일 |

| `useAccounts.ts` | 회계 |

| `useMembers.ts` | 멤버 |

## 3. 작업 순서

```
STEP 1 (EntitySelector 헤더 이동)
  ↓
STEP 2 (엔티티 ent_id 컬럼 추가) ← 병렬 가능
  ↓
STEP 3 + STEP 4 + STEP 5 (Controller + Service + Module ← 도메인별 일괄)
  ↓
STEP 6 (프론트엔드 Hook)
  ↓
타입 체크 검증
```

## 4. 사이드 임팩트 분석

| 영역 | 영향 | 대응 |
|------|------|------|
| 기존 데이터 (ent_id = NULL) | 법인 선택 후 조회 불가 | 필요 시 마이그레이션 스크립트로 기본 법인 배정 |
| 기존 API 시그니처 | entityId 파라미터 추가 | optional로 추가하여 하위 호환 유지 |
| EntitySelector 중복 제거 | 도메인별 레이아웃 변경 | MainLayout 단일 배치로 통일 |
| Billing 도메인 | 이미 적용 완료 | 변경 없음 |
| KMS 도메인 | 기존 `user.entityId!` 사용 | `req.entityId`로 통일 |
