# PLAN-포털무료체험자동계정생성및구글연동-작업계획-20260403

## 1. 시스템 개발 현황 분석 (System Baseline)

### 1.1 현재 구현 상태
1. portal register는 portal-api에서 customer/client 생성까지만 수행한다.
 - 근거: [apps/portal-api/src/domain/auth/service/portal-auth.service.ts](apps/portal-api/src/domain/auth/service/portal-auth.service.ts#L23)
2. AMA 계정 생성은 admin 수동 브릿지 액션으로만 수행한다.
 - 근거: [apps/api/src/domain/portal-bridge/controller/portal-bridge.controller.ts](apps/api/src/domain/portal-bridge/controller/portal-bridge.controller.ts#L28)
 - 근거: [apps/web/src/domain/portal-bridge/pages/PortalCustomerPage.tsx](apps/web/src/domain/portal-bridge/pages/PortalCustomerPage.tsx#L241)
3. 전화번호는 프론트/DTO에서 필수이나 DB 컬럼은 nullable이다.
4. Google OAuth는 portal-web/portal-api auth 경로에 미구현 상태다.

### 1.2 목표 구현 방향
1. 자동화 대상은 기존 수동 createInternalAccount 로직을 재사용 가능한 도메인 서비스로 분리해 연계한다.
2. Google 연동은 신규 OAuth 엔드포인트 + 콜백 + 온보딩 단계(회사/국가/패스워드)로 설계한다.

## 4. 사이드 임팩트 분석 (Side Impact)

1. 인증/가입 경로 영향
- portal 가입 로직 변경으로 기존 이메일 인증/비밀번호 재설정 플로우와 충돌 가능.

2. 운영 부하 영향
- 자동 프로비저닝과 메일 발송이 register 트랜잭션에 결합되면 실패 전파/지연 가능.

3. 데이터 정합성 영향
- portal customer 생성 성공 후 AMA 계정 생성 실패 시 반쪽 상태 발생 가능.

4. 관리자 업무 영향
- 기존 수동 승인 업무는 감소하나, 실패 재처리 기능 중요도가 증가.

5. 보안 영향
- Google OAuth 도입 시 state 검증/redirect allowlist 미흡하면 계정 탈취 리스크.

## 5. DB 마이그레이션 계획 (DB Migration)

### 5.1 필요 여부
1. 전화번호 옵션화만 반영 시 DB 변경은 불필요(기존 nullable).
2. Google 연동 메타데이터 및 온보딩 상태를 DB에 저장하면 컬럼 추가가 필요할 수 있음.

### 5.2 예상 마이그레이션 항목
1. amb_svc_portal_customers
 - provider 구분 컬럼(예: pct_auth_provider)
 - provider 사용자 식별자(예: pct_provider_user_id)
 - 온보딩 상태(예: pct_onboarding_status)
2. 필요 시 unique index 조정
 - 이메일/프로바이더 조합 정책 확정 후 반영

### 5.3 실행 원칙
1. 스테이징/프로덕션은 수동 SQL 선반영 필수.
2. 코드 배포 전 DB 적용 -> 배포 -> 후속 검증 순서 준수.

## 6. 검증 계획 (Validation)

1. 기능 검증
- 일반 register -> 자동 AMA 계정 생성 -> 안내 메일 -> ama 최초 비번변경 성공
- register phone 미입력 성공
- Google 연동 -> 온보딩 완료 -> ama 로그인 성공

2. 오류 검증
- 중복 이메일
- 메일 발송 실패
- Google callback 실패(state mismatch, token 오류)

3. 회귀 검증
- 기존 portal 로그인/비밀번호 재설정
- 기존 수동 portal-bridge create-account 기능 유지
