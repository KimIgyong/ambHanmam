## 시스템 현황 분석

### 핵심 문제
CMS 컨트롤러 6개가 모두 `req.user.entityId` (JWT 토큰 고정값)를 사용.
- USER_LEVEL 사용자: JWT에 `entityId` 포함 → 정상 동작
- ADMIN_LEVEL 사용자: JWT에 `entityId` 없음 (undefined) → 조회 실패

### 기존 법인 전환 인프라 (이미 구축됨)
| 구성요소 | 상태 | 설명 |
|---------|------|------|
| `EntitySelector` 컴포넌트 | 완료 | 헤더에서 법인 선택 UI |
| `entity.store.ts` (Zustand) | 완료 | 선택 법인 상태 관리 + localStorage 저장 |
| `api-client.ts` 인터셉터 | 완료 | 모든 요청에 `X-Entity-Id` 헤더 자동 주입 |
| `resolveEntityId()` 유틸리티 | 완료 | query param → `X-Entity-Id` 헤더 → JWT 순으로 entityId 해석 |

**결론: 프론트엔드는 수정 불필요. 백엔드 컨트롤러에서 entityId 소스만 변경하면 됨.**

단, React Query 캐시 키에 entityId가 없어 법인 변경 시 이전 법인 데이터가 표시되는 문제가 있으므로 훅도 수정 필요.

## 구현 계획

### 1단계: 백엔드 - CMS 컨트롤러 entityId 소스 변경

**공통 변경 패턴:**
```typescript
// Before
const entId = req.user.entityId;

// After
const entId = resolveEntityId(req.headers['x-entity-id'], req.user);
```

`resolveEntityId`는 entity-settings 모듈에 이미 존재 (`apps/api/src/domain/entity-settings/util/resolve-entity-id.ts`). X-Entity-Id 헤더 → JWT entityId 순으로 fallback. UUID 유효성 검증 포함.

#### 1-1. CmsMenuController 수정
**파일**: `apps/api/src/domain/site-management/controller/cms-menu.controller.ts`

- import `resolveEntityId` 추가
- 5개 메서드에서 `req.user.entityId` → `resolveEntityId(req.headers['x-entity-id'], req.user)` 변경:
  - `getMenuTree()`

### 2단계: 프론트엔드 - React Query 캐시 키에 entityId 추가

법인 변경 시 이전 법인 데이터가 남아있지 않도록 queryKey에 entityId를 포함.

#### 2-1. useMenus.ts 수정
**파일**: `apps/web/src/domain/site-management/hooks/useMenus.ts`

```typescript
import { useEntityStore } from '@/domain/hr/store/entity.store';

const menuKeys = {
  all: (entityId?: string) => ['cms-menus', entityId] as const,
  tree: (entityId?: string) => [...menuKeys.all(entityId), 'tree'] as const,
};

export const useMenuTree = () => {
  const entityId = useEntityStore((s) => s.currentEntity?.entityId);
  return useQuery({
    queryKey: menuKeys.tree(entityId),
    queryFn: () => cmsMenuService.getMenuTree(),
    enabled: !!entityId,
    staleTime: 1000 * 60,
  });
};
```

- `useCreateMenu`, `useUpdateMenu`, `useReorderMenus`, `useDeleteMenu`에서도 invalidate 시 entityId 포함

#### 2-2. usePages.ts 수정
**파일**: `apps/web/src/domain/site-management/hooks/usePages.ts`

동일 패턴:
- `pageKeys`에 entityId 추가
- `usePageList`에 `enabled: !!entityId` 추가
- mutation의 invalidateQueries에 entityId 반영

#### 2-3. usePosts.ts 수정
**파일**: `apps/web/src/domain/site-management/hooks/usePosts.ts`

동일 패턴 적용.

#### 2-4. useSections.ts 수정
**파일**: `apps/web/src/domain/site-management/hooks/useSections.ts`

동일 패턴 적용.

## 사이드 임팩트 분석

| 항목 | 영향 | 대응 |
|------|------|------|
| CmsPublicController | 영향 없음 | 인증 불필요 API, `req.user` 미사용 |
| `getAllMenuTree()` | 영향 없음 | 공개 사이트용, entityId 파라미터 미사용 |
| USER_LEVEL 사용자 | 영향 없음 | `resolveEntityId`가 JWT entityId를 fallback으로 사용 |
| api-client 인터셉터 | 영향 없음 | 이미 `X-Entity-Id` 헤더 주입 중 |
| 기존 페이지 에디터 | 영향 없음 | 페이지 ID 기반 조회이므로 법인 전환과 무관 |
| cms.store.ts | 영향 없음 | 에디터 상태만 관리, 법인 컨텍스트 없음 |
