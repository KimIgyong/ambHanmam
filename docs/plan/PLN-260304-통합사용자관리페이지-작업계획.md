## 2. 구현 단계

### Phase 1: 백엔드 — Admin 모듈 생성

#### Step 1.1: Admin Controller 생성
**새 파일**: `apps/api/src/domain/admin/controller/admin.controller.ts`

```
@Controller('admin')
@UseGuards(AdminLevelGuard)
```

**엔드포인트 4개:**

1. **`GET /admin/users`** — 전체 사용자 목록
   - Query: `page`, `limit`, `search` (이름/이메일), `entity_id` (법인 필터), `status`, `role`
   - 응답: 페이징 된 사용자 목록 + 법인 정보 포함
   - JOIN: `amb_users` LEFT JOIN `amb_hr_entity_user_roles` + `amb_hr_entities`

2. **`GET /admin/entity-user-roles`** — 전체 법인-사용자 역할
   - Query: `page`, `limit`, `search`, `entity_id`
   - JOIN: `amb_hr_entity_user_roles` + `amb_hr_entities` + `amb_users`

3. **`GET /admin/unit-user-roles`** — 전체 부서-사용자 역할
   - Query: `page`, `limit`, `search`, `unit_id`
   - JOIN: `amb_user_unit_roles` + `amb_units` + `amb_users`

4. **`DELETE /admin/unit-user-roles/:uurId`** — 부서 역할 삭제

#### Step 1.2: Admin Service 생성
**새 파일**: `apps/api/src/domain/admin/service/admin.service.ts`

- `findAllUsers(params)` — QueryBuilder로 법인 필터 없는 전체 조회, 페이징
- `findAllEntityUserRoles(params)` — 3테이블 JOIN 조회
- `findAllUnitUserRoles(params)` — 3테이블 JOIN 조회
- `deleteUnitUserRole(uurId)` — 삭제

#### Step 1.3: Admin Module 생성
**새 파일**: `apps/api/src/domain/admin/admin.module.ts`


#### Step 1.4: 응답 DTO 정의

## 4. 사이드 임팩트 분석

### 4.1 영향 없음 (안전)
- 기존 `/settings/members` 페이지에 영향 없음 (별도 경로)
- 기존 `/settings/portal-bridge` 페이지에 영향 없음
- 기존 API는 수정하지 않고 새 엔드포인트만 추가

### 4.2 주의 사항
- **AdminLevelGuard**: 새 `/admin/*` 엔드포인트 전체에 적용하여 일반 사용자 접근 차단
- **EntityGuard 미적용**: Admin 엔드포인트는 법인 필터 없이 전체 조회 — 의도적 설계
- **데이터 규모**: 사용자 수가 적은 내부 시스템이므로 서버 사이드 페이징은 기본 50건으로 설정

## 6. 검증 방법

1. `npm run build` — 빌드 성공 확인
2. Settings 페이지에서 "Total User Management" 카드 표시 확인
3. `/settings/total-users` 접근 → 5개 탭 정상 렌더링
4. Tab 2: 전체 사용자 목록 조회, 법인별 필터, 검색, 상태 변경, 역할 변경
5. Tab 3: 법인-사용자 역할 목록, 삭제
6. Tab 4: 부서-사용자 역할 목록, 삭제
7. Tab 1/5: 기존 기능 정상 동작 확인
8. ADMIN_LEVEL이 아닌 사용자 접근 시 차단 확인
9. 스테이징 배포 후 실제 테스트
