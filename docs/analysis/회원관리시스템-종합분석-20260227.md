## 2. 회원 초대 및 가입 플로우

### 2.1 초대 생성

**백엔드**: `apps/api/src/domain/invitation/service/invitation.service.ts` → `create()`

플로우:
1. 이미 PENDING 초대 존재 → 오류
2. 이미 가입된 이메일 → 오류
3. UUID 토큰 생성, 유효기간 **7일**
4. 초대 레코드 저장 (이메일, 역할, 부서, 그룹, 소속법인, 자동승인 여부)
5. 이메일 발송 (실패해도 초대 레코드는 유지)

**초대 엔티티** (`amb_invitations`) 주요 필드:
| 필드 | 설명 |
|------|------|
| `inv_email` | 초대 대상 이메일 |
| `inv_token` | UUID 토큰 (고유) |
| `inv_role` | 할당 역할 |
| `inv_department` | 할당 부서 |
| `inv_group_id` | 할당 그룹 |
| `inv_group_code` | ADMIN_GROUP / USER_GROUP |
| `inv_company_id` | 소속 법인 |
| `inv_auto_approve` | 자동 승인 여부 |
| `inv_status` | PENDING / ACCEPTED / CANCELLED |
| `inv_expires_at` | 만료 시각 |
| `inv_send_count` | 이메일 발송 횟수 |

**프론트엔드**: `apps/web/src/domain/members/components/InvitationFormModal.tsx`
- 입력 필드: 이메일, 역할(MEMBER/MANAGER/VIEWER/ADMIN), 부서, 그룹(선택)
- 부서 목록은 `DEPARTMENTS` 상수에서 로드

### 2.2 초대 수락

**경로**: `/invite/:token` (Public route, 인증 불필요)

**프론트엔드**: `apps/web/src/domain/auth/pages/InviteAcceptPage.tsx`

플로우:
1. 토큰으로 초대 정보 조회 (`GET /invitations/validate/:token`)
2. 유효하지 않거나 만료 → 오류 표시
3. 사용자가 이름, 비밀번호 입력 후 제출
4. `POST /invitations/token/:token/accept` 호출
5. 비밀번호 규칙: 8자 이상, 영문+숫자 조합

**백엔드**: `invitation.service.ts` → `accept()`
1. 토큰 검증 (존재, PENDING, 미만료, 미가입)
2. 사용자 생성: 초대에 지정된 역할/부서/그룹코드/법인 적용
3. `autoApprove=true` → **ACTIVE** / `false` → **PENDING** (관리자 승인 필요)
4. 초대 상태 → `ACCEPTED`

### 2.3 일반 회원가입

**경로**: `/register`

**백엔드**: `apps/api/src/domain/auth/service/auth.service.ts` → `register()`

플로우:
1. 이메일 중복 체크
2. 초대 토큰이 있으면 → 초대 정보(역할/부서/그룹)로 세팅
3. 초대 토큰 없으면 → 기본값: `MEMBER / USER_GROUP / PENDING`
4. `autoApprove=true` → ACTIVE, `false` → PENDING
5. 사용자 저장 후 JWT 토큰 발급
6. **회사 메일 계정 자동 생성** (Postal API) — 실패해도 가입은 진행

### 2.4 로그인

**경로**: `/login`

**백엔드**: `auth.service.ts` → `login()`

상태별 처리:
- `WITHDRAWN` → 로그인 차단
- `SUSPENDED` → 로그인 차단
- `INACTIVE` → 로그인 차단
- `PENDING` → **로그인 허용** (프론트에서 `/pending` 리다이렉트)
- `ACTIVE` → 정상 로그인

JWT Payload 포함 정보:
```typescript
{ sub, email, group, role, status, companyId, isHq, mustChangePw, tokenVersion }
```

### 2.5 인증 가드 (프론트엔드)

**파일**: `apps/web/src/router/AuthGuard.tsx`

| 사용자 상태 | 리다이렉트 |
|-------------|-----------|
| 미인증 | `/login` |
| PENDING | `/pending` |
| INACTIVE/SUSPENDED | `/inactive` |
| mustChangePw=true | `/force-change-password` |
| ACTIVE | 정상 진입 |

## 11. 전체 데이터 흐름 요약

### 11.1 사용자 가입~활성화 플로우

```
[관리자] 초대 생성
    ↓
[시스템] 이메일 발송 (토큰 포함 / 7일 유효)
    ↓
[사용자] /invite/:token → 이름/비밀번호 입력
    ↓
[시스템] 사용자 생성 (INVITE 방식)
    ├─ autoApprove=true → ACTIVE (바로 사용)
    └─ autoApprove=false → PENDING (관리자 승인 대기)
                                ↓
                    [관리자] /settings/members → 승인/거부
                                ↓
                    [시스템] PENDING → ACTIVE / WITHDRAWN
```

### 11.2 권한 계층 구조

```
1. 시스템 역할 (usr_role)
   └─ SUPER_ADMIN > ADMIN > MANAGER > MEMBER > VIEWER

2. 그룹 코드 (usr_group_code)
   ├─ ADMIN_GROUP — 전체 메뉴 접근 (수정 불가)
   └─ USER_GROUP — 역할별 차등 접근

3. 법인 역할 (entity_user_role.eur_role)
   └─ SYSTEM_ADMIN / ADMIN / MEMBER — 법인별 데이터 접근 제어

4. 부서 역할 (user_dept_role.udr_role)
   └─ DEPARTMENT_HEAD > TEAM_LEAD > MEMBER — 부서 내 직급 구분

5. 그룹 (amb_groups)
   └─ 법인 소속 그룹 — 그룹별 메뉴 접근 권한 추가 설정 가능
```

### 11.3 메뉴 접근 판정 플로우

```
메뉴 enabled? ──[NO]──→ 숨김
      │
     [YES]
      │
ADMIN_GROUP? ──[YES]──→ 접근 허용
      │
     [NO]
      │
사용자별 설정 있음? ──[YES]──→ 설정값 적용
      │
     [NO]
      │
역할별 기본 설정 적용
      │
그룹별 필터 있음? ──[YES]──→ 소속 그룹 확인 → 접근/차단
      │
     [NO]
      │
접근 허용/차단
```
