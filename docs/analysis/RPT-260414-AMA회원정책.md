## 1. 회원 레벨 체계

AMA 시스템은 **4단계 회원 레벨**로 구분되며, 각 레벨은 조직 범위(scope)에 따라 데이터 접근 범위가 결정된다.

| 레벨 | 코드 | 소속 | 데이터 범위 | 설명 |
|------|------|------|------------|------|
| **HQ 관리자** | `ADMIN_LEVEL` | ROOT(본사) | `ALL` — 전체 법인 | 시스템 전체 관리, `usr_company_id = NULL` |
| **법인 사용자** | `USER_LEVEL` | SUBSIDIARY(자회사) | `OWN_ORG` — 소속 법인 | `ent_id` 기반 격리, 자기 법인만 CRUD |
| **파트너** | `PARTNER_LEVEL` | 파트너 조직 | `PARTNER` — 파트너 범위 | `partner_id` 기반 격리 |
| **고객사** | `CLIENT_LEVEL` | 고객사 | `CLIENT` — 고객 범위 | `cli_id` 기반 격리, 배정 프로젝트만 접근 |

> **소스**: `packages/types/src/user-level.types.ts`, `apps/api/src/domain/auth/entity/user.entity.ts`

## 2. 레벨별 역할(Role) 정의

각 레벨 내에서 허용되는 역할(Role)은 엄격히 정의되어 있으며, 레벨-역할 조합 외의 할당은 불가하다.

### ADMIN_LEVEL (HQ 관리자)

| 역할 | 권한 |
|------|------|
| `SUPER_ADMIN` | 시스템 전체 제어 (설정, 플랜, 모든 법인 관리) |
| `ADMIN` | 전체 운영 관리 (일부 시스템 설정 제외) |

### USER_LEVEL (법인 사용자)

| 역할 | 권한 |
|------|------|
| `MASTER` | 법인 관리자 — 회원 관리, 법인 설정, 부서 관리 |
| `MANAGER` | 부서 관리자 — 부서 내 업무 관리 |
| `MEMBER` | 일반 사원 — 기본 업무 수행 |
| `VIEWER` | 읽기 전용 — 데이터 조회만 가능 |

### PARTNER_LEVEL (파트너)

| 역할 | 권한 |
|------|------|
| `PARTNER_ADMIN` | 파트너 앱 및 파트너 직원 관리 |
| `PARTNER_MEMBER` | 앱 상태 조회만 가능 |

### CLIENT_LEVEL (고객사)

| 역할 | 권한 |
|------|------|
| `CLIENT_ADMIN` | 고객사 직원 초대, 프로젝트/이슈 관리 |
| `CLIENT_MEMBER` | 배정된 프로젝트 조회, 이슈 제출 |

### 부서 내 역할 (UserUnitRole)

법인 사용자는 부서(`amb_units`) 소속 시 추가 역할이 부여된다:

| 부서 역할 | 설명 |
|-----------|------|
| `UNIT_HEAD` | 유닛장 (Level 1 조직) |
| `TEAM_LEAD` | 팀장 (Level 2 조직) |
| `MEMBER` | 부서 일반 구성원 |

> PRIMARY 지정된 부서 역할이 JWT에 `departmentId`, `departmentRole`로 포함된다.

## 3. 회원 상태 라이프사이클

```
가입/초대 ──► PENDING ──┬──► ACTIVE ──┬──► INACTIVE (비활성화)
                        │             │
                        │             └──► SUSPENDED (정지)
                        │
                        └──► WITHDRAWN (거절/탈퇴)
```

| 상태 | 설명 | API 접근 | 에러 코드 |
|------|------|---------|----------|
| `PENDING` | 가입 승인 대기 | 차단 | `E1010` |
| `ACTIVE` | 정상 활동 | 허용 | - |
| `INACTIVE` | 관리자에 의한 비활성화 | 차단 | `E1004` |
| `SUSPENDED` | 계정 정지 (구독 정지 등) | 차단 | `E1003` |
| `WITHDRAWN` | 거절 또는 자진 탈퇴 | 차단 | `E1002` |

### 상태 전이 규칙

- **PENDING → ACTIVE**: MASTER/ADMIN이 승인 (`approve()`) — `usr_approved_by`, `usr_approved_at` 기록
- **PENDING → WITHDRAWN**: MASTER/ADMIN이 거절 (`reject()`)
- **ACTIVE → INACTIVE**: 관리자 비활성화 처리
- **ACTIVE → SUSPENDED**: 구독 정지 또는 관리자 정지 처리
- **초대 + `auto_approve=true`**: PENDING 없이 바로 ACTIVE

> **소스**: `apps/api/src/domain/auth/guard/level-role.guard.ts`, `apps/api/src/domain/members/service/member.service.ts`

## 4. 가입 경로별 플로우

AMA 시스템은 6가지 가입 경로(`usr_join_method`)를 지원한다.

### 4.1 SEED (시스템 시드)

- 초기 시스템 부트스트래핑용 슈퍼관리자 생성
- 코드 레벨에서만 실행

### 4.2 REGISTER (자가 가입)

```
사용자 → 회원가입 폼 제출 → UserEntity 생성
  ├─ 기본 역할: MEMBER
  ├─ 기본 레벨: USER_LEVEL
  └─ 상태: PENDING (MASTER 승인 대기)
```

### 4.3 INVITE (초대 가입)

```
관리자/MASTER → 초대 생성 (InvitationService.create)
  ├─ inv_email, inv_role, inv_level_code, inv_company_id 지정
  ├─ inv_auto_approve 옵션 설정
  └─ 유효기간: 7일

초대 이메일 발송 (UUID 토큰 포함)

수신자 → 초대 링크 클릭 → 회원가입 폼
  ├─ 역할/레벨/법인: 초대 정보에서 자동 상속
  ├─ auto_approve=true → 즉시 ACTIVE
  └─ auto_approve=false → PENDING (승인 대기)

가입 완료 → EntityUserRoleEntity 생성 → 기본 "Holding" 유닛 배정
```

**초대 상태 관리**: `PENDING → ACCEPTED / EXPIRED / CANCELLED`

### 4.4 PORTAL (포털 가입)

```
고객 → 포털 회원가입 → 이메일 OTP 인증 (6자리, 10분 TTL)
  ├─ verify_token 발급 (JWT, 10분)
  └─ 비밀번호: 자동 생성 (16자 UUID 기반)

포털 계정 생성 (PortalCustomerEntity)
  └─ 동시에 SvcClientEntity 생성 (PROSPECT 상태)

Portal Bridge 자동 프로비저닝:
  ├─ 내부 UserEntity 생성 (USER_LEVEL, ACTIVE)
  ├─ usr_must_change_pw = true
  └─ eurIsOwner = true (자동 생성 법인의 소유자)

Auto-login Token 발급 (5분 TTL) → 메인 앱으로 리디렉트
  └─ initialSetup: 이름 + 비밀번호 변경 (must_change_pw 해제)
```

### 4.5 PARTNER_INVITE (파트너 초대)

- 초대 시 `inv_level_code = PARTNER_LEVEL` 지정
- 파트너 조직(`inv_partner_id`) 연결
- 나머지 플로우는 INVITE와 동일

### 4.6 CLIENT_INVITE (고객사 초대)

- 초대 시 `inv_level_code = CLIENT_LEVEL` 지정
- 고객사(`inv_cli_id`) 연결
- 나머지 플로우는 INVITE와 동일

> **소스**: `apps/api/src/domain/invitation/service/invitation.service.ts`, `apps/api/src/domain/auth/service/auth.service.ts`

## 5. 인증 및 보안 정책

### 5.1 JWT 토큰 구조

| 항목 | 값 |
|------|-----|
| Access Token 유효기간 | 4시간 |
| Refresh Token 유효기간 | 7일 |
| 전달 방식 | httpOnly Cookie (`access_token`, `refresh_token`) + Bearer Header |
| Cookie maxAge | 15분 (쿠키 갱신 주기, JWT 자체는 4시간) |

**JWT Payload 구성**:
```
sub, email, level, role, status, companyId, isHq,
mustChangePw, entityId?, cliId?, partnerId?,
tokenVersion, timezone, locale
```

### 5.2 토큰 폐기 (Revocation)

- `usr_token_version` 필드 활용
- 서버 측에서 `tokenVersion` 증가 시 기존 모든 토큰 즉시 무효화
- Refresh Token 갱신 시에도 `tokenVersion` 증가 (토큰 로테이션)

### 5.3 로그인 플로우

| 시나리오 | 동작 |
|---------|------|
| `entity_code` 포함 로그인 | 해당 법인에서 (email, entId) 매칭 → JWT 발급 |
| `entity_code` 없음, ADMIN 계정 | ADMIN_LEVEL 계정 → 즉시 JWT 발급 |
| `entity_code` 없음, 단일 USER 매칭 | 해당 계정 → 즉시 JWT 발급 |
| `entity_code` 없음, 복수 USER 매칭 | 에러 반환 → entity_code 선택 필요 |

### 5.4 계정 잠금 정책

| 항목 | 설정값 |
|------|--------|
| 잠금 임계값 | **15회** 연속 로그인 실패 |
| 잠금 기간 | **10분** (`usr_locked_until`) |
| 해제 조건 | 잠금 기간 경과 후 다음 성공 로그인 시 자동 해제 |

### 5.5 비밀번호 변경 강제

- `usr_must_change_pw = true` 설정 시 모든 API 요청 차단 (에러 `E1011`)
- 비밀번호 변경 API만 허용
- 포털 가입 사용자는 초기 설정(initialSetup) 완료 시 해제

### 5.6 Cookie 보안 설정

| 환경 | secure | sameSite | httpOnly |
|------|--------|----------|----------|
| Production | `true` | `lax` | `true` |
| Development | `false` | `lax` | `true` |

### 5.7 포털 OAuth 2.0

- **Google OAuth** 지원 (`pct_auth_provider = GOOGLE`)
- OAuth state JWT (`portal_google_oauth_state`) 발급 후 Google 인증 → 코드 교환 → 사용자 생성/조회

> **소스**: `apps/api/src/domain/auth/service/auth.service.ts`, `apps/api/src/domain/auth/strategy/jwt.strategy.ts`

## 6. 데이터 격리 및 접근 제어

### 6.1 Guard 체계

```
요청 → JwtAuthGuard → LevelRoleGuard → OwnEntityGuard → 컨트롤러
                                              ↓
                                   DataScopeInterceptor
                                              ↓
                                     서비스 (DataScope 적용)
```

| Guard / Interceptor | 역할 |
|---------------------|------|
| `JwtAuthGuard` | JWT 검증 (`@Public()` 데코레이터 시 스킵) |
| `LevelRoleGuard` | 상태 검증 + 레벨/역할 접근 제어 |
| `OwnEntityGuard` | 법인 범위 접근 제어 (ADMIN 바이패스, USER는 MASTER만 통과) |
| `DataScopeInterceptor` | 레벨별 자동 데이터 스코프 설정 |
| `AccessControlGuard` | 워크 아이템 단위 세밀한 권한 검사 |

### 6.2 데코레이터 단축 표현

| 데코레이터 | 동작 |
|-----------|------|
| `@Auth()` | JWT + 상태/비밀번호 검증 |
| `@AdminOnly()` | ADMIN_LEVEL만 허용 |
| `@MasterOrAdmin()` | MASTER, ADMIN, SUPER_ADMIN만 허용 |
| `@PartnerOnly()` | PARTNER_LEVEL만 허용 |
| `@PartnerAdminOnly()` | PARTNER_LEVEL + PARTNER_ADMIN만 허용 |
| `@AuthAllowPending()` | JWT만 검증 (PENDING 상태 허용) |

### 6.3 DataScope 자동 매핑

| 레벨 | scope | 필터 키 |
|------|-------|---------|
| ADMIN_LEVEL | `ALL` | 없음 (전체 접근) |
| USER_LEVEL | `OWN_ORG` | `companyId` |
| PARTNER_LEVEL | `PARTNER` | `partnerId` |
| CLIENT_LEVEL | `CLIENT` | `cliId` |

### 6.4 엔티티 ID 결정 로직 (`resolveEntityId`)

```
Query Parameter entity_id 있음? → 해당 값 사용
                 없음? → JWT의 entityId fallback
```

> **소스**: `apps/api/src/domain/auth/guard/`, `apps/api/src/domain/auth/interceptor/data-scope.interceptor.ts`

## 7. 구독 플랜과 회원 제한

### 7.1 플랜 체계

| 플랜 | 기본 사용자 수 | 특징 |
|------|--------------|------|
| `FREE` | `pln_free_user_count` (기본 5명) | 무료 기본 기능 |
| `BASIC` | 설정 가능 `pln_max_users` | 유료 기본 기능 + 확장 |
| `PREMIUM` | 설정 가능 `pln_max_users` | 전체 기능 |

### 7.2 구독 상태

| 상태 | 설명 |
|------|------|
| `TRIALING` | 체험 기간 (`sbn_trial_end_at`까지) |
| `ACTIVE` | 정상 구독 |
| `PAST_DUE` | 결제 지연 |
| `SUSPENDED` | 구독 정지 |
| `CANCELLED` | 구독 해지 |

### 7.3 사용자 수 제한

- 법인(Entity)당 1개 구독 (`SubSubscriptionEntity`)
- `sbn_user_count`가 `pln_max_users` 초과 시 사용자 추가 차단
- 좌석 단위: `pln_user_slot_size` 단위로 증감
- 최소 좌석: `pln_min_users`

### 7.4 AI 토큰 정책

| 지급 유형 | 설명 |
|----------|------|
| `ONETIME_GRANT` | 가입 시 1회 지급 (`pln_token_onetime`) |
| `MONTHLY_GRANT` | 매월 사용자당 지급 (`pln_token_per_user_monthly`) |
| `ADDON_PURCHASE` | 추가 구매 |
| `REFERRAL_REWARD` | 추천 보상 (`pln_referral_reward_tokens`) |

- 월간 리셋 여부: `pln_is_token_monthly_reset`
- 토큰 월렛 유형: `BASE`, `ADDON`, `REFERRAL` (법인별 분리 관리)

### 7.5 스토리지 정책

| 항목 | 설명 |
|------|------|
| 기본 용량 | `pln_storage_base_gb` |
| 최대 용량 | `pln_storage_max_gb` |
| 경고 임계 | `pln_storage_warn_pct` (%) |
| 차단 임계 | `pln_storage_block_pct` (%) |
| 업로드 차단 | `sqt_is_upload_blocked` (차단 임계 초과 시) |

### 7.6 단계별 과금 (Tiered Pricing)

사용자 수 구간별 차등 단가 적용:
- `plt_users_min` ~ `plt_users_max` 구간 정의
- 구간별 월간/연간 단가, 월간 토큰 배정량 차등

> **소스**: `apps/api/src/domain/subscription/entity/`

## 8. 포털 회원 관리

포털(`portal-api`)은 메인 앱과 **별도의 사용자 테이블**(`amb_svc_portal_customers`)을 사용한다.

### 8.1 포털 인증 방식

| 방식 | 설명 |
|------|------|
| Email + Password | 이메일 OTP 인증 후 가입, 임시 비밀번호 자동 생성 |
| Google OAuth 2.0 | Google 계정 연동 로그인/가입 |

### 8.2 포털 가입 플로우

1. 이메일 입력 → 6자리 OTP 발송 (인메모리 저장, 10분 TTL)
2. OTP 검증 → `verify_token` 발급 (JWT, 10분)
3. 가입 정보 제출 (`verify_token` 포함) → 포털 계정 생성
4. 필수 동의: `terms_agreed`, `privacy_agreed` (마케팅은 선택)
5. 포털 계정 상태: 즉시 `ACTIVE` (승인 절차 없음)

### 8.3 포털 → 메인앱 브릿지

```
PortalCustomerEntity (포털)
     │
     ├──► SvcClientEntity (PROSPECT 상태)
     │
     └──► Portal Bridge 자동 프로비저닝
              │
              ├──► UserEntity (USER_LEVEL, ACTIVE, must_change_pw=true)
              ├──► HrEntityEntity (자동 생성 법인)
              ├──► EntityUserRoleEntity (eurIsOwner=true)
              └──► PortalUserMappingEntity (매핑 기록)
```

### 8.4 포털-내부 사용자 매핑

`amb_portal_user_mappings` 테이블로 포털 고객과 내부 사용자 간 매핑:
- 상태: `ACTIVE` / `REVOKED`
- 감사 추적 포함

> **소스**: `apps/portal-api/src/domain/auth/`, `apps/api/src/domain/portal-bridge/`

## 9. 워크 아이템 ACL

업무 항목(이슈, 태스크, 프로젝트 등) 단위의 세밀한 접근 제어:

### 9.1 가시성 (Visibility)

| 레벨 | 범위 |
|------|------|
| `PRIVATE` | 작성자만 |
| `SHARED` | 명시적으로 공유된 사용자 |
| `UNIT` | 소속 유닛 전체 |
| `CELL` | 소속 셀 전체 |
| `ENTITY` | 법인 전체 |
| `PUBLIC` | 전사 공개 |

### 9.2 공유 권한 (Permission Hierarchy)

```
VIEW (1) < COMMENT (2) < EDIT (3) < ADMIN (4)
```

| 권한 | 설명 |
|------|------|
| `VIEW` | 조회만 가능 |
| `COMMENT` | 조회 + 댓글 작성 |
| `EDIT` | 조회 + 댓글 + 수정 |
| `ADMIN` | 전체 권한 (삭제, 공유 관리 포함) |

### 9.3 공유 대상 유형

| 타입 | 설명 |
|------|------|
| `USER` | 개별 사용자 |
| `UNIT` | 유닛 전체 |
| `TEAM` | 팀 전체 |
| `CELL` | 셀 전체 |
| `ENTITY` | 법인 전체 |

- 공유별 만료일(`wis_expires_at`) 설정 가능
- `AccessControlGuard`에서 요청 시점 권한 검증

> **소스**: `apps/api/src/domain/acl/`

## 주요 보안 체크포인트

| 항목 | 정책 |
|------|------|
| 동일 이메일 복수 법인 가입 | 허용 (Unique: `email + company_id` WHERE `deleted_at IS NULL`) |
| 토큰 즉시 폐기 | `tokenVersion` 증가로 기존 토큰 무효화 |
| 브루트포스 방어 | 15회 실패 → 10분 잠금 |
| 비밀번호 변경 강제 | `must_change_pw=true` 시 모든 API 차단 |
| 법인 간 데이터 격리 | `ent_id` 기반 WHERE 필터 필수 |
| Soft Delete | `usr_deleted_at` — 물리 삭제 없음 |
| 쿠키 보안 | httpOnly + secure(Prod) + sameSite:lax |
