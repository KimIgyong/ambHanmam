## 1. PostgreSQL `date` 타입 → toISOString() 오류

### 증상
```
TypeError: entity.xxxDate.toISOString is not a function
```
`GET /api/v1/{resource}` 또는 `GET /api/v1/{resource}/:id` 요청 시 500 에러.

### 원인
- TypeORM의 PostgreSQL `date` 컬럼 (`type: 'date'`)은 **Date 객체가 아닌 문자열** (`"2025-01-15"`)을 반환하는 경우가 있음
- `timestamp` / `timestamptz` 타입은 항상 Date 객체를 반환하므로 안전
- Mapper에서 `entity.field.toISOString()` 호출 시 문자열에는 해당 메서드가 없어 런타임 에러 발생

### 영향 범위
`type: 'date'`로 정의된 모든 엔티티 컬럼 (약 45개):

| 도메인 | 엔티티 | 필드 |
|--------|--------|------|
| accounting | BankAccountEntity | `bacOpeningDate` |
| accounting | TransactionEntity | `txnDate` |
| asset | AssetEntity | `astPurchaseDate` ✅ |
| billing | ContractEntity | `ctrStartDate`, `ctrEndDate` |
| billing | ContractMilestoneEntity | `mstDueDate` |
| billing | InvoiceEntity | `invDate`, `invDueDate`, `invServicePeriodStart/End`, `invPaidDate` |
| billing | PaymentEntity | `payDate` |
| billing | SowEntity | `sowPeriodStart`, `sowPeriodEnd` |
| hr | EmployeeEntity | `empStartDate`, `empEndDate` |
| hr | HolidayEntity | `holDate` |
| hr | OtRecordEntity | `otrDate` |
| hr | PayrollPeriodEntity | `pypPaymentDate` |
| hr | SalaryHistoryEntity | `slhEffectiveDate` |
| hr | SystemParamEntity | `hspEffectiveFrom`, `hspEffectiveTo` |
| hr | TimesheetEntity | `tmsWorkDate` |
| issues | IssueEntity | `issStartDate`, `issDueDate` |
| meeting-notes | MeetingNoteEntity | `mtnMeetingDate` ✅ |
| project | ProjectEntity | `pjtStartDate`, `pjtEndDate` |
| schedule | ScheduleExceptionEntity | `sceOriginalDate` |
| schedule | ScheduleRecurrenceEntity | `scrEndDate` |
| service-mgmt | ServiceEntity | `svcLaunchDate` |
| service-mgmt | SubscriptionEntity | `subStartDate`, `subEndDate`, `subTrialEndDate` |
| todo | TodoEntity | `tdoDueDate` ✅ |
| work-schedule | WorkScheduleEntity | `wksDate` ✅ |

✅ = 수정 완료

### 해결 패턴 (안전한 날짜 변환)

```typescript
// ❌ 위험 - date 타입은 문자열일 수 있음
purchaseDate: entity.astPurchaseDate?.toISOString().split('T')[0]

// ✅ 안전 - instanceof 체크 후 변환
purchaseDate: entity.astPurchaseDate
  ? (entity.astPurchaseDate instanceof Date
      ? entity.astPurchaseDate.toISOString().split('T')[0]
      : String(entity.astPurchaseDate))
  : null
```

### 예방 규칙
> **새 Mapper 작성 시**: `type: 'date'` 컬럼은 반드시 `instanceof Date` 체크 후 `.toISOString()` 호출. 또는 `String(value)`로 통일.

## 2. 삭제 후 캐시된 Detail 쿼리 404 오류

### 증상
```
GET /api/v1/{resource}/{id} 404 (Not Found)
```
리소스 삭제 직후 콘솔에 404 에러가 반복 출력됨.

### 원인
- TanStack Query의 `invalidateQueries({ queryKey: keys.all })`이 **모든 관련 쿼리**(list + detail + exceptions)를 무효화
- Detail 쿼리가 무효화되면 **이미 삭제된 리소스를 다시 GET** 시도
- `setSelectedId(null)` 상태 업데이트와 쿼리 invalidation 간 **타이밍 이슈**

### 발생 사례
- `/schedule` — 일정 삭제 시 → `GET /schedules/:id` 404 ✅ 수정
- 잠재적: `/todos`, `/issues`, `/meeting-notes` 등 동일 패턴 사용 모듈

### 해결 패턴

```typescript
// ❌ 위험 - detail 쿼리도 invalidate → 삭제된 리소스 refetch
onSuccess: () => {
  queryClient.invalidateQueries({ queryKey: keys.all });
}

// ✅ 안전 - detail 쿼리는 제거, list만 갱신
onSuccess: () => {
  queryClient.removeQueries({
    queryKey: [...keys.all, 'detail'],
    exact: false,
  });
  queryClient.invalidateQueries({
    queryKey: [...keys.all, 'list'],
  });
}
```

### 예방 규칙
> **삭제 Mutation 작성 시**: `onSuccess`에서 detail/exceptions 쿼리는 `removeQueries`로 캐시 제거, list 쿼리만 `invalidateQueries`로 갱신.

## 3. 권한 부족 403 (모달 컴포넌트 즉시 로딩)

### 증상
```
GET /api/v1/{resource} 403 (Forbidden)
```
특정 페이지 진입 시, 사용하지 않는 모달 내의 데이터 페칭이 즉시 실행됨.

### 원인
- React 컴포넌트가 **항상 렌더링**되면 내부 훅(`useQuery`)이 **즉시 API 호출**
- 모달 내부에서 `useGroupList()` 등 **권한이 필요한 API를 호출**하는 경우, 모달이 닫혀있어도 호출됨
- `MANAGER` / `ADMIN` 전용 엔드포인트를 일반 사용자가 호출하면 403 발생

### 발생 사례
- `/meeting-notes` — `MeetingNoteFormModal` 내 `useGroupList()` → `GET /groups` 403 ✅ 수정

### 해결 패턴

```tsx
// ❌ 위험 - 항상 렌더링 → 훅 즉시 실행
<MeetingNoteFormModal isOpen={showModal} onClose={...} onSubmit={...} />

// ✅ 안전 - 조건부 렌더링 → 필요할 때만 훅 실행
{showModal && (
  <MeetingNoteFormModal isOpen={showModal} onClose={...} onSubmit={...} />
)}
```

### 예방 규칙
> **모달/패널 컴포넌트**: 내부에서 API를 호출하는 모달은 반드시 `{isOpen && <Modal />}` 패턴으로 조건부 렌더링. 또는 훅에 `enabled: isOpen` 옵션 추가.

## 4. 존재하지 않는 테이블 참조 500

### 증상
```
QueryFailedError: relation "amb_xxx" does not exist
```
새 엔티티 추가 후 마이그레이션/동기화 없이 배포.

### 원인
- TypeORM `synchronize: true`를 사용하더라도, **엔티티를 app.module.ts에 등록하지 않으면** 테이블이 자동 생성되지 않음
- 엔티티를 모듈의 `TypeOrmModule.forFeature()`에만 등록하고 **app.module.ts 전역 entities 배열에 누락**하는 경우 발생

### 발생 사례
- `amb_site_settings` 테이블 미생성 (2026-02-26 확인)

### 해결 패턴

```typescript
// ✅ 반드시 두 곳에 등록
// 1. 해당 모듈
@Module({
  imports: [TypeOrmModule.forFeature([NewEntity])],
})

// 2. app.module.ts 전역 entities 배열
entities: [...existingEntities, NewEntity],
```

### 예방 규칙
> **새 엔티티 추가 시**: ① 모듈 `TypeOrmModule.forFeature()` + ② `app.module.ts` entities 배열 **양쪽 모두** 등록 확인.

## 5. Nullable Entity 필드의 안전하지 않은 접근

### 증상
```
TypeError: Cannot read properties of null/undefined (reading 'xxx')
```

### 원인
- Entity 필드가 `nullable: true`이지만, Mapper/Service에서 null 체크 없이 프로퍼티 접근
- 관계(Relation) 미로드 상태에서 `entity.relation.field` 접근

### 자주 발생하는 패턴

```typescript
// ❌ relation이 로드되지 않았을 때 에러
entity.user.usrName

// ✅ optional chaining 사용
entity.user?.usrName || ''

// ❌ nullable 필드에 메서드 호출
entity.nullableDate.toISOString()

// ✅ null 체크 후 호출
entity.nullableDate?.toISOString() || null
```

### 예방 규칙
> **Mapper 작성 시**: 모든 관계 필드는 `?.` (optional chaining), nullable 컬럼은 조건부 접근. TypeORM `relations` 옵션에서 실제로 로드하는 관계만 non-null로 간주.
