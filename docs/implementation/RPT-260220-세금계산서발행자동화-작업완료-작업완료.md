## 1. 작업 개요

한국 법인(KR)의 매출 세금계산서를 팝빌(Popbill) ASP를 통해 국세청(NTS)에 자동 발행하는 기능을 구현하였다. 기존 베트남 E-Invoice(TVAN) 구현 패턴(`EinvoiceService` + `TVAN_API_SERVICE` 팩토리)을 그대로 따르며, 팝빌 계정 미보유 상태이므로 Mock 어댑터를 우선 구현하였다.

| 번호 | 기능 | 설명 |
|------|------|------|
| 1 | 세금계산서 정발행 | APPROVED_ADMIN 상태의 KRW 청구서에 대해 팝빌 API를 통한 정발행 |
| 2 | 수정세금계산서 발행 | 6가지 수정사유코드 지원, 원본 청구서 승인번호 연계 |
| 3 | 역발행 요청 | 공급받는자가 발행을 요청하는 역발행 플로우 |
| 4 | 발행 취소 | ISSUED 상태의 세금계산서 취소 |
| 5 | 상태 조회/동기화 | 팝빌 API를 통한 실시간 상태 동기화 (stateCode 매핑) |

### 3.2 DB 스키마 확장

`amb_bil_invoices` 테이블에 9개 컬럼을 추가하였다. TypeORM `synchronize: true` 설정에 의해 자동 마이그레이션된다.

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| `inv_nts_mgt_key` | varchar(24) | 팝빌 관리번호 (`AMB-{invNumber}`) |
| `inv_nts_confirm_num` | varchar(30) | 국세청 승인번호 |
| `inv_nts_status` | varchar(20) | 상태 (NONE/PENDING/ISSUED/ACCEPTED/REJECTED/FAILED/CANCELLED) |
| `inv_nts_issued_at` | timestamptz | 발행 일시 |
| `inv_nts_sent_at` | timestamptz | 국세청 전송 일시 |
| `inv_nts_result_code` | varchar(10) | 국세청 처리결과 코드 |
| `inv_nts_error` | text | 오류 메시지 |
| `inv_nts_modify_code` | varchar(2) | 수정사유코드 (1~6) |
| `inv_nts_original_key` | varchar(24) | 원본 관리번호 (수정세금계산서용) |

### 3.6 프론트엔드 UI

`NtsSection` 컴포넌트는 KR 법인 + KRW 통화 조건에서만 표시되며, 다음 내용을 포함한다:

- **상태 뱃지**: 7종 상태별 색상 구분 (NONE=회색, PENDING=파랑, ISSUED=초록, ACCEPTED=에메랄드, REJECTED=빨강, FAILED=빨강, CANCELLED=회색)
- **정보 표시**: 관리번호, 국세청 승인번호, 발행일시, 수정사유 (해당 시)
- **액션 버튼**: 세금계산서 발행, 수정세금계산서 발행, 발행 취소, 상태 새로고침
- **수정세금계산서 모달**: 수정사유코드 드롭다운(6종) + 원본 청구서 ID 입력

## 7. 비고

- 모든 신규 파일은 프로젝트 파일 네이밍 컨벤션을 준수한다 (백엔드 kebab-case, 프론트엔드 PascalCase).
- API 엔드포인트는 `/api/v1` base path 규칙을 준수한다.
- 다국어 키는 하드코딩 없이 `useTranslation()` 훅의 `t()` 함수를 통해 사용하며, i18n 규칙을 준수한다.
- `PopbillModule`은 `@Global()` 데코레이터로 등록되어 전체 애플리케이션에서 `POPBILL_SERVICE` 주입이 가능하다.
- 기존 E-Invoice(VN) 기능에 대한 사이드 임팩트는 없다. NTS 필드와 엔드포인트는 완전히 별도로 동작한다.
- `NtsSection` 컴포넌트는 `currentEntity.country === 'KR' && invoice.currency === 'KRW'` 조건에서만 렌더링되므로 VN 법인 UI에 영향 없다.
