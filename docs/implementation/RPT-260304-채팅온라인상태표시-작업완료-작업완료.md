## 1. 구현 내용 요약

### 1-1. 백엔드 — Presence 서비스

**PresenceService** (`presence.service.ts`)
- 인메모리 `Map<string, { lastHeartbeat: Date, status: 'online' | 'offline' }>` 기반 상태 관리
- `heartbeat(userId)` — 하트비트 수신 시 online 상태 업데이트, 기존 offline이었으면 SSE 이벤트 발행
- `getUserStatus(userId)` / `getStatus(userIds)` — 단일/일괄 상태 조회
- `@Cron('*/30 * * * * *')` — 30초마다 유휴 사용자 체크, 60초 이상 하트비트 없으면 offline 전환 + SSE 이벤트 발행

**PresenceController** (`presence.controller.ts`)
- `POST /talk/presence/heartbeat` — 하트비트 수신 (204 No Content)
- `GET /talk/presence/status?user_ids=...` — 사용자 상태 일괄 조회
- `SSE /talk/presence/events` — 전역 Presence SSE 스트림

**TalkSseService 확장** (`talk-sse.service.ts`)
- 기존 채널별 Subject에 더해 전역 `presence$` Subject 추가
- `emitPresence(userId, status)` — Presence 이벤트 발행
- `subscribePresence()` — Presence SSE 스트림 구독 (Observable)

### 1-2. 프론트엔드 — Presence 훅/서비스/스토어

**talk.service.ts 확장**
- `sendHeartbeat()` → POST `/talk/presence/heartbeat`
- `getPresenceStatus(userIds)` → GET `/talk/presence/status?user_ids=...`

**usePresence 훅** (`usePresence.ts`)
- `usePresenceHeartbeat()` — 30초마다 하트비트 전송
  - `visibilitychange` 감지: 탭 비활성 시 하트비트 중단, 활성화 시 재시작
  - `mousemove`, `keydown` 감지: 5분 유휴 시 하트비트 중단, 활동 재개 시 즉시 재시작
- `usePresenceStatus(userIds)` — React Query 기반 상태 조회 (30초 staleTime + refetchInterval)
- `usePresenceSSE()` — 전역 Presence SSE 연결, `presence:update` 이벤트 수신 → Zustand store 업데이트
  - 지수 백오프 재연결 (최대 5회)

**talk.store.ts 확장**
- `presenceMap: Record<string, 'online' | 'offline'>` — 사용자별 상태 맵
- `setPresence(userId, status)` — 개별 상태 업데이트
- `setPresenceMap(map)` — 일괄 상태 업데이트

## 3. 아키텍처 설계 결정

| 항목 | 결정 | 이유 |
|------|------|------|
| 통신 방식 | SSE + Heartbeat REST | 기존 SSE 아키텍처 활용, WebSocket 도입 불필요 |
| 저장소 | 인메모리 Map | DB 불필요 (내부 시스템, 동시 접속 50명 이하), 서버 재시작 시 30초 내 복구 |
| 하트비트 주기 | 30초 | 네트워크 부하 최소화 + 적절한 상태 갱신 주기 |
| 유휴 타임아웃 | 서버 60초 / 클라이언트 5분 | 클라이언트: 5분 유휴 → 하트비트 중단, 서버: 60초 미수신 → offline 전환 |
| SSE 스트림 | 전역 별도 스트림 | 기존 채널별 SSE에 영향 없음, 독립적 Presence 이벤트 관리 |
| ChannelList DM 점 | 미구현 | TalkChannelResponse에 member ID 없음, 향후 백엔드 확장 시 추가 가능 |

## 4. 사이드 임팩트

### 4.1 안전한 변경
- 기존 채널별 SSE 스트림에 영향 없음 (별도 Presence SSE)
- 기존 채팅 기능(메시지 CRUD, 번역, 리액션 등) 변경 없음
- 인메모리 Map이므로 DB 마이그레이션 불필요
- `ScheduleModule`은 `app.module.ts`에 이미 등록됨

### 4.2 주의사항
- **서버 재시작 시**: 모든 사용자 offline 상태 → 30초 후 하트비트로 자동 복구
- **SSE 연결 수**: 기존 채널별 SSE + 전역 Presence SSE = 동시 2개 연결
- **네트워크 부하**: 하트비트 30초 주기 (경량 POST, Content 없는 204 응답) → 미미
