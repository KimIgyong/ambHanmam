# 커스텀앱 & 앱스토어 현황 분석 리포트

> **작성일**: 2026-03-30  
> **작성 목적**: 커스텀앱(사용자 등록 외부서비스)과 앱스토어(아메바/파트너 공식앱)의 현재 구현 상태 분석 및 분리 운영 방향 수립

## 1. 현재 아키텍처 개요

### 1.1 페이지 구조

```
┌─ 사이드바 ─────────────────────────────────────────────────────────┐
│                                                                     │
│  [커스텀앱 영역] ← GET /my (사이드바용 활성앱 목록)                    │
│    ├── app-car-manager                                              │
│    └── my-custom-tool                                               │
│                                                                     │
│  [메뉴]                                                             │
│    ├── /menu/custom-apps         → MenuCategoryPage (카드뷰)        │
│    ├── /app-store                → AppStorePage (iframe: 공식)      │
│    └── /app-store-stg            → AppStorePage (iframe: 스테이징)   │
│                                                                     │
│  [법인설정]                                                          │
│    ├── /entity-settings/custom-apps → EntityCustomAppsTabPage       │
│    │   ├── [탭: 커스텀앱]      → EntityCustomAppsPage (CRUD)        │
│    │   ├── [탭: 공인앱스토어]   → iframe (apps.amoeba.site)         │
│    │   └── [탭: 테스트앱스토어] → iframe (stg-apps.amoeba.site)     │
│    │                                                                 │
│    └── /entity-settings/apps     → EntityAppsPage (마켓플레이스)     │
│                                                                     │
│  [앱 실행]                                                           │
│    └── /apps/:appCode            → CustomAppHostPage (iframe 호스트) │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 핵심 문제: 커스텀앱과 앱스토어가 혼재

| 구분 | 커스텀앱 (사용자 등록) | 앱스토어 (아메바/파트너 공식) |
|------|----------------------|--------------------------|
| **성격** | Entity가 직접 등록하는 외부 서비스 | 아메바 회사와 파트너사가 만든 앱 |
| **관리** | Entity MASTER가 CRUD | 앱스토어에서 구독 신청 → 관리자 승인 |
| **데이터** | `amb_entity_custom_apps` 테이블 | `amb_partner_apps` + `amb_partner_app_installs` |
| **현재 상태** | ⚠️ 하나의 탭 페이지에 통합 | ⚠️ iframe으로 임베디드 + ImportFromStore 혼재 |

## 2. 현재 구현 상태 상세

### 2.1 프론트엔드 구현 현황

#### A. EntityCustomAppsTabPage (법인설정 > 커스텀앱)

| 탭 | 구현 내용 | 상태 | 문제점 |
|----|----------|------|--------|
| **커스텀앱** | `EntityCustomAppsPage` — CRUD, 카드 UI, 수정/삭제/연결테스트 | ✅ 완성 | "앱스토어에서 가져오기" 버튼이 여기에 위치 |
| **공인앱스토어** | iframe → `apps.amoeba.site` | ✅ 동작 | 단순 iframe, 상호작용 없음 |
| **테스트앱스토어** | iframe → `stg-apps.amoeba.site` | ✅ 동작 | 스테이징 전용, 개발 목적 |

#### B. ImportFromStoreModal (앱스토어에서 가져오기)

- **현재 위치**: 커스텀앱 탭의 "앱스토어에서 가져오기" 버튼 클릭 시 오픈
- **데이터 소스**: 내부 DB (`amb_partner_app_installs`) 직접 쿼리
- **기능**: 파트너앱 목록 표시, 검색, 등록/등록됨 상태 표시, 클릭 시 Custom App으로 등록
- **문제**: 
  - 앱스토어앱을 커스텀앱 테이블에 복제하여 "커스텀앱"으로 취급
  - 앱스토어앱과 사용자 직접 등록 앱의 경계가 모호

#### C. 기타 앱 관련 페이지

| 라우트 | 컴포넌트 | 설명 | 상태 |
|--------|---------|------|------|
| `/app-store` | `AppStorePage` | 사이드바에서 접근하는 독립 앱스토어 페이지 (iframe) | ✅ 동작 |
| `/app-store-stg` | `AppStorePage` | 스테이징 앱스토어 | ✅ 동작 |
| `/menu/custom-apps` | `MenuCategoryPage` | 커스텀앱 카드뷰 (사이드바 "Custom Apps" 섹션 클릭) | ✅ 동작 |
| `/entity-settings/apps` | `EntityAppsPage` | 마켓플레이스 앱 설치/관리 | ✅ 동작 |
| `/apps/:appCode` | `CustomAppHostPage` | 개별 앱 iframe 호스트 | ✅ 동작 |

### 2.2 백엔드 구현 현황

#### A. Custom Apps API (`/api/v1/entity-settings/custom-apps`)

| 엔드포인트 | 용도 | 데이터 테이블 | 상태 |
|-----------|------|-------------|------|
| `GET /` | 관리용 앱 목록 | `amb_entity_custom_apps` | ✅ |
| `POST /` | 앱 등록 | `amb_entity_custom_apps` | ✅ |
| `PATCH /:id` | 앱 수정 | `amb_entity_custom_apps` | ✅ |
| `DELETE /:id` | 앱 삭제 (soft) | `amb_entity_custom_apps` | ✅ |
| `GET /my` | 사이드바용 목록 | `amb_entity_custom_apps` | ✅ |
| `POST /:id/token` | JWT/API키 발급 | `amb_entity_custom_apps` | ✅ |
| `POST /:id/health` | 연결 테스트 | `amb_entity_custom_apps` | ✅ |
| `GET /importable` | 앱스토어 구독앱 | `amb_partner_app_installs` + `amb_partner_apps` | ⚠️ 내부DB |
| `POST /import/:papId` | 앱스토어앱 등록 | ← → `amb_entity_custom_apps` | ⚠️ 복제 방식 |

#### B. Service 의존성

```
EntityCustomAppService
  ├── customAppRepo     (EntityCustomAppEntity)       ← 커스텀앱 데이터
  ├── partnerAppRepo    (PartnerAppEntity)            ← 앱스토어 앱 마스터 (내부 DB)
  ├── installRepo       (PartnerAppInstallEntity)     ← 앱스토어 구독 (내부 DB)
  └── jwtService        (JwtService)                  ← 토큰 발급
```

### 2.3 DB 테이블 현황

| 테이블 | 용도 | Entity 격리 |
|--------|------|------------|
| `amb_entity_custom_apps` | 커스텀앱 + import된 앱스토어앱 | ✅ `ent_id` |
| `amb_partner_apps` | 파트너앱 마스터 (앱스토어 원본) | ❌ 전체 공유 |
| `amb_partner_app_installs` | 법인별 앱 구독/설치 기록 | ✅ `pai_entity_id` |

**관련 컬럼**:
- `eca_source_pap_id` (UUID, nullable) — 앱스토어에서 import한 경우 원본 partner_app ID 추적
- `eca_source_app_slug` — 작업계획서에서 추가 예정 (외부 API 연동용)

## 3. 문제점 분석

### 3.1 아키텍처 혼재 문제

```
❌ 현재: 앱스토어앱이 "커스텀앱"으로 복제됨
                                                
  앱스토어 (apps.amoeba.site)                   
    └── app-car-manager (구독 ACTIVE)           
         │                                       
         ▼  [Import From Store]                  
  커스텀앱 테이블 (amb_entity_custom_apps)       
    ├── my-erp-tool      ← 사용자 직접 등록     
    ├── app-car-manager  ← 앱스토어에서 복제! ⚠️ 
    └── my-dashboard     ← 사용자 직접 등록     
```

**문제**:
1. 앱스토어앱의 업데이트(URL, 이름, 아이콘 변경)가 커스텀앱에 반영 안 됨
2. 앱스토어에서 구독 해지해도 커스텀앱에서는 그대로 유지
3. 사용자가 "내가 등록한 앱"과 "앱스토어 앱"을 구분할 수 없음
4. 관리자가 실수로 앱스토어앱을 수정/삭제할 수 있음

### 3.2 데이터 소스 이원화

| 항목 | `ImportFromStoreModal` | `앱스토어 iframe` |
|------|----------------------|-------------------|
| 데이터 소스 | 내부 DB (로컬 쿼리) | 외부 AppStore 사이트 |
| 실시간성 | ❌ 동기화 지연 가능 | ✅ 실시간 |
| 구독 상태 | `paiIsActive` (내부) | 실제 구독 상태 |
| 활용 | 커스텀앱 테이블에 복제 | 시각적 열람만 |

### 3.3 라우팅/UI 중복

| 항목 | 위치 1 | 위치 2 | 위치 3 |
|------|--------|--------|--------|
| 앱스토어 보기 | `/entity-settings/custom-apps` (공인앱스토어 탭) | `/app-store` (사이드바) | `/app-store-stg` |
| 커스텀앱 관리 | `/entity-settings/custom-apps` (커스텀앱 탭) | `/menu/custom-apps` (카드뷰) | - |
| 앱 설치 관리 | `/entity-settings/apps` (EntityAppsPage) | ImportFromStoreModal | - |

## 4. 새 앱스토어 Subscription API

`docs/AppStore-Subscription-API-for-AMA.md`에 정의된 Public API가 제공됨:

```
GET /api/v1/platform/subscriptions/entity/{ent_id}
```

| 항목 | 값 |
|------|---|
| 인증 | 불필요 (Public) |
| 응답 | Entity의 전체 앱 목록 + 구독 상태 (null=미구독, PENDING/ACTIVE/SUSPENDED) |
| 도메인 | stg-apps.amoeba.site (스테이징) / apps.amoeba.site (프로덕션) |

**현재 미활용**: 이 API가 있지만, 현행 코드는 여전히 내부 DB 쿼리 방식을 사용 중.

## 5. 분리 운영을 위한 TO-BE 아키텍처 방향

```
✅ 목표: 커스텀앱과 앱스토어를 명확히 분리

[커스텀앱] — Entity 사용자가 직접 등록하는 외부 서비스
  ├── 데이터: amb_entity_custom_apps (eca_source_pap_id IS NULL)
  ├── 관리: Entity MASTER가 직접 CRUD
  ├── 페이지: /entity-settings/custom-apps → 독립 커스텀앱 관리
  └── 사이드바: "Custom Apps" 섹션에 표시

[앱스토어] — 아메바 회사와 파트너사가 만든 앱
  ├── 데이터: 외부 AppStore API (실시간 조회)
  ├── 관리: 앱스토어에서 구독 → 자동 활성화
  ├── 페이지: /entity-settings/app-store → 전용 앱스토어 관리 페이지  
  │   ├── 구독 현황 조회 (Subscription API 연동)
  │   ├── 구독 상태 표시 (ACTIVE/PENDING/SUSPENDED)
  │   └── 앱스토어 바로가기
  └── 사이드바: "App Store" 또는 별도 섹션에 표시
```

## 6. 현재 상태 요약 (Status Matrix)

| 기능 | 구현 상태 | 데이터 소스 | 비고 |
|------|----------|-----------|------|
| 커스텀앱 CRUD | ✅ 완성 | 내부 DB | 정상 동작 |
| 커스텀앱 토큰 발급 | ✅ 완성 | 내부 DB | JWT/API Key |
| 커스텀앱 연결 테스트 | ✅ 완성 | HTTP fetch | health check |
| 커스텀앱 사이드바 | ✅ 완성 | 내부 DB + 역할 필터 | `/my` 엔드포인트 |
| 앱스토어 iframe 뷰 | ✅ 완성 | 외부 iframe | apps.amoeba.site |
| 앱스토어 Import | ⚠️ 혼재 | 내부 DB (로컬 파트너앱 테이블) | 커스텀앱에 복제 |

| 앱스토어 전용 관리 페이지 | ❌ 없음 | - | iframe만 존재 |
| 커스텀앱/앱스토어 분리 UI | ❌ 미분리 | - | 탭으로만 구분 |
| 앱스토어 구독 상태 조회 | ❌ 미구현 | - | Subscription API 활용 필요 |

## 7. 권장 후속 작업

1. **커스텀앱 페이지 독립화**: "앱스토어에서 가져오기" 버튼 및 `ImportFromStoreModal` 제거, 순수 커스텀앱 CRUD만 유지
2. **앱스토어 구독관리 페이지 신설**: Subscription API 연동하여 구독 현황 조회/표시 전용 페이지
3. **사이드바 분리**: 커스텀앱 / 앱스토어앱 섹션 분리
4. **라우팅 정리**: 중복 라우트 통합, 앱스토어 관련 전용 경로 신설
5. **내부 DB 의존 제거**: `PartnerAppEntity`, `PartnerAppInstallEntity` 참조 제거 가능 여부 확인 후 정리
