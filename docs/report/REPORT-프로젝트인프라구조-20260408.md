## 1. 프로젝트 개요

다법인 글로벌 기업의 미션·태스크·이슈·프로젝트를 지식 자산으로 성장시키고, 채팅·노트·공지의 기본 번역 협업과 Today AI Analysis 기반 일일 업무 분석으로 글로벌 운영 효율을 높이는 엔터프라이즈 플랫폼.

### 기술 스택

| 영역 | 기술 |
|------|------|
| Frontend | React 18 + TypeScript 5 + TailwindCSS 3 + Vite 5 |
| Backend | NestJS 10 + TypeORM + PostgreSQL 15 (pgvector) |
| AI | Anthropic Claude API (`@anthropic-ai/sdk`) |
| 실시간 통신 | SSE (Server-Sent Events) |
| 모노레포 | npm workspaces + Turborepo |
| 컨테이너 | Docker Compose |
| 리버스 프록시 | Nginx |

## 4. 로컬 개발 환경

### 4.1 NPM 스크립트

```bash
npm run dev             # api + web + portal 전체 동시 실행 (turbo)
npm run dev:api         # @amb/api 단독 실행 (포트 3009)
npm run dev:web         # @amb/web 단독 실행 (포트 5179)
npm run dev:portal      # portal-api + portal-web 동시 실행
npm run dev:portal-api  # @amb/portal-api 단독 실행 (포트 3010)
npm run dev:portal-web  # @amb/portal-web 단독 실행 (포트 5180)
npm run dev:mobile      # @amb/mobile 실행
npm run dev:all         # 전체 실행
npm run build           # 전체 빌드
npm run lint            # 린트 검사
npm run lint:fix        # 린트 자동 수정
npm run test            # 테스트 실행
npm run db:up           # PostgreSQL + Adminer Docker 실행
npm run db:down         # PostgreSQL + Adminer Docker 중지
npm run migration:run   # TypeORM 마이그레이션 실행
npm run migration:generate  # TypeORM 마이그레이션 생성
```

## 7. DB 관리

### 7.1 DB 접속 정보

| 환경 | 호스트 | 포트 | DB명 | 사용자 | 비밀번호 |
|------|--------|:----:|------|--------|---------|
| 로컬 | localhost | 5432 | db_amb | amb_user | amb_password |
| 스테이징 | Docker 내부 | 5432 | db_amb | amb_user | (서버 .env) |
| 프로덕션 | Docker 내부 | 5432 | db_amb | amb_user | (서버 .env) |

### 7.2 DB 접속 방법

```bash
# 로컬
psql -h localhost -U amb_user -d db_amb

# 스테이징
ssh amb-staging "docker exec -it amb-postgres-staging psql -U amb_user -d db_amb"

# 프로덕션
ssh amb-production "docker exec -it amb-postgres-production psql -U amb_user -d db_amb"
```

### 7.3 DB 마이그레이션 주의사항

- TypeORM `synchronize`는 **프로덕션/스테이징에서 비활성화** (`NODE_ENV=production`일 때)
- 새 엔티티/컬럼 추가 시 반드시 **수동 SQL 마이그레이션** 필요
- SQL 파일: `sql/` 디렉토리에 번호 순서로 관리
- 마이그레이션 순서: **SQL 실행 → 코드 배포** (역순 시 500 에러 발생)

```bash
# 스테이징에 SQL 실행 예시
cat sql/005-create-plan-features-tiers-addons.sql | \
  ssh amb-staging "docker exec -i amb-postgres-staging psql -U amb_user -d db_amb"

# 프로덕션에 SQL 실행 예시
cat sql/005-create-plan-features-tiers-addons.sql | \
  ssh amb-production "docker exec -i amb-postgres-production psql -U amb_user -d db_amb"
```

## 8. Git 브랜치 전략

### 8.1 브랜치 구조

| 브랜치 | 용도 | 배포 환경 | 보호 규칙 |
|--------|------|-----------|-----------|
| `main` | 개발 통합 (기본 브랜치) | 스테이징 서버 | PR 필수, 1명 승인 |
| `production` | 프로덕션 릴리즈 | 프로덕션 서버 | PR 필수, 1명 승인 |
| `feature/*` | 기능 개발 | 로컬 | — |
| `hotfix/*` | 긴급 버그 수정 | — | — |

### 8.2 개발 플로우

```
feature/* ──(Squash Merge PR)──→ main ──(Merge Commit PR)──→ production
                                  │                            │
                              스테이징 배포                 프로덕션 배포
```

1. `main`에서 `feature/{이름}` 브랜치 생성
2. 작업 완료 후 `main`으로 PR → **Squash Merge**
3. 스테이징 배포 + 테스트
4. `main` → `production`으로 PR → **Merge Commit**
5. 프로덕션 배포

### 8.3 Hotfix 플로우

```
production ──→ hotfix/* ──→ production (직접 머지)
                  └──────→ main (체리픽 또는 머지)
```

### 8.4 커밋 메시지 규칙

```
{type}: {설명}

type: feat | fix | docs | style | refactor | test | chore | hotfix
예: feat: 사용자 프로필 페이지 추가
```
