# 클라이언트 레벨 사용자 로그인 페이지 기능 구현 현황 리포트

**작성일**: 2026-03-20  
**대상 시스템**: AMB Management - Client Portal  
**접속 URL**: `https://ama.amoeba.site/client/login`  

---

## 1. 개요

AMB Management의 **CLIENT_LEVEL** 사용자 인증 시스템은 고객사 담당자가 배정된 프로젝트 범위 내에서 이슈를 등록·추적할 수 있는 전용 포탈이다. 기존 내부 사용자(ADMIN_LEVEL / USER_LEVEL) 인증과 분리된 독립 인증 체계로 운영된다.

**총 구현 완성도: 95%** (이메일 발송 미구현)

### 4.2 회원가입 (✅ 완료 — 초대 기반)

| 항목 | 상세 | 상태 |
|------|------|------|
| **초대 토큰 검증** | URL `?token=` → `GET /invitation/:token` 자동 검증 | ✅ |
| **토큰 만료** | 72시간 (만료 시 EXPIRED 상태 전이) | ✅ |
| **폼 필드** | 이메일(읽기전용), 이름(필수), 비밀번호(≥8자), 비밀번호 확인, 직급(선택), 전화번호(선택) | ✅ |
| **비밀번호 해싱** | bcrypt (salt rounds) | ✅ |
| **중복 가입 방지** | 동일 이메일 CLIENT_LEVEL 계정 존재 시 차단 | ✅ |
| **초대 상태 업데이트** | PENDING → ACCEPTED + `civ_accepted_at` 기록 | ✅ |
| **자동 로그인** | 가입 완료 즉시 JWT 발급 → 대시보드 진입 | ✅ |
| **Rate Limiting** | 5회/분 | ✅ |

**초대 상태 전이:**
```
[생성] PENDING ─→ [가입완료] ACCEPTED
        │
        └──→ [72시간 만료] EXPIRED
```

## 9. 사용자 타입 체계

```
ADMIN_LEVEL (본사)
├─ SUPER_ADMIN
└─ ADMIN

USER_LEVEL (법인 소속)
├─ MASTER
├─ MANAGER
├─ MEMBER
└─ VIEWER

CLIENT_LEVEL (고객사) ← 본 리포트 대상
├─ CLIENT_ADMIN    (고객사 관리자)
└─ CLIENT_MEMBER   (고객사 일반 담당자)
```

## 10. 인증 플로우 다이어그램

```
【초대 → 가입 → 로그인 전체 플로우】

[담당자 MANAGER]
    │
    │ POST /client-management/invite
    │ { email, cli_id, name?, role? }
    ↓
[초대 생성]
    │ token: crypto.randomBytes(32).hex
    │ 만료: +72시간
    │ 상태: PENDING
    ↓
[📧 이메일 발송] ← TODO: Phase 5 구현 예정
    │ 링크: /client/register?token=xxx
    ↓
[고객 담당자]
    │ 초대 링크 클릭
    ↓
[ClientRegisterPage]
    │ → GET /client-auth/invitation/:token (자동 검증)
    │ → 이메일(읽기전용) + 이름 + 비밀번호 입력
    │ → POST /client-auth/register
    ↓
[계정 생성]
    │ usr_level_code = 'CLIENT_LEVEL'
    │ usr_status = 'ACTIVE'
    │ usr_join_method = 'CLIENT_INVITE'
    │ 초대 상태 → ACCEPTED
    ↓
[JWT 발급]
    │ Access Token: 4시간 (httpOnly cookie)
    │ Refresh Token: 7일 (httpOnly cookie)
    ↓
[자동 로그인 → /client 대시보드]

─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─

[이후 로그인]
    │ /client/login
    │ → 이메일 + 비밀번호 입력
    │ → POST /client-auth/login
    ↓
[인증 검증]
    │ ① 이메일로 사용자 조회 (level='CLIENT_LEVEL')
    │ ② bcrypt.compare(password, hash)
    │ ③ 상태 검증 (WITHDRAWN/SUSPENDED/INACTIVE 차단)
    │ ④ usr_last_login_at 갱신
    ↓
[JWT 발급 → 쿠키 설정 → /client 리다이렉트]
```

| **2FA/MFA** | ❌ 미구현 | 낮음 | TOTP/SMS 인증 없음 |
| **계정 잠금 정책** | ❌ 미구현 | 중간 | 연속 로그인 실패 시 계정 잠금 없음 (Throttle만 적용) |
| **다중 기기 세션 관리** | ❌ 미구현 | 낮음 | 동시 로그인 제어 없음 |
| **로그인 이력 (Client)** | ❌ 미구현 | 낮음 | 내부 사용자용 `amb_login_history`는 있으나 CLIENT_LEVEL 미적용 |
| **초대 재발송** | ❌ 미구현 | 중간 | 만료된 초대 재발송 UI/API 미구현 |

## 12. 결론

CLIENT_LEVEL 로그인 시스템은 초대 기반 가입 → JWT 인증 → 프로젝트/이슈 접근 제어 전체 플로우가 **안정적으로 구현**되어 있다. 보안 측면에서 httpOnly 쿠키, bcrypt 해싱, Rate Limiting, ClientGuard 데이터 격리 등 주요 보안 요소가 적용되어 있다.

**즉시 해결 필요 항목**: 초대 이메일 발송 (현재 토큰을 수동으로 전달해야 함)  
**향후 개선 권장**: 비밀번호 찾기, 계정 잠금 정책, 로그인 이력 기록
