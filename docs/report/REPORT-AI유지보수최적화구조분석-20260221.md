4. **응답 포맷 이중 처리** — TransformInterceptor 존재하나 컨트롤러에서 수동 래핑 병행
5. **@amb/common 사실상 미사용** — 프로젝트 전체에서 2곳만 import, 공유 유틸리티 체계 부재

## 2. Backend 분석 (NestJS 10 + TypeORM)

### 2.1 현황

- **파일 수**: 467 TypeScript 파일
- **도메인 모듈**: 22개 (`auth`, `billing`, `chat`, `hr`, `project`, `kms`, `notices`, `webmail`, `accounting` 등)
- **엔티티**: 80개 이상 (app.module.ts 단일 배열에 등록)
- **패턴 일관성**: 8.5 / 10

### 2.2 강점

| 항목 | 상세 |
|------|------|
| **모듈 구조** | 도메인별 `controller/service/entity/dto` 4단 분리 일관 적용 |
| **에러 처리** | `BusinessException` + `HttpExceptionFilter` 이중 레이어, 에러 코드 체계(E1xxx~E9xxx) |
| **가드/데코레이터** | `@Public()`, `@Roles()`, `@CurrentUser()`, `@RequireAccess()` 선언적 접근 제어 |
| **AI 서비스** | `ClaudeService`에서 API 키 우선순위(env→DB→mock) + 5분 캐시 TTL 구현 |
| **트랜잭션 처리** | `postal-webhook.service.ts` 등에서 멱등성(idempotent) 처리 패턴 적용 |

### 2.3 개선 필요 사항

#### 2.3.1 응답 포맷 이중 처리 ⚠️ 심각도: 높음

**현황**: `TransformInterceptor`가 자동으로 `{ success, data, timestamp }` 래핑하지만, 다수 컨트롤러에서 수동 래핑 병행

```typescript
// ❌ 현재: 컨트롤러에서 수동 래핑 (이중 처리 가능성)
return { success: true, data: result };

// ✅ 개선: 인터셉터에 일임, 컨트롤러는 raw data만 반환
return result;
```

**AI 관점**: 패턴이 불일치하면 AI가 새 엔드포인트 생성 시 어느 패턴을 따라야 할지 혼란. 일관된 단일 패턴 필수.

#### 2.3.2 타입 안전성 갭 ⚠️ 심각도: 높음

**현황**: 일부 컨트롤러에서 `req: any`, `query: any` 사용

```typescript
// ❌ 현재
@Get()
async findAll(@Req() req: any, @Query() query: any) { ... }

// ✅ 개선
@Get()
async findAll(@CurrentUser() user: UserPayload, @Query() query: FindAllQueryDto) { ... }
```

**영향 받는 파일**:
- `apps/api/src/domain/billing/controller/contract.controller.ts`
- `apps/api/src/domain/project/controller/project.controller.ts`
- 기타 다수

**AI 관점**: `any` 타입은 AI가 변경의 사이드 이펙트를 추론할 수 없게 만드는 가장 큰 장벽.

#### 2.3.3 God Function 분리 ⚠️ 심각도: 중간

**현황**: `AuthService.register()` — 약 70줄, 7단계 로직이 단일 메서드에 집중

```
1. 이메일 중복 확인
2. 비밀번호 해시
3. 사용자 생성
4. 프로필 생성
5. 권한 할당
6. 환영 메시지 발송
7. 감사 로그 기록
```

**개선 방향**: 각 단계를 private 메서드로 분리하거나, 이벤트 기반(EventEmitter2)으로 후처리 분리

**AI 관점**: 70줄 이상 함수는 AI가 부분 수정 시 전체 로직 파악이 필요해 컨텍스트 비용 증가.

#### 2.3.4 app.module.ts 엔티티 배열 ⚠️ 심각도: 낮음

**현황**: 80개 엔티티가 `entities: [...]` 단일 배열에 나열

**개선 방향**: 각 도메인 모듈에서 `TypeOrmModule.forFeature([...])` 사용 (이미 부분 적용), `autoLoadEntities: true` 옵션 활용

**현황**: 84개 수동 import 문 — 언어/네임스페이스 추가 시마다 3곳 수정 필요

```typescript
// ❌ 현재: 수동 import × 84
import authEn from './locales/en/auth.json';
import authKo from './locales/ko/auth.json';
import authVi from './locales/vi/auth.json';
// ... 81개 더
```

**개선 방향**: `i18next-http-backend` 또는 Vite의 `import.meta.glob` 활용
```typescript
// ✅ 개선: 자동 로딩
const modules = import.meta.glob('./locales/**/*.json', { eager: true });
Object.entries(modules).forEach(([path, mod]) => {
  const [, lang, ns] = path.match(/\.\/locales\/(\w+)\/(\w+)\.json/) || [];
  if (lang && ns) i18n.addResourceBundle(lang, ns, mod.default);
});
```

**AI 관점**: 새 네임스페이스 추가 시 JSON 파일만 생성하면 자동 등록. AI 작업 단계 3→1로 감소.

#### 3.3.4 Zustand Store 사용 기준 명확화 🟡 심각도: 중간

**현황**: 일부 도메인은 store 있고 일부는 없음. React Query와의 역할 분담 기준 불명확.

**개선 방향**: 명확한 가이드라인 수립
```
Zustand Store: UI 상태 전용 (사이드바 열림/닫힘, 선택된 탭, 필터 값)
React Query: 서버 상태 전용 (API 데이터 캐싱, 로딩/에러 상태)
```

## 5. AI 유지보수 최적화를 위한 핵심 원칙

### 5.1 AI 친화적 코드의 3대 원칙

#### 원칙 1: 파일 크기 제한 (Context Window Efficiency)

```
✅ 컴포넌트: ≤ 200줄
✅ 서비스 메서드: ≤ 40줄
✅ 타입 정의 파일: ≤ 300줄
✅ 라우트 파일: ≤ 50줄
❌ God File: 500줄 이상 단일 파일
```

**근거**: AI 모델의 컨텍스트 윈도우(128K~200K 토큰)에서 단일 파일이 차지하는 비율이 높을수록 다른 관련 파일을 동시에 참조할 수 없음. 200줄 이하면 관련 파일 5~10개를 동시에 컨텍스트에 로드 가능.

#### 원칙 2: 패턴 일관성 (Pattern Predictability)

```
✅ 모든 컨트롤러가 동일한 응답 패턴 사용
✅ 모든 도메인 디렉토리가 동일한 구조
✅ 모든 컴포넌트가 동일한 파일 구성
❌ 같은 목적의 코드가 2가지 이상 패턴으로 구현
```

**근거**: AI는 패턴 매칭으로 코드를 생성함. 프로젝트 내 패턴이 일관적이면 AI가 기존 코드를 레퍼런스로 삼아 정확한 코드를 생성. 불일치하면 잘못된 패턴을 선택할 확률 증가.

#### 원칙 3: 명시적 타입 (Explicit Contract)

```
✅ 모든 함수 파라미터에 타입 명시
✅ 모든 API 응답에 타입 정의
✅ DTO로 입출력 스키마 명확화
❌ any, unknown 없이 런타임 타입 확인
❌ 암묵적 타입 추론에 의존
```

**근거**: AI가 코드 변경의 영향 범위를 파악하는 가장 중요한 단서는 타입. `any`는 "영향 범위를 모른다"와 동일.

### 5.2 AI 유지보수를 위한 프로젝트 메타데이터

현재 `CLAUDE.md`가 잘 구성되어 있으나, AI 유지보수 효율을 극대화하기 위해 추가 권장:

```markdown
# CLAUDE.md에 추가 권장 사항

## 도메인별 담당 파일 맵 (AI 탐색 시간 단축)
| 도메인 | 백엔드 | 프론트엔드 | 타입 |
|--------|--------|-----------|------|
| auth | apps/api/src/domain/auth/ | apps/web/src/domain/auth/ | packages/types/src/auth.types.ts |
| billing | apps/api/src/domain/billing/ | apps/web/src/domain/billing/ | packages/types/src/billing.types.ts |
| ... | ... | ... | ... |
```

## 7. AI 유지보수 효율 극대화를 위한 추가 제안

### 7.1 AI 작업 패턴 표준화

각 도메인이 다음 "골든 템플릿"을 따르면, AI가 새 도메인을 추가할 때 기존 도메인을 복사·수정하는 것만으로 완성 가능:

```
domain/{name}/
├── controller/
│   └── {name}.controller.ts     (CRUD 엔드포인트, ≤150줄)
├── service/
│   └── {name}.service.ts        (비즈니스 로직, ≤300줄)
├── entity/
│   └── {name}.entity.ts         (TypeORM 엔티티, ≤100줄)
├── dto/
│   ├── request/
│   │   ├── create-{name}.request.ts
│   │   └── update-{name}.request.ts
│   └── response/
│       └── {name}.response.ts
├── {name}.module.ts             (NestJS 모듈, ≤30줄)
└── README.md                    (도메인 설명, 선택)
```

### 7.2 변경 영향도 추적 강화

```typescript
// packages/types/src/auth.types.ts 상단에 주석 추가
/**
 * @consumers
 *   - apps/api/src/domain/auth/ (backend)
 *   - apps/web/src/domain/auth/ (frontend)
 *   - apps/api/src/domain/hr/   (employee→user 관계)
 * @lastModified 2026-02-21
 */
```

AI가 타입 변경 시 `@consumers` 주석을 보고 영향 받는 모든 위치를 자동으로 수정 가능.

### 7.3 코드 생성 스니펫

AI가 반복 작업을 할 때 사용할 수 있는 프로젝트 전용 스니펫:

```
spec/templates/
├── new-domain.template.md       (새 도메인 모듈 생성 가이드)
├── new-page.template.md         (새 프론트엔드 페이지 생성 가이드)
├── new-entity.template.md       (새 엔티티 + 마이그레이션 가이드)
└── new-api-endpoint.template.md (새 API 엔드포인트 생성 가이드)
```

## 8. 결론

### 8.1 현재 상태 평가

AMB Management 프로젝트는 NestJS + React 기반 엔터프라이즈 시스템으로서 **전반적인 아키텍처와 모듈 구조가 우수**합니다. 특히:

- 22개 도메인 모듈의 일관된 백엔드 구조
- 에러 코드 체계와 선언적 접근 제어
- TypeScript strict mode 전역 적용
- CLAUDE.md를 통한 AI 지침 이미 구비

그러나 **AI가 지속적으로 유지보수**하는 관점에서 보면, 몇 가지 핵심 병목이 존재합니다:

1. **대형 파일**(api.types.ts 2,100줄, DocumentsPage 695줄, router 354줄)이 AI 컨텍스트 윈도우를 비효율적으로 소비
2. **패턴 불일치**(응답 포맷 이중 처리, Store 사용 기준 부재)가 AI의 코드 생성 정확도를 저하
3. **타입 안전성 갭**(`any` 타입)이 AI의 변경 영향 분석을 불가능하게 함

### 8.2 기대 효과

Phase 1~4를 순차적으로 적용하면:

| 단계 | 누적 효과 | AI 작업 시간 단축 | 오류율 감소 |
|------|----------|-------------------|------------|
| Phase 1 완료 | 기본 최적화 | ~30% | ~20% |
| Phase 2 완료 | 구조적 개선 | ~50% | ~40% |
| Phase 3 완료 | 고도화 | ~70% | ~60% |
| Phase 4 완료 | 완전 최적화 | ~80% | ~75% |

### 8.3 최우선 실행 항목 (이번 주 권장)

1. ✅ `api.types.ts` 도메인별 분리 (AI 컨텍스트 효율 즉시 개선)
2. ✅ 컨트롤러 `any` 타입 제거 (변경 영향 분석 가능하게)
3. ✅ 응답 포맷 TransformInterceptor로 통일 (패턴 일관성 확보)

이 3가지만 실행해도 AI의 코드 생성·수정 정확도가 체감할 수 있을 정도로 향상될 것입니다.
