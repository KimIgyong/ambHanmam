# 요구사항 분석서 (REQ)

- 문서명: 회사 자산관리 시스템
- 작성일: 2026-02-24
- 기준 요구: 사용자 제공 REQ-001 ~ REQ-055
- 대상 시스템: AMB Management (NestJS + React + PostgreSQL, 모노레포)

## 1) 요구사항 배경

사용자 제공 명세는 자산 등록/신청/승인/반납/통계/감사까지 포함한 **엔드투엔드 자산관리 플랫폼**을 요구한다. 현재 AMB Management는 협업·업무관리 중심 시스템이며 자산관리 전용 도메인은 부재하므로, 기존 공통 모듈을 재사용하되 신규 도메인 구축이 필요하다.

## 2) AS-IS 현황 분석 (현재 구현 기준)

### 2.1 존재하는 기반 기능
- 권한/메뉴 제어
  - `settings` 도메인에 메뉴 코드, 역할별/그룹별/사용자별 권한 매핑 체계가 존재.
  - 신규 메뉴 코드 추가 시 UI 표시/숨김 및 접근 통제 확장 가능.
- 검색 인프라
  - `search` 도메인이 `amb_work_items` 기반 전역 검색을 제공.
  - 모듈 이벤트(`MODULE_DATA_EVENTS`) 연계 시 신규 도메인 검색 편입 가능.
- 이력/로그 패턴
  - `todo`, `issues`, `project`, `translation`, `kms` 등에서 상태 이력/버전 히스토리 저장 패턴 존재.
  - 감사 로그성 엔티티(`access-audit-log`) 구조 존재.
- 스케줄러/자동화
  - `@nestjs/schedule` + `Cron` 사용 사례(태그 배치, 메일 동기화, 빌링 자동화) 존재.
- 파일 업로드
  - 공통 파일 업로드 모듈(`infrastructure/file`) 존재.
- 엑셀 UX 패턴
  - 프론트 `accounting`에 `ExcelImportModal` 패턴 존재(직접 재사용보다는 패턴 참고 용도).

### 2.2 부재/미구현 영역
- 자산관리 전용 백엔드 도메인(`asset`, `asset-request`, `meeting-room`) 부재.
- 자산 엔티티, 신청 엔티티, 승인 이력 엔티티, 반납 엔티티 부재.
- 회의실 예약 캘린더/시간 슬롯 충돌 검증 로직 부재.
- 자산 전용 통계/리포트 API 부재.
- 자산 변경 immutable 로그(5년 보존 정책 포함) 부재.
- SSO 연동 및 로그인 실패 잠금 정책은 현 시스템 기본 구현과 별도 검토 필요.

## 3) TO-BE 요구사항 해석

사용자 명세(REQ-001~055)는 다음 6개 서브도메인으로 분해된다.

1. 자산 마스터 관리 (등록/수정/상태/이력/검색)
2. 자산 사용 신청 (유형별 신청/검증/개인 목록)
3. 승인/배정/반납 운영 (부서장·관리자 프로세스)
4. 회의실 예약 운영 (캘린더/중복검증/자동종료)
5. 통계·리포트·감사 로그
6. 권한/보안/성능/가용성 비기능 요구

| 검색 연동 | work_item 기반 공통 검색 존재 | 자산/신청 검색 포함 | 이벤트/모듈 매핑 추가 필요 |

| 권한 | 메뉴 권한 체계 존재 | 일반/부서장/관리자 권한 분기 | 메뉴코드 + 데이터 스코프 정책 추가 필요 |
| 감사로그 5년 보존 | 일부 로그 테이블 존재 | 변경불가 장기보존 정책 필요 | 별도 보존전략/DB정책 설계 필요 |

## 5) 권장 아키텍처 방향

### 5.1 도메인 분리 원칙
- 기존 `work-schedule`(출퇴근/근무표)과 자산관리 일정/예약은 목적이 다르므로 **완전 분리**한다.
- 신규 도메인 예시:
  - `asset` (자산 마스터)
  - `asset-request` (신청/승인/반납)
  - `meeting-room` (회의실 자산 + 시간 예약)
  - `asset-report` (통계/리포트)

### 5.2 DB 모델 초안 (개념)
- `amb_assets`
  - 코드, 명칭, 구분, 상태, 보유형태, 부서, 담당자, 위치, 구매/감가 정보, 바코드/RFID
- `amb_asset_requests`
  - 신청번호, 신청자, 유형, 자산지정방식, 자산코드(선택), 사용기간, 장소, 상태
- `amb_meeting_reservations`
  - 요청 FK, 회의제목, 참석인원, 회의성격, 시작/종료일시, 필요장비
- `amb_asset_approval_histories`
  - 단계, 승인자, 결과, 의견, 처리일시
- `amb_asset_change_logs`
  - 변경항목, 이전값/변경값, 변경사유, 변경자, 변경일시
- `amb_asset_request_logs`
  - 신청 상태 변경 로그(전/후 상태, 사유, 처리자)

### 5.3 상태머신 초안
- 자산 상태: `보관 → 예약됨 → 사용중 → 수리중 → 폐기예정 → 폐기완료`
- 신청 상태: `임시저장 → 제출(승인대기) → 1차승인 → 최종승인 → 사용중 → 사용완료`, 예외 `반려/취소`

## 6) 요구사항별 구현 적합성 분석 (요약)

### 6.1 높은 적합도 (기존 패턴 재사용 가능)
- REQ-023~029(목록/승인/반려): 빌링/프로젝트 승인 흐름 패턴 재사용 가능
- REQ-040~044(권한/메뉴): `settings` 권한 체계 재사용 가능
- REQ-009 일부(검색 UI): 전역 검색 바 구조 재사용 가능
- REQ-030 일부(알림 트리거): `Cron + Mail/TODO` 패턴 재사용 가능

### 6.2 중간 적합도 (확장 필요)
- REQ-007/008/045/046/048: 이력 테이블/조회 패턴은 있으나 자산 도메인 전용 스키마 필요
- REQ-005: 프론트 엑셀 UX 패턴은 있으나 서버 검증/오류엑셀 생성 신규 구현 필요
- REQ-035~037: 기존 대시보드/통계 컴포넌트 패턴은 있으나 지표 산출 로직 신규 필요

### 6.3 낮은 적합도 (신규 구축 필요)
- REQ-001~004, 006, 010~022: 자산/신청 핵심 기능 대부분 신규
- REQ-017/038/039: 회의실 시간 단위 충돌 및 캘린더 운영 신규
- REQ-047: 5년 보존 + 수정/삭제 불가 정책(운영/컴플라이언스 수준) 신규 설계 필요
- REQ-051: SSO 연동 및 계정 잠금 정책은 현 인증체계와 별도 통합 과제

## 7) 기술 제약 및 설계 유의사항

1. 기존 API 규칙 유지
   - Request DTO `snake_case`, Response DTO `camelCase`, 표준 응답 포맷 유지.
2. 멀티 엔티티 스코프
   - `ent_id` 기반 분리 및 권한 스코프 강제.
3. 공개범위/검색 연계
   - 필요 시 `WorkItem` 동기화 이벤트를 사용해 검색 통합.
4. 회의실 중복 검증
   - DB 레벨에서 시간 중복 방지(트랜잭션 + 락/제약) 설계 필요.
5. 알림 신뢰성
   - 종료예정 알림(3일/1일) 및 자동종료는 스케줄러 장애 대비 재처리 전략 필요.

## 9) 수용 기준 (분석서 기준)

- AC-1: 자산관리 도메인이 기존 `work-schedule`과 분리된 메뉴/테이블/API로 설계된다.
- AC-2: 사용자 제공 REQ-001~055를 구현 우선순위(Phase)로 추적 가능하다.
- AC-3: 재사용 가능한 AS-IS 공통 기능(권한/검색/이력/스케줄러)이 명확히 식별된다.
- AC-4: 신규 구축이 필요한 고갭 영역(회의실 시간중복, 엑셀검증, 5년 보존, SSO)이 문서화된다.

## 10) 결론

현재 코드베이스는 자산관리 시스템을 즉시 제공하는 상태는 아니며, **공통 기반은 충분하지만 핵심 도메인은 신규 구축이 필수**다. 따라서 본 요구사항은 단일 기능 추가가 아니라 신규 업무 도메인 도입 과제로 분류해야 하며, MVP 우선 범위를 명확히 나눠 단계적으로 구현하는 것이 적절하다.
