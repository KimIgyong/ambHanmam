# 요구사항분석서: 법인별 구글드라이브 설정 기능

**작성일**: 2026-03-14  
**요청**: HQ의 구글드라이브 설정정보를 VN01에서도 사용할 수 있도록 하는 기능

---

## 1. AS-IS 현황 분석

### 1.1 아키텍처 개요

| 구분 | 경로 | 기능 | Guard |
|------|------|------|-------|
| **Admin Drive** | `/admin/drive` | 전역(시스템) Drive 설정 관리 | AdminGuard |
| **Entity Drive** | `/entity-settings/drive` | 법인별 Drive 설정 관리 | OwnEntityGuard |

### 1.2 데이터 저장 구조

#### (A) 인증 정보 (서비스 계정)
- **저장 위치**: 서버 환경변수로만 관리 (DB 미저장)
- `GOOGLE_SERVICE_ACCOUNT_KEY_PATH` — JSON 키 파일 경로
- `GOOGLE_SERVICE_ACCOUNT_KEY_JSON` — JSON 문자열 / Base64
- **특징**: 전역 싱글턴 (`GoogleDriveService`), 법인별 분리 불가

#### (B) 운영 설정 (DB)
- **테이블**: `amb_drive_settings`

| 컬럼 | 설명 | 비고 |
|------|------|------|
| `drs_id` (PK) | 설정 UUID | |
| `ent_id` (FK) | 법인 ID | null = 전역 설정 |
| `drs_impersonate_email` | 위임 이메일 | Domain-Wide Delegation |
| `drs_billing_root_folder_id` | 빌링 루트 폴더 ID | Google Drive 폴더 |
| `drs_billing_root_folder_name` | 루트 폴더 표시명 | |
| `drs_updated_at` | 수정 시각 | |
| `drs_updated_by` | 수정자 | |

#### (C) 법인(Entity) 구조
- **테이블**: `amb_hr_entities`
- HQ: `ent_level='ROOT'`, `ent_is_hq=true`, `ent_parent_id=null`
- VN01: `ent_level='SUBSIDIARY'`, `ent_parent_id=HQ의 ent_id`

### 1.3 현재 동작 방식

```
                    ┌──────────────┐
                    │ /admin/drive │  → DriveSettingsService.getSettings()     → entityId=없음 → 전역
                    └──────────────┘
                    ┌────────────────────────┐
                    │ /entity-settings/drive │  → DriveSettingsService.getSettings(entityId) → 법인별
                    └────────────────────────┘
```

- **전역 설정** (entityId=null): admin 페이지에서 관리, env fallback 지원
- **법인별 설정** (entityId 지정): entity-settings에서 관리, **fallback 없음**
  - 해당 법인 레코드가 없으면 `configured=false`, 모든 값 `null` 반환
- `GoogleDriveService`: NestJS 싱글턴, 서버 전체에서 하나의 서비스 계정 + impersonate email 사용

### 1.4 프론트엔드 현황

| 페이지 | 상태 | 기능 |
|--------|------|------|
| `/admin/drive` (DriveSettingsPage) | ✅ 활성 | 풍부한 UI: 연결 테스트, Shared Drive 브라우저, 폴더 트리 탐색 |

### 1.5 Drive 사용처

| 모듈 | 사용 방식 | entityId 인식 |
|------|----------|--------------|

## 2. TO-BE 요구사항

### 2.1 핵심 요구사항

> **HQ 법인의 구글드라이브 설정정보를 VN01 법인에서도 사용할 수 있어야 한다.**

이는 다음을 의미합니다:
1. HQ에 이미 설정된 Drive 설정을 VN01이 **상속/공유**하거나
2. VN01에서 HQ의 설정을 **복사**하여 사용하거나
3. 법인별로 **자체 설정**도 가능해야 한다

### 2.2 구체적 요구사항

| # | 요구사항 | 우선순위 |
|---|---------|---------|
| R1 | Entity Settings에 독립적인 Drive 설정 페이지 제공 (`/entity-settings/drive`) | 필수 |
| R2 | 법인별 Drive 설정이 없을 때 **상위 법인(HQ) 설정으로 fallback** | 필수 |
| R3 | Fallback 중인 상태를 UI에서 명확히 표시 (예: "HQ 설정 사용 중") | 필수 |
| R4 | 법인 독자 설정 시 HQ 설정 대신 자체 설정 사용 | 필수 |
| R5 | API Keys 페이지(`/entity-settings/api-keys`)와 동일한 UI 패턴 | 권장 |
| R6 | 연결 테스트 기능 (현재 admin 페이지에만 있음) | 권장 |
| R7 | HQ 설정 복사 버튼 ("HQ 설정 가져오기") | 선택 |

## 4. 구현 방안

### 4.1 방안 비교

| 방안 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A: Fallback 방식** | 법인 설정 없으면 자동으로 HQ 설정 사용 | 단순, 즉시 동작 | 하위 법인이 HQ에 의존 |
| **B: 복사 방식** | HQ 설정을 VN01로 복사하여 독립 레코드 생성 | 완전 독립 | 원본 변경 시 동기화 안 됨 |
| **C: 혼합 (권장)** | 기본 fallback + 자체 설정 오버라이드 가능 | 유연, 즉시 동작 + 커스터마이즈 | 약간 복잡 |

### 4.2 권장안: 방안 C (Fallback + Override)

```
                         VN01 설정 조회 요청
                               │
                  ┌────────────▼────────────┐
                  │  VN01 자체 설정 있는가?  │
                  └────────┬────────────┬───┘
                     YES   │            │  NO
                  ┌────────▼───┐   ┌────▼──────────────┐
                  │ 자체 설정  │   │ HQ (parent) 설정   │
                  │ source:own │   │ source:inherited   │
                  └────────────┘   └────────────────────┘
```

**동작 규칙:**
1. 법인(VN01)에 `amb_drive_settings` 레코드가 **있으면** → 자체 설정 사용 (`source: 'own'`)
2. 법인 레코드가 **없으면** → 상위 법인(HQ) 설정 조회 (`source: 'inherited'`)
3. 상위 법인도 **없으면** → 전역 설정(entId=null) fallback (`source: 'global'`)
4. "자체 설정 저장" 시 VN01 전용 레코드 생성 → 이후 HQ 변경 무관
5. "HQ 설정으로 되돌리기" 시 VN01 레코드 삭제 → fallback 복귀

### 4.3 백엔드 변경

#### (1) `DriveSettingsService.getSettings()` 수정

```typescript
async getSettings(entityId?: string): Promise<DriveSettingsResponse> {
  if (entityId) {
    // 1) 법인 자체 설정 확인
    const own = await this.driveSettingsRepo.findOne({ where: { entId: entityId } });
    if (own) return this.toResponse(own, 'own');

    // 2) 상위 법인 설정 fallback
    const parentEntity = await this.hrEntityRepo.findOne({ where: { entId: entityId } });
    if (parentEntity?.entParentId) {
      const parent = await this.driveSettingsRepo.findOne({ where: { entId: parentEntity.entParentId } });
      if (parent) return this.toResponse(parent, 'inherited');
    }

    // 3) 전역 설정 fallback
    const global = await this.driveSettingsRepo.findOne({ where: {} });
    if (global?.drsImpersonateEmail) return this.toResponse(global, 'global');

    return { configured: false, source: 'none', ... };
  }
  // 기존 전역 조회 로직
}
```

## 5. 기술 제약사항

### 5.1 GoogleDriveService 싱글턴 제약

- 현재 `GoogleDriveService`는 NestJS **싱글턴**으로, `reinitialize()` 호출 시 서버 전체에서 하나의 impersonate email만 사용
- **법인마다 다른 impersonate email이 필요한 경우**: 매 요청마다 임시 auth client를 생성하는 패턴 필요 (현재 구조에서는 큰 리팩토링)
- **현실적 상황**: 동일한 Google Workspace의 서비스 계정을 사용하므로 HQ와 VN01이 같은 impersonate email을 공유해도 문제 없음 → **현재 구조 유지 가능**

### 5.2 서비스 계정 키

- 서비스 계정 JSON 키는 **환경변수로만 관리** (DB 저장 안 함)
- HQ와 VN01이 같은 Google Workspace를 사용한다면 동일한 서비스 계정으로 충분
- 다른 Workspace를 사용한다면 별도 서비스 계정 필요 → 현재 아키텍처로는 지원 불가 (향후 확장 과제)

### 5.3 Billing Root Folder

- 법인마다 **다른 빌링 루트 폴더**를 사용할 가능성이 높음 (법인별 회계 문서 분리)
- 이 경우 VN01이 HQ 폴더를 그대로 상속하면 안 될 수 있음
- → fallback은 impersonate email에만 적용하고, folder ID는 법인별 필수 입력으로 할 수도 있음

## 6. 사용자 플로우

### 6.1 VN01이 HQ 설정을 상속받는 경우 (기본)

1. VN01 관리자가 `/entity-settings/drive` 접속
2. HQ 설정이 자동 로드됨, 배너에 "HQ 설정 사용 중" 표시
3. "Test Connection" 클릭 → 연결 확인
4. 별도 조치 없이 Drive 기능 사용 가능

### 6.2 VN01이 자체 설정을 하는 경우

1. VN01 관리자가 `/entity-settings/drive` 접속
2. "Use Own Settings" 또는 폼 값 수정 후 "Save"
3. VN01 전용 레코드 생성, 배너 "자체 설정 사용 중"으로 변경
4. 이후 HQ 설정 변경과 무관하게 동작

### 6.3 VN01이 HQ 설정으로 되돌리는 경우

1. VN01 관리자가 "Revert to HQ Settings" 클릭
2. 확인 다이얼로그 표시
3. VN01 레코드 삭제 → HQ fallback 복귀
