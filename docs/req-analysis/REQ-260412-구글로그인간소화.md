## 2. AS-IS 현황 분석

### 2.1 현재 구글 가입 흐름 (6단계)

```
[1] 사용자: "Continue with Google" 클릭 (Register 또는 Login 페이지)
     ↓
[2] Portal-API: GET /portal/auth/google/start → Google OAuth URL 반환
     ↓
[3] 팝업 or 페이지 리다이렉트: Google 계정 선택 → 인증 코드 발급
     ↓
[4] Portal-API: GET /portal/auth/google/callback
    - Google Access Token 교환 → 사용자 정보 획득
    - Portal Customer 생성 (랜덤 비밀번호, ACTIVE, GOOGLE provider)
    - onboarding_token 발급 (20분 TTL)
    - 콜백 페이지로 리다이렉트
     ↓
[5] GoogleOnboardingPage: 온보딩 폼 표시 ★ 현재 병목
    - 회사명 (필수) + 국가 (필수) + 비밀번호 (필수) + 전화번호 + 약관 동의
    - POST /portal/auth/google/onboarding 호출
    - Portal Customer 업데이트 (비밀번호 저장)
    - auto-provision 호출 → AMA 계정 생성 (또 다른 임시 비밀번호)
    - auto-login-token 발급
     ↓
[6] Success 화면: "Go to AMA" 클릭 → AMA auto-login
```

### 2.2 현재 비밀번호 생성 시점 (총 3개)

| 시점 | 비밀번호 | 용도 |
|------|---------|------|
| Google Callback | `uuid().slice(0,16)` 랜덤 | Portal Customer 생성 시 placeholder |
| Google Onboarding | 사용자 직접 입력 | Portal Customer에 저장 (위 랜덤 교체) |
| AMA Auto-Provision | `uuid().slice(0,12)` 랜덤 | AMA User 생성, 이메일로 발송 |

→ **Google 사용자인데도 비밀번호를 3번 생성**하는 구조 (불필요한 복잡도)

## 3. TO-BE 변경 설계

### 3.1 변경 후 구글 가입 흐름 (3단계로 단축)

```
[1] 사용자: "Continue with Google" 클릭
     ↓  (페이지 리다이렉트, 팝업 없음)
[2] Google 인증 → Portal-API Callback
    - Portal Customer 생성 (비밀번호 없음/null, GOOGLE provider)
    - SvcClient 생성
    - AMA auto-provision 즉시 실행 (비밀번호 없이)
    - auto-login-token 발급
    - 리다이렉트: ama.amoeba.site/{entityCode}/login?alt={token}
     ↓
[3] AMA 자동 로그인 → /today 대시보드 진입
    - mustChangePw = false (비밀번호 변경 불필요)
    - InitialSetupPage 스킵
```

### 3.2 핵심 변경 원칙

1. **비밀번호 제거**: Google 사용자는 비밀번호 없이 OAuth 토큰 기반 인증
2. **온보딩 제거**: 회사명/국가는 Google 프로필 또는 AMA InitialSetup에서 선택적 수집
3. **즉시 프로비저닝**: Callback에서 바로 Portal + AMA 계정 동시 생성
4. **자동 리다이렉트**: 성공 화면 없이 바로 AMA로 이동

## 4. 변경 영향 분석 (사이드 임팩트)

### 4.1 🔴 위험도 높음 (CRITICAL)

#### SI-1: Portal 로그인 시 비밀번호 인증 실패
| 항목 | 내용 |
|------|------|
| **영향** | Google 가입 사용자가 www.amoeba.site에서 이메일+비밀번호로 로그인 시도 불가 |
| **원인** | 비밀번호를 null/빈값으로 저장하면 bcrypt.compare() 실패 |
| **대응** | Portal-API `login()` 메서드에 Google 사용자 분기 추가 → `pctAuthProvider === 'GOOGLE'`이면 "Use Google Sign-in" 안내 에러 반환. 또는 Google 사용자 로그인 시 자동으로 Google OAuth 리다이렉트 |
| **파일** | `portal-auth.service.ts` login(), `LoginPage.tsx` 에러 처리 |

#### SI-2: AMA 로그인 시 비밀번호 인증 실패
| 항목 | 내용 |
|------|------|
| **영향** | AMA에서 이메일+비밀번호 로그인 불가 (EntityLoginPage) |
| **원인** | AMA User의 `usrPassword`가 null이면 일반 로그인 불가 |
| **대응** | **(방안 A)** AMA에도 Google OAuth 로그인 버튼 추가 → Portal-API 경유 인증 **(방안 B)** AMA 로그인은 항상 Portal auto-login-token 경유 (Google 사용자) **(방안 C)** AMA User에 `usrAuthProvider` 필드 추가, Google 사용자는 비밀번호 로그인 차단 |
| **파일** | `auth.service.ts` login(), `user.entity.ts`, EntityLoginPage |

#### SI-3: "비밀번호 찾기" 기능 오작동
| 항목 | 내용 |
|------|------|
| **영향** | Google 사용자가 "Forgot Password" 클릭 시 비밀번호 재설정 이메일 수신 후 비밀번호 설정하면 이중 인증 경로 발생 |
| **원인** | 비밀번호 null인 사용자에게 리셋 링크를 보내면 비밀번호를 새로 설정하게 되어 Google-only 원칙 위반 |
| **대응** | Portal + AMA 비밀번호 찾기에서 Google 사용자 차단. "Google 계정으로 로그인하세요" 안내 표시 |
| **파일** | Portal: `ForgotPasswordPage.tsx`, AMA: `ForgotPasswordPage.tsx`, 백엔드 리셋 API |

## 6. 구현 단계 (권장 순서)

### Phase 1: 백엔드 기반 작업 (AMA-API + Portal-API)
1. AMA-API: `user.entity.ts`에 `usrAuthProvider` 컬럼 추가 (nullable, default null)
2. AMA-API: `portal-bridge.service.ts` — Google 사용자 프로비저닝 시 `usrPassword = null`, `usrMustChangePw = false`, `usrAuthProvider = 'GOOGLE'`
3. AMA-API: `auth.service.ts` — login()에서 password null 사용자 차단, initialSetup() 비밀번호 optional 처리
4. Portal-API: `handleGoogleCallback()` — Callback에서 바로 프로비저닝 + auto-login-token 발급
5. Portal-API: 기존 Google 사용자 재로그인 처리 (email_exists → 자동 로그인)

### Phase 2: 프론트엔드 변경 (Portal-Web + AMA-Web)
6. Portal-Web: Step1AccountForm 팝업 로직 제거 → 페이지 리다이렉트 통일
7. Portal-Web: Google 버튼 클릭 전 약관 동의 수집 UI
8. Portal-Web: GoogleCallbackPage / GoogleOnboardingPage 단순화 또는 제거
