## 2. AS-IS 현황 분석

### 2.1 로그인 플로우 (현행)

```
[모든 사용자] → /login (공통 LoginPage)
    ↓ authService.login(email, password, entityCode)
    ↓ JWT 발급 (level, role 포함)
[LoginPage] → navigate('/select-entity')
    ↓
[EntitySelectPage]
    ├── isAdmin() → navigate('/admin')
    └── !isAdmin() → navigate('/')
```

**문제점:**
1. ADMIN_LEVEL 사용자가 일반 사용자와 동일한 `/login` 경로로 진입
2. 로그인 시 법인(entity) 선택이 요구되지만, 어드민은 법인에 소속되지 않음 → 불필요한 EntitySelectPage 경유
3. 어드민 전용 URL이 없어 북마크/직접 접근 불가
4. 일반 사용자 UI와 혼재되어 어드민 업무 진입이 불편

### 2.2 어드민 대시보드 (현행)

| 항목 | 현황 |
|------|------|
| **구성** | 4개 통계 카드 (Active Services, Entities, Total Users, AI Usage) — 대부분 하드코딩 "—" |
| **Quick Actions** | `SETTINGS_CARDS` 배열 基반 그리드 표시 |
| **실제 데이터** | 서비스 수만 실제 조회, 나머지 placeholder |

**문제점:**
- Quick Actions가 SETTINGS_CARDS 재활용으로 전체 어드민 메뉴를 대시보드에서 직관적으로 파악하기 어려움
- 통계 데이터가 대부분 미연동

### 2.3 유저 관리 (현행)

| 페이지 | 경로 | 대상 | 기능 |
|--------|------|------|------|
| TotalUserManagementPage | `/admin/total-users` | 전체 사용자 | 7탭 (Portal Customers, Users, Entity Roles, Unit Roles, Invitations, Drive, AI) |
| EntityManagementPage | `/admin/user-management` | 법인별 관리 | 법인 목록 → 상세 (Summary, Services, AI Usage) |
| MembersSettingPage | `/admin/members` | 법인 멤버 | 내부 멤버 관리 |

**문제점:**
1. **ADMIN_LEVEL 사용자 전용 관리 페이지 없음** — 어드민 계정 생성/수정/삭제/역할변경이 별도 UI 없이 DB 직접 조작
2. **PARTNER_LEVEL 사용자 전용 관리 페이지 없음** — 파트너 사용자가 InvitationsTab에 혼재되어 관리 불편
3. TotalUserManagementPage의 "Users" 탭이 모든 레벨의 사용자를 혼합 표시

### 2.4 파트너 초대 (현행)

| 항목 | 현황 |
|------|------|
| **초대 진입점** | PartnerManagementPage (`/admin/partners`) 내 `Send` 버튼 |
| **초대 모달** | `InvitePartnerUserModal` (인라인 컴포넌트, 페이지 내부 정의) |
| **입력 필드** | email + role(PARTNER_MEMBER/PARTNER_ADMIN)만 |
| **API** | 기존 `entitySettingsService.inviteMember()` 재사용 (`level_code: 'PARTNER_LEVEL'`) |

**문제점:**
1. 초대가 파트너 조직 목록 페이지 안에 종속 — 직관적인 "초대 관리" UI 부재
2. 초대 이력은 TotalUserManagementPage > InvitationsTab에서만 확인 가능
3. 다수의 파트너 사용자를 초대할 때 파트너사 ↔ 초대 현황 확인이 불편
4. 초대 후 상태 추적(PENDING/ACCEPTED/EXPIRED)이 파트너별로 필터링 안 됨

## 3. TO-BE 요구사항 상세

### REQ-01: ADMIN_LEVEL 전용 로그인 경로 분리

#### 3.1.1 기능 설명

ADMIN_LEVEL 사용자를 위한 전용 로그인 페이지(`/admin/login`)를 구현한다. 어드민은 법인(entity) 소속이 아니므로 법인 선택 단계를 생략하고, 이메일 + 비밀번호만으로 바로 로그인한다.

#### 3.1.2 플로우 (TO-BE)

```
[어드민] → /admin/login (AdminLoginPage)
    ↓ adminAuthService.login(email, password)  ← entity_code 미전달
    ↓ JWT 발급 (level='ADMIN_LEVEL')
    ↓ 로그인 성공 → navigate('/admin')
    ↓
[AdminDashboardPage] (바로 진입, EntitySelectPage 경유 없음)
```

#### 3.1.3 기능 상세

| # | 항목 | 설명 |
|---|------|------|
| 1 | **URL** | `/admin/login` — PublicRoute (이미 어드민 로그인 상태면 `/admin` 리다이렉트) |
| 2 | **로그인 폼** | 이메일 + 비밀번호 (법인 선택 없음) |
| 3 | **백엔드 처리** | 기존 `/api/v1/auth/login` 엔드포인트 사용. `entity_code` 미전달 시 ADMIN_LEVEL 사용자 자동 매칭 로직 이미 존재 |
| 4 | **로그인 성공** | `/admin`으로 직접 이동 (EntitySelectPage 경유 없음) |
| 5 | **로그인 실패** | "이메일 또는 비밀번호가 올바르지 않습니다" 에러 표시 |
| 6 | **비어드민 차단** | 로그인 후 JWT의 `level`이 `ADMIN_LEVEL`이 아닌 경우 → "이 페이지는 관리자 전용입니다" 에러 표시 + 입력 유지 |
| 7 | **UI 디자인** | 기존 LoginPage 스타일 기반, 브랜딩: "AMB ADMIN" 로고/타이틀 |
| 8 | **다국어** | ko/en/vi 지원 (`admin` 네임스페이스에 번역키 추가) |
| 9 | **기존 경로** | `/login` 기존 로그인은 그대로 유지 (일반 USER_LEVEL + CLIENT_LEVEL 용) |

#### 3.1.4 라우터 변경

| 변경 | 내용 |
|------|------|
| 추가 | `/admin/login` → `AdminLoginPage` (PublicRoute로 감싸, 어드민 인증 시 `/admin` redirect) |
| 유지 | 기존 `/login`, `/select-entity` 경로 및 로직 그대로 유지 |
| 유지 | 기존 ADMIN_LEVEL 사용자의 `/login` 경로 접근도 허용 (하위 호환) |

#### 3.1.5 사이드 임팩트

- 기존 `/login` 경로로 어드민 로그인하는 플로우는 **하위 호환 유지** (코드 삭제 없음)
- EntitySelectPage에서 `isAdmin()` 분기 로직 유지
- 백엔드 API 변경 없음 (기존 `entity_code` 미전달 로직 활용)

### REQ-03: 어드민레벨 유저 관리 페이지

#### 3.3.1 기능 설명

ADMIN_LEVEL 사용자(시스템 운영자)를 전용 관리하는 페이지를 구현한다. 현재 어드민 계정은 DB 직접 조작으로만 관리 가능하며, UI를 통한 관리 수단이 없다.

#### 3.3.2 UI 구성 — `/admin/admin-users`

**목록 테이블:**

| 컬럼 | 설명 |
|------|------|
| 이름 | 사용자 이름 |
| 이메일 | 로그인 이메일 |
| 역할 | SUPER_ADMIN / ADMIN |
| 상태 | ACTIVE / INACTIVE / PENDING |
| 마지막 로그인 | 최종 로그인 일시 |
| 가입일 | 계정 생성일 |
| 액션 | 수정 / 비활성화 / 삭제 |

**기능:**

| # | 기능 | 설명 |
|---|------|------|
| 1 | **목록 조회** | ADMIN_LEVEL 사용자만 필터링하여 표시 |
| 2 | **검색** | 이름, 이메일 검색 |
| 3 | **상태 필터** | ALL / ACTIVE / INACTIVE / PENDING 탭 |
| 4 | **어드민 계정 생성** | 이메일 + 이름 + 비밀번호 + 역할(SUPER_ADMIN/ADMIN) 입력 모달 |
| 5 | **역할 변경** | SUPER_ADMIN ↔ ADMIN 역할 전환 (SUPER_ADMIN만 가능) |
| 6 | **상태 변경** | ACTIVE ↔ INACTIVE 토글 |
| 7 | **비밀번호 초기화** | 임시 비밀번호 발급 + `mustChangePw` 플래그 설정 |
| 8 | **삭제** | Soft delete (confirm 필수) |

#### 3.3.3 권한 제어

| 기능 | SUPER_ADMIN | ADMIN |
|------|:-----------:|:-----:|
| 목록 조회 | ✅ | ✅ |
| 어드민 생성 | ✅ | ❌ |
| 역할 변경 | ✅ | ❌ |
| 상태 변경 | ✅ | ❌ |
| 비밀번호 초기화 | ✅ | ✅ (본인만) |
| 삭제 | ✅ | ❌ |
| 자기 자신 삭제 | ❌ | ❌ |

#### 3.3.4 백엔드 API (신규)

| Method | Path | Guard | 설명 |
|--------|------|-------|------|
| GET | `/api/v1/admin/admin-users` | @AdminOnly | 어드민 사용자 목록 |
| POST | `/api/v1/admin/admin-users` | @SuperAdminOnly | 어드민 계정 생성 |
| GET | `/api/v1/admin/admin-users/:id` | @AdminOnly | 어드민 상세 |
| PATCH | `/api/v1/admin/admin-users/:id` | @SuperAdminOnly | 어드민 수정 (역할/상태) |

### REQ-05: 파트너 초대 전용 페이지

#### 3.5.1 기능 설명

파트너 사용자 초대를 위한 전용 페이지를 구현한다. 현재는 PartnerManagementPage 내 인라인 모달로만 가능하여 다수 초대 관리와 이력 추적이 불편하다.

#### 3.5.2 UI 구성 — `/admin/partner-invitations`

**2탭 구조:**

##### Tab 1: 새 초대 발송

| 필드 | 타입 | 필수 | 설명 |

| 파트너사 | 드롭다운 (검색 가능) | ✅ | 활성 파트너사 목록 |
| 이메일 | 텍스트 입력 | ✅ | 초대 대상 이메일 |
| 역할 | 라디오 (PARTNER_ADMIN / PARTNER_MEMBER) | ✅ | 기본값: PARTNER_MEMBER |
| 비고 | 텍스트 (선택) | ❌ | 초대 메모 |

- **다건 초대**: "추가" 버튼으로 여러 건 입력 후 일괄 발송
- **발송**: `entitySettingsService.inviteMember()` 기존 API 재사용 (`level_code: 'PARTNER_LEVEL'`)

| 컬럼 | 설명 |

| 필터 | 옵션 |

## 5. 사용자 플로우

### 5.1 어드민 로그인 플로우 (TO-BE)

```
1. 어드민이 /admin/login 접속
2. 이메일 + 비밀번호 입력
3. 로그인 버튼 클릭
4. [백엔드] entity_code 없이 login → ADMIN_LEVEL 사용자 매칭 → JWT 발급
5. [프론트] level === 'ADMIN_LEVEL' 확인 → /admin 이동
   ├── level !== 'ADMIN_LEVEL' → 에러 메시지 "관리자 전용 페이지입니다"
6. Admin Dashboard (카드뷰) 렌더링
```

### 5.2 어드민 유저 관리 플로우

```
1. Admin Dashboard → "어드민 유저 관리" 카드 클릭 → /admin/admin-users
2. 어드민 사용자 목록 조회 (ADMIN_LEVEL만)
3. [생성] "새 어드민 추가" 버튼 → 모달 (이메일, 이름, 비밀번호, 역할)
   ├── SUPER_ADMIN만 생성 가능
   ├── 생성 시 mustChangePw = true 설정
4. [수정] 목록에서 사용자 클릭 → 역할 변경 / 상태 변경
5. [비밀번호 초기화] 확인 후 임시 비밀번호 생성 → 이메일 발송
6. [삭제] SUPER_ADMIN만 가능, 자기 자신 삭제 불가
```

### 5.3 파트너 유저 관리 플로우

```
1. Admin Dashboard → "파트너 유저 관리" 카드 클릭 → /admin/partner-users
2. 파트너사 드롭다운으로 필터 선택 (전체 or 특정)
3. 파트너 사용자 목록 조회 (PARTNER_LEVEL만)
4. [역할 변경] PARTNER_ADMIN ↔ PARTNER_MEMBER 전환
5. [비활성화] ACTIVE → INACTIVE 전환
6. [비밀번호 초기화] 확인 후 임시 비밀번호 생성
7. [빠른 초대] "파트너 초대" 버튼 → /admin/partner-invitations 이동
```

### 5.4 파트너 초대 플로우

```
1. Admin Dashboard → "파트너 초대" 카드 클릭 → /admin/partner-invitations
2. [새 초대 탭]
   a. 파트너사 선택 (드롭다운 검색)
   b. 이메일 입력
   c. 역할 선택 (PARTNER_ADMIN / PARTNER_MEMBER)
   d. (옵션) 추가 버튼으로 다건 입력
   e. "초대 발송" 버튼 → API 호출 → 이메일 발송
3. [초대 이력 탭]
   a. 파트너사/상태/기간 필터 적용
   b. PENDING 초대 → 재발송 / 취소 가능
   c. EXPIRED 초대 → 재발송 가능
```

## 6. 기술 제약사항

| # | 항목 | 설명 |
|---|------|------|
| 1 | **백엔드 API 호환** | `/api/v1/auth/login`에서 `entity_code` 미전달 시 ADMIN_LEVEL 자동 매칭 로직 이미 존재 → 백엔드 로그인 API 변경 불필요 |
| 2 | **JWT 쿠키** | 어드민 로그인 시 동일한 `access_token` 쿠키 사용 (기존과 동일) |
| 3 | **Auth Store 공유** | 어드민 로그인도 기존 `useAuthStore` 사용 (별도 스토어 분리 불필요) |
| 4 | **하위 호환** | 기존 `/login` 경로를 통한 어드민 로그인 플로우 유지 (코드 삭제 없음) |
| 5 | **권한 체계** | 어드민 유저 생성/역할변경/삭제는 `SUPER_ADMIN` 역할만 허용 |
| 6 | **기존 API 재사용** | 파트너 초대는 기존 `inviteMember` API 재사용 (`level_code: 'PARTNER_LEVEL'`) |
| 7 | **i18n** | 모든 UI 텍스트는 `admin` 네임스페이스 번역 키 사용 (ko/en/vi) |
