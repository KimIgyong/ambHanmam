# REQ-법인별데이터필터링 요구사항 분석서

**작성일**: 2026-02-19
**요청자**: 관리자
**대상**: 전체 메뉴 화면

## 1. 배경 및 문제

법인/소속(Entity) 기능이 나중에 구현되어, 기존 기능들은 법인/소속 구분 없이 모든 데이터를 노출하고 있음.

### 1.1 현재 문제점 (AS-IS)

| 구분 | 현황 | 문제 |
|------|------|------|
| 공지사항 | 전체 공지 노출 | 법인 A 사용자가 법인 B 공지를 볼 수 있음 |
| 할일(Todo) | 전체 할일 노출 | 다른 법인 소속 할일이 혼재됨 |
| AI 대화 | 전체 대화 노출 | 법인 구분 없이 모든 대화 노출 |
| 회의록 | 전체 회의록 노출 | 다른 법인 회의록 접근 가능 |
| 근무일정 | 전체 일정 노출 | 법인 간 일정 구분 불가 |
| 회계(계좌) | 전체 계좌 노출 | 법인별 계좌 분리 안 됨 |
| 멤버/그룹 | 전체 멤버 노출 | 다른 법인 멤버가 목록에 표시 |
| 작업항목 | 전체 작업 노출 | 법인 구분 없이 모든 작업 표시 |
| KMS 태그 | 전체 태그 노출 | 법인별 태그 구분 불가 |
| 검색 | 전체 검색 | 다른 법인 데이터도 검색됨 |

### 1.2 요구 상태 (TO-BE)

모든 메뉴 화면에서 **현재 선택된 법인 소속 데이터만** 노출되어야 함.

## 2. 요구사항 목록

### 2.1 핵심 요구사항

| ID | 요구사항 | 우선순위 |

| R-04 | 백엔드 Service에 법인별 데이터 필터링 로직 추가 | 필수 |
| R-05 | 프론트엔드 React Query 훅에 entityId 의존성 추가 | 필수 |
| R-06 | 법인 전환 시 자동 데이터 갱신 | 필수 |
| R-07 | 신규 데이터 생성 시 현재 법인 ID 자동 저장 | 필수 |

## 3. 기존 시스템 분석

### 3.1 이미 구현된 인프라

| 구성 요소 | 설명 | 위치 |
|----------|------|------|
| `EntityGuard` | X-Entity-Id 헤더 검증 → `req.entityId` 설정 | `apps/api/src/domain/hr/guard/entity.guard.ts` |
| `useEntityStore` | Zustand 기반, localStorage 영속화 | `apps/web/src/domain/hr/store/entity.store.ts` |
| `EntitySelector` | 법인 선택 드롭다운 컴포넌트 | `apps/web/src/domain/hr/components/EntitySelector.tsx` |
| `apiClient` 인터셉터 | 모든 요청에 `X-Entity-Id` 헤더 자동 주입 | `apps/web/src/lib/api-client.ts` |
| DB `synchronize: true` | 엔티티 수정만으로 스키마 자동 반영 | `apps/api/src/config/typeorm.config.ts` |

### 3.2 모범 사례 (이미 적용된 패턴)

Billing 도메인의 Partner/Invoice가 이미 법인 필터링을 적용하고 있음:

- **Controller**: `@UseGuards(EntityGuard)` + `req.entityId` 전달
- **Service**: `findAll(entityId, ...)` → `where: { entId: entityId }`
- **Hook**: `queryKey`에 `entityId` 포함 + `enabled: !!entityId`

### 3.3 대상 도메인 (11개)

| # | 도메인 | 대상 엔티티 | Controller | Service |
|---|--------|------------|------------|---------|
| 1 | 공지사항 | NoticeEntity | NoticeController | NoticeService |
| 2 | 할일 | TodoEntity | TodoController | TodoService |
| 3 | AI 대화 | ConversationEntity | ChatController | ChatService |
| 4 | 회의록 | MeetingNoteEntity | MeetingNoteController | MeetingNoteService |
| 5 | 근무일정 | WorkScheduleEntity | WorkScheduleController | WorkScheduleService |
| 6 | 회계 | BankAccountEntity | AccountController | AccountingService |
| 7 | 멤버 | (UserEntity 기반) | MemberController | MemberService |
| 8 | 그룹 | GroupEntity | GroupController | GroupService |
| 9 | 작업항목 | WorkItemEntity | WorkItemController | WorkItemService |
| 10 | KMS | (기존 entId) | KmsController, TagCloudController | KmsTagService |
| 11 | 검색 | (기존 EntityGuard) | SearchController | SearchService |

### 3.4 관련 데이터 모델

```
HrEntityEntity (amb_hr_entities) ── 법인 마스터
  │
  ├─ 1:N → NoticeEntity.ent_id
  ├─ 1:N → TodoEntity.ent_id
  ├─ 1:N → ConversationEntity.ent_id
  ├─ 1:N → MeetingNoteEntity.ent_id
  ├─ 1:N → WorkScheduleEntity.ent_id
  ├─ 1:N → BankAccountEntity.ent_id
  ├─ 1:N → WorkItemEntity.ent_id (기존)
  └─ 1:N → EntityUserRoleEntity → UserEntity (멤버 소속)
```

## 4. 갭 분석

| 레이어 | AS-IS | TO-BE | 갭 |
|--------|-------|-------|-----|
| DB 스키마 | 6개 엔티티에 ent_id 컬럼 없음 | ent_id (UUID, nullable) 추가 | 컬럼 추가 필요 |
| Controller | EntityGuard 미적용 | 11개 Controller에 적용 | Guard 추가 + req.entityId 전달 |
| Service | 법인 필터링 없음 | where 조건에 entId 추가 | 서비스 로직 수정 |
| Module | HrModule 미임포트 | EntityGuard 사용을 위한 HrModule 의존성 | 모듈 임포트 추가 |
| Frontend Hook | entityId 미포함 | queryKey에 entityId + enabled 조건 | 훅 수정 |
| UI | EntitySelector가 도메인별 배치 | 전역 헤더에 단일 배치 | 레이아웃 변경 |

## 5. 사용자 플로우

```
1. 사용자 로그인
   └→ 2. MainLayout 헤더의 EntitySelector에서 법인 선택 (자동 선택 또는 수동)
      └→ 3. useEntityStore에 currentEntity 저장 + localStorage 영속화
         └→ 4. apiClient 인터셉터가 X-Entity-Id 헤더 자동 주입
            └→ 5. 각 메뉴 접근 시 해당 법인 데이터만 조회
               └→ 6. 데이터 생성 시 현재 법인 ID 자동 저장

법인 전환 시:
   └→ useEntityStore 업데이트
      └→ React Query queryKey 변경 (entityId 포함)
         └→ 자동으로 새 법인 데이터 재조회
```

## 6. 기술 제약사항

| 항목 | 제약 |
|------|------|
| 기존 데이터 | ent_id = NULL인 기존 데이터는 법인 선택 후 조회되지 않음 (마이그레이션 필요 시 별도 처리) |
| nullable 설계 | ent_id는 nullable로 설정하여 마이그레이션 호환성 확보 |
| TypeORM synchronize | `synchronize: true` 설정으로 별도 마이그레이션 스크립트 불필요 |
| 하위 호환 | 기존 API 시그니처 변경 시 entityId는 optional 파라미터로 추가 |
