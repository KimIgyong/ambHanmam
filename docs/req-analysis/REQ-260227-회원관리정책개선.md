## 1. AS-IS 현황 분석

### 1.1 용어 체계 현황 (혼란의 근원)

현재 시스템에서 사용되는 **권한 관련 용어 5가지**가 명확한 기준 없이 혼용되고 있다.

| # | 용어 | DB 컬럼 / 테이블 | 값(예시) | 용도 | 문제점 |
|---|------|------------------|----------|------|--------|
| 1 | **Group Code** (그룹코드) | `usr_group_code` | `ADMIN_GROUP`, `USER_GROUP` | 사용자 등급 (Admin Level / User Level) | UI에서 "Level"로 표현되지만 DB는 "group_code" |
| 2 | **Role** (역할) | `usr_role` | `SUPER_ADMIN`, `ADMIN`, `MANAGER`, `MEMBER`, `VIEWER` | 시스템 전역 역할 | 초대 시 `USER` 사용, 등록 시 `MEMBER` 기본값 → 불일치 |
| 3 | **Function** (직무) | `usr_department` | `LEGAL`, `ACCOUNTING`, `HR`... (12개) | AI 에이전트 할당용 부서 코드 | UI 라벨은 "Function/직무"이나 DB는 "department" |
| 4 | **Group** (그룹) | `amb_groups` / `amb_user_groups` | `DevOPS`, `BPO`... | 메뉴 권한 필터링 + 데이터 접근 그룹 | `usr_group_code`의 "그룹"과 혼동 |
| 5 | **Department** (부서) | `amb_departments` / `amb_user_dept_roles` | `기술연구소`, `경영지원팀`... | 조직도 기반 부서 (법인별) | `usr_department`의 "부서"와 혼동 |

추가적으로 **Entity(법인)별 역할**(`amb_entity_user_roles`)과 **부서 내 역할**(`amb_user_dept_roles`)이 별도로 존재한다.

| # | 추가 역할 | 테이블 | 값(예시) | 용도 |
|---|-----------|--------|----------|------|
| 6 | **Entity Role** (법인역할) | `amb_entity_user_roles` | `STAFF`, `HR_ADMIN`, `ADMIN` | 법인별 접근 권한 |
| 7 | **Department Role** (부서역할) | `amb_user_dept_roles` | `MEMBER`, `TEAM_LEAD`, `DEPARTMENT_HEAD` | 부서 내 직책 |

### 1.2 `USER` vs `MEMBER` 역할 불일치

| 항목 | 현재 상태 |
|------|-----------|
| **공식 역할 enum** (`packages/types`) | `SUPER_ADMIN`, `ADMIN`, `MANAGER`, `MEMBER`, `VIEWER` |
| **User Entity 기본값** | `usrRole = 'MEMBER'` |
| **초대 폼 선택지** | `USER`, `MANAGER`, `ADMIN` |
| **역할 변경 모달 선택지** | `USER`, `MANAGER`, `ADMIN` |
| **RolesGuard hierarchy** | `USER: 2, MEMBER: 2` (같은 레벨로 처리) |
| **DEFAULT_PERMISSIONS** | `MEMBER`과 `USER` 별도 정의 (동일한 권한) |

**결과**: DB에 `usr_role = 'USER'`인 사용자와 `usr_role = 'MEMBER'`인 사용자가 공존. 논리적으로는 동일하나 문자열이 다름.

### 1.5 권한 시스템 구조 (3계층)

```
┌─────────────────────────────────────────────────────────┐
│                    Menu Configuration                    │
│  (ON/OFF 토글 + 정렬순서)                                │
│  → 메뉴가 OFF면 모든 사용자에게 숨김                        │
├─────────────────────────────────────────────────────────┤
│                    Role Permissions                      │
│  ┌─────────────┐    ┌─────────────────────────────────┐ │
│  │ Admin Level  │    │ User Level                      │ │
│  │ (잠금 상태)   │    │ MANAGER / MEMBER / VIEWER       │ │
│  │ SA ✓  A ✓   │    │ 역할별 메뉴 ON/OFF 설정           │ │
│  └─────────────┘    │ + 그룹별 접근 제한 (Group Filter)  │ │
│                     └─────────────────────────────────┘ │
├─────────────────────────────────────────────────────────┤
│                    User Permissions                      │
│  개별 사용자 Override (최우선 적용)                         │
│  → 역할 권한과 무관하게 특정 사용자에게 부여/차단            │
└─────────────────────────────────────────────────────────┘
```

**우선순위**: Menu OFF → 숨김 > Admin 역할 → 전체 접근 > User Override → 최우선 > Role 권한 → 기본 > Group 필터 → 제한적

### 1.6 초대(Invitation) 플로우 현황

```
초대 폼 입력:
  ├─ email (필수)
  ├─ role: USER | MANAGER | ADMIN   ← "USER"가 MEMBER과 불일치
  ├─ department: LEGAL ~ PLANNING   ← UI: "Function/직무", DB: "usr_department"  
  └─ group_id: DevOPS, BPO 등       ← 선택사항
```

**DB 저장 시**: 
- `usr_role` = 선택한 role (`USER` | `MANAGER` | `ADMIN`)
- `usr_department` = 선택한 department
- `usr_group_code` = role이 ADMIN이면 `ADMIN_GROUP`, 그 외 `USER_GROUP`
- `amb_user_groups`에 group_id 연결

## 2. TO-BE 요구사항

### 2.1 용어 통일 및 표준화

| 현재 용어 | 문제점 | 개선안 |
|-----------|--------|--------|
| `usr_group_code` = `ADMIN_GROUP`/`USER_GROUP` | "그룹"이 `amb_groups`와 혼동 | **라벨**: "Admin Level / User Level" 유지, 코드는 현행 유지 (DB 변경 최소화) |
| 초대 폼 `role` = `USER` | `MEMBER`와 불일치 | **`USER` → `MEMBER`로 통일** |
| 초대 폼 `department` = "Function/직무" | `usr_department`와 이름 혼동 | UI 라벨을 **"Function (AI Agent)"**으로 명확히 표기 |
| `amb_groups` "그룹" | `usr_group_code`와 혼동 | UI에서 **"Access Group"** 으로 구분 표기 |

### 2.2 `USER` 역할 폐기 및 `MEMBER` 통일

1. **초대 폼**: `USER` → `MEMBER`로 변경
2. **역할 변경 모달**: `USER` → `MEMBER`로 변경, `VIEWER` 옵션 추가
3. **RolesGuard**: `USER` backward-compat 유지 (기존 DB 데이터 호환)
4. **DEFAULT_PERMISSIONS**: `USER` 항목은 유지하되 주석으로 deprecated 표기
5. **기존 DB의 `usr_role = 'USER'` 데이터**: 마이그레이션으로 `MEMBER`로 일괄 변경 (선택사항)

### 2.3 SUPER_ADMIN 체크 누락 수정

모든 프론트엔드 관리자 체크를 `isAdminRole()` 헬퍼 함수로 통일:

```typescript
// 공통 헬퍼
const isAdminRole = (role: string) => role === 'SUPER_ADMIN' || role === 'ADMIN';
```

수정 대상:
- `GroupListTab.tsx`
- `NoticesPage.tsx`
- `NoticeDetailPage.tsx`
- `RoleBadge` (모든 역할에 대한 색상 맵 추가)

### 2.4 메뉴 카테고리 재분류

**SCHEDULE, PROJECT_MANAGEMENT → WORK_TOOL로 이동**:

```
// menu-config.service.ts 시드 데이터 변경
{ menuCode: 'SCHEDULE', category: 'MODULE' → 'WORK_TOOL', sortOrder: 750 }
{ menuCode: 'PROJECT_MANAGEMENT', category: 'MODULE' → 'WORK_TOOL', sortOrder: 760 }
```

**WORK_ITEMS(Issues)도 WORK_TOOL로 확인** — 현재 `WORK_ITEMS`는 시드에 미포함. 추가 필요 시 `WORK_TOOL` 카테고리로 시드.

### 2.5 CHAT MENUS를 Role Permissions에서 숨김

Role Permissions 탭에서 `CHAT_*` 메뉴 9개를 **필터링하여 숨김**:
- 이유: 채팅 메뉴는 별도 관리 체계 (에이전트 설정)
- 구현: 프론트엔드 `RolePermissionTab.tsx`에서 `CHAT_` prefix 메뉴 숨김

### 2.6 법인(Entity) 기반 부서/그룹 관리 강화

현재 부서와 그룹은 법인별로 구분 가능하나, UI에서 법인 선택이 명확하지 않은 경우 있음:
- **Department**: 이미 법인별 (`entityId` FK) — 현행 유지
- **Group**: `entId` nullable — 법인 없는 글로벌 그룹 허용 → **법인 필수로 변경** 검토

```

  ├─ email 입력
  ├─ Role 선택: MANAGER / MEMBER / VIEWER   ← USER 제거, VIEWER 추가
```

## 5. 기술 제약사항

1. **DB 마이그레이션**: `usr_role = 'USER'` → `'MEMBER'` 일괄 변경 시 기존 데이터 영향
2. **Backward Compatibility**: `RolesGuard`의 `USER: 2` 매핑은 유지해야 함 (전환 기간)
3. **메뉴 카테고리 변경**: DB `amb_menu_config`에 이미 시드된 데이터 → `onModuleInit`에서 upsert 또는 별도 마이그레이션
4. **CHAT 메뉴 분리**: 백엔드 DEFAULT_PERMISSIONS에는 CHAT 메뉴 권한이 존재 → 삭제하지 않고 프론트엔드에서만 숨김
