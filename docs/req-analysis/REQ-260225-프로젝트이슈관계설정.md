## 1. 요구사항 개요

프로젝트 관리와 이슈 관리 간의 연관 관계를 설정하고, 프로젝트 제안/승인 프로세스 및 이슈 타입/상태 체계를 개선한다. 또한 현재 레드마인에 등록된 프로젝트·태스크·이슈 데이터의 마이그레이션을 고려한 스키마를 설계한다.

### 1.1 핵심 요구사항 요약

| # | 요구사항 | 영역 |
|---|---------|------|
| R1 | 프로젝트 제안은 모든 사용자가 가능 | 프로젝트 권한 |
| R2 | 프로젝트 제안 승인은 MANAGER 이상만 가능 | 프로젝트 권한 |
| R3 | 프로젝트는 법인(Entity) 구분 없이 전체 노출 | 프로젝트 조회 |
| R4 | 프로젝트 리스트에서 관련 이슈 목록 확인 가능 | 프로젝트-이슈 관계 |
| R5 | 이슈 등록 시 관련 프로젝트 선택 가능 | 이슈-프로젝트 관계 |
| R6 | 이슈의 승인/진행 상태 표현 | 이슈 상태 |
| R7 | 이슈 타입: 버그/기능개선/의견/태스크/기타 | 이슈 타입 |
| R8 | 레드마인 기존 프로젝트/태스크/이슈 데이터 마이그레이션 호환 설계 | 마이그레이션 |

## 2. AS-IS 현황 분석

### 2.1 현재 프로젝트 관리 (`kms_projects`)

#### 엔티티 구조
- **테이블**: `kms_projects`
- **PK**: `pjt_id` (UUID)
- **법인 연결**: `ent_id` (필수, NOT NULL) → 법인별 격리
- **상태 흐름**: `DRAFT → SUBMITTED → REVIEW → APPROVED → IN_PROGRESS → ON_HOLD → COMPLETED | CANCELLED`
- **제안자**: `pjt_proposer_id` (UUID, 필수)
- **승인 관련**: `pjt_submitted_at`, `pjt_approved_at`, `pjt_rejected_at`

#### 프로젝트 API 현황
| 엔드포인트 | 메서드 | 설명 | 권한 |
|-----------|--------|------|------|
| `/project/projects` | GET | 목록 조회 | EntityGuard (법인 필터) |
| `/project/projects` | POST | 프로젝트 생성 | EntityGuard |
| `/project/projects/:id` | PATCH | 수정 | EntityGuard |
| `/project/projects/:id/submit` | POST | 제안 제출 | EntityGuard |
| `/project/projects/:id` | DELETE | 삭제 | EntityGuard |

#### 프로젝트 관련 문제점
1. **법인 필터링**: `findAll()`에서 `entId` 필수 → 타 법인 프로젝트 볼 수 없음
2. **권한 미분리**: 제안/승인/수정 모두 EntityGuard만 적용, 역할 기반 제어 없음
3. **이슈 연결 없음**: 프로젝트와 이슈 간 관계 테이블/컬럼 부재

### 2.2 현재 이슈 관리 (`amb_issues`)

#### 엔티티 구조
- **테이블**: `amb_issues`
- **PK**: `iss_id` (UUID)
- **타입**: `iss_type` VARCHAR(20) — 현재 `BUG | FEATURE | ENHANCEMENT`
- **상태**: `iss_status` VARCHAR(20) — 현재 `OPEN | APPROVED | IN_PROGRESS | RESOLVED | CLOSED | REJECTED`
- **프로젝트 연결**: **없음**

#### 이슈 상태 전이 (현행)
```
OPEN → APPROVED | REJECTED
APPROVED → IN_PROGRESS | REJECTED
IN_PROGRESS → RESOLVED
RESOLVED → CLOSED | OPEN (reopened)
REJECTED → OPEN (재등록)
CLOSED → (종료)
```

#### 이슈 API 현황
| 엔드포인트 | 메서드 | 설명 | 권한 |
|-----------|--------|------|------|
| `/issues` | GET | 목록 조회 | EntityGuard |
| `/issues` | POST | 생성 | EntityGuard |
| `/issues/:id` | PATCH | 수정 | EntityGuard |
| `/issues/:id/status` | PATCH | 상태 변경 | EntityGuard |
| `/issues/:id/approve` | PATCH | 승인 | RolesGuard(MANAGER) |
| `/issues/:id/reject` | PATCH | 반려 | RolesGuard(MANAGER) |
| `/issues/:id` | DELETE | 삭제 | RolesGuard(MANAGER) |

#### 이슈 관련 문제점
1. **타입 제한**: `BUG | FEATURE | ENHANCEMENT` 3종만 지원 → 의견/태스크/기타 없음
2. **프로젝트 연결 없음**: 이슈가 어떤 프로젝트에 속하는지 추적 불가
3. **상태 표현 부족**: 승인 상태와 진행 상태가 혼합 → UI에서 구분 표현 미흡

### 2.3 프론트엔드 현황

#### 프로젝트 UI
- `ProjectListPage.tsx`: 승인된 프로젝트만 표시 (`APPROVED`, `IN_PROGRESS`, `ON_HOLD`, `COMPLETED`)
- `ProposalListPage.tsx`: 제안 목록 별도 관리
- 관련 이슈 표시 UI 없음

#### 이슈 UI
- `IssuesPage.tsx`: 리스트/칸반/간트 3개 뷰 지원
- `IssueFormModal.tsx`: 타입 선택 `BUG | FEATURE | ENHANCEMENT`
- 프로젝트 선택 필드 없음
- 상태 배지는 있으나 승인/진행 상태 구분 표현 없음

### 3.2 AMB 맞춤 설계 방향

경쟁 솔루션들은 이슈가 프로젝트에 **필수 소속**되는 구조이나, AMB Management는 다부서 통합 관리 시스템이므로:

1. **선택적 연결**: 이슈 등록 시 프로젝트 연결은 **선택 사항** (프로젝트 무관 이슈도 존재)
2. **크로스 법인**: 프로젝트는 법인 구분 없이 전사 공유 (다국가 법인 간 협업)
3. **제안 민주화**: 모든 사용자가 프로젝트를 제안 가능 (Jira/Redmine에는 없는 AMB 고유 기능)
4. **승인 워크플로우**: 기존 `ProjectReview` 테이블 활용, MANAGER 이상 승인 권한

## 4. TO-BE 요구사항 상세

### 4.1 [R1] 프로젝트 제안 - 모든 사용자 가능

#### 현황
- `ProjectController.create()`: EntityGuard만 적용, 역할 제한 없음 → **이미 충족**

#### 변경 사항
- 기존 구현이 이미 요구사항을 충족하므로, **코드 변경 불필요**
- 단, UI에 "모든 구성원이 제안 가능" 안내 문구 추가 권장

### 4.2 [R2] 프로젝트 제안 승인 - MANAGER 이상

#### 현황
- `ProjectController.update()`: EntityGuard만 적용 → 상태 변경(승인/반려) 시 역할 검증 없음
- `ProjectReviewController`: 별도 리뷰 컨트롤러 존재하나, 역할 가드 미적용

#### 변경 사항

| 항목 | 내용 |
|------|------|
| **API 변경** | `update()` 메서드에서 status가 `APPROVED`/`REJECTED`로 변경될 때 `@Roles('MANAGER')` 검증 추가 |
| **대안 접근** | 승인/반려 전용 엔드포인트 분리 (`POST /:id/approve`, `POST /:id/reject`)에 RolesGuard 적용 |
| **영향 범위** | `project.controller.ts`, `project.service.ts` |

#### 상태 전이 제어 매트릭스

| 상태 전이 | 필요 권한 | 비고 |
|-----------|----------|------|
| DRAFT → SUBMITTED | 모든 사용자 (제안자 본인) | 기존 유지 |
| SUBMITTED → REVIEW | MANAGER 이상 | 검토 시작 |
| REVIEW → APPROVED | MANAGER 이상 | **신규 권한 적용** |
| REVIEW → REJECTED | MANAGER 이상 | **신규 권한 적용** |
| APPROVED → IN_PROGRESS | MANAGER 이상 | 실행 개시 |
| 기타 전이 | MANAGER 이상 | 안전 장치 |

### 4.6 [R6] 이슈 승인/진행 상태 표현 개선

#### 현황
- 이슈 상태: `OPEN → APPROVED → IN_PROGRESS → RESOLVED → CLOSED | REJECTED`
- UI에서 상태 배지로 표시되나, **승인 단계**와 **작업 진행 단계**의 구분이 불명확

#### 변경 사항

##### 4.6.1 상태 2트랙 표현 (Jira 방식 참고)

이슈 상태를 **승인 상태(Approval Track)**와 **진행 상태(Progress Track)**로 시각적 구분:

| 단계 | 상태 | 트랙 | 배지 색상 |
|------|------|------|----------|
| 등록 | OPEN | 승인 대기 | 🟡 Yellow |
| 승인됨 | APPROVED | 승인 완료 | 🟢 Green |
| 반려됨 | REJECTED | 승인 반려 | 🔴 Red |
| 진행중 | IN_PROGRESS | 작업 진행 | 🔵 Blue |
| 해결됨 | RESOLVED | 작업 완료 | 🟣 Purple |
| 종료 | CLOSED | 최종 종료 | ⚫ Gray |

##### 4.6.2 UI 개선
- `IssueListView`, `IssueKanbanCard`: 상태 배지에 트랙 구분 아이콘/색상 적용
- `IssueDetailModal`: 상태 변경 이력(`IssueStatusHistory`)에 트랙 구분 표시
- 칸반 보드: 승인 트랙 / 진행 트랙 구분 가능한 레인 옵션

| Immediate | 5 | 1 | `CRITICAL` |

#### 10.2.3 코멘트/이력 매핑 (`journals` → `amb_issue_comments` + `amb_issue_status_logs`)

| 레드마인 필드 | AMB 대상 | 변환 규칙 |
|-------------|---------|----------|
| `journals.notes` (텍스트 있음) | `amb_issue_comments.isc_content` | 코멘트로 저장 |
| `journal_details` (status_id 변경) | `amb_issue_status_logs` | 상태 변경 이력 분리 저장 |
| `journals.user_id` | `isc_author_id` / `isl_changed_by` | 사용자 UUID 룩업 |
| `journals.created_on` | `isc_created_at` / `isl_created_at` | 타임스탬프 변환 |

#### 10.2.4 사용자 매핑 (`users` → `amb_users`)

| 전략 | 설명 |
|------|------|
| **이메일 기반 매칭** | 레드마인 `users.mail` = AMB `amb_users.usr_email` |
| **매칭 실패 시** | 마이그레이션 로그 기록, `iss_reporter_id`를 시스템 사용자로 대체 |
| **사용자 매핑 테이블** | `amb_migration_user_map(redmine_user_id, amb_user_id)` 임시 테이블 |

### 10.3 스키마 확장 (마이그레이션 호환)

레드마인 마이그레이션 지원을 위해 추가해야 할 컬럼:

```sql
-- 프로젝트: 레드마인 원본 ID + 계층 구조
ALTER TABLE kms_projects
  ADD COLUMN pjt_redmine_id INT NULL,
  ADD COLUMN pjt_parent_id UUID NULL REFERENCES kms_projects(pjt_id);

CREATE UNIQUE INDEX idx_projects_redmine_id ON kms_projects(pjt_redmine_id) WHERE pjt_redmine_id IS NOT NULL;
CREATE INDEX idx_projects_parent ON kms_projects(pjt_parent_id);

-- 이슈: 레드마인 원본 ID + 진행률 + 일정
ALTER TABLE amb_issues
  ADD COLUMN iss_redmine_id INT NULL,
  ADD COLUMN iss_start_date DATE NULL,
  ADD COLUMN iss_due_date DATE NULL,
  ADD COLUMN iss_done_ratio INT DEFAULT 0 CHECK (iss_done_ratio BETWEEN 0 AND 100);

CREATE UNIQUE INDEX idx_issues_redmine_id ON amb_issues(iss_redmine_id) WHERE iss_redmine_id IS NOT NULL;
CREATE INDEX idx_issues_dates ON amb_issues(iss_start_date, iss_due_date);

-- 마이그레이션 사용자 매핑 (임시)
CREATE TABLE amb_migration_user_map (
  redmine_user_id INT PRIMARY KEY,
  amb_user_id UUID NOT NULL REFERENCES amb_users(usr_id),
  redmine_login VARCHAR(100),
  redmine_email VARCHAR(200),
  mapped_at TIMESTAMPTZ DEFAULT NOW()
);

-- 마이그레이션 로그
CREATE TABLE amb_migration_logs (
  mgl_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  mgl_source VARCHAR(20) NOT NULL DEFAULT 'REDMINE',
  mgl_entity_type VARCHAR(20) NOT NULL, -- PROJECT | ISSUE | COMMENT | USER
  mgl_source_id INT NOT NULL,
  mgl_target_id UUID NULL,
  mgl_status VARCHAR(20) NOT NULL DEFAULT 'PENDING', -- PENDING | SUCCESS | FAILED | SKIPPED
  mgl_error_message TEXT NULL,
  mgl_created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_migration_logs_source ON amb_migration_logs(mgl_source, mgl_entity_type, mgl_source_id);
```

### 10.4 마이그레이션 프로세스 설계

#### 10.4.1 마이그레이션 순서 (의존 관계 기반)

```
1. 사용자 매핑 (users → amb_migration_user_map)
   ↓
2. 프로젝트 마이그레이션 (projects → kms_projects)
   - parent_id 계층은 2차 패스로 처리
   ↓
3. 이슈 마이그레이션 (issues → amb_issues)
   - project_id → pjt_id 룩업
   - assigned_to_id, author_id → amb_user_id 룩업
   ↓
4. 코멘트/이력 마이그레이션 (journals → amb_issue_comments + amb_issue_status_logs)
   ↓
5. 첨부파일 마이그레이션 (attachments → 파일 스토리지)
   - 선택 사항, 파일 복사 필요
   ↓
6. 검증 및 정합성 확인
```

#### 10.4.2 마이그레이션 API 설계

| 엔드포인트 | 메서드 | 설명 | 권한 |
|-----------|--------|------|------|
| `POST /admin/migration/redmine/import` | POST | 레드마인 JSON 데이터 일괄 임포트 | SUPER_ADMIN |
| `POST /admin/migration/redmine/preview` | POST | 마이그레이션 프리뷰 (dry-run) | SUPER_ADMIN |
| `GET /admin/migration/logs` | GET | 마이그레이션 로그 조회 | ADMIN+ |
| `POST /admin/migration/redmine/rollback` | POST | 마이그레이션 롤백 (redmine_id 기반 삭제) | SUPER_ADMIN |

#### 10.4.3 데이터 소스 옵션

| 방식 | 설명 | 장/단점 |
|------|------|--------|
| **Redmine REST API** | `/projects.json`, `/issues.json` 등 API 호출 | 실시간, 인증 필요, 페이징 처리 필요 |
| **DB 직접 연결** | MySQL/PostgreSQL 직접 쿼리 | 빠르고 완전하지만 DB 접근 필요 |
| **JSON 파일 임포트** | API로 추출한 JSON 파일 업로드 | 오프라인 가능, 가장 범용적 |
| **CSV 임포트** | 레드마인 CSV 내보내기 활용 | 간편하지만 관계 정보 제한적 |

> **권장**: JSON 파일 임포트 방식 (레드마인 REST API로 사전 추출)

### 10.5 마이그레이션 고려사항

#### 10.5.1 데이터 정합성

| 항목 | 대응 방안 |
|------|----------|
| **중복 방지** | `pjt_redmine_id`, `iss_redmine_id` UNIQUE 인덱스로 재실행 시 중복 방지 |
| **사용자 미매칭** | 매핑 실패 시 시스템 사용자 대체 + `amb_migration_logs`에 SKIPPED 기록 |
| **프로젝트 계층** | 2-pass 처리 (1차: 프로젝트 생성, 2차: parent_id 업데이트) |
| **타임스탬프 보존** | `iss_created_at`, `iss_updated_at`에 레드마인 원본 시간 저장 (TypeORM `@CreateDateColumn` 우회) |
| **Textile 마크업** | 레드마인 Textile 포맷 → Markdown 또는 Plain Text 변환 |
| **커스텀 필드** | `iss_ai_analysis` 컬럼에 JSON으로 일괄 저장 또는 별도 매핑 규칙 정의 |

#### 10.5.2 이슈 시작/마감일 확장

레드마인 이슈는 `start_date`, `due_date`, `done_ratio`를 지원하므로, AMB 이슈에도 해당 필드를 추가:

| 신규 필드 | 타입 | 용도 | 비고 |
|----------|------|------|------|
| `iss_start_date` | DATE | 이슈 시작 예정일 | 간트 차트 연동 |
| `iss_due_date` | DATE | 이슈 마감 예정일 | 간트 차트 연동 |
| `iss_done_ratio` | INT (0~100) | 진행률 | 프로젝트 진행률 집계 |

이 필드들은 마이그레이션뿐 아니라, 기존 간트 뷰(`IssueGanttView.tsx`)에서도 바로 활용 가능.

#### 10.5.3 프로젝트 계층 구조

레드마인은 프로젝트 간 parent-child 계층을 지원. AMB에도 `pjt_parent_id` 추가:

```
[AMB Platform] (parent)
  ├── [AMB Web Frontend]  (child)
  ├── [AMB API Backend]   (child)
  └── [AMB KMS Module]    (child)
```

- 프로젝트 리스트에서 트리 뷰 또는 인덴트 표시 가능
- 하위 프로젝트의 이슈를 상위 프로젝트에서 집계 가능 (향후)

#### 10.5.4 롤백 전략

| 시나리오 | 롤백 방법 |
|---------|----------|
| **전체 롤백** | `DELETE FROM amb_issues WHERE iss_redmine_id IS NOT NULL` + 프로젝트도 동일 |
| **부분 롤백** | `amb_migration_logs` 기반으로 특정 배치 롤백 |
| **타임스탬프 보호** | 마이그레이션 데이터는 `iss_created_at` < 특정 기준일로 식별 가능 |
