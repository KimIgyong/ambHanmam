# REQ-260405 포털 랜딩페이지 교체

## 1. 요구사항 요약

`temp/www-landing.html` 파일의 AMA 제품 중심 랜딩페이지 디자인을 `portal-web`의 기존 랜딩페이지(`LandingPage.tsx`)로 교체한다. 새 디자인의 시각적 요소, 레이아웃, 인터랙션을 그대로 구현해야 한다.

## 2. AS-IS 현황 분석

### 2.1 현재 랜딩페이지 (`LandingPage.tsx`)
- **파일**: `apps/portal-web/src/pages/public/LandingPage.tsx` (565줄)
- **컨셉**: Amoeba Company 전체 소개 (회사·제품 종합)
- **컬러 톤**: Orange 중심 (#F97316), Indigo primary (#6366F1)
- **구조**: 단일 모놀리식 컴포넌트, 내부 `translations` 객체로 i18n 관리 (i18next 미사용)
- **섹션**: Hero → Why Amoeba (6 feature cards) → Trust Bar (마퀴) → About → Products (4 cards) → Stats (4개) → CTA

### 2.2 레이아웃 구조
- **라우트**: `/` → `PublicLayout` → `LandingPage`
- **PublicLayout**: `CmsHeader` → `<Outlet>` → `CmsFooter` + 쿠키 배너
- **CmsHeader**: 투명→흰색 전환, 언어 드롭다운, CMS 메뉴 로드
- **CmsFooter**: CMS 설정 기반 푸터

### 2.3 스타일링
- **TailwindCSS 3**: primary color Indigo 기반
- **globals.css**: Tailwind + 커스텀 애니메이션 (amb-float, amb-pulse, amb-orbit, amb-marquee, amb-glow)
- **폰트**: Pretendard (CDN), Plus Jakarta Sans

### 2.4 i18n 구조
- **i18n.ts**: `common` 네임스페이스 단일 사용 (ko/en/vi)
- **LandingPage**: `useTranslation()` 훅 사용하지만, 실제 번역은 **내부 객체** `translations`로 직접 관리
- 3개 언어: en, ko, vi

## 3. TO-BE 요구사항 (새 디자인 from `www-landing.html`)

### 3.1 새 디자인 컨셉
- **제품 중심**: AMA (AI Management Assistant) 단일 제품에 집중
- **컬러 톤**: Navy (#08122A) + Blue (#2563EB, #3B82F6) 중심, Orange 악센트 (#F37021)
- **폰트**: Sora (헤딩), DM Sans (본문), JetBrains Mono (코드/숫자)
- **인터랙션**: 슬라이더(Pain Cards), Fade-Up 스크롤 애니메이션

### 3.4 인터랙션
- **Pain Slider**: 자동 재생 (4.2초 간격), prev/next 버튼, dot 네비, hover 시 정지
- **Fade-Up**: IntersectionObserver 기반 (threshold 0.1), delay 클래스 (.d1~.d4)
- **Nav 스크롤**: 40px 이후 더 진한 배경
- **언어 스위처**: 3버튼 (KO/EN/VI), 즉시 전환

### 3.5 다국어 번역 데이터
- HTML 내 `T` 객체에 ko/en/vi 3개 언어 완전한 번역 데이터 내장 (약 200개 키)
- `data-i18n` 속성 기반 DOM 직접 치환 방식 → **React 컴포넌트에서는 내부 객체로 관리** (기존 패턴 유지)

### 3.6 Nav와 Footer
- **Nav**: 현재 HTML에 자체 nav 포함 → portal-web에서는 **CmsHeader 재사용** 또는 대체 결정 필요
- **Footer**: HTML에 자체 footer 포함 → portal-web에서는 **CmsFooter 재사용** 또는 대체 결정 필요
  - **결정**: Nav/Footer는 `PublicLayout`에서 제공하는 `CmsHeader`/`CmsFooter`를 유지하되, 랜딩페이지는 기존 헤더/푸터를 함께 쓰는 구조 유지. LandingPage 내부에서 별도 Nav/Footer를 렌더링하지 않음.

## 6. 기술 제약사항

1. **PublicLayout 유지**: CmsHeader/CmsFooter 재사용 → LandingPage 내부에 자체 nav/footer 없음
2. **i18n 방식**: 기존 LandingPage처럼 내부 translations 객체 유지 (i18n JSON 파일 사용하지 않음)
3. **외부 라이브러리 없음**: 슬라이더는 자체 구현 (Swiper 등 추가 불필요)
4. **TailwindCSS 활용**: HTML의 인라인 스타일을 가능한 Tailwind 클래스로 변환, 불가능한 것은 인라인 스타일 허용
5. **Google Fonts**: Sora, DM Sans, JetBrains Mono를 `globals.css` 또는 `index.html`에서 로드
6. **CmsHeader anchor 조정**: 기존 `LANDING_ANCHORS`가 `['why', 'products', 'about', 'contact']` → 새 디자인에 맞게 anchor 조정 필요하지만, CmsHeader는 CMS 메뉴 기반이므로 LandingPage 자체 anchor만 변경
7. **단일 파일 유지**: 현재 LandingPage.tsx가 단일 모놀리식 → 동일 패턴 유지 (컴포넌트 분리 불필요)
8. **SVG 인라인**: HTML의 SVG 아이콘은 lucide-react 아이콘으로 대체 가능한 것은 대체, 나머지는 인라인 JSX
