# 요구사항 분석서: Today Dashboard (오늘 대시보드)

- **문서번호**: REQ-오늘대시보드-20260306
- **작성일**: 2026-03-06
- **참조 원본**: `reference/today-dashboard-requirements.md` (v1.0.0)

---

## 1. 개요

AMB Management 사용자가 로그인 후 자신과 팀/법인의 오늘 할일, 이슈, 지연 항목을 한눈에 파악할 수 있는 통합 현황 페이지.

- **경로**: `/` (로그인 후 기본 랜딩 페이지)
- **추가 경로**: `/today` (사이드바 메뉴에서도 접근 가능)
- **메뉴코드**: `TODAY`
- **카테고리**: `WORK_TOOL`
- **탭 구조**: [나의 오늘] / [셀의 오늘] / [모두의 오늘]
- **기존 DashboardPage 대체**: 현재 `/` 인덱스의 `DashboardPage`를 `TodayPage`로 교체

## 2. 현행 시스템 분석 (AS-IS)

### 2.1 데이터 모델 — 원본 문서 vs 실제 시스템 대조

| 원본 문서 참조 | 실제 시스템 | 차이점 |
|----------------|-------------|--------|
| `amb_todos` (tod_owner_id, tod_due_date, tod_status) | `amb_todos` (usr_id, tdo_due_date, tdo_status) | 컬럼 prefix가 `tdo_` (tod_ 아님), owner 컬럼은 `usr_id` |
| `amb_work_items` (wit_is_deleted) | `amb_work_items` (wit_deleted_at) | `wit_is_deleted` 없음, Soft Delete(`wit_deleted_at`) 사용 |
| `amb_departments` (dep_id, dep_parent_id) | `amb_units` (unt_id, unt_parent_id, unt_level) | 테이블명/컬럼명 다름. Level 1=유닛, 2=팀 |
| `amb_department_members` (dpm_is_primary) | `amb_user_unit_roles` (uur_is_primary) + `amb_user_cells` | 부서 매핑이 2계층 (유닛 역할 + 셀 멤버) |
| `amb_work_schedules` (wsc_type) | **미구현** | 근무 스케줄 엔티티 없음. `amb_calendars` 존재하나 용도 다름 |
| `amb_talk_channels` (chn_type='DM') | `amb_talk_channels` (chn_type='DIRECT') | DM 타입값이 `DIRECT` (DM 아님) |
| rank 숫자 (VIEWER=1, MEMBER=2...) | level + role 문자열 | 수치 rank 없음, `hasRole()`, `isMaster()` 등 헬퍼 사용 |

### 2.2 기존 API 현황 (재활용 가능)

| API | 경로 | 용도 | Today에서 활용 |
|-----|------|------|---------------|
| 할일 목록 | `GET /todos` | 개인 할일 (status, date 필터) | 나의 오늘: 오늘 마감/진행중/지연 조회 |
| 셀 할일 | `GET /todos/cell` | 소속 셀 전원 할일 | 셀의 오늘: 팀원별 집계 |
| 회사 할일 | `GET /todos/company` | 법인 전체 할일 | 모두의 오늘: 전 직원 집계 |
| 할일 수정 | `PATCH /todos/:id` | 상태 변경 포함 | 인라인 완료 체크 |
| 내 이슈 | `GET /issues/my` | 담당/보고 이슈 | 미종료 이슈 건수 |
| 이슈 목록 | `GET /issues` | 법인 이슈 목록 | 전 직원 이슈 건수 |
| DM 채널 | `POST /talk/channels/dm` | Find-or-Create DM | Talk 버튼 (이미 구현됨) |

### 2.3 기존 DashboardPage 분석 (대체 대상)

**파일**: `apps/web/src/domain/dashboard/pages/DashboardPage.tsx`

현재 `/` 인덱스에 `DashboardPage`가 렌더링되며, 이미 다음 정보를 표시 중:
- 오늘 마감 할일 (최대 5건)
- 최근 회의록 (최대 5건)
- 나의 이슈 (미종료)
- 공지사항 (최대 5건)
- 은행잔고 (ADMIN_LEVEL 전용)

**대체 전략**: DashboardPage를 TodayPage로 **교체**. 기존 DashboardPage의 회의록/공지/은행잔고 위젯은 [나의 오늘] 탭 하단에 통합하거나, 사이드바 위젯으로 분리.

### 3.3 데이터 모델 GAP — 쿼리 수정 필요

원본 문서의 SQL 쿼리를 실제 테이블/컬럼에 맞게 수정해야 합니다:

**`/today/me` 쿼리 수정 사항:**

| 원본 | 실제 |
|------|------|
| `tod_id` | `tdo_id` |
| `tod_title` | `tdo_title` |
| `tod_status` | `tdo_status` |
| `tod_due_date` | `tdo_due_date` |
| `tod_owner_id = :userId` | `usr_id = :userId` |
| `tod_deleted_at IS NULL` | `tdo_deleted_at IS NULL` |
| `tod_created_at` | `tdo_created_at` |

**`/today/all` 쿼리 수정 사항:**

| 원본 | 실제 |
|------|------|
| `amb_departments` → `dep_name` | `amb_units` → `unt_name`, `unt_level` |
| `amb_department_members` → `dpm_is_primary` | `amb_user_unit_roles` → `uur_is_primary` |
| `cell_dept.dep_name` (Level2) | `unit.unt_name WHERE unt_level = 2` |
| `unit_dept.dep_name` (Level1) | `parent_unit.unt_name WHERE unt_level = 1` |
| `amb_work_schedules` | **제외** (미구현) |
| `amb_work_items.wit_is_deleted = FALSE` | 사용 안 함 → `amb_issues.iss_deleted_at IS NULL` 기준 |
| `wit_type IN ('TODO','TASK')` | `iss_status NOT IN ('RESOLVED','CLOSED')` |

**이슈 카운트 — 원본 vs 실제 시스템:**

원본 문서는 `amb_work_items`에서 이슈를 카운트하지만, 실제 시스템의 이슈는 `amb_issues` 테이블에서 관리됩니다.

```sql
-- 원본 (work_items 기반)
SELECT COUNT(*) FROM amb_work_items wi
WHERE wi.wit_owner_id = u.usr_id
  AND wi.wit_type IN ('TODO', 'TASK')
  AND wi.wit_is_deleted = FALSE

-- 실제 (issues 기반으로 변경)
SELECT COUNT(*) FROM amb_issues i
WHERE (i.iss_reporter_id = u.usr_id OR i.iss_assignee_id = u.usr_id)
  AND i.iss_status NOT IN ('RESOLVED', 'CLOSED')
  AND i.ent_id = :entId
  AND i.iss_deleted_at IS NULL
```

## 4. TO-BE 요구사항 (실제 시스템 기준 보정)

### 4.1 탭 접근 제어

| 탭 | 접근 조건 | 구현 방법 |
|----|-----------|-----------|
| 나의 오늘 | 전체 사용자 | 기본 활성 탭 |
| 셀의 오늘 | `hasRole('MANAGER','MASTER','ADMIN','SUPER_ADMIN')` | 조건 미충족 시 탭 미노출 |
| 모두의 오늘 | `isMaster() \|\| isAdmin()` | USER_LEVEL MASTER + ADMIN_LEVEL |

### 4.2 API 설계 (실제 시스템 반영)

#### `GET /api/v1/today/me`
- **Guard**: `@Auth()` (VIEWER 이상)
- **데이터 소스**: `amb_todos` (usr_id = currentUser), `amb_issues` (reporter/assignee = currentUser)
- **응답**: todayDue[], inProgress[], overdue[], issueCount, summary

#### `GET /api/v1/today/team`
- **Guard**: `@Auth()` + MANAGER 이상 체크
- **데이터 소스**: `amb_user_unit_roles` (uur_is_primary=true) → 동일 유닛 사용자 → 할일/이슈 집계
- **참고**: 원본의 "셀(Level2 팀)" 개념은 실제 시스템에서 `amb_units.unt_level = 2` 또는 `amb_cells` + `amb_user_cells`로 대응

#### `GET /api/v1/today/all`
- **Guard**: `@Auth()` + MASTER 이상 체크
- **데이터 소스**: `amb_hr_entity_user_roles` (ent_id) → 법인 전 직원 → 할일/이슈 집계
- **제외 필드**: `workScheduleType` (amb_work_schedules 미구현)

#### `POST /api/v1/talk/channels/dm` — 기존 API 그대로 활용
- **이미 구현됨**: `channel.service.ts` → `findOrCreateDm()`
- **채널 타입**: `DIRECT` (원본의 `DM`이 아닌 실제 값)
