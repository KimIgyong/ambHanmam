# REQ-구성원관리UI개선-20260329

## 1. 요구사항 요약

`/entity-settings/members` 구성원 관리 페이지의 UI를 개선한다:

| # | 요구사항 | 설명 |
|---|---------|------|
| R1 | 초대 탭 분리 | "대기중 초대" 화면을 **초대 대기**(PENDING)와 **승인 고객**(ACCEPTED) 서브탭으로 분리 |

## 2. AS-IS 현황 분석

### 2.1 프론트엔드 구조

**파일**: `apps/web/src/domain/entity-settings/pages/EntityMemberPage.tsx` (931줄)

- **탭 구조**: `members` | `invitations` (2개 탭, 하위 구분 없음)
- **멤버 리스트**: USER_LEVEL / CLIENT_LEVEL 구분 없이 전체 통합 테이블
  - 컬럼: Email, Name, Role, Unit, Status, JoinedAt
  - Role 값은 MASTER/MANAGER/MEMBER/VIEWER 만 정의
- **초대 리스트**: PENDING/ACCEPTED/CANCELLED/EXPIRED 모두 단일 카드 리스트
- **멤버 상세 모달** (`MemberDetailModal`):
  - 읽기 전용: Email, Name, Level, JoinMethod
  - 편집 가능: Role (MASTER/MANAGER/MEMBER/VIEWER), Unit, Status
  - CLIENT_LEVEL 고객사 정보 표시 없음
- **비밀번호 초기화**: 단일 버튼 → 이메일 발송만 지원

### 2.2 백엔드 구조

**파일**: `apps/api/src/domain/entity-settings/service/entity-member.service.ts`

- `findMembers()`: `EntityUserRoleEntity` + `usrCompanyId` 기반 합집합
  - CLIENT_LEVEL 멤버는 `usrCompanyId`가 NULL일 수 있으므로 EntityUserRole에 등록된 경우만 조회됨
  - 응답에 `levelCode` 필드 포함
- `resetMemberPassword()`: 임시 비밀번호 생성 + 이메일 발송만 지원
  - `usrMustChangePw = true` 설정
  - 응답: `{ userId, emailSent: boolean }`
  - **관리자가 임시 비밀번호를 직접 확인하는 기능 없음**

### 2.3 데이터 모델

**UserEntity** (`amb_users`):
- `usr_level_code`: `ADMIN_LEVEL | USER_LEVEL | CLIENT_LEVEL | PARTNER_LEVEL`
- `usr_role`: `SUPER_ADMIN | ADMIN | MASTER | MANAGER | MEMBER | VIEWER`
- `usr_cli_id`: CLIENT_LEVEL 전용, 고객사 ID (SvcClientEntity FK)
- `usr_company_id`: USER_LEVEL의 법인 ID
- `usr_join_method`: `SEED | REGISTER | INVITE`

**SvcClientEntity** (`amb_svc_clients`):
- `cli_id`, `cli_code`, `cli_company_name`, `cli_ent_id` (속한 법인)
- CLIENT_LEVEL 사용자의 `usr_cli_id`로 연결

**InvitationEntity** (`amb_invitations`):
- `inv_level_code`: `USER_LEVEL | PARTNER_LEVEL` (CLIENT_LEVEL은 별도 client-invitation)
- `inv_status`: `PENDING | ACCEPTED | CANCELLED | EXPIRED`

### 2.4 i18n

- `entitySettings.json`: 멤버/초대 관련 키 존재
- CLIENT_LEVEL 관련 번역 키 없음
- 비밀번호 초기화: 이메일 발송 관련만 존재

## 3. TO-BE 요구사항

### 3.1 R1: 초대 탭 분리

**현재**: invitations 탭에 모든 상태 혼재 (PENDING + ACCEPTED + CANCELLED + EXPIRED)

**변경**:
```
구성원 관리
├─ Tab: 구성원 (members)
│   ├─ Sub-tab: 마스터 | 매니저 | 멤버 | 뷰어
│   └─ Sub-tab: 클라이언트-매니저 | 클라이언트-멤버 | 연결된 고객사
├─ Tab: 초대 대기 (pending)
│   └─ PENDING 상태만 표시
└─ Tab: 승인 고객 (approved)
    └─ ACCEPTED 상태만 표시
```

### 3.2 R2: 레벨별 구성원 리스트 구분

**구성원 탭 내부 서브탭 구조**:

| 서브탭 그룹 | 서브탭 | 필터 조건 |
|------------|--------|----------|
| USER_LEVEL | 마스터 | `levelCode=USER_LEVEL && role=MASTER` |
| USER_LEVEL | 매니저 | `levelCode=USER_LEVEL && role=MANAGER` |
| USER_LEVEL | 멤버 | `levelCode=USER_LEVEL && role=MEMBER` |
| USER_LEVEL | 뷰어 | `levelCode=USER_LEVEL && role=VIEWER` |
| CLIENT_LEVEL | 클라이언트-매니저 | `levelCode=CLIENT_LEVEL && role includes MANAGER` |
| CLIENT_LEVEL | 클라이언트-멤버 | `levelCode=CLIENT_LEVEL && role includes MEMBER` |
| CLIENT_LEVEL | 연결된 고객사 | 고객사 목록 (SvcClientEntity) |

### 3.3 R3: 클라이언트 상세 정보

CLIENT_LEVEL 멤버 상세 모달에 표시할 추가 정보:
- 연결된 고객사명 (`cliCompanyName`)
- 고객사 코드 (`cliCode`)
- 고객사 상태 (`cliStatus`)

### 3.4 R4: 상세/수정 정보 일치화

모달의 읽기 전용/편집 가능 필드를 일치화:
- **공통 노출**: Level, Join Method, Role, Unit, Status
- **CLIENT_LEVEL 추가**: 연결된 고객사 정보

### 3.5 R5: 비밀번호 초기화 2가지 방식

| 방식 | 설명 | 액션 |
|------|------|------|
| 이메일 발송 | 임시 비밀번호 생성 후 사용자 이메일로 발송 | 기존 방식, 변경 없음 |
| 관리자 생성 | 임시 비밀번호 생성 후 관리자에게 직접 표시 | **신규**: 화면에 임시 비밀번호 표시 (복사 가능) |

## 5. 사용자 플로우

### 5.1 구성원 관리 탭 플로우
```
1. /entity-settings/members 접근
2. 메인 탭: [구성원] [초대 대기 (N)] [승인 기록]
3. 구성원 탭 선택 시:
   3a. 서브탭 그룹1: [마스터] [매니저] [멤버] [뷰어]  ← USER_LEVEL
   3b. 구분선 |
   3c. 서브탭 그룹2: [클라이언트-매니저] [클라이언트-멤버] [연결된 고객사] ← CLIENT_LEVEL
4. 서브탭 선택 → 해당 역할/레벨 멤버만 필터 표시
5. 멤버 클릭 → 상세 모달
```

### 5.2 비밀번호 초기화 플로우
```
1. 멤버 상세 모달 하단 "비밀번호 초기화" 섹션
2. 2개 버튼 표시:
   [이메일로 발송] ← 기존 방식
   [임시 비밀번호 생성] ← 신규
3. "이메일로 발송" 클릭:
   → confirm → API 호출 → 성공 toast
4. "임시 비밀번호 생성" 클릭:
   → confirm → API 호출 → 응답에서 tempPassword 수신
   → 화면에 임시 비밀번호 표시 (코드 블록 + 복사 버튼)
   → "이 비밀번호는 이 화면을 닫으면 다시 볼 수 없습니다" 경고
```

## 6. 기술 제약사항

1. **데이터 격리**: 모든 멤버 조회는 `entityId` 기반 필터링 필수 (OwnEntityGuard)
2. **CLIENT_LEVEL 데이터**: `usr_cli_id`로 `amb_svc_clients` 조인 필요 → 백엔드 `findMembers()` 수정
3. **비밀번호 노출**: 관리자 직접 확인 방식에서 임시 비밀번호를 응답에 포함해야 하므로, HTTPS 전송 필수, 로그에 비밀번호 기록 금지
4. **TypeORM synchronize OFF**: 스테이징/프로덕션은 `synchronize: false` → 새 컬럼/테이블 추가 시 수동 SQL 필요
5. **i18n**: 3개 언어(ko/en/vi) 모두 번역 키 추가 필요
6. **기존 멤버 호환**: CLIENT_LEVEL 멤버가 현재 `EntityUserRole`에 없을 수 있으므로 `usrCliId`가 있는 사용자도 조회 대상에 포함해야 함
