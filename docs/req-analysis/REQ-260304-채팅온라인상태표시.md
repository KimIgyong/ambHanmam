## 2. TO-BE 요구사항

### 2.1 Acceptance Criteria

#### AC-1: 채팅 사용자 목록에 온라인 표시
- MemberPanel에서 각 사용자 아바타에:
  - **초록 점** = Active (온라인)
  - **회색 점** = Inactive (오프라인)

#### AC-2: 1:1 DM 채팅 헤더
- DM 채널 헤더에 상대방 상태 표시:
  - 온라인: **"Active now"**
  - 오프라인: **"Offline"**

#### AC-3: 자동 상태 전환
- 사용자 연결 해제 또는 **X분 이상 유휴** 시 → Inactive 전환

### 2.2 사용자 플로우

```
1. 사용자 A가 Amoeba Talk 페이지 진입
   → 서버에 "온라인" 하트비트 전송 시작
   → 다른 사용자에게 A의 상태가 "Active"로 업데이트

2. MemberPanel에서 멤버 목록 확인
   → 온라인 사용자: 초록 점 + 이름
   → 오프라인 사용자: 회색 점 + 이름

3. DM 채널 선택
   → 헤더에 상대방 이름 + "Active now" 또는 "Offline"

4. 사용자 A가 5분간 활동 없음 (또는 브라우저 탭 비활성)
   → 서버가 A를 "Inactive"로 전환
   → 다른 사용자 화면에서 A의 초록 점 → 회색 점

5. 사용자 A가 페이지 이탈/브라우저 닫기
   → 하트비트 중단 → 서버가 A를 "Offline"로 전환
```

## 3. 기술 분석

### 3.1 Presence 구현 방식 비교

| 방식 | 장점 | 단점 | 적합성 |
|------|------|------|--------|
| **WebSocket** | 양방향, 실시간 | 기존 SSE 아키텍처 변경 필요 | △ |
| **SSE 기반 + Heartbeat** | 기존 아키텍처 활용, 서버→클라 이벤트 | 클라→서버는 REST | ★ 권장 |
| **Polling** | 구현 간단 | 서버 부하, 지연 | △ |

### 3.2 권장 아키텍처: SSE + Heartbeat API

기존 SSE 아키텍처를 확장하여 구현:

```
[프론트엔드]                          [백엔드]
  │                                     │
  ├─ POST /talk/presence/heartbeat ──→ UserPresence 업데이트
  │   (30초마다)                        (인메모리 Map)
  │                                     │
  ├─ GET /talk/presence/status ──────→ 사용자 상태 조회
  │   (멤버 패널 진입 시)                │
  │                                     │
  └─ SSE: presence:update ←─────────── 상태 변경 이벤트 발행
      (실시간 업데이트)
```

### 3.3 핵심 설계 결정사항

1. **저장소**: 인메모리 Map (DB 불필요 — 사용자 수 적은 내부 시스템)
   - Key: `userId`, Value: `{ lastHeartbeat: Date, status: 'online' | 'offline' }`
   - 서버 재시작 시 모든 사용자 offline (허용 가능)

2. **하트비트 주기**: 30초
   - 클라이언트: 30초마다 POST `/talk/presence/heartbeat`
   - 서버: 60초 이내 하트비트 없으면 offline 전환

3. **유휴 감지**: 브라우저 `visibilitychange` + `mousemove`/`keydown`
   - 5분간 활동 없으면 하트비트 중단 → 서버에서 offline 전환

4. **SSE 이벤트**: 기존 SSE 스트림에 `presence:update` 이벤트 추가
   - 채널 멤버의 상태 변경 시 해당 채널에 이벤트 발행

## 5. 기술 제약사항

### 5.1 SSE 연결 제한
- SSE는 채널별 연결 (`/talk/channels/:channelId/events`)
- Presence는 **글로벌** 개념 → 별도 SSE 스트림 또는 기존 채널 SSE에 통합 필요
- **해결**: 전역 Presence SSE 엔드포인트 추가 (`/talk/presence/events`)

### 5.2 서버 스케일링
- 인메모리 Map은 **단일 서버**에서만 동작
- 현재 AMB Management는 단일 서버 (스테이징/프로덕션 각 1대)
- 향후 다중 서버 시 Redis 전환 필요 (현재는 불필요)

### 5.3 데이터 규모
- 내부 시스템이므로 동시 접속자 수 제한적 (50명 이하 예상)
- 인메모리 Map으로 충분한 성능

### 5.4 브라우저 호환성
- `visibilitychange` API: 모던 브라우저 전체 지원
- `EventSource` (SSE): 모던 브라우저 전체 지원
