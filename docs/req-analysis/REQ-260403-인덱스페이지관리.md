| 항목 | 내용 |
|------|------|
| **요청** | stg-ama.amoeba.site 루트(`/`) 접속 시 커스텀 인덱스 페이지(랜딩 페이지) 표시 기능 |

## 2. AS-IS 현황 분석

### 2.1 현재 라우팅 구조

**Nginx (docker/staging/nginx.conf)**:
- `location /` → `try_files $uri /index.html` (SPA fallback)
- `/api/` → `proxy_pass http://amb-api:3009`
- 현재 루트(`/`)는 SPA index.html 로드 → React Router 동작 → AuthGuard → `/login` 또는 대시보드

**React Router (apps/web/src/router/index.tsx)**:
- 인증되지 않은 사용자 → `/login` 리다이렉트
- 인증된 사용자 → `/today` (대시보드)
- **별도 랜딩/인덱스 페이지 없음**

### 2.2 기존 Site Settings

### 2.3 인증 구조
- SPA 내부: `AuthGuard` → JWT 토큰 확인 → 미인증 시 `/login`
- 백엔드: `AdminGuard` → `user.role === 'ADMIN' || 'SUPER_ADMIN'`
- 공개 API 엔드포인트: `/api/v1/open/*` (인증 불필요, 기존 패턴 존재)

## 3. TO-BE 요구사항

### 3.1 R1: 인덱스 페이지 사용/미사용 설정
- `amb_site_settings`에 `sts_index_enabled` (boolean, default: false) 컬럼 추가
- 관리자가 사용 ON → 루트(`/`) 접속 시 커스텀 HTML 페이지 표시
- 사용 OFF → 현재처럼 SPA 로드 (기존 동작)

### 3.2 R2: 인덱스 페이지 HTML 편집
- `amb_site_settings`에 `sts_index_html` (text, nullable) 컬럼 추가
- `/admin/site-settings` 페이지에 HTML 편집기 (코드 에디터 또는 textarea) 추가
- 미리보기 기능 (새 탭에서 렌더링 결과 확인)
- 저장 시 DB에 HTML 콘텐츠 저장

### 3.3 R3: 인덱스 페이지 공개 서빙
- **공개 API**: `GET /api/v1/open/index-page` → 인덱스 설정(enabled, html) 반환
- **프론트 라우팅**: SPA 최상위에 IndexPage 컴포넌트 추가
  - 앱 초기 로드 시 공개 API로 인덱스 설정 조회
  - `indexEnabled === true` + `indexHtml` 존재 → 커스텀 HTML 렌더링
  - `indexEnabled === false` → 기존 로직 (AuthGuard → 로그인/대시보드)
  - 인덱스 페이지에 "로그인" 또는 "시작하기" 버튼으로 SPA 진입 가능

### 3.4 아키텍처

```
[비로그인 사용자] → GET /
    │
    ├─ SPA 로드 → React IndexGate 컴포넌트
    │     │
    │     ├─ GET /api/v1/open/index-page
    │     │     │
    │     │     ├─ { enabled: true, html: "..." }
    │     │     │     → 커스텀 HTML 렌더링 + "로그인" 링크
    │     │     │
    │     │     └─ { enabled: false }
    │     │           → 기존 AuthGuard 로직 진행
    │     │
    │     └─ API 에러 → 기존 AuthGuard 로직 진행 (graceful fallback)
```

## 5. 사용자 플로우

### 5.1 관리자: 인덱스 페이지 설정
1. `/admin/site-settings` 접속
2. "인덱스 페이지" 섹션에서 "사용" 토글 ON
3. HTML 에디터에 랜딩 페이지 HTML 입력 (또는 기본 템플릿 사용)
4. "미리보기" 버튼 → 새 탭에서 결과 확인
5. "저장" 버튼 → DB 저장

### 5.2 일반 방문자: 인덱스 페이지 접근
1. `https://stg-ama.amoeba.site` 접속
2. SPA 로드 → IndexGate가 `/api/v1/open/index-page` 호출
3. 인덱스 활성화 → 커스텀 HTML 렌더링
4. 페이지 내 "로그인" 링크 클릭 → `/login`으로 이동

### 5.3 인덱스 비활성화 시
1. `https://stg-ama.amoeba.site` 접속
2. IndexGate → API 응답 `enabled: false`
3. 기존대로 AuthGuard 동작 → `/login` 또는 대시보드

## 6. 기술 제약사항

### 6.1 보안
- HTML 편집 시 XSS 위험 → `<script>` 태그 허용 여부 결정 필요
  - **결정**: 관리자(ADMIN) 전용 기능이므로 HTML 그대로 저장/렌더링 허용
  - `dangerouslySetInnerHTML` 사용하되, 관리자만 편집 가능
  - 공개 API에서 반환되는 HTML은 관리자가 작성한 것이므로 trust
- 공개 API는 인증 불필요 → rate limiting 고려 (기존 `/open/*` 패턴과 동일)

### 6.2 DB 마이그레이션
- 스테이징/프로덕션: TypeORM synchronize 비활성화
- 컬럼 추가 수동 SQL 필요:
  ```sql
  ALTER TABLE amb_site_settings 
    ADD COLUMN sts_index_enabled BOOLEAN DEFAULT false,
    ADD COLUMN sts_index_html TEXT;
  ```

### 6.3 멀티테넌시
- `amb_site_settings`는 `ent_id` 기반 → 법인별 인덱스 페이지 설정 가능
- **단, 공개 API에서는 어떤 법인의 설정을 반환할지** 결정 필요
  - 방안: 첫 번째 법인(기존 `resolveEntId` 패턴) 또는 별도 "마스터" 설정
  - **결정**: 기존 `resolveEntId` 패턴 활용 (첫 번째 Entity 기준)

### 6.4 캐시
- 인덱스 페이지 API는 빈번히 호출될 수 있음 → 적절한 HTTP 캐시 헤더 또는 React Query staleTime 설정
- 관리자 저장 시 프론트 캐시 무효화

### 6.5 HTML 크기 제한
- TEXT 타입 사용 (PostgreSQL 무제한), 단 DTO에서 적절한 최대 길이 검증 (예: 1MB)
