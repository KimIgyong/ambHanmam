# REQ-사이트서비스관리설정이동-20260305

## 1. 요구사항 개요

| 항목 | 내용 |
|------|------|
| **제목** | 사이트관리/서비스관리 설정 카드뷰 이동 |
| **요청일** | 2026-03-05 |
| **대상 페이지** | MainLayout 사이드바, SettingsPage, `/site/*`, `/service/*` |
| **우선순위** | 높음 |

### 요구사항 요약
1. **사이트관리**(`/site`)와 **서비스관리**(`/service`)를 좌측 사이드바에서 제거
2. 두 모듈을 ADMIN_LEVEL **설정 페이지** 카드뷰로 이동
3. **사이트관리 진입 구조 변경**: 대시보드(법인별 사이트 사용 통계) + 법인 목록(선택 시 법인별 사이트 관리)

## 2. AS-IS 현황 분석

### 2-1. 사이드바 메뉴 구조

| 메뉴 | menuCode | 카테고리 | 위치 | 접근 권한 |
|------|----------|---------|------|-----------|
| 서비스 관리 | `SERVICE_MANAGEMENT` | MODULE | 사이드바 workModules | ADMIN_LEVEL만 (ADMIN_MODULE_CODES) |
| 사이트 관리 | `SITE_MANAGEMENT` | MODULE | 사이드바 workModules | ADMIN_LEVEL만 (ADMIN_MODULE_CODES) |

**MainLayout.tsx 사이드바 필터링 로직:**
- `CHAT_`, `SETTINGS_`, `ENTITY_` prefix → 사이드바에서 제외
- `WORK_TOOL_CODES` 배열 포함 → workTools 섹션
- 나머지 → workModules 섹션 (여기에 SITE/SERVICE 표시)

**현재 문제:**
- SITE_MANAGEMENT, SERVICE_MANAGEMENT는 ADMIN_MODULE_CODES에 정의되어 ADMIN 전용이지만, 사이드바에 MODULE 카테고리로 노출됨
- 일반 사용자 사이드바에 표시되지 않으나(권한 없으므로), ADMIN 사이드바가 길어지는 원인

### 2-2. 기존 설정 페이지 카드뷰

**SettingsPage.tsx** (`/settings`):
- 16개 설정 카드를 2열 그리드로 표시
- `useMyPermissions()` 기반 권한 필터링
- `SETTINGS_CARDS` 배열 정의: `{ menuCode, titleKey, descriptionKey, icon, iconColor, bgColor, path }`
- 하단에 MySignature 섹션

**이미 존재하는 사이트 관련 카드:**
- `SETTINGS_SITE` → `/settings/site-settings` (사이트 설정 - 도메인, 로고 등 전역 설정)

### 2-3. 현재 라우트 구조

```
/service (MenuGuard 'SERVICE_MANAGEMENT')
├─ /service/dashboard
├─ /service/services, /service/services/:id
├─ /service/clients, /service/clients/:id
└─ /service/subscriptions, /service/subscriptions/:id

/site (MenuGuard 'SITE_MANAGEMENT')
├─ /site/menus
├─ /site/pages, /site/pages/:pageId
├─ /site/posts (Coming Soon)
└─ /site/subscribers (Coming Soon)
```

### 2-4. 사이트관리 법인 격리 현황

- CMS 컨트롤러 6개 모두 `req.user.entityId` (JWT 고정값) 사용
- ADMIN 사용자: JWT에 entityId 없음 → 조회 실패
- `X-Entity-Id` 헤더는 프론트엔드에서 이미 주입 중이나 백엔드가 미사용
- React Query 캐시 키에 entityId 미포함 → 법인 변경 시 stale 데이터

## 6. 기술 제약사항 및 고려사항

### 6-1. 메뉴 코드 카테고리 변경 불필요
- `SERVICE_MANAGEMENT`, `SITE_MANAGEMENT`의 backend category를 'MODULE'에서 'SETTINGS'로 변경하면 자동으로 사이드바에서 사라짐 (SETTINGS_ prefix 필터)
- **그러나 menuCode 자체는 변경하지 않음** → 기존 권한 데이터, ADMIN_MODULE_CODES 등 영향 최소화
- 프론트엔드 사이드바 필터에서만 제외 처리

### 6-2. 기존 라우트 유지
- `/service/*` 라우트: 기존 그대로 유지 (MenuGuard, ServiceLayout)
- `/site/*` 라우트: 최상위에 SiteManagementPage 추가, 하위 라우트는 그대로 유지
- `/settings/site-settings`: 기존 사이트 설정 카드와 구분 (도메인/로고 등 전역 설정)

### 6-3. 사이트 대시보드 통계
- 법인별 CMS 통계 조회 API 필요: 각 법인의 메뉴/페이지/게시글/구독자 수
- ADMIN은 모든 법인 통계 조회, MASTER는 소속 법인만
- 기존 CMS 엔티티(CmsMenuEntity, CmsPageEntity, CmsPostEntity, CmsSubscriberEntity)에서 `entId` 기준 group by count

### 6-4. 법인 전환 시 사이트 진입
- 사이트 대시보드에서 법인 클릭 → EntitySelector의 currentEntity를 해당 법인으로 변경 → `/site/menus`로 이동
- 또는 `/site/menus?entityId={id}` 형태로 직접 이동 (SiteLayout에서 EntitySelector 자동 동기화)

### 6-5. 기존 PLAN과의 관계
- `PLAN-사이트메뉴법인별관리-작업계획-20260305.md`의 백엔드 수정(resolveEntityId 적용, React Query entityId 추가)은 여전히 필요
- 본 요구사항은 **진입점 변경**에 해당하며, 법인 격리 백엔드 로직과 병행 진행
