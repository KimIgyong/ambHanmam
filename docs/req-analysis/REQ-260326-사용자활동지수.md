## 2. TO-BE 요구사항 구체화

### 2.1 콘텐츠 별 평점 확장

#### 2.1.1 평점 적용 대상

| 콘텐츠 | 평점 대상 | 작성자 필드 | 비고 |
|---------|----------|------------|------|
| **이슈** | 이슈 본문 | `iss_reporter_id` | 이슈 보고자 = 작성자 |
| **노트/회의록** | 노트 본문 | `usr_id` | ✅ 이미 구현됨 |
| **이슈 코멘트** | 코멘트 | `isc_author_id` | 이슈에 달린 코멘트 |
| **Todo 코멘트** | 코멘트 | `tcm_author_id` | 태스크에 달린 코멘트 |
| **회의록 코멘트** | 코멘트 | `mnc_author_id` | 회의록에 달린 코멘트 |
| **태스크(Todo)** | 태스크 본문 | `usr_id` | Todo 작성자 |

> **대화방 채팅은 별 평점 비적용** — 이미 이모지 반응이 있고, 실시간 메시지 하나하나에 별점은 UX 상 부자연스러움. 대신 이모지 반응 횟수를 호응도에 반영.

#### 2.1.2 평점 규칙

- **별점 범위**: 1~5점 (0.5 단위 불가, 정수만)
- **자기 평가 금지**: 콘텐츠 최초 작성자는 해당 콘텐츠에 평가 불가 (API + UI 모두 차단)
- **1인 1평가**: 동일 콘텐츠에 대해 사용자당 1개 평가만 (변경/삭제 가능)
- **표시 정보**: 평균 별점, 평가 참여자 수, 내 평점

#### 2.1.3 새 테이블 설계

```
(A) amb_issue_ratings (isr_)
    - isr_id (PK, UUID)
    - iss_id (FK → amb_issues)
    - usr_id (FK → amb_users, 평가자)
    - isr_rating (smallint, CHECK 1~5)
    - isr_created_at, isr_updated_at
    - UNIQUE(iss_id, usr_id)

(B) amb_issue_comment_ratings (icr_ 충돌 → isrc_)
    → 검토 결과: 이슈 코멘트에는 이미 이모지 반응(IssueCommentReaction)이 있으므로,
      별도 별점보다는 이모지 반응을 호응도 계산에 활용하는 것이 효율적.
    → 코멘트 별 평점은 "코멘트 퀄리티" 측정 목적이므로, 별도 테이블 생성.

    amb_comment_ratings (cmr_)  ← 통합 코멘트 평점 테이블
    - cmr_id (PK, UUID)
    - cmr_target_type (varchar: 'ISSUE_COMMENT' | 'TODO_COMMENT' | 'NOTE_COMMENT')
    - cmr_target_id (UUID, 대상 코멘트 ID)
    - usr_id (FK → amb_users, 평가자)
    - cmr_rating (smallint, CHECK 1~5)
    - cmr_created_at, cmr_updated_at
    - UNIQUE(cmr_target_type, cmr_target_id, usr_id)

(C) amb_todo_ratings (tdr_)
    - tdr_id (PK, UUID)
    - tdo_id (FK → amb_todos)
    - usr_id (FK → amb_users, 평가자)
    - tdr_rating (smallint, CHECK 1~5)
    - tdr_created_at, tdr_updated_at
    - UNIQUE(tdo_id, usr_id)
```

> **설계 결정 — 통합 vs 개별 테이블**:
> - 이슈, 태스크, 노트(기존)는 **각각 개별 테이블** (FK 참조 무결성 유지)
> - 코멘트 3종은 **통합 테이블**(amb_comment_ratings)로 운영 (구조 동일, 관리 편의)

### 2.2 사용자 활동 지수 (Activity Index)

#### 2.2.1 추적 항목 및 기본 가중치

| 항목 | 설명 | 기본 가중치 | 카운트 단위 |
|------|------|------------|------------|
| 이슈 작성 | `amb_issues` 생성 | **5** | 건 |
| 노트/회의록 작성 | `amb_meeting_notes` 생성 | **4** | 건 |
| 코멘트 작성 | 이슈+Todo+노트 코멘트 합산 | **4** | 건 |
| 태스크 작성 | `amb_todos` 생성 | **2** | 건 |
| 대화방 채팅 | `amb_talk_messages` 전송 | **1** | 건 |

**일일 활동 지수 = Σ (항목별 건수 × 가중치)**

#### 2.2.2 가중치 설정 테이블

```
amb_activity_weight_configs (awc_)
- awc_id (PK, UUID)
- ent_id (FK → amb_entities, 법인별 설정)
- awc_category ('ISSUE' | 'MEETING_NOTE' | 'COMMENT' | 'TODO' | 'CHAT_MESSAGE')
- awc_weight (smallint, 1~10, DEFAULT는 위 기본값)
- awc_engagement_weight (smallint, 1~10, 호응도 가중치 - 아래 2.3절)
- awc_updated_by (FK → amb_users, 수정한 관리자)
- awc_created_at, awc_updated_at
- UNIQUE(ent_id, awc_category)
```

#### 2.2.3 일일 활동 스냅샷 테이블

```
amb_daily_activity_stats (das_)
- das_id (PK, UUID)
- ent_id (FK → amb_entities)
- usr_id (FK → amb_users)
- das_date (date, 집계 날짜)
- das_issue_count (int, 이슈 작성 수)
- das_note_count (int, 노트/회의록 작성 수)
- das_comment_count (int, 코멘트 작성 수)
- das_todo_count (int, 태스크 작성 수)
- das_chat_count (int, 채팅 메시지 수)
- das_activity_score (decimal, 가중치 적용 활동 지수)
- das_rating_sum (decimal, 받은 별점 합계)
- das_rating_count (int, 받은 별점 수)
- das_engagement_score (decimal, 가중치 적용 호응도 지수)
- das_total_score (decimal, 활동 + 호응도 합산)
- das_created_at
- UNIQUE(ent_id, usr_id, das_date)
```

> **집계 방식**: 매일 자정(cron) 또는 온디맨드 집계. 당일 지수는 실시간 계산, 과거는 스냅샷 조회.

### 2.3 호응도 지수 (Engagement/Appreciation Index)

#### 2.3.1 호응도 구성 요소

| 항목 | 측정 값 | 호응도 기본 가중치 |
|------|--------|-------------------|
| 이슈 별점 | 별점 합계 / 평가자 수 | **5** |
| 노트/회의록 별점 | 별점 합계 / 평가자 수 | **4** |
| 코멘트 별점 | 별점 합계 / 평가자 수 | **4** |
| 태스크 별점 | 별점 합계 / 평가자 수 | **2** |
| 대화방 이모지 반응 | 이모지 반응 합계 / 반응자 수 | **1** |

**일일 호응도 지수 = Σ ((항목별 평균별점 × 평가자수) × 호응도 가중치)**

> 단순 평균만 하면 평가자가 1명인 5점이 10명에게 4.5점 받은 것보다 높게 나옴. 
> **가중 공식**: `(별점합계) × 가중치` 로 평가 참여 규모 반영.

#### 2.3.2 호응도 계산 상세

```
해당 날짜 기준, 사용자 X가 작성한 모든 콘텐츠에 대해:

Category별 호응도 = (해당 카테고리 콘텐츠들의 받은 별점 총합) × 해당 카테고리 호응도 가중치

일일 호응도 지수 = Σ category별_호응도

예시)
- 이슈 2건 작성 → 별점 총합 (4+5+3) = 12, 평가자 3명 → 호응도: 12 × 5 = 60
- 코멘트 5건 작성 → 별점 총합 (5+4) = 9, 평가자 2명 → 호응도: 9 × 4 = 36
- 일일 호응도 지수 = 60 + 36 = 96
```

## 4. 사용자 플로우

### 4.1 콘텐츠 평가 플로우

```
[사용자 A] → 이슈 상세 페이지 열기
  → 이슈 작성자 = 사용자 A인 경우: 별점 UI 비활성화 (안내 툴팁)
  → 이슈 작성자 ≠ 사용자 A인 경우: ★★★★★ 클릭으로 평가
    → PUT /api/v1/issues/{id}/rating { rating: 4 }
    → 응답: { avgRating: 4.2, ratingCount: 5, myRating: 4 }
    → UI 갱신: 평균 별점 + 평가자 수 표시
```

### 4.2 관리자 가중치 설정 플로우

```
[법인 관리자] → 법인 설정 > 사용자 활동지수 관리
  → 탭 1: 가중치 설정
    → 이슈 활동 가중치: 5 → 7로 변경
    → [저장] 클릭
    → PUT /api/v1/entity-settings/activity-weights { weights: [...] }
    → 이후 집계부터 새 가중치 적용
```

### 4.3 활동 통계 조회 플로우

```
[법인 관리자] → 법인 설정 > 사용자 활동지수 관리
  → 탭 2: 직원별 활동 통계
    → 기간: 2026-03-01 ~ 2026-03-26
    → 부서: 전체
    → 정렬: 종합지수 높은순
    → 테이블 표시: 이름, 각 항목 카운트, 활동지수, 호응도지수, 종합지수
    → 홍길동 클릭 → 개인 상세 (일별 추이 차트 + 카테고리별 브레이크다운)
```

## 5. 기술 제약사항 및 고려사항

### 5.1 성능
- **일일 스냅샷 집계**: Cron Job (매일 자정)으로 전일 데이터 배치 집계
- **당일 실시간 지수**: 쿼리 기반 계산 (인덱스 최적화 필수)
- **인덱스**: `(ent_id, usr_id, created_at)` 복합 인덱스 필요 (각 콘텐츠 테이블)

### 5.2 데이터 정합성
- 콘텐츠 삭제(soft delete) 시 해당 평점 처리: 평점 유지 (삭제된 콘텐츠도 과거 기여도로 인정)
- 사용자 탈퇴 시: 평점 데이터 익명화 (통계에는 반영, 개인 식별 불가)

### 5.3 스테이징/프로덕션 DB 마이그레이션
- TypeORM `synchronize: false` (staging/production)
- 4개 신규 테이블 + 인덱스: **수동 SQL 마이그레이션 필수**
- 배포 순서: SQL 먼저 실행 → 코드 배포

### 5.4 기존 미팅노트 평점과의 호환
- `amb_meeting_note_ratings`는 그대로 유지 (마이그레이션 불필요)
- 자기 평가 방지 로직만 추가 (`upsertRating`에서 작성자 체크)
- 호응도 지수 집계 시 기존 데이터도 포함

### 5.5 i18n
- 새 네임스페이스: `activityIndex` (가중치 설정, 통계 화면 전용)
- 기존 네임스페이스 확장: `issues`, `todo`, `meetingNotes` (각각 rating 관련 키 추가)

## 6. 추가 의견 및 제안

### 6.1 강력 권장 사항

#### (A) 대화방 채팅은 별 평점 대신 이모지 반응 카운트 활용
- 채팅 메시지에 별 평점은 UX상 과도함 (메시지 하나하나에 1~5점?)
- 이미 `TalkReactionEntity`로 이모지 반응 인프라가 존재
- **제안**: 이모지 반응 받은 횟수를 호응도 지수에 반영 (별점 1건 = 이모지 반응 1건 동등 취급, 또는 별도 환산 비율 설정)

#### (B) 자기 평가 방지의 범위 명확화
- **이슈**: `iss_reporter_id` (보고자)만 차단? `iss_assignee_id` (담당자)도 차단?
  - **제안**: 보고자만 차단 (담당자는 이슈를 작성한 것이 아니므로 평가 가능)
- **코멘트**: 코멘트 작성자만 차단 (원글 작성자는 자기 글의 코멘트에 평가 가능)

#### (C) 코멘트 평점 vs 코멘트 이모지 반응의 관계
- 현재 이슈 코멘트에는 이모지 반응이 있음
- **제안**: 코멘트에 별 평점을 새로 추가하되, 기존 이모지 반응은 그대로 유지. 호응도 계산에는 **별 평점만** 사용 (이모지는 감성 표현 용도)
- 또는: 코멘트에 별 평점 대신, 기존 이모지 반응 수를 호응도에 반영 (구현 공수 절감)

### 6.2 선택적 제안 사항

#### (D) 기간별 통계 리셋/구간 비교
- 월별/분기별 리셋 옵션 → "이번 달 활동왕", "이번 분기 호응도 1위"
- 전월 대비 증감률 표시 (↑12%, ↓5%)

#### (E) 부서(Cell)/팀 단위 집계
- 개인별뿐 아니라 Cell/부서 단위 합산 통계
- 부서 간 활동 비교 차트

#### (F) 대시보드 위젯
- 로비(Today) 페이지에 개인 활동 지수 미니 위젯 표시
- "이번 주 내 활동 지수: 85 (상위 20%)"

#### (G) 알림 연동
- 주간 활동 리포트 이메일/알림 발송 (선택적)
- "이번 주 활동 지수가 지난주 대비 30% 증가했습니다"

#### (H) 콘텐츠 품질 지표 추가 고려
- 이슈: 해결률(iss_status → CLOSED 비율)도 활동 품질 지표로 고려
- 태스크: 기한 내 완료율도 활동 품질 지표로 고려
- 현재는 "작성 건수"만 카운트하므로, 향후 "품질 가중치" 도입 가능

#### (I) 랭킹 표시 주의사항
- 과도한 랭킹 공개는 조직 문화에 부정적 영향 가능
- **제안**: 개인에게는 자기 순위만 보이고, 관리자에게만 전체 랭킹 공개
- 또는: 상위 3명만 표시 (순위 공개 범위 설정 가능)

#### (J) 채팅 메시지 스팸 방지
- 채팅 가중치가 1이어도, 의미 없는 메시지 대량 발송으로 점수 올리기 시도 가능
- **제안**: 일일 채팅 카운트 상한(cap) 설정 가능 (예: 최대 50건까지만 지수 반영)
- 가중치 설정 테이블에 `awc_daily_cap` 컬럼 추가
