### 2.3 현재 문제점

1. **툴바 접근성**: 에디터가 모달 내부에 있고 모달 본문이 `overflow-y-auto`로 스크롤됨. 내용이 길어지면 툴바가 스크롤 밖으로 밀려남 → 사용자가 서식 변경할 때마다 상단으로 스크롤 필요
2. **에디터 높이 무한 확장**: `resize-y` 속성이 있지만 maxHeight 없음. 내용이 늘어나면 모달 전체가 길어짐
3. **min-h Tailwind 동적 클래스 문제**: `[&_.ProseMirror]:min-h-[${minHeight}]` — Tailwind JIT는 동적 클래스명을 감지하지 못함. `style` prop이 아닌 클래스 문자열 보간이므로, safelist에 없으면 빌드 시 누락될 수 있음

## 3. TO-BE 요구사항

### 3.1 R1 — Sticky Toolbar

**목표**: 에디터 내용이 길어져도 서식 도구 메뉴가 항상 보이도록 함

**구현 방안**: RichTextEditor 내부 구조 변경
```
<div> (wrapper - border, flex flex-col)
  ├── <div> (toolbar - sticky top-0 z-10, border-b, bg-white) ← sticky 적용
  └── <div> (editor-body - overflow-y-auto, max-h 제한)
       └── <EditorContent>
</div>
```

- 에디터 wrapper에 `maxHeight` prop을 추가하여 에디터 전체 높이를 제한
- 에디터 본문 영역만 `overflow-y-auto`로 스크롤
- 툴바는 `sticky top-0`으로 항상 상단에 고정
- **모달 내부 배치 시**: 모달 자체 스크롤과 에디터 내부 스크롤이 중복되지 않도록 에디터에 적절한 maxHeight 설정

### 3.2 R2 — 기본 3라인 높이

**목표**: 본문 편집 에디터의 기본 최소 높이를 3라인에 해당하는 값으로 설정

- 기본값: `minHeight='150px'` (현행, 약 3라인 상당) → 유지 가능
- Todo/Mission의 `100px`은 약 2라인 → `150px`로 통일 검토
- Tailwind 동적 클래스 문제 해결: `className` 보간 대신 `style={{ minHeight }}` 사용

### 3.3 R3 — 내부 스크롤

**목표**: 에디터 높이가 maxHeight를 초과하면 에디터 내부에서만 스크롤

- `maxHeight` prop 추가 (기본값: `'300px'` — 약 12~15줄)
- 본문 편집 시: `maxHeight='300px'` ~ `'400px'`
- 에디터 본문 영역에 `overflow-y-auto` + `max-h` 적용
- `resize-y` 제거 (maxHeight 기반 자동 스크롤로 대체)

### 3.4 R4 — 코멘트 1라인 유지

**목표**: 코멘트 에디터는 기존 소형 사이즈 유지

- `minHeight='80px'`, `maxHeight` 미설정 또는 `'150px'`
- 코멘트 에디터는 sticky toolbar 미적용 (화면이 작으므로 불필요)
- 현행 동작 유지

### 3.5 R5 — 마크다운 지원 검토

#### 기술 검토 결과

| 방안 | 패키지 | 설명 | 장점 | 단점 |
|------|--------|------|------|------|
| **A. tiptap-markdown** | `tiptap-markdown` | TipTap 확장으로 MD 입출력 지원 | 기존 에디터에 자연스럽게 통합, WYSIWYG + MD 입력 모두 가능 | 추가 패키지 설치 필요, HTML↔MD 변환 시 일부 서식 손실 가능 |
| **B. Markdown 단축키** | 내장 기능 | TipTap StarterKit에 이미 마크다운 단축키 일부 내장 (`#` → Heading, `-` → 리스트 등) | 추가 설치 없음, 이미 동작 중 | 전체 마크다운 문법 미지원, 출력은 HTML |
| **C. 별도 마크다운 에디터** | `react-markdown` + 별도 textarea | 순수 마크다운 입력 모드 토글 | 완전한 마크다운 경험 | 이중 에디터 관리 복잡, UX 혼란 |

**추천**: **방안 B (현행 유지)** — TipTap StarterKit이 이미 지원하는 마크다운 단축키 활용

현재 동작하는 마크다운 단축키:
- `# ` → Heading 1, `## ` → Heading 2
- `- ` 또는 `* ` → 불릿 리스트
- `1. ` → 번호 리스트
- `[ ] ` → 체크리스트 (TaskList 확장)
- `` ` `` → 인라인 코드
- `---` → 수평선
- `> ` → 인용

**향후 확장 시**: `tiptap-markdown` 패키지를 추가하면:
- 마크다운 텍스트를 붙여넣으면 자동으로 리치텍스트로 변환
- `editor.storage.markdown.getMarkdown()`으로 마크다운 출력 가능
- 약 5KB gzip 추가 (`tiptap-markdown`)

## 6. 기술 제약사항

### 6.1 Tailwind CSS 동적 클래스

- Tailwind JIT는 `[&_.ProseMirror]:min-h-[${variable}]` 형태의 동적 문자열 보간을 인식하지 못함
- **해결**: `style` prop으로 변경 (`style={{ minHeight, maxHeight }}`)

### 6.2 모달 스크롤과 에디터 스크롤 충돌

- 모달 본문: `max-h-[75vh] overflow-y-auto` (모달 내 폼 전체 스크롤)
- 에디터: `overflow-y-auto` + `maxHeight` (에디터 내부 스크롤)
- **이중 스크롤** 발생 가능 → 모달 스크롤 중 에디터 영역 위에 커서가 있으면 에디터가 스크롤 이벤트를 먹을 수 있음
- **해결**: `overscroll-behavior: contain`으로 에디터 스크롤이 부모로 전파되지 않도록 처리

### 6.3 `resize-y` 제거

- 현재 EditorContent에 `resize-y` 적용 → 사용자가 수동으로 에디터 높이 조절 가능
- maxHeight + 내부 스크롤 적용 시 resize-y는 혼란을 줄 수 있음
- **해결**: `resize-y` 제거하고 maxHeight 기반 자동 스크롤로 대체

### 6.4 브랜치별 파편화

- RichTextEditor가 여러 브랜치 폴더에 복제되어 있음 (`inspiring-allen/`, `loving-shockley/`, `peaceful-lovelace/` 등)
- 수정은 **`apps/web/src/domain/meeting-notes/components/RichTextEditor.tsx`** (메인)만 대상
- 다른 브랜치 폴더는 별도 관리 대상

### 6.5 영향 범위

- `RichTextEditor` 컴포넌트 1개 파일만 수정
- **Props 인터페이스 변경** (`maxHeight` 추가) — 기존 사용처는 optional이므로 하위 호환성 유지
- **사용처 변경 불필요** — 기존 `minHeight` prop은 그대로 동작, `maxHeight`는 기본값 제공
