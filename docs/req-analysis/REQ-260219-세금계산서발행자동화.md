## 1. AS-IS 현황 분석

### 1-1. 현재 시스템의 세금계산서 관련 기능

현재 AMB Management 시스템에는 이미 세금계산서 발행의 **기반 데이터 구조**가 상당 부분 구축되어 있다.

#### Invoice 엔티티 (`amb_bil_invoices`)
| 필드 | 설명 | 현재 활용 |
|------|------|-----------|
| `invNumber` | 청구서 번호 | KR+KRW: `TAX-{YYYYMM}-{seq}` 자동생성 |
| `invTaxRate` | 세율 (%) | 입력 가능 (decimal 5,2) |
| `invTaxAmount` | 세금액 | 입력 가능 (decimal 15,2) |
| `invSubtotal` | 공급가액 | 입력 가능 |
| `invTotal` | 합계 (공급가액 + 세액) | 자동 계산 |
| `invTaxInvoiceType` | `HOMETAX` / `SYSTEM` | **존재하지만 실제 홈택스 연동 없음** |
| `invServicePeriodStart/End` | 용역 기간 | 입력 가능 |
| `invDate` | 작성일자 | 입력 가능 |

#### Partner 엔티티 (`amb_bil_partners`)
| 필드 | 설명 | 현재 활용 |
|------|------|-----------|
| `ptnTaxId` | 사업자등록번호 | 입력 가능 (검증 로직 없음) |
| `ptnBizType` | 업태 | 입력 가능 |
| `ptnBizCategory` | 종목 | 입력 가능 |
| `ptnCeoName` | 대표자명 | 입력 가능 |
| `ptnCompanyName` | 상호 | 입력 가능 |
| `ptnAddress` | 주소 | 입력 가능 |
| `ptnContactEmail` | 이메일 | 입력 가능 |

#### 기존 워크플로우
```
계약 → 청구서 생성(수동/자동) → 승인 워크플로우 → PDF 생성 → 이메일 발송
         (DRAFT)              (4단계 승인)        (한국 세금계산서 형식)
```

#### Invoice PDF 생성 (`InvoicePdfService`)
- KR+KRW 전용 세금계산서 형식 PDF 이미 구현
- 사업자번호, 대표자명, 업태/종목 표시
- 승인 완료 시 도장/서명 이미지 삽입

### 1-2. 현재 부족한 부분

| 영역 | 현황 | 필요사항 |
|------|------|----------|
| **국세청 전송** | 미구현 | ASP 연동을 통한 국세청 자동 전송 |
| **전자서명** | 미구현 | 공동인증서 기반 전자서명 (ASP 위임) |
| **국세청승인번호** | 미저장 | 국세청 승인번호 수신 및 저장 |
| **발행 상태 추적** | `invTaxInvoiceType`만 존재 | 국세청 전송 상태, 승인번호, 전송일시 등 |
| **사업자번호 검증** | 미구현 | 10자리 형식 검증 + 국세청 진위확인 |
| **수정세금계산서** | 미구현 | 6가지 수정사유에 따른 수정발행 |
| **역발행** | 미구현 | 공급받는자가 요청하는 역발행 프로세스 |
| **매입세금계산서 수집** | 미구현 | 홈택스 매입자료 자동 수집 |

## 2. 한국 전자세금계산서 제도 요약

### 2-1. 법적 요건

- **근거법:** 부가가치세법 제32조
- **의무대상:** 모든 법인사업자 (개인사업자는 공급가액 8,000만원 이상)
- **발행 채널:** 홈택스 직접 / ASP(중개사업자) API / ERP 연동
- **국세청 직접 발행 API는 제공하지 않음** → 반드시 ASP 경유 필요

### 2-2. 필수 기재사항

| 구분 | 필수 항목 |
|------|-----------|
| **공급자** | 사업자등록번호, 상호, 대표자명, 주소, 업태, 종목 |
| **공급받는자** | 사업자등록번호, 상호, 대표자명 |
| **금액** | 공급가액, 세액, 합계금액 |
| **일자** | 작성일자, 공급일자 |
| **품목** | 품명, 규격, 수량, 단가, 공급가액, 세액 (최대 99건) |

### 2-3. 기한 및 가산세

| 행위 | 기한 | 가산세 |
|------|------|--------|
| 정상 발급 | 공급월 익월 10일까지 | - |
| 지연 발급 | 익월 10일 초과 ~ 25일 | 공급가액의 **1%** |
| 미발급 | 익월 25일 초과 | 공급가액의 **2%** |
| 국세청 지연전송 | 발급일 다음날 초과 | 공급가액의 **0.3%** |
| 국세청 미전송 | 과세기간 말 익월 11일 초과 | 공급가액의 **0.5%** |

### 2-4. 공동인증서 요건

- **전자세금용 인증서** 필요 (연 약 4,400원) 또는 범용 인증서 (연 약 110,000원)
- ASP 서버에 1회 등록 → 이후 API 호출 시 ASP가 자동 서명 처리
- 매 호출마다 인증서를 직접 처리할 필요 없음

### 3-2. 추천 순위

#### 1순위: 팝빌 (Popbill) - **권장**
- **장점:** 공식 Node.js SDK (`npm install popbill`), 가장 풍부한 문서/샘플, 높은 시장 점유율, 안정적 서비스 이력
- **단점:** 건당 200원 (볼타 대비 2.2배)
- **연동 기간:** 약 3~5일
- **API 문서:** https://developers.popbill.com/reference/taxinvoice/node/api

#### 2순위: 볼타 (Bolta) - 비용 최적화 시
- **장점:** 최저 단가 (90원/건), 현대적 RESTful API, 깔끔한 문서
- **단점:** SDK 없음 (REST 직접 호출), 상대적 신생 서비스
- **연동 기간:** 약 1~3일
- **API 문서:** https://docs.bolta.io/

#### 3순위: 바로빌 (Barobill) - 비추천
- **단점:** SOAP 방식으로 Node.js/TypeScript 환경에 부적합, 공식 SDK 없음
- 금융/공공기관 대상에는 적합하나 본 프로젝트에는 부적절

## 6. 구현 단계 제안

### Phase 1: 기반 구축 (1~2주)

| 순서 | 작업 | 설명 |
|------|------|------|
| 1 | ASP 계정 개설 | 팝빌 회원가입 + 테스트 환경 연동 ID/Secret 발급 |
| 2 | 인증서 준비 | 전자세금용 공동인증서 발급 + ASP 서버 등록 |
| 3 | DB 스키마 확장 | Invoice 엔티티에 NTS 관련 필드 추가 (마이그레이션) |
| 4 | ASP 어댑터 구현 | `PopbillAdapter` - SDK 래핑 서비스 |
| 5 | 환경 변수 추가 | `POPBILL_LINK_ID`, `POPBILL_SECRET_KEY`, `POPBILL_IS_TEST` |

### Phase 2: 정발행 구현 (1주)

| 순서 | 작업 | 설명 |
|------|------|------|
| 1 | `TaxInvoiceService` 구현 | Invoice → 팝빌 TaxInvoice 데이터 매핑 + 발행 |
| 2 | 승인 후 발행 연동 | 관리자 최종 승인(APPROVED_ADMIN) 시 자동 or 수동 발행 |
| 3 | Webhook 엔드포인트 | 팝빌 → 우리 시스템으로 상태 변경 수신 |
| 4 | 프론트엔드 UI | 발행 버튼, 상태 뱃지, 국세청승인번호 표시 |

### Phase 3: 확장 기능 (1~2주)

| 순서 | 작업 | 설명 |
|------|------|------|
| 1 | 수정세금계산서 | 6가지 수정사유 처리 (기재사항착오, 공급가액변동 등) |
| 2 | 역발행 프로세스 | 공급받는자 역발행 요청 → 승인 → 발행 |
| 3 | 사업자번호 검증 | 팝빌 API로 사업자 진위확인 |
| 4 | 설정 페이지 | ASP 연결 상태, 인증서 만료일 표시 |

### Phase 4: 고도화 (선택)

| 순서 | 작업 | 설명 |
|------|------|------|
| 1 | 매입세금계산서 수집 | 홈택스 매입자료 자동 수집 |
| 2 | 대량 발행 | 다건 일괄 발행 기능 |
| 3 | 발행 기한 알림 | 익월 10일 전 미발행 건 자동 알림 |
| 4 | 세금계산서 대장 | 기간별 발행 현황, 국세청 전송 현황 리포트 |

| 테스트 환경 한계 | 국세청 실제 전송 테스트 불가 | ASP 테스트 환경에서 발행까지만 검증 |
| 팝빌 SDK 콜백 방식 | Promise 미지원 (콜백 기반) | `promisify` 래퍼 또는 직접 Promise 구현 |

### 8-2. 비즈니스 리스크

| 리스크 | 확률 | 영향 | 대응 |
|--------|------|------|------|
| 발행 기한 초과 | 중 | 가산세 (1~2%) | 자동 알림 + 대시보드 |
| ASP 서비스 장애 | 저 | 일시적 발행 불가 | 수동 홈택스 발행 fallback |
| 인증서 만료 미인지 | 중 | 발행 불가 | 만료 30일 전 알림 |

## 9. 결론 및 권고

### 권고사항

1. **ASP 선정: 팝빌 (1순위)**
   - 공식 Node.js SDK 지원으로 NestJS 환경에 최적
   - 풍부한 레퍼런스와 안정적 서비스 이력
   - 건당 200원은 법인 규모에서 부담 없는 수준

2. **구현 우선순위: Phase 1~2 먼저 진행**
   - 정발행(매출 세금계산서) 자동화가 가장 시급
   - 수정/역발행은 Phase 3에서 점진적 확장

3. **기존 시스템 활용 극대화**
   - Invoice, Partner 엔티티에 세금계산서 필수 데이터 대부분 이미 존재
   - 승인 워크플로우(4단계)를 세금계산서 발행 트리거로 활용
   - `InvoiceNumberingService`의 KR+KRW 번호 체계 재활용

4. **사전 준비 사항**
   - [ ] 팝빌 회원가입 + 연동 ID 발급
   - [ ] 전자세금용 공동인증서 발급
   - [ ] 테스트 환경 연동 확인
