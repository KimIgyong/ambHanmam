### 1.2 현재 문제점

| # | 문제 | 상세 |
|---|------|------|
| 1 | **섹션 구분 불명확** | "기한 초과"와 "오늘의 할일" 2개 섹션으로만 구분. COMPLETED/IN_PROGRESS/SCHEDULED가 "오늘의 할일"에 혼재 |
| 2 | **수정 진입 불편** | 제목 클릭 불가, 우측 연필(Pencil) 아이콘 클릭 필요 → 모달 열림 |

## 2. TO-BE 요구사항

### 요구사항 #1: 3칸 레이아웃 변경 (완료 | 오늘 할일 | 예정)

**현재**: 기한 초과 / 오늘의 할일 (2섹션, 수직 나열)
**변경**: 완료 / 오늘 할일 / 예정 (3컬럼, 칸반 보드 스타일)

```
┌────────────────────────────────────────────────────────┐
│  오늘의 할일                              [+ 할일 추가] │
│  [나의 할일] [내 그룹] [회사]                           │
│  ──────────────────────────────────────────────────── │
│                                                        │
│  ┌──완료──────┐ ┌──오늘 할일────┐ ┌──예정──────────┐  │
│  │ ✓ 할일A    │ │ ◑ 할일C      │ │ ○ 할일E        │  │
│  │ ✓ 할일B    │ │ ○ 할일D      │ │ ○ 할일F        │  │
│  │            │ │ ⚠ 기한초과G  │ │                │  │
│  └────────────┘ └──────────────┘ └────────────────┘  │
└────────────────────────────────────────────────────────┘
```

**분류 기준**:
- **완료**: `status === 'COMPLETED'`
- **오늘 할일**: `status !== 'COMPLETED' && dueDate <= today` (진행 중 + 기한 초과 포함)
- **예정**: `status !== 'COMPLETED' && dueDate > today` (미래 마감일)

**기한 초과 표시**: "오늘 할일" 칸 내에서 `dueDate < today`인 항목은 빨간 테두리로 표시 유지

### 요구사항 #2: 제목 클릭 → 바로 수정화면

**현재**: 제목은 텍스트만 표시, 수정은 연필 아이콘 → TodoFormModal
**변경**: 제목 클릭 시 바로 TodoFormModal(수정 모드) 오픈

- 제목에 `cursor-pointer` + `hover:underline` 스타일 적용
- 클릭 시 `onEdit(todo)` 호출 (기존 연필 아이콘과 동일 동작)
- 연필 아이콘 제거 (불필요)

### 요구사항 #3: 관련자 등록 기능

**신규 DB 테이블**: `amb_todo_participants`

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `tpt_id` | uuid (PK) | 참여자 레코드 ID |
| `tdo_id` | uuid (FK) | 할일 ID |
| `usr_id` | uuid (FK) | 사용자 ID |
| `tpt_role` | varchar(20) | 역할: `ASSIGNEE`(담당자), `WATCHER`(참조자) |
| `tpt_created_at` | timestamp | 등록일시 |

**UI**:
- TodoFormModal에 "관련자" 필드 추가
- 같은 법인 사용자 목록에서 선택 (멀티 셀렉트)
- 역할 지정: 담당자 / 참조자
- TodoItem에 관련자 아바타(이니셜) 표시

**비즈니스 규칙**:
- 할일 작성자는 자동으로 소유자(기존 `usr_id`)
- 관련자는 해당 할일을 자신의 "나의 할일"에서도 볼 수 있음 (읽기 전용 또는 ASSIGNEE는 상태 변경 가능)
- 관련자 등록/삭제는 소유자만 가능

### 요구사항 #4: 코멘트 알림 → AI 에이전트 알림

**현재**: 코멘트 작성 시 DB 저장만, 작성자에게 알림 없음
**변경**: 내 할일에 코멘트가 달리면 AI 에이전트가 채팅으로 알림

**알림 흐름**:
```
1. 타인이 내 할일에 코멘트 작성
   ↓
2. TodoService.addTodoComment() 완료 후 이벤트 발행
   ↓
3. EventEmitter2: 'todo.comment.created' 이벤트
   ↓
4. TodoNotificationListener 감지
   ↓
5. AI 에이전트를 통해 할일 소유자에게 알림 메시지 전송
   (예: "홍길동님이 '서버 배포 준비' 할일에 코멘트를 남겼습니다: ...")
```

**알림 채널**:
- **1차 구현**: Amoeba Talk SSE 기반 실시간 알림 (TalkSseService 활용)
- **향후 확장**: 이메일, 푸시 알림

## 4. 사용자 플로우

### 4.1 할일 생성 (관련자 포함)
```
사용자 → [+ 할일 추가] → TodoFormModal 열림
  → 제목, 설명, 마감일 입력
  → [관련자 추가] 클릭 → 사용자 검색/선택 → 역할 지정(담당자/참조자)
  → [저장] → 할일 생성 + 관련자 등록 + 번역 다이얼로그
```

### 4.2 할일 수정 (제목 클릭)
```
사용자 → 할일 제목 클릭 → TodoFormModal(수정 모드) 열림
  → 필드 수정 → [저장]
```

### 4.3 코멘트 알림 수신
```
타인 → 내 할일 상세 → 코멘트 입력 → [등록]
  → 시스템: 이벤트 발행 → AI 에이전트 알림 생성
  → 나: Amoeba Talk 알림 채널에서 실시간 수신
  → 알림 클릭 → 해당 할일 상세 모달 열림
```

## 5. 기술 제약사항

1. **3컬럼 레이아웃**: 모바일 반응형 고려 필요 (좁은 화면에서 세로 스택)
2. **관련자 테이블**: 기존 `amb_todos`와의 관계 설정, 마이그레이션 필요
3. **알림 시스템**: Amoeba Talk 모듈(`TalkSseService`)이 채널 기반. 개인 알림 채널 개념 필요
4. **성능**: 관련자 포함 할일 조회 시 JOIN 증가 → 쿼리 최적화 필요
5. **권한**: 관련자로 등록된 사용자의 할일 접근 범위 명확히 정의 필요

```typescript
async addTodoComment(todoId, content, userId) {
  const todo = await this.checkTodoAccess(todoId);
  if (todo.usrId === userId) throw ForbiddenException; // 본인 할일에는 코멘트 불가
  const comment = this.commentRepository.create({ ... });
  const saved = await this.commentRepository.save(comment);
```

**변경 후**:
```typescript
async addTodoComment(todoId, content, userId) {
  // ... 기존 로직 동일
  const saved = await this.commentRepository.save(comment);
  // ✅ 이벤트 발행 추가
  this.eventEmitter.emit('todo.comment.created', {
    todoId,
    todoTitle: todo.tdoTitle,
    todoOwnerId: todo.usrId,
    commentAuthorId: userId,
    commentAuthorName: withAuthor.author.usrName,
    commentContent: content,
  });
  return TodoMapper.toCommentResponse(withAuthor);
}
```

## 9. 리스크 및 고려사항

| 리스크 | 영향 | 완화 방안 |
|--------|------|----------|
| 3컬럼 레이아웃 모바일 대응 | 좁은 화면에서 수평 스크롤 | `md:` 브레이크포인트 이하에서 세로 스택 |

| 기존 할일 데이터 마이그레이션 | `amb_todo_participants` 신규 테이블이므로 기존 데이터 영향 없음 | 없음 |
