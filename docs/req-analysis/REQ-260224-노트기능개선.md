# 요구사항 분석서: 노트 기능 개선 (회의록/메모 → 노트)

- **작성일**: 2026-02-24
- **모듈**: 회의록/메모 (`meeting-notes`)
- **우선순위**: 높음

---

## 1. 배경 및 목적

기존 "회의록/메모" 기능을 "노트"로 통합하고, 작성 유형(회의록/메모)별 입력/저장 규칙을 명확히 분리한다.

**범위**: 사이드메뉴 명칭, 작성 버튼, 작성 폼, 저장 스키마, 목록/상세 표시 규칙, 공개범위 기준 변경.

**핵심 문제**:
- 회의록과 메모를 구분 없이 동일한 폼으로 작성 → 메모에도 불필요한 회의일 입력 강제
- 공개범위가 "부서" 기반 → 시스템의 "그룹" 기반 공유 체계와 불일치
- UI 텍스트가 "회의록/메모"로 혼재 → 통일된 "노트" 용어 필요

### 2.1 메뉴 및 네이밍

| 항목 | 현재 상태 |
|------|----------|
| 사이드 메뉴 | "회의록/메모" |
| 생성 버튼 | "+새 메모" |
| 페이지 타이틀 | "회의록/메모" |
| URL | `/meeting-notes` |
| i18n 네임스페이스 | `meetingNotes` |

### 2.2 폼 필드

| 필드 | 현재 | 비고 |
|------|------|------|
| 제목 | 필수 | `mtn_title` VARCHAR(200) |
| 회의일 | **항상 필수** | `mtn_meeting_date` DATE NOT NULL |
| 공개범위 | PRIVATE / DEPARTMENT / PUBLIC | GROUP은 Entity에 존재하나 UI 미노출 |
| 부서 | DEPARTMENT 선택 시 노출 | `mtn_department` VARCHAR(30) |
| 내용 | TipTap 리치텍스트 | `mtn_content` TEXT (HTML) |
| **유형 선택** | **없음** | 회의록/메모 구분 불가 |

### 2.3 공개범위

| 값 | UI 노출 | 동작 |
|----|---------|------|
| PRIVATE | ✅ | 본인만 조회 |
| DEPARTMENT | ✅ | 같은 부서원 조회 (부서 선택 필요) |
| GROUP | ❌ (UI 미노출) | Entity/DTO에는 존재, `mtn_group_id` 컬럼 있음 |
| PUBLIC | ✅ | 전체 조회 |

### 2.4 데이터 모델 이슈

- `mtn_meeting_date`가 NOT NULL — 메모 작성 시에도 회의일 필수 입력
- 유형 구분 필드(`type`) 없음
- `MEETING_VISIBILITY` 상수에 `GROUP` 누락 (Entity/DTO 불일치)
- `MeetingNoteResponse`에 `groupId` 필드 매핑 누락

## 3. TO-BE 요구사항

### 3.1 UX/명칭 변경

| 항목 | AS-IS | TO-BE |
|------|-------|-------|
| 사이드 메뉴 | "회의록/메모" | **"노트"** |
| 생성 버튼 | "+새 메모" | **"+새 노트"** |
| 페이지 타이틀 | "회의록/메모" | **"노트"** |
| 빈 상태 메시지 | "회의록이 없습니다" | **"노트가 없습니다"** |
| 토스트 메시지 | "메모가 저장되었습니다" | **"노트가 저장되었습니다"** |

| 다국어 | ko/en/vi 동시 변경 | 모든 키 "노트" 용어 통일 |

### 3.2 노트 유형(Type) 선택

#### 3.2.1 유형 종류

| 유형 | 값 | 설명 |
|-----|----|------|
| 회의록 | `MEETING_NOTE` | 회의록 양식, 회의일 필수, 템플릿 제공 |
| 메모 | `MEMO` | 간단한 메모, 자유 형식, 회의일 없음 |

#### 3.2.2 유형 선택 UI

- 작성 시작 시 **노트 유형 선택 필수**
- 폼 모달 상단에 2버튼 탭 또는 세그먼트 컨트롤
- 기본값: `MEMO`
- **유형 선택 후 폼이 동적으로 변경됨**:
  - `MEETING_NOTE`: 회의일 필드 노출 + 템플릿 삽입
  - `MEMO`: 회의일 필드 숨김 + 빈 내용

#### 3.2.3 유형 전환 시 동작

- 내용이 비어있으면: 즉시 전환
- 내용이 있으면: **"유형을 변경하면 내용이 초기화됩니다. 계속하시겠습니까?"** 확인 다이얼로그
  - 확인: 내용 초기화 + (회의록이면) 템플릿 삽입
  - 취소: 유형 유지
- **사용자가 수정한 내용은 자동 템플릿 재삽입으로 덮어쓰지 않음**

#### 3.2.4 DB 변경

| 변경사항 | 내용 |
|---------|------|
| 신규 컬럼 | `mtn_type VARCHAR(20) NOT NULL DEFAULT 'MEMO'` |
| 값 범위 | `'MEETING_NOTE'` / `'MEMO'` |
| 마이그레이션 | 기존 데이터 → `'MEETING_NOTE'` (기존이 회의록 기반이므로) |

### 3.3 저장/표시 필드 규칙

#### 3.3.1 공통 저장 필드

| 필드 | 설명 |
|------|------|
| `type` | 노트 유형 (`MEETING_NOTE` / `MEMO`) |
| `title` | 제목 (필수) |
| `content` | 본문 (HTML) |
| `visibility` | 공개범위 (그룹 기반) |
| `createdAt` | 작성일 (시스템 자동 저장) |

#### 3.3.2 메모 (`MEMO`)

| 항목 | 동작 |
|------|------|
| 회의일 필드 | **숨김** (UI 미노출, 입력 불가) |
| `meeting_date` 저장 | **작성일(현재 날짜) 자동 저장** |
| 서버 방어 | `type=MEMO`인데 `meeting_date` 전달 시 → **무시**, 현재 날짜로 덮어씀 |
| 상세 페이지 | "작성일" 레이블로 `createdAt` 표시 |

#### 3.3.3 회의록 (`MEETING_NOTE`)

| 항목 | 동작 |
|------|------|
| 회의일 필드 | **노출** (날짜 선택 **필수**) |
| `meeting_date` 저장 | 사용자가 선택한 회의일 저장 |
| `createdAt` | 별도 자동 저장 (회의일과 독립) |
| 상세 페이지 | "회의일: YYYY-MM-DD" 표시 + "작성일" 별도 표시 |

#### 3.3.4 DB 변경

| 변경사항 | 내용 |
|---------|------|
| `mtn_meeting_date` | `NOT NULL` **유지** (MEMO는 작성일 자동 입력) |
| DTO 변경 | `meeting_date`를 `@IsOptional()`로 변경 (MEMO 시 미전송) |
| 서비스 로직 | `type === 'MEMO'` && `meeting_date` 없으면 → **현재 날짜 자동 설정** |
| 서비스 로직 | `type === 'MEETING_NOTE'` && `meeting_date` 없으면 → 검증 오류 |

### 3.4 공개범위 그룹 기반 전환

#### 3.4.1 공개범위 옵션

| AS-IS | TO-BE |
|-------|-------|
| PRIVATE / DEPARTMENT / PUBLIC | PRIVATE / **GROUP** / **ENTITY** / PUBLIC |

> **DEPARTMENT 제거, GROUP 추가, ENTITY(법인 전체) 추가**

| 값 | 설명 | 선택 시 추가 UI |
|----|------|----------------|
| `PRIVATE` | 본인만 | 없음 |
| `GROUP` | 특정 그룹원 | **단일 그룹 선택** 드롭다운 |
| `ENTITY` | 법인 전체 | 없음 (현재 접속 법인 기준) |
| `PUBLIC` | 전체 공개 | 없음 |

> **결정**: 공개범위는 **단일 그룹** 선택으로 제한 (복수 그룹 미지원, 향후 확장 가능)

#### 3.4.2 DB 변경

| 변경사항 | 내용 |
|---------|------|
| `mtn_visibility` | `DEPARTMENT` → `ENTITY` 마이그레이션 |
| `mtn_department` | deprecated (향후 제거), 기존 값 유지 |
| `mtn_group_id` | 기존 컬럼 활용 (이미 존재) |
| `MEETING_VISIBILITY` 상수 | `GROUP`, `ENTITY` 추가, `DEPARTMENT` 제거 |

#### 3.4.3 기존 데이터 마이그레이션 정책

| 기존 값 | 전환 정책 | 근거 |
|---------|----------|------|
| `DEPARTMENT` | → `ENTITY` | 부서 공개 → 법인 전체 공개로 안전 확대 (축소 시 접근 불가 우려) |
| `PRIVATE` | 유지 | 변경 없음 |
| `PUBLIC` | 유지 | 변경 없음 |
| `GROUP` (기존 데이터 있을 경우) | 유지 | `mtn_group_id`와 함께 유지 |

#### 3.4.4 서비스 조회 로직

- `ENTITY` 공개: 같은 `entId`의 모든 사용자 조회 가능
- `GROUP` 공개: `GroupAccessService.getUserGroupIds`로 그룹 멤버 조회
- `DEPARTMENT` 하위 호환: 기존 로직 유지 (마이그레이션 후 해당 데이터 없어야 함)

### 3.5 회의록 템플릿

#### 3.5.1 기본 템플릿

유형을 `MEETING_NOTE`로 선택하면 내용 영역에 아래 템플릿이 자동 삽입된다 (최초 1회):

```
회의 정보
  회의제목 :
  참석자 :
  주제 :

회의 내용
  (자유 작성)

결정 사항
  (자유 작성)

Action Items
  • 
```

> TipTap HTML 형태:
```html
<h2>회의 정보</h2>
<p><strong>회의제목 :</strong> </p>
<p><strong>참석자 :</strong> </p>
<p><strong>주제 :</strong> </p>

<h2>회의 내용</h2>
<p></p>

<h2>결정 사항</h2>
<p></p>

<h2>Action Items</h2>
<ul>
  <li></li>
</ul>
```

#### 3.5.2 템플릿 동작

| 항목 | 동작 |
|------|------|
| 자동 삽입 시점 | 유형 `MEETING_NOTE` 선택 즉시 (내용이 비어있을 때만) |
| 수정 가능 여부 | 완전 자유 편집 (템플릿은 초기값일 뿐) |
| 재삽입 방지 | 사용자가 수정한 내용은 자동 재삽입으로 덮어쓰지 않음 |
| 유형 전환 시 | 확인 다이얼로그 → 확인 시 내용 초기화 + (회의록이면) 템플릿 삽입 |
| i18n | 템플릿 내용도 번역 키로 관리 (ko/en/vi) |

## 4. API/데이터 계약

### 4.1 요청 DTO 변경

**CreateMeetingNoteRequest**:

| 필드 | 타입 | 필수 | 변경사항 |
|------|------|------|---------|
| `type` | `'MEETING_NOTE' \| 'MEMO'` | **YES** | **신규** |
| `title` | string | YES | 유지 |
| `content` | string | YES | 유지 |
| `meeting_date` | string (date) | MEETING_NOTE만 필수 | `@IsOptional()` 변경 (MEMO 시 서버 자동설정) |
| `visibility` | `'PRIVATE' \| 'GROUP' \| 'ENTITY' \| 'PUBLIC'` | NO | GROUP/ENTITY 추가 |
| `group_id` | string (UUID) | GROUP 시 | 유지 |

**UpdateMeetingNoteRequest**: 동일 구조, 모든 필드 optional

### 4.2 응답 DTO 변경

**MeetingNoteResponse**:

| 필드 | 타입 | 변경사항 |
|------|------|---------|
| `type` | `'MEETING_NOTE' \| 'MEMO'` | **신규** |
| `meetingDate` | `string` | 유지 (MEMO는 작성일, MEETING_NOTE는 회의일) |
| `groupId` | `string \| null` | **신규 매핑** (기존 누락 수정) |
| `createdAt` | string | 유지 (작성일) |
| `updatedAt` | string | 유지 |

### 4.3 서버 방어 로직

| 케이스 | 처리 |
|--------|------|
| `type=MEMO` + `meeting_date` 전달 | `meeting_date` **무시** (현재 날짜로 자동 저장) |
| `type=MEETING_NOTE` + `meeting_date` 미전달 | **400 Bad Request** 검증 오류 |
| `visibility=GROUP` + `group_id` 미전달 | **400 Bad Request** 검증 오류 |

### 4.4 Controller 쿼리 파라미터 추가

| 파라미터 | 타입 | 설명 |
|---------|------|------|
| `type` | `'MEETING_NOTE' \| 'MEMO'` | 유형별 필터 (optional) |

## 7. 수용 기준 (Acceptance Criteria)

| # | 기준 | 검증 방법 |
|---|------|----------|
| AC-1 | 메뉴/버튼 문구가 모두 "노트"로 변경된다 | UI 확인 (ko/en/vi) |
| AC-2 | 새 노트 작성 시 유형 선택이 가능하다 | 폼 모달에서 MEMO/MEETING_NOTE 선택 |
| AC-3 | 메모 선택 시 회의일이 보이지 않고 작성일이 자동 저장된다 | MEMO 저장 후 DB 확인: `mtn_meeting_date = 작성일` |
| AC-4 | 회의록 선택 시 회의일이 보이며 별도 작성일이 저장된다 | MEETING_NOTE 저장 후 DB 확인: `mtn_meeting_date` + `mtn_created_at` 독립 |
| AC-5 | 회의록 선택 시 템플릿이 초기 노출된다 | 폼 모달 내용 영역에 "회의제목/참석자/주제" 삽입 확인 |
| AC-6 | 공개범위가 그룹 기준으로 정상 저장/조회된다 | GROUP visibility + group_id 저장, 해당 그룹원만 조회 |
| AC-7 | 목록에 유형 뱃지가 표시된다 | 📝/🗓 뱃지 확인 |
| AC-8 | 유형 필터가 동작한다 | 전체/회의록/메모 필터링 확인 |
| AC-9 | 기존 데이터가 정상 마이그레이션된다 | 기존 DEPARTMENT → ENTITY, type → MEETING_NOTE 확인 |
| AC-10 | 서버 방어 로직이 동작한다 | MEMO+meeting_date → 무시(작성일 저장), MEETING_NOTE-meeting_date → 400 |

## 10. 추가 고려사항

### 10.1 확정 사항 (본 요구사항 기준)

| 항목 | 결정 |
|------|------|
| 공개범위 그룹 | **단일 그룹** 선택으로 제한 (1:1, 복수 미지원) |
| 템플릿 삽입 시점 | 유형 선택 **즉시** (내용 비어있을 때) |
| 기존 DEPARTMENT 데이터 | → `ENTITY`로 안전 확대 전환 |
| MEMO의 meeting_date | **작성일(현재 날짜) 자동 저장** (NOT NULL 유지) |
| 노트 유형 값 | `MEETING_NOTE` / `MEMO` |

### 10.2 향후 확장 가능성

- 추가 유형: `TODO`, `DAILY_REPORT` 등 확장 가능하도록 `mtn_type` VARCHAR 설계
- 복수 그룹 공유: UI에서 그룹 다중 선택 (M:N 테이블 필요)
- 커스텀 템플릿: 사용자/부서별 회의록 템플릿 관리 기능
- 태그/라벨: 노트에 태그 부여 기능

### 10.3 마이그레이션 전략

| 기존 데이터 | 마이그레이션 |
|------------|-------------|
| `mtn_type` NULL (컬럼 미존재) | → `'MEETING_NOTE'` (기존 데이터는 모두 회의록 취급) |
| `mtn_visibility = 'DEPARTMENT'` | → `'ENTITY'` (부서→법인 전체로 안전 확대) |
| `mtn_meeting_date` NOT NULL | 유지 (MEMO는 작성일 자동, 기존 값 유지) |
| 기존 `mtn_department` 데이터 | 유지 (deprecated, 향후 DROP COLUMN) |
