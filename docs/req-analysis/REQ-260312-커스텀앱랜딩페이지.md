## 1. 요구사항 원문

### v1.0 원문 (초기)
> `https://stg-ama.amoeba.site/apps/{appname}` 커스텀앱별로 페이지를 제공한다.  
> 각 페이지는 커스텀앱 이름과 내용 설명  
> `https://stg-ama.amoeba.site/entity-settings/custom-apps` App Name과 Description을 노출한다.  
> 앱 연결 시 먼저 이 설명 페이지로 이동해서 iframe을 지원하는 사이트는 바로 이동하고,  
> iframe을 지원하지 않는 서비스는 이 페이지에서 "Open in the new tab" 버튼을 제공한다.

### v1.1 추가 요구사항
> **앱 설정에 iframe / open new tab 설정 기능 추가**  
> - iframe 허용된(설정된) 앱은 바로 오픈  
> - 이외 앱은 페이지 이동 후 클릭 형태  
> 
> **Authentication 방식별로 입력받아야 할 항목 노출 및 저장 기능**  
> - JWT Token  
> - API Key  
> - None

## 2. AS-IS 현황 분석

### 2.1 현재 `/apps/:appCode` 동작

`CustomAppHostPage` 컴포넌트(`apps/web/src/domain/custom-apps/pages/CustomAppHostPage.tsx`) 기준:

| 단계 | 현재 동작 |
|------|---------|
| 앱 접근 | `/apps/:appCode` 접속 즉시 iframe 로드 시작 |
| 설명 노출 | 없음 (앱 이름만 상단 바에 표시) |
| iframe 차단 감지 | 3초 타임아웃: `onLoad` 미발생 시 `iframeBlocked = true` |
| 차단 시 UI | 앱 이름 + 설명 카드 + "Open in new tab" 버튼 |
| 성공 시 UI | 상단 바(이름 + 새탭 버튼) + iframe 전체 렌더링 |

**현재 플로우:**
```
앱 클릭 → /apps/:appCode → (즉시) iframe 로드 시도
                              ├─ 3초 내 로드 성공 → iframe 전체화면
                              └─ 3초 내 로드 실패 → 설명 카드 + 새탭 버튼
```

### 2.2 현재 DB 구조 (`amb_entity_custom_apps`)

| 컬럼 | 타입 | 현황 |
|------|------|------|
| `eca_name` | varchar(200) | ✅ 존재 |
| `eca_description` | text nullable | ✅ 존재 |
| `eca_icon` | varchar(100) | ✅ 존재 |
| `eca_url` | varchar(500) | ✅ 존재 |
| `eca_code` | varchar(50) | ✅ 존재 |
| `eca_auth_mode` | varchar(20) — `jwt/none/api_key` | ✅ 존재 |
| **iframe/새탭 오픈 방식 설정** | — | ❌ **없음 → 신규** |
| **API Key 값 저장** | — | ❌ **없음 → 신규** |

### 2.3 현재 설정 폼 (`EntityCustomAppsPage.tsx`)

- `auth_mode` 선택: JWT / None / API Key — 라디오 버튼 3개만 존재
- **JWT Token**: 추가 입력 없음 (서버에서 자동 생성)
- **API Key**: 현재 별도 키 입력 필드 없음 ❌
- **None**: 추가 입력 없음
- **오픈 방식 설정(iframe/새탭)**: 현재 없음 ❌

## 3. TO-BE 요구사항

### 3.1 앱 오픈 방식 설정 (신규)

#### FR-01. 오픈 방식 설정 옵션 추가

앱 등록/수정 폼에 **"오픈 방식"** 설정 항목 추가:

| 옵션 | 값 | 동작 |
|------|-----|------|
| iframe 내에서 열기 | `iframe` | 앱 진입 시 즉시 iframe 전체화면 |
| 새 탭에서 열기 | `new_tab` | 앱 진입 시 랜딩 페이지(설명 카드 + "Open in new tab" 버튼) 표시 |

#### FR-02. 오픈 방식에 따른 `/apps/:appCode` 동작 분기

**`open_mode = 'iframe'`인 경우:**
```
사용자 클릭 → /apps/:appCode → 즉시 iframe 전체화면
    ├─ onLoad 발생 → 정상 표시
    └─ 3초 후 onLoad 없음 → 자동 폴백: 설명 카드 + "Open in new tab"
```

**`open_mode = 'new_tab'`인 경우:**
```
사용자 클릭 → /apps/:appCode → 랜딩 페이지 표시 (앱 이름 + 설명)
    → "Open in new tab" 버튼 클릭 → 새 탭에서 URL 오픈
```

> v1.0에서 제안한 3초 타임아웃 자동 감지 후 자동 전환은 제거. 관리자가 오픈 방식을 명시적으로 설정함.  
> 단, `iframe` 설정 앱이 실제 차단될 경우 폴백(fallback) 처리는 유지.

### 3.2 Authentication 방식별 입력 항목 (신규)

#### FR-03. JWT Token 방식

| 항목 | 내용 |
|------|------|
| 추가 입력 필드 | 없음 |
| 동작 | 앱 접근 시 서버에서 JWT 자동 발급 (`/entity-settings/custom-apps/:id/token`) |
| URL 구성 | `{eca_url}?ama_token={jwt}&locale={lang}` |
| 안내 문구 | "접속 시 자동으로 JWT 토큰이 발급되어 URL에 전달됩니다." |

#### FR-04. API Key 방식 (신규)

| 항목 | 내용 |
|------|------|
| 추가 입력 필드 | **API Key 값** (필수, 암호화 저장) |
| 표시 방식 | 입력 시 마스킹 (`type="password"`), 저장 후 `••••••••{last4}` 표시 + "변경" 버튼 |
| URL 구성 | `{eca_url}?api_key={value}&locale={lang}` |
| 안내 문구 | "API Key가 URL 쿼리 파라미터로 전달됩니다. HTTPS 환경에서만 사용하세요." |
| 보안 주의 | API Key는 서버에 암호화 저장. 응답 시 마스킹 처리 (`••••••••{last4}`) |

#### FR-05. None 방식

| 항목 | 내용 |
|------|------|
| 추가 입력 필드 | 없음 |
| 동작 | 인증 없이 URL 그대로 열기 |
| URL 구성 | `{eca_url}?locale={lang}` |

### 3.3 entity-settings/custom-apps 설정 UI 개선

#### FR-06. 앱 카드 정보 강조

- App Name: `text-base font-semibold` 강조
- Description: 최대 2줄 (`line-clamp-2`)
- 오픈 방식 뱃지: `iframe` / `새 탭` 표시
- "앱 보기 →" 링크: `/apps/:appCode` 이동

#### FR-07. 앱 등록/수정 폼 구성 개선

```
[기본 정보]
 - 앱 코드 (생성 시만) / 앱 이름 * / 앱 URL * / 설명 / 아이콘

[오픈 방식]  ← 신규
  ( • iframe 내에서 열기 )  ( • 새 탭에서 열기 )
  ↳ iframe 선택 시: "iframe 미지원 시 자동으로 새 탭 안내로 전환됩니다." 안내

[인증 설정]  ← 개선
  ( • JWT Token )  ( • API Key )  ( • None )
  ↳ JWT Token: 안내 문구만
  ↳ API Key:   [API Key 입력 필드] (필수)
  ↳ None:      추가 입력 없음

[접근 역할] / [정렬 순서] / [활성 상태 (수정 시)] / [연결 테스트 (수정 시)]
```

## 5. 사용자 플로우

### 5.1 iframe 오픈 방식 앱 (`open_mode = 'iframe'`)

```
사용자 클릭 → /apps/:appCode
    ↓
즉시 iframe 전체화면 시도
    ├─ onLoad 발생 (3초 내) → 정상 iframe 표시
    └─ 타임아웃 (3초 후) → 자동 폴백: 설명 카드 + "Open in new tab" 버튼
```

### 5.2 새 탭 오픈 방식 앱 (`open_mode = 'new_tab'`)

```
사용자 클릭 → /apps/:appCode
    ↓
[랜딩 페이지] 표시
  - 앱 아이콘 + 이름 + 설명
  - "Open in new tab" 버튼
    ↓ 클릭
외부 URL 새 탭 열기
```

### 5.3 API Key 방식 앱 등록 플로우

```
설정 폼 → 인증 설정: "API Key" 선택
    ↓
API Key 입력 필드 표시
    ↓ 저장
서버: AES-256 암호화 후 eca_api_key_enc에 저장
    ↓
앱 접근 시 → 서버에서 복호화 → URL 쿼리 파라미터로 추가
```

### 5.4 관리자 앱 설정 변경

```
entity-settings/custom-apps → [수정] 버튼
    ↓
오픈 방식 변경: iframe ↔ new_tab
인증 방식 변경: 해당 방식 입력 필드 표시/숨김
    ↓ 저장
즉시 반영 (사이드바 캐시 2분 후 갱신)
```

## 6. 기술 제약사항

### 6.1 iframe 차단 감지 (open_mode = 'iframe' 전용 폴백)

브라우저 보안 정책상 JavaScript로 `X-Frame-Options`를 직접 읽을 수 없음.  
`open_mode = 'iframe'` 설정 앱에 한해 3초 타임아웃 + `onLoad` 감지로 폴백 처리.

### 6.2 API Key 보안

- **저장**: AES-256-GCM + 환경 변수 `ENCRYPTION_KEY`로 암호화
- **응답**: 항상 `••••••••{last4}` 마스킹 반환 — 원문 노출 금지
- **전달**: HTTPS URL 쿼리 파라미터(`?api_key=...`)로 전달  
  ※ 서버 로그 노출 가능성 — HTTPS + 외부 앱 로그 처리 안전성 전제
- **변경 시**: 입력 필드가 비어있으면 기존 키 유지 (명시적 입력 시만 덮어쓰기)

### 6.3 오픈 방식 기본값

- 신규 컬럼 추가 시 기존 앱: `DEFAULT 'iframe'` 적용 → 기존 동작 유지
- 신규 등록 폼 기본값: `iframe`

### 6.4 암호화 키 관리

- 환경 변수 `ENCRYPTION_KEY` (32바이트 hex) 추가 필요
- `env/backend/.env.development`, staging/production `.env`에 각각 추가

## 7. 엣지 케이스

| 케이스 | 처리 방법 |
|--------|---------|
| 앱 코드 존재하지 않음 | 에러 페이지 (현행 유지) |
| 앱 비활성 상태 | 에러 페이지 (현행 유지) |
| `open_mode = 'iframe'`이나 실제 차단됨 | 3초 폴백: 설명 카드 + "Open in new tab" |
| `open_mode = 'new_tab'` | 항상 랜딩 페이지, iframe 시도 없음 |
| API Key 방식에서 Key 미입력 | 폼 유효성 검사 — 저장 불가 |
| API Key 변경 시 입력 필드 비워두면 | 기존 Key 유지 |
| JWT 토큰 발급 실패 | 에러 표시 + 원본 URL 새탭 버튼 |
| 기존 앱(컬럼 추가 전 데이터) | `DEFAULT 'iframe'` 적용 — 기존 동작 유지 |
| 설명(`eca_description`) 없음 | 설명 영역 숨김 |
| 유효하지 않은 Lucide 아이콘명 | `AppWindow` 폴백 |
