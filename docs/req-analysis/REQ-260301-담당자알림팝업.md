### 3.2 백엔드 설계

#### 3.2.1 DB 테이블 — `amb_notifications`

```sql
CREATE TABLE amb_notifications (
    ntf_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    ntf_type VARCHAR(30) NOT NULL,        -- 'TODO_ASSIGNED', 'ISSUE_ASSIGNED', 'NOTE_ASSIGNED', 'CALENDAR_INVITED'
    ntf_title VARCHAR(500) NOT NULL,       -- 알림 제목
    ntf_message TEXT,                      -- 알림 본문
    ntf_recipient_id UUID NOT NULL,        -- 수신자 user ID (FK → amb_users)
    ntf_sender_id UUID NOT NULL,           -- 발신자 user ID (FK → amb_users)
    ntf_resource_type VARCHAR(30) NOT NULL, -- 'TODO', 'ISSUE', 'MEETING_NOTE', 'CALENDAR'
    ntf_resource_id UUID NOT NULL,          -- 대상 리소스 ID
    ntf_is_read BOOLEAN DEFAULT FALSE,     -- 읽음 여부
    ntf_read_at TIMESTAMP WITH TIME ZONE,  -- 읽은 시간
    ent_id UUID NOT NULL,                  -- 법인 ID
    ntf_created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    ntf_deleted_at TIMESTAMP WITH TIME ZONE
);
CREATE INDEX idx_ntf_recipient ON amb_notifications(ntf_recipient_id, ntf_is_read);
CREATE INDEX idx_ntf_resource ON amb_notifications(ntf_resource_type, ntf_resource_id);
```

#### 3.2.2 이벤트 페이로드

```typescript
// 담당자 지정 이벤트 (EventEmitter2)
interface AssigneeAssignedEvent {
  type: 'TODO_ASSIGNED' | 'ISSUE_ASSIGNED' | 'NOTE_ASSIGNED' | 'CALENDAR_INVITED';
  resourceType: 'TODO' | 'ISSUE' | 'MEETING_NOTE' | 'CALENDAR';
  resourceId: string;         // 대상 항목 ID
  resourceTitle: string;      // 대상 항목 제목
  senderId: string;           // 지정한 사용자 ID
  senderName: string;         // 지정한 사용자 이름
  recipientIds: string[];     // 수신 대상 사용자 ID 배열
  entityId: string;           // 법인 ID
  role: 'ASSIGNEE' | 'PARTICIPANT';  // 역할
}
```

#### 3.2.3 글로벌 SSE 엔드포인트

```typescript
// NotificationSseService — Talk SSE 패턴 재사용
@Controller('api/v1/notifications')
export class NotificationController {
  @Sse('stream')              // GET /api/v1/notifications/stream
  stream(@CurrentUser() user): Observable<MessageEvent> {
    return this.notificationSseService.subscribe(user.usrId);
  }
}
```

#### 3.3.2 SSE 연결

```typescript
// hooks/useNotificationSSE.ts
const useNotificationSSE = () => {
  useEffect(() => {
    const eventSource = new EventSource('/api/v1/notifications/stream');
    eventSource.onmessage = (event) => {
      const notification = JSON.parse(event.data);
      // 1. 토스트 팝업 표시
      toast(notification.title, { description: notification.message });
      // 2. 스토어 업데이트 (미읽음 카운트)
      useNotificationStore.getState().addNotification(notification);
    };
    return () => eventSource.close();
  }, []);
};
```

## 5. 사용자 플로우

```
1. 사용자 A가 이슈를 생성하면서 담당자로 사용자 B를 지정
      │
2. IssueService.createIssue() 실행
      │
3. EventEmitter.emit('assignee.assigned', { 
      type: 'ISSUE_ASSIGNED', recipientIds: [B.id], ...
   })
      │
4. NotificationListener.onAssigneeAssigned()
      ├── 4a. amb_notifications 테이블에 INSERT
      └── 4b. NotificationSseService.emit(B.id, payload)
              │
5. 사용자 B의 브라우저 (SSE 연결 중)
      ├── 5a. Toast 팝업: "사용자 A님이 이슈의 담당자로 지정했습니다. [이슈 제목]"
      └── 5b. 헤더 벨 아이콘 미읽음 카운트 +1
              │
6. 사용자 B가 토스트 또는 드롭다운에서 클릭
      │
7. 해당 이슈 상세 페이지로 이동 + 알림 읽음 처리
```

## 6. 기술 제약사항

| 제약 | 영향 | 대응 |
|------|------|------|
| SSE는 단방향 통신 | 서버 → 클라이언트만 가능 | 읽음 처리는 REST API로 |
| SSE 연결 수 제한 (브라우저당 6개) | 탭 여러 개 열면 제한 | SharedWorker 또는 BroadcastChannel 고려 |
| 사용자 오프라인 시 SSE 수신 불가 | 실시간 알림 미수신 | DB 저장으로 다음 접속 시 미읽음 조회 |
| EventEmitter2는 프로세스 내 이벤트 | 멀티 인스턴스 환경 불가 | 현재 단일 인스턴스이므로 문제없음, 이후 Redis Pub/Sub 고려 |
