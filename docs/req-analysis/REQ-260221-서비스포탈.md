## 1. AS-IS 현황 분석

### 1.1 현재 시스템 구조

현재 AMB Management(`mng.amoeba.site`)는 **내부 직원 전용** 시스템으로, 외부 고객 대응 기능이 전무하다.

```
현재 운영 중:
  mng.amoeba.site (내부 관리) ─── PostgreSQL ─── Postal Mail
    ├── HR / Payroll / Leave / Timesheet
    ├── Billing (계약·송장·결제·세금계산서)
    ├── KMS (지식관리·태깅·AI 분석)
    ├── Chat (9개 부서 AI 에이전트)
    ├── Mail (Postal 웹메일)
    ├── Project / Service / Documents
    ├── Accounting / Notices / Todos
    └── Settings (API Key, SMTP, Drive)

미구축:
  www.amoeba.site (서비스 포탈) ─── 전무
    ├── 서비스 소개/마케팅 ❌
    ├── 온라인 구독/결제 ❌
    ├── 고객 셀프서비스 포탈 ❌
    └── 서비스 프로비저닝 ❌
```

### 4.1 아키텍처 선택지

| 방안 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A) 모노레포 확장** | 현재 `ambManagement` 모노레포에 `apps/portal` 추가 | 코드 공유 용이, 빌링 엔티티 직접 참조, 인프라 통합 | 모노레포 규모 증가, 배포 복잡성 |

#### 권장: **방안 A — 모노레포 확장**

```
ambManagement/ (기존 모노레포)
├── apps/
│   ├── api/                ← 기존 내부 API (mng.amoeba.site 전용)
│   ├── web/                ← 기존 내부 프론트엔드
│   ├── portal-api/         ← 🆕 서비스 포탈 API (www.amoeba.site 전용)
│   └── portal-web/         ← 🆕 서비스 포탈 프론트엔드
├── packages/
│   ├── types/              ← 공유 타입 (기존 + 포탈용 확장)
│   ├── common/             ← 공유 유틸리티
│   └── billing-core/       ← 🆕 빌링 코어 (엔티티, 서비스 공유)
```

**이유**:
1. `bil_partners`, `bil_contracts`, `bil_invoices` 엔티티를 `billing-core` 패키지로 추출하여 양쪽 API에서 공유
2. CryptoService, MailService 등 인프라 코드 재사용
3. 동일 PostgreSQL DB 사용 → 실시간 데이터 동기화 불필요 (직접 참조)
4. Turborepo 빌드 캐싱 활용
5. CI/CD 파이프라인 통합

### 4.3 Stripe 연동 설계

```
Stripe 연동 흐름:

1. 고객 가입 → Stripe Customer 생성
   stripe.customers.create({ email, name, metadata: { cst_id } })

2. 플랜 선택 + 결제 → Stripe Checkout Session
   stripe.checkout.sessions.create({
     customer: stripe_customer_id,
     line_items: [{ price: pln_stripe_price_id, quantity: 1 }],
     mode: 'subscription',
     subscription_data: { trial_period_days: 14 }
   })

3. 결제 완료 → Webhook: checkout.session.completed
   → svc_subscriptions 생성
   → bil_partners 생성 (ptn_source: SERVICE_PORTAL)
   → bil_contracts 생성 (ctr_type: SUBSCRIPTION)
   → API Key 발급
   → Welcome email 발송

4. 월 청구 → Webhook: invoice.paid
   → bil_invoices 동기화
   → 사용량 초과분 포함

5. 결제 실패 → Webhook: invoice.payment_failed
   → 고객 알림 이메일
   → Dunning 프로세스 시작

6. 구독 변경 → Webhook: customer.subscription.updated
   → svc_subscriptions 상태 업데이트
   → bil_contracts 동기화
```

## 7. 기술적 제약사항 및 결정사항

### 7.1 아키텍처 결정

| 결정 | 근거 |
|------|------|
| 고객 계정은 `svc_customers`로 내부 사용자(`amb_users`)와 완전 분리 | 내부 직원과 외부 고객의 인증/인가 체계가 근본적으로 다름 |
| `portal-api`는 별도 NestJS 앱으로 분리 | 내부 API와 외부 API의 보안 수준, Rate Limiting, 인증 방식이 다름 |
| 빌링 엔티티는 `billing-core` 공유 패키지로 추출 | 양쪽 API에서 동일 DB 테이블 참조, 코드 중복 방지 |
| 신규 테이블은 `svc_` prefix 사용 | 기존 `amb_` prefix와 구분, 서비스 포탈 전용 데이터 식별 |
| PCI-DSS는 Stripe Checkout/Customer Portal에 위임 | 카드 데이터를 자체 서버에 저장하지 않음 |

### 7.2 구현 시 주의사항

1. **Stripe Webhook 멱등성** (NFR-012): event ID 기반 중복 처리 필수 — Postal webhook 처리에서 `msg_external_id` 중복 체크 패턴 참조
2. **bil_partners/contracts 동기화**: 포탈에서 생성한 데이터가 내부 관리 화면에도 즉시 표시되어야 함 — 동일 DB이므로 자동 해결
3. **API Center 미구축 대응**: Phase 3 시작 전 Canonical Provisioning Interface를 TypeScript interface로 사전 정의, Mock 구현체로 개발 진행
4. **환경 분리**: portal-api는 별도 포트(예: 3010)로 운영, 프로덕션에서는 별도 Cloud Run 인스턴스
