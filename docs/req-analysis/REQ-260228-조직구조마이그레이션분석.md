## 1. 분석 개요

### 1.1 목적
Department/Group 모델 → Unit/Cell/Team 모델로의 마이그레이션이 현재 시스템과 호환되는지 검증하고, 마이그레이션 SQL 실행 시 발생할 수 있는 문제점을 사전 식별한다.

### 1.2 분석 범위
| 영역 | 분석 내용 |
|------|----------|
| DB 스키마 | 테이블명, 컬럼명, FK 관계, 인덱스, 제약조건 |
| 마이그레이션 SQL | Phase 1~5 각 단계별 SQL 검증 |
| 백엔드 코드 | TypeORM 엔티티, 서비스, 컨트롤러, DTO |
| 프론트엔드 코드 | 컴포넌트, 라우팅, 상수, i18n |
| 공유 타입 | packages/types의 enum/interface 정의 |

### 1.3 결론 요약

> **마이그레이션 SQL을 현재 상태로 실행하면 실패합니다.**
>
> ~~CRITICAL 등급 4건, HIGH 등급 7건, MEDIUM 등급 5건의 문제가 확인되었습니다.~~
>
> **[재검토 결과] CRITICAL 7건, HIGH 12건, MEDIUM 11건 — 총 30건의 이슈가 확인되었습니다.**
> 최초 분석 대비 CRITICAL 3건, HIGH 5건, MEDIUM 6건이 추가 발견되었습니다.
> `amb_todos`, `amb_notices`, `amb_invitations`, `amb_users`, `amb_assets`, `amb_calendars` 등 다수 테이블/서비스의 department/group 참조가 누락되어 있었습니다.
> SQL 실행 자체가 실패하는 CRITICAL 이슈를 수정하지 않으면 마이그레이션 진행이 불가능합니다.

## 2. CRITICAL 이슈 (마이그레이션 SQL 실행 실패)

### 🔴 C-1. `amb_menu_permissions` 컬럼명 불일치 — SQL 실행 실패

**현상**: 마이그레이션 Phase 4.3 및 Phase 5 INSERT문이 존재하지 않는 컬럼명을 참조

| 구분 | 마이그레이션 SQL 사용값 | 실제 시스템 컬럼명 |
|------|----------------------|------------------|
| 메뉴코드 | `mpr_menu_code` | `mpm_menu_code` |
| 역할 | `mpr_role` | `mpm_role` |
| 접근수준 | `mpr_access_level` (VARCHAR) | `mpm_accessible` (BOOLEAN) |

**영향**:
```sql
-- Phase 4.3: 이 UPDATE는 실패함 (mpr_menu_code 컬럼 없음)
UPDATE amb_menu_permissions
SET mpr_menu_code = 'UNITS'           -- ✗ 실제: mpm_menu_code
WHERE mpr_menu_code = 'DEPARTMENTS';  -- ✗ 실제: mpm_menu_code

-- Phase 5: 이 INSERT도 실패함 (컬럼명 + 데이터 타입 불일치)
INSERT INTO amb_menu_permissions (mpr_menu_code, mpr_role, mpr_access_level)
VALUES ('TEAMS', 'ADMIN', 'FULL');
-- ✗ 실제 컬럼: (mpm_menu_code, mpm_role, mpm_accessible)
-- ✗ mpm_accessible는 BOOLEAN이지 'FULL' VARCHAR가 아님
```

**수정 방안**:
```sql
-- Phase 4.3 수정
UPDATE amb_menu_permissions
SET mpm_menu_code = 'UNITS'
WHERE mpm_menu_code = 'DEPARTMENTS';

-- Phase 5 수정
INSERT INTO amb_menu_permissions (mpm_menu_code, mpm_role, mpm_accessible)
VALUES
    ('TEAMS', 'ADMIN', TRUE),
    ('TEAMS', 'MANAGER', TRUE),
    ('TEAMS', 'USER', TRUE);
```

### 🔴 C-2. `amb_groups` 기존 컬럼 미처리 — 중복 컬럼 발생

**현상**: 현재 `amb_groups` 테이블에 `grp_created_at`, `grp_updated_at`, `grp_deleted_at`이 이미 존재하나, 마이그레이션에서 리네이밍하지 않음

| 현재 존재 컬럼 | 마이그레이션 동작 | 결과 |
|--------------|----------------|------|
| `grp_created_at` | 리네이밍 안 함 | `grp_created_at`이 방치됨 (cel_ prefix 아님) |
| `grp_updated_at` | ADD COLUMN `cel_updated_at` | 두 개의 updated_at 컬럼 공존 → **혼란** |
| `grp_deleted_at` | ADD COLUMN `cel_deleted_at` | 두 개의 deleted_at 컬럼 공존 → **혼란** |

**수정 방안**: Phase 2에 다음 추가
```sql
-- 기존 컬럼을 먼저 리네이밍
ALTER TABLE amb_cells RENAME COLUMN grp_created_at TO cel_created_at;
ALTER TABLE amb_cells RENAME COLUMN grp_updated_at TO cel_updated_at;
ALTER TABLE amb_cells RENAME COLUMN grp_deleted_at TO cel_deleted_at;

-- cel_updated_at, cel_deleted_at ADD COLUMN 제거 (이미 존재하므로)
```

### 🔴 C-3. `amb_menu_group_permissions` 테이블 완전 누락

**현상**: 마이그레이션 SQL 전체에서 `amb_menu_group_permissions` 테이블에 대한 처리가 없음

**현재 테이블 구조**:
```
amb_menu_group_permissions
├── mgp_id          UUID (PK)
├── mgp_menu_code   VARCHAR(50)
├── grp_id          UUID → amb_groups(grp_id)  ← FK 참조!!
├── mgp_accessible  BOOLEAN
├── mgp_created_at  TIMESTAMP
└── mgp_updated_at  TIMESTAMP
```

**문제점**:
1. `grp_id` FK가 `amb_groups(grp_id)`를 참조 → 테이블이 `amb_cells`로 변경되면 FK 무결성 위반
2. 컬럼명 `grp_id`가 새 네이밍 컨벤션(`cel_id`)과 불일치
3. 테이블명 자체가 "group"을 포함 → "cell"로 변경 필요

**수정 방안**: Phase 2에 다음 추가
```sql
-- 2.8 Rename menu-group-permissions → menu-cell-permissions
ALTER TABLE amb_menu_group_permissions RENAME COLUMN grp_id TO cel_id;
ALTER TABLE amb_menu_group_permissions RENAME COLUMN mgp_id TO mcp_id;
ALTER TABLE amb_menu_group_permissions RENAME COLUMN mgp_menu_code TO mcp_menu_code;
ALTER TABLE amb_menu_group_permissions RENAME COLUMN mgp_accessible TO mcp_accessible;
ALTER TABLE amb_menu_group_permissions RENAME COLUMN mgp_created_at TO mcp_created_at;
ALTER TABLE amb_menu_group_permissions RENAME COLUMN mgp_updated_at TO mcp_updated_at;
ALTER TABLE amb_menu_group_permissions RENAME TO amb_menu_cell_permissions;
```

### 🟠 H-5. Response DTO 네이밍 패턴 급변

**현재 시스템 응답 형식** (`packages/types`):
```typescript
interface DepartmentResponse {
  departmentId: string;    // 도메인 의미 camelCase
  name: string;
  nameLocal: string | null;
  level: number;
  ...
}
```

**스펙 제안 응답 형식**:
```json
{
  "untId": "...",          // DB 컬럼 기반 camelCase
  "untName": "개발Unit",
  "untNameLocal": "Development Unit",
  "untLevel": 1
}
```

**문제점**: 응답 필드명이 `departmentId` → `untId`, `name` → `untName`으로 변경되면 프론트엔드 전체의 API 응답 파싱 로직을 재작성해야 함. 기존 `camelCase` 도메인 네이밍 패턴(`departmentId`, `name`)이 DB 컬럼 패턴(`untId`, `untName`)으로 바뀌는 것은 API 설계 컨벤션의 일관성을 해침.

**권장**: 기존 패턴 유지
```typescript
interface UnitResponse {
  unitId: string;          // departmentId → unitId (도메인 의미 유지)
  name: string;            // 변경 없음
  nameLocal: string | null;
  level: number;
  ...
}
```
