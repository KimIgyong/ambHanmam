- **작성일**: 2026-03-09
- **보고자**: 사용자 (스테이징 테스트 중 발견)

## 1. 현상

- **URL**: `https://stg-ama.amoeba.site/entity-settings/custom-apps`
- **API**: `POST /api/v1/entity-settings/custom-apps`
- **응답**: `400 Bad Request`
- 커스텀앱 등록 시 400 오류 발생

## 2. 원인 분석

### 2.1 API 흐름

```
프론트엔드 → POST /entity-settings/custom-apps (body: DTO)
                    ↓
컨트롤러 → resolveEntityId(queryEntityId, user)
                    ↓
queryEntityId = undefined (쿼리 파라미터 없음)
user.entityId = ???
```

### 2.2 resolveEntityId() 동작

```typescript
// resolve-entity-id.ts
const entityId = queryEntityId || user.entityId;
if (!entityId) throw new BadRequestException('entity_id is required');
```

### 2.3 JWT 토큰의 entityId 설정 로직

```typescript
// jwt.strategy.ts
1) JWT payload에서 entityId 추출
2) 없으면 primaryRole.unit?.entId 로 fallback (부서 소속 법인)
3) 둘 다 없으면 entityId = undefined
```

### 2.4 스테이징 DB 현황 — entityId 미할당 사용자

| 사용자 | 레벨 | 역할 | unit | entity | 결과 |
|--------|------|------|------|--------|------|
| `admin@amoeba.group` | ADMIN_LEVEL | SUPER_ADMIN | NULL | NULL | **400 오류** |
| `hyunwook.c@gmail.com` | USER_LEVEL | MASTER | NULL | NULL | **400 오류** |
| `gray.kim@amoeba.group` | USER_LEVEL | MASTER | 있음 | 있음 | 정상 |
| `sunnyson.vn@gmail.com` | USER_LEVEL | MASTER | 있음 | 있음 | 정상 |

### 2.5 근본 원인

**`/entity-settings/custom-apps` 페이지는 EntitySettingsGuard(MASTER or ADMIN 허용)로 보호되어 있어 ADMIN_LEVEL 사용자도 접근 가능하지만, ADMIN_LEVEL 사용자는 entityId가 없어 `resolveEntityId()`에서 400 오류 발생.**

추가로 MASTER이지만 unit/entity 미할당 사용자도 같은 문제.

### 2.6 DTO 유효성 검사 가능성

별도로, DTO 유효성 검사에서도 400이 발생할 수 있음:
- `url` 필드: `@IsUrl({ protocols: ['https'], require_protocol: true })` → HTTP URL 입력 시 거부
- `code` 필드: `/^[a-z0-9-]+$/` regex → 대문자/특수문자 포함 시 거부
- `forbidNonWhitelisted: true` → DTO에 정의되지 않은 필드 전송 시 거부

## 3. 사이드 임팩트 분석

### 3.1 이전 작업과의 관계

커스텀앱 관리 화면 분리 작업(PLAN-커스텀앱관리화면분리-20260309)에서 수정한 내용:

| 변경 사항 | 사이드 임팩트 |
|----------|-------------|
| `entity-settings.service.ts`: entityId 파라미터 추가 | **없음** — optional, 기존 호출 시 undefined → params 미전송 |
| `useEntitySettings.ts`: 훅에 entityId 지원 | **없음** — 기존 EntityCustomAppsPage는 인자 없이 호출 |
| `useCustomApps` enabled 조건 `entityId !== ''` | **없음** — undefined !== '' = true (기존과 동일 동작) |

**결론**: 이전 작업의 사이드 임팩트 아님. 이 400 오류는 기존부터 존재하던 문제.

### 3.2 영향 받는 기능

| 기능 | 영향 |
|------|------|
| USER_LEVEL 커스텀앱 관리 (entity 할당된 MASTER) | 영향 없음 — 정상 동작 |
| USER_LEVEL 커스텀앱 관리 (entity 미할당 MASTER) | **400 오류** |
| ADMIN_LEVEL이 USER_LEVEL 페이지 접근 | **400 오류** |
| ADMIN_LEVEL 커스텀앱 관리 (신규 `/admin/custom-apps`) | 영향 없음 — entityId 선택 필수 |

## 4. 요구사항

### 4.1 프론트엔드 개선

1. **EntityCustomAppsPage에서 entityId 없는 사용자 처리**: entity 미할당 사용자에게 안내 메시지 표시
2. **ADMIN_LEVEL이 USER_LEVEL 페이지 접근 시**: `/admin/custom-apps`로 리다이렉트하거나 에러 방지

### 4.2 백엔드 개선 (선택)

1. **resolveEntityId()에서 더 명확한 에러 메시지**: 현재 "entity_id is required"는 사용자에게 불친절
2. **OwnEntityGuard에서 entityId 사전 검증**: Guard 단계에서 entityId 존재 확인

### 4.3 DTO 유효성 검사 에러 UX 개선

1. **프론트엔드 폼 유효성 검사 강화**: URL HTTPS 체크, code 형식 체크를 프론트에서 사전 검증
2. **API 에러 메시지 표시**: 400 응답의 상세 validation 에러를 사용자에게 보여주기
