# REQ-이슈상태변경UI개선-20260331

## 1. 요구사항 요약

이슈 상세보기(`/issues`)의 상태 변경 UI를 개선한다.

**핵심 요구사항:**
- 다음 단계 상태 변경 **버튼** (좌측 정렬) + 모든 상태 **드롭다운** (우측 정렬) 동시 노출
- 현재: 드롭다운 `<select>` 하나로만 상태 변경
- TO-BE: `[OPEN] [APPROVED]` 같은 다음 단계 버튼 + `[모든 상태 드롭다운]` 분리 배치

## 2. AS-IS 현황 분석

### 2.1 현재 UI (IssueDetailModal.tsx #664-703)
```
┌─────────────────────────────────────────┐
│ Workflow                                │
│ ┌─────────────────────────────────────┐ │
│ │ ▼ 상태 변경 (select dropdown)      │ │
│ │   - 현재 상태 제외 모든 상태 노출   │ │
│ └─────────────────────────────────────┘ │
│ ┌─────────────────────────────────────┐ │
│ │ 워크플로우 노트 입력                │ │
│ └─────────────────────────────────────┘ │
└─────────────────────────────────────────┘
```

### 2.2 현재 상태 전이 규칙 (ISSUE_VALID_TRANSITIONS)
| 현재 상태 | 유효 다음 상태 |
|-----------|---------------|
| OPEN | APPROVED, REJECTED |
| APPROVED | IN_PROGRESS, REJECTED |
| IN_PROGRESS | TEST, RESOLVED |
| TEST | REOPEN, RESOLVED |
| REOPEN | IN_PROGRESS |
| RESOLVED | CLOSED, OPEN |
| CLOSED | REOPEN |
| REJECTED | OPEN |

### 2.3 권한 규칙 (프론트 + 백엔드 동시 적용)
| 상태 | 권한 조건 |
|------|----------|
| APPROVED | Manager만 (MANAGER/ADMIN/MASTER/SUPER_ADMIN) |
| REJECTED | Manager만 |
| CLOSED | Reporter 또는 MASTER/ADMIN만 |
| 기타 | Manager 또는 관련자 (reporter/assignee/participant) |

### 2.4 관련 코드 파일
| 파일 | 역할 |
|------|------|
| `apps/web/src/domain/issues/components/IssueDetailModal.tsx` | 이슈 상세 모달 (상태 변경 UI 포함) |
| `packages/types/src/domain.types.ts` | `ISSUE_VALID_TRANSITIONS` 상수 |
| `apps/web/src/locales/{ko,en,vi}/issues.json` | 번역 파일 (status, workflow 키) |
| `apps/api/src/domain/issues/service/issue.service.ts` | 백엔드 상태 변경 검증 |

### 2.5 현재 문제점
1. **드롭다운만 존재** — 다음 단계 상태를 빠르게 선택하기 어려움
2. **프론트 유효 전이 미적용** — `ISSUE_VALID_TRANSITIONS`를 사용하지 않고 전체 상태 나열 (백엔드에서만 검증)
3. **UX 비직관** — 워크플로우 흐름에 맞지 않는 상태가 모두 보임

## 3. TO-BE 요구사항

### 3.1 UI 레이아웃
```
┌───────────────────────────────────────────────────┐
│ Workflow                                          │
│ ┌──────────────────────────┐  ┌─────────────────┐ │
│ │ [승인] [반려]  (버튼들)  │  │ [▼ 상태 변경]   │ │
│ │ ← 좌측 정렬              │  │ → 우측 정렬     │ │
│ └──────────────────────────┘  └─────────────────┘ │
│ ┌───────────────────────────────────────────────┐ │
│ │ 워크플로우 노트 입력                          │ │
│ └───────────────────────────────────────────────┘ │
└───────────────────────────────────────────────────┘
```

### 3.2 좌측: 다음 단계 버튼 (Quick Actions)
- `ISSUE_VALID_TRANSITIONS[현재상태]`에서 가져온 유효 전이 상태만 버튼으로 노출
- 각 버튼은 상태별 색상 적용 (STATUS_COLORS 활용)
- 권한 조건 적용 (APPROVED/REJECTED → Manager만, CLOSED → Reporter/Master만)
- 클릭 시 즉시 `handleWorkflowAction(status)` 호출

### 3.3 우측: 모든 상태 드롭다운
- 현재 상태 제외 모든 상태 표시 (기존 로직 유지)
- 기존 권한 필터링 유지
- "▼ 상태 변경" 라벨 유지

### 3.4 워크플로우 노트
- 기존과 동일하게 하단에 유지

## 5. 사용자 플로우

1. 사용자가 이슈 상세 모달을 연다
2. Workflow 섹션에서:
   - **좌측**: 현재 상태의 유효 다음 상태가 버튼으로 표시됨
     - 예: 현재 OPEN → `[승인]` `[반려]` 버튼
     - 예: 현재 IN_PROGRESS → `[검수]` `[해결]` 버튼
   - **우측**: 드롭다운으로 모든 상태(권한 필터 적용) 선택 가능
3. 버튼 또는 드롭다운에서 상태 선택
4. 워크플로우 노트 입력 (선택사항)
5. 상태 변경 완료 → 토스트 메시지

## 6. 기술 제약사항

- **프론트엔드만 수정** — 백엔드 API, DB 변경 없음
- `ISSUE_VALID_TRANSITIONS`는 `@amb/types`에서 import하여 사용
- 기존 `handleWorkflowAction()` 함수 재사용
- i18n: 기존 `issues:status.{STATUS}` 키 재사용, 신규 키 최소 추가
- STATUS_COLORS 상수 재사용하여 버튼 스타일링
