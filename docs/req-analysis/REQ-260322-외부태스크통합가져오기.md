# 요구사항 분석서: 통합 외부 태스크 가져오기 (External Task Import)

- **문서번호**: REQ-외부태스크통합가져오기-20260322
- **작성일**: 2026-03-22
- **상태**: 작성 완료
- **범위**: USER_LEVEL (Entity별 설정)
- **지원 프로바이더**: Asana, Redmine, Jira, Notion (확장 가능)

---

## 1. 요구사항 요약

Entity(법인) 단위로 외부 프로젝트 관리 도구(Asana, Redmine, Jira, Notion)의 태스크/이슈를 AMA Issue로 가져오는 **통합 가져오기 프레임워크**를 구현한다.

### 설계 원칙: 공통 로직 + 서비스별 개별 설정

> **연동 로직(가져오기·변환·등록)은 공통 프레임워크로 1벌만 구현하고,**
> **각 서비스(Asana, Redmine, Jira 등)의 연결 설정(URL, Token, 인증)은 Entity Settings에서 서비스별로 개별 등록·관리한다.**

- **공통 영역**: Provider 인터페이스, 태스크 탐색 UI, Issue 변환/등록 로직, 매핑 테이블, Import 로그
- **서비스별 개별 영역**: 연결 정보(URL, Token), 인증 방식, API 호출 구현체, 필드 매핑 규칙
- **설정 위치**: Entity Settings → "외부 태스크 도구" 메뉴에서 서비스별 등록/설정/관리

### 핵심 변경사항
1. **기존 Redmine 마이그레이션** (ADMIN_LEVEL, SUPER_ADMIN 전용) → **USER_LEVEL Entity별 가져오기**로 이관
2. **Asana 가져오기** 신규 구현 (VN01 Entity에 PAT 연결 완료 상태)
3. **Jira, Notion** 추가 확장 가능한 **Provider 패턴** 설계
4. **외부 태스크 도구 설정** → Entity Settings에서 서비스별 개별 등록/설정/관리
5. 통합 UI: 프로바이더 선택 → 프로젝트/보드 탐색 → 태스크 선택 → AMA Issue 등록

## 2. AS-IS 현황 분석

### 2.1 기존 Redmine 마이그레이션 시스템 (ADMIN_LEVEL)

| 항목 | 현황 |
|------|------|
| **위치** | `apps/api/src/domain/migration/` |
| **접근 권한** | `@Roles('SUPER_ADMIN')` — 시스템 관리자 전용 |
| **인증 정보** | `amb_api_keys` 테이블 (provider: REDMINE, REDMINE_URL), 시스템 레벨 저장 |
| **기능** | 전체 가져오기(import-all), 선택 가져오기(import-selected), 재가져오기(reimport), 롤백 |
| **ID 추적** | `amb_issues.iss_redmine_id` (INT) |
| **사용자 매핑** | `amb_migration_user_maps` 테이블 (이메일 기반) |
| **로그** | `amb_migration_logs` 테이블 (batch_id 기반) |
| **프론트엔드** | `apps/web/src/domain/admin/pages/RedmineMigrationPage.tsx` |
| **문제점** | Entity 격리 없음, SUPER_ADMIN만 접근 가능, 단일 Redmine 서버만 지원 |

### 2.2 Entity Custom App 시스템 (활용 대상)

| 항목 | 현황 |
|------|------|
| **테이블** | `amb_entity_custom_apps` |
| **인증 모드** | `jwt`, `none`, `api_key` |
| **API Key 저장** | AES-256-GCM 암호화 (`eca_api_key_enc`) |
| **복호화** | `EntityCustomAppService.generateAppToken()` → `decrypt()` |
| **현재 설정** | VN01 Entity에 `eca_code: 'asana'`, `eca_auth_mode: 'api_key'` 로 PAT 저장 완료 |

### 2.4 Entity Settings UI 현황

| 항목 | 현황 |

| **문제점** | Custom Apps는 범용 외부 앱 연동용 — 외부 태스크 도구 전용 설정 UI 없음 |

| 항목 | 현황 |

## 3. TO-BE 요구사항

### 3.1 아키텍처 방향: Provider Pattern

프로바이더 추가 시 코드 변경을 최소화하기 위한 **플러그인 아키텍처**:

```
┌─────────────────────────────────────────────┐
│          ExternalTaskImportModule             │
│                                              │
│  ┌─────────────────────────────────────┐     │
│  │     ExternalTaskImportService        │     │
│  │  (통합 오케스트레이터)                │     │
│  └──────────┬──────────────────────────┘     │
│             │                                │
│  ┌──────────▼──────────────────────────┐     │
│  │     TaskImportProviderRegistry       │     │
│  │  (프로바이더 레지스트리)              │     │
│  └──┬────┬────┬────┬───────────────────┘     │
│     │    │    │    │                         │
│  ┌──▼─┐┌─▼──┐┌▼──┐┌▼────┐                   │
│  │Asana││Redm││Jira││Notion│  ← Provider     │
│  │Prov.││Prov.││Prov││Prov. │    구현체       │
│  └────┘└────┘└───┘└─────┘                   │
│                                              │
│  공통 인터페이스: ExternalTaskProvider        │
│  - fetchProjects()                           │
│  - fetchSections/Boards/Lists()              │
│  - fetchTasks()                              │
│  - convertToIssue()                          │
└─────────────────────────────────────────────┘
```

### 3.2 기능 요구사항 (Functional)

#### FR-01. 통합 Provider 인터페이스

| 항목 | 설명 |
|------|------|
| **설명** | 모든 외부 도구가 같은 인터페이스를 구현하여 통합 처리 |
| **인터페이스** | `ExternalTaskProvider` (아래 상세 정의) |
| **등록** | NestJS Provider 패턴으로 자동 등록, `PROVIDER_TYPE` 상수로 식별 |

```typescript
interface ExternalTaskProvider {
  readonly providerType: ProviderType;  // 'asana' | 'redmine' | 'jira' | 'notion'
  readonly displayName: string;
  readonly icon: string;

  // 연결 테스트
  testConnection(config: ProviderConnectionConfig): Promise<ConnectionTestResult>;

  // 계층 탐색 (프로젝트 → 보드/섹션 → 태스크)
  fetchProjects(config: ProviderConnectionConfig): Promise<ExternalProject[]>;
  fetchGroups(config: ProviderConnectionConfig, projectId: string): Promise<ExternalGroup[]>;
  fetchTasks(config: ProviderConnectionConfig, groupId: string, options?: FetchTaskOptions): Promise<PaginatedResult<ExternalTask>>;

  // 변환
  convertToIssueData(task: ExternalTask, defaults: ImportDefaults): CreateIssueFromExternalDto;
}

// 프로바이더별 "그룹" 개념 매핑
// Asana: Section
// Redmine: Tracker/Version/Category
// Jira: Sprint/Epic/Board
// Notion: Database View / Filter
```

#### FR-02. Entity Settings에서 서비스별 연결 설정 관리

| 항목 | 설명 |
|------|------|
| **설명** | Entity 관리자가 **Entity Settings → 외부 태스크 도구** 메뉴에서 서비스별 연결 정보를 등록/설정/관리 |
| **위치** | Entity Settings 메인 → "외부 태스크 도구" 카드 클릭 → 서비스별 설정 페이지 |
| **저장** | Entity Custom App 시스템 활용 (`eca_code`: 'asana', 'redmine', 'jira', 'notion') |
| **인증 방식** | 프로바이더별 상이 (아래 표 참조) |
| **복수 설정** | 같은 프로바이더도 Entity당 여러 인스턴스 가능 (예: Redmine 서버 2개) |
| **서비스별 개별 관리** | 각 서비스(탭)마다 독립된 설정 폼 — URL, Token, 연결 테스트 등 서비스 특성에 맞는 UI |
| **권한** | Entity MASTER / MANAGER |

**프로바이더별 인증 방식:**

| Provider | 인증 방식 | 저장 필드 | API Base URL |
|----------|----------|----------|--------------|
| **Asana** | PAT (Bearer Token) | `eca_api_key_enc` = PAT | `https://app.asana.com/api/1.0` |
| **Redmine** | API Key | `eca_api_key_enc` = API Key, `eca_url` = Server URL | `{eca_url}` (자체 서버) |
| **Jira** | Email + API Token (Basic Auth) | `eca_api_key_enc` = `email:token` (Base64) | `https://{domain}.atlassian.net/rest/api/3` |
| **Notion** | Internal Integration Token | `eca_api_key_enc` = `secret_xxx` | `https://api.notion.com/v1` |

#### FR-06. UI 분리: 설정 vs 가져오기

외부 태스크 연동의 **설정**과 **사용(가져오기)**은 분리된 UI에서 수행한다.

**A. 설정 UI — Entity Settings**

| 항목 | 설명 |
|------|------|
| **위치** | Entity Settings → "외부 태스크 도구" 메뉴 |
| **기능** | 서비스별(Asana/Redmine/Jira/Notion) 연결 등록·수정·삭제, 연결 테스트, 활성/비활성 전환 |
| **권한** | Entity MASTER / MANAGER |
| **접근** | Entity Settings 메인 페이지 카드 목록에 "외부 태스크 도구" 카드 추가 |

**B. 가져오기 UI — Issue 영역**

| 항목 | 설명 |
|------|------|
| **위치** | Issue 목록 상단 "외부 가져오기" 또는 Issue 메뉴 하위 |
| **기능** | (설정 완료된) 프로바이더 선택 → 프로젝트/그룹 탐색 → 태스크 선택 → AMA Issue 등록 |
| **권한** | Entity MASTER / MANAGER |
| **전제조건** | Entity Settings에서 최소 1개 서비스 연결이 활성 상태여야 함 |
| **미설정 안내** | 연결된 서비스 없을 시 "Entity Settings에서 외부 태스크 도구를 먼저 설정하세요" 안내 + 설정 이동 링크 |

## 6. API 설계

### 6.1 서비스 설정 엔드포인트 (Entity Settings 영역)

기존 Entity Custom App API(`/api/v1/entity-settings/custom-apps`)를 활용하되, 외부 태스크 도구 전용 조회 API를 추가한다.

| Method | Path | 설명 |
|--------|------|------|
| `GET` | `/api/v1/entity-settings/external-task-tools` | 등록된 외부 태스크 도구 목록 (Custom App 중 provider 코드 필터) |
| `POST` | `/api/v1/entity-settings/external-task-tools` | 외부 태스크 도구 연결 등록 (Custom App 생성 래퍼) |
| `PATCH` | `/api/v1/entity-settings/external-task-tools/:id` | 연결 설정 수정 |
| `DELETE` | `/api/v1/entity-settings/external-task-tools/:id` | 연결 삭제 |
| `POST` | `/api/v1/entity-settings/external-task-tools/:id/test` | 연결 테스트 (프로바이더별 API 호출) |

> 💡 내부적으로 `amb_entity_custom_apps` 테이블을 사용하되, `eca_code`가 `asana`, `redmine`, `jira`, `notion` 중 하나인 레코드만 필터링한다.

### 6.2 태스크 가져오기 엔드포인트 (Issue 영역)

| Method | Path | 설명 |
|--------|------|------|
| `GET` | `/api/v1/issues/external/providers` | 사용 가능한 프로바이더 목록 (Entity Settings에서 등록+활성 상태인 것만) |
| `GET` | `/api/v1/issues/external/:provider/projects` | 프로젝트 목록 조회 |
| `GET` | `/api/v1/issues/external/:provider/projects/:projectId/groups` | 그룹(섹션/보드) 목록 |
| `GET` | `/api/v1/issues/external/:provider/groups/:groupId/tasks` | 태스크 목록 (미리보기) |
| `POST` | `/api/v1/issues/external/:provider/import` | 선택한 태스크 일괄 가져오기 |
| `GET` | `/api/v1/issues/external/imported` | 가져온 Issue 목록 (매핑 정보 포함) |
| `GET` | `/api/v1/issues/external/logs` | 가져오기 실행 이력 |

### 6.3 Provider URL Parameter

`:provider` 값: `asana` | `redmine` | `jira` | `notion`

Query parameter `?app_id={eca_id}`로 같은 프로바이더의 복수 연결 중 특정 연결 지정.

### 6.4 공통 응답 형식

```typescript

interface ExternalProject {

  description?: string;

  id: string;
  name: string;
  taskCount?: number;    // 태스크 수 (지원 시)
}

// 태스크 (통합 형식)
interface ExternalTask {
  id: string;            // 외부 시스템 ID
  title: string;
  description?: string;
  assignee?: string;     // 담당자 이름
  assigneeEmail?: string;
  status?: string;       // 외부 시스템 상태 (원본 텍스트)
  isCompleted: boolean;
  priority?: string;     // 외부 시스템 우선순위 (원본 텍스트)
  startDate?: string;    // ISO date
  dueDate?: string;      // ISO date
  tags?: string[];
  url?: string;          // 외부 시스템 permalink
  createdAt?: string;
  updatedAt?: string;
  // AMA 등록 여부 (서버에서 매핑 테이블 조회 후 표시)

  importedIssueId?: string;   // 이미 등록된 경우 AMA Issue ID

// 페이지네이션 (통합)
interface PaginatedResult<T> {

  hasMore: boolean;
  nextCursor?: string;   // offset 또는 cursor token
```

## 7. 프로바이더별 상세 사양

### 7.1 Asana Provider

| 항목 | 사양 |
|------|------|
| **인증** | `Authorization: Bearer {PAT}` |
| **Base URL** | `https://app.asana.com/api/1.0` |
| **프로젝트 조회** | `GET /projects?workspace={workspace_gid}&opt_fields=name,notes` |
| **섹션(그룹) 조회** | `GET /projects/{project_gid}/sections` |
| **태스크 조회** | `GET /sections/{section_gid}/tasks?opt_fields=name,notes,assignee,assignee.name,assignee.email,due_on,start_on,completed,tags,tags.name,permalink_url,created_at,modified_at` |
| **페이지네이션** | offset 토큰 기반, limit max 100 |
| **Rate Limit** | 1,500 req/min |
| **특이사항** | workspace_gid 필요 (PAT 소유자의 기본 워크스페이스 사용 또는 Custom App URL에서 파싱) |

### 7.2 Redmine Provider

| 항목 | 사양 |
|------|------|
| **인증** | `X-Redmine-API-Key: {api_key}` |
| **Base URL** | `{eca_url}` (Entity Custom App URL 필드 활용) |
| **프로젝트 조회** | `GET /projects.json?limit=100&offset=0` |
| **그룹 조회** | Tracker 또는 Version 기반 — `GET /projects/{id}/versions.json` |
| **태스크(이슈) 조회** | `GET /issues.json?project_id={id}&status_id=*&limit=100&offset=0` |
| **태스크 상세** | `GET /issues/{id}.json?include=journals` |
| **페이지네이션** | offset/limit 기반, limit max 100 |
| **Rate Limit** | 서버별 상이 |
| **타입 매핑** | 기존 `TRACKER_MAP`, `STATUS_MAP`, `PRIORITY_MAP` 재활용 |
| **특이사항** | 기존 `RedmineApiService` 로직 대부분 재활용 가능 |

### 7.3 Jira Provider

| 항목 | 사양 |
|------|------|
| **인증** | Basic Auth: `email:api_token` → Base64 → `Authorization: Basic {encoded}` |
| **Base URL** | `https://{domain}.atlassian.net/rest/api/3` |
| **프로젝트 조회** | `GET /project/search?maxResults=50&startAt=0` |
| **그룹 조회** | Board/Sprint: `GET /rest/agile/1.0/board/{boardId}/sprint` 또는 Epic/Label 필터 |
| **태스크(이슈) 조회** | `GET /search?jql=project={key} AND sprint={sprintId}&fields=summary,description,assignee,priority,status,duedate,created,updated&maxResults=50&startAt=0` |
| **페이지네이션** | `startAt` + `maxResults` (offset 기반) |
| **Rate Limit** | 자체 관리 (429 시 Retry-After) |
| **타입 매핑** | Bug→BUG, Story→FEATURE_REQUEST, Task→TASK, Epic→FEATURE_REQUEST |
| **특이사항** | Description이 ADF(Atlassian Document Format) → 텍스트 변환 필요 |

**Jira Custom App 설정:**
- `eca_code`: `jira`
- `eca_url`: `https://{domain}.atlassian.net` (Jira Cloud 도메인)
- `eca_api_key_enc`: `email:api_token` 형태로 저장 (Base64 인코딩 전 원본)

### 7.4 Notion Provider

| 항목 | 사양 |
|------|------|
| **인증** | `Authorization: Bearer {internal_integration_token}`, `Notion-Version: 2022-06-28` |
| **Base URL** | `https://api.notion.com/v1` |
| **프로젝트 조회** | `POST /search` (filter: { object: "database" }) — Integration에 공유된 DB 목록 |
| **그룹 조회** | Database의 Status/Select property 값 목록 (DB 스키마에서 추출) |
| **태스크(페이지) 조회** | `POST /databases/{database_id}/query` (filter/sorts 지원) |
| **페이지네이션** | cursor 기반 (`start_cursor`, `has_more`) |
| **Rate Limit** | 3 req/sec (Average), Burst 허용 |
| **타입 매핑** | Database property 이름 기반 동적 매핑 |
| **특이사항** | Property 이름이 프로젝트마다 다름 → 사용자가 필드 매핑 설정 필요 |

**Notion Custom App 설정:**
- `eca_code`: `notion`
- `eca_url`: (사용 안 함 — API 고정)
- `eca_api_key_enc`: `secret_xxx` (Internal Integration Token)

## 8. 시스템 흐름도

### 8.1 설정 흐름 (Entity Settings)

```
Entity 관리자 → Entity Settings → "외부 태스크 도구" 카드 클릭
  → Frontend: ExternalTaskToolsPage 표시 (/entity-settings/external-task-tools)
  → 서비스별 탭 표시: [Asana] [Redmine] [Jira] [Notion]
  → Asana 탭 선택:
    → Step 1: 연결 이름 입력 + PAT 입력
    → Step 2: [연결 테스트] 클릭
      → API: POST /entity-settings/external-task-tools/:id/test
      → Backend: PAT로 Asana /users/me 호출 → 워크스페이스 정보 반환
    → Step 3: [저장] → Custom App 레코드 생성
      → API: POST /entity-settings/external-task-tools
      → eca_code='asana', eca_auth_mode='api_key', eca_api_key_enc=encrypt(PAT)
  → 결과: Asana 연결 🟢 활성
```

### 8.2 가져오기 흐름 (Issue 영역)

```
사용자 → Issue 목록 → "외부 가져오기" 클릭
  → Frontend: ExternalTaskImportPage 표시
  → Step 1: 프로바이더 선택
    → API: GET /issues/external/providers
    → Entity Settings에서 등록+활성 상태인 프로바이더 목록 반환
    → [Asana ✓] [Redmine ✓] [Jira ✗] [Notion ✗]  (✓=설정됨)
    → 미설정 시: "Entity Settings에서 외부 태스크 도구를 먼저 설정하세요" 안내
  → Step 2: 프로젝트 탐색
    → API: GET /issues/external/asana/projects?app_id=xxx
    → Backend: Custom App에서 PAT 복호화 → Asana API 호출
    → 프로젝트 드롭다운 표시
  → Step 3: 그룹(섹션) 선택
    → API: GET /issues/external/asana/projects/{gid}/groups?app_id=xxx
    → 섹션 드롭다운 표시
  → Step 4: 태스크 미리보기
    → API: GET /issues/external/asana/groups/{gid}/tasks?app_id=xxx
    → 태스크 목록 + 중복 표시 (amb_external_task_mappings 조회)
  → Step 5: 선택 & 등록
    → API: POST /issues/external/asana/import
    → 결과: { imported: 5, skipped: 2, failed: 0 }
```

### 8.3 Provider 호출 내부 흐름

```
Controller (엔드포인트)
  → query/body에서 provider, app_id 추출
  → ExternalTaskImportService.fetchTasks(provider, appId, groupId, options)
    → ProviderRegistry.getProvider(provider)  → AsanaProvider 인스턴스
    → EntityCustomAppService에서 eca_id=appId 조회 → PAT 복호화
    → provider.fetchTasks(config, groupId, options)
      → HTTP fetch('https://app.asana.com/api/1.0/sections/{gid}/tasks', { headers })
      → 응답 → ExternalTask[] 표준 형식 변환
    → amb_external_task_mappings에서 기 등록 여부 조회
    → 결과에 alreadyImported 플래그 부착
  → 응답 반환
```

## 10. 보안 요구사항

| 항목 | 구현 방법 |
|------|----------|
| **인증 정보** | Entity Custom App의 AES-256-GCM 암호화 (기존 인프라) |
| **서버 사이드** | 모든 외부 API 호출은 Backend에서만 수행, Token 프론트엔드 노출 없음 |
| **SSRF 방지** | Redmine: URL 허용 목록 또는 내부 IP 차단 검증 |
| **권한** | Entity MASTER/MANAGER 이상만 가져오기 기능 접근 |
| **감사** | `amb_external_import_logs` 테이블에 모든 실행 기록 |
| **Rate Limit** | 프로바이더별 제한 준수, 429 응답 시 Retry-After 적용 |

## 11. 기존 Redmine 마이그레이션 이관 전략

### 11.1 공존 기간 (Transition)

| 항목 | AS-IS (유지) | TO-BE (신규) |
|------|-------------|-------------|
| **경로** | `/migration/redmine/*` | `/issues/external/redmine/*` |
| **권한** | SUPER_ADMIN | Entity MASTER/MANAGER |
| **인증** | `amb_api_keys` (시스템) | `amb_entity_custom_apps` (Entity) |
| **ID 추적** | `iss_redmine_id` (컬럼) | `amb_external_task_mappings` (테이블) |
| **UI** | Admin → Redmine Migration | Issue → 외부 가져오기 → Redmine |

### 11.2 데이터 호환

- 기존 `iss_redmine_id` 데이터는 유지 (삭제하지 않음)
- 신규 가져오기는 `amb_external_task_mappings`에 기록
- 중복 체크: 새 시스템에서도 `iss_redmine_id` 체크하여 기존 마이그레이션 건과 중복 방지

### 11.3 RedmineProvider에서 기존 코드 재활용 목록

| 기존 코드 | 재활용 방식 |
|----------|------------|
| `RedmineApiService.fetchProjects()` | `RedmineProvider.fetchProjects()` 내부 구현 |
| `RedmineApiService.fetchIssues()` | `RedmineProvider.fetchTasks()` 내부 구현 |
| `RedmineApiService.get()` HTTP 헬퍼 | 그대로 활용 (인증 방식 동일) |
| `TRACKER_MAP`, `STATUS_MAP`, `PRIORITY_MAP` | `RedmineProvider.convertToIssueData()` 에서 활용 |
| `RedmineMigrationService.importSelectedIssues()` | `ExternalTaskImportService.importTasks()` 로 통합 |

## 12. 기술 제약사항 및 리스크

| # | 제약/리스크 | 영향 | 대응 |
|---|-----------|------|------|
| 1 | Asana PAT은 개인 토큰 | 권한 범위가 PAT 소유자에 한정 | Service Account 또는 관리자 PAT 사용 권장 |
| 2 | Jira ADF(Description) | 복잡한 마크업 → 텍스트 변환 필요 | 간단한 ADF→텍스트 변환기 구현 또는 라이브러리 |
| 3 | Notion Property 이름 가변 | DB마다 프로퍼티 이름이 다름 | Title property 자동 감지 + 필드 매핑 설정 UI(Phase 2) |
| 4 | Redmine 자체 서버 접근 | VPN/방화벽 제약 가능 | 서버에서 직접 접근 가능해야 함 (SSRF 주의) |
| 5 | 대량 가져오기 타임아웃 | 100건 이상 시 HTTP 타임아웃 | 50건 이하: 동기 처리, 이상: 백그라운드 + 폴링 |
| 6 | 담당자 매핑 불일치 | 외부 사용자 ≠ AMA 사용자 | 이메일 기반 자동 매핑, 미매핑 시 null |
| 7 | Rate Limit 초과 | Notion 3 req/sec 엄격 | 프로바이더별 throttle 적용 |
| 8 | 기존 마이그레이션과 중복 | Redmine 동일 이슈 이중 등록 | `iss_redmine_id` + `amb_external_task_mappings` 모두 체크 |

## 13. 범위 외 사항 (Out of Scope)

- AMA Issue → 외부 시스템 역방향 Push (양방향 동기화)
- Webhook 기반 실시간 자동 동기화 (수동 가져오기만 Phase 1)
- 외부 시스템 코멘트 동기화
- 외부 시스템 첨부파일 다운로드/업로드
- 서브태스크 재귀 가져오기 (1 depth만)
- Notion 페이지 본문 전체 가져오기 (Property 기반 데이터만)
- GitHub Issues 연동 (기존 `iss_github_id` 있지만 별도 요구사항)
- Linear, ClickUp, Monday.com 등 추가 프로바이더 (Provider 패턴으로 추후 확장 가능)

## 14. 구현 우선순위

| Phase | 프로바이더 | 사유 |
|-------|-----------|------|
| **Phase 1** | **Asana** + **Redmine** | Asana: PAT 이미 설정됨. Redmine: 기존 코드 재활용 가능 |
| **Phase 2** | **Jira** | 가장 널리 사용되는 프로젝트 관리 도구 |
| **Phase 3** | **Notion** | Property 매핑 복잡도 높음, 별도 설정 UI 필요 |

통합 프레임워크(인터페이스, 테이블, 컨트롤러, UI)는 Phase 1에서 모두 구축하고, Phase 2/3에서는 Provider 구현체만 추가하는 구조.
