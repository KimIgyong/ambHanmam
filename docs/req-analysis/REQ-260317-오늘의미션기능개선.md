## 1. AS-IS 현황 분석

### 1.1 데이터 모델 (`amb_daily_missions`)

| 컬럼 | 타입 | 설명 | 비고 |
|------|------|------|------|
| `msn_id` | UUID | PK | |
| `ent_id` | UUID | 엔티티(법인) FK | |
| `usr_id` | UUID | 사용자 FK | |
| `msn_date` | DATE | 미션 날짜 | 복합 유니크: (ent_id, usr_id, msn_date) |
| `msn_content` | TEXT | 미션 HTML 콘텐츠 (TipTap 리치 텍스트) | nullable |
| `msn_check_result` | VARCHAR(20) | 달성도 결과 | `HALF` / `PARTIAL` / `ALL_DONE` / `EXCEED` |
| `msn_check_score` | SMALLINT | 달성도 점수 | `50` / `75` / `100` / `110` |

**핵심 발견**:
- `msn_registered_lines` JSONB 컬럼이 **미존재** → 신규 추가 필요
- `msn_check_score`는 고정 4단계(50/75/100/110)만 허용 → 자동 계산(0~100) 범위로 확장 필요
- `msn_check_result`는 4종(`HALF`/`PARTIAL`/`ALL_DONE`/`EXCEED`) → 초안 제안의 4종(`ZERO`/`PARTIAL`/`GOOD`/`PERFECT`)과 **값이 다름** — 하위 호환 전환 필요

### 1.2 현재 어제 미션 체크 방식

#### 백엔드 흐름

| 엔드포인트 | 메서드 | 설명 |
|-----------|--------|------|
| `GET /api/v1/today/me` | GET | 오늘 미션 + 어제 미평가 미션 반환 |
| `PATCH /api/v1/today/mission/:date/check` | PATCH | 체크 결과 저장 |

**현재 check API Body:**
```typescript
{ result: string; score: number }
// result ∈ ['HALF', 'PARTIAL', 'ALL_DONE', 'EXCEED']
// score  ∈ [50, 75, 100, 110]
```

**현재 `saveCheck` 서비스** (`mission.service.ts` 93~107줄):
- 단순히 `msn_check_result`, `msn_check_score` 컬럼에 매핑 저장
- 라인별 분석, Todo 생성, carry-over 등 **일체 없음**
- 유효성 검증: 컨트롤러에서 result/score 값 in-list 체크만 수행

#### 프론트엔드 흐름

**파일**: `apps/web/src/domain/today/components/mission/MissionSection.tsx` (260줄)

**현재 Step 구조:**
```
Step 1: 어제 미션 체크 (yesterdayMission 존재 && 미평가 시)
  ├─ 어제 미션 HTML 전체 표시
  ├─ 4개 버튼: 📉50% | 📈75% | ✅100% | 🚀110%
  └─ 클릭 → useSaveMissionCheck.mutate({ date, result, score })

Step 2: 오늘 미션 작성 (mission = null)
  ├─ RichTextEditor
  └─ 저장 / 넘기기

Step 3: 미션 완성 상태 (mission.msnContent 존재)
  ├─ HTML 렌더링 + 수정 + 번역
  └─ 달성도 표시 (msnCheckScore %)

Step 4: 넘기기 완료 (mission.msnContent = null)
  └─ "오늘은 미션을 넘겼습니다"
```

**문제점 (초안 P-01 ~ P-04와 일치):**
1. 어제 미션을 **통째로** 표시하고 **하나의 버튼**으로 전체 평가 → 개별 라인 세부 평가 불가
2. 미완료 항목에 대한 **후속 처리 없음** (Task 이관, 오늘 미션 이월 등)
3. 달성도가 **사용자 주관적 선택** → 데이터 신뢰도 저하
4. 메모/해당없음 구분 없이 **모든 내용이 평가 대상**

### 1.3 연관 시스템 현황

#### Todo 모듈 (최근 개선 완료)
- **Computed Status 6단계** 도입 완료 (`OVERDUE`/`DUE_TODAY`/`IN_PROGRESS`/`UPCOMING_SOON`/`UPCOMING`/`COMPLETED`)
- `PATCH /api/v1/todos/:id/complete` / `PATCH /api/v1/todos/:id/reopen` 엔드포인트 존재
- `tdo_start_date` DATE nullable 컬럼 존재
- `tdo_due_date` DATE nullable 변경 완료
- `tdo_completed_at` TIMESTAMP 컬럼 존재 (완료 시각)
- `tdo_source` 컬럼 **미존재** — 미션에서 생성된 Todo 추적 불가

#### Today Service (`today.service.ts`)
- `getMissionsByUsers()` — 팀/전사 미션 조회 시 `msn_check_score`, `msn_check_result` 반환
- 점수 범위가 0~100으로 확장되면 기존 팀 통계의 평균 계산에 영향 없음 (이미 number 타입)

#### Snapshot 시스템
- 미션 저장 시 자동 스냅샷 생성 (`SnapshotService.createSnapshot`)
- 체크 결과 저장 시 스냅샷 **미생성** — 변경사항 없음

| ID | 요구사항 | 우선순위 | 적용 가능성 | 현재 시스템 상태 |

**msn_check_result 매핑 변경:**

| 범위 | 현재(AS-IS) | 변경 후(TO-BE) |
|------|------------|--------------|
| 100% | `ALL_DONE` | `PERFECT` |
| 75~99% | `PARTIAL` | `GOOD` |
| 1~74% | `HALF` | `PARTIAL` |
| 0% | — (미존재) | `ZERO` |
| 110% (초과) | `EXCEED` | 제거 (자동 계산 시 max 100%) |
| null | — | `null` (전체 NA) |

**하위 호환 처리**:
- 기존 저장된 `HALF`/`PARTIAL`/`ALL_DONE`/`EXCEED` 값은 읽기 전용 유지
- 팀 통계에서 `msnCheckScore`를 number로 이미 취급하므로 범위 확장에 문제 없음
- 프론트엔드에서 `msnCheckResult` 라벨 표시 시 기존+신규 값 모두 매핑

### 4.2 서버 처리 플로우

```
PATCH /today/mission/:date/check
  │
  ├─ Request Body: { lines: MissionCheckLineDto[], carryOverText: string | null }
  │
  ├─ Validation
  │   ├─ 모든 line.state ∈ ['done', 'incomplete', 'na']
  │   ├─ state = 'incomplete' → subChoice ∈ ['mission', 'task'] 필수
  │   └─ state = 'done' / 'na' → subChoice = null
  │
  ├─ Line Processing (순회)
  │   ├─ state = 'done'
  │   │   └─ amb_todos INSERT: title=text, tdo_completed_at=NOW(), tdo_start_date=어제, tdo_status='COMPLETED'
  │   ├─ state = 'incomplete' + subChoice = 'task'
  │   │   └─ amb_todos INSERT: title=text, tdo_start_date=오늘, tdo_completed_at=NULL, tdo_status='IN_PROGRESS'
  │   ├─ state = 'incomplete' + subChoice = 'mission'
  │   │   └─ carryOverLines에 추가 (DB Todo INSERT 안함)
  │   └─ state = 'na'
  │       └─ skip (DB 작업 없음)
  │
  ├─ Score Calculation
  │   ├─ evalTotal = count(state != 'na')
  │   ├─ doneCount = count(state == 'done')
  │   ├─ score = evalTotal > 0 ? Math.round(doneCount / evalTotal * 100) : null
  │   └─ result = computeCheckResult(score)
  │
  ├─ DB UPDATE
  │   └─ amb_daily_missions SET msn_check_score, msn_check_result, msn_registered_lines, msn_carry_over_text
  │
  └─ Response: { msnId, checkScore, checkResult, registeredLines[], carryOverText }
```

## 5. 기술 제약사항

### 5.1 TipTap HTML 파싱 이슈

- 미션은 TipTap 리치 텍스트 에디터로 작성됨 → HTML 구조가 다양
- `<p>`, `<li>`, `<h1>`~`<h6>` 등 다양한 블록 요소 처리 필요
- `<br>` 단독은 빈 줄로 간주하여 제외
- 인라인 요소(`<strong>`, `<em>`, `<a>` 등)는 텍스트에 포함하여 plain text 추출
- **파싱 전략**: 프론트엔드 DOMParser로 처리, 서버에는 파싱된 결과(lineIndex + text)만 전송

### 5.2 Todo 생성 시 필드 매핑

| Todo 필드 | DONE 시 | INCOMPLETE + Task 시 |
|----------|---------|---------------------|
| `tdo_title` | 라인 plain text | 라인 plain text |
| `tdo_start_date` | 어제 날짜 | 오늘 날짜 |
| `tdo_due_date` | NULL (이미 완료) | NULL (사용자가 추후 설정) |
| `tdo_completed_at` | NOW() | NULL |
| `tdo_status` | `'COMPLETED'` | `'IN_PROGRESS'` (computed) |
| `tdo_visibility` | `'PRIVATE'` (기본) | `'PRIVATE'` |
| `usr_id` | 현재 사용자 | 현재 사용자 |
| `ent_id` | 현재 법인 | 현재 법인 |

### 5.3 동시성 / 멱등성

- 하루에 미션 체크는 **1회만** 가능 (이미 `msn_check_result != NULL`이면 재체크 불가)
- 저장 API는 **트랜잭션으로 묶어야** 함 (부분 Todo INSERT 후 실패 시 롤백)
  - 단, 초안 NFR-MCK-003의 "개별 실패 허용"도 고려 → **정책 결정 필요**
  - **권장**: 전체 트랜잭션 (일부 Todo 실패 시 전체 롤백이 데이터 정합성에 유리)

### 5.4 모듈 의존성

- `today.module.ts`에서 `TodoEntity`는 이미 import되어 있음
- Todo 생성을 위해 `TodoService`를 주입하거나, `TodoEntity` Repository를 직접 사용
- **권장**: `TodoEntity` Repository 직접 사용 (TodoService의 복잡한 로직 불필요, 단순 INSERT만 필요)
