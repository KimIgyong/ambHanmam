# REQ-260413 이슈 코멘트 고객사 공개 제어

## 1. 요구사항 요약

프로젝트 관리 이슈의 코멘트(댓글)에 **고객사 공개 여부 제어** 기능을 추가한다.

- 내부 사용자(USER_LEVEL)가 이슈에 코멘트를 작성할 때, 프로젝트 매니저(또는 작성자)가 **"고객사에게 공개"** 옵션을 선택할 수 있어야 한다.
- CLIENT_LEVEL 사용자가 이슈 상세 페이지에서 코멘트를 조회할 때, **공개로 설정된 코멘트만** 표시되어야 한다.
- 고객사가 직접 작성한 코멘트는 자동으로 공개 상태로 처리한다.

---

## 2. AS-IS 현황 분석

### 2.1 이슈 코멘트 엔티티 (DB)

- **테이블**: `amb_issue_comments`
- **주요 컬럼**:

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `isc_id` | UUID (PK) | 코멘트 ID |
| `iss_id` | UUID (FK) | 이슈 ID |
| `isc_author_id` | UUID (FK) | 작성자 ID |
| `isc_author_type` | varchar(10) | 'USER' / 'AI' / 'SYSTEM' |
| `isc_content` | text | 코멘트 내용 |
| `isc_issue_status` | varchar(20) | 작성 시 이슈 상태 |
| `isc_parent_id` | UUID (nullable) | 부모 코멘트 ID (대댓글) |
| `isc_created_at` | timestamp | 생성일시 |

- **고객사 공개 여부 컬럼이 존재하지 않음** → 모든 코멘트가 동일하게 취급됨

### 2.2 내부 사용자 이슈 코멘트 API

- **엔드포인트**: `POST /api/v1/issues/:id/comments`
- **DTO**: `CreateIssueCommentRequest` → `{ content, parent_id? }`
- **서비스**: `IssueService.addIssueComment(issueId, content, userId, authorType, parentId)`
- **컨트롤러**: `IssueController` (`@UseGuards(EntityGuard)`)
- **공개 여부 파라미터 없음**

### 2.3 클라이언트 이슈 코멘트 API

- **이슈 상세 조회**: `GET /api/v1/client/issues/:id` → `ClientIssueService.findIssueById()`
  - 이슈 조회 시 `relations: ['comments', 'comments.author']`로 **모든 코멘트를 무조건 반환**
  - 공개/비공개 필터링 없음
- **코멘트 추가**: `POST /api/v1/client/issues/:id/comments` → `ClientIssueService.addComment()`
  - 고객이 작성하는 코멘트도 동일 구조
- **컨트롤러**: `ClientIssueController` (`@UseGuards(ClientGuard)`)

### 2.4 프론트엔드 (내부 사용자 - apps/web)

- **이슈 코멘트 UI**: `IssueDetailModal.tsx`
  - Rich Text Editor로 코멘트 작성
  - 대댓글, 리액션, 번역 지원
  - **공개 여부 선택 UI 없음**

### 2.5 프론트엔드 (클라이언트 - apps/web/client-portal)

- **이슈 상세 페이지**: `ClientIssueDetailPage.tsx`
  - 이슈 상세 응답의 `comments` 배열을 **전체 렌더링**
  - **공개/비공개 필터링 없음**
  - Rich Text Editor로 코멘트 작성

### 2.6 이슈 Visibility vs 코멘트 Visibility

- 이슈 자체에는 `iss_visibility` 컬럼 존재 (PRIVATE / CELL / ENTITY)
- 코멘트에는 별도 visibility 개념 없음
- 클라이언트 이슈 목록에서는 `issVisibility = 'ENTITY'` 필터 적용 중

## 3. TO-BE 요구사항

### 3.1 DB 변경

`amb_issue_comments` 테이블에 고객사 공개 여부 컬럼 추가:

| 컬럼 | 타입 | 기본값 | 설명 |
|------|------|--------|------|
| `isc_client_visible` | boolean | false | 고객사(CLIENT_LEVEL) 공개 여부 |

- 기본값 `false` → 내부 사용자가 작성하는 코멘트는 기본적으로 고객사 비공개
- CLIENT_LEVEL 사용자가 작성한 코멘트는 서비스 로직에서 자동으로 `true` 설정

### 3.2 백엔드 API 변경

#### A) 내부 사용자 코멘트 작성 API 수정

- **POST** `/api/v1/issues/:id/comments`
- **DTO 추가 필드**: `client_visible?: boolean` (optional, 기본값 false)
- **서비스 로직**: `IssueService.addIssueComment()`에 `clientVisible` 파라미터 추가

#### B) 내부 사용자 코멘트 공개 토글 API (신규)

- **PATCH** `/api/v1/issues/:id/comments/:commentId/client-visible`
- **DTO**: `{ client_visible: boolean }`
- **권한**: 코멘트 작성자 또는 프로젝트 매니저만 변경 가능
- 이미 작성된 코멘트의 공개 여부를 사후에 변경할 수 있는 기능

#### C) 클라이언트 이슈 상세 API 수정

- **GET** `/api/v1/client/issues/:id`
- **변경**: `findIssueById()`에서 코멘트 반환 시 `isc_client_visible = true`인 코멘트만 필터링
- 클라이언트가 작성한 코멘트(자기 코멘트)도 당연히 표시

#### D) 클라이언트 코멘트 작성 시 자동 공개

- **POST** `/api/v1/client/issues/:id/comments`
- **변경**: `ClientIssueService.addComment()`에서 `isc_client_visible = true` 자동 설정
- 고객이 작성한 코멘트는 항상 고객에게 표시되어야 하므로

#### E) 내부 사용자 이슈 코멘트 응답에 공개 여부 포함

- `IssueCommentResponse`에 `clientVisible: boolean` 필드 추가
- 내부 사용자 UI에서 공개/비공개 상태를 확인 및 토글 가능

### 3.3 프론트엔드 변경 (내부 사용자)

#### IssueDetailModal.tsx 변경

- 코멘트 작성 시 **"고객사에게 공개" 체크박스** 추가
- 기존 코멘트 목록에서 **공개 여부 뱃지/아이콘** 표시
  - 공개: 🔓 아이콘 + "Client visible" 태그
  - 비공개: 🔒 아이콘 (또는 표시 없음)
- 코멘트별 **공개/비공개 토글 버튼** (작성자 또는 PM만)

### 3.4 프론트엔드 변경 (클라이언트 포털)

#### ClientIssueDetailPage.tsx 변경

- **변경 없음** (백엔드에서 공개 코멘트만 필터링하여 반환하므로)
- 고객은 공개/비공개 여부를 알 필요 없음

## 5. 사용자 플로우

### 5.1 내부 사용자 (USER_LEVEL) - 코멘트 작성

```
1. 이슈 상세 모달 열기
2. 코멘트 입력 (Rich Text Editor)
3. [선택] "고객사에게 공개" 체크박스 ON/OFF
4. "댓글 작성" 버튼 클릭
5. API 호출: POST /issues/:id/comments { content, client_visible }
6. 코멘트 저장 (client_visible 값 포함)
7. 코멘트 목록 갱신 → 공개 여부 뱃지 표시
```

### 5.2 내부 사용자 (USER_LEVEL) - 기존 코멘트 공개 전환

```
1. 이슈 상세 모달에서 코멘트 목록 확인
2. 비공개 코멘트의 "고객 공개" 토글 버튼 클릭
3. API 호출: PATCH /issues/:id/comments/:commentId/client-visible
4. 공개 상태 변경 → 뱃지 업데이트
```

### 5.3 클라이언트 (CLIENT_LEVEL) - 코멘트 조회

```
1. 프로젝트 > 이슈 상세 페이지 열기
2. API 호출: GET /client/issues/:id
3. 서버에서 client_visible = true인 코멘트만 반환
4. 공개 코멘트만 렌더링 (비공개 코멘트는 존재 자체를 모름)
```

### 5.4 클라이언트 (CLIENT_LEVEL) - 코멘트 작성

```
1. 이슈 상세 페이지에서 코멘트 입력
2. API 호출: POST /client/issues/:id/comments
3. 서버에서 자동으로 client_visible = true 설정
4. 코멘트 저장 + 목록 갱신
```

## 6. 기술 제약사항

### 6.1 DB 마이그레이션

- 스테이징/프로덕션은 TypeORM `synchronize: false` → **수동 SQL 필수**
- 기존 코멘트의 `isc_client_visible` 기본값 → `false` (안전한 기본값, 비공개 처리)
- 마이그레이션 SQL:
  ```sql
  ALTER TABLE amb_issue_comments 
    ADD COLUMN isc_client_visible BOOLEAN NOT NULL DEFAULT false;
  ```

### 6.2 기존 데이터 호환성

- 기존 모든 코멘트는 `false`(비공개)로 설정됨
- 고객사가 이전에 볼 수 있었던 코멘트가 갑자기 안 보일 수 있음
- **이전 전략 옵션**:
  - A) 기존 코멘트 모두 `false` → PM이 수동 공개 전환 (안전, 권장)
  - B) 기존 코멘트 모두 `true` → 비밀 코멘트 노출 위험
  - C) CLIENT_LEVEL 사용자가 작성한 기존 코멘트만 `true` → 부분 이전
  - **권장: 옵션 C** (고객 작성 코멘트는 공개, 내부 코멘트는 비공개)

### 6.3 권한 정책

- **코멘트 공개 전환 권한**: 코멘트 작성자 또는 프로젝트의 PM(매니저)
- PM 여부 판단: `ProjectMemberEntity`에서 `pjm_role = 'PM'` 또는 `ProjectEntity.pjt_manager_id`
- ADMIN_LEVEL은 모든 코멘트 공개 전환 가능

### 6.4 대댓글(Reply) 처리

- 부모 코멘트가 비공개인데 대댓글이 공개일 경우 → 문맥 단절
- **정책**: 부모 코멘트가 비공개면 대댓글도 자동 비공개 처리
- 대댓글 작성 시 부모의 공개 상태를 상속

### 6.5 AI 코멘트 처리

- `isc_author_type = 'AI'` 코멘트 → 기본 비공개
- PM이 수동으로 공개 전환 가능

### 6.6 성능

- 클라이언트 이슈 상세 쿼리에 WHERE 조건 추가 → 인덱스 불필요 (코멘트 수가 이슈당 소량)
- 내부 사용자 쿼리에는 영향 없음 (필터링 없이 전체 반환)

### 6.7 i18n

- 새 UI 요소에 대한 번역 키 추가 필요 (ko/en/vi 3개 언어)
  - `issue.clientVisible`, `issue.clientHidden`, `issue.toggleClientVisible` 등
