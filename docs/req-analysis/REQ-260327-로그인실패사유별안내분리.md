> **작성일**: 2026-03-27

## 1. 요구사항 요약

### 1.1 로그인 실패 사유별 안내 메시지 분리

모든 로그인 페이지에서 로그인 실패 시, 실패 사유에 따라 안내 메시지를 분리한다.

| 실패 사유 | 안내 메시지 |
|-----------|------------|
| 이메일은 존재하지만 비밀번호가 틀림 | "비밀번호를 정확히 입력하세요" |
| 이메일 자체가 존재하지 않음 | "등록되지 않은 이메일입니다. 정확히 입력하시거나 회원가입을 해주세요" |

### 1.2 패스워드 리셋 페이지 Entity 정보 필수화

패스워드 리셋(forgot-password) 페이지에서도 로그인 페이지와 동일한 기준을 적용한다.
- **USER_LEVEL / CLIENT_LEVEL**: Entity(법인) 선택 **필수**
- **PARTNER_LEVEL**: Partner Code **필수**
- **ADMIN_LEVEL**: Entity 불요 (기존 유지)

> 현재 `/forgot-password` 페이지는 이미 Entity 필수로 구현 완료(이전 커밋). Client/Partner 포탈은 별도 forgot-password 페이지가 없으므로 신규 추가가 필요하다.

### 1.3 Reset Password 페이지에 Entity Name 노출

`/reset-password?token=X` 페이지에서 사용자가 어떤 법인 계정의 비밀번호를 재설정하는지 식별할 수 있도록 **Entity Name(=법인명)**을 화면에 표시한다.

- 토큰으로 사용자 조회 → 사용자의 `usrCompanyId`로 Entity 조회 → Entity Name 반환
- 프론트엔드에서 페이지 로드 시 토큰 검증 API 호출하여 Entity Name + 사용자 이메일 노출
- 토큰이 무효하거나 만료된 경우 에러 표시하고 로그인으로 리디렉트

### 3.1 백엔드 에러 체계

#### A. 메인 Auth (`POST /auth/login`) — `/login`, `/admin/login` 공용

### 4.5 Reset Password 페이지 Entity Name 노출

#### 현재 문제
- `/reset-password?token=X` 페이지에서 사용자가 어떤 법인 계정의 비밀번호를 리셋하는지 알 수 없음
- 동일 이메일로 여러 법인에 가입된 사용자의 경우 혼동 가능

#### 구현 방안

**백엔드: 토큰 검증 API 신규 추가**
- **엔드포인트**: `GET /auth/verify-reset-token?token=X` (Public)
- **응답**: `{ valid: true, email: "user@...co.kr", entityName: "아메바 베트남", entityCode: "VN01" }`
- **로직**: `amb_password_resets` 테이블 → `usr_id` → `UserEntity` → `usrCompanyId` → `HrEntityEntity.entName`
- **보안**: 만료/사용된 토큰은 `{ valid: false }` 반환 (에러 코드 포함)
- ADMIN_LEVEL 사용자(usrCompanyId=null)의 경우 `entityName: "System Admin"` 표시

**프론트엔드: ResetPasswordPage 개선**
- 페이지 로드 시 `verify-reset-token` API 호출
- 유효한 토큰이면 "**{Entity Name}**의 **{maskedEmail}** 계정 비밀번호 재설정" 형태로 표시
- 이메일은 마스킹 처리: `us****@amoeba.co.kr`
- 토큰 무효/만료 시 에러 페이지 표시 + 로그인으로 리다이렉트

## 5. 보안 고려사항 (중요)

### 5.1 사용자 열거(User Enumeration) 공격 위험

> **주의**: 이메일 존재 여부를 구분 응답하면 공격자가 유효한 이메일 주소를 수집할 수 있습니다(OWASP: Authentication Testing - User Enumeration).

#### 위험 수준 평가

| 로그인 페이지 | Entity/Partner Code 필요 | 열거 위험 |
|-------------|--------------------------|---------|
| `/login` | ✅ Entity Code 필수 | **낮음** — 법인코드 + 이메일 조합이 필요하여 무차별 열거 어려움 |
| `/client/login` | ✅ Entity Code 필수 | **낮음** — 법인코드 + 이메일 조합 필요 |
| `/admin/login` | ❌ 불요 | **중간** — 이메일만으로 존재 여부 확인 가능 |
| `/partner/login` | ✅ Partner Code 필수 | **낮음** — 파트너코드 + 이메일 조합 필요 |

#### 대응 방안

- **Rate Limiting 이미 적용됨**: 모든 로그인 엔드포인트에 `@Throttle({ default: { ttl: 60000, limit: 5 } })` 설정 → 분당 5회로 무차별 열거 방지
- **계정 잠금 이미 적용됨**: 5회 연속 실패 시 15분 잠금 (`MAX_LOGIN_ATTEMPTS = 5`)
- `/admin/login`은 관리자 전용이므로 일반 사용자 접근 제한 환경 (내부 네트워크)
- **결론**: 기존 Rate Limiting + Account Lockout으로 충분히 방어됨. 사용자 경험 개선 효과가 보안 위험 대비 높다고 판단

### 5.2 Entity/Partner Code 미존재 시

법인코드 또는 파트너코드가 유효하지 않은 경우에는 기존대로 `E1002`(일반적인 "Invalid credentials")를 반환하여 법인/파트너 존재 여부를 노출하지 않는다.

### 6.2 사이드 임팩트

- **하위 호환**: `E1002`는 기존대로 유지. 새 에러 코드(`E1014`, `E1015`)를 추가하는 방식이므로, 프론트엔드가 미배포되어도 기존 `E1002` 폴백으로 동작
- **Client/Partner Portal**: 현재 `UnauthorizedException` → `BusinessException`으로 변경 필요. 응답 포맷이 표준 포맷(`{ success, data, error: { code, message } }`)으로 통일됨
- **Rate Limiting**: 기존 설정 유지 (분당 5회), 추가 조치 불요

### 8.1 에러 코드 상수 추가

```typescript
// apps/api/src/global/constant/error-code.constant.ts
EMAIL_NOT_FOUND: { code: 'E1014', message: 'Email not registered.' },
PASSWORD_INCORRECT: { code: 'E1015', message: 'Incorrect password.' },
```

### 8.2 백엔드 에러 분기 로직 (핵심)

```typescript
// 공통 패턴 (모든 포탈)
// Step 1: 사용자 조회
const user = await this.userRepository.findOne({ where: { usrEmail: email, ... } });

// Step 2: 이메일 미존재 → E1014
if (!user) {
  throw new BusinessException(
    ERROR_CODE.EMAIL_NOT_FOUND.code,
    ERROR_CODE.EMAIL_NOT_FOUND.message,
    HttpStatus.UNAUTHORIZED,
  );
}

// Step 3: 비밀번호 불일치 → E1015
if (!(await bcrypt.compare(password, user.usrPassword))) {
  await this.recordFailedLogin(user); // 실패 횟수 기록 (메인 auth만)
  throw new BusinessException(
    ERROR_CODE.PASSWORD_INCORRECT.code,
    ERROR_CODE.PASSWORD_INCORRECT.message,
    HttpStatus.UNAUTHORIZED,
  );
}
```

### 8.3 프론트엔드 에러 코드 파싱 로직 (공통 유틸)

```typescript
// 에러 응답에서 에러 코드 추출하는 공통 헬퍼
function getLoginErrorMessage(error: unknown, t: TFunction): string {
  const code = error?.response?.data?.error?.code;
  switch (code) {
    case 'E1014': return t('loginErrorEmailNotFound');
    case 'E1015': return t('loginErrorPasswordIncorrect');
    case 'E1020': return t('loginErrorAccountLocked');
    default:      return t('loginFailed');
  }
}
```

### 8.4 Client/Partner 포탈 예외 처리 통일

현재 Client/Partner Auth Service는 `UnauthorizedException`(NestJS 기본)을 사용하여 에러 코드가 없음.  
→ `BusinessException`으로 변경하여 표준 에러 응답 포맷(`{ success, data, error: { code, message } }`)으로 통일.
