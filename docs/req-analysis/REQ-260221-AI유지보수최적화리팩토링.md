## 0. 문서 목적 및 핵심 목표

### 0.1 핵심 목표

AMB Management / ambKMS 프로젝트는 **AI(Claude Code)가 설계·구현한 시스템**이다. 이 프로젝트의 유지보수 역시 AI가 수행하되, 다음 프로세스를 통해 **안전하고 추적 가능한 AI 자율 유지보수 체계**를 구축하는 것이 본 문서의 목표이다.

```
┌──────────────────────────────────────────────────────────────────┐
│                   AI 자율 유지보수 파이프라인                       │
│                                                                  │
│  ① 신고/요청 → ② 관리자 확인 → ③ AI 분석 → ④ AI 구현 → ⑤ 검증    │
│       ↑              ↓              ↓           ↓          ↓     │
│   사용자/운영자    승인/반려     계획서 작성   코드 수정   자동 테스트  │
│                                                    ↓          ↓  │
│                                              ⑥ 관리자 승인 → ⑦ 배포 │
│                                                              ↓   │
│                                                      ⑧ 모니터링   │
│                                                       ↓         │
│                                               버그 재발 시 ① 복귀  │
└──────────────────────────────────────────────────────────────────┘
```

### 0.2 이 문서의 범위

| 범위 | 내용 |
|------|------|
| **A. AI 파이프라인 인프라** | 이슈 수집 → 승인 → AI 실행 → 검증 → 배포 전 과정 |
| **B. 코드 구조 최적화** | AI가 코드를 읽고 수정하기 쉬운 구조로 개선 |
| **C. 사용자 UX 개선** | 서비스 품질 향상 (토스트, 에러 번역, AI 응답 모드) |
| **D. 운영 체계 개선** | 모니터링, 감사 로그, RBAC, 배포 안정화 |

### 1.4 현재 AI 작업 방식 (AS-IS)

```
현재 프로세스 (수동·세션 기반):

1. 사용자가 Claude Code 세션을 시작
2. 구두/텍스트로 버그 또는 기능 요청 전달
3. AI가 CLAUDE.md 5단계 워크플로우에 따라:
   a. 요구사항 분석서 작성 (docs/analysis/REQ-*.md)
   b. 작업 계획서 작성 (docs/plan/PLAN-*.md)
   c. 코드 구현
   d. 테스트 케이스 문서 작성 (docs/test/TC-*.md) — 실행 가능 테스트 아님
   e. 완료 보고서 작성 (docs/implementation/RPT-*.md)
4. 세션 로그 기록 (docs/log/YYYY-MM-DD/)
5. 사용자가 검토 후 수동으로 커밋/배포

문제점:
- 이슈 추적 불가 (세션 종료 시 이슈 맥락 소실)
- 테스트 없이 배포 (검증 단계 부재)
- 배포 후 문제 감지 불가 (모니터링 부재)
- 이전 세션 작업 이력 참조 어려움 (세션 간 단절)
```

**상태 흐름**:
```
OPEN ──→ APPROVED ──→ IN_PROGRESS ──→ RESOLVED ──→ CLOSED
  │         │                             │
  └→ REJECTED                         └→ OPEN (재오픈)

관리자 역할:
  OPEN → APPROVED: 관리자가 이슈를 확인하고 AI 작업을 승인
  OPEN → REJECTED: 관리자가 이슈를 반려 (사유 기재)
  RESOLVED → CLOSED: 관리자가 AI 수정 결과를 확인하고 최종 완료

AI 역할:
  APPROVED → IN_PROGRESS: AI가 작업 시작 (자동 또는 수동 트리거)
  IN_PROGRESS → RESOLVED: AI가 수정 완료 + PR 생성
```

**수용 기준**:
- [ ] 이슈 CRUD API + 프론트 UI (`/issues`, `/issues/:id`)
- [ ] 상태 전이 시 자동 로그 기록 (`amb_issue_status_logs`)
- [ ] AI 코멘트 (`isc_author_type: 'AI'`)로 분석 결과 기록
- [ ] 관리자만 승인/반려 가능 (`@Roles('ADMIN', 'MANAGER')`)

## 5. 갭 분석

### 5.1 AI 자율 유지보수 파이프라인 갭

| 단계 | AS-IS | TO-BE | 갭 | 위험도 |
|------|-------|-------|-----|--------|
| ① 신고 | GitHub 템플릿만 | 내부 이슈 관리 + GitHub 동기화 | **대** | 중 |
| ② 승인 | 프로세스 없음 | 관리자 승인 UI + 상태 머신 | **대** | 중 |
| ③ 분석 | CLAUDE.md + 문서 | 도메인 맵 + 템플릿 보강 | 소 | 낮 |
| ④ 구현 | 코드 구조 비최적 | 파일 분리 + 패턴 일관화 | **대** | 중 |
| ⑤ 검증 | 테스트 0개 | 단위+통합 테스트 + hook | **최대** | **최고** |
| ⑥ 리뷰 | PR 프로세스 없음 | PR 템플릿 + 영향 분석 | 중 | 중 |
| ⑦ 배포 | 수동 SSH | GitHub Actions 자동화 | 중 | 중 |
| ⑧ 모니터링 | 전무 | 헬스체크 + 로깅 + 대시보드 | **대** | **높** |

### 5.2 핵심 병목 (AI 자율 유지보수 불가 사유 Top 3)

```
1위: 테스트 전무 (⑤)
  → AI가 수정 후 "맞는지" 검증할 방법이 없음
  → 결과: 모든 변경이 수동 검증에 의존

2위: 이슈 추적 부재 (①②)
  → 버그 신고가 체계적으로 관리되지 않음
  → 결과: AI에게 "무엇을 수정할지" 전달하는 표준 경로가 없음

3위: 모니터링 부재 (⑧)
  → 배포 후 문제 발생을 자동 감지할 수 없음
  → 결과: 버그 재발 → 재신고 → 재수정의 비효율 루프
```

## 8. 기술 제약 및 의존관계

### 8.1 파이프라인 의존관계

```
P-07 (hook) ──→ P-06 (테스트) ──→ P-10 (CI) ──→ P-10 (CD)
                                                    ↓
P-01 (이슈) ──→ P-03 (승인 UI) ──→ P-09 (PR)    배포 자동화
                                                    ↓
P-11 (헬스체크) ────────────────────────────→ P-12 (스모크/롤백)
```

### 8.2 트랙 간 시너지

```
S-03 (응답 통일) ──→ U-01 (토스트) ──→ U-02 (에러 번역)
  "에러 발생 → 표준 응답 → 토스트 표시 → 번역된 메시지" 파이프라인

S-05 (i18n 자동 로딩) ──→ U-02 (에러 번역)
  "errors.json 추가만으로 자동 등록"

P-06 (테스트) ──→ S-01~S-03 (코드 구조)
  "테스트가 있어야 구조 리팩토링 시 회귀 검증 가능"

P-01 (이슈) + O-02 (감사 로그) ──→ 전체 추적성
  "누가 신고 → 누가 승인 → AI가 수정 → 결과" 전 과정 추적
```

### 8.3 핵심 제약

| 제약 | 설명 | 대응 |
|------|------|------|
| **테스트 우선** | 코드 구조 변경(S) 전에 테스트(P-06) 선행 필수 | Phase 1에서 테스트 먼저 |
| **하위 호환** | `@amb/types` import 경로 유지 | index.ts re-export |
| **배포 중단 최소화** | Phase 3(코드 구조) 작업 시 배포 영향 | 점진적 마이그레이션 |
