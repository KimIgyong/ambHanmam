# REQ-사이트메뉴법인별관리-20260305

## 1. 요구사항 개요

| 항목 | 내용 |
|------|------|
| **제목** | 사이트 메뉴 법인별 관리 기능 |
| **요청일** | 2026-03-05 |
| **대상 페이지** | `/site/menus` (SiteMenuPage) |
| **우선순위** | 높음 |

### 요구사항 요약
현재 사이트 관리(`/site/*`) 페이지는 로그인 사용자의 JWT에 포함된 `entityId` 하나만으로 메뉴/페이지를 관리함. **ADMIN/MASTER 사용자가 법인을 선택하여 각 법인의 사이트 메뉴를 독립적으로 관리**할 수 있어야 함.

## 2. AS-IS 현황 분석

### 2-1. 백엔드 현황

| 항목 | 현재 상태 |
|------|-----------|
| **컨트롤러** | `CmsMenuController` (`/cms/menus`) |
| **엔티티 ID 소스** | `req.user.entityId` (JWT에서 추출) |
| **가드** | `JwtAuthGuard` + `ManagerGuard` (SUPER_ADMIN/ADMIN/MASTER/MANAGER) |
| **엔티티 격리** | `entId` 컬럼 기반 (CmsMenuEntity, CmsPageEntity 모두 `entId` 보유) |
| **문제점** | `req.user.entityId`는 JWT 토큰 발급 시점의 사용자 소속 법인으로 고정됨. ADMIN이 다른 법인의 메뉴를 관리할 수 없음 |

**현재 엔티티 ID 결정 흐름:**
```
JWT 토큰 → jwt.strategy.ts → user.entityId 설정
  ├─ USER_LEVEL: JWT payload의 entityId 사용
  └─ ADMIN_LEVEL: entityId 없음 (undefined)

CmsMenuController → req.user.entityId 사용 (ADMIN은 undefined → 조회 실패)
```

### 2-2. 프론트엔드 현황

| 항목 | 현재 상태 |
|------|-----------|
| **페이지** | `SiteMenuPage.tsx` |
| **API 호출** | `cmsMenuService.getMenuTree()` (엔티티 파라미터 없음) |
| **법인 선택 UI** | 없음 (헤더의 `EntitySelector`와 무관) |
| **X-Entity-Id 헤더** | `api-client.ts` 인터셉터에서 자동 주입되지만, 백엔드에서 이 헤더를 사용하지 않음 |

### 2-3. 기존 법인 전환 패턴 (참고)

시스템에는 이미 두 가지 법인 컨텍스트 패턴이 존재:

| 패턴 | 사용처 | 메커니즘 |
|------|--------|----------|
| **EntityGuard + X-Entity-Id** | HR 모듈 전체 | 헤더에서 entityId 추출, EntityGuard가 권한 검증 |
| **OwnEntityGuard + query param** | entity-settings 모듈 | `resolveEntityId(queryEntityId, user)`로 결정 |

**CMS 모듈은 두 패턴 모두 사용하지 않고 JWT 고정값만 사용 중.**

## 3. TO-BE 요구사항

### 3-1. 핵심 요구사항

1. **법인 선택**: ADMIN/MASTER 사용자가 `/site/menus` 페이지에서 관리할 법인을 선택할 수 있어야 함
2. **법인별 격리**: 선택한 법인의 메뉴만 조회/생성/수정/삭제되어야 함
3. **기존 UX 유지**: 일반 사용자(MEMBER, VIEWER)는 기존과 동일하게 자신의 법인 메뉴만 관리
4. **사이트 전체 적용**: 메뉴뿐 아니라 페이지, 포스트, 구독자 등 사이트 관리 전체에 법인 컨텍스트 적용

### 3-2. 사용자 시나리오

**시나리오 1: ADMIN이 베트남 법인 사이트 메뉴 관리**
1. ADMIN 사용자 로그인
2. 헤더의 EntitySelector에서 "VN01 - Amoeba Vietnam" 선택
3. `/site/menus` 접속 → VN01 법인의 메뉴 트리만 표시
4. 메뉴 추가/수정/삭제 → VN01 법인에만 적용

**시나리오 2: MASTER가 자기 법인 메뉴 관리**
1. MASTER 사용자 로그인 (KR01 법인 소속)
2. `/site/menus` 접속 → KR01 법인의 메뉴 트리 표시
3. EntitySelector에서 다른 법인 선택 불가 (소속 법인만 표시)

**시나리오 3: ADMIN이 법인 미선택 상태**
1. ADMIN 사용자 로그인, EntitySelector에서 법인 미선택
2. `/site/menus` 접속 → "법인을 선택하세요" 안내 메시지 표시

## 4. 갭 분석

### 4-1. 백엔드 갭

| # | 현재 | 목표 | 변경 필요 |
|---|------|------|-----------|
| 1 | `req.user.entityId` (JWT 고정) | `X-Entity-Id` 헤더 또는 쿼리 파라미터 | CmsMenuController에서 엔티티 ID 소스 변경 |
| 2 | EntityGuard 미사용 | EntityGuard 사용 | 컨트롤러에 EntityGuard 추가 또는 자체 resolveEntityId 로직 |
| 3 | 동일 패턴이 CmsPage, CmsSection, CmsPost, CmsSubscriber 컨트롤러에도 적용 필요 | 사이트 관리 전체 컨트롤러 수정 | 4개 컨트롤러 모두 수정 |

### 4-2. 프론트엔드 갭

| # | 현재 | 목표 | 변경 필요 |
|---|------|------|-----------|
| 1 | EntitySelector 미연동 | EntitySelector의 선택 법인으로 API 호출 | `X-Entity-Id` 헤더 이미 자동 주입 → 백엔드만 수정하면 연동 완료 |
| 2 | 법인 미선택 시 빈 화면 | "법인을 선택하세요" 안내 | SiteLayout 또는 SiteMenuPage에 가드 추가 |
| 3 | ADMIN의 entityId가 undefined | EntitySelector 연동 필수 | 이미 `api-client.ts` 인터셉터에서 `X-Entity-Id` 주입 중 |

### 4-3. 핵심 발견

**프론트엔드의 `api-client.ts`는 이미 `X-Entity-Id` 헤더를 모든 요청에 자동 주입하고 있음.** 따라서:
- 프론트엔드 API 서비스 수정 불필요
- 백엔드 컨트롤러에서 `req.user.entityId` 대신 `req.headers['x-entity-id']` 또는 EntityGuard의 `req.entityId`를 사용하도록 변경하면 됨

## 6. 기술 제약사항 및 고려사항

### 6-1. EntityGuard vs 자체 로직
- **EntityGuard 사용 권장**: HR 모듈에서 검증된 패턴, 권한 검증 포함
- **단, SiteManagementModule에 EntityGuard 의존성 추가 필요** (HrModule의 EntityService)
- 대안: `resolveEntityId` 유틸리티 + OwnEntityGuard 패턴 (entity-settings 방식)

### 6-2. 모듈 의존성
- 현재 `SiteManagementModule`은 `HrModule`에 의존하지 않음
- EntityGuard를 사용하려면 HrModule의 EntityService를 import 해야 함
- 또는 entity-settings처럼 `resolveEntityId` 유틸리티로 간단하게 처리 가능

### 6-3. 사이트 관리 전체 영향 범위
CMS 관련 컨트롤러 **5개 모두** `req.user.entityId` 사용:
- `CmsMenuController` - 메뉴 관리
- `CmsPageController` - 페이지 관리
- `CmsSectionController` - 섹션 관리
- `CmsPostController` - 게시글 관리
- `CmsSubscriberController` - 구독자 관리

→ **5개 컨트롤러 모두 동일하게 수정 필요**

### 6-4. Public API 영향 없음
- `CmsPublicController`는 인증 없이 모든 법인 데이터를 조회하므로 변경 불필요
- `getAllMenuTree()`는 공개 사이트용으로 기존 동작 유지

### 6-5. React Query 캐시 키
- 법인이 변경되면 기존 캐시 데이터가 무효화되어야 함
- `X-Entity-Id` 헤더 변경 시 React Query의 `queryKey`에 entityId를 포함하거나, 법인 변경 시 전체 invalidate 필요

## 7. 구현 방안 (권장)

### 접근 방식: `resolveEntityId` 패턴 (entity-settings와 동일)

| 장점 | 설명 |
|------|------|
| 모듈 의존성 최소화 | HrModule import 불필요 |
| 검증된 패턴 | entity-settings에서 이미 사용 중 |
| 간단한 변경 | 컨트롤러에서 entityId 소스만 변경 |

**변경 요약:**
1. 백엔드: CMS 컨트롤러 5개에서 `req.user.entityId` → `resolveEntityId()` 변경
2. 프론트엔드: `X-Entity-Id` 헤더 이미 자동 주입 → API 서비스 변경 불필요
3. 프론트엔드: 법인 미선택 시 안내 UI 추가 (SiteLayout 또는 각 페이지)
4. 프론트엔드: React Query 캐시 키에 entityId 추가 (법인 변경 시 refetch)
