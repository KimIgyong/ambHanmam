## 1. AS-IS 현황 분석

### 1.1 현재 이슈 관리 시스템 구조

#### 프론트엔드 구성
| 항목 | 현황 |

| **훅** | useIssues.ts (useIssueList, useCreateIssue, useUpdateIssue, useDeleteIssue 등) |

#### 백엔드 구성
| 항목 | 현황 |
|------|------|
| **엔티티** | IssueEntity, IssueCommentEntity, IssueStatusLogEntity |
| **컨트롤러** | IssueController (12개 엔드포인트) |
| **서비스** | IssueService |
| **DTO** | CreateIssueRequest, UpdateIssueRequest, UpdateIssueStatusRequest, CreateIssueCommentRequest |
| **매퍼** | IssueMapper |
| **에러코드** | E23001(Not Found), E23002(Invalid Transition), E23003(Comment Not Found) |

#### 이슈 데이터 모델
- **타입**: BUG, FEATURE, ENHANCEMENT
- **심각도**: CRITICAL, MAJOR, MINOR
- **상태**: OPEN → APPROVED → IN_PROGRESS → RESOLVED → CLOSED (또는 REJECTED)
- **우선순위**: 1(최상)~5(최하)
- **담당자 필드**: `iss_assignee` VARCHAR(50) - 단순 문자열 (사용자 엔티티와 관계 없음)

### 1.2 현재 뷰 방식
- **목록 뷰만 존재**: IssueItem 컴포넌트로 리스트 형태 표시
- **칸반 뷰 없음**: 상태별 컬럼 뷰 미구현
- **간트차트 뷰 없음**: 일정 기반 뷰 미구현
- **탭 전환 없음**: 단일 뷰만 지원

### 1.3 현재 담당자 지정 방식
- 담당자 필드: 단순 텍스트 입력 (`<input type="text">`)
- 조직원 목록 연동 없음
- 담당자 검색/선택 기능 없음
- 기본값: 빈 문자열 (미지정)
- DB 컬럼: `iss_assignee` VARCHAR(50) - 이름 문자열 저장

### 1.4 PATCH 400 Bad Request 에러 분석

**원인**: 프론트엔드에서 이슈 수정 시 `type` 필드를 함께 전송하지만, `UpdateIssueRequest` DTO에 `type` 필드가 정의되어 있지 않음.

**프론트엔드 코드** (`IssuesPage.tsx:49-80`):
```typescript
handleFormSubmit → formData에 type 포함 → updateIssue.mutate({ id, data: formData })
```

**백엔드 DTO** (`update-issue.request.ts`):
- `title`, `description`, `severity`, `priority`, `affected_modules`, `resolution`, `ai_analysis`, `assignee` 만 허용
- `type` 필드 미정의 → class-validator의 `whitelist: true` 또는 `forbidNonWhitelisted: true` 설정에 의해 400 반환

**해결 방안**:
1. 백엔드 `UpdateIssueRequest`에 `type` 필드 추가, 또는
2. 프론트엔드에서 수정 시 `type` 필드 제외

## 2. TO-BE 요구사항

### 2.1 이슈 목록 3가지 뷰 (탭 전환)

#### 2.1.1 목록 보기 (List View) - 기존 유지/개선
- 현재 IssueItem 기반 리스트 유지
- 탭 UI 추가

#### 2.1.2 칸반 보기 (Kanban View) - 신규
- 상태별 컬럼: OPEN, APPROVED, IN_PROGRESS, RESOLVED, CLOSED, REJECTED
- 각 컬럼에 해당 상태의 이슈 카드 표시
- 드래그 앤 드롭으로 상태 변경 (유효한 전환만 허용)
- 카드에 타입, 심각도, 우선순위, 담당자 표시
- 카드 클릭 시 상세 모달 표시

#### 2.1.3 간트차트 보기 (Gantt Chart View) - 신규
- 이슈별 일정 타임라인 표시
- X축: 날짜, Y축: 이슈 목록
- 이슈 생성일 ~ 해결일 또는 현재일 기간 바 표시
- 상태/타입별 색상 구분
- 마우스 오버 시 상세 정보 툴팁

### 2.2 담당자 지정 개선

#### 2.2.1 담당자 선택 UI
- 텍스트 입력 → 조직원 검색 드롭다운으로 변경
- 회사 조직원(멤버) 리스트에서 선택
- 검색 기능: 이름/이메일로 검색
- 기본값: 현재 로그인 사용자 (이슈 생성 시)
- 미지정 옵션 유지

#### 2.2.2 담당자 데이터 모델 변경
- `iss_assignee` VARCHAR(50) → `iss_assignee_id` UUID (FK to amb_users.usr_id)
- 기존 문자열 기반에서 사용자 ID 참조로 변경
- 기존 데이터 마이그레이션 필요

#### 2.2.3 백엔드 변경
- 멤버 목록 조회 API 활용: `GET /api/v1/members`
- 이슈 응답에 `assigneeId`, `assigneeName` 추가
- 이슈 생성/수정 DTO에 `assignee_id` 필드 추가

## 4. 사용자 플로우

### 4.1 이슈 목록 조회 (탭 전환)
```
사용자 → /issues 접속
  → 기본: 목록 보기 탭 활성
  → [칸반 보기] 탭 클릭 → 상태별 컬럼 렌더링
  → [간트차트 보기] 탭 클릭 → 타임라인 렌더링
  → 각 뷰에서 필터/검색 공유
```

### 4.2 칸반 보기 상태 변경
```
사용자 → 칸반 보기 탭 선택
  → 이슈 카드 드래그 시작
  → 유효한 대상 컬럼만 드롭 가능 표시
  → 카드 드롭 → 상태 변경 API 호출
  → (필요시) 상태 변경 메모 입력 모달
  → 칸반 뷰 갱신
```

### 4.3 이슈 등록 (담당자 선택)
```
사용자 → [이슈 등록] 버튼 클릭
  → 이슈 폼 모달 표시
  → 담당자 필드: 현재 사용자 기본 선택됨
  → 담당자 변경 시: 검색 입력 → 조직원 드롭다운 표시
  → 이름/이메일로 검색 → 선택
  → 나머지 필드 입력 → 등록
```

### 4.4 이슈 수정 (담당자 변경)
```
사용자 → 이슈 상세 → [수정] 버튼
  → 수정 폼: 기존 담당자 표시
  → 담당자 필드 클릭 → 검색/선택 드롭다운
  → 변경 → 저장
```

## 5. 기술 제약사항

### 5.1 프론트엔드
- 칸반 드래그 앤 드롭: `@dnd-kit/core` 또는 `react-beautiful-dnd` 등 라이브러리 필요
- 간트차트: `gantt-task-react` 또는 커스텀 SVG/Canvas 기반 구현
- 담당자 검색 드롭다운: 기존 컴포넌트 패턴 확인 필요 (다른 모듈에서 유사 UI 존재 여부)
- 새 라이브러리 추가 시 번들 사이즈 영향 고려

### 5.2 백엔드
- 담당자 필드 변경: DB 마이그레이션 필요 (VARCHAR → UUID)
- 기존 데이터 마이그레이션: 기존 이름 문자열 → 사용자 ID 매핑
- 간트차트용 날짜 필드: 현재 `createdAt`/`resolvedAt`으로 대체 가능 (별도 시작일/종료일 추가는 선택)
- 이슈 응답에 `assigneeId`, `assigneeName` 추가 필요

### 5.3 상태 전환 제약
- 칸반 드래그 시 `ISSUE_VALID_TRANSITIONS` 규칙 준수
- MANAGER 이상만 가능한 작업: APPROVED, REJECTED, CLOSED
- 일반 사용자: IN_PROGRESS, RESOLVED만 전환 가능

### 5.4 성능 고려
- 칸반 뷰: 전체 이슈를 한번에 로드 필요 (페이징 제거 또는 상태별 그룹 쿼리)
- 간트차트: 대량 이슈 시 렌더링 성능 고려
- 담당자 검색: 디바운싱 적용, 캐싱 활용
