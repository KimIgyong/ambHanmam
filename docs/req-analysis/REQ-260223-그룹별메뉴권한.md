## 1. 요구사항 요약

6가지 요구사항을 통합 구현한다:

**A. 그룹별 메뉴 권한**: 등급별 권한 탭에서 USER_GROUP 역할(MANAGER/MEMBER/VIEWER) 오른쪽에 **그룹별 체크박스 열**을 추가. 각 메뉴를 특정 그룹에게만 노출/사용 가능하도록 설정. 그룹은 멤버 상세 "소속그룹"과 동일한 `amb_groups` 테이블을 참조.

**B. 법인별 그룹/부서 관리**: 법인 관리 페이지(`/settings/entities`)에 그룹 등록/편집/삭제 및 부서 등록/편집/삭제 기능 추가.

**C. 멤버 정보 편집**: 멤버 상세 페이지(`/settings/members/:id`)에서 이름, 역할, 직무 항목을 수정 가능하게 변경.

**D. 초대메일 발송 로그 추가**: 초대 목록 탭(`/settings/members` → 초대 탭)에 이메일 발송 날짜/시간 컬럼을 추가한다.

**E. 회원가입 비밀번호 개선**: RegisterPage(`/register?invitation_token=`)에 비밀번호 확인 필드 추가 + 비밀번호 보기(눈 아이콘) 토글. InviteAcceptPage(`/invite/:token`)에도 비밀번호 보기 토글 추가.

**F. 승인대기 워크플로우**: 초대 회원가입 후 즉시 승인 처리하지 않고 **승인대기(PENDING)** 상태로 저장. 관리자가 초대 멤버 리스트에서 내용 확인 후 승인하면 로그인 가능.

## 2. AS-IS 현황

### 2-1. 권한 판정 체계 (3단계)
```
0단계: 메뉴 config enabled=false → 전체 숨김
1단계: ADMIN_GROUP(SUPER_ADMIN/ADMIN) → 모든 활성 메뉴 접근
2단계: 사용자별 개별 설정 (amb_user_menu_permissions) → 최우선
3단계: 역할별 설정 (amb_menu_permissions) → 등급 기본값
```

### 2-2. 현 DB 구조
| 테이블 | 용도 | 키 차원 |
|--------|------|---------|
| `amb_menu_config` | 메뉴 활성/비활성, 순서 | menuCode |
| `amb_menu_permissions` | 역할별 접근권한 | (menuCode, role) |
| `amb_user_menu_permissions` | 사용자별 개별 재정의 | (menuCode, userId) |

### 2-3. 그룹/부서 시스템 현황

**두 테이블의 역할 구분**:
| 테이블 | 용도 | 권한 연동 |
|--------|------|----------|
| **`amb_groups`** | **메뉴 접근권한 그룹** — 멤버 상세 "소속그룹"과 동일 | ✅ 메뉴 권한 판정에 사용 |
| **`amb_departments`** | **소속부서 관리** — 조직도/인사 목적 | ❌ 메뉴 권한과 무관 |

- **`amb_groups`**: 그룹 관리 (grp_id, grp_name, grp_description, ent_id)
- **`amb_user_groups`**: 사용자-그룹 N:M (ugr_id, usr_id, grp_id) → 메뉴 권한 판정 시 참조
- **`amb_departments`**: 소속부서 관리 (dep_id, dep_name, ent_id) — 현재 **데이터 없음**(0행)
- **`amb_user_dept_roles`**: 사용자-부서 N:M (서비스/API 구현 완료) → 인사/조직 관리 목적
- `amb_hr_entities` (법인): HQ(GL), VN01(VN), KR01(KR) 3개 법인 활성
- 그룹 API (`/groups`): CRUD + 멤버 배정/제거 구현 완료
- 부서 API (`/departments`): CRUD + 트리 + 멤버 배정 구현 완료
- **현재 메뉴 권한과 그룹/부서는 완전히 분리됨** — 교차점 없음
- 멤버 상세 "소속그룹"은 `amb_groups` + `amb_user_groups` 사용

### 2-4. RolePermissionTab UI 구조
```
[등급별 권한 탭]
├── 서브탭: ADMIN_GROUP | USER_GROUP
├── ADMIN_GROUP: Lock 아이콘 + disabled 토글 (전체 ON)
└── USER_GROUP: MANAGER | MEMBER | VIEWER 3열 토글 매트릭스
    ├── 채팅 메뉴 섹션
    ├── 업무 도구 섹션
    ├── 업무 모듈 섹션
    └── 설정 메뉴 섹션
```

## 3. TO-BE 요구사항

### 3-1. 기능 개요
USER_GROUP 탭의 각 메뉴 행에 **역할별 토글 3개(MANAGER/MEMBER/VIEWER)** 외에 **그룹별 체크박스**가 추가된다.

### 3-2. UI 변경

**현재(AS-IS)**:
```
| 메뉴명         | MANAGER | MEMBER | VIEWER |
|----------------|---------|--------|--------|
| 프로젝트 관리  |  [ON]   | [OFF]  | [OFF]  |
```

**변경 후(TO-BE)**:
```
| 메뉴명         | MANAGER | MEMBER | VIEWER | MS | DevOPS | BPO | 기술연구소 |
|----------------|---------|--------|--------|-----|--------|-----|-----------|
| 프로젝트 관리  |  [ON]   | [OFF]  | [OFF]  | [✓] |  [✓]  | [ ] |    [✓]    |
```

> **역할 토글**: ON/OFF → 해당 등급 전체 접근 가능 여부  
> **그룹 체크박스**: ✓ → 해당 그룹(부서) 소속 사용자만 접근 가능 (역할 권한 + 그룹 권한 AND 조건)

### 3-3. 권한 판정 로직 변경 (4단계)

```
0단계: 메뉴 config enabled=false → 전체 숨김
1단계: ADMIN_GROUP → 모든 활성 메뉴 접근 (변경 없음)
2단계: 사용자별 개별 설정 → 최우선 (변경 없음)
3단계: 역할별 설정 → 등급 기본값 (변경 없음)
4단계(신규): 그룹 필터 — 3단계에서 접근 가능 판정된 메뉴에 대해:
  - 해당 메뉴에 그룹 설정이 없으면 → 모든 그룹 접근 가능 (기존 호환)
  - 해당 메뉴에 그룹 설정이 있으면 → 사용자 소속 그룹이 체크된 경우만 접근
```

### 3-4. 데이터 모델 변경

#### 신규 테이블: `amb_menu_group_permissions`
| 컨럼 | 타입 | 설명 |
|------|------|------|
| `mgp_id` | UUID (PK) | |
| `mgp_menu_code` | varchar(50) | 메뉴 코드 |
| `grp_id` | UUID (FK → amb_groups) | 그룹 ID |
| `mgp_accessible` | boolean (default: true) | 접근 허용 |
| `mgp_created_at` | timestamp | |
| `mgp_updated_at` | timestamp | |
| **UNIQUE** | `(mgp_menu_code, grp_id)` | |

> 참고: 기존 `amb_groups` 테이블을 그대로 활용. 그룹 데이터는 법인 관리 페이지의 그룹 관리 UI에서 등록/편집/삭제.

#### 멤버 상세 편집용 컨럼 추가

## 5. 사용자 플로우

### 5-1. 관리자: 그룹별 메뉴 권한 설정
```
1. 설정 > 접근권한 관리 > 등급별 권한 탭 진입
2. USER_GROUP 서브탭 선택
3. 각 메뉴 행의 오른쪽에 부서별 체크박스가 표시됨
4. 특정 메뉴의 특정 부서 체크 → 해당 부서 소속 사용자만 접근 가능
5. 체크박스가 하나도 선택되지 않으면 → 모든 그룹 접근 가능 (기존 호환)
6. 저장 버튼 클릭 → 역할 권한 + 그룹 권한 일괄 저장
```

### 5-2. 일반 사용자: 메뉴 접근
```
1. 사용자 로그인
2. getMyMenus() 호출 → 권한 판정
3. 역할(MANAGER 등) OK + 소속 그룹이 해당 메뉴에 체크됨 → 메뉴 노출
4. 역할 OK but 소속 그룹이 미체크 → 메뉴 숨김
5. 메뉴에 그룹 설정이 없으면 → 역할 권한만으로 판정 (기존 동작)
```

## 6. 기술 제약사항 및 고려사항

### 6-1. 그룹 데이터 선행 필요
- 법인 관리 페이지에서 그룹 등록 UI를 통해 그룹 생성 후 권한 설정 가능
- 사용자도 그룹에 배정되어야 함 (멤버 상세 "소속그룹" 또는 `amb_user_groups`)

### 6-2. 기존 호환성
- 그룹 설정이 없는 메뉴 → 기존과 동일하게 역할 권한만 적용
- ADMIN_GROUP은 그룹 필터 무시 (항상 전체 접근)
- 사용자별 개별 권한도 그룹 필터보다 우선

### 6-3. 다중 그룹 소속
- `amb_user_groups`로 사용자가 여러 그룹에 소속 가능
- 소속 그룹 중 하나라도 체크되어 있으면 접근 허용 (OR 조건)

### 6-4. UI 확장성
- 그룹이 추가되면 자동으로 열이 추가되어야 함
- 법인별 그룹핑 표시 필요 (VN: MS/DevOPS/BPO, KR: 기술연구소)

### 6-5. `amb_departments`는 소속부서 관리 전용
- 메뉴 접근권한 판정에는 `amb_groups` 사용 (멤버 상세 "소속그룹"과 동일)
- `amb_departments`는 인사/조직도 목적의 소속부서 관리에만 사용
- 법인 관리 페이지에서 그룹과 부서를 각각 별도 탭으로 관리

### 6-5. 성능
- getMyMenus()에서 추가 조인 1회 발생 — 메뉴 수 × 그룹 수 규모이므로 영향 미미

## 7. 추가 요구사항 A: 법인별 그룹/부서 관리 기능

### 7-1. 요구사항
법인 관리 페이지(`/settings/entities`)에 법인별 **그룹 관리** 및 **부서 관리** 기능을 추가한다.

### 7-2. AS-IS
- 법인 관리 페이지: 법인 정보 CRUD만 존재 (이름, 코드, 국가, 통화, 주소 등)
- 부서/그룹 관리 UI 없음
- 백엔드: 그룹 API (`/groups`) 및 부서 API (`/departments`)는 이미 구현되어 있음

### 7-3. TO-BE
법인 카드(또는 법인 상세) 하위에 **2개 탭** 추가:

#### 그룹 관리 탭
| 기능 | 설명 |
|------|------|
| 그룹 목록 | 해당 법인 소속 그룹(`amb_groups` where `ent_id`) 테이블 표시 |
| 그룹 등록 | 그룹명, 설명 입력 → `POST /groups` |
| 그룹 수정 | 그룹명, 설명 인라인/모달 편집 → `PUT /groups/:id` |
| 그룹 삭제 | 확인 후 삭제 → `DELETE /groups/:id` |

#### 부서 관리 탭
| 기능 | 설명 |
|------|------|
| 부서 목록 | 해당 법인 소속 부서(`amb_departments` where `ent_id`) 트리/리스트 표시 |
| 부서 등록 | 부서명, 현지명, 상위부서, 레벨 입력 → `POST /departments` |
| 부서 수정 | 부서명, 현지명 편집 → `PUT /departments/:id` |
| 부서 삭제 | 확인 후 삭제 → `DELETE /departments/:id` |

### 7-4. 관련 DB 테이블

**`amb_groups`** (그룹 — 메뉴 권한에서 참조하는 핵심 테이블):

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `grp_id` | UUID (PK) | 그룹 ID |
| `grp_name` | varchar | 그룹명 |
| `grp_description` | varchar, nullable | 설명 |
| `ent_id` | UUID (FK → `amb_hr_entities`), nullable | 소속 법인 |
| `grp_created_at` | timestamp | |
| `grp_updated_at` | timestamp | |

**`amb_user_groups`** (사용자-그룹 N:M):

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `ugr_id` | UUID (PK) | |
| `usr_id` | UUID (FK → `amb_users`) | 사용자 |
| `grp_id` | UUID (FK → `amb_groups`) | 그룹 |
| `ugr_created_at` | timestamp | |

> **핵심**: 메뉴 접근권한의 "그룹별 권한"은 `amb_departments`가 아닌 **`amb_groups`** 테이블을 참조해야 함  
> (멤버 상세 페이지 "소속그룹"과 동일한 데이터 소스)

## 8. 추가 요구사항 B: 멤버 상세 - 이름/역할/직무 수정

### 8-1. 요구사항
멤버 상세 페이지(`/settings/members/:id`)에서 **이름**, **역할**, **직무** 항목을 수정할 수 있어야 한다.

### 8-2. AS-IS

| 필드 | 표시 | 수정 가능 | DB 컬럼 |
|------|------|----------|---------|
| 이름 (name) | `InfoItem` 읽기전용 | ❌ | `usr_name` (varchar 50) |
| 역할 (role) | `InfoItem` 읽기전용 | ❌ (상세에서) | `usr_role` (varchar 20) |
| 직무 (job title) | **미표시** | ❌ | **컬럼 없음** |
| 회사 이메일 | 인라인 편집 | ✅ | `usr_company_email` |

- 역할 변경: 백엔드 `PATCH /members/:id/role` + 프론트 `useUpdateMemberRole` 존재하나, **목록 페이지 모달에서만 사용**되며 상세 페이지에서는 미사용
- 이름 변경: 백엔드/프론트 모두 API 없음
- 직무: `amb_users` 테이블에 컬럼 자체가 없음 (HR `emp_position`만 존재)

### 8-3. TO-BE

| 부서 | inv_department |
| 상태 | inv_status (PENDING/ACCEPTED/CANCELLED) |
