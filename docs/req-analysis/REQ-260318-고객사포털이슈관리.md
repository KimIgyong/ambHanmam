## 1. AS-IS 현황 분석

### 1.1 현행 시스템 구조

현재 AMB Management에는 **3개의 독립 앱**이 존재한다:

| 앱 | 경로 | 포트 | 대상 | 인증 체계 |
|----|------|------|------|-----------|
| **Main Web** (`apps/web`) | `https://stg-ama.amoeba.site` | 5179 | 내부 법인 직원 | `UserEntity` + JWT (법인코드+이메일+비밀번호) |
| **Portal Web** (`apps/portal-web`) | `https://stg-ama.amoeba.site/portal` | 5180 | 외부 SaaS 고객 | `PortalCustomerEntity` + JWT (이메일+비밀번호) |
| **Main API** (`apps/api`) | 포트 3009 | — | 내부 시스템 | `JwtAuthGuard` + 역할/법인 가드 |
| **Portal API** (`apps/portal-api`) | 포트 3010 | — | 포털 시스템 | `PortalJwtAuthGuard` |

### 1.2 현행 사용자 역할 체계

```
ADMIN_LEVEL: SUPER_ADMIN(6) > ADMIN(5)        ← ROOT(HQ) 법인 전용
USER_LEVEL:  MASTER(4) > MANAGER(3) > MEMBER(2) > VIEWER(1) ← 하위 법인
```

- **가입 경로**: `SEED` | `REGISTER` | `INVITE` | `PORTAL`
- `PORTAL` 가입 경로는 Portal-Bridge 모듈에서 포털 고객을 내부 사용자로 전환할 때만 사용

### 1.3 현행 프로젝트/이슈 접근 구조

| 항목 | 현재 접근 제어 |
|------|---------------|
| **프로젝트 목록** | `entityId`(법인) 기반 필터링만 적용. 해당 법인 소속 모든 사용자가 전체 프로젝트 조회 가능 |
| **프로젝트 멤버** | `ProjectMember` 엔티티 존재 (PM, LEAD, MEMBER, REVIEWER, OBSERVER 역할), 단 멤버 여부로 조회를 제한하지 않음 |
| **이슈 생성** | 인증된 사용자라면 누구나 생성 가능 (역할 무관) |
| **이슈 상태 변경** | `APPROVED`/`REJECTED`/`CLOSED`는 MANAGER 이상만 가능 |
| **이슈 가시성** | `ENTITY`(법인 전체) / `GROUP`(Cell 소속) / `PRIVATE`(보고자/담당자/참여자만) |

### 1.4 현행 외부 고객 관련 엔티티

| 엔티티 | 테이블 | 용도 | 인증 가능 여부 |
|--------|--------|------|---------------|
| `PartnerEntity` | `amb_bil_partners` | 빌링 거래처 (계약 상대방) | ✗ 없음 |
| `SvcClientEntity` | `amb_svc_clients` | 서비스 고객사 (SaaS 구독 관리) | ✗ 없음 |
| `SvcClientContactEntity` | `amb_svc_client_contacts` | 고객사 담당자 연락처 | ✗ 없음 (이름/이메일/전화만) |
| `PortalCustomerEntity` | `amb_svc_portal_customers` | SaaS 포털 고객 계정 | ✓ 별도 JWT 인증 |

### 1.5 현행 Portal 시스템의 범위

현재 Portal(`portal-api` + `portal-web`)은 **SaaS 서비스 구독/결제 포털** 전용:
- 서비스 카탈로그 조회, 구독 관리, 결제(Stripe/VNPay), 사용량 확인
- **프로젝트/이슈 기능 없음**
- Portal-Bridge 모듈로 포털 고객을 내부 사용자로 전환만 제공

### 1.6 갭 분석 요약

| 요구사항 | 현행 상태 | 갭 |
|---------|----------|-----|

| 고객사 프로젝트 조회 | 내부 사용자 전용 (entityId 기반) | **고객-프로젝트 연결 및 접근 제어 필요** |

## 2. TO-BE 요구사항 정의

### 2.1 구현 전략: 두 가지 접근법 비교

#### Option A: 기존 Portal 확장
기존 `portal-api` + `portal-web`에 프로젝트/이슈 기능을 추가한다.

| 장점 | 단점 |
|------|------|
| 별도 인증 체계 이미 존재 | SaaS 구독과 프로젝트 관리가 혼재 |
| `PortalCustomer` → `SvcClient` 연결 구조 있음 | Portal은 구독/결제 목적으로 설계됨 |
| 별도 앱으로 로그인 페이지 분리 완료 | 프로젝트/이슈 엔티티가 Main API에만 존재 |

#### Option B: Client Portal (신규 앱) — **권장**
`apps/client-web` + Main API에 Client 전용 엔드포인트를 추가한다.

| 장점 | 단점 |
|------|------|
| SaaS Portal과 용도 완전 분리 | 신규 프론트엔드 앱 필요 |
| Main API의 프로젝트/이슈 서비스 재사용 | 인증 체계 추가 필요 |
| 고객 역할 제한이 명확 | — |
| 확장성 (향후 고객 대시보드 등) | — |

#### Option C: Main API에 CLIENT 역할 추가 + 별도 프론트엔드 경로 — **가장 실용적**
기존 `apps/api`의 인증 체계에 `CLIENT` 역할을 추가하고, `apps/web` 내에 `/client` 라우트 그룹으로 분리한다.

| 장점 | 단점 |
|------|------|
| 기존 인프라 최대 활용 | Main Web에 라우팅 분기 추가 |
| 프로젝트/이슈 서비스 변경 최소화 | 역할 체계 확장 필요 |
| 배포/운영 복잡도 최소 | 완전한 분리가 아닌 논리적 분리 |
| `DataScopeInterceptor` 확장만으로 데이터 격리 | — |

### 2.2 권장 전략: Option C (CLIENT 역할 + 별도 라우트)

**근거:**
1. 고객사 담당자도 결국 `amb_users` 테이블의 사용자로 관리하는 것이 프로젝트 멤버/이슈 보고자 등의 FK 관계와 일관성 유지
2. 기존 `ProjectMember`, `IssueEntity.reporterId` 등 FK가 모두 `UserEntity`를 참조하므로 별도 테이블을 쓰면 FK 모순
3. Portal-Bridge의 `createInternalAccount` 패턴(포털 고객 → 내부 사용자 전환)이 이미 선례

## 3. 상세 요구사항

### 3.1 사용자 역할 확장

#### 3.1.1 CLIENT 레벨 추가

```
현행 UserLevelCode: ADMIN_LEVEL | USER_LEVEL
추가: CLIENT_LEVEL
```

| 레벨 | 허용 역할 | 설명 |
|------|----------|------|
| `CLIENT_LEVEL` | `CLIENT_ADMIN`, `CLIENT_MEMBER` | 고객사 사용자 전용 |

- `CLIENT_ADMIN`: 고객사 내 관리자. 자사 소속 고객 직원 초대 가능
- `CLIENT_MEMBER`: 고객사 일반 직원. 배정된 프로젝트 조회, 이슈 등록만 가능

#### 3.1.2 역할 계층 확장

```
현행: VIEWER(1) < MEMBER(2) < MANAGER(3) < MASTER(4) < ADMIN(5) < SUPER_ADMIN(6)
추가: CLIENT_MEMBER(0.5) < CLIENT_ADMIN(0.8)  ← 모든 내부 역할보다 하위
```

> CLIENT 역할은 내부 역할 계층에 포함하지 않고 **별도 분기**로 처리하는 것을 권장. 내부 역할과 비교할 일이 없으므로 `isClient()` 체크만으로 충분.

### 3.2 고객사-프로젝트 연결 구조

#### 3.2.1 프로젝트 클라이언트 매핑 (신규 테이블)

```sql
CREATE TABLE kms_project_clients (
  pcl_id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  pjt_id      UUID NOT NULL REFERENCES kms_projects(pjt_id),
  cli_id      UUID NOT NULL REFERENCES amb_svc_clients(cli_id),
  pcl_status  VARCHAR(20) DEFAULT 'ACTIVE',   -- ACTIVE | REVOKED
  pcl_created_at  TIMESTAMPTZ DEFAULT NOW(),
  pcl_updated_at  TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(pjt_id, cli_id)
);
```

- 하나의 프로젝트에 여러 고객사가 연결될 수 있음 (파트너사 포함)
- 고객사 사용자는 자신의 `SvcClient`에 연결된 프로젝트만 조회 가능

#### 3.2.2 고객 사용자-클라이언트 연결

`amb_users` 테이블에 추가 컬럼:

```sql
ALTER TABLE amb_users ADD COLUMN usr_cli_id UUID REFERENCES amb_svc_clients(cli_id);
```

- `CLIENT_LEVEL` 사용자에게만 설정
- 내부 사용자(`USER_LEVEL`, `ADMIN_LEVEL`)는 NULL

#### 3.3.3 가입/초대 플로우

고객사 사용자는 **셀프 가입이 아닌 초대 기반**으로 생성:

```
1. 내부 PM/Admin이 프로젝트에 고객사(SvcClient) 연결
2. 고객사 담당자 이메일로 초대 링크 발송
3. 초대 링크로 접속 → /client/register?token=xxx
4. 이름/비밀번호 설정 → 계정 생성 (usr_level_code='CLIENT_LEVEL', usr_cli_id 설정)
5. 자동 로그인 → /client/dashboard
```

### 3.6 고객사 이슈 등록

#### 3.6.1 이슈 생성 폼 (고객용 - 간소화)

| 필드 | 필수 | 설명 |
|------|------|------|
| `type` | O | `BUG` / `FEATURE_REQUEST` / `OPINION` / `OTHER` (TASK 제외) |
| `title` | **O** | 이슈 제목 |
| `description` | O | 상세 설명 (리치텍스트) |
| `severity` | O | `CRITICAL` / `MAJOR` / `MINOR` |
| `priority` | X | 기본값 3 (고객이 변경 불가) |
| `project_id` | 자동 | 프로젝트 상세에서 진입 시 자동 설정 |
| `epic_id` / `component_id` | X | 선택적 (WBS 그룹 선택) |
| `attachments` | X | 파일 첨부 (스크린샷, 문서) |

**고객이 설정 불가한 필드:**
- `assignee_id`: 내부 담당자 지정은 내부 직원만 가능
- `visibility`: 고객 이슈는 자동으로 `ENTITY` (해당 법인 내부 전원 조회 가능)
- `status`: 초기값 `OPEN` 고정, 고객이 상태 변경 불가
- `done_ratio`: 내부 직원만 업데이트

## 6. 보안 설계

### 6.1 인증 분리

```typescript
// JWT payload 확장
interface ClientJwtPayload {
  sub: string;        // userId
  email: string;
  level: 'CLIENT_LEVEL';
  role: 'CLIENT_ADMIN' | 'CLIENT_MEMBER';
  cliId: string;      // 소속 고객사 ID
  status: string;
}
```

- 기존 `JwtAuthGuard`를 확장하여 `CLIENT_LEVEL` 사용자도 인증 통과
- `ClientGuard` 신규 가드: `CLIENT_LEVEL`인지 확인
- `DataScopeInterceptor` 확장: `CLIENT_LEVEL` → `scope: 'CLIENT'`, `cliId` 포함

### 6.2 데이터 접근 제어

| 데이터 | 고객 접근 범위 |
|--------|---------------|
| **프로젝트** | `kms_project_clients`에서 자신의 `cli_id`로 연결된 프로젝트만 |
| **이슈** | 해당 프로젝트의 이슈 중 `visibility = 'ENTITY'` 또는 본인이 `reporter`인 이슈 |
| **코멘트** | 해당 이슈의 코멘트 (읽기), 본인 코멘트 (쓰기) |
| **사용자 정보** | 담당자 이름만 노출 (이메일, 부서 등 비공개) |
| **프로젝트 멤버** | PM 이름만 노출 |
| **예산/파일** | 비공개 (고객 공유 설정된 파일만 예외) |

### 6.3 API 격리

```typescript

@UseGuards(JwtAuthGuard, ClientGuard)
```

## 9. 기술 제약사항 및 고려사항

### 9.1 기존 시스템과의 호환성

| 제약 | 대응 |
|------|------|
| `ProjectMember.usrId` FK → `UserEntity` | CLIENT 사용자도 `amb_users`에 존재하므로 호환 |
| `Issue.reporterId` FK → `UserEntity` | 동일하게 호환 |
| `EntityGuard`: `entityId` 필수 체크 | CLIENT 사용자는 `entityId` 없이 `cliId` 기반 → 별도 Guard 필요 |
| `DataScopeInterceptor`: ADMIN/USER만 처리 | CLIENT scope 분기 추가 |
| 기존 API 엔드포인트 | 변경 없음 ← Client 전용 API를 별도 경로(`/client/`)에 추가 |

### 9.2 알림 시스템

- 고객이 등록한 이슈의 상태 변경 시 → 이메일 알림 발송
- 내부 담당자가 코멘트 작성 시 → 고객에게 이메일 알림
- 고객이 코멘트 작성 시 → 내부 담당자에게 인앱 알림

### 9.3 보안 고려사항

- 고객 사용자는 내부 직원 정보 열람 불가 (이름만 노출)
- 고객 이슈의 `visibility`는 강제 `ENTITY` (해당 법인 내부 직원은 모두 조회 가능)
- 고객 사용자 간 서로 다른 고객사의 이슈/프로젝트 접근 불가
- 초대 토큰 만료: 72시간 (보안상 적정 만료 시간)
- Rate limiting: 로그인 시도 10회/분

### 9.4 다국어 지원

- 기존 i18n 체계(ko/en/vi) 동일 적용
- 번역 키 네임스페이스: `client` (신규)
- 고객 포털 UI 텍스트도 `locales/*/client.json`으로 관리

### 9.5 성능 고려사항

- 고객 프로젝트 조회: `kms_project_clients` JOIN이 추가되므로 인덱스 필수
- 이슈 목록: 기존 쿼리에 `WHERE reporter_id = :userId OR pjt_id IN (...)` 조건 추가
