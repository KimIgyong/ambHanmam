# REQ-260412 내부 사용자 삭제 기능

## 1. 개요

통합 사용자 관리 페이지(`/settings/total-users`)의 **내부 사용자** 탭에서 사용자 삭제 기능을 개선한다.
현재 단순 `window.confirm()` 기반의 소프트 삭제만 존재하며, 삭제 미리보기·연관 데이터 정리·완전 삭제가 없음.

**페이지 경로**: `https://ama.amoeba.site/settings/total-users` → "내부 사용자" 탭

## 3. TO-BE 요구사항

### 3.1 기능 요구사항

#### FR-1: 삭제 옵션 모달 (포털 고객 패턴 동일)
- 사용자 이름 클릭 시 **삭제 방법 선택 모달** 표시
  - 사용자 정보 표시 (이름, 이메일, 법인, 역할)
  - **소프트 삭제** 옵션: 사용자 비활성화 + 연관 데이터 정리
  - **완전 삭제** 옵션: 영구 삭제 모달로 전환
- 기존 Trash2 아이콘 버튼 제거

#### FR-2: 소프트 삭제 개선
- 기존: `usr_deleted_at` 설정만 수행
- 개선: 소프트 삭제 시 다음 데이터도 함께 처리
  - `EntityUserRole` 상태를 `INACTIVE`로 변경
  - `TalkChannelMember` 상태를 `LEFT`로 변경
  - 활성 `PushSubscription` 삭제
  - `OAuthToken` / `AuthorizationCode` 삭제

#### FR-3: 삭제 프리뷰 API
- 삭제 전 연관 데이터 건수를 미리 조회하여 표시
  - 법인 역할 수, 대화 수, 투두 수, 워크아이템 수, 캘린더 수, 회의록 수

#### FR-4: 완전 삭제 (Permanent Delete)
- Level 1 (사용자 계정만 삭제):
  - 조직/역할 매핑 삭제
  - 인증/세션 데이터 삭제
  - 알림/푸시 구독 삭제
  - 통계/추적 데이터 삭제
  - 사용자 레코드 물리 삭제
  - **보존**: 대화, 투두, 워크아이템, 캘린더, 회의록, 토크 메시지 (작성자 참조만 NULL로 변경)
- Level 2 (전체 데이터 삭제):
  - Level 1 포함
  - 사용자 소유 대화/메시지 삭제
  - 사용자 소유 투두/워크아이템 삭제
  - 사용자 소유 캘린더 일정 삭제
  - 사용자 소유 회의록 삭제
  - 사용자 소유 업무 보고서 삭제
- 이메일 확인 검증 필수 (포털 고객 패턴과 동일)
- 트랜잭션 처리 (원자성 보장)
- 감사 로그 기록 (`logger.warn`)

#### FR-5: 보안 제약
- `ADMIN_LEVEL` + `ADMIN` 역할 이상만 접근 가능
- `SUPER_ADMIN` 사용자는 삭제 불가 (시스템 보호)
- 자기 자신 삭제 불가
- 삭제 확인 시 대상 사용자 이메일 입력 필수

### 3.2 비기능 요구사항

| 항목 | 요구 |
|------|------|
| **트랜잭션** | 완전 삭제는 단일 트랜잭션 내 실행 (일부 실패 시 전체 롤백) |
| **성능** | 프리뷰 조회 3초 이내, 삭제 실행 10초 이내 |
| **감사** | 모든 삭제 작업 로그 기록 (관리자, 대상, 레벨, 시각) |
| **i18n** | ko/en/vi 3개 언어 지원 |

## 5. 사용자 플로우

### 5.1 소프트 삭제 플로우
```
이름 클릭 → 삭제 옵션 모달 → "소프트 삭제" 선택
→ confirm 다이얼로그 → 확인
→ DELETE /api/v1/members/{id} (기존 개선)
→ 성공 toast → 목록 갱신
```

### 5.2 완전 삭제 플로우
```
이름 클릭 → 삭제 옵션 모달 → "완전 삭제" 선택
→ 완전 삭제 모달 열림
→ GET /api/v1/admin/users/{id}/deletion-preview (프리뷰 로드)
→ 연관 데이터 건수 표시
→ Level 1/2 선택
→ 이메일 입력 확인
→ "완전 삭제 실행" 클릭
→ DELETE /api/v1/admin/users/{id}/permanent
→ 성공 toast → 목록 갱신
```

### 5.3 삭제 불가 케이스
```
SUPER_ADMIN 사용자 → 삭제 옵션 모달에서 두 버튼 모두 비활성화 + 안내 메시지
자기 자신 → 삭제 옵션 모달에서 두 버튼 모두 비활성화 + 안내 메시지
```

## 6. 기술 제약사항

### 6.1 FK 제약 vs 작성자 보존
- 다른 사용자가 참여하는 데이터(이슈 댓글, 토크 메시지 등)에서 작성자 ID가 삭제되면 "알 수 없는 사용자"로 표시해야 함
- Level 1에서는 작성자 참조를 NULL로 SET → 프론트에서 `user?.name || '탈퇴한 사용자'` 패턴 필요
- Level 2에서는 소유 데이터를 통째로 삭제

### 6.2 Unique 제약
- `amb_users` UNIQUE INDEX: `(usr_email, usr_company_id) WHERE usr_deleted_at IS NULL`
- 소프트 삭제 시 동일 이메일로 재가입 가능 (기존 동작 유지)
- 완전 삭제 시 레코드 자체가 사라지므로 문제 없음

### 6.3 토크 메시지 보존 정책
- Level 1: 토크 메시지 보존 (작성자 NULL) → 대화 맥락 유지
- Level 2: 토크 메시지도 삭제 → 대화 맥락 손상 가능 (경고 표시 필요)

### 6.4 기존 포털 고객 삭제와의 일관성
- 동일한 UI 패턴 (이름 클릭 → 삭제 옵션 모달 → 완전 삭제 모달)
- 동일한 보안 패턴 (이메일 확인, 트랜잭션, 감사 로그)
- 연관 테이블 수가 포털 고객(5개)보다 훨씬 많음(30개+) → 레벨 구분이 더 중요
