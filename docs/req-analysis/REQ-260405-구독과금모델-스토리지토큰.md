## 3. TO-BE Requirements / TO-BE 요구사항

### 3.1 Subscription Lifecycle / 구독 생명주기

1. **Entity 가입 시 FREE 플랜 자동 부여**
   - BASE 토큰 지갑에 50,000 토큰 1회 지급, 스토리지 1GB 할당

2. **BASIC 전환 플로우**
   - POST /api/v1/subscriptions/checkout → Polar.sh Checkout URL
   - Polar webhook `subscription.active` → 내부 구독 활성화 + 토큰 지급 + 스토리지 확장

3. **구독 상태 관리**: TRIALING → ACTIVE → PAST_DUE → SUSPENDED → CANCELLED

4. **사용자 수 변경**: 5명 단위 증감, Polar.sh 구독 수정 연동

### 3.2 Token Wallet System / 토큰 지갑 시스템

1. **3종 지갑**: BASE (월 지급), ADDON (유료 충전, 이월), REFERRAL (보상, 이월)
2. **차감 순서**: BASE → ADDON → REFERRAL (만료 예정 토큰 우선 소비)
3. **월 리셋 (BASIC)**: 매월 1일 00:00 KST, BASE 잔여 소멸 + 신규 지급
4. **잔액 가드**: AI 요청 전 총잔액 검사, 0이면 E7001 차단

### 3.3 Storage Quota System / 스토리지 쿼터 시스템

1. **Entity별 할당**: base_gb (플랜 기본) + addon_gb (추가 구매) = total_gb
2. **업로드 검사**: projected > 120% → E7002 차단, > 100% → 경고 헤더
3. **사용량 추적**: 업로드/삭제 시 sqt_used_bytes 증감 (pessimistic lock)
4. **추가 구매**: 5GB 단위, max_gb 상한 내에서 추가

### 3.4 Payment Integration (Polar.sh) / 결제 연동

1. **구독 결제**: Polar Checkout → subscription.active webhook → 프로비저닝
2. **토큰 추가 충전**: Polar Checkout → order.created webhook → ADDON 지갑 적립
3. **스토리지 추가 구매**: Polar Checkout → order.created webhook → addon_gb 증가
4. **웹훅 보안**: Svix HMAC-SHA256 서명 검증, 멱등성 보장 (webhook-id 기반)

### 3.5 Data Model (New Tables) / 데이터 모델 (신규 테이블)

기존 `amb_svc_plans`(spl_)/`amb_svc_subscriptions`(sub_) 테이블은 **변경하지 않는다**.
신규 SaaS 구독 테이블은 `amb_sub_` 도메인 prefix를 사용한다.

| 테이블 | Prefix | 용도 |
|---|---|---|
| `amb_sub_plans` | pln_ | 요금제 마스터 (FREE/BASIC/PREMIUM) |
| `amb_sub_subscriptions` | sbn_ | Entity별 구독 현황 |
| `amb_sub_token_wallets` | tkw_ | 토큰 잔액 지갑 (타입별) |
| `amb_sub_token_ledgers` | tkl_ | 토큰 변동 원장 (불변 이력) |
| `amb_sub_storage_quotas` | sqt_ | 스토리지 할당/사용량 |
| `amb_pg_subscriptions` | pgs_ | Polar.sh 구독 미러 |
| `amb_pg_webhook_events` | pgw_ | Polar.sh 웹훅 이벤트 로그 |

**충돌 방지 전략**: `amb_sub_` prefix로 기존 B2B 서비스관리(`amb_svc_*`)와 완전 분리. 기존 코드/테이블 수정 불필요.

## 5. User Flows / 사용자 플로우

### 5.1 Free → Basic 구독 전환 플로우
1. Entity 관리자가 구독 관리 페이지에서 "Basic으로 업그레이드" 클릭
2. 사용자 수 선택 (6명~ / 5명 단위)
3. POST /api/v1/subscriptions/checkout → Polar Checkout URL 반환
4. Polar 결제 화면에서 카드 등록 + 결제 완료
5. Polar → POST /webhooks/polar (subscription.active)
6. 서버: 구독 ACTIVE 처리 + BASE 토큰 지급 + 스토리지 3GB 확장
7. 사용자에게 success_url 리다이렉트

### 5.2 AI 토큰 차감 플로우
1. 사용자가 AI 채팅 메시지 전송
2. TokenBalanceGuard: 총잔액 검사 → 0이면 E7001 차단
3. Claude API 호출 → 응답 + usage tokens
4. TokenWalletService.deductAiUsage: BASE → ADDON → REFERRAL 순서로 차감
5. TokenLedger DEBIT 기록

### 5.3 스토리지 업로드 플로우
1. 사용자가 파일 업로드 시도
2. StorageQuotaService.checkUpload: used > 120% → E7002 차단, > 100% → 경고
3. 파일 저장 완료 → addUsage(ent_id, file_size_bytes)
4. sqt_is_upload_blocked 플래그 갱신

### 5.4 토큰 추가 충전 플로우
1. 사용자가 토큰 충전 클릭 → 수량 선택 (10,000/50,000 토큰 단위)
2. POST /api/v1/subscriptions/tokens/purchase → Polar Checkout URL
3. 결제 완료 → order.created webhook
4. ADDON 지갑에 토큰 적립 + TokenLedger CREDIT 기록

### 5.5 월 토큰 리셋 (Cron)
1. 매월 1일 00:00 KST
2. ACTIVE + BASIC 구독 목록 조회
3. 각 테넌트: BASE 잔여 소멸(DEBIT RESET) → 신규 지급(CREDIT MONTHLY_GRANT)

## 6. Technical Constraints / 기술 제약사항

1. **테이블 분리**: 기존 service-management 도메인(`amb_svc_plans`/`amb_svc_subscriptions`)은 변경하지 않고, 신규 구독 테이블은 `amb_sub_` prefix를 사용하여 완전 분리한다.

2. **기존 quota 시스템 공존**: `EntityAiConfigEntity` + `ApiQuotaInterceptor` 와 새로운 `TokenBalanceGuard`가 이행 기간 동안 공존해야 함.

3. **스테이징/프로덕션 수동 SQL**: TypeORM synchronize 비활성화 환경이므로 모든 스키마 변경은 수동 SQL 필수.

4. **Mixed storage backends**: 로컬 + Google Drive 공존. 스토리지 쿼터는 로컬 파일만 1차 적용하고, Drive 파일은 향후 확장.

5. **Polar.sh rawBody 필요**: NestJS에서 webhook 서명 검증 위해 rawBody 파싱 활성화 필요.

6. **i18n**: 모든 경고/차단/충전 안내 메시지는 ko/en/vi 번역 키 기반 제공.

7. **데이터 격리**: 모든 신규 API는 `@Auth()` + `@UseGuards(OwnEntityGuard)` + `resolveEntityId()` 필수 적용.
