# 요구사항 분석서: 이메일 구분 체계 정립

- **문서번호**: REQ-이메일구분체계-20260307
- **작성일**: 2026-03-07
- **관련 문서**: REQ-포털MASTER계정이슈-20260305

## 1. 배경

하나의 이메일 주소가 시스템 내에서 3가지 서로 다른 컨텍스트로 사용될 수 있다:

| # | 컨텍스트 | 엔티티 | 테이블 | 유니크 제약 |
|---|----------|--------|--------|-------------|
| 1 | 아메바 포털 사용자 | `PortalCustomerEntity` | `amb_portal_customers` | `pct_email` UNIQUE |
| 2 | 아메바 매니지먼트 서비스 계정 | `UserEntity` | `amb_users` | `usr_email` UNIQUE |
| 3 | 법인별 초대 이메일 | `InvitationEntity` | `amb_invitations` | `inv_email + inv_company_id` 복합 |

**핵심 요구**: 동일 이메일이 위 3가지 컨텍스트에서 각각 독립적으로 존재할 수 있어야 하며, 법인 내에서만 동일 이메일 중복이 방지되어야 한다.

## 2. AS-IS 현황 분석

### 2.1 포털 ↔ 매니지먼트 관계

```
PortalCustomerEntity (pctEmail: UNIQUE)
    │
    └─ PortalUserMappingEntity (pctId ↔ usrId)
           │
           └─ UserEntity (usrEmail: UNIQUE)
```

- 포털과 매니지먼트는 **완전히 분리된 시스템**
- `PortalCustomerEntity`와 `UserEntity`는 별도 테이블, 별도 UNIQUE 제약
- 동일 이메일이 포털과 매니지먼트 양쪽에 존재 가능 → **이미 지원됨**

### 2.2 매니지먼트 계정 ↔ 법인 관계

```
UserEntity (usrEmail: UNIQUE)
    │
    ├─ EntityUserRoleEntity (usrId + entId: UNIQUE) → 법인 A
    ├─ EntityUserRoleEntity (usrId + entId: UNIQUE) → 법인 B
    └─ EntityUserRoleEntity (usrId + entId: UNIQUE) → 법인 C
```

- 한 사용자(`UserEntity`)가 여러 법인에 소속 가능
- `EntityUserRoleEntity`의 UNIQUE 제약은 `(entId, usrId)` → 동일 법인 내 같은 사용자 중복 방지
- **이미 1:N 구조 지원됨**

### 2.3 초대 시스템 현황

```typescript
// invitation.service.ts — 중복 체크 로직
const existingInvitation = await this.invitationRepo.findOne({
  where: {
    invEmail: dto.email,
    invCompanyId: companyId,  // 법인별 체크
    invStatus: In(['PENDING', 'ACCEPTED']),
  },
});
```

- 초대는 **법인 단위**로 이메일 중복 체크
- 같은 이메일 + 같은 법인 = 거부
- 같은 이메일 + 다른 법인 = 허용
- **이미 요구사항 충족**

### 2.4 회원가입 흐름

```typescript
// auth.service.ts — register()
const existingUser = await this.userRepo.findOne({
  where: { usrEmail: dto.email },
});
if (existingUser) {
  // 이미 존재하는 사용자 → EntityUserRoleEntity만 추가
} else {
  // 신규 사용자 생성 + EntityUserRoleEntity 생성
}
```

- 초대 토큰으로 회원가입 시, 이미 존재하는 이메일이면 기존 UserEntity에 새 법인 역할만 추가
- **이미 다법인 소속 지원**

## 4. 결론

### 4.1 이메일 구분 체계

**현재 시스템은 이미 3단계 이메일 구분을 올바르게 지원하고 있다:**

1. **포털 이메일** (`PortalCustomerEntity.pctEmail`): 포털 시스템 내 독립 관리
2. **서비스 계정 이메일** (`UserEntity.usrEmail`): 매니지먼트 시스템 내 글로벌 유니크
3. **법인별 초대 이메일** (`InvitationEntity.invEmail`): 법인 단위로 유니크 체크

### 4.2 실제 수정 필요 사항

현재 구조적으로는 이메일 구분 체계가 정상이며, 실제 이슈는 **포털 브릿지 계정 생성 시 누락 데이터** 문제이다.

이 이슈는 이미 `REQ-포털MASTER계정이슈-20260305`에서 분석 완료되었으며, `PLAN-포털MASTER계정이슈-작업계획-20260305`에 구현 계획이 수립되어 있다.

### 4.3 구현 우선순위

| 순위 | 작업 | 관련 문서 |
|------|------|-----------|
| 1 | portal-bridge.service.ts에 EntityUserRoleEntity 생성 추가 | PLAN-포털MASTER계정이슈 |
| 2 | portal-bridge.service.ts에 EmployeeEntity 자동 생성 추가 | PLAN-포털MASTER계정이슈 |
| 3 | 다법인 MASTER 생성 시 usrCompanyId 처리 로직 검토 | 본 문서 3.3 |

## 5. 사용자 플로우

### 5.1 정상 플로우: 초대를 통한 다법인 소속

```
1. 법인 A 관리자 → user@email.com 초대
2. user@email.com → 초대 토큰으로 회원가입 → UserEntity 생성 + EntityUserRoleEntity(법인A)
3. 법인 B 관리자 → user@email.com 초대 (다른 법인이므로 허용)
4. user@email.com → 초대 토큰으로 가입 → 기존 UserEntity에 EntityUserRoleEntity(법인B) 추가
5. 로그인 시 법인 A, B 모두 접근 가능
```

### 5.2 포털 브릿지 플로우 (수정 필요)

```
1. 포털 고객(portal@email.com) → ADMIN이 내부 계정 생성
2. UserEntity 생성/업데이트
3. ★ EntityUserRoleEntity 생성 (현재 누락 → 수정 필요)
4. ★ EmployeeEntity 생성 (현재 누락 → 수정 필요)
5. PortalUserMappingEntity 생성
6. 임시 비밀번호 이메일 발송
```
