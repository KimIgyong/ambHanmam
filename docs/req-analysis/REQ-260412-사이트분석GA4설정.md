   - www.amoeba.site → `G-H83THZBDHT` (Portal)
   - ama.amoeba.site → `G-S8K2WQYGM9` (App)

## 3. 갭 분석 (GAP Analysis)

### 3.1 핵심 갭: 데이터 소스 불일치

| 항목 | Portal-web (클라이언트) | Admin 대시보드 (서버) | 갭 |
|------|------------------------|---------------------|-----|
| **데이터 전송 대상** | GA4 (`window.gtag`) only | `amb_site_event_logs` 테이블에서 조회 | ⚠️ **Portal-web이 백엔드 이벤트 API를 미호출** |
| **이벤트 종류** | 16개 세분화된 퍼널 이벤트 | 4개 (page_view, login, register_visit, subscription) | ⚠️ **이벤트 타입 미매칭** |
| **데이터 흐름** | portal-web → GA4 | admin-web ← api ← `amb_site_event_logs` (비어있음) | ⚠️ **데이터가 수집되지 않음** |

> **Root Cause**: Portal-web은 GA4 클라이언트 트래킹만 구현되어 있고, 백엔드 `POST /analytics/events` API를 호출하지 않음. 따라서 Admin 대시보드의 Portal 탭은 데이터가 비어있는 상태.

## 4. TO-BE 요구사항

### 4.1 GA4 설정 확장 (GA Settings 탭)

#### R1: 2개 사이트 Measurement ID 관리

**DB 변경**: `amb_site_settings` 테이블에 `sts_app_ga_measurement_id` VARCHAR(30) 컬럼 추가

**UI 변경**: GA Settings 페이지에 App 사이트(ama.amoeba.site) 입력 필드 추가

| 항목 | 라벨 | 기본값 | 설명 |
|------|------|--------|------|
| Portal Site | www.amoeba.site | `G-H83THZBDHT` | 기존 필드 유지 |
| App Site | ama.amoeba.site | `G-S8K2WQYGM9` | **신규 추가** |

**백엔드 변경**:
- `SiteSettingsEntity`에 `stsAppGaMeasurementId` 필드 추가
- `getGaSettings()`: 2개 ID 반환
- `updateGaSettings()`: `app_ga_measurement_id` 파라미터 추가 → DB 저장

### 4.2 Portal 이벤트 수집 파이프라인 구축

#### R2: Portal-web에서 백엔드 이벤트 API 호출

Portal-web의 핵심 이벤트 시점에 `POST /analytics/events` 호출을 추가하여, GA4 클라이언트 트래킹과 병행으로 백엔드에도 데이터 수집.

**수집 대상 이벤트** (admin 대시보드 표시용):

| 클라이언트 이벤트 | 백엔드 event_type | page_path | metadata |
|------------------|------------------|-----------|----------|
| `landing_page_view` | `page_view` | `/` | `{ source: 'landing' }` |
| `register_page_view` | `register_visit` | `/register` | `{ step: number }` |
| `signup_go_ama_click` | `ama_navigation` | `/signup-complete` | — |
| `newsletter_subscribe_success` | `subscription` | `/` | `{ email_domain }` |
| (모든 페이지 진입) | `page_view` | 현재 pathname | `{ referrer }` |

**구현 방식**: portal-web에 이벤트 전송 유틸(`trackServerEvent`)을 생성하고, ga-events.ts의 기존 클라이언트 트래킹 함수 내에서 병렬 호출.

**API 엔드포인트**: 기존 `POST /api/v1/analytics/events` 활용 (인증 불필요 - 익명 이벤트 허용됨)

> **CORS 주의**: `www.amoeba.site` (portal-web) → `ama.amoeba.site/api/v1` (api)로의 크로스도메인 요청이므로, API 서버의 CORS 설정에 `www.amoeba.site` 허용 필요.
> 또는 portal-api를 경유하여 api로 프록시하는 방안 검토.

### 4.3 Portal Analytics 대시보드 개선

#### R3: 퍼널 핵심 지표 카드 개선

현재 서머리 카드를 요청된 3가지 지표 중심으로 재구성:

| 카드 | 이벤트 타입 | 아이콘 | 색상 |
|------|-----------|--------|------|
| 랜딩페이지 조회수 | `page_view` (page_path = `/`) | Eye | blue |
| 가입 페이지 진입 | `register_visit` | UserPlus | purple |
| AMA 이동 횟수 | `ama_navigation` | ExternalLink | green |
| 뉴스레터 구독 | `subscription` | Mail | orange |

#### R4: Registration Funnel 시각화 (선택사항)

퍼널 단계별 전환율을 보여주는 차트:
```
Landing (100%) → Register (xx%) → Step1 (xx%) → Step2 (xx%) → Complete (xx%) → Go AMA (xx%)
```

> 구현 우선순위: 서머리 카드 개선이 필수이며, 퍼널 시각화는 2차 작업으로 분류.

### 4.4 Portal-web 이벤트 트래킹 API 호출 경로

**방안 A: portal-web → api 직접 호출 (CORS)**
- 장점: 단순한 아키텍처
- 단점: CORS 설정 필요, api URL을 portal-web 환경변수에 추가 필요

**방안 B: portal-web → portal-api → api 프록시**
- 장점: 동일 도메인(www.amoeba.site) 내 통신, CORS 불필요
- 단점: portal-api에 프록시 엔드포인트 추가 필요

**결정**: 방안 B 채택 — portal-api에 `POST /analytics/events` 프록시 엔드포인트를 추가하고, portal-api가 내부적으로 DB에 직접 쓰거나 api를 호출.

> portal-api도 동일 DB를 사용하므로, portal-api에서 직접 `amb_site_event_logs`에 INSERT 하는 것이 가장 효율적.

## 6. 기술 제약사항

1. **CORS**: portal-web(www.amoeba.site) → api(ama.amoeba.site) 크로스도메인 → portal-api 프록시로 우회
2. **GA4 Data API 미사용**: Google Cloud 서비스 계정 설정 / OAuth2 복잡도 회피 → 자체 이벤트 로그 방식 유지
3. **이벤트 볼륨**: 현재 트래픽 수준에서는 `amb_site_event_logs` 직접 쿼리로 충분 (추후 집계 테이블 검토)
4. **GA4 index.html 동적 반영**: 현재 하드코딩으로 정상 동작. DB 설정값의 실시간 반영은 범위 외 (빌드 타임 인라인)
5. **portal-api DB 접근**: portal-api가 `amb_site_event_logs` 테이블에 직접 INSERT 가능 (동일 DB)
