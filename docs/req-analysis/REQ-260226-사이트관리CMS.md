### 1.3 메뉴/권한 시스템 현황

ambKMS는 DB 기반 메뉴 설정 + 3계층 권한 시스템을 사용:

| 테이블 | 역할 |
|--------|------|
| `amb_menu_config` | 사이드바 메뉴 항목 (menuCode, labelKey, icon, path, category, sortOrder) |
| `amb_menu_permissions` | 역할(Role)별 메뉴 접근 권한 (SUPER_ADMIN/ADMIN/MANAGER/MEMBER/VIEWER) |
| `amb_menu_group_permissions` | 그룹별 메뉴 접근 권한 (추가 필터) |
| `amb_user_menu_permissions` | 사용자별 개별 메뉴 접근 권한 (최우선) |

**메뉴 등록 방식**: `menu-config.service.ts`의 `DEFAULT_MENU_CONFIGS` 배열에 항목 추가 → 서버 시작 시 `onModuleInit()`에서 DB에 없는 항목 자동 INSERT.

**권한 등록 방식**: `menu-code.constant.ts`의 `MENU_CODES` 객체 + `DEFAULT_PERMISSIONS` 객체에 각 역할별 기본값 추가 → 서버 시작 시 자동 INSERT.

**사이드바 렌더링**: `MainLayout.tsx`에서 `useMyMenus()` → `GET /settings/permissions/me` API 호출 → 허용된 메뉴만 사이드바에 표시. `ICON_MAP` + `COLOR_MAP`에 아이콘 매핑 필요.

**어드민 설정 메뉴**: `settings/permissions` 페이지의 "메뉴 설정" 탭에서 활성화/순서 관리, "역할별 권한" 탭에서 역할별 접근 토글 → 신규 menuCode가 `DEFAULT_MENU_CONFIGS`에 등록되면 자동으로 관리 UI에 노출.

### 2.3 데이터 모델 (9개 신규 테이블)

```
amb_cms_menus (cmn_)          — CMS 메뉴 (2단계 트리, self-ref)
  ├── 1:1 → amb_cms_pages (cmp_)          — 페이지 (유형 8종, 상태, 메타데이터)

  │         ├── 1:N → amb_cms_page_versions (cpv_)   — 발행 버전 스냅샷
  │         ├── 1:N → amb_cms_sections (cms_)         — 섹션 (LANDING/SERVICE_INFO)
  │         ├── 1:N → amb_cms_posts (cpt_)            — 게시글 (BOARD/BLOG)
  │         │         └── 1:N → amb_cms_post_attachments (cpa_) — 첨부파일
  │         ├── 1:N → amb_cms_post_categories (cpg_) — 게시글 카테고리
  │         └── 1:N → amb_cms_subscribers (csb_)      — 구독자 (SUBSCRIPTION)
  └── self-ref → cmn_parent_id (상위 메뉴)
```

### 2.4 API 엔드포인트 (30+ 신규)

**Admin API** (인증 필요, Admin/Manager Guard):
- 메뉴: GET/POST/PUT/PATCH/DELETE `/api/v1/cms/menus` (5개)
- 페이지: GET/PUT (목록, 상세, 메타수정, 콘텐츠저장) `/api/v1/cms/pages` (4개)
- 발행: POST `/api/v1/cms/pages/:id/publish|unpublish|preview` (3개)
- 버전: GET/POST `/api/v1/cms/pages/:id/versions|rollback` (2개)
- 섹션: GET/POST/PUT/PATCH/DELETE `/api/v1/cms/pages/:id/sections` (5개)
- 게시글: GET/POST/PUT/DELETE `/api/v1/cms/pages/:pageId/posts` (4개)
- 구독자: GET (목록, CSV) `/api/v1/cms/pages/:pageId/subscribers` (2개)

**Public API** (인증 불필요, portal-web 공개 Read-only):
- GET `/api/v1/cms/public/menus|pages|posts|preview` (6개)
- POST `/api/v1/cms/public/pages/:slug/subscribe` (1개)

## 4. 사용자 플로우 (이벤트 시나리오 5건)

### 시나리오 1: 서비스 안내 페이지 생성

```
Admin → 사이트관리 > 메뉴관리 진입
  → [+ 메뉴 추가] 클릭
  → 메뉴명: "AmoebaTalk", slug: "talk", 아이콘 선택
  → 페이지 유형: "서비스 안내" 선택
  → [생성] → 메뉴 + 페이지 + 기본 섹션 4개 자동 생성 (Hero/Features/FAQ/CTA)
  → 페이지 편집 화면으로 자동 이동
  → 각 섹션 편집 (Hero: 제목/부제/CTA/배경, Features: 기능항목 등)
  → [미리보기] → 새 탭에서 portal-web 프리뷰
  → [발행] → 확인 모달 → 버전 1 생성, PUBLISHED 상태
```

**예외**: slug 중복 시 "이미 사용 중인 URL" 에러, 이미지 6MB 업로드 시 크기 초과 에러, 콘텐츠 없이 발행 시 에러

```
Admin → 메뉴 추가: "공지사항", slug: "notices", 유형: "게시판"
  → 메뉴 + 게시판 페이지 + "공지" 기본 카테고리 생성
```

| 이미지 업로드 | 현재 `50mb` body limit, Google Drive 연동 사용 중 | 기존 Drive 업로드 API 재사용 |
| TipTap 버전 | 3.x 계열 (image, link, placeholder, underline, react, starter-kit) | 추가 extension 필요 시 호환성 확인 |
| @dnd-kit 버전 | core 6.x, sortable 10.x, utilities 3.x | 트리 구현 시 sortable 기반 중첩 가능 |
| CORS | Public API 사용 시 `www.amoeba.site` 도메인 허용 필요 | `CORS_ORIGINS` 환경변수에 추가 |
| 다국어 | 시스템 UI는 en/ko/vi 3개, CMS 콘텐츠는 en/ko 2개 | 콘텐츠 언어와 UI 언어 분리 관리 |
