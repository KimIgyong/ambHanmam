# REQ-이슈리플라이멘션이모지알림-20260402

## 1. 요구사항 요약

이슈 시스템의 알림 기능을 강화하여 사용자 간 인터랙션 시 적절한 알림이 발송되도록 개선한다.

| # | 요구사항 | 설명 |
|---|---------|------|
| R1 | 리플라이 멘션/이모티콘 | 답글 에디터에 멘션(@) 자동완성 및 이모지 리액션 버튼 추가 |
| R2 | 리플라이 자동 멘션 알림 | 답글에 별도 @멘션 없으면, 원래 코멘트 작성자에게 자동 알림 |
| R3 | 이모지 리액션 알림 | 이슈/코멘트에 이모지 클릭 시 "OO님이 당신의 게시물에 👍를 눌렀습니다" 멘션 알림 |
| R4 | 이슈 Rating 알림 | 별점 등록 시 "OO님이 당신의 이슈에 별 N개를 주셨습니다" 알림 |

## 2. AS-IS 현황 분석

### 2.1 답글(Reply) 시스템

| 항목 | 현황 |
|------|------|
| **DB 엔티티** | `amb_issue_comments.isc_parent_id` (self-join, 1-depth 제한) |
| **백엔드** | `IssueService.addIssueComment(issueId, content, userId, authorType, parentId?)` |
| **프론트엔드** | `IssueDetailModal.tsx` — `replyingTo` state로 답글 대상 추적 |
| **답글 에디터** | `RichTextEditor` 사용하지만 **`enableMention` prop 미설정** (= 멘션 비활성) |
| **이모지 리액션** | **답글에는 이모지 버튼 미렌더링** (부모 코멘트만 이모지 표시) |

### 2.2 멘션(@) 시스템

| 항목 | 현황 |
|------|------|
| **프론트엔드** | `RichTextEditor.tsx` — `enableMention={true}` 시 `@` 입력하면 `MentionPortalDropdown` 표시 |
| **백엔드 파싱** | `mention.util.ts` — `extractMentionedUserIds(content, users, authorId)` 정규식 `/@([^\s@]+)/g` |
| **알림 트리거** | `issue.service.ts:L814-L852` — 코멘트 생성 시 멘션 감지 → `MENTION_EVENT` emit |
| **멘션 없을 때** | assignee + participants 전원에게 `MENTION_EVENT` emit |
| **리플라이 알림** | ⚠️ **원래 코멘트 작성자에게 별도 알림 미구현** — 리플라이도 일반 코멘트와 동일 로직 적용 |

### 2.3 이모지 리액션 시스템

| 항목 | 현황 |
|------|------|
| **DB** | `amb_issue_comment_reactions` (icr_id, isc_id, usr_id, icr_type) |
| **타입** | LIKE, CHECK, PRAY, GRIN, LOVE (5종) |
| **백엔드** | `IssueService.toggleCommentReaction()` — 토글 방식 (있으면 삭제, 없으면 생성) |
| **프론트** | 각 코멘트 아래 5개 이모지 버튼, `useToggleCommentReaction` 훅 |
| **알림** | ⚠️ **리액션 시 알림 미구현** — 리액션 토글만 수행, 이벤트 미발행 |

### 2.4 별점(Rating) 시스템

| 항목 | 현황 |
|------|------|
| **DB** | `amb_issue_ratings` (isr_id, iss_id, usr_id, isr_rating CHECK 1-5) |
| **백엔드** | `ContentRatingService.upsertIssueRating()` — upsert + 자기 이슈 평가 금지 |
| **프론트** | `StarRating` 컴포넌트, `useUpsertIssueRating` 훅 |
| **알림** | ⚠️ **별점 등록 시 알림 미구현** — 레이팅만 저장, 이벤트 미발행 |

### 2.5 알림 시스템

| 항목 | 현황 |
|------|------|
| **알림 유형** | TODO_ASSIGNED, ISSUE_ASSIGNED, ISSUE_STATUS_CHANGED, NOTE_ASSIGNED, CALENDAR_INVITED, COMMENT_MENTION, TALK_MESSAGE |
| **이벤트** | EventEmitter2: ASSIGNEE_EVENT, MENTION_EVENT, ISSUE_STATUS_EVENT |
| **리스너** | `NotificationListener` — 이벤트 수신 → DB 저장 → SSE 전송 → Web Push |
| **SSE** | `NotificationSseService` — `Subject<NotificationSseEvent>` (RxJS) → `GET /notifications/stream` |

## 3. TO-BE 요구사항

### 3.1 R1: 답글 에디터 멘션/이모지

**프론트엔드 변경**:
- `IssueDetailModal.tsx` 답글 `RichTextEditor`에 `enableMention={true}` 추가
- 답글 렌더링에 이모지 리액션 버튼 추가 (부모 코멘트와 동일 UI)
- 답글 이모지 데이터는 이미 `buildCommentReactionMap`에서 `commentIds`에 답글 ID도 포함되므로 별도 API 불필요

### 3.2 R2: 리플라이 자동 멘션 알림

**백엔드 변경**:
- `issue.service.ts:addIssueComment()` — 답글(`parentId` 있음)인 경우:
  1. 멘션 감지 (`extractMentionedUserIds`)
  2. 멘션 없으면 → **원래 코멘트 작성자(`parentComment.iscAuthorId`)에게 자동 알림**
  3. 기존 로직(assignee+participants 전원 알림)은 답글이 아닌 경우에만 적용

### 3.3 R3: 이모지 리액션 알림

**새 이벤트 타입 추가**:
- `REACTION_EVENT` = 'comment.reaction'
- `NOTIFICATION_TYPE.COMMENT_REACTION` = 'COMMENT_REACTION'
- 페이로드: `{ resourceType, resourceId, commentId, reactionType, senderId, senderName, recipientId, entityId }`

**백엔드 변경**:
- `IssueService.toggleCommentReaction()` — 리액션 **추가** 시 (삭제 아닐 때) 이벤트 emit
- 코멘트 작성자 본인이 자기 코멘트에 리액션 → 알림 미발송
- `NotificationListener` — `@OnEvent(REACTION_EVENT)` 핸들러 추가
- 알림 메시지: `"{senderName}님이 코멘트에 {emoji}를 눌렀습니다."`

### 3.4 R4: 이슈 Rating 알림

**새 이벤트 타입 추가**:
- `RATING_EVENT` = 'content.rating'
- `NOTIFICATION_TYPE.CONTENT_RATING` = 'CONTENT_RATING'
- 페이로드: `{ resourceType, resourceId, resourceTitle, rating, senderId, senderName, recipientId, entityId }`

**백엔드 변경**:
- `ContentRatingService.upsertIssueRating()` — 레이팅 저장 후 이벤트 emit
- 수신자: 이슈 `reporterId` (작성자)
- `NotificationListener` — `@OnEvent(RATING_EVENT)` 핸들러 추가
- 알림 메시지: `"{senderName}님이 이슈에 ⭐{rating}개를 주셨습니다."`
- ContentRatingService에 EventEmitter2 주입 필요
- User 정보 조회 필요 (sender 이름)

## 5. 사용자 플로우

### 5.1 답글 멘션 플로우
```
사용자가 답글에서 @입력 → MentionPortalDropdown 표시 → 멤버 선택 →
제출 → 백엔드 멘션 파싱 → MENTION_EVENT → 멘션된 사용자에게 알림
```

### 5.2 답글 자동 알림 플로우
```
사용자가 답글 작성 (멘션 없음) → 제출 →
백엔드: parentId 있음 → parentComment.authorId 조회 →
작성자 ≠ 원래 코멘트 작성자이면 → MENTION_EVENT emit (원래 작성자에게)
```

### 5.3 이모지 리액션 알림 플로우
```
사용자가 코멘트/답글의 이모지 클릭 → toggleCommentReaction API →
리액션 추가됨(삭제 아님) → REACTION_EVENT emit →
NotificationListener → 코멘트 작성자에게 알림 + SSE + Push
```

### 5.4 별점 알림 플로우
```
사용자가 이슈에 별점 등록 → upsertIssueRating API →
RATING_EVENT emit → NotificationListener →
이슈 작성자(reporterId)에게 알림 + SSE + Push
```

## 6. 기술 제약사항

| 항목 | 제약 |
|------|------|
| `buildCommentReactionMap` | 답글 commentId도 `commentIds` 배열에 포함시켜야 리액션 데이터를 받을 수 있음 → 기존 코드 확인 필요 |
| ContentRatingService DI | EventEmitter2 주입 + UserRepository 주입 필요 (sender 이름 조회용) |
| 알림 중복 방지 | 같은 이모지 연속 토글 시 알림 도배 방지 → 추가 시에만 발송, 삭제 시 미발송 |
| 자기 자신 제외 | 본인 코멘트에 리액션/본인 이슈에 별점 → 알림 미발송 |
| 답글-답글 멘션 | 답글의 원래 코멘트 작성자 알림은 새 동작, 기존 "멘션 없을 때 assignee+participants 전원" 로직과 분리 필요 |
| DB 스키마 변경 | 없음 — 기존 `amb_notifications` 테이블의 `ntf_type`은 varchar이므로 새 타입 추가 시 DDL 불필요 |
