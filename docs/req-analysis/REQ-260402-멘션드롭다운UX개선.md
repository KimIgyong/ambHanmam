# REQ-멘션드롭다운UX개선-20260402

## 1. 요구사항 요약

코멘트 에디터에서 `@` 멘션 시 사용자 선택 드롭다운이 에디터 내부에 갇혀 표시되어 선택이 불편한 문제를 개선한다.

## 2. AS-IS 현황 분석

### 2.1 현재 구현 구조

```
RichTextEditor.tsx (line 576)
┌─ <div className="overflow-hidden rounded-lg border ...">  ← 🔴 overflow-hidden!
│  ├─ <div className="flex-1 overflow-y-auto">               ← 스크롤 컨테이너
│  │  ├─ <div> (toolbar: B I U H2 ...)                        ← sticky toolbar
│  │  ├─ <div className="relative">                           ← 에디터 영역
│  │  │  ├─ <EditorContent editor={editor} />                 ← ProseMirror 에디터
│  │  │  ├─ 🔴 멘션 드롭다운 (absolute top-full z-50)         ← 에디터 밖으로 나갈 수 없음
│  │  │  └─ WikiLink 드롭다운 (absolute)
│  │  └─ ...
│  └─ ...
└─ </div>
```

### 2.2 문제 원인

| # | 원인 | 영향 |
|---|------|------|
| 1 | 최외곽 `overflow-hidden` | absolute 요소가 에디터 영역 밖으로 나갈 수 없음 |
| 2 | `overflow-y-auto` 내부 배치 | 스크롤 컨테이너 안에서도 드롭다운이 잘림 |
| 3 | IssueDetailModal `overflow-y-auto` | 모달 자체도 스크롤 영역이라 추가 클리핑 발생 |

### 2.3 영향 범위

- Issues 코멘트 (IssueDetailModal.tsx → RichTextEditor enableMention)
- Meeting Notes 코멘트 (NoteCommentSection.tsx → RichTextEditor enableMention)
- Reply 입력에서도 동일 문제 발생

## 3. 업계 표준 멘션 UX 벤치마킹

### 3.1 주요 SaaS 제품의 멘션 패턴

| 제품 | 패턴 | 드롭다운 위치 | 기술 방식 |
|------|------|--------------|----------|
| **Slack** | Portal + Floating | 커서 위치 기준 위/아래 자동 | React Portal + @floating-ui |
| **Notion** | Portal + Command Palette | 커서 인라인 아래 | Portal to body + getBoundingClientRect |
| **GitHub** | Portal Overlay | 입력 아래 고정 | Portal + offset 계산 |
| **Linear** | Portal Popup | 커서 위치 기준 | Tippy.js + Portal |
| **Jira** | Portal Dropdown | @ 타이핑 위치 아래 | Atlaskit + Layer Manager |
| **Google Docs** | Inline Popup | 커서 바로 아래 | 자체 레이어 매니저 |

### 3.2 공통 핵심 패턴: **React Portal**

모든 주요 제품이 사용하는 핵심 기법은 **React Portal(`createPortal`)을 통해 `document.body`에 드롭다운을 렌더링** 하는 것이다.

```
장점:
- 어떤 overflow 설정이든 영향받지 않음
- z-index 스택 컨텍스트에서 자유로움
- 뷰포트 경계 감지로 위/아래 자동 전환 가능
```

### 3.3 위치 결정 전략 비교

| 전략 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. 커서 추적형** | ProseMirror 커서 좌표 기반 | 가장 자연스러운 UX, Notion/Slack 스타일 | 구현 복잡도 높음, 커서 이동 시 위치 재계산 |
| **B. 에디터 하단 고정형** | 에디터 영역 바로 아래 고정 | 단순 구현, 안정적, GitHub 스타일 | 에디터가 길면 드롭다운이 멀리 떨어짐 |
| **C. 에디터 상단 고정형** | 에디터 영역 바로 위 | 현재와 유사, 스크롤 영향 적음 | 상단에 공간 없으면 잘림 |

## 4. AMA에 맞는 개선안

### 4.1 권장안: **B안 (에디터 하단 Portal)** + 뷰포트 감지 flip

#### 선택 근거
1. AMA의 코멘트 에디터는 minHeight=80px로 작은 편 → 에디터 바로 아래/위가 자연스러움
2. 커서 추적형(A안)은 tiptap `@tiptap/suggestion` 플러그인 전면 교체 필요 → 영향 범위 과대
3. Portal만 추가하면 현재 코드 변경 최소화 가능
4. GitHub/Linear 스타일의 깔끔한 UX 달성 가능

#### 동작 방식
```
1. @를 입력하면 멘션 드롭다운 활성화
2. ReactDOM.createPortal로 document.body에 렌더링
3. 에디터 wrapper의 getBoundingClientRect()로 위치 계산
4. 기본: 에디터 아래 | 아래 공간 부족 시: 에디터 위로 flip
5. 뷰포트 밖으로 넘치지 않도록 clamp 처리
```

#### 개선 전후 비교

**Before (현재)**
```
┌─────────────────────────┐
│ RichTextEditor          │
│  B I U H2 ...           │
│  @Admin                 │
│  ┌──────────────┐       │  ← 에디터 안에 갇혀있음
│  │ Admin Holding│       │
│  └──────────────┘       │
└─────────────────────────┘
          ↑ overflow-hidden으로 잘릴 수 있음
```

**After (개선)**
```
┌─────────────────────────┐
│ RichTextEditor          │
│  B I U H2 ...           │
│  @Admin                 │
└─────────────────────────┘
┌────────────────────────────┐  ← Portal로 body에 렌더링
│ 👤 Admin        Holding    │
│ 👤 Admin2       IT Dept    │
│ 👤 AdminUser    Sales      │
└────────────────────────────┘
     ↑ overflow 제약 없음, 넉넉한 공간
```

```

```

## 5. 기술 제약사항

1. **tiptap 공식 Mention extension 미사용**: 현재 커스텀 구현이므로 공식 Suggestion 플러그인으로 전환 시 대규모 리팩터링 필요 → 현재 구조 유지하면서 Portal만 추가하는 것이 효율적
2. **z-index 충돌**: IssueDetailModal(z-50) 위에 Portal이 표시되려면 z-index: 9999+ 필요
3. **SSR 호환**: `document.body` 참조는 클라이언트에서만 → 현재 SPA이므로 문제없음
