## 1. AS-IS 현황 분석

### 1.1 문제 현상
- 검색 필드에서 사용자가 텍스트를 입력하다가 잠시 멈추면(300ms), 입력이 완료되지 않았음에도 시스템이 즉시 검색을 실행
- 검색 실행 시 목록이 리렌더링되면서 **입력 필드의 포커스가 유실**되거나 **입력이 일시적으로 차단**될 수 있음
- Debounce가 적용되지 않은 페이지에서는 **매 키입력마다** API 호출 발생

### 1.2 검색 필드 현황 분류

#### A. Debounce 300ms 적용 (5건) - 너무 짧은 타이머
| 컴포넌트 | 파일 | 패턴 |
|---------|------|------|
| GlobalSearchBar | `domain/search/components/GlobalSearchBar.tsx` | `setTimeout 300ms → setDebouncedQuery` |
| IssueFilters | `domain/issues/components/IssueFilters.tsx` | `setTimeout 300ms → onChange(filters)` |
| IssueFormModal (상위이슈검색) | `domain/issues/components/IssueFormModal.tsx` | `setTimeout 300ms → issueService.getIssues()` |
| DocumentsPage | `domain/documents/pages/DocumentsPage.tsx` | `setTimeout 300ms → setSearchQuery` |
| ScopedTodoList | `domain/todos/components/ScopedTodoList.tsx` | `setTimeout 300ms → setDebouncedSearch` |

#### B. Debounce 미적용 - 즉시 검색 (API 호출) (8건+)
| 페이지 | 파일 | 영향 |
|--------|------|------|
| 계약 목록 | `domain/billing/pages/ContractListPage.tsx` | 매 키입력 시 API 호출 |
| 직원 목록 | `domain/hr/pages/EmployeeListPage.tsx` | 매 키입력 시 API 호출 |
| 프로젝트 목록 | `domain/project/pages/ProjectListPage.tsx` | 매 키입력 시 API 호출 |
| 고객 목록 | `domain/service-management/pages/ClientListPage.tsx` | 매 키입력 시 API 호출 |
| 파트너 목록 | `domain/billing/pages/PartnerListPage.tsx` | 매 키입력 시 API 호출 |
| 매출/매입 목록 | `domain/billing/pages/` | 매 키입력 시 API 호출 |
| 관리자 사용자 | `domain/admin/components/InternalUsersTab.tsx` | 매 키입력 시 API 호출 |

#### C. 클라이언트 필터링 (문제 없음)
| 컴포넌트 | 방식 |
|---------|------|

| NewDmModal | useMemo 로컬 필터링 |

### 1.3 근본 원인
1. **Debounce 타이머 300ms가 너무 짧음**: 한글 입력 시 자음+모음 조합 과정에서 300ms는 쉽게 초과
2. **Debounce 미적용 페이지 존재**: 목록 페이지의 검색 필드에서 `onChange` 즉시 상위 state 변경 → API 호출
3. **검색 실행 시 리렌더링으로 포커스 유실**: 부모 컴포넌트 state 변경 → 자식 컴포넌트 리마운트 가능성

## 2. TO-BE 요구사항

### 2.1 핵심 요구사항
1. **검색 중 입력 필드 계속 수정 가능**: 검색 API 호출 중에도 입력 필드의 포커스와 텍스트 편집이 유지되어야 함
2. **적절한 Debounce 적용**: 모든 API 기반 검색 필드에 500ms debounce 적용
3. **일관된 UX**: 프로젝트 전체에서 동일한 검색 동작 패턴 적용

### 2.2 상세 요구사항

| # | 요구사항 | 우선순위 |
|---|---------|---------|
| R1 | 기존 300ms debounce를 500ms로 상향 | 높음 |
| R2 | Debounce 미적용 목록 페이지에 debounce 추가 | 높음 |
| R3 | 검색 실행 중 입력 필드 포커스/값 유지 보장 | 높음 |
| R4 | 공용 `useDebounce` 훅으로 패턴 통일 | 중간 |

## 3. 갭 분석

| 항목 | AS-IS | TO-BE | 갭 |

| Debounce 타이머 | 300ms (일부) / 없음 (일부) | 500ms 일괄 적용 | debounce 타이머 변경 + 미적용 페이지 추가 |

| 입력 중 포커스 | 리렌더링 시 유실 가능 | 로컬 state로 포커스 유지 | 검색어 state를 로컬로 분리 |

## 4. 기술 제약사항

- **한글 IME 특성**: 조합형 입력 중 onChange 이벤트가 자음/모음 단위로 발생하므로 최소 500ms 이상 debounce 필요
- **React Query staleTime**: 현재 `staleTime: 1000 * 60`이므로 동일 쿼리 중복 호출은 캐시로 방어됨
- **호환성**: 기존 필터 동작(드롭다운, 체크박스 등)은 즉시 반영 유지해야 함 (debounce는 텍스트 검색에만 적용)
