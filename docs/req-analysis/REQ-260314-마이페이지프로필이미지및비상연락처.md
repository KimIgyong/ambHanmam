## 3. TO-BE 요구사항

### 3.1 프로필 이미지 (아바타)

| 항목 | 스펙 |
|------|------|
| 저장 방식 | PostgreSQL `bytea` (기존 서명이미지와 동일 패턴) |
| 파일 형식 | JPEG, PNG |
| 최대 크기 | 500KB |
| 표시 위치 | ProfileSection 상단 중앙 또는 좌측 |
| 업로드 방식 | 클릭 → 파일 선택 다이얼로그 → 즉시 미리보기 → 저장 버튼 |
| 없을 때 기본값 | 이니셜(이름 첫 글자) 아바타 |

### 3.2 비상연락용 전화번호

| 항목 | 스펙 |
|------|------|
| 필드명 | `phone` |
| 형식 | 자유 텍스트 (국가코드 포함 가능, 예: +84-xxx, 010-xxxx) |
| 최대 길이 | 30자 |
| 필수 여부 | 선택 입력 (nullable) |
| 표시 위치 | ProfileSection 내 기존 필드 하단 |
| 편집 방식 | 인라인 편집 또는 편집 버튼 → 폼 전환 |

### 3.3 API 요구사항

- `PATCH /users/me/profile` : 전화번호 수정
- `POST /users/me/profile-image` : 프로필 이미지 업로드
- `GET /users/me/profile-image` : 프로필 이미지 조회 (binary response)
- `DELETE /users/me/profile-image` : 프로필 이미지 삭제

## 6. 기술 제약사항

- 프로필 이미지는 기존 서명 이미지와 동일하게 `bytea` DB 저장 방식 적용
- S3 도입은 이번 범위 외 (추후 리팩토링 시 고려)
- `PATCH /users/me/profile`은 향후 이름, 직함 등 추가 필드 수정을 고려한 범용 설계
- TypeORM `synchronize: false` → 수동 SQL 마이그레이션 필요
