## 2. AS-IS 현황 분석

### 2.1 계약 금액 입력 (R1)

**현재 구현**: `ContractDetailPage.tsx` L339–346

```tsx
<input
  type="number"
  value={form.amount}
  onChange={(e) => handleChange('amount', parseFloat(e.target.value) || 0)}
  className="..."
/>
```

| 구분 | 현황 |
|------|------|
| 입력 타입 | `type="number"` — 숫자만 입력, 천 단위 구분자 없음 |
| 내부 상태 | `amount: number` (ContractForm 인터페이스) |
| DB 저장 | `ctr_amount DECIMAL(15,2)` |
| 목록 표시 | `ContractListPage`에서는 `toLocaleString()`으로 포맷 표시 |
| **문제점** | 입력 시 `1234567890`처럼 나열되어 가독성 매우 낮음 |

### 2.2 결제 조건 (R2)

**현재 구현**: `ContractDetailPage.tsx` L367–404 (Billing Options 섹션)

| 필드 | DB 컬럼 | UI | 설명 |
|------|---------|-----|------|
| 청구일 | `ctr_billing_day` INT | `type="number"` min=1 max=31 | 1~31 정수만 입력 |
| 청구 기준 | `ctr_billing_period` VARCHAR(20) | `<select>` CURRENT/PREVIOUS_MONTH | 드롭다운 |
| 자동 생성 | `ctr_auto_generate` BOOLEAN | 체크박스 | 인보이스 자동 생성 여부 |
| 자동 갱신 | `ctr_auto_renew` BOOLEAN | 체크박스 | 계약 자동 갱신 |

**현재 구조의 한계**:

1. **정기/비정기 결제 구분 없음**: 단일 billing_day/billing_period 필드만 존재
2. **비정기 결제 회차 관리 불가**: 회차별 개별 청구일/청구기준/청구금액 설정 불가
3. **청구 금액 필드 없음**: 계약 전체 금액(`ctr_amount`)만 있고, 각 청구 시 실제 청구할 금액 필드가 없음
4. **청구일 필드**: 숫자(일)만 입력 가능, 날짜(date) 형식 입력 불가

**기존 Milestone 기능과의 관계**:

현재 `MILESTONE` 타입 계약은 `amb_bil_contract_milestones` 테이블에서 회차별 관리 지원:
- `mst_seq` (순번), `mst_label` (항목명), `mst_percentage` (비율), `mst_amount` (금액), `mst_due_date` (예정일)
- 그러나 **결제 조건(billing_period, billing_day) 필드가 없으므로** "청구 기준(전월/당월)" 개념을 담을 수 없음
- Milestone은 "마일스톤 기반 분할 결제"에 초점, 요구사항의 "정기/비정기 결제 스케줄"과는 목적 다름

### 2.3 파일 업로드 (R3)

**현재 구현**: `ContractDetailPage.tsx` L533–535

```tsx
{/* Documents */}
{!isNew && id && (
  <DocumentManager refType="CONTRACT" refId={id} />
)}
```

| 구분 | 현황 |
|------|------|
| 신규 등록 시 | `isNew=true` → DocumentManager **렌더링 안 됨** |
| 수정 시 | `isNew=false && id` → DocumentManager **정상 렌더링** |
| DocumentManager | 7개 문서 유형 지원 (SIGNED_CONTRACT, APPENDIX, SOW 등) |
| 업로드 방식 | 파일 업로드 + URL 추가 지원 |
| **문제점** | 계약 등록 시 파일 업로드 불가 → 등록 완료 후 수정 페이지 진입 필요 |
| **원인** | 신규 등록 시 `refId`(계약 ID)가 아직 존재하지 않아 DocumentManager에 전달 불가 |

## 3. TO-BE 요구사항

### 3.1 R1: 계약 금액 천 단위 구분자

**목표**: 금액 입력 시 실시간으로 천 단위 구분자(,)가 자동 표시

| 구분 | TO-BE |
|------|-------|
| 입력 UX | `type="text"`로 변경, 입력 시 천단위 `,` 자동 삽입 |
| 표시 형태 | `1,234,567,890` (입력 중에도 포맷 유지) |
| 내부 상태 | `amount: number` 유지 (포맷은 UI 레이어에서만 처리) |
| DB 저장 | `DECIMAL(15,2)` 변경 없음 |
| 소수점 | 통화에 따라 소수점 2자리 허용 (예: `1,234.56`) |
| 적용 범위 | 계약 금액, 단가(unit_price), 청구 금액 등 모든 금액 필드 |

**구현 방향**: 재사용 가능한 `CurrencyInput` 컴포넌트 생성

### 3.2 R2: 결제 조건 구조 개선

**목표**: 정기/비정기 결제를 구분하고, 각각에 맞는 스케줄 입력 UI 제공

#### 3.2.1 정기 결제 (Regular Payment)

| 필드 | 설명 | 예시 |
|------|------|------|
| 결제 유형 | "정기 결제" 선택 | `REGULAR` |
| 주기 | 매월 (고정) | `MONTHLY` |
| 청구일 | 날짜 입력 (date picker) | `15` (매월 15일) |
| 청구 기준 | 전월/당월 선택 | `PREVIOUS_MONTH` / `CURRENT_MONTH` |
| 청구 금액 | 매월 청구할 금액 | `10,000,000` |

#### 3.2.2 비정기 결제 (Irregular Payment)

| 필드 | 설명 | 예시 |
|------|------|------|
| 결제 유형 | "비정기 결제" 선택 | `IRREGULAR` |
| 결제 회차 | 1, 2, 3, … 회차 | 최대 N회 |
| 청구일 | 각 회차별 날짜 (date picker) | `2026-05-15` |
| 청구 기준 | 각 회차별 전월/당월 | `CURRENT_MONTH` |
| 청구 금액 | 각 회차별 금액 | `5,000,000` |

#### DB 설계 방향

**옵션 A: 기존 테이블 확장** — `amb_bil_contracts`에 결제 유형 컬럼 추가 + 스케줄 테이블 신규 생성

```
amb_bil_contracts 추가 컬럼:
  ctr_payment_type     VARCHAR(20)   -- 'REGULAR' | 'IRREGULAR' | null(기존 호환)
  ctr_billing_amount   DECIMAL(15,2) -- 정기 결제 시 매월 청구 금액

amb_bil_payment_schedules (신규 테이블):
  pms_id               UUID          PK
  ctr_id               UUID          FK → amb_bil_contracts
  pms_seq              INT           결제 회차 (비정기)
  pms_billing_date     DATE          청구일
  pms_billing_period   VARCHAR(20)   청구 기준 (CURRENT_MONTH|PREVIOUS_MONTH)
  pms_amount           DECIMAL(15,2) 청구 금액
  pms_status           VARCHAR(20)   상태 (PENDING|INVOICED|PAID)
  pms_created_at       TIMESTAMP
  pms_updated_at       TIMESTAMP
```

### 3.3 R3: 계약 등록 시 파일 업로드

**목표**: `/billing/contracts/new` 페이지에서 계약서 파일을 첨부할 수 있도록 개선

| 구분 | TO-BE |
|------|-------|
| 신규 등록 | 임시 파일 보관 → 계약 생성 후 업로드 자동 처리 |

## 6. 기술 제약사항

### 6.1 금액 포맷 (R1)
- `type="text"`로 변경 시 모바일 키보드에서 숫자 패드 비활성 → `inputMode="numeric"` 속성 추가 필요
- 커서 위치 관리: 포맷 변경 시 커서가 끝으로 이동하는 문제 → 커서 위치 계산 로직 필요
- 소수점 통화 처리: VND(소수점 없음), USD/KRW(소수점 2자리) 등 통화별 분기

### 6.2 결제 스케줄 (R2)
- **기존 데이터 호환**: `ctr_billing_day`, `ctr_billing_period` 기존 값이 있는 계약은 `payment_type=null`로 기존 로직 유지
- **Milestone과의 관계**: `MILESTONE` 타입 계약은 기존 마일스톤 편집기 유지, 결제 스케줄과 병행 가능 여부 결정 필요
- **인보이스 자동 생성**: `auto_generate` 플래그와 결제 스케줄의 연동 — 향후 스케줄러 구현 시 활용

### 6.3 파일 업로드 (R3)
- **계약 ID 미존재**: 신규 등록 시 `refId`가 없으므로 DocumentManager 직접 사용 불가
- **해결 방안**: 계약 생성 API → 파일 업로드 API 순차 호출 (2-step)
- **실패 처리**: 계약 생성 성공 → 파일 업로드 실패 시, 계약은 생성된 상태로 유지하고 에러 토스트 표시 (재업로드는 수정 페이지에서 가능)
- **파일 크기 제한**: 기존 DocumentManager 설정 따름

### 6.4 데이터 격리
- 모든 API는 기존 `@Auth()` + `OwnEntityGuard` + `resolveEntityId()` 패턴 유지
- 신규 `amb_bil_payment_schedules` 테이블은 `amb_bil_contracts`의 `ent_id`를 통해 간접 격리 (contract FK 기반)

### 6.5 i18n
- 추가 번역 키: 결제 유형, 회차, 청구 금액 등 약 15~20개 키 추가 예상
- 지원 언어: ko / en / vi
