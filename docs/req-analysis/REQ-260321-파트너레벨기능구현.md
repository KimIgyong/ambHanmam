## 3. TO-BE 요구사항 상세

### 3.1 REQ-01: PARTNER_LEVEL 타입 시스템 추가

#### 3.1.1 레벨 구조 변경

```
 AS-IS (3 레벨)                        TO-BE (4 레벨)
 ─────────────                         ─────────────
 ADMIN_LEVEL                           ADMIN_LEVEL
 USER_LEVEL                            PARTNER_LEVEL ← 신규
 CLIENT_LEVEL                          USER_LEVEL
                                       CLIENT_LEVEL
```

#### 3.1.2 타입 추가 사항

**`packages/types/src/user-level.types.ts` 변경**:

| 상수 | 추가 항목 |
|------|----------|
| `USER_LEVEL_CODE` | `PARTNER_LEVEL: 'PARTNER_LEVEL'` |
| `USER_ROLE` | `PARTNER_ADMIN: 'PARTNER_ADMIN'`, `PARTNER_MEMBER: 'PARTNER_MEMBER'` |
| `VALID_LEVEL_ROLES` | `PARTNER_LEVEL: ['PARTNER_ADMIN', 'PARTNER_MEMBER']` |
| `JOIN_METHOD` | `PARTNER_INVITE: 'PARTNER_INVITE'` |

**역할 계층 (ROLE_HIERARCHY)** 변경:

```
Level 6: SUPER_ADMIN    ← ADMIN_LEVEL
Level 5: ADMIN           ← ADMIN_LEVEL
Level 4: MASTER          ← USER_LEVEL
Level 3: MANAGER         ← USER_LEVEL
Level 2: PARTNER_ADMIN   ← PARTNER_LEVEL (신규)
Level 2: MEMBER          ← USER_LEVEL
Level 1: PARTNER_MEMBER  ← PARTNER_LEVEL (신규)
Level 1: VIEWER          ← USER_LEVEL
Level 0: CLIENT_ADMIN    ← CLIENT_LEVEL (격리)
Level 0: CLIENT_MEMBER   ← CLIENT_LEVEL (격리)
```

> **참고**: PARTNER_ADMIN/PARTNER_MEMBER의 계층 수치(2/1)는 시스템 내부 가드 비교용이며, 실제 접근 제어는 `PARTNER_LEVEL` 레벨 분기로 처리한다. PARTNER 역할은 USER_LEVEL 역할과 동일 계층에서 비교되지 않으며, CLIENT와 마찬가지로 레벨 단위 격리를 적용한다.

#### 3.1.3 데이터 스코프 추가

**`packages/types/src/permission.types.ts` 변경**:

| 상수 | 추가 항목 |
|------|----------|
| `DATA_SCOPE` | `PARTNER: 'PARTNER'` |
| `ROLE_DATA_SCOPE` | `PARTNER_ADMIN: 'PARTNER'`, `PARTNER_MEMBER: 'PARTNER'` |

**DataScopeInterceptor 추가 분기**:
```typescript
if (level === PARTNER_LEVEL && partnerId) → scope = PARTNER
```

#### 3.1.4 JWT Payload 확장

### 3.3 REQ-03: 파트너 로그인 및 마이페이지

#### 3.3.1 기능 설명

#### 3.3.2 파트너 로그인 페이지 (`/partner/login`)

**UI 구성**:

| 요소 | 설명 |
|------|------|
| 타이틀 | "AMA Partner Portal" |
| 이메일 입력 | 파트너 계정 이메일 |
| 비밀번호 입력 | 비밀번호 |
| 로그인 버튼 | Submit |
| 회원가입 링크 | "초대를 받으셨나요? 회원가입" → `/partner/register?invitation_token=...` |
| 비밀번호 찾기 | `/partner/forgot-password` |

**로그인 로직**:
- 기존 `POST /api/v1/auth/login` API 사용 (entity_code 불필요)
- 백엔드에서 이메일로 사용자 검색 → PARTNER_LEVEL 사용자 매칭
- JWT 발급 시 `level: 'PARTNER_LEVEL'`, `partnerId: 'xxx'` 포함

**로그인 시나리오**:
```
1. 이메일 + 비밀번호 입력
2. POST /api/v1/auth/login { email, password }
3. 서버: PARTNER_LEVEL 사용자 검색
   - 매칭 → JWT 발급 (level=PARTNER_LEVEL, partnerId=xxx)
   - 미매칭 → 에러: "파트너 계정이 아닙니다"
4. 프론트: partnerAuthStore에 저장 → /partner/dashboard 리다이렉트
```

**Auth 분기 전략 (2가지 옵션)**:

| 옵션 | 설명 | 장단점 |
|------|------|--------|
| **A. 기존 AuthService 공유** | `/login`과 같은 API + 프론트에서 level 체크 후 라우팅 | 단순, 관리 용이 |
| **B. 별도 PartnerAuthService** | `/partner-auth/login` 전용 API | CLIENT와 동일 패턴, 격리 |

> **권장: 옵션 B** — CLIENT_LEVEL이 `/client-auth/login` 별도 API를 사용하는 패턴과 일치. 파트너 전용 로그인 검증 로직(파트너사 활성 상태 확인 등) 추가 용이.

### 3.4 REQ-04: 파트너 커스텀앱 개발 신청/등록

#### 3.4.1 기능 설명

파트너가 AMA 플랫폼에서 사용할 커스텀앱(플러그인)을 개발하고, 앱스토어 등록을 신청하는 기능이다.

#### 3.4.2 앱 라이프사이클

```
[DRAFT] → [SUBMITTED] → [IN_REVIEW] → [APPROVED] → [PUBLISHED]
                │                           │              │
                └── [REJECTED] ←────────────┘              │
                       │                                   │
                       └── 수정 후 재제출 ────────────────→┘
                                              [SUSPENDED] ← Admin 정지
                                              [DEPRECATED] ← 파트너/Admin 폐기
```

**상태 정의**:

| 상태 | 설명 | 주체 |
|------|------|------|
| `DRAFT` | 작성 중 (아직 제출 안 됨) | 파트너 |
| `SUBMITTED` | 심사 요청 제출 완료 | 파트너 |
| `IN_REVIEW` | Admin이 심사 중 | Admin |
| `APPROVED` | 심사 승인 (발행 가능) | Admin |
| `REJECTED` | 심사 거절 (사유 포함) | Admin |
| `PUBLISHED` | 앱스토어에 발행됨 (Entity에서 설치 가능) | Admin |
| `SUSPENDED` | Admin이 일시 정지 | Admin |
| `DEPRECATED` | 폐기 (더 이상 설치 불가) | 파트너/Admin |

| 파트너사 격리 | 파트너 사용자는 자기 파트너사의 앱만 조회/수정 가능 (DataScope: PARTNER) |
| 앱 URL 검증 | HTTPS만 허용, 내부 IP 접근 차단 (SSRF 방지) |
| JWT 격리 | 파트너 JWT는 파트너 포털 API만 접근 가능 |
| 초대 토큰 | UUID + 7일 만료 + 일회 사용 (기존 시스템 그대로) |
| Admin 심사 | 파트너 앱은 반드시 Admin 승인 후 발행 |
| API Key 암호화 | 기존 AES-256-GCM 시스템 재사용 |
