## 1. AS-IS 현황 분석

### 1.1 기존 빌링 모듈 구조

현재 AMB Management 빌링 모듈은 다음 기능을 제공한다:

| 기능 | 현황 | 비고 |
|------|------|------|
| 인보이스 CRUD | O | DRAFT → ISSUED → SENT → PAID 워크플로우 |
| 결재 워크플로우 | O | 4단계(검토→관리자→최종승인), 최종승인 시 자동 PDF+이메일 |
| PDF 생성 | O | pdfkit 기반, VN/KR 레이아웃 분리 |
| 인보이스 번호 채번 | O | VN: `No.{누적순번}`, KR: `TAX-YYYYMM-seq` |
| 이메일 발송 | O | PDF 첨부, Postal SMTP 사용 |
| Google Drive 연동 | O | 승인 시 PDF 자동 저장 |
| 계약 기반 자동 생성 | O | FIXED/USAGE_BASED 계약 크론 자동 인보이스 |
| 세금계산서(한국) | 부분 | `inv_tax_invoice_type` 필드 존재(HOMETAX/SYSTEM), 홈택스 연동 미구현 |

### 1.2 VN 법인 인보이스 현재 처리 방식

- `ent_id`가 VN01인 경우 `buildVnInvoice()` 레이아웃으로 영문 Commercial Invoice PDF 생성
- 인보이스 번호: `No.{전체 누적순번}` 형식
- **세율**: 자동 적용 없음 (수동 입력)
- **전자세금계산서 연동**: 없음 (PDF를 수동으로 TVAN 업체에 전달하는 방식으로 추정)

### 1.3 기존 InvoiceEntity 관련 필드

| 필드 | 용도 | 전자세금계산서 활용 가능 여부 |
|------|------|------|
| `inv_number` | 인보이스 번호 | X (GDT 별도 번호 체계 필요) |
| `inv_tax_rate` | 세율 | O (0/5/8/10% 매핑 가능) |
| `inv_tax_amount` | 세액 | O |
| `inv_tax_invoice_type` | HOMETAX/SYSTEM | 확장 필요 (TVAN 업체 추가) |
| `inv_status` | 발행 상태 | 확장 필요 (GDT 전송 상태 추가) |
| `inv_gsheet_url` | Google Sheet URL | 관련 없음 |

## 2. TO-BE 요구사항

### 2.1 핵심 요구사항

#### R-001: TVAN 업체 API 연동
- 공인 전자세금계산서 서비스 업체(TVAN) API를 통해 GDT(총국세청)에 전자세금계산서를 발행
- **1차 연동 대상 업체**: 미정 (EasyInvoice, MISA, Viettel S-Invoice 중 선정 필요)
- 연동 흐름: AMB 인보이스 → TVAN API → GDT 인증코드 수신 → AMB에 결과 저장

#### R-002: 전자세금계산서 발행 워크플로우
- 기존 인보이스 최종승인(`APPROVED_ADMIN`) 이후 전자세금계산서 발행 단계 추가
- 워크플로우: `APPROVED_ADMIN` → `EINV_PENDING` → `EINV_ISSUED` (또는 `EINV_FAILED`)
- 발행 실패 시 재시도 가능

#### R-003: GDT 표준 XML 생성
- Decree 123/2020 + Circular 32/2025 규격에 맞는 XML 데이터 생성
- 필수 항목: 판매자/구매자 세무코드(MST), 품목명, 수량, 단가, 세율, 세액, 총액
- 청구서 기호(Ký hiệu): `1C{YY}T{XX}` 형식 (VAT Invoice, 코드 있음, 일반)

#### R-004: 전자세금계산서 번호 관리
- TVAN 업체에서 부여하는 공식 전자세금계산서 번호 별도 저장
- 기존 `inv_number`(내부 인보이스 번호)와 분리
- GDT 인증코드(Mã CQT) 저장

#### R-005: 전자세금계산서 조회/다운로드
- TVAN에서 발행된 서명 PDF 및 XML 다운로드
- AMB UI에서 전자세금계산서 상태 조회
- GDT 포털 조회 URL 링크 제공

#### R-006: 부가세율 자동 적용
- VN 법인 인보이스 생성 시 VAT 세율 선택 UI 제공
- 지원 세율: 0%, 5%, 8%(임시감면, ~2026.12.31), 10%, 면세(KCT)
- 계약 유형/카테고리에 따른 기본 세율 추천

#### R-007: 거래처 세무정보 확장
- 거래처(PartnerEntity)에 베트남 세무 관련 필드 추가
- 세무코드(MST), 대표자명, 사업장 주소 등 전자세금계산서 필수 정보

### 2.2 비기능 요구사항

| 구분 | 요구사항 |
|------|----------|
| **보안** | TVAN API 키/인증정보는 환경변수로 관리, DB에 평문 저장 금지 |
| **가용성** | TVAN API 장애 시 인보이스 기본 기능(CRUD/결재/PDF)은 정상 동작 |
| **보관** | 발행된 전자세금계산서 XML/PDF 10년 보관 (법적 의무) |
| **감사추적** | 전자세금계산서 발행/취소/수정 이력 로깅 |
| **다국어** | 전자세금계산서 관련 UI는 ko/en/vi 3개 언어 지원 |

## 4. 사용자 플로우

### 4.1 전자세금계산서 발행 플로우

```
[1] 인보이스 생성 (DRAFT)
 ↓
[2] 항목 입력 (품목, 수량, 단가, 세율)
 ↓
[3] 결재 요청 → 검토 → 관리자승인 → 최종승인 (APPROVED_ADMIN)
 ↓  ※ 최종승인 시 기존 동작: PDF 생성 + Drive 저장 + 이메일 발송
[4] "전자세금계산서 발행" 버튼 클릭
 ↓
[5] 시스템: XML 생성 → TVAN API 전송 → GDT 인증코드 수신
 ↓
[6-성공] inv_einv_status = ISSUED, 번호/인증코드 저장
[6-실패] inv_einv_status = FAILED, 에러 메시지 저장 → "재시도" 가능
 ↓
[7] UI에서 발행 결과 확인 (번호, GDT 코드, 조회 URL)
[8] 서명된 XML/PDF 다운로드 가능
```

### 4.2 전자세금계산서 취소 플로우

```
[1] 발행된 전자세금계산서 선택
 ↓
[2] "취소" 버튼 → 취소 사유 입력
 ↓
[3] 시스템: TVAN API 취소 요청
 ↓
[4] inv_einv_status = CANCELLED
[5] 필요 시 수정된 인보이스로 재발행
```

## 5. 기술 제약사항

### 5.1 TVAN 업체 선정 필요
- 현재 어떤 TVAN 업체와 계약되어 있는지 확인 필요
- API 문서 및 테스트 계정 확보 필요
- **권장**: 추상 인터페이스(`TvanApiService`)를 설계하여 업체 교체 가능하게 구현

### 5.2 디지털 서명
- GDT 승인 CA 발급 인증서 필요 (VNPT-CA, Viettel-CA 등)
- 인증서 파일(.p12/.pfx)을 서버에 안전하게 보관
- XML 서명은 TVAN 업체 측에서 처리 (일반적)

### 5.3 네트워크 의존성
- TVAN API 호출은 외부 네트워크 의존 → 타임아웃/재시도 정책 필요
- 비동기 처리 권장 (발행 요청 → 상태 폴링 또는 콜백)

### 5.4 기존 기능 영향 최소화
- 전자세금계산서 기능은 VN 법인(`ent_country = 'VN'`)에만 적용
- KR 법인 인보이스 워크플로우에는 영향 없어야 함
- TVAN 미설정 시 기존 인보이스 기능 정상 동작

### 5.5 데이터 매핑 주의사항
- AMB 인보이스의 `inv_subtotal`/`inv_tax_amount`/`inv_total` → GDT XML의 `TgTCThue`/`TgTThue`/`TgTTTBSo`
- AMB 거래처의 `ptn_tax_id` → GDT XML의 `MST`
- AMB 인보이스 항목의 `itm_description`/`itm_quantity`/`itm_unit_price`/`itm_amount` → GDT XML의 `THHDVu`/`SLuong`/`DGia`/`ThTien`
