## 3. TO-BE 요구사항 상세

### 3.1 R1: MASTER/HR 출퇴근 수정 권한

#### 3.1.1 권한 확장
- **현재**: MASTER만 승인/반려, 멤버관리 가능. 출퇴근 수정/삭제는 본인만 가능
- **변경**: MASTER 또는 HR 역할을 가진 사용자가 **다른 사용자**의 출퇴근 기록을 수정(Amendment) 가능

#### 3.1.2 Amendment(수정 이력) 개념
- 기존 출퇴근 레코드(`amb_attendances`)는 **직접 수정하지 않음** (원본 보존)
- 별도의 **`amb_attendance_amendments`** 테이블에 수정 이력을 추가(Append)
- UI에서는 원본 하단에 수정 이력을 표시

#### 3.1.3 새 테이블: `amb_attendance_amendments`

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `aam_id` | UUID (PK) | 수정 이력 ID |
| `att_id` | UUID (FK) | 원본 출퇴근 레코드 ID |
| `aam_type` | VARCHAR(20) | 수정된 출퇴근 타입 |
| `aam_start_time` | VARCHAR(5) | 수정된 출근 시간 |
| `aam_end_time` | VARCHAR(5) | 수정된 퇴근 시간 |
| `aam_note` | TEXT | 변경 사유/메모 |
| `aam_amended_by` | UUID | 수정자 (MASTER/HR 사용자) |
| `aam_created_at` | TIMESTAMPTZ | 수정 일시 |

- 하나의 출퇴근 레코드에 **여러 개의 Amendment** 가능 (시간순 누적)
- 가장 마지막 Amendment가 **현재 유효한 상태**를 나타냄

#### 3.1.4 수정 대상 필드
- 출퇴근 타입 (`att_type` → `aam_type`)
- 출근 시간 (`att_start_time` → `aam_start_time`)
- 퇴근 시간은 타입+출근시간에 따라 자동 계산

#### 3.1.5 유효 상태 결정 로직
| 상황 | 유효 상태 |
|------|----------|
| Amendment 없음 | 원본(`amb_attendances`)의 값 |
| Amendment 1건 이상 | 가장 최근 Amendment의 값 |

#### 3.1.6 날짜 범위 제한
- MASTER/HR의 Amendment는 **과거 날짜 포함** 모든 날짜에 대해 가능
- 일반 사용자의 본인 수정은 기존 규칙(다음주 월~1개월) 유지

### 3.2 R2: 주말 근무 등록 허용

#### 3.2.1 변경 사항
- **현재**: `validateWeekday()` 함수가 토요일(6), 일요일(0) 등록 시 `E12006` 에러
- **변경**: `validateWeekday()` 제거 또는 비활성화하여 주말 등록 허용

#### 3.2.2 주말 등록 규칙
- 주말 근무도 **일반 출퇴근과 동일한 타입** 사용 (WORK, REMOTE 등)
- REMOTE 월 1회 제한은 주말 포함하여 적용
- 시간 계산 로직 동일 (start_time + 9h = end_time)
- 승인 워크플로우: 이번 주 REMOTE → PENDING (주말도 동일)

#### 3.2.3 프론트엔드 변경
- 현재: 토/일 그리드 셀이 표시는 되나 `isDateEditable()` 함수에서 편집 불가 처리될 수 있음
- 변경: 토/일 셀도 편집 가능하도록 허용
- WeekBulkFormModal: 토/일도 등록 대상에 포함

### 3.3 R3: Note 변경 이력 기록

#### 3.3.1 Amendment Note
- `amb_attendance_amendments.aam_note` 컬럼에 변경 사유 기록
- 예시: "재택근무에서 출근으로 변경 (고객 미팅)", "오전반차로 변경"
- **필수 입력**: Amendment 생성 시 Note 필수

#### 3.3.2 UI 표시
- 그리드 셀 내 또는 셀 클릭 시 상세에서 Amendment 이력 확인
- 각 Amendment마다: 변경 타입 + 변경 시간 + Note + 수정자 + 수정일시

## 5. 사용자 플로우

### 5.1 MASTER/HR가 타 사용자 출퇴근 수정 (R1, R3)

```
1. MASTER/HR 사용자가 출퇴근 페이지(/attendance) 접속
2. 팀 출퇴근 그리드에서 타인의 셀 클릭
3. [기존 등록 정보 표시] — 기존 타입, 시간, 등록일시
4. [수정 이력이 있으면] — Amendment 목록 표시 (시간순)
5. "수정(Amend)" 버튼 클릭 (MASTER/HR에게만 표시)
6. 수정 폼 표시: 
   - 새 타입 선택
   - 새 출근시간 선택 (해당 시)
   - Note 입력 (필수)
7. "저장" 클릭
8. POST /attendances/:id/amend 호출
9. 새 Amendment 생성 + 그리드 갱신
10. 셀에 최신 상태 반영 + 수정 아이콘 표시
```

### 5.2 주말 근무 등록 (R2)

```
1. 사용자가 출퇴근 페이지 접속
2. 토요일/일요일 셀 클릭
3. 출퇴근 등록 폼 표시 (기존 평일과 동일)
4. 타입, 시간 선택 후 저장
5. POST /attendances (기존 API)
6. 주말에도 등록 성공, 그리드에 표시
```

### 5.3 변경 이력 조회 (R3)

```
1. 사용자가 출퇴근 그리드에서 Amendment된 셀 확인 (시각적 표시)
2. 셀 클릭
3. 상세 패널에:
   - 원본 정보: [타입] [시간] [등록일시]
   - Amendment #1: [변경타입] [변경시간] [Note] [수정자] [수정일시]
   - Amendment #2: ...
4. 최신 Amendment가 현재 유효 상태
```

## 6. 기술 제약사항 및 고려사항

### 6.1 DB 스키마 변경
- **스테이징/프로덕션**: TypeORM `synchronize: false` → 수동 SQL 마이그레이션 필요
- 새 테이블 `amb_attendance_amendments` → `CREATE TABLE` SQL
- FK 제약: `att_id` → `amb_attendances.att_id`

### 6.2 Unique 제약 유지
- `amb_attendances`의 `(usr_id, att_date)` Unique 제약은 유지
- Amendment는 별도 테이블이므로 제약과 무관

### 6.3 HR 역할 판별
- `EntityUserRoleEntity.eurRole`에서 'HR' 값 존재 여부 확인 필요
- 현재 MASTER는 `user.role`로 JWT에 포함, HR은 `eurRole` 조회 필요할 수 있음
- EntityGuard가 이미 entityId를 세팅하므로, eurRole 조회를 추가하는 방식으로 구현

### 6.4 Amendment 적용 시 원본 보존
- `amb_attendances` 레코드는 원본 그대로 유지
- 유효 상태 조회 시: 해당 attendance의 최신 amendment가 있으면 amendment 값, 없으면 원본 값
- 조회 성능: `att_id`에 대한 amendment 조회가 빈번하므로 인덱스 필요

### 6.5 프론트엔드 영향
- `AttendanceResponse` 타입에 amendment 관련 필드 추가 → 기존 코드 호환성 유지(optional)
- 그리드 셀 렌더링: 유효 상태(amendment 반영)를 표시 + amendment 존재 여부 아이콘
- MASTER/HR 판별: 프론트에서 역할 체크 후 "수정" 버튼 표시 여부 결정

### 6.6 에러 코드 정리
- `E12006` (ATTENDANCE_WEEKEND_NOT_ALLOWED): 더 이상 사용되지 않으나 에러코드 상수 자체는 유지 (하위 호환)
- 신규 에러코드:
  - `E12007`: ATTENDANCE_AMENDMENT_PERMISSION_DENIED — Amendment 권한 없음
  - `E12008`: ATTENDANCE_AMENDMENT_NOTE_REQUIRED — Note 미입력
  - `E12009`: ATTENDANCE_ORIGINAL_NOT_FOUND — 수정 대상 원본 없음

### 7.2 사이드 임팩트
- 기존 출퇴근 등록/수정/삭제 플로우에는 영향 없음 (Amendment는 별도 경로)
- `validateWeekday()` 제거로 기존 평일 로직 변경 없음
- `AttendanceResponse` 확장 시 optional 필드로 추가하여 기존 소비자 코드에 영향 없음
- 팀 출퇴근 조회 시 Amendment JOIN이 추가되어 약간의 조회 비용 증가

## 8. 구현 우선순위

| 순서 | 단계 | 내용 | 의존성 |
|------|------|------|--------|
| 1 | DB 스키마 | `amb_attendance_amendments` 테이블 + 마이그레이션 SQL | 없음 |

| 5 | Frontend Amendment | 모달, 이력 표시, 권한 분기, i18n | 단계 3, 4 |
| 6 | 통합 테스트 | 전체 플로우 검증 | 단계 2~5 |
