## 1. 현행 엔티티 격리 아키텍처

### 격리 메커니즘 (3계층)

| 계층 | 구현 | 설명 |
|------|------|------|
| **프론트엔드** | `useEntityStore` + API 인터셉터 | `X-Entity-Id` 헤더 자동 주입 |
| **API Guard** | `EntityGuard` (`@UseGuards`) | `req.entityId` 설정 + 권한 검증 |
| **서비스/DB** | QueryBuilder WHERE 조건 | `entId = :entityId` 필터링 |

### EntityGuard 동작
1. `X-Entity-Id` 헤더 추출
2. SUPER_ADMIN/ADMIN → 바이패스 (모든 법인 접근)
3. 일반 사용자 → `EntityUserRoleEntity`에서 접근 권한 확인
4. `req.entityId`, `req.entityRole` 설정

## 3. 수정 패턴

### 패턴 A: Controller에 EntityGuard 추가
```typescript
// AS-IS: EntityGuard 없음
@Controller('hr/employees')
export class EmployeeController { ... }

// TO-BE: EntityGuard 추가
@UseGuards(EntityGuard)
@Controller('hr/employees')
export class EmployeeController { ... }
```

### 패턴 B: Service에 entityId 파라미터 추가 + WHERE 조건
```typescript
// AS-IS: entityId 파라미터 없음
async getEmployees(status?: string) {
  return this.repo.find({ where: { empStatus: status } });
}

// TO-BE: entityId 필터링 추가
async getEmployees(entityId: string, status?: string) {
  const where: any = { entId: entityId };
  if (status) where.empStatus = status;
  return this.repo.find({ where });
}
```

### 패턴 C: Controller → Service entityId 전달
```typescript
// AS-IS: entityId 미전달
@Get()
async getEmployees(@Query('status') status?: string) {
  return this.service.getEmployees(status);
}

// TO-BE: req.entityId 전달
@Get()
async getEmployees(@Query('status') status?: string, @Req() req: any) {
  return this.service.getEmployees(req.entityId, status);
}
```

## 5. 기술 제약사항

1. **AdminGuard와 EntityGuard 병행**: AdminGuard가 이미 적용된 컨트롤러에 EntityGuard 추가 시 Guard 순서 주의
2. **SUPER_ADMIN 호환**: EntityGuard는 SUPER_ADMIN/ADMIN을 자동 바이패스하므로 기존 관리자 기능 영향 없음
3. **웹메일 엔티티 분리**: MailAccountEntity에 `entId` 컬럼이 없을 경우 마이그레이션 필요 (확인 필요)
4. **프론트엔드**: 이미 `X-Entity-Id` 헤더를 자동 주입하므로 FE 변경 없음
5. **하위호환**: 기존에 단일 법인만 사용 중이면 데이터 유실 없음, 다중 법인 운영 시 데이터가 이미 혼재된 경우 데이터 정리 필요
