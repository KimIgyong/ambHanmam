## 1. 요구사항 요약 (Requirement Summary)

기존 포털 가입폼(단일 페이지 9필드)을 **2-Step 위저드**로 개선하고, 인라인 이메일 인증·국가 아이콘 그리드·비밀번호 강도 표시·AMA CTA 완료 화면을 도입하여 가입 전환율과 사용자 경험을 향상시킨다.

### 핵심 변경 사항
| # | 항목 | 설명 |
|---|------|------|
| 1 | 2-Step 구조 | Step 1(계정 정보) → Step 2(회사·약관) |
| 2 | 인라인 이메일 인증 | 4자리 코드 발송·확인 (10분 TTL, 180초 재발송 타이머) |
| 3 | 국가 아이콘 그리드 | 11개국 프리셋 + "기타 국가" 검색 오버레이 |
| 4 | 비밀번호 강도 바 | 4단계 색상 (빨강→주황→연녹→초록) |
| 5 | 약관 동의 개선 | 전체 동의 체크박스 + 필수/선택 뱃지 + 전문보기 링크 |
| 6 | 즉시 계정 생성 | 기존 이메일 링크 인증 대기 제거 → 가입 즉시 활성화 |
| 7 | AMA 비밀번호 분리 | 포털 비밀번호와 AMA 서비스 비밀번호 분리 (AMA 첫 로그인 시 설정) |
| 8 | 완료 화면 | AMA CTA + 아메바 서비스 카드 소개 |
| 9 | UI 디자인 리뉴얼 | Navy/Blue 그래디언트 헤더, 480px 카드 레이아웃, 스텝 진행 바 |

## 2. AS-IS 현황 분석 (Current State Analysis)

### 2.1 프론트엔드 — 가입 페이지

**파일**: `apps/portal-web/src/pages/auth/RegisterPage.tsx` (~350줄)

- **구조**: 단일 페이지 폼, 모든 필드를 한 화면에 나열
- **폼 필드** (9개):
  - `name` (필수), `email` (필수), `password` (필수)
  - `company_name` (필수), `country` (필수, 텍스트 입력), `phone` (선택)
  - `terms_agreed` (필수), `privacy_agreed` (필수), `marketing_agreed` (선택)
- **유효성 검사**: Zod 스키마 + React Hook Form
- **비밀번호 규칙**: 8자 이상, 대소문자+숫자+특수문자 (강도 표시 없음)
- **국가 선택**: 단순 텍스트 입력 (드롭다운 없음, 최대 10자)
- **이메일 인증**: 가입 후 별도 페이지(`VerifyEmailSentPage`)로 이동 → 이메일 링크 클릭 방식
- **약관**: 개별 체크박스, 전체 동의 없음
- **완료 후 이동**: `/verify-email-sent` 페이지 (대기 안내)
- **Google 가입**: OAuth 연동 제공
- **i18n**: `auth` 네임스페이스 (ko/en/vi)

#### 2.4.2 PortalAuthService — Register 흐름 (~480줄)
1. 이메일 중복 확인 → `ConflictException`
2. bcrypt 비밀번호 해싱 (salt rounds: 12)
3. UUID 이메일 인증 토큰 생성 (24시간 유효)
4. **Client 레코드 생성** (`amb_svc_clients` — type: COMPANY, status: PROSPECT)
5. **PortalCustomer 레코드 생성** (`amb_svc_portal_customers`)
6. **Auto Provision 트리거** → AMA API의 PortalBridge 호출
7. 실패 시 rollback (customer + client 삭제)

#### 2.4.3 VerifyEmail 흐름
- 이메일 링크의 UUID 토큰으로 검증
- 만료 확인 → `pct_email_verified = true`, client의 `cli_email_verified = true`

### 2.5 백엔드 — AMA API (PortalBridge)

**파일**: `apps/api/src/domain/portal-bridge/`

#### Auto Provision 흐름
1. Portal-API → `POST /portal-bridge/customers/:pctId/auto-provision` (internal token 인증)
2. `resolveOrCreateEntityForPortalCustomer()` — 법인(HrEntity) 생성
3. `createInternalAccount()`:
   - `amb_users` 레코드 생성 (usrJoinMethod: 'PORTAL', usrMustChangePw: true)
   - `EntityUserRole` 생성 (ENTITY_ADMIN)
   - HR Employee 자동 생성
   - `PortalUserMapping` 생성 (pctId → usrId)
4. **임시 비밀번호** 이메일 발송

### 3.2 Step 1 — 계정 정보

| 필드 | 타입 | 필수 | 유효성 | 비고 |
|------|------|------|--------|------|
| 이름 | text | ✅ | 2자 이상, 특수문자 불가 | User 아이콘 |
| 이메일 | email | ✅ | RFC 5322, 중복 체크 | Mail 아이콘, 인증까지 완료해야 진행 |
| 인증코드 | 4자리 숫자 | ✅ | 10분 TTL | 인라인 4칸 코드 박스 |
| 비밀번호 | password | ✅ | 8자+, 대소문자·숫자·특수문자 | Lock 아이콘, "아메바 포털 전용" 라벨 |

#### 이메일 인증 UX
1. 이메일 입력 → "인증코드 발송" 인라인 버튼 클릭
2. `POST /api/v1/portal/auth/send-verify-code` { email }
3. 4칸 코드 입력 UI 표시 + 180초 재발송 타이머
4. 코드 입력 → "확인" 클릭 → `POST /api/v1/portal/auth/verify-email-code` { email, code }
5. 성공: "✅ 이메일 인증 완료" 뱃지, 이메일 필드 잠금, verify_token 저장
6. 실패: "인증코드가 올바르지 않습니다" 인라인 에러

#### 비밀번호 강도
| 점수 | 색상 | 조건 |
|------|------|------|
| 1 | 빨강 | 8자 미만 또는 단일 문자 유형 |
| 2 | 주황 | 2가지 문자 유형 조합 |
| 3 | 연녹 | 3가지 문자 유형 조합 |
| 4 | 초록 | 4가지 조합 + 12자 이상 |

#### 다음 단계 진입 조건
- 이름 입력됨 (2자 이상)
- 이메일 인증 완료 (`emailVerified === true`)
- 비밀번호 강도 ≥ 2

### 3.7 AMA 비밀번호 분리 정책

- **포털 비밀번호** → `amb_svc_portal_customers.pct_password` (가입 시 설정)
- **AMA 비밀번호** → `amb_users.usr_password` (AMA 첫 로그인 시 설정)
- Phase 1 구현 (플래그 방식):
  1. 가입 시 `usr_password` = 포털 비밀번호 해시로 임시 저장

  3. AMA 로그인 시 플래그 확인 → 비밀번호 설정 페이지 리다이렉트
  4. 설정 완료 → `usr_ama_pw_set_required = false`

## 6. 기술 제약사항 (Technical Constraints)

### 6.1 호환성
- 기존 Google OAuth 플로우 유지 (`GoogleCallbackPage`, `GoogleOnboardingPage`)
- 기존 `auth.store.ts`의 `register()`, `login()`, `refreshAuth()` 인터페이스 유지 (내부 구현만 변경)
- 기존 Portal-API ↔ AMA API (PortalBridge) Auto Provision 메커니즘 유지

### 6.2 보안
- 인증코드: 4자리 숫자, 10분 TTL, 동일 이메일 5회/시간 발송 제한
- verify_token: JWT 또는 UUID, 10분 TTL, 일회용 (사용 후 무효화)
- Rate Limiting: send-verify-code 5회/시간/이메일, register 5회/분/IP
- `country_code`: ISO 3166-1 alpha-2 화이트리스트 검증

### 6.3 성능
- 이메일 발송: Postal SMTP 사용 (기존 인프라)
- 인증코드 저장: Redis(우선) 또는 DB 임시 테이블 (인프라에 따라 결정)
- 국가 목록: 클라이언트 사이드 정적 데이터 (ISO 3166-1, ~250개국)

### 6.4 DB 마이그레이션
- 스테이징/프로덕션: TypeORM synchronize 비활성화 → 수동 SQL 필수
- `amb_user_agreements`: 신규 테이블 CREATE
- `amb_users.usr_ama_pw_set_required`: ALTER TABLE ADD COLUMN

### 6.5 기존 가입자 영향
- 기존 `VerifyEmailSentPage`, `VerifyEmailPage` 페이지는 당분간 유지 (기존 이메일 링크 미처리 사용자 대응)
- 기존 `pct_email_verify_token` 컬럼 유지 (하위 호환)

### 6.6 i18n
- 새 네임스페이스 `portalSignup` 추가 → `i18n.ts`에 등록
- 기존 `auth` 네임스페이스 키 유지 (LoginPage 등에서 공유)
- 3개 언어: ko / en / vi
