## 1. AS-IS 현황 분석

### 1.1 계정 생성 이메일 현황

**발송 시점**: `/admin/portal-bridge` 페이지에서 포탈 고객을 내부 계정으로 전환할 때  
**발송 코드 위치**: `apps/api/src/domain/portal-bridge/service/portal-bridge.service.ts`  
**발송 메서드**: `MailService.sendRawEmail()` (범용 raw HTML 발송)

**현재 이메일 템플릿 (AS-IS)**:
```html
<h2>Welcome to AMB Management System</h2>
<p>Hello {userName},</p>
<p>Your account has been created based on your portal subscription.</p>
<p><strong>Login Email:</strong> {email}</p>
<p><strong>Temporary Password:</strong> {tempPassword}</p>
<p>Please change your password after first login.</p>
<p>Entity: {entityName}</p>
<p>Role: {role}</p>
```

**문제점**:
- 로그인 URL이 없어 수신자가 어느 주소로 접속해야 하는지 알 수 없음
- 템플릿이 서비스 코드에 하드코딩되어 있어 수정 시 배포 필요
- 브랜딩이 없음 (단순 텍스트 위주)

### 1.2 초대 이메일 (`sendInvitationEmail`) 현황

**발송 위치**: `apps/api/src/infrastructure/external/mail/mail.service.ts`  
**특징**:
- `FRONTEND_URL` 환경변수를 이용해 초대 링크(`/register?invitation_token=...`) 생성
- styled HTML 템플릿 (브랜드 색상, 로고 포함)
- 회원 초대 시 사용 (포탈 브릿지와 별개)

### 1.3 기존 Settings 인프라 현황

| 설정 | 테이블 | 경로 | 설명 |
|------|--------|------|------|
| SMTP | `amb_smtp_settings` | `/admin/smtp` | 이메일 발송 서버 설정 |
| 사이트 | `amb_site_settings` | `/admin/site/...` | 로고, 포탈 URL 등 |
| 메뉴 권한 | `amb_menu_permissions` | `/admin/permissions` | 메뉴 접근 권한 |

**현재 Settings 카드에 이메일 템플릿 관리 항목 없음.**

### 1.4 환경 변수 현황

| 변수 | 스테이징 | 프로덕션 | 설명 |
|------|---------|---------|------|
| `FRONTEND_URL` | `http://stg-ama.amoeba.site` | `https://ama.amoeba.site` | 프론트엔드 접속 URL |

**`FRONTEND_URL`이 docker-compose에 이미 정의되어 있으나, 포탈 브릿지 이메일에서 사용되지 않음.**

## 2. TO-BE 요구사항

### 2.1 [즉시] 계정 생성 이메일에 로그인 URL 추가

**요구사항**:
- 포탈 고객 → 내부 계정 전환 시 발송되는 이메일에 로그인 URL 추가
- 로그인 URL 포맷: `{FRONTEND_URL}/login`
- 임시 비밀번호 아래에 로그인 버튼 또는 링크 표시
- 환경별 URL 자동 적용 (환경변수 `FRONTEND_URL` 사용)

**기대 이메일 (TO-BE)**:
```
Welcome to AMB Management System

Hello {userName},
Your account has been created based on your portal subscription.

Login Email:      {email}
Temporary Password: {tempPassword}

[Login Now] → {FRONTEND_URL}/login

Please change your password after first login.
Entity: {entityName}
Role: {role}
```

### 2.2 [단기] Admin Settings - 이메일 템플릿 관리 기능

**요구사항**:
- `/admin/settings` 대시보드에 "이메일 템플릿" 설정 카드 추가
- `/admin/email-templates` 페이지 신설
- 관리자가 이메일 제목 및 본문 편집 가능
- 변수(플레이스홀더) 시스템 지원: `{{userName}}`, `{{tempPassword}}`, `{{loginUrl}}` 등
- DB에 템플릿 저장 (실시간 반영, 배포 불필요)

**관리 대상 템플릿**:

| 템플릿 코드 | 설명 | 사용처 |
|------------|------|--------|
| `ACCOUNT_CREATED` | 내부 계정 생성 안내 | 포탈 브릿지 계정 전환 |
| `INVITATION` | 회원 초대 | 회원 초대 기능 (기존) |

**지원 변수**:

| 변수 | 설명 | 사용 템플릿 |
|------|------|------------|
| `{{userName}}` | 수신자 이름 | ACCOUNT_CREATED, INVITATION |
| `{{userEmail}}` | 수신자 이메일 | ACCOUNT_CREATED |
| `{{tempPassword}}` | 임시 비밀번호 | ACCOUNT_CREATED |
| `{{loginUrl}}` | 로그인 URL | ACCOUNT_CREATED |
| `{{entityName}}` | 법인명 | ACCOUNT_CREATED |
| `{{role}}` | 사용자 역할 | ACCOUNT_CREATED |
| `{{inviterName}}` | 초대자 이름 | INVITATION |
| `{{inviteLink}}` | 초대 수락 링크 | INVITATION |
| `{{companyName}}` | 회사명 | INVITATION |

### 2.3 [옵션] 로그인 도메인 주소 관리

**요구사항** (옵션):
- `FRONTEND_URL` 환경변수를 Admin Settings에서 오버라이드 가능
- `amb_smtp_settings` 또는 `amb_site_settings` 테이블에 `login_url` 컬럼 추가 고려
- DB에 저장된 값이 있으면 환경변수보다 우선 적용

## 4. 사용자 플로우

### 4.1 Phase 1 - 계정 생성 이메일 로그인 URL 추가

```
Admin → portal-bridge 페이지
  → "계정 생성" 버튼 클릭
  → POST /portal-bridge/customers/:pctId/create-account
  → PortalBridgeService.createInternalAccount()
  → FRONTEND_URL + "/login" 조합
  → MailService.sendAccountCreatedEmail() 호출 (신규 메서드)
  → 이메일 발송 (로그인 URL 포함된 styled HTML)
```

### 4.2 Phase 2 - 이메일 템플릿 편집

```
Admin → /admin/settings → "이메일 템플릿" 카드 클릭
  → /admin/email-templates 페이지
  → 템플릿 목록 (ACCOUNT_CREATED, INVITATION)
  → 템플릿 선택 → 편집 (제목/본문)
  → 저장 → PUT /api/v1/settings/email-templates/:code
  → DB 업데이트 (amb_email_templates)
  → 이후 이메일 발송 시 DB 템플릿 사용
```

### 4.3 Phase 2 - 이메일 발송 흐름 (템플릿 적용 후)

```
계정 생성 이벤트 발생
  → EmailTemplateService.getTemplate('ACCOUNT_CREATED') 호출
  → DB에서 템플릿 로드 (없으면 기본값 사용)
  → 변수 치환: {{userName}} → 실제 이름, {{loginUrl}} → FRONTEND_URL/login 등
  → MailService.sendRawEmail() 호출 (렌더링된 HTML)
```

## 5. 기술 제약사항

### 5.1 Backend
- NestJS `ConfigService`를 통한 환경변수 접근: `FRONTEND_URL`
- TypeORM 마이그레이션 필요 (`amb_email_templates` 테이블 신규 생성)
- `PortalBridgeModule`에 `ConfigModule` 이미 사용 가능 여부 확인 필요
- `SmtpSettingsService`가 `MailService`에 주입되어 있음 → 유사 패턴으로 `EmailTemplateService` 설계

### 5.2 Frontend
- i18n: 새 번역 키 `settingsPage.emailTemplates.*` 추가 필요
- HTML 에디터 선택: 단순 `<textarea>` 또는 Monaco Editor 고려
  - 권장: `<textarea>`로 시작 (추후 WYSIWYG 전환 가능)
- 변수 힌트 UI: 클릭 시 커서 위치에 변수 삽입 기능 (UX 개선)

### 5.3 보안
- HTML 템플릿 편집 허용 시 XSS 위험 → Admin Level 사용자에게만 허용 (_이미 AdminGuard 적용됨_)
- 로그인 URL 검증: SSRF 방지 위해 허용된 도메인만 수용 (선택사항)
