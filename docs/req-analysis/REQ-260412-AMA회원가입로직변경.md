## 1. AS-IS 현황 분석

### 1.1 현재 이메일 가입 플로우 (6단계)

```
[www.amoeba.site 방문]
  ↓ 무료체험 CTA 클릭
[/register — Step 1: 계정 정보]
  ├─ 이름 입력
  ├─ 이메일 입력 → 인증코드 발송 → 4자리 코드 검증 → verify_token 발급
  └─ 비밀번호 입력 (8자+, 대/소/숫자/특수)     ← ★ 포탈 비밀번호
  ↓ Next Step
[/register — Step 2: 회사 & 약관]
  ├─ 회사명 (필수)
  ├─ 국가 선택 (필수) — CountryIconGrid 11개국 + 기타
  ├─ 전화번호 (선택)
  └─ 약관 동의 (이용약관+개인정보 필수, 마케팅 선택)
  ↓ Submit
[Backend: register()]
  ├─ amb_svc_clients 생성 (cliCode: C-YYYYMM-NNN)
  ├─ amb_svc_portal_customers 생성 (포탈 비밀번호 = 사용자 입력)
  └─ Auto-Provision 호출 →
      ├─ amb_hr_entities 생성 (entCode: VN1234)
      ├─ amb_users 생성 (AMA 비밀번호 = 임시 12자리 랜덤)  ← ★ 포탈 비밀번호와 다름!
      ├─ amb_entity_user_roles, amb_hr_employees 생성
      └─ 이메일 발송 (임시 비밀번호 포함)
  ↓
[가입 완료 화면]
  ↓ "AMA 로그인하기" 클릭
[ama.amoeba.site/{entityCode}/login]
  ├─ 이메일 + 임시 비밀번호로 로그인
  └─ usrMustChangePw: true → 비밀번호 변경 강제
  ↓
[AMA 사용 시작]
```

### 1.2 현재 문제점

| # | 문제 | 사용자 임팩트 |

| P3 | **이메일로 임시 비밀번호 전달** — 이메일 도달 지연, 스팸함 분류 시 로그인 불가 | 전환율 저하 |

## 2. TO-BE 요구사항

### 2.1 개선 가입 플로우 (4단계)

```
[www.amoeba.site 방문]
  ↓ 무료체험 CTA 클릭
[/register — 단일 스텝]
  ├─ 이메일 입력 → 인증코드 발송 → 4자리 코드 검증
  └─ (비밀번호 입력 없음, 이름 입력 없음)
  ↓ Submit
[Backend: 간소화된 register()]
  ├─ amb_svc_portal_customers 생성 (임시 비밀번호, 회사/국가 없음)
  ├─ amb_svc_clients 생성 (최소 정보)
  └─ Auto-Provision 호출 →
      ├─ amb_hr_entities 생성
      ├─ amb_users 생성 (비밀번호 = 공통 임시 or 사용자 설정)
      └─ AMA 로그인 자동 처리
  ↓
[자동 로그인으로 AMA 진입]
  ↓
[모달: 회사명 / 국가 입력 (옵션)]
  ↓ 입력 또는 건너뛰기
[AMA 사용 시작]
```

### 2.2 핵심 변경 사항

| # | 변경 | 설명 |
|---|------|------|
| C1 | **Step 2 제거** | 회사명, 국가, 전화, 약관을 가입 시 수집하지 않음 |
| C2 | **비밀번호 입력 제거** (Step1에서) | 포탈 비밀번호를 사용자가 입력하지 않음 |
| C3 | **이름 입력 제거** (Step1에서) | 이메일에서 추출 또는 AMA 내에서 설정 |
| C4 | **자동 로그인** | 가입 완료 후 AMA에 자동으로 로그인 |
| C5 | **회사/국가 → AMA 내 모달** | 옵션으로 AMA 최초 진입 시 모달로 수집 |

## 3. 장단점 분석

### 3.1 장점

| # | 장점 | 상세 |
|---|------|------|
| ✅ 1 | **가입 전환율 대폭 향상** | 입력 항목 9개 → 1개(이메일만). 이탈 지점(Step2) 제거 |
| ✅ 2 | **비밀번호 혼란 해소** | 포탈/AMA 이중 비밀번호 문제 제거. 자동 로그인으로 비밀번호 자체가 불필요 |
| ✅ 3 | **가입→사용 시간 단축** | 현재 5~10분 → 1~2분. 이메일 인증 후 즉시 AMA 사용 가능 |
| ✅ 4 | **이메일 의존도 감소** | 임시 비밀번호 이메일 수신 대기 불필요 |
| ✅ 5 | **모바일 가입 경험 개선** | 긴 폼 작성 없이 이메일만으로 시작 가능 |
| ✅ 6 | **Step2 이탈 문제 해소** | Step2 자체가 없으므로 이탈 미발생 |

### 3.2 단점 / 리스크

| # | 단점 | 심각도 | 대응방안 |
|---|------|--------|---------|
| ⚠️ 1 | **회사/국가 정보 미수집** | 중 | AMA 내 모달로 옵션 수집 + 나중에 설정에서 입력 가능 |
| ⚠️ 2 | **약관 동의 누락** | **높음** | 이메일 인증 단계에 약관 동의 체크박스 포함 필수 (법적 요구) |
| ⚠️ 3 | **Entity 생성 시 회사명/국가 없음** | 중 | 이메일 도메인에서 추정 or 기본값("My Company" / 접속 IP 기반 국가) |
| ⚠️ 4 | **자동 로그인 구현 복잡도** | 중 | Portal → AMA 크로스도메인 토큰 전달 메커니즘 필요 |
| ⚠️ 5 | **이름 없이 사용자 생성** | 낮음 | 이메일 로컬파트 사용 (예: `john@company.com` → "john") 또는 모달에서 수집 |
| ⚠️ 6 | **비밀번호 정책 변경** | 중 | AMA 직접 비밀번호 설정 플로우 필요 (최초 진입 시 or Magic Link or 비밀번호 설정 이메일) |
| ⚠️ 7 | **기존 사용자 마이그레이션** | 낮음 | 기존 가입자에게는 영향 없음 (점진적 적용) |
| ⚠️ 8 | **GA4 퍼널 재설계** | 낮음 | 기존 Step1/Step2 이벤트 → 단일 스텝 이벤트로 변경 |
| ⚠️ 9 | **Google OAuth 플로우 재설계** | 중 | Google 가입도 동일한 간소화 플로우로 통합 |

### 3.3 핵심 쟁점

#### 쟁점 1: 약관 동의를 어디서 받을 것인가?

| 방안 | 장점 | 단점 |
|------|------|------|
| A) 이메일 인증 화면에 포함 | 한 화면에서 완결 | 화면이 다소 복잡해짐 |
| B) AMA 최초 진입 시 모달 | 가입 자체는 최소화 | 법적으로 서비스 사용 전 동의 필요 |
| **C) 이메일 인증 + 약관만 (권장)** | 법적 요건 충족 + 최소 입력 | 약간의 추가 입력 |

#### 쟁점 2: 비밀번호 없이 AMA 로그인을 어떻게 처리할 것인가?

| 방안 | 장점 | 단점 |
|------|------|------|
| A) 자동 로그인 토큰 | 즉시 사용 가능 | 크로스도메인 토큰 전달 구현 필요 |
| B) Magic Link (이메일) | 보안 강화 | 이메일 수신 대기 필요 (현재와 비슷) |
| C) 가입 시 비밀번호 1회 입력 → 포탈+AMA 공유 | 간단 | 비밀번호 입력 1회 남음 (현재보다는 개선) |
| **D) 자동 로그인 + AMA 내 비밀번호 설정 (권장)** | 즉시 사용 + 보안 | 구현 복잡도 중간 |

#### 쟁점 3: Entity/User 생성 시 회사명 없으면?

| 방안 | 장점 | 단점 |
|------|------|------|
| A) 이메일 도메인 사용 | 자동 | 개인 이메일(gmail 등)이면 무의미 |
| **B) 기본값 + 모달에서 수정 (권장)** | 즉시 생성 가능 | Entity 이름 변경 로직 필요 |
| C) Entity 생성 지연 (모달 완료 후) | 정확한 정보 | 자동 로그인과 충돌 (Entity 없이 로그인 불가) |

## 7. 자동 로그인 기술 방안

### 7.1 방안 A: One-Time Token (권장)

```
[Portal-API]                      [AMA-API]
    │                                  │
    │── register() 완료 후 ──────────▶│
    │   생성: autoLoginToken (JWT)     │
    │   payload: { usrId, entCode,     │
    │             type: 'auto_login',  │
    │             exp: 5분 }           │
    │                                  │
[Portal-Web]                      [AMA-Web]
    │                                  │
    │── redirect ─────────────────────▶│
    │   URL: ama.amoeba.site/{entCode}/login?alt={token}
    │                                  │
    │                                  │── 토큰 검증 ──▶ JWT 발급
    │                                  │── 자동 로그인 완료
```

- Portal-API가 auto-provision 후 AMA-API로부터 one-time 토큰을 받음
- 프론트엔드가 이 토큰을 URL 파라미터로 AMA에 전달
- AMA 로그인 페이지에서 `alt` 파라미터 감지 → 자동 로그인 처리
- **장점**: 구현 상대적 단순, 보안 우수 (1회용, 5분 만료)
- **단점**: AMA-Web 로그인 페이지 수정 필요

### 7.2 방안 B: Portal Bridge Token

```
[Portal-API]
    │── register() 완료 후
    │   생성: portalBridgeToken (Portal-API 서명)
    │
[Portal-Web → AMA-Web]
    │── redirect: ama.amoeba.site/{entCode}/login?pbt={token}
    │
[AMA-Web → AMA-API]
    │── POST /auth/portal-bridge-login { token }
    │   AMA-API가 Portal-API에 토큰 검증 요청
    │── JWT 쌍 발급
```

- **장점**: AMA-API가 Portal-API에 직접 검증 요청 → 보안 강화
- **단점**: 서비스 간 추가 호출, 구현 복잡

## 8. 기술 제약사항

| # | 제약 | 영향 |
|---|------|------|
| 1 | **크로스도메인** — www.amoeba.site ↔ ama.amoeba.site | 쿠키 공유 불가, URL 파라미터 또는 postMessage 필요 |
| 2 | **약관 동의 법적 요건** — GDPR/개인정보보호법 | 서비스 이용 전 반드시 동의 수집 필수 |
| 3 | **Entity 생성 시 최소 정보** — entCode 생성에 국가 코드 필요 | 기본값 필요 (예: 접속 IP 기반 추정 or "XX" prefix) |
| 4 | **비밀번호 없는 계정 보안** — 추후 재로그인 시 비밀번호 필요 | AMA 내 비밀번호 설정 기능 또는 Magic Link 필요 |
| 5 | **기존 이메일 템플릿** — 임시 비밀번호 발송 로직 | 자동 로그인 시 임시 비밀번호 이메일 불필요, 웰컴 이메일로 대체 |
| 6 | **GA4 퍼널 재설계** — 기존 이벤트 호환 | 새 퍼널 이벤트 설계 필요 |

## 9. 권장 TO-BE 확정안

분석 결과, 다음 **절충안**을 권장합니다:

### 최소 가입 화면 (단일 스텝)

```
[/register — 단일 스텝]
  ├─ 이메일 입력 → 인증코드 발송 → 코드 검증
  ├─ 약관 동의 (이용약관 + 개인정보 체크)     ← 법적 필수
  └─ Submit
```

### AMA 자동 로그인 후 모달

```
[AMA 최초 진입 — 모달]
  ├─ 비밀번호 설정 (필수) — AMA 재로그인용
  ├─ 이름 (필수)
  ├─ 회사명 (선택, 기본값: 이메일 도메인 또는 "My Company")
  ├─ 국가 (선택, 기본값: 접속 IP 기반 추정)
  └─ "시작하기" 버튼
```

### 이 방안의 보안 밸런스

- **약관**: 가입 시 수집 (법적 요건 충족)
- **비밀번호**: AMA 최초 진입 시 설정 (보안 확보)
- **이름/회사/국가**: AMA 내 모달에서 수집 (옵션)
- **자동 로그인**: One-Time Token 방식 (크로스도메인 안전)
