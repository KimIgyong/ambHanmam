# REQ-슬랙연동사용자기본프로필이미지-20260401

## 1. 요구사항 요약
- 배경
  - 외부연동(슬랙) 기반 대화에서 프로필 이미지가 없는 사용자(또는 발신자)가 다수 존재하여 UI 일관성이 떨어짐
- 요구사항
  1. 신규 사용자 생성 시 기본 프로필 이미지(아이콘)를 자동 등록
  2. 기존 사용자 중 프로필 이미지가 없는 계정에 기본 이미지 일괄 등록(backfill)
  3. 외부연동-슬랙 시나리오에서도 프로필 이미지 누락으로 인한 404/빈 아바타 노이즈 최소화
- 기대효과
  - 아바타 표시 일관성 확보
  - 프로필 이미지 조회 404 감소
  - 사용자 식별성 개선

## 2. AS-IS 현황 분석
### 2.1 프로필 이미지 저장/조회 구조
- 사용자 엔티티
  - `apps/api/src/domain/auth/entity/user.entity.ts`
  - `usr_profile_image`는 `bytea`, nullable
- 조회 API
  - `GET /api/v1/users/:userId/profile-image` (`apps/api/src/domain/auth/controller/user-profile.controller.ts`)
  - 이미지가 없으면 404 반환
- 내 프로필 조회 API
  - `GET /api/v1/users/me/profile`에서 `usrProfileImage` 존재 시에만 `profileImageUrl` 제공
  - 파일: `apps/api/src/domain/auth/controller/user.controller.ts`

### 2.2 Amoeba Talk 아바타 로딩 방식
- 프론트 아바타 컴포넌트
  - `apps/web/src/domain/amoeba-talk/components/UserAvatar.tsx`
- 동작
  - `fetch(/api/v1/users/{userId}/profile-image)` 호출
  - 404면 이니셜 아바타 fallback 렌더링
- 영향
  - 기능은 fallback으로 유지되지만, 네트워크 404가 다량 발생 가능

### 2.3 슬랙 연동 메시지 저장 방식
- 슬랙 inbound 메시지는 실제 사용자 계정이 아닌 시스템 사용자 ID로 저장
  - `apps/api/src/domain/slack-integration/service/slack-event.service.ts`
  - `usrId: '00000000-0000-0000-0000-000000000000'`, `senderName: [Slack] {displayName}`
- 해당 ID가 실제 `amb_users`에 없거나 이미지가 없으면 아바타 이미지 조회 시 404 가능

### 2.4 사용자 생성 경로별 기본 이미지 미설정
아래 주요 생성 경로에서 `usrProfileImage` 기본값을 설정하지 않음:
- 일반 가입/복구: `apps/api/src/domain/auth/service/auth.service.ts`
- 초대 수락: `apps/api/src/domain/invitation/service/invitation.service.ts`
- 클라이언트 포털 가입: `apps/api/src/domain/client-portal/service/client-auth.service.ts`
- 파트너 포털 가입: `apps/api/src/domain/partner-portal/service/partner-auth.service.ts`
- 포털 브리지 생성: `apps/api/src/domain/portal-bridge/service/portal-bridge.service.ts`
- 어드민 생성: `apps/api/src/domain/admin/service/admin.service.ts`

## 3. TO-BE 요구사항
1. 모든 신규 사용자 생성 시 기본 프로필 이미지가 자동 저장되어야 한다.
2. 기존 `usr_profile_image IS NULL` 사용자에 대해 일괄 기본 이미지를 채워야 한다.
3. 슬랙 연동 대화에서 아바타 표시가 비어 보이지 않도록 해야 한다.
4. 기존 수동 업로드(내 프로필 PNG/JPEG 업로드) 동작은 유지되어야 한다.
5. 대량 백필 시 서비스 성능/잠금 영향이 최소화되어야 한다.

## 4. 갭 분석
- 데이터 갭
  - AS-IS: `usr_profile_image` nullable, 기본값 없음
  - TO-BE: 생성 시 기본 이미지 강제 + 기존 null 데이터 정리
- 연동 갭
  - AS-IS: 슬랙 메시지 sender는 시스템 사용자 ID 기반으로 일반 프로필 조회 대상과 불일치
  - TO-BE: 시스템 사용자 처리 규칙(기본 아바타 고정 또는 조회 스킵) 명확화 필요
- API 갭
  - AS-IS: 조회 API Content-Type이 사실상 JPEG 고정
  - TO-BE: SVG를 저장할 경우 MIME 처리 전략 필요

## 5. 사용자 플로우
1. 사용자가 회원가입/초대수락/포털연동으로 계정 생성
2. 서버가 기본 프로필 아이콘을 자동 생성/주입하여 `usr_profile_image`에 저장
3. Amoeba Talk 및 기타 화면에서 사용자 아바타 조회
4. 이미지가 즉시 표시되고, 누락 사용자의 fallback/404가 감소
5. 운영자는 백필 배치 실행 후 누락 사용자 0건 상태를 확인

## 6. 기술 제약사항
- 저장 포맷 제약
  - 현재 `usr_profile_image`는 바이너리만 저장하며 MIME 타입 컬럼이 없음
  - 조회 API는 `Content-Type: image/jpeg`로 응답
- 정책 충돌 가능성
  - 요구사항은 "SVG 아이콘" 명시, 그러나 현재 API 구조는 JPEG/PNG 친화적
- 권장 해석
  - 옵션 A(권장): SVG 템플릿으로 생성하되 최종 저장은 PNG/JPEG로 통일(스키마 변경 없음)
  - 옵션 B(정밀): SVG 원본 저장을 위해 MIME 컬럼 추가 + 조회 API content-type 동적 처리
- 운영 제약
  - staging/production은 `synchronize=false`이므로 스키마 변경 시 수동 SQL 필수

## 7. 영향 범위

  - 프로필 이미지 조회 컨트롤러

- Frontend

- DB
  - 옵션 A: 스키마 변경 없음
  - 옵션 B: MIME 컬럼 추가 필요

## 8. 수용 기준(AC)
- AC-1: 신규 생성 사용자 100%가 기본 프로필 이미지를 가진다.
- AC-2: 기존 `usr_profile_image IS NULL` 사용자 백필 후 누락 건수가 0건이다.
- AC-3: Amoeba Talk에서 프로필 이미지 관련 404가 유의미하게 감소한다.
- AC-4: 사용자 수동 프로필 이미지 업로드/삭제 기능은 기존처럼 동작한다.
- AC-5: 슬랙 연동 대화에서 아바타 표시 일관성이 유지된다.

## 9. 결정 필요사항
1. 저장 포맷 확정: `SVG 원본 저장` vs `SVG 기반 PNG/JPEG 저장`
2. 시스템 사용자 ID(`00000000-...`) 처리 방식 확정
   - A: DB에 시스템 사용자 레코드/기본 이미지 구성
   - B: 프론트에서 시스템 사용자 ID는 이미지 fetch 스킵 + 전용 아이콘 렌더
3. 백필 실행 방식 확정: SQL 일괄 업데이트 vs 앱 배치 스크립트
