## 1. 요구사항 요약

### 요구사항 1: 사용자 로그인 시 활동 모달
매니저/멤버/뷰어 로그인 시 개인 활동 요약 모달 표시:
- **내가 받은 호응**: 별점 건수·점수, 코멘트 이모지 건수
- **내가 반응한 호응**: 별점 부여 횟수, 코멘트 반응 횟수
- **어제 작업 내역**: 등록 이슈 건수, 처리(상태변경) 이슈 건수, 작성 태스크 수, 작성 노트 수

### 요구사항 2: 마스터 법인설정 > 업무통계 > 구성원활동
기존 구성원활동 테이블에서 **이름**과 **Unit** 사이에 3가지 지수 컬럼 추가:
- 활동지수 | 호응지수 | 종합지수

## 2. AS-IS 현황 분석

### 2.1 로그인 후 모달 현황

| 항목 | 현재 상태 |
|------|-----------|
| MASTER 로그인 모달 | `ServiceUsageModal.tsx` — AI 사용량, 직원 현황, 7일간 활동 통계 표시 |
| MANAGER/MEMBER/VIEWER 모달 | **없음** — 로그인 후 바로 대시보드 이동 |
| 로그인 플로우 | `LoginPage` → `EntitySelectPage` → 대시보드 (자동 선택 시) |

#### ServiceUsageModal (MASTER 전용) 현재 표시 내용
- 직원 현황: 전체 멤버 수, 활동 사용자 수, 활성 비율
- AI 사용 현황: 일일/월간 토큰 사용량, 쿼타 진행바
- 7일간 업무 통계: 이슈·할일·회의록·댓글 건수
- **개인 활동 데이터 없음**

### 2.2 호응 시스템 현황

#### 별점(Rating) 시스템
| 대상 | 엔티티 | 테이블 | 컬럼 prefix |
|------|--------|--------|------------|
| 이슈 | `IssueRatingEntity` | `amb_issue_ratings` | `isr_` |
| 할일(Todo) | `TodoRatingEntity` | `amb_todo_ratings` | `tdr_` |
| 댓글(통합) | `CommentRatingEntity` | `amb_comment_ratings` | `cmr_` |
| 회의록 | `MeetingNoteRatingEntity` | `amb_meeting_note_ratings` | `mnr_` |

- 별점 범위: 1~5점, 자기 콘텐츠 평가 차단
- API: `PUT /content-ratings/issues/{id}`, `PUT /content-ratings/todos/{id}`, 등

#### 이모지 반응(Reaction) 시스템
| 대상 | 엔티티 | 테이블 |
|------|--------|--------|
| 이슈 댓글 | `IssueCommentReactionEntity` | `amb_issue_comment_reactions` |
| Talk 메시지 | `TalkReactionEntity` | `amb_talk_reactions` |

- 반응 유형: LIKE(👍), CHECK(✅), PRAY(🙏), GRIN(😀), LOVE(❤️)

### 2.3 활동지수 시스템 현황

| 구성요소 | 현재 상태 |
|----------|-----------|
| `DailyActivityStatEntity` | 일일 활동 통계 집계 (`amb_daily_activity_stats`) |
| `ActivityWeightConfigEntity` | 법인별 가중치 설정 (`amb_activity_weight_configs`) |
| `ActivityStatsService` | 멤버 통계 집계 · 조회 기능 |
| `EntityActivityIndexPage` | 구성원활동 페이지 — Stats탭 + Weights탭 |

#### DailyActivityStatEntity 필드
```
das_issue_count, das_note_count, das_comment_count, das_todo_count, das_chat_count
das_activity_score, das_rating_sum, das_rating_count, das_reaction_count
das_engagement_score, das_total_score
```

#### 기존 구성원활동 테이블 컬럼 구조
```
# | 이름 | 이슈 | 노트 | 댓글 | 할일 | 채팅 | 활동점수 | 참여도점수 | 총점수
```

### 2.4 업무 통계 API 현황

| API | 용도 |
|-----|------|
| `GET /activity-stats/summary` | 멤버별 활동지수 통계 (기간별) |
| `GET /activity-stats/:userId/daily` | 사용자 일별 통계 |
| `POST /activity-stats/aggregate` | 수동 집계 트리거 |
| `GET /entity-settings/work-statistics/members` | 구성원 업무 활동 조회 |

### 2.5 사용자 Unit 정보

- `UserEntity.usrUnit` (varchar 30) — 부서명 직접 저장
- `UserUnitRoleEntity` — `amb_user_unit_roles` 테이블, Unit과 역할 매핑
- `MemberActivityStat` 인터페이스에 **unit 정보 없음** (현재 `userName`만 존재)

## 3. TO-BE 요구사항

### 3.1 요구사항 1: 사용자 로그인 활동 모달

#### 대상 사용자
- MANAGER, MEMBER, VIEWER (USER_LEVEL 사용자)
- MASTER는 기존 `ServiceUsageModal` 유지

#### 모달 표시 조건
- 로그인 성공 후 대시보드 진입 시 자동 표시
- "오늘 하루 보지 않기" 옵션 (localStorage 기반)
- ESC / X 버튼으로 닫기

#### 표시 데이터 구조

**A. 내가 받은 호응**
| 항목 | 데이터 소스 | 설명 |
|------|-------------|------|
| 별점 건수 | `amb_issue_ratings` + `amb_todo_ratings` + `amb_comment_ratings` + `amb_meeting_note_ratings` | 내 콘텐츠에 달린 별점 총 건수 |
| 별점 평균 점수 | 위 테이블들 | 내가 받은 별점의 평균 |
| 이모지 반응 건수 | `amb_issue_comment_reactions` | 내 댓글에 달린 이모지 총 건수 |

**B. 내가 반응한 호응**
| 항목 | 데이터 소스 | 설명 |
|------|-------------|------|
| 별점 부여 횟수 | 위 rating 테이블들 (usr_id = me) | 내가 다른 사람에게 준 별점 총 건수 |
| 코멘트 반응 횟수 | `amb_issue_comment_reactions` (usr_id = me) | 내가 단 이모지 반응 총 건수 |

**C. 어제 작업 내역**
| 항목 | 데이터 소스 | 설명 |
|------|-------------|------|
| 등록 이슈 건수 | `amb_issues` (reporter = me, created_at = yesterday) | 어제 내가 등록한 이슈 |
| 처리 이슈 건수 | `amb_issue_status_logs` (changed_by = me, created_at = yesterday) | 어제 상태 변경한 이슈 |
| 작성 태스크 수 | `amb_todos` (created_by = me, created_at = yesterday) | 어제 작성한 할일 |
| 작성 노트 수 | `amb_meeting_notes` (author = me, created_at = yesterday) | 어제 작성한 노트 |

### 3.2 요구사항 2: 구성원활동 지수 노출

#### 변경 대상
`EntityActivityIndexPage.tsx`의 Stats 테이블

#### 변경 내용
기존 테이블: `# | 이름 | 이슈 | 노트 | 댓글 | 할일 | 채팅 | 활동점수 | 참여도점수 | 총점수`

변경 테이블: `# | 이름 | 활동지수 | 호응지수 | 종합지수 | Unit | 이슈 | 노트 | 댓글 | 할일 | 채팅 | 활동점수 | 참여도점수 | 총점수`

- **이름** 뒤에 **활동지수**, **호응지수**, **종합지수** 3개 컬럼 추가
- **종합지수** 뒤에 **Unit(부서)** 컬럼 추가
- `MemberActivityStat`에 `unit` 필드 추가 필요 (백엔드 응답 확장)

#### 지수 정의
| 지수 | 계산 | 표시 |
|------|------|------|
| 활동지수 | `activityScore` | 기존 `das_activity_score` |
| 호응지수 | `engagementScore` | 기존 `das_engagement_score` |
| 종합지수 | `totalScore` | 기존 `das_total_score` |

> 참고: 기존 데이터에 이미 3개 지수가 계산되어 있으므로 **표시 위치만 변경**하는 것이 핵심

## 5. 사용자 플로우

### 5.1 로그인 → 활동 모달 플로우

```
사용자 로그인 (MANAGER/MEMBER/VIEWER)
  → EntitySelectPage → 법인 선택
  → 대시보드 진입
  → localStorage 'user-activity-modal-dismissed-{date}' 확인
    → 미설정: UserActivityModal 표시
      → 내가 받은 호응 (별점 건수·평균, 이모지 건수)
      → 내가 반응한 호응 (별점 부여 횟수, 이모지 반응 횟수)
      → 어제 작업 내역 (이슈 등록/처리, 태스크, 노트)
      → [X] 닫기 / [오늘 하루 보지 않기]
    → 설정됨: 모달 미표시, 대시보드 바로 사용
```

### 5.2 구성원활동 지수 노출 플로우

```
MASTER 로그인 → 법인설정 → 업무통계 → 구성원활동 탭
  → 기간 설정 (기본: 최근 30일)
  → 테이블 표시:
    # | 이름 | 활동지수 | 호응지수 | 종합지수 | Unit | 이슈 | 노트 | 댓글 | 할일 | 채팅
  → 정렬: 종합지수 내림차순 (기본)
```

## 6. 기술 제약사항

### 6.1 데이터 조회 성능
- 호응 통계는 4개 rating 테이블 + 1개 reaction 테이블 JOIN 필요
- 어제 작업 통계는 이슈, status_log, 할일, 노트 테이블 각각 COUNT 쿼리
- **최적화**: 단일 API 호출로 모든 통계를 집계하여 반환 (N+1 방지)

### 6.2 데이터 격리
- 모든 통계 쿼리에 `ent_id` 필터 적용 필수
- 개인 모달: `usr_id` + `ent_id` 범위
- 구성원활동: `ent_id` 범위 내 전체 멤버

### 6.3 기존 시스템 호환
- `DailyActivityStatEntity`의 `activityScore`, `engagementScore`, `totalScore` 재사용
- 구성원활동 테이블의 기존 데이터 구조 유지 (컬럼 추가만, 삭제 없음)

### 6.4 DB 변경
- 스키마 변경 없음 (기존 테이블/컬럼으로 충분)
- 신규 API만 추가
