## 1. AS-IS 현황 분석

### 1.1 관련 기존 모듈 현황

| 모듈 | 경로 | 관련성 |
|------|------|--------|
| 회계(Accounting) | `apps/api/src/domain/accounting/` | 거래(Transaction), 정기지출(RecurringExpense) 엔티티 보유 |
| 계약/청구(Billing) | `apps/api/src/domain/billing/` | 인보이스 다단계 승인 워크플로우 참조 가능 |
| HR | `apps/api/src/domain/hr/` | 직원·법인 엔티티(`HrEntityEntity`) 참조 |
| 멤버 | `apps/api/src/domain/members/` | 사용자 역할·조직 정보 |

### 1.2 기존 역할 계층

```
SUPER_ADMIN (6) ─┐
ADMIN       (5) ─┤ ADMIN_LEVEL (시스템 전체 관리자)
                 │
MASTER      (4) ─┐
MANAGER     (3) ─┤ USER_LEVEL
MEMBER      (2) ─┤
VIEWER      (1) ─┘
```

> **매핑**: 요구사항의 "Director" = 시스템 역할 `MASTER` (법인 최고 관리자)

### 1.3 기존 정기지출(RecurringExpense) 현황

기존 `amb_recurring_expenses` 테이블은 **회계 계좌의 정기 출금 내역**을 단순 기록하는 용도로만 사용됨.  
승인 워크플로우, 작성자/승인자 개념, 결의서 문서화 개념이 **전혀 없음**.

### 1.4 기존 승인 워크플로우 참조

Billing 모듈의 인보이스 승인은 아래 4단계로 구현되어 있음:

```
DRAFT → REVIEW → APPROVED → PAID
```

## 2. TO-BE 요구사항

### 2.1 기능 목표

회사 업무상 발생하는 지출 건에 대해 **사전 또는 사후**로 결의서를 작성하고,  
계층적 역할 기반의 **다단계 승인 워크플로우**를 통해 지출을 관리한다.

### 2.2 핵심 요구사항 목록

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| REQ-01 | 지출결의서 작성 (사전/사후 구분) | P0 |
| REQ-02 | 정기/부정기 지출 구분 및 전환 | P0 |
| REQ-03 | 정기 지출 주기 및 종료예정일 설정 | P0 |
| REQ-04 | 역할 기반 다단계 승인 (MEMBER→MANAGER→MASTER), 자가 승인 허용 | P0 |
| REQ-05 | 지출 항목 복수 등록 (품목/금액/수량), 결의서당 단일 통화 | P0 |
| REQ-06 | 첨부파일 등록 — 파일 업로드 및 링크(URL) 첨부 동시 지원 | P1 |
| REQ-07 | 결의서 목록/상세 조회 및 필터링 | P1 |
| REQ-08 | 승인 반려 사유 기록 | P1 |
| REQ-09 | 정기 지출 결의서 지정일 자동 생성 (Cron Job), 금액 수정 가능 | P1 |
| REQ-10 | 승인 요청·완료·반려 시 앱 내 알림 + 이메일 동시 발송 | P1 |
| REQ-11 | 결의서 번호 자동 생성 (EXP-YYYYMM-NNNN 형식) | P2 |
| REQ-12 | 지출 통계/요약 대시보드 | P2 |
| REQ-13 | 승인 완료 후 실제 집행 정보 입력 (집행일·방법·영수증/세금계산서) | P1 |
| REQ-14 | 월별 실제 지출 실적 통계 리포트 | P1 |
| REQ-15 | 익월 지출 예정 리포트 (정기 지출 기반 예상 금액, 추가 항목 입력 가능) | P2 |

## 3. 기능 요구사항 상세

### 3.1 지출결의서 작성

#### 3.1.1 기본 정보

| 필드 | 설명 | 필수 여부 |
|------|------|---------|
| 제목 | 결의서 제목 (예: "2026년 3월 사무용품 구입") | 필수 |
| 지출 유형 | 사전 신청 (PRE_APPROVAL) / 사후 승인 (POST_APPROVAL) | 필수 |
| 지출 구분 | 부정기 (ONE_TIME) / 정기 (RECURRING) | 필수 |
| 지출 카테고리 | 교통비, 접대비, 사무용품, 식대, 소프트웨어, 기타 등 | 필수 |
| 지출 일자 | 실제 지출 발생일 또는 예정일 | 필수 |
| 지출 내용 | 지출 목적 및 상세 내용 | 필수 |
| 신청 이유 | 지출의 업무적 필요성 설명 | 선택 |
| 결의서 번호 | 자동 생성 (EXP-YYYYMM-NNNN), 제출 시 부여 | 자동 |
| 첨부파일 | 파일 직접 업로드 또는 외부 링크(URL) 첨부, 복수 등록 가능 | 선택 |

#### 3.1.2 지출 항목 (Line Items)

결의서 1건에 복수의 지출 항목을 등록할 수 있다.

| 필드 | 설명 | 필수 여부 |
|------|------|---------|
| 항목명 | 품목 또는 서비스명 | 필수 |
| 수량 | 수량 (기본값: 1) | 필수 |
| 단가 | 단위 금액 | 필수 |
| 세금 | VAT 포함 여부 또는 세금 금액 | 선택 |
| 비고 | 항목별 메모 | 선택 |

> **통화 정책**: 결의서 1건당 단일 통화로 고정 (VND / USD / KRW 중 1개 선택). 모든 항목은 해당 통화로 입력된다.  
**총 합계** = 각 항목의 (단가 × 수량 + 세금) 합산

#### 3.1.3 정기 지출 추가 필드

지출 구분이 `RECURRING`인 경우 아래 필드 추가:

| 필드 | 설명 | 필수 여부 |
|------|------|---------|
| 반복 주기 | MONTHLY(월) / QUARTERLY(분기) / SEMI_ANNUAL(반년) / ANNUAL(1년) | 필수 |
| 시작일 | 정기 지출 시작 기준일 | 필수 |
| 종료 예정일 | 정기 지출 종료 예정일 | 필수 |
| 결제일 | 매 주기에서 지출이 발생하는 일자 (예: 매월 25일) | 선택 |

> **정책**: 부정기(`ONE_TIME`)로 등록 후 추후 정기(`RECURRING`)로 수정 가능하며, 이 경우 정기 필드(주기, 종료예정일)를 추가 입력한다.

#### 3.1.4 승인자 지정

작성자는 최소 1명의 1차 승인자를 직접 지정해야 한다.

| 작성자 역할 | 1차 승인자 | 2차 승인자 (최종) |
|-----------|----------|-----------------|
| MEMBER | MANAGER | MASTER (Director) |
| MANAGER | MASTER (Director) | — (1단계) |
| MASTER | 동급 MASTER, ADMIN, 또는 **본인 자가 승인** | — (1단계) |

**승인자 지정 규칙:**
- MEMBER는 반드시 MANAGER급 이상의 1차 승인자를 선택한다.
- MANAGER는 반드시 MASTER급 이상의 승인자를 선택한다 (1단계 승인).
- MASTER 이상 작성 시 동급 또는 상위 역할의 승인자를 선택하거나 **본인이 직접 자가 승인**할 수 있다.
- 승인자는 동일 법인(Entity) 소속이어야 한다.
- **자가 승인(자기 결재)**: MASTER 이상 역할에 한해 허용. 승인 이력에 `eap_is_self_approval = true`로 기록.

### 3.2 결의서 상태 흐름

```
          ┌─────────┐
          │  DRAFT  │ ── 작성 중 (저장만, 미제출)
          └────┬────┘
               │ 제출
               ▼
        ┌──────────────┐
        │  PENDING     │ ── 승인 대기 중
        └──────┬───────┘
               │
    ┌──────────┴──────────┐
    │ MEMBER 작성건         │ MANAGER/MASTER 작성건
    ▼                      ▼
┌──────────────┐    ┌──────────────┐
│ APPROVED_L1  │    │   APPROVED   │ ── 최종 승인
│ (MANAGER     │    └──────────────┘
│  1차 승인)   │
└──────┬───────┘
       │ MASTER 최종 승인
       ▼
┌──────────────┐
│   APPROVED   │ ── 최종 승인 완료
└──────────────┘

※ 모든 PENDING, APPROVED_L1 상태에서 반려 가능
┌──────────────┐
│   REJECTED   │ ── 반려됨 (사유 필수)
└──────────────┘

┌──────────────┐
│  CANCELLED   │ ── 작성자가 직접 취소 (DRAFT, PENDING 상태만)
└──────────────┘
```

**상태 전이 규칙:**

| 현재 상태 | 가능한 전이 | 권한 |
|---------|-----------|------|
| DRAFT | → PENDING (제출) | 작성자 본인 |
| DRAFT | → CANCELLED (취소) | 작성자 본인 |
| PENDING | → APPROVED_L1 (1차 승인) | 지정된 1차 승인자 (MANAGER) |
| PENDING | → APPROVED (직접 최종 승인) | 지정된 승인자 (MANAGER 작성건) |
| PENDING | → REJECTED (반려) | 지정된 승인자 |
| APPROVED_L1 | → APPROVED (최종 승인) | 지정된 2차 승인자 (MASTER) |
| APPROVED_L1 | → REJECTED (반려) | 지정된 2차 승인자 (MASTER) |
| PENDING | → CANCELLED (취소) | 작성자 본인 |

### 3.3 조회 및 필터

#### 3.3.1 목록 조회 권한

| 역할 | 조회 가능 범위 |
|------|-------------|
| MEMBER | 본인이 작성한 결의서 |
| MANAGER | 본인 작성 + 본인이 승인자로 지정된 결의서 |
| MASTER / ADMIN | 동일 법인 전체 결의서 |

#### 3.3.2 필터 조건

- 상태 (DRAFT / PENDING / APPROVED_L1 / APPROVED / REJECTED / CANCELLED)
- 지출 구분 (ONE_TIME / RECURRING)
- 지출 유형 (PRE_APPROVAL / POST_APPROVAL)
- 지출 카테고리
- 지출 일자 범위 (시작일 ~ 종료일)
- 작성자
- 금액 범위

---

### 3.4 수정 가능 범위

| 상태 | 수정 가능 여부 | 가능 항목 |
|------|-------------|---------|
| DRAFT | ✅ 전체 수정 가능 | 모든 필드 |
| PENDING | ⚠️ 제한적 수정 | 첨부파일, 메모만 (단, 승인자 동의 시 전체 수정 후 재제출 가능) |
| APPROVED_L1 | ⚠️ 제한적 수정 | 첨부파일 추가만 허용 |
| APPROVED / REJECTED / CANCELLED | ❌ 수정 불가 | — |

> **예외**: MANAGER가 PENDING 상태 결의서를 반려하면, 작성자는 수정 후 재제출 가능 (상태 DRAFT로 복귀)

### 3.5 지출 집행 정보 입력 (REQ-13)

결의서가 **최종 승인(APPROVED)** 된 후, 실제 지출이 집행되면 담당자(작성자 또는 MANAGER 이상)가 집행 정보를 입력한다.

#### 3.5.1 집행 정보 필드

| 필드 | 설명 | 필수 여부 |
|------|------|---------|
| 집행 일자 | 실제 지출이 발생한 날짜 | 필수 |
| 집행 방법 | CARD(카드) / CASH(현금) / TRANSFER(계좌이체) / OTHER(기타) | 필수 |
| 집행 방법 메모 | 집행 방법이 OTHER인 경우 직접 입력 | 조건부 필수 |
| 집행 금액 | 실제 집행된 금액 (결의서 승인 금액과 다를 수 있음) | 필수 |
| 영수증/세금계산서 타입 | RECEIPT(영수증) / TAX_INVOICE(세금계산서) / NONE(없음) | 필수 |
| 증빙 번호 | 세금계산서 번호 또는 영수증 번호 | 선택 |
| 증빙 파일 | 영수증 사진, 세금계산서 파일 (파일 업로드 또는 링크) | 선택 |
| 메모 | 집행 관련 추가 메모 | 선택 |

#### 3.5.2 집행 상태 정의

| 상태 | 설명 |
|------|------|
| 미집행 (APPROVED) | 최종 승인되었으나 집행 정보 미입력 |
| 집행완료 (EXECUTED) | 집행 정보 입력 완료 |

> **결의서 상태 추가**: `APPROVED` → `EXECUTED` 전이 추가  
> 집행 정보는 수정 가능 (집행완료 후에도 수정 허용, 이력 기록)

#### 3.5.3 집행 방법 Enum

```
CARD       카드 결제 (법인카드, 개인카드)
CASH       현금 지출
TRANSFER   계좌이체
OTHER      기타 (텍스트 직접 입력)
```

#### 3.5.4 증빙 유형 Enum

```
RECEIPT      영수증 (사진 첨부 가능)
TAX_INVOICE  세금계산서 (파일 첨부 + 번호 입력)
NONE         증빙 없음
```

### 3.6 월별 실제 지출 실적 리포트 (REQ-14)

#### 3.6.1 리포트 개요

집행 완료된 지출결의서를 기준으로 **매월 실제 지출 실적**을 집계하여 리포트를 제공한다.

#### 3.6.2 리포트 조회 조건

| 조건 | 설명 |
|------|------|
| 기준 연월 | 집행 일자 기준 (예: 2026년 3월) |
| 법인 | 로그인 사용자의 소속 법인 |
| 통화 | 통화별 분리 집계 |
| 카테고리 | 카테고리별 소계 |

#### 3.6.3 리포트 구성 항목

**요약 섹션:**
- 해당 월 총 집행 금액 (통화별)
- 전월 대비 증감 금액 및 비율
- 집행 건수
- 미집행 건수 (APPROVED 상태 잔존)

**카테고리별 집계:**

| 카테고리 | 건수 | 결의 금액 | 집행 금액 | 차이 |
|---------|------|---------|---------|------|
| 교통비 | 3 | 1,500,000 | 1,450,000 | -50,000 |
| 사무용품 | 2 | 800,000 | 800,000 | 0 |
| ... | | | | |
| **합계** | **5** | **2,300,000** | **2,250,000** | **-50,000** |

**집행 방법별 집계:**
- 카드 / 현금 / 계좌이체 / 기타 별 금액 및 비율

**상세 내역 목록:**
- 결의서 번호, 제목, 카테고리, 집행일, 결의금액, 집행금액, 집행방법, 증빙 타입

#### 3.6.4 내보내기

- Excel 파일 다운로드 (`.xlsx`)
- 인쇄용 보기 (Print-friendly)

### 3.7 익월 지출 예정 리포트 (REQ-15)

#### 3.7.1 리포트 개요

등록된 **정기 지출 결의서(RECURRING, APPROVED 상태)**와 전월 실적을 기반으로 **익월 예상 지출 리포트**를 자동 생성하고, 비정기 예상 지출 항목을 추가 입력받는다.

#### 3.7.2 자동 계산 로직

| 항목 | 계산 방법 |
|------|----------|
| 정기 지출 예상 금액 | 해당 월에 `exr_next_due_date`가 도래하는 정기 결의서의 `exr_total_amount` 합산 |
| 전월 실적 금액 | 전월 집행완료(EXECUTED) 결의서 기준 카테고리별 합산 |
| 예상 참고값 | 정기 지출은 직전 집행 금액 표시 (수정 가능) |

#### 3.7.3 리포트 구성

**헤더:**
- 대상 연월 (예: 2026년 4월 지출 예정)
- 자동 생성 기준: `정기 지출 + 추가 입력 항목`
- 작성자, 최종 수정일

**정기 지출 예상 항목 (자동 구성, 금액 수정 가능):**

| 결의서번호 | 제목 | 카테고리 | 주기 | 전월 집행금액 | 예상 금액 (수정) | 유의사항 |
|---------|------|---------|------|------------|--------------|--------|
| EXP-202601-0005 | AWS 서버비 | 소프트웨어 | 월별 | USD 500 | [USD 500] | — |
| EXP-202601-0008 | 사무실 임차료 | 임차료 | 월별 | VND 15,000,000 | [VND 15,000,000] | — |

**추가 예정 항목 (직접 입력):**
- 제목, 카테고리, 예상 금액, 통화, 메모
- 여러 건 추가 가능

**요약:**

| 통화 | 정기 예상 합계 | 추가 항목 합계 | 총 예상 합계 |
|------|-------------|-------------|------------|
| VND | 20,000,000 | 5,000,000 | 25,000,000 |
| USD | 500 | 0 | 500 |

#### 3.7.4 저장 및 공유

- 리포트 초안 저장 (`amb_expense_forecast_reports` 테이블)
- 저장된 리포트는 이후 조회 및 수정 가능
- Excel / PDF 내보내기
- MANAGER 이상 공유 가능

## 4. 비기능 요구사항

| 항목 | 요구사항 |
|------|---------|
| 접근 제어 | 법인(Entity) 격리 — 타 법인 결의서 접근 불가 |
| 다국어 | 한국어(ko) / 영어(en) / 베트남어(vi) 지원 |
| 감사 로그 | 상태 변경 이력(누가, 언제, 어떤 액션) 추적, 자가 승인 여부 표시, 집행 정보 수정 이력 |
| 알림 | 승인 요청·완료·반려 시 **앱 내 Notification + 이메일(SMTP) 동시 발송** |
| 통화 | 결의서 1건당 단일 통화 (VND / USD / KRW), 항목별 개별 통화 불가 |
| 페이지네이션 | 기본 20건, 최대 100건 |
| 소프트 삭제 | `exr_deleted_at` 컬럼으로 Soft Delete |
| 자동 생성 | 정기 지출 결의서는 `exr_next_due_date` 기준 Cron Job으로 자동 생성, 금액 수정 가능 |
| 결의서 번호 | 제출 시 `EXP-YYYYMM-NNNN` 형식으로 자동 부여 (법인·월별 순번) |
| 리포트 집계 기준 | 월별 실적 리포트는 **집행 일자(exd_executed_at)** 기준 집계 |
| 예정 리포트 저장 | 익월 지출 예정 리포트는 수정 후 저장 가능, 이전 리포트 조회 가능 |

### 6.2 amb_expense_requests

결의서 상태에 `EXECUTED` 추가:
```
DRAFT → PENDING → APPROVED_L1 → APPROVED → EXECUTED
```

```sql
CREATE TABLE amb_expense_requests (
  exr_id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  ent_id           UUID NOT NULL,                    -- 법인 ID
  exr_requester_id UUID NOT NULL,                    -- 작성자 (usr_id)

  -- 기본 정보
  exr_title        VARCHAR(200) NOT NULL,            -- 제목
  exr_type         VARCHAR(20) NOT NULL,             -- PRE_APPROVAL | POST_APPROVAL
  exr_frequency    VARCHAR(20) NOT NULL DEFAULT 'ONE_TIME', -- ONE_TIME | RECURRING
  exr_category     VARCHAR(50) NOT NULL,             -- 지출 카테고리
  exr_expense_date DATE NOT NULL,                    -- 지출 일자
  exr_description  TEXT,                             -- 지출 내용
  exr_reason       TEXT,                             -- 신청 이유

  -- 결의서 번호 (제출 시 자동 부여: EXP-YYYYMM-NNNN)
  exr_number       VARCHAR(30) UNIQUE,               -- 예: EXP-202603-0001

  -- 금액 요약 (item 합산 캐싱)
  exr_total_amount DECIMAL(18,2) NOT NULL DEFAULT 0,
  exr_currency     VARCHAR(10) NOT NULL DEFAULT 'VND', -- 결의서 단위 단일 통화

  -- 정기 지출 필드 (RECURRING인 경우)
  exr_period       VARCHAR(20),                      -- MONTHLY | QUARTERLY | SEMI_ANNUAL | ANNUAL
  exr_start_date   DATE,                             -- 정기 시작일
  exr_end_date     DATE,                             -- 정기 종료 예정일
  exr_payment_day  INTEGER,                          -- 매 주기 결제일 (예: 25)
  exr_next_due_date DATE,                            -- 다음 실행 예정일 (자동 계산)

  -- 상태
  exr_status       VARCHAR(30) NOT NULL DEFAULT 'DRAFT',
                   -- DRAFT | PENDING | APPROVED_L1 | APPROVED | EXECUTED | REJECTED | CANCELLED

  -- 정기 지출 자동 생성 추적
  exr_parent_id    UUID REFERENCES amb_expense_requests(exr_id), -- 원본 정기 결의서 ID (자동 생성본)
  exr_is_auto_generated BOOLEAN NOT NULL DEFAULT FALSE,          -- Cron 자동 생성 여부

  -- 승인자 (지정된 단계별 승인자, NULL이면 자가 승인)
  exr_approver1_id UUID,                             -- 1차 승인자 (MANAGER급)
  exr_approver2_id UUID,                             -- 2차 승인자 (MASTER급, MEMBER 작성건만)

  -- 메타
  exr_created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  exr_updated_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  exr_deleted_at   TIMESTAMPTZ
);
```

## 7. 사용자 플로우

### 7.1 MEMBER의 지출결의서 제출 → 최종 승인 흐름

```
[MEMBER]
  1. 지출결의서 작성 (DRAFT)
     - 지출 유형 선택 (사전/사후)
     - 지출 구분 선택 (부정기/정기)
     - 지출 항목 입력 (복수 가능)
     - 1차 승인자(MANAGER) 지정
     - 2차 승인자(MASTER/Director) 지정
  2. 제출 → 상태: PENDING

[MANAGER (1차 승인자)]
  3. 승인 요청 알림 수신
  4. 결의서 검토
  5-a. 승인 → 상태: APPROVED_L1
  5-b. 반려 → 상태: REJECTED (사유 입력 필수)
       └─ MEMBER는 수정 후 재제출 가능

[MASTER/Director (2차 승인자)]
  6. 최종 승인 요청 알림 수신 (APPROVED_L1에서)
  7. 결의서 최종 검토
  8-a. 최종 승인 → 상태: APPROVED
  8-b. 반려 → 상태: REJECTED (사유 입력 필수)
```

### 7.2 MANAGER의 지출결의서 제출 → 승인 흐름

```
[MANAGER]
  1. 지출결의서 작성 (DRAFT)
     - 승인자(MASTER/Director) 지정
  2. 제출 → 상태: PENDING

[MASTER/Director (단독 승인자)]
  3. 승인 요청 알림 수신
  4. 결의서 검토
  5-a. 승인 → 상태: APPROVED (1단계로 최종 승인)
  5-b. 반려 → 상태: REJECTED
```

### 7.3 부정기 → 정기 전환 플로우

```
[MEMBER/MANAGER]
  1. 기존 ONE_TIME 결의서 조회
  2. 수정 진입 (DRAFT 혹은 REJECTED 상태만 가능)
  3. 지출 구분을 ONE_TIME → RECURRING으로 변경
  4. 추가 필드 입력:
     - 반복 주기 (MONTHLY/QUARTERLY/SEMI_ANNUAL/ANNUAL)
     - 시작일
     - 종료 예정일
  5. 저장 및 재제출
```

### 7.4 정기 지출 자동 생성 플로우 (Cron Job)

```
[Cron Job — 매일 자정 실행]
  1. exr_status = APPROVED AND exr_frequency = RECURRING 인 결의서 조회
  2. exr_next_due_date = 오늘인 건 필터링
  3. 각 건에 대해 새 결의서 자동 생성:
     - exr_parent_id = 원본 결의서 ID
     - exr_is_auto_generated = true
     - exr_status = DRAFT (작성자가 검토·금액 수정 후 제출)
     - exr_total_amount = 원본 금액 (수정 가능)
     - 항목(items) 동일하게 복사
     - 승인자 동일하게 복사
  4. 다음 실행일 계산 후 원본의 exr_next_due_date 업데이트
  5. 작성자에게 "정기 지출 결의서가 생성되었습니다" 알림 + 이메일 발송
  6. exr_end_date 초과 시 더 이상 자동 생성 안 함
```

> **자동 생성 정책**: 자동 생성된 결의서는 DRAFT 상태로 생성되며, 작성자가 금액 등을 수정한 뒤 직접 제출해야 승인 흐름이 시작된다.

### 7.5 집행 정보 입력 플로우

```
[작성자 또는 MANAGER 이상]
  1. 결의서 상세 화면 접근 (exr_status = APPROVED)
  2. [집행 정보 입력] 버튼 클릭
  3. 집행 정보 입력:
     - 집행 일자 (필수)
     - 집행 방법 선택: 카드 / 현금 / 계좌이체 / 기타(텍스트 입력)
     - 실제 집행 금액 (필수, 결의 금액 자동 채워짐, 수정 가능)
     - 증빙 타입 선택: 영수증 / 세금계산서 / 없음
     - 증빙 번호 입력 (선택)
     - 증빙 파일 업로드 또는 링크 첨부 (선택)
     - 메모 (선택)
  4. 저장 → 결의서 상태: EXECUTED
  5. 집행 완료 후에도 정보 수정 가능
```

### 7.6 월별 실적 리포트 조회 플로우

```
[MANAGER 이상]
  1. 지출결의서 메뉴 → [월별 실적 리포트] 탭
  2. 조회 연월 선택 (기본값: 전월)
  3. 집행완료(EXECUTED) 결의서 기준 자동 집계:
     - 카테고리별 집행 금액 / 건수
     - 집행 방법별 집계
     - 결의 금액 vs 집행 금액 비교
  4. 상세 내역 목록 확인
  5. Excel 다운로드 또는 인쇄
```

### 7.7 익월 지출 예정 리포트 작성 플로우

```
[MANAGER 이상]
  1. 지출결의서 메뉴 → [지출 예정 리포트] 탭
  2. 대상 연월 선택 (기본값: 익월)
  3. 시스템 자동 구성:
     - 정기 결의서 중 해당 월에 exr_next_due_date가 도래하는 건 자동 목록화
     - 각 항목에 전월 집행 금액 표시 (참고값)
  4. 각 항목의 예상 금액 수정 가능
  5. 추가 예정 항목 직접 입력 (제목, 카테고리, 금액, 통화)
  6. 저장 → amb_expense_forecast_reports + amb_expense_forecast_items 저장
  7. Excel / PDF 내보내기
  8. 저장된 리포트는 이후 수정 및 재조회 가능
```

## 9. 기술 제약사항

### 9.1 백엔드

- **NestJS 모듈**: `apps/api/src/domain/expense-request/` 신규 생성
- **DB**: PostgreSQL — TypeORM 엔티티 **7개** 신규 추가 (기존 4 + 집행·예정리포트 3)
- **파일 컬럼 prefix**: `exr_`, `eri_`, `eap_`, `eat_`, `exd_`, `efr_`, `efi_`
- **App Module**: `AppModule`에 `ExpenseRequestModule` 등록 필요
- **메뉴 코드**: `MENU_CODES`에 `EXPENSE_REQUEST` 추가
- **ACL**: 기존 ACL 미들웨어에 `EXPENSE_REQUEST` 메뉴 권한 추가
- **법인 격리**: 모든 조회 쿼리에 `ent_id` 필터 필수
- **Cron Job**: `@nestjs/schedule` 활용, 매일 자정 실행 — 정기 지출 결의서 자동 생성
- **결의서 번호 채번**: `EXP-{YYYYMM}-{4자리 순번}` — 법인+월 기준 시퀀스 관리
- **이메일 발송**: 기존 `SmtpModule` / `EmailTemplateModule` 연계, 지출결의서 전용 템플릿 4종 (승인 요청, 승인 완료, 반려, 정기 자동 생성)
- **파일 업로드**: `Multer` 처리, 서버 로컬 또는 기존 파일 스토리지 정책 따름
- **리포트 집계**: TypeORM QueryBuilder로 집행 일자 기준 월별 GROUP BY 집계
- **Excel 내보내기**: `exceljs` 라이브러리 활용 (기존 회계 모듈 패턴 참조)

### 9.2 프론트엔드

- **도메인 경로**: `apps/web/src/domain/expense-request/`
- **라우팅**: 
  - `/expense-request` — 목록
  - `/expense-request/new` — 작성
  - `/expense-request/:id` — 상세
  - `/expense-request/reports/monthly` — 월별 실적 리포트
  - `/expense-request/reports/forecast` — 익월 예정 리포트
- **i18n**: `locales/ko/expenseRequest.json`, `locales/en/expenseRequest.json`, `locales/vi/expenseRequest.json` 신규 추가
- **스토어**: Zustand 기반 `useExpenseRequestStore`
- **API 클라이언트**: `expenseRequest.service.ts` 신규 추가

### 9.3 공유 타입

- **packages/types/**: `ExpenseRequestType`, `ExpenseItemType`, `ExpenseApprovalType`, `ExpenseExecutionType`, `ExpenseForecastReportType` 추가
