# REQ-스토리지추가구매모델-20260405

## 1. Requirements Summary / 요구사항 요약

### 1.1 Business Goal / 비즈니스 목표
- EN: Extend the current AI-token-centered monetization model with paid storage add-ons.
- KR: 현재 AI 토큰 중심 과금 모델에 스토리지 추가 구매 모델을 결합한다.

### 1.2 Core Requested Outcomes / 핵심 요청 결과
- EN: Introduce purchasable storage capacity (HDD storage) per entity.
- KR: Entity 단위로 추가 스토리지 용량(유료)을 구매할 수 있어야 한다.
- EN: Separate storage space by entity folder/bucket scope.
- KR: 저장 공간을 Entity별 폴더(또는 버킷 prefix)로 분리해야 한다.
- EN: Trigger over-usage alerts and provide seamless upsell path.
- KR: 사용량 임계치 도달/초과 시 알림을 제공하고, 즉시 추가 구매로 연결해야 한다.

### 1.3 Scope in This Document / 본 문서 범위
- EN: Requirement analysis only (AS-IS, TO-BE, GAP, user flow, constraints).
- KR: 본 문서는 요구사항 분석 단계이며 구현은 포함하지 않는다.

## 3. TO-BE Requirements / TO-BE 요구사항

### 3.1 Functional Requirements / 기능 요구사항

1. Storage Product Catalog
   - EN: Define storage add-on products (for example 10GB, 50GB, 100GB).
   - KR: 스토리지 추가 구매 상품(예: 10GB, 50GB, 100GB)을 관리한다.

2. Entity-level Storage Quota
   - EN: Each entity has base included storage + purchased storage.
   - KR: 각 Entity는 기본 제공량 + 추가 구매량으로 총 스토리지 한도를 가진다.

3. Entity-level Physical Segmentation
   - EN: Separate storage physically/logically by entity.
   - KR: Entity별 물리/논리 분리 저장 경로를 강제한다.
   - Local example: `uploads/entities/{ent_id}/...`
   - Drive example: entity billing root folder under own drive settings

4. Storage Usage Metering
   - EN: Track used bytes per entity and aggregate daily/monthly usage.
   - KR: Entity별 사용 바이트를 수집하고 일/월 집계를 제공한다.

5. Threshold Alerting
   - EN: Alert at configurable thresholds (recommended: 80%, 90%, 100%).
   - KR: 임계치(권장: 80%, 90%, 100%) 도달 시 단계별 알림을 발송한다.

6. Overage Action Policy
   - EN: Support both SOFT limit (warning only) and HARD limit (block uploads).
   - KR: SOFT 제한(경고)과 HARD 제한(업로드 차단) 정책을 지원한다.

7. Upsell Purchase Flow
   - EN: From warning state, user can immediately purchase additional storage.
   - KR: 경고 상태에서 추가 스토리지 상품 구매로 즉시 전환할 수 있어야 한다.

8. Admin + Entity Visibility
   - EN: Admin can monitor all entities; entity admin can monitor own usage/quota/history.
   - KR: 관리자 전체 모니터링, 법인 관리자는 자기 법인 사용량/한도/구매이력 조회가 가능해야 한다.

### 3.2 Data and Domain Requirements / 데이터·도메인 요구사항

1. New domain entities (proposed)
   - `amb_storage_quota_products`
   - `amb_storage_quota_topups`
   - `amb_entity_storage_configs`
   - `amb_storage_usage_daily` (entity/day summary)
   - `amb_storage_usage_events` (optional raw delta log)

2. Required key fields (proposed)
   - quota limit bytes, used bytes, warning thresholds, action_on_exceed
   - product size bytes, product price/currency, active flag
   - purchase status and refund rollback relation with PG transaction

3. Existing integration pattern reuse
   - EN: Reuse payment event-driven top-up mechanism used by AI quota.
   - KR: AI 쿼터에서 이미 사용하는 결제 이벤트 기반 충전 메커니즘을 재사용한다.

### 3.3 API and UX Requirements / API 및 UX 요구사항

1. Entity settings page
   - EN: Add storage usage card, threshold warnings, and purchase CTA.
   - KR: 스토리지 사용량 카드, 임계치 경고, 추가 구매 CTA를 제공한다.

2. Admin page
   - EN: Add storage product management and entity usage dashboard.
   - KR: 스토리지 상품 관리 및 법인별 사용량 대시보드를 제공한다.

3. API conventions
   - EN: Keep request snake_case / response camelCase with standard envelope.
   - KR: 기존 API 규칙(request snake_case, response camelCase, 표준 응답)을 유지한다.

## 5. User Flows / 사용자 플로우

### 5.1 Entity Admin Flow / 법인 관리자 플로우
1. 법인 관리자가 스토리지 사용량 페이지를 연다.
2. 현재 사용량/한도/잔여량을 확인한다.
3. 임계치(예: 80%, 90%, 100%) 도달 시 경고를 확인한다.
4. 추가 구매 버튼을 눌러 스토리지 상품을 선택한다.
5. 결제 완료 후 용량이 즉시(또는 짧은 지연 후) 증가한다.

### 5.2 Upload Enforcement Flow / 업로드 제한 플로우
1. 사용자가 파일 업로드를 시도한다.
2. 서버가 엔터티 기준 사용량+예상증가량을 검사한다.
3. SOFT 정책이면 업로드 허용 + 경고 기록/노출.
4. HARD 정책이면 업로드 차단 + 구매 유도 메시지 반환.

### 5.3 Admin Oversight Flow / 시스템 관리자 플로우
1. 관리자가 전체 법인 스토리지 대시보드를 조회한다.
2. 임계치 초과/근접 법인을 정렬/필터링해 확인한다.
3. 상품 활성화/가격/정렬을 조정한다.
4. 환불 발생 시 용량 롤백 이력까지 추적한다.

## 6. Technical Constraints / 기술 제약사항

1. Multi-tenant isolation is mandatory.
   - KR: USER_LEVEL 범위에서 ent_id 기반 데이터 격리를 반드시 유지해야 한다.

2. Staging/Production DB migration is manual.
   - KR: 스테이징/프로덕션은 TypeORM synchronize 비활성화 환경이므로 수동 SQL 마이그레이션이 필수다.

3. Accuracy vs performance tradeoff.
   - KR: 업로드 이벤트 즉시 누적 + 주기적 재조정(배치) 혼합 전략이 필요하다.

4. Mixed storage backends.
   - KR: 로컬 파일과 Google Drive가 공존하므로 단일 기준 사용량 계산 규칙(바이트 기준)을 먼저 확정해야 한다.

5. Refund rollback consistency.
   - KR: 결제 취소/환불 시 구매 용량을 되돌릴 때 현재 사용량이 이미 높은 경우 정책(즉시 차단/유예기간)이 필요하다.

6. i18n and UX consistency.
   - KR: 경고/차단/구매 유도 문구는 ko/en/vi 번역 키 기반으로 제공해야 하며 하드코딩을 금지한다.

## 7. Recommended Design Direction / 권장 설계 방향

### 7.1 Architecture Option Comparison / 아키텍처 옵션 비교

1. Option A: Local-first metering
   - 장점: HDD 과금 모델과 직접적
   - 단점: Drive 파일까지 포함한 총량 과금에 한계

2. Option B: Drive-first metering
   - 장점: 엔터티 폴더 구조와 정합성 높음
   - 단점: 로컬 첨부를 별도 처리해야 함

3. Option C: Hybrid unified ledger (Recommended)
   - 장점: 로컬+Drive 혼합 환경에서 총량 과금 가능
   - 단점: 집계 구현 복잡도 증가

### 7.2 Recommendation
- EN: Adopt Hybrid unified ledger and keep storage usage as a billing source of truth.
- KR: 하이브리드 통합 원장 방식으로 스토리지 사용량을 과금 기준 단일 진실원으로 운영하는 것을 권장한다.

## 9. Open Questions / 의사결정 필요사항

1. 기본 제공 스토리지 정책
   - 각 Entity의 기본 제공량을 동일하게 줄지, 플랜별 차등 적용할지 결정 필요

2. 과금 방식
   - 선불 패키지(Top-up)만 사용할지, 사용량 종량(후불) 혼합할지 결정 필요

3. 초과 시 처리
   - HARD 차단 시 업로드 전면 차단인지, 특정 모듈만 차단인지 정책 정의 필요

4. 환불/다운그레이드 정책
   - 환불 후 한도가 사용량보다 낮아질 때 유예 기간/정리 정책 필요

5. 저장소 범위
   - Drive 파일까지 과금 포함할지, 로컬 HDD만 과금할지 1차 범위 결정 필요
