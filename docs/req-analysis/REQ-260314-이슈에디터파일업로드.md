| 항목 | 현재 상태 |
|------|----------|
| **위치** | `apps/web/src/domain/meeting-notes/components/RichTextEditor.tsx` |

### 1.2 현재 툴바 구성

```
[ Bold ] [ Italic ] [ Underline ] | [ H2 ] | [ 이미지 ▼ ] [ 링크 ] | [ Undo ] [ Redo ] | [ Source ]
                                              ├── 파일 업로드
                                              └── URL 입력
```

- 이미지 버튼 클릭 시 드롭다운: "파일 업로드" / "URL 입력"
- input `accept="image/*"` 로 이미지만 허용
- 드래그 앤 드롭, 클립보드 붙여넣기 모두 이미지만 처리

### 1.3 이미지 업로드 흐름

```
사용자 → 이미지 선택/드롭/붙여넣기
  ├─ Google Drive 연동 시 → driveApiService.uploadFile() → /api/v1/drive/files/{fileId}/view
  └─ 미연동 시 (Fallback) → POST /api/v1/files/upload → /api/v1/files/{storedName}
      → editor에 <img src="..." /> 삽입
```

### 1.4 백엔드 파일 서비스

| 항목 | 현재 상태 |
|------|----------|
| **위치** | `apps/api/src/infrastructure/file/` |
| **업로드 API** | `POST /api/v1/files/upload` |
| **파일 제공 API** | `GET /api/v1/files/:filename` (Public) |
| **저장소** | 로컬 파일시스템 (`process.cwd()/uploads`) |
| **최대 크기** | 10MB |
| **허용 MIME** | 이미지 (jpeg, png, gif, webp), PDF, Word, Excel, PowerPoint |

**중요**: 백엔드는 이미 문서 파일(PDF, Word, Excel, PPT) 업로드를 지원하고 있으며 다운로드(sendFile)도 가능. 프론트엔드 에디터에서만 이미지 외 파일 업로드를 차단하고 있는 상태.

### 1.5 이슈 Description/Comment 렌더링

- **Description**: `dangerouslySetInnerHTML={{ __html: currentIssue.description }}` (HTML 직접 렌더링)
- **Comment**: `dangerouslySetInnerHTML={{ __html: comment.content }}` (HTML 직접 렌더링)
- HTML 내 `<a>` 태그 링크가 자연스럽게 렌더링됨 → 파일 다운로드 링크도 정상 작동 가능

## 4. 구현 방안

### 4.1 방안 A: RichTextEditor에 파일 업로드 버튼 추가 (권장)

에디터 컴포넌트에 파일 업로드 전용 버튼을 추가하고, 업로드 완료 시 다운로드 링크를 에디터에 삽입합니다. 

#### 4.1.1 프론트엔드 변경

**RichTextEditor.tsx 수정사항:**

1. **파일 업로드 버튼 추가** (📎 아이콘 사용)
   ```
   [ Bold ] [ Italic ] [ Underline ] | [ H2 ] | [ 이미지 ▼ ] [ 📎 파일 ] [ 링크 ] | [ Undo ] [ Redo ] | [ Source ]
   ```

2. **파일 업로드 핸들러 추가**
   ```typescript
   // 새 file input (accept 제한 없음 or 문서 유형만 허용)
   <input type="file" accept=".pdf,.doc,.docx,.xls,.xlsx,.ppt,.pptx" />
   
   // 업로드 처리
   const uploadAndInsertFile = async (file: File) => {
     const formData = new FormData();
     formData.append('file', file);
     const res = await apiClient.post('/files/upload', formData);
     const { storedName, originalName, fileSize } = res.data.data;
     const downloadUrl = `${baseURL}/files/${storedName}/download`;
     const sizeStr = formatFileSize(fileSize);
     
     // 에디터에 다운로드 링크 삽입
     editor.chain().focus()
       .insertContent(`<a href="${downloadUrl}" target="_blank" rel="noopener">📎 ${originalName} (${sizeStr})</a>&nbsp;`)
       .run();
   };
   ```

3. **드래그 앤 드롭 확장** (선택)
   - 기존: 이미지 파일만 처리 → 비이미지 파일 드롭 시 파일 업로드로 처리
   - 드롭 영역 메시지: "이미지를 여기에 놓으세요" → "파일을 여기에 놓으세요"

#### 4.1.2 백엔드 변경

**FileController에 다운로드 엔드포인트 추가:**

```typescript
@Public()
@Get(':filename/download')
downloadFile(@Param('filename') filename: string, @Res() res: Response): void {
  // 파일명 검증 (path traversal 방지)
  if (filename.includes('..') || filename.includes('/') || filename.includes('\\')) {
    throw new BadRequestException('Invalid filename.');
  }
  const filePath = this.fileService.getFilePath(filename);
  if (!existsSync(filePath)) {
    throw new NotFoundException('File not found.');
  }
  // originalName은 DB에 없으므로 storedName으로 다운로드
  // (또는 query param으로 원본명 전달)
  res.download(filePath, filename);
}
```

**개선안: 원본 파일명 보존을 위한 옵션**
- 프론트에서 링크 생성 시 원본 파일명을 query parameter로 포함:
  ```
  /api/v1/files/{storedName}/download?name={originalName}
  ```
- 백엔드에서 `res.download(filePath, queryName || filename)` 처리

#### 4.1.3 업로드된 파일 링크의 HTML 형태

에디터에 삽입되는 HTML:
```html
<a href="/api/v1/files/{uuid}.xlsx/download?name=매출보고서.xlsx" target="_blank" rel="noopener">
  📎 매출보고서.xlsx (245KB)
</a>
```

이슈 Description 및 Comment 렌더링 시 이 HTML이 그대로 출력되어 클릭 시 파일 다운로드가 실행됩니다.

## 5. 사용자 플로우

### 5.1 이슈 작성 시 파일 업로드

```
1. 사용자가 /issues 에서 "New Issue" 클릭
2. IssueFormModal 열림 → Description 에디터 표시
3. 에디터 툴바에서 📎 (파일 첨부) 버튼 클릭
4. 파일 선택 다이얼로그 열림 (PDF, Word, Excel, PPT 허용)
5. 파일 선택
6. 로딩 인디케이터 표시
7. POST /api/v1/files/upload → 파일 업로드
8. 응답의 storedName, originalName, fileSize로 다운로드 링크 생성
9. 에디터에 "📎 파일명.xlsx (245KB)" 형태의 링크 삽입
10. 이슈 저장 시 Description HTML에 링크가 포함되어 저장
```

### 5.2 이슈 상세 댓글 작성 시 파일 업로드

```
1. 사용자가 이슈 상세 모달 하단 댓글 에디터에 접근
2. 동일한 RichTextEditor → 동일한 📎 버튼 사용
3. 파일 업로드 → 다운로드 링크 삽입
4. "Send" 버튼으로 댓글 추가
5. POST /api/v1/issues/:id/comments (content에 HTML 포함)
```

### 5.3 파일 다운로드

```
1. 이슈 Description 또는 Comment에서 "📎 파일명.xlsx (245KB)" 링크 표시
2. 사용자 클릭
3. GET /api/v1/files/{storedName}/download?name={originalName}
4. Content-Disposition: attachment; filename="파일명.xlsx"
5. 브라우저 다운로드 시작
```

## 6. 기술 제약사항 및 고려사항

### 6.1 기술 제약

| 항목 | 설명 |
|------|------|
| **TipTap 링크 삽입** | `insertContent()` 로 `<a>` 태그 삽입 가능 (Link extension 이미 활성화됨) |
| **파일 크기 제한** | 현재 10MB — 문서 파일 기준으로 충분. 변경 불필요 |
| **MIME 타입** | 이미 PDF/Word/Excel/PPT 지원됨. 추가 타입 필요 시 서비스 코드 수정 |
| **파일 저장소** | 로컬 파일시스템 사용 중. Docker 볼륨 매핑 필요 (스테이징/프로덕션) |
| **파일 영속성** | 로컬 저장소이므로 컨테이너 재생성 시 파일 손실 가능 — Docker volume 확인 필요 |

### 6.2 보안 고려

| 항목 | 대응 |
|------|------|
| **Path Traversal** | 기존 `filename.includes('..')` 체크 유지 |
| **MIME 타입 검증** | 서버 사이드 whitelist 방식 유지 |
| **파일명 난독화** | UUID 기반 storedName 유지 |
| **XSS** | 에디터 HTML 내 `<a>` href는 업로드 API 경로로 제한 |
| **다운로드 API 인증** | 현재 `@Public()` 데코레이터 사용 — 링크를 아는 사람은 누구나 다운로드 가능 (기존 이미지와 동일) |

### 6.3 영향 범위

RichTextEditor는 공통 컴포넌트이므로 파일 업로드 버튼 추가 시 다음 모듈에도 자동 적용:

| 모듈 | 컴포넌트 | 영향 |
|------|---------|------|
| **이슈** | IssueFormModal, IssueDetailModal | ✅ 대상 |
| **회의록** | MeetingNoteFormModal | ✅ 자동 적용 |
| **공지사항** | NoticeFormModal | ✅ 자동 적용 |
| **Todo** | TodoFormModal | ✅ 자동 적용 |
| **Today** | MissionSection | ✅ 자동 적용 |

→ 이슈 외 다른 모듈에서도 파일 첨부가 가능해지는 **긍정적 부수 효과**
