# REQ-사용자목록및코멘트기능개선-20260402

## 1. 요구사항 요약

### 요구사항 1: 사용자 리스트 선택 개선
이슈 작성 시 Assignee, Participants 선택과 @멘션 등 **모든 사용자 선택 UI**에서:
- Active 상태 사용자만 표시
- 회사 직원(USER_LEVEL) 리스트 우선 노출 (CLIENT_LEVEL 하단 배치)

### 요구사항 2: 코멘트 기능 개선
이슈, 노트, 태스크 등에서 사용하는 코멘트 기능:
- **2.1** 코멘트 @멘션 드롭다운이 위로 표시되어 에디터 메뉴에 가려짐 → **아래로 표시**
- **2.2** 코멘트 리스트 가독성 개선 — 작성자명+날짜 라인 배경색을 본문보다 진하게
- **2.3** 코멘트 Reply(답글) 기능 추가 — Reply 배경색은 원 코멘트보다 진하게

## 2. AS-IS 현황 분석

### 2.1 사용자 목록 데이터 흐름

| 계층 | 파일 | 역할 |
|------|------|------|
| Hook | `apps/web/src/domain/members/hooks/useMembers.ts` | `useMemberList()` → `membersService.getMembers()` |
| Service | `apps/web/src/domain/members/service/members.service.ts` | `GET /api/v1/members` |
| Backend Controller | `apps/api/src/domain/members/controller/member.controller.ts` | `findAll(req.entityId)` |
| Backend Service | `apps/api/src/domain/members/service/member.service.ts` | EntityId 기반 사용자 조회 |

**현재 백엔드 `findAll()` 로직:**
1. EntityUserRole `eurStatus='ACTIVE'`인 사용자 ID 수집
2. `usrCompanyId=entityId`인 사용자 ID 수집
3. CLIENT_LEVEL 고객사 사용자 ID 수집
4. 합집합으로 `usrCreatedAt DESC` 정렬 조회
5. **`usrStatus` 필터 없음** → INACTIVE/SUSPENDED/PENDING 사용자도 반환됨

**현재 MemberResponse 반환 필드:**
```typescript
interface MemberResponse {
  userId, email, name, unit, role,
  levelCode?: string,   // USER_LEVEL | CLIENT_LEVEL | ADMIN_LEVEL
  status?: string,      // ACTIVE | INACTIVE | SUSPENDED | PENDING | WITHDRAWN
  cells: { cellId, name }[],
  createdAt: string
}
```

## 3. TO-BE 요구사항

### 3.1 사용자 리스트 개선
1. **백엔드**: `member.service.ts`의 `findAll()`에서 `usrStatus='ACTIVE'` 필터 추가
2. **프론트엔드**: `useMemberList()` 반환값을 USER_LEVEL 우선 정렬
   - 정렬 순서: USER_LEVEL → CLIENT_LEVEL → 기타 → 이름 순
3. **적용 범위**: 모든 사용자 선택 컴포넌트 (AssigneeSelector, ParticipantSelector, MentionInput, RichTextEditor, TalkMessageInput)

### 3.2 멘션 드롭다운 방향
- RichTextEditor의 멘션 드롭다운: `bottom-full` → `top-full` (아래로 표시)
- MentionInput은 사용 상황에 따라 `bottom-full` 유지 또는 변경 검토

### 3.3 코멘트 가독성 개선
- **작성자/날짜 라인**: `bg-gray-100` 배경 + 좌우 padding
- **코멘트 본문**: `bg-gray-50` 배경 유지
- **적용 대상**: Issue 코멘트, Meeting Note 코멘트 (RichTextEditor 사용 모듈)

### 3.4 코멘트 Reply 기능
- **DB**: `amb_issue_comments`에 `isc_parent_id` 컬럼 추가, `amb_meeting_note_comments`에 `mnc_parent_id` 컬럼 추가
- **백엔드 API**: 코멘트 생성 시 `parent_id` 파라미터 지원, 조회 시 트리 구조 반환
- **프론트엔드**: Reply 버튼 → 중첩 표시, Reply 배경 `bg-gray-100` (원 코멘트 `bg-gray-50`보다 진하게)

## 5. 사용자 플로우

### 5.1 이슈 생성 시 Assignee 선택
```
사용자 → Assignee 클릭 → 드롭다운 표시
→ [Active USER_LEVEL 직원 목록] (이름순)
→ [Active CLIENT_LEVEL 고객사 목록] (이름순)
→ 검색 입력 시 필터링 → 선택
```

### 5.2 코멘트 Reply
```
코멘트 목록 → 특정 코멘트에 Reply 버튼 클릭
→ 해당 코멘트 하단에 Reply 입력 에디터 표시
→ 내용 작성 → 저장
→ 원 코멘트 아래에 들여쓰기+진한 배경으로 Reply 표시
```

### 5.3 멘션 (@)
```
코멘트 에디터에서 @ 입력
→ 에디터 아래쪽에 사용자 드롭다운 표시
→ Active USER_LEVEL 먼저 → CLIENT_LEVEL 순서
→ 검색/선택
```

## 6. 기술 제약사항

1. **DB 스키마 변경**: `isc_parent_id`, `mnc_parent_id` 컬럼 추가 필요 → 스테이징/프로덕션은 수동 SQL 실행
2. **TypeORM synchronize 비활성화**: 스테이징/프로덕션은 `NODE_ENV=production`으로 `synchronize: false`
3. **사용자 목록 정렬**: 백엔드에서 정렬하면 전체 API 영향범위 넓음 → 프론트 `useMemberList()` 후처리 정렬이 안전
4. **코멘트 공유 컴포넌트**: 현재 모듈별 개별 구현 → 통합 리팩토링 대신 각 모듈별 개별 수정 (스코프 제한)
5. **Reply 깊이**: 1 depth만 허용 (무한 중첩 방지)
