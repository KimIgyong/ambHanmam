# 요구사항 분석서: 웹메일 계정-사용자 관계 설정 기능 개선

**작성일**: 2026-02-21
**요청 배경**: 웹메일 계정과 시스템 사용자 간 관계를 명확히 확인/관리할 수 있어야 하며, 멤버 상세 페이지에서 회사 이메일을 수정할 수 있어야 함

---

## 1. AS-IS 현황 분석

### 1-1. 웹메일 계정 관리 (`/settings/mail-accounts`)
- 계정 목록에서 `userName`, `userRole`, `userEmail` 정보를 JOIN하여 표시 중
- 그러나 **카드 UI에 사용자 시스템 이메일(로그인 이메일)이 표시되지 않음**
- 사용자 클릭 시 해당 멤버 상세로 이동하는 네비게이션 없음

### 1-2. 멤버 상세 (`/settings/members/:id`)
- `companyEmail` 필드가 읽기 전용(InfoItem)으로 표시됨
- DB 테이블 `amb_users.usr_company_email` 컬럼 존재
- **수정 API(PATCH)가 없음** — `updateRole`만 존재
- 회사 이메일을 변경하려면 DB 직접 수정 필요

### 1-3. 데이터 모델
| 테이블 | 컬럼 | 설명 |
|--------|------|------|
| `amb_users` | `usr_email` | 로그인 이메일 (시스템 계정) |
| `amb_users` | `usr_company_email` | 회사 이메일 (표시용) |
| `amb_mail_accounts` | `mac_email` | 웹메일 이메일 주소 |
| `amb_mail_accounts` | `mac_user_id` → `amb_users.usr_id` | 사용자 연결 |

---

## 2. TO-BE 요구사항

### 요구사항 A: 웹메일 계정 카드에서 사용자 관계 명확화
- 카드에 **시스템 이메일(userEmail)** 추가 표시
- 사용자 이름 클릭 시 `/settings/members/:userId`로 이동 링크

### 요구사항 B: 멤버 상세에서 회사 이메일 수정
- `companyEmail` InfoItem을 인라인 편집 가능하게 변경
- 백엔드 `PATCH /members/:id/company-email` API 추가
- 프론트엔드 mutation 훅 + UI 수정

## 4. 기술 제약사항
- `usr_company_email`은 unique 제약조건이 있으므로 중복 검증 필요
- 메일 계정 카드에서 멤버 링크는 ADMIN/MANAGER 권한 사용자만 이동 가능 (기존 MenuGuard 활용)

## 5. 영향 범위
