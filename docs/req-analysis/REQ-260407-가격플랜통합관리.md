# REQ-260407 — 가격 플랜 통합 관리 (Price Plan Unified Management)

## 1. 요구사항 요약

### 배경
현재 AMA 구독 페이지(`/entity-settings/subscription` → Price Plans 탭)와 포털 요금제 페이지(`www.amoeba.site/page/pricing`)가 **서로 다른 데이터 소스**를 사용하여 가격 정보가 불일치할 위험이 있다.

### 핵심 요구사항
1. **데이터 통합**: AMA Price Plans와 Portal Pricing 페이지가 **동일한 DB 테이블 데이터**를 사용
2. **Admin 가격 관리 페이지**: ADMIN_LEVEL 전용 `/admin/service/priceplan` 페이지 신규 생성
3. **관리 기능**: 가격 플랜의 항목별 단위, 단가, 수량, 가격, 옵션을 테이블 형태로 입력/수정
4. **조건 동기화**: Portal Pricing 페이지의 모든 항목/조건이 DB 기반으로 동적 표시

**PricingPage.tsx 하드코딩 데이터**:

#### 섹션 1: Plan Cards (3개)
- FREE: $0, 50,000T 1회, 1GB, 최대 5명
- BASIC: $2/user/mo, 20,000T/user/mo, 3GB(~100GB), 6-49명 (Recommended)
- PREMIUM: Custom, Custom, Custom, 50+명

#### 섹션 2: Main Comparison Table (`comparisonRows` 배열)
14행 비교표: 월간요금, 사용자구성, 월정액, 토큰기본, 토큰추가방법, 토큰리셋, 스토리지기본, 스토리지추가, 스토리지한도, 추천보상, 연간할인, 전담지원, SLA, 커스텀연동

#### 섹션 3: BASIC Tier Detail Table (`BASIC_TIERS` 배열)
9개 티어 (Tier 1~9): 사용자 6-49명 구간, 월/연간 요금, 절감액, 토큰/월

#### 섹션 4: Annual Billing Comparison
월간/연간 결제 비교 카드 (2개월 무료 혜택)

#### 섹션 5: ADD-ON Pricing Table
토큰 추가(10K/$1), 스토리지 추가(1GB/$1), 추천보상(50KT 무료)

#### 섹션 6: Storage Policy Table
기본 스토리지, 추가 스토리지, 최대한도, Google Drive 정책

#### 섹션 7: Referral Reward Section
초대 조건(10명), 보상(50,000T), 제한사항

#### 섹션 8: Premium Cases + FAQ
Premium 적용 사례, FAQ 5개

### 3.2 Admin 가격 관리 페이지

**경로**: `/admin/service/priceplan`
**대상**: ADMIN_LEVEL 전용
**기능**:

| 기능 | 설명 |
|------|------|
| 플랜 기본 정보 관리 | 코드, 이름, 티어, 가격/user, 활성 여부, 정렬 순서 |
| 토큰 정책 관리 | 1회 지급량, 월/user 지급량, 추가 단위/단가, 월간 리셋 여부 |
| 스토리지 정책 관리 | 기본 용량, 최대 용량, 추가 단위/단가, 경고/차단 임계값 |
| 사용자 정책 관리 | 최소/최대 인원, 슬롯 크기, 무료 포함 인원 |
| 추천 정책 관리 | 활성 여부, 보상 토큰, 필요 초대 수 |
| 연간 청구 관리 | 연간 가용 여부, 할인 개월 수 |
| 플랜 비교 항목 관리 | 비교표 행 데이터 (feature key, free/basic/premium 값) |
| BASIC Tier 상세 관리 | 사용자 구간별 월/연간 요금, 절감액, 토큰량 |
| ADD-ON 상품 관리 | 추가 상품 (토큰, 스토리지, 추천보상) 항목/단가/단위 |

### 3.3 Portal Pricing 동적화

현재 PricingPage.tsx의 모든 하드코딩 데이터를 API 기반으로 교체:
- Plan Cards → `GET /api/v1/subscriptions/plans` 데이터 활용
- Comparison Table → `GET /api/v1/subscriptions/plans/comparison` (신규 API)
- BASIC Tiers → `GET /api/v1/subscriptions/plans/tiers` (신규 API)
- ADD-ON Table → plans 데이터의 addon 필드 활용
- Storage Policy → plans 데이터의 storage 필드 활용
- Referral Section → plans 데이터의 referral 필드 활용

## 5. 사용자 플로우

### 5.1 Admin: 가격 플랜 관리
```
Admin → /admin → 사이드바 → Service > Price Plan
  → 플랜 목록 테이블 (FREE/BASIC/PREMIUM)
  → 행 클릭 → 인라인 편집 or 상세 패널
    → 기본 정보: 이름, 가격/user, 활성 여부
    → 토큰 정책: 1회지급, 월지급, 추가단위/단가
    → 스토리지 정책: 기본GB, 최대GB, 추가단위/단가
    → 사용자 정책: min/max, 슬롯, 무료인원
    → 추천 정책: 활성, 보상토큰, 필요초대수
    → 연간 청구: 가용, 무료개월
  → 비교 항목 관리 탭
    → 14행 비교표 데이터 (feature key, free/basic/premium 값)
    → 행 추가/수정/삭제/순서변경

    → 사용자구간별 티어 테이블
    → 행 추가/수정/삭제
  → 저장 → toast 성공 알림
```

## 6. 기술 제약사항

### 6.1 보안
- Admin CRUD API는 **ADMIN_LEVEL 전용** (`@Auth()` + ADMIN_LEVEL 검증)
- 조회 API (`GET /plans`, `/plans/comparison`, `/plans/tiers`)는 **`@Public()`** (Portal에서 비인증 접근)
- 가격 변경 이력 추적 필요 (audit trail)

### 6.2 성능
- 플랜 데이터 변경 빈도 매우 낮음 → `staleTime: 10분` 캐싱 유지
- Portal PricingPage는 SSR 없이 CSR → API 호출 latency 감안 (skeleton UI)

### 6.3 호환성
- 기존 `GET /subscriptions/plans` API 응답 형식 유지 (하위 호환)
- 기존 `amb_sub_plans` 테이블 스키마 유지 (컬럼 추가만, 삭제/변경 없음)
- Portal PricingPage i18n 키 구조 유지 (하드코딩 값만 동적 교체)

### 6.4 데이터 정합성
- Portal 표시 데이터와 DB 시드 데이터 불일치 해소 → **DB 시드를 Portal 기준으로 업데이트**
- BASIC $2/user/mo가 최종 가격인지, DB $5/user가 최종인지 **사업 확인 필요**

### 6.5 배포
- 스테이징/프로덕션: TypeORM synchronize 비활성 → **수동 SQL 마이그레이션 필수**
- 새 테이블 생성 + 시드 데이터 삽입 SQL 별도 관리
