# REQ-이슈삭제권한정책정리-20260331

## 1. 요구사항 요약

이슈 삭제 기능의 권한 정책을 3단계로 세분화하여, 생성자/프로젝트매니저/MASTER 각각의 역할에 맞는 삭제 권한을 부여한다.

| 역할 | 삭제 유형 | 조건 |
|------|-----------|------|
| 이슈 생성자 (Reporter) | Soft Delete | 다른 사람에게 할당(assignee)되지 않은 경우에만 |
| 프로젝트 매니저 (PM) | Soft Delete / 숨김 | 해당 프로젝트에 속한 이슈 |
| MASTER 레벨 사용자 | Hard Delete (영구 삭제) | 모든 이슈 |

## 2. AS-IS 현황 분석

### 2.1 백엔드 (API)

**컨트롤러**: `apps/api/src/domain/issues/controller/issue.controller.ts`
```typescript
@Delete(':id')
@UseGuards(RolesGuard)
@Roles('MANAGER')
@HttpCode(HttpStatus.NO_CONTENT)
async deleteIssue(@Param('id') id: string) {
  await this.issueService.deleteIssue(id);
}
```
- `@Roles('MANAGER')`: 시스템 역할 MANAGER 이상만 삭제 가능 (hierarchy level ≥ 3)
- **문제**: 요청자의 userId를 서비스에 전달하지 않아, 생성자/PM 판별 불가
- **문제**: `OwnEntityGuard` 미적용 → ent_id 격리 없음

**서비스**: `apps/api/src/domain/issues/service/issue.service.ts`
```typescript
async deleteIssue(id: string): Promise<void> {
  const entity = await this.issueRepository.findOne({ where: { issId: id } });
  if (!entity) throw new BusinessException(...);
  await this.issueRepository.softRemove(entity);
}
```
- 오직 `softRemove()` 한 가지 방식만 존재
- Hard Delete(영구 삭제) 로직 없음
- 권한 검증 없음 (생성자/PM/할당여부 무관하게 삭제)

**엔티티**: `apps/api/src/domain/issues/entity/issue.entity.ts`
- `iss_reporter_id`: 이슈 생성자 (UUID)
- `iss_assignee_id`: 담당자 (UUID, nullable)
- `pjt_id`: 프로젝트 ID (UUID, nullable)
- `iss_deleted_at`: Soft Delete 컬럼 (`@DeleteDateColumn`)

### 2.2 프론트엔드 (Web)

**삭제 버튼 노출 조건** — `IssueDetailModal.tsx`:
```typescript
const isManager = user?.role === 'MANAGER' || user?.role === 'ADMIN' || user?.role === 'MASTER' || user?.role === 'SUPER_ADMIN';
const isInvolved = isReporter || user?.userId === currentIssue.assigneeId || isParticipant;

{(isManager || isInvolved) && ( <button onClick={() => onDelete(currentIssue)}> ... </button> )}
```
- MANAGER 이상이거나 관련자(생성자/담당자/참여자)이면 삭제 버튼 표시

**삭제 확인 모달** — `IssueDeleteConfirmModal.tsx`:
```typescript
<button onClick={onConfirm} disabled={!isMaster} ... />
{!isMaster && ( <p>"이슈 삭제는 마스터 권한 소유자만 가능합니다"</p> )}
```
- **isMaster만 삭제 버튼 활성화** → 실질적으로 MASTER만 삭제 가능
- 생성자/PM은 삭제 버튼은 보이지만 모달에서 차단됨

**IssuesPage.tsx**:
```typescript
const isMaster = useAuthStore((s) => s.isMaster());
<IssueDeleteConfirmModal isMaster={isMaster} ... />
```

### 2.3 현재 동작 요약

| 역할 | 삭제 버튼 표시 | 모달 확인 가능 | API 성공 | 실제 결과 |
|------|---------------|---------------|---------|----------|
| 생성자 (MEMBER) | ✅ | ❌ (disabled) | ❌ (403) | 삭제 불가 |
| 생성자 (MANAGER) | ✅ | ❌ (disabled) | ✅ | 모달에서 차단 |
| PM (MANAGER) | ✅ | ❌ (disabled) | ✅ | 모달에서 차단 |
| MASTER | ✅ | ✅ | ✅ | Soft Delete |
| ADMIN | ✅ | ❌ (disabled) | ✅ | 모달에서 차단 |

### 2.4 프로젝트 매니저 판별 기준

- `kms_projects.pjt_manager_id`: 프로젝트 매니저 사용자 ID
- `kms_project_members.pmb_role`: 프로젝트 내 역할 ('MANAGER', 'MEMBER', 'VIEWER')
- 이슈의 `pjt_id`로 프로젝트 연결, 해당 프로젝트의 `pjt_manager_id` 또는 `pmb_role === 'MANAGER'`로 PM 여부 판별

## 3. TO-BE 요구사항

### 3.1 삭제 권한 3단계 정책

#### Level 1: 생성자 삭제 (Reporter Soft Delete)
- **대상**: 이슈 생성자 (`iss_reporter_id === userId`)
- **조건**: `iss_assignee_id`가 NULL인 경우에만 (다른 사람에게 할당되지 않은 이슈)
- **동작**: Soft Delete (`iss_deleted_at` 설정)
- **UI**: 삭제 확인 모달에서 바로 확인 가능

#### Level 2: PM 삭제 (Project Manager Soft Delete)
- **대상**: 이슈가 속한 프로젝트의 매니저
  - `kms_projects.pjt_manager_id === userId` OR
  - `kms_project_members` 에서 `pjt_id + usr_id + pmb_role === 'MANAGER'`
- **조건**: 이슈의 `pjt_id`가 해당 프로젝트에 속해야 함
- **동작**: Soft Delete (`iss_deleted_at` 설정)
- **UI**: 삭제 확인 모달에서 바로 확인 가능

#### Level 3: MASTER 영구 삭제 (Hard Delete)
- **대상**: MASTER 역할 사용자 (`user.role === 'MASTER'`)
- **조건**: 없음 (모든 이슈 대상)
- **동작**: Hard Delete (`repository.remove()` — DB에서 완전 제거)
- **UI**: 삭제 확인 모달에서 "영구 삭제" 구분 표시, 경고 강화

### 3.2 API 변경

**기존 단일 엔드포인트 → 삭제 유형별 분리**

| Method | Path | 권한 | 동작 |
|--------|------|------|------|
| DELETE | `/api/v1/issues/:id` | MEMBER+ | Soft Delete (생성자/PM 조건 검증) |
| DELETE | `/api/v1/issues/:id/permanent` | MASTER+ | Hard Delete (영구 삭제) |

### 3.3 프론트엔드 변경

- 삭제 확인 모달을 조건에 따라 동적으로 표시
  - 생성자/PM: "이 이슈를 삭제하시겠습니까?" + 확인 버튼 활성
  - MASTER: "영구 삭제" 옵션 추가 (Soft Delete 기본, Hard Delete 선택 가능)
  - 권한 없는 사용자: 삭제 버튼 비노출
- 삭제 버튼 노출 조건 변경:
  - 생성자 && assignee 없음 → 표시
  - PM → 프로젝트 이슈에 대해 표시
  - MASTER → 항상 표시

## 5. 사용자 플로우

### 5.1 생성자가 자기 이슈 삭제
```
생성자가 이슈 상세 열기
→ assignee가 없는 이슈 → 삭제 버튼 표시
→ 삭제 클릭 → 확인 모달 ("이 이슈를 삭제하시겠습니까?")
→ 확인 → DELETE /api/v1/issues/:id (soft delete)
→ 성공 토스트 + 목록 갱신

생성자가 이슈 상세 열기
→ assignee가 있는 이슈 → 삭제 버튼 비표시
```

### 5.2 PM이 프로젝트 이슈 삭제
```
PM이 이슈 목록에서 자기 프로젝트 이슈 클릭
→ 삭제 버튼 표시 (프로젝트 매니저이므로)
→ 삭제 클릭 → 확인 모달
→ 확인 → DELETE /api/v1/issues/:id (soft delete)
→ 성공 토스트 + 목록 갱신
```

### 5.3 MASTER가 영구 삭제
```
MASTER가 이슈 상세 열기
→ 삭제 버튼 표시 (항상)
→ 삭제 클릭 → 확인 모달:
  - [삭제] 버튼 (Soft Delete, 기본)
  - [영구 삭제] 버튼 (Hard Delete, 빨간색 경고)
→ 영구 삭제 클릭 → 2차 확인 ("정말로 영구 삭제하시겠습니까? 복구할 수 없습니다.")
→ 확인 → DELETE /api/v1/issues/:id/permanent (hard delete)
→ 성공 토스트 + 목록 갱신
```

## 6. 기술 제약사항

- **DB 마이그레이션**: 스키마 변경 없음 (기존 `iss_deleted_at` 컬럼 활용)
- **Hard Delete 시 참조 무결성**: 이슈 코멘트(`amb_issue_comments`), 참여자(`amb_issue_participants`), 상태 로그(`amb_issue_status_logs`), 평가(`amb_issue_ratings`) 등 하위 데이터도 함께 삭제 필요
- **프로젝트 매니저 판별**: `kms_projects` + `kms_project_members` 조인 필요 → 쿼리 비용 최소화 고려
- **OwnEntityGuard 적용**: 기존 삭제 API에 ent_id 격리 누락 → 반드시 추가
- **캐스케이드 삭제**: Hard Delete 시 FK 제약조건에 따른 하위 테이블 처리 필요
