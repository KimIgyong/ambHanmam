## 1. 개요

### 1.1 목적

참조 문서 `PRJ-REQ-WBS-1.2.1`의 WBS(Work Breakdown Structure) 기능 요구사항을 현행 AMA 시스템의 코드·스키마·UI와 대조 분석하여, **현 시스템에 적용 가능한 구현 범위와 갭(Gap)을 식별**한다.

### 1.2 분석 범위

| 구분 | 대상 |
|------|------|
| **Backend** | Issue Entity (`amb_issues`), Project Entity (`kms_projects`), Issue Service/Controller, DTO |

#### 3.1.1 신규 테이블: `amb_project_epics`

```sql
CREATE TABLE amb_project_epics (
    epc_id          UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    ent_id          UUID         NOT NULL,
    pjt_id          UUID         NOT NULL,          -- 현행 kms_projects.pjt_id 참조
    epc_title       VARCHAR(200) NOT NULL,
    epc_description TEXT,
    epc_status      VARCHAR(20)  NOT NULL DEFAULT 'PLANNED',
    epc_color       VARCHAR(7),
    epc_start_date  DATE,
    epc_due_date    DATE,
    epc_created_by  UUID         NOT NULL,
    epc_created_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    epc_updated_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    epc_deleted_at  TIMESTAMPTZ,

    CONSTRAINT fk_amb_project_epics_entities FOREIGN KEY (ent_id)
        REFERENCES amb_hr_entities(ent_id),
    CONSTRAINT fk_amb_project_epics_projects FOREIGN KEY (pjt_id)
        REFERENCES kms_projects(pjt_id),              -- ⚠️ 현행 테이블명 반영
    CONSTRAINT fk_amb_project_epics_users    FOREIGN KEY (epc_created_by)
        REFERENCES amb_users(usr_id)
);

CREATE INDEX idx_amb_project_epics_pjt_id ON amb_project_epics(pjt_id, epc_status)
    WHERE epc_deleted_at IS NULL;
CREATE INDEX idx_amb_project_epics_ent_id ON amb_project_epics(ent_id)
    WHERE epc_deleted_at IS NULL;
```

#### 3.1.3 기존 테이블 변경: `amb_issues`

```sql
-- Epic/Component FK 컬럼 추가
ALTER TABLE amb_issues
    ADD COLUMN epc_id UUID,
    ADD COLUMN cmp_id UUID;

-- FK 제약 추가
ALTER TABLE amb_issues
    ADD CONSTRAINT fk_amb_issues_epics FOREIGN KEY (epc_id)
        REFERENCES amb_project_epics(epc_id),
    ADD CONSTRAINT fk_amb_issues_components FOREIGN KEY (cmp_id)
        REFERENCES amb_project_components(cmp_id);

-- Epic/Component 동시 지정 방지 CHECK
ALTER TABLE amb_issues
    ADD CONSTRAINT ck_amb_issues_epic_or_component
    CHECK (NOT (epc_id IS NOT NULL AND cmp_id IS NOT NULL));

-- 인덱스
CREATE INDEX idx_amb_issues_epc_id ON amb_issues(epc_id)
    WHERE epc_id IS NOT NULL;
CREATE INDEX idx_amb_issues_cmp_id ON amb_issues(cmp_id)
    WHERE cmp_id IS NOT NULL;
```

> **참조 문서 대비 변경점:**
> - 테이블명: `amb_project_issues` → `amb_issues` (현행)
> - FK 컬럼: `prj_id` → `pjt_id` (현행)
> - FK 제약명: 현행 테이블명 기준으로 조정
> - CHECK 제약명: `ck_amb_issues_epic_or_component`

## 7. 기술 제약사항

### 7.1 TypeORM 제약

- **서브쿼리 orderBy 제한**: TypeORM의 `.orderBy()`는 서브쿼리 별칭을 직접 사용할 수 없음 (이전 세션에서 확인). WBS 트리 조회 시 복잡한 정렬은 **JavaScript 후처리**로 해결 필요.
- **트리 구조 쿼리**: WBS 트리 조회는 Epic 목록 + Component 목록 + 미지정 이슈 목록을 별도 쿼리로 조회 후 JavaScript에서 트리 조합하는 방식 권장.

### 7.2 현행 테이블명 차이

참조 문서의 SQL/API 설계에서 사용하는 테이블명과 현행 시스템의 테이블명이 다르므로, **모든 FK 참조 및 쿼리를 현행 테이블명 기준으로 변환**해야 한다:

| 참조 문서 | 현행 시스템 |
|-----------|-------------|
| `amb_projects` | `kms_projects` |
| `amb_projects.prj_id` | `kms_projects.pjt_id` |
| `amb_project_issues` | `amb_issues` |
| `amb_project_issues.prj_id` | `amb_issues.pjt_id` |

### 7.3 이슈 계층 구조 공존

현행 Parent-Child 계층 (`iss_parent_id`)과 신규 Epic/Component 그룹핑은 **독립적으로 공존**한다:
- **Level 0**: Project
- **Level 1**: Epic / Component / [미지정] (신규)
- **Level 2**: Issue (기존 `pjt_id`로 프로젝트 연결)
- **Level 3**: Sub-Issue (기존 `iss_parent_id`로 부모 연결)

하나의 이슈가 Epic에 속하면서 동시에 다른 이슈의 Sub-Issue일 수 있다.

## 8. 결론

### 8.1 적용 가능성 평가

| 항목 | 판정 | 근거 |
|------|------|------|
| **DB 스키마** | ✅ 적용 가능 | 신규 테이블 2개 + ALTER 1개. 기존 데이터 영향 없음 (nullable 컬럼) |
| **Epic/Component CRUD API** | ✅ 적용 가능 | 기존 프로젝트 도메인 내 모듈 추가. 기존 API 영향 없음 |
| **WBS 탭 UI** | ✅ 적용 가능 | 기존 5개 탭 → 6개 탭으로 확장. 기존 탭 영향 없음 |
| **이슈 폼 그룹 필드** | ✅ 적용 가능 | optional 필드 추가. 기존 폼 동작 유지 |
| **간트 차트 개선** | ⚠️ 주의 필요 | 현행 parentIssueId 기반 그룹핑과 Epic/Component 기반 그룹핑 공존 설계 필요 |
| **드래그&드롭** | ✅ 적용 가능 | @dnd-kit 이미 칸반에서 사용 중. 동일 패턴 확장 |

### 8.2 핵심 주의사항

1. **테이블명/컬럼명 변환**: 참조 문서의 모든 SQL을 현행 테이블명 기준으로 변환해야 함
2. **TypeORM Entity 설계**: 참조 문서의 raw SQL을 TypeORM Entity + Migration으로 변환
3. **간트 차트 기존 로직**: parentIssueId 기반 계층을 유지하면서 Epic/Component 그룹을 상위 레이어로 추가하는 이중 계층 설계 필요
4. **권한 분리**: 이슈 일반 수정과 Epic/Component 지정 수정의 권한이 다르므로, updateIssue 내에서 필드별 권한 분기 로직 필요
