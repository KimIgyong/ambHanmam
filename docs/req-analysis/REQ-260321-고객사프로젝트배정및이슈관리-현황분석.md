> **작성일**: 2026-03-21  
> **목적**: 초대된 고객사 직원에게 프로젝트를 배정하고, 고객 포털에서 이슈를 등록·관리할 수 있는 기능의 현재 구현 현황 분석

## 1. 요구사항 요약

| # | 기능 | 설명 |
|---|------|------|

## 2. 전체 아키텍처 현황

### 2.1 관련 시스템 구성

```
┌─────────────────────────────────────────────────────────┐
│                   관리자(AMA) 측                          │
│                                                          │
│  Entity Settings > 고객관리    Project > 프로젝트 관리     │
│  ├─ 고객사 등록/초대            ├─ 프로젝트 CRUD           │
│  └─ 고객사 직원 초대 메일        ├─ 프로젝트 멤버 관리       │
│                                └─ 프로젝트-고객사 배정 API  │
└─────────────┬───────────────────────────┬───────────────┘
              │                           │
              ▼                           ▼
┌─────────────────────┐    ┌─────────────────────────────┐
│   amb_svc_clients    │    │   kms_project_clients        │
│  (고객사 정보)        │◄──│  (프로젝트-고객사 매핑)        │
└─────────────────────┘    └─────────────────────────────┘
              │                           │
              ▼                           ▼
┌─────────────────────────────────────────────────────────┐
│                고객 포털 (/client)                        │
│                                                          │
│  대시보드 │ 프로젝트 목록/상세 │ 이슈 목록/생성/상세 │ 프로필│
│           │                  │ 코멘트/해결확인      │       │
└─────────────────────────────────────────────────────────┘
```

### 2.2 데이터 흐름

```
1. Entity Settings에서 고객사 등록 (Billing Partners 기반)
2. 고객사 직원에게 초대 메일 발송
3. 직원이 초대 수락 → CLIENT_LEVEL 사용자 생성
4. 관리자가 프로젝트에 고객사 배정 (kms_project_clients)          ← ⚠️ 배정 UI 미구현
5. 고객사 직원이 /client/projects에서 배정된 프로젝트 조회
6. 프로젝트 내 이슈 등록/조회/코멘트/해결확인
```

### 3.4 이슈 상태 워크플로우

```
OPEN → APPROVED → IN_PROGRESS → RESOLVED → CLOSED
  ↑                                  │         
  └──────────────────────────────────┘ (재오픈)
  
OPEN → REJECTED → OPEN (재오픈)
```

| 상태 전이 | 관리자 API | 고객 포탈 API |
|-----------|-----------|--------------|
| OPEN → APPROVED | ✅ `/issues/:id/status` | ❌ 없음 |
| APPROVED → IN_PROGRESS | ✅ | ❌ 없음 |
| IN_PROGRESS → RESOLVED | ✅ | ❌ 없음 |
| **RESOLVED → CLOSED** | ✅ | ✅ `confirmResolution` |
| RESOLVED → OPEN (재오픈) | ✅ | ❌ 없음 |
| OPEN → REJECTED | ✅ | ❌ 없음 |

**고객 포털에서의 상태 변경**: RESOLVED → CLOSED (해결 확인)만 가능. 그 외의 상태 전이는 관리자 측에서만 수행.

## 7. 핵심 발견 사항

### 7.1 🔴 Critical: 프로젝트-고객사 배정 UI 부재

**현상**: 백엔드에 프로젝트-고객사 배정/해제 API (`/project/projects/:id/clients`)가 완전히 구현되어 있으나, 관리자 프론트엔드에서 이 API를 호출하는 UI가 없습니다.

**영향**: 
- 관리자가 프로젝트에 고객사를 배정할 수 없음
- 고객 포털에서 프로젝트 목록이 항상 비어 있음 (kms_project_clients 데이터 0건)
- 이슈 등록 시 프로젝트 선택 드롭다운이 비어 있음

**필요 작업**:
1. 프론트엔드 `project.service.ts`에 클라이언트 배정 API 메서드 추가
2. `ProjectOverviewTab.tsx` 또는 전용 탭에 고객사 배정 UI 추가
3. Entity의 고객사 목록을 조회하여 선택 → 배정하는 UX 구현

### 7.2 🟡 Warning: 고객 포탈 이슈 상태 변경 제한

**현상**: 고객은 RESOLVED → CLOSED (해결 확인)만 가능. 기타 상태 변경 API가 고객 포털에 없음.

**검토 필요**: 
- 고객이 변경할 수 있어야 하는 상태 범위 정의 필요
- 현재 구조상 합리적 범위: RESOLVED → CLOSED (해결 확인), OPEN 상태에서 이슈 취소 정도

### 7.3 🟢 Info: 인증 및 접근 제어 정상

- `ClientGuard`로 CLIENT_LEVEL 사용자만 고객 포털 접근 가능
- `ProjectClientEntity`로 고객사별 프로젝트 접근 권한 분리
- 이슈 조회 시 프로젝트 배정 여부 + reporter 여부 이중 검증
