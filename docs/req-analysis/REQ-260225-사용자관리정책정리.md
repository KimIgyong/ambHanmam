# REQ-사용자관리정책정리-20260225

## 1. 요구사항 개요

사용자관리(KMS) 시스템의 구분 체계를 정리하고, 명칭 통일 및 부서/그룹 관리 기능을 `/settings/entities` 경로에 탭으로 통합한다.

---

## 2. AS-IS 현황 분석

### 2.1 역할(Role) 체계

| 구분 | 값 | DB 컬럼 | 설명 |
|------|-----|---------|------|
| **그룹 코드** | `ADMIN_GROUP`, `USER_GROUP` | `amb_users.usr_group_code` | 사용자 레벨 그룹 |
| **시스템 역할** | `SUPER_ADMIN`, `ADMIN`, `MANAGER`, `MEMBER`, `VIEWER` | `amb_users.usr_role` | 5단계 역할 |
| **법인별 역할** | `SYSTEM_ADMIN`, `ADMIN`, `STAFF`, `MANAGER`, `USER` | `amb_hr_entity_user_roles.eur_role` | 법인 내 역할 |
| **부서별 역할** | `MEMBER`, `TEAM_LEAD`, `DEPARTMENT_HEAD` | `amb_user_dept_roles.udr_role` | 부서 내 역할 |

### 2.2 현재 DB 구조

#### `amb_users` (사용자)
| 컬럼 | 타입 | 용도 |
|------|------|------|
| `usr_id` (PK) | UUID | 사용자 고유 ID |

| `usr_group_code` | VARCHAR(30) | 그룹 코드 (USER_GROUP 기본) |
| `usr_job_title` | VARCHAR(100) | 직무 |
| `usr_company_id` | UUID FK | 소속 조직 |
| `usr_status` | VARCHAR(20) | PENDING/ACTIVE/INACTIVE/SUSPENDED/WITHDRAWN |

#### `amb_hr_entities` (법인/조직)
| 컬럼 | 타입 | 용도 |
|------|------|------|

| `eur_role` | VARCHAR(20) | SYSTEM_ADMIN / ADMIN / STAFF / MANAGER / USER |

#### `amb_departments` (부서)
| 컬럼 | 타입 | 용도 |

### 2.3 현재 UI 구조

#### 설정 > 사용자관리 (`/settings/members`)
- **Members 탭**: 사용자 목록 (이름, 이메일, 역할, 부서, 상태)
- **Groups 탭**: 그룹 CRUD (그룹명, 설명, 소속 법인, 멤버 관리)
- **Invitations 탭**: 초대 발송/관리

#### 설정 > 사용자 상세 (`/settings/members/:id`)
- **기본 정보**: Name, Email, Role, Department(레거시 텍스트), Job Title, Company Email
- **소속 법인**: Entity Name, Entity Code, Role, Status — 추가/삭제 가능
- **소속 부서**: Department Name, Role, isPrimary, Entity Name — 추가/삭제 가능
- **소속 그룹**: Group Name, Entity Name — 추가/삭제 가능
- **HR 직원 연결**: Employee Code, 부서, 직급 등 — 연결/해제

#### 설정 > 조직관리 (`/settings/entities`)
- 법인 CRUD (법인코드, 법인명, 국가, 통화, 대표자 등)
- 법인 상세 펼침 시 서브탭: **그룹 관리** / **부서 관리**

#### 설정 > 부서관리 (`/settings/departments`)
- 별도 페이지로 부서 트리 표시 (독립 경로)

#### 설정 > 권한관리 (`/settings/permissions`)
- **메뉴 설정 탭**: 메뉴 ON/OFF
- **등급별 권한 탭**: ADMIN_GROUP / USER_GROUP 서브탭
  - ADMIN_GROUP: SUPER_ADMIN, ADMIN — 전체 잠금 (항상 ON)
  - USER_GROUP: MANAGER, MEMBER, VIEWER — 토글 + 그룹별 체크박스
- **사용자별 권한 탭**: 개별 사용자 권한 오버라이드

### 2.4 현재 문제점

| # | 문제 | 상세 |
|---|------|------|
| 1 | **명칭 혼란** | `ADMIN_GROUP` / `USER_GROUP`이 "관리자 그룹" / "일반 사용자 그룹"으로 표시되어, `amb_groups` 테이블의 "그룹"(MS, BPO, DevOPS 등)과 혼동됨 |
| 2 | **부서 중복 관리** | `/settings/departments`(독립 페이지)와 `/settings/entities`(법인 상세 서브탭) 두 곳에서 부서 관리 가능 |
| 3 | **Function 필드 누락** | 요구사항에 Function(직능) 필드가 있으나, DB에 `usr_function` 컬럼 없음. `usr_department`(레거시 텍스트)가 유사 용도로 사용 중 |
| 4 | **역할 명칭 혼재** | 법인 역할(SYSTEM_ADMIN/ADMIN/STAFF/MANAGER/USER)과 시스템 역할(SUPER_ADMIN/ADMIN/MANAGER/MEMBER/VIEWER)이 별도 체계인데 UI에서 구분이 불분명 |
| 5 | **그룹 관리 분산** | `/settings/members` > Groups 탭과 `/settings/entities` > 법인 상세 > 그룹 관리 — 두 곳에서 그룹 CRUD 가능 |

## 3. TO-BE 요구사항

### 3.1 명칭 변경

| AS-IS | TO-BE | 적용 범위 |
|-------|-------|----------|
| `Admin Group` / `관리자 그룹` | **`Admin Level Group`** / **`관리자 등급 그룹`** | 권한관리 UI + i18n |
| `User Group` / `일반 사용자 그룹` | **`User Level Group`** / **`사용자 등급 그룹`** | 권한관리 UI + i18n |

> **주의**: DB 값 `ADMIN_GROUP` / `USER_GROUP`은 변경하지 않음 (하위 호환). 표시 명칭만 변경.

### 3.2 사용자 기본 정보 유지

| 필드 | DB 컬럼 | 비고 |
|------|---------|------|
| Name | `usr_name` | 편집 가능 (기 구현) |
| Email | `usr_email` | 고유값 |
| Role | `usr_role` | SUPER_ADMIN ~ VIEWER |
| Function | `usr_department` | **레거시 컬럼을 "Function"으로 재활용** |
| Job Title | `usr_job_title` | 편집 가능 (기 구현) |

### 3.3 조직관리 페이지 통합 (`/settings/entities`)

현재 법인 상세 내 서브탭(그룹/부서) — **유지하되 명확히 정리**:

| 탭 | 위치 | 기능 |
|----|------|------|
| **법인 목록** | 메인 영역 | 법인 CRUD (기 구현) |
| **부서 / Departments** | 법인 상세 서브탭 | 해당 법인 소속 부서 관리. 1인 1부서 원칙 |
| **그룹 / Groups** | 법인 상세 서브탭 | 해당 법인 소속 그룹 관리. 1인 다수 그룹 가능 (태그 개념) |

### 3.4 부서 정책

- 부서는 법인에 소속 (필수: `ent_id`)
- **1인 1부서 원칙**: 한 사용자는 하나의 부서에만 Primary 소속
- 부서 역할: `MEMBER`, `TEAM_LEAD`, `DEPARTMENT_HEAD`
- 부서 계층: Department(Level 1) → Team(Level 2) — 기 구현

### 3.5 그룹 정책

- 그룹은 법인에 소속 가능 (선택: `ent_id` nullable)
- **1인 다수 그룹**: 한 사용자가 여러 그룹에 속할 수 있음 (태그 개념)
- 그룹 예시: MS, BPO, DevOPS, 기술연구소
- 그룹용 권한: 메뉴별 그룹 접근 제한 (권한관리의 체크박스 — 기 구현)

### 3.6 기존 구조 유지 항목

- **Entities**: 법인 CRUD — 유지
- **Departments**: 부서 CRUD — 유지 (법인 상세 내)
- **Groups**: 그룹 CRUD — 유지 (법인 상세 내 + 사용자관리 탭)
- **권한관리**: 등급별/사용자별 권한 — 유지

### 4.2 Function 필드 (중간 리스크)

- `usr_department` 컬럼이 레거시 텍스트 형태로 "Function" 역할 가능
- MemberDetailPage에서 이미 `department` 필드로 표시 중
- **라벨만 "Function"으로 변경하면 됨** (또는 별도 `usr_function` 컬럼 추가)
- 기본 정보 섹션에서: Name, Email, Role, Function(=department 라벨), Job Title 순서로 표시

### 4.3 부서/그룹 관리 통합 (낮은 리스크)

- 이미 `/settings/entities` 법인 상세에 그룹/부서 서브탭 구현됨
- `/settings/departments` 독립 페이지 존재 → **중복 제거 또는 유지 결정 필요**
- `/settings/members` > Groups 탭 존재 → 전체 그룹 관리 (법인 무관)

### 4.4 1인 1부서 정책 (중간 리스크)

- 현재 `amb_user_dept_roles`는 다대다 관계 (1인 다수 부서 가능, `udr_is_primary`로 구분)
- **1인 1부서 원칙** 적용 시: 새 부서 할당 시 기존 부서 자동 해제 로직 필요
- 또는 Primary만 1개로 제한하고 기존 다대다 유지

## 5. 사용자 플로우

### 5.1 관리자가 사용자를 관리하는 플로우

```
[설정] → [사용자 관리] → 사용자 목록
  ├── 사용자 클릭 → 상세 페이지
  │     ├── 기본 정보: Name, Email, Role, Function, Job Title
  │     ├── 소속 법인: Entity Name, Code, Role — [추가] [삭제]
  │     ├── 소속 부서: Dept Name, Role, Entity — [추가] [삭제]  
  │     ├── 소속 그룹: Group Name, Entity — [추가] [삭제]
  │     └── HR 연결: Employee 정보
  └── Groups 탭 → 전체 그룹 CRUD
```

### 5.2 관리자가 조직을 관리하는 플로우

```
[설정] → [조직 관리] → 법인 목록
  └── 법인 상세 펼침
        ├── [그룹 관리 탭] → 해당 법인 그룹 CRUD
        └── [부서 관리 탭] → 해당 법인 부서 CRUD (트리 구조)
```

### 5.3 관리자가 권한을 관리하는 플로우

```
[설정] → [권한 관리]
  ├── 메뉴 설정 탭: 메뉴 ON/OFF
  ├── 등급별 권한 탭
  │     ├── [Admin Level Group] → SUPER_ADMIN, ADMIN (잠금)
  │     └── [User Level Group] → MANAGER, MEMBER, VIEWER (토글 + 그룹 체크박스)
  └── 사용자별 권한 탭: 개별 오버라이드
```

## 6. 기술 제약사항

1. **DB 값 변경 불가**: `usr_group_code`의 `ADMIN_GROUP` / `USER_GROUP` 값은 JWT 토큰, EntityGuard, DataScopeInterceptor 등 다수 백엔드 로직에서 사용. 표시 명칭만 변경.
2. **기존 라우트 유지**: `/settings/entities`, `/settings/members`, `/settings/permissions`, `/settings/departments` 경로 유지
3. **i18n 3개 언어**: ko, en, vi 모두 동시 업데이트 필요
4. **권한관리 RolePermissionTab**: 현재 `sortedGroups.length > 0`일 때만 그룹 컬럼 표시 — `useGroupList` 훅의 데이터 로딩 확인 필요 (403 이슈 가능)
