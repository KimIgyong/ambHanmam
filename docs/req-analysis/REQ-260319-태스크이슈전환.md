## 1. 배경 및 아이디어

### 1.1 현재 업무 흐름 (AS-IS)

```
[오늘의 미션] → [어제 미션 리뷰] → [태스크(Todo)]
              done → 완료 태스크 자동 생성
              incomplete + task → 오늘 태스크 자동 생성
              incomplete + mission → 다음날 미션으로 이월
```

- 미션은 **개인 업무 관리** 도구로, 매일 목표를 세우고 체크하는 구조
- 미션 리뷰를 통해 태스크(Todo)로 전환하면 **조직 내 공유** 가능 (GROUP/ENTITY visibility)
- 그러나 태스크에서 **다른 동료와의 협업/전달**이 필요한 경우, 이를 이슈로 격상시킬 수단이 없음

### 1.2 요구사항 (TO-BE)

```
[오늘의 미션] → [태스크(Todo)] → [이슈(Issue)]
                 ↑ 기존 흐름      ↑ 신규 기능
```

**태스크를 이슈로 전환** 기능을 통해:
- 개인 업무(태스크)를 **팀/조직 단위 추적이 필요한 이슈**로 격상
- 담당자 배정, 상태 추적(OPEN→APPROVED→IN_PROGRESS→RESOLVED→CLOSED), 심각도/우선순위 Part 관리 가능
- 이슈 전환 후 원본 태스크에는 연결된 이슈(`iss_id`)가 자동 링크

### 2.2 기존 연결 메커니즘

#### Todo → Issue 방향 (이미 존재)
- `amb_todos.iss_id` FK로 이슈 연결 가능
- TodoFormModal에서 프로젝트 선택 시 해당 프로젝트의 OPEN 이슈 드롭다운 제공
- TodoItem/TodoDetailModal에서 연결된 이슈 제목 표시 (`issueTitle`)
- **UpdateTodoRequest DTO에 `issue_id` 필드가 이미 존재** → 업데이트 시 이슈 연계 가능

#### Issue → Todo 방향 (존재하지 않음)
- IssueEntity에 `tdo_id` FK 없음
- 이슈에서 원본 태스크를 추적할 수 없음 (단방향 관계)

### 2.3 필드 매핑 가능성

| Todo 필드 | → | Issue 필드 | 매핑 방식 |
|-----------|---|-----------|----------|
| `tdo_title` | → | `iss_title` | 직접 복사 |
| `tdo_description` | → | `iss_description` | 직접 복사 (비어있으면 title 사용) |
| `tdo_start_date` | → | `iss_start_date` | 직접 복사 |
| `tdo_due_date` | → | `iss_due_date` | 직접 복사 |
| `tdo_tags` | → | `iss_affected_modules` | 자동 변환 불가 (수동 입력) |
| `tdo_visibility` | → | `iss_visibility` | PRIVATE→PRIVATE, GROUP→CELL, ENTITY→ENTITY |
| `tdo_cell_id` | → | `iss_cell_id` | 직접 복사 |
| `pjt_id` | → | `pjt_id` | 직접 복사 |
| `usr_id` (소유자) | → | `iss_reporter_id` | 태스크 소유자 = 이슈 보고자 |
| — | → | `iss_type` | 기본값 **TASK** (사용자 변경 가능) |
| — | → | `iss_severity` | 사용자 선택 필요 (필수값) |
| — | → | `iss_priority` | 기본값 3 |
| — | → | `iss_assignee_id` | 사용자 선택 |
| 참여자 목록 | → | 참여자 목록 | 선택적 복사 가능 |

## 4. 사용자 플로우

### 4.1 기본 전환 플로우

```
사용자: Tasks 페이지 > Mine 탭 > 특정 태스크 클릭
       ↓
시스템: TodoItem 액션 메뉴에 "이슈로 전환" 버튼 표시
       (태스크 소유자만 / 이미 이슈 연결 시 비활성)
       ↓
사용자: "이슈로 전환" 클릭
       ↓
시스템: IssueFormModal 팝업 (태스크 정보 프리필)
       - type: TASK (기본값)
       - title: 태스크 제목 (프리필, 수정 가능)
       - description: 태스크 설명 (프리필, 수정 가능)
       - severity: 선택 필요 (필수)
       - priority: 3 (기본값)
       - start_date: 태스크 시작일 (프리필)
       - due_date: 태스크 마감일 (프리필)
       - project_id: 태스크 프로젝트 (프리필)
       - visibility: 태스크 visibility 매핑 (프리필)
       - assignee_id: 비어있음 (사용자 선택)
       ↓
사용자: 추가 정보 입력 후 "생성" 클릭
       ↓
시스템: ① Issue 생성 API 호출
       ② Todo 업데이트 API 호출 (issue_id 연결)
       ③ 성공 토스트 표시
       ④ TodoItem에 이슈 링크 아이콘 표시 (기존 기능 활용)
```

### 4.2 엣지 케이스

| # | 케이스 | 처리 방안 |
|---|--------|----------|
| E1 | 이미 이슈가 연결된 태스크 | "이슈로 전환" 버튼 비활성 + 툴팁 "이미 연결된 이슈가 있습니다" |
| E2 | 완료된(COMPLETED) 태스크 | 전환 허용 (완료 후에도 이슈화 필요할 수 있음) |
| E3 | 이슈 생성 성공 후 Todo 업데이트 실패 | 이슈는 생성되었으므로 사용자에게 알림, 수동 연결 안내 |
| E4 | 다른 사용자의 태스크 (Scoped View) | 현재 ScopedTodoItem은 읽기 전용 → 전환 불가 (소유자만 가능) |
| E5 | Todo description이 비어있음 | Issue description에 title을 복사하여 필수값 충족 |
| E6 | 권한: MASTER가 타인의 태스크 전환 | MASTER는 TodoItem 렌더링 → 전환 버튼 표시 가능, reporter_id는 태스크 소유자로 설정 |

## 5. 사이드 이팩트 분석

### 5.1 영향도 평가

| # | 영역 | 사이드 이팩트 | 심각도 | 대응 |
|---|------|-------------|--------|------|
| S1 | **KMS WorkItem** | 이슈 생성 시 `module.data.created` 이벤트 자동 발행 → KMS WorkItem에 이슈 등록됨. 원본 태스크 WorkItem과 **중복 등록** 가능 | 🟡 중간 | 허용 (태스크와 이슈는 별개 모듈 추적 대상) |
| S2 | **알림 시스템** | 이슈 생성 시 assignee에게 알림 발송 (기존 로직) → 의도된 동작이므로 문제 없음 | 🟢 낮음 | 기존 로직 유지 |
| S3 | **Todo-Issue 양방향 표시** | Todo에서 `issueTitle` 표시 (기존), Issue에서의 원본 Todo 표시 없음 → UX 일관성 이슈 | 🟡 중간 | Phase 1에서는 단방향만, Phase 2에서 역방향 링크 고려 |
| S4 | **칸반/간트 뷰** | 새 이슈가 칸반의 OPEN 컬럼에 자동 추가됨 → 정상 동작 | 🟢 낮음 | 영향 없음 |
| S5 | **이슈 필터 기본값** | 기본 필터에서 OPEN 포함 → 새 이슈 즉시 목록에 노출 → 정상 | 🟢 낮음 | 영향 없음 |
| S6 | **태스크 상태 변경** | 전환 후 태스크 상태를 어떻게 할 것인가? → 유지(변경 없음)가 적절. 이슈는 별디 추적 단위 | 🟡 중간 | 태스크 상태 변경 없이 iss_id만 연결 |
| S7 | **반복 태스크** | 반복 태스크에서 전환 시, 이슈는 1회성. 다음 반복 인스턴스는 이슈 미연결 | 🟢 낮음 | 자연스러운 동작 |
| S8 | **태스크 참여자 → 이슈 참여자** | 태스크 참여자를 이슈 참여자로 자동 복사할지 결정 필요 | 🟡 중간 | 폼에서 사전 선택 상태로 표시, 사용자가 수정 가능 |
| S9 | **AI 분석** | 이슈에 AI 분석 자동 트리거 없음 (사용자가 수동으로 요청) → 문제 없음 | 🟢 낮음 | 영향 없음 |
| S10 | **TodoFormModal의 이슈 드롭다운** | 태스크 편집 시 연결 이슈 드롭다운에 방금 생성한 이슈가 나타남 → 정상 동작 | 🟢 낮음 | 영향 없음 |

### 5.2 데이터 정합성

| 시나리오 | 분석 |
|---------|------|
| 이슈 삭제 시 Todo의 iss_id | IssueEntity 소프트 삭제(`iss_deleted_at`) → Todo의 `iss_id` FK는 남아있으나, JOIN 시 deleted 필터로 자연 처리. **issueTitle이 null로 표시됨** → 허용 가능 |
| Todo 삭제 시 연결 이슈 | Todo 삭제해도 Issue는 독립 엔티티로 유지 → 문제 없음 |
| 동일 Todo에서 중복 전환 시도 | iss_id 존재 여부로 버튼 비활성 처리 → 중복 생성 방지 |

## 6. 구현 가능성 검토

### 6.1 기존 인프라 활용도

| 항목 | 활용 가능 여부 | 설명 |
|------|--------------|------|
| Todo `iss_id` FK | ✅ 이미 존재 | 이슈 연결에 바로 사용 가능 |
| UpdateTodoRequest `issue_id` | ✅ 이미 존재 | PATCH API로 이슈 연결 가능 |
| IssueFormModal `defaultProjectId` | ✅ 확장 가능 | 초기값 전달 패턴이 이미 있음 |
| TodoItem 액션 버튼 | ✅ 확장 가능 | edit/delete 옆에 전환 버튼 추가 |
| 이슈 타입 `TASK` | ✅ 이미 존재 | 태스크에서 전환된 이슈의 기본 타입으로 적합 |
| KMS 이벤트 시스템 | ✅ 자동 동작 | 이슈 생성 시 자동으로 WorkItem 등록 |
| 알림 시스템 | ✅ 자동 동작 | 이슈 담당자 배정 시 자동 알림 |

### 6.2 구현 방안 비교

#### 방안 A: 프론트엔드 2-Step (권장)
```
프론트엔드:
  1. IssueFormModal 열기 (Todo 데이터 프리필)
  2. 사용자가 추가 정보 입력 후 이슈 생성 API 호출
  3. 성공 시 Todo 업데이트 API 호출 (issue_id 세팅)
```

**장점**: 백엔드 변경 최소화, 기존 API 100% 재사용
**단점**: 프론트엔드에서 2번의 API 호출, 부분 실패 가능성

#### 방안 B: 백엔드 전용 API
```
POST /todos/:id/convert-to-issue
  Body: { type, severity, priority, assignee_id, ... }
  
백엔드:
  1. Todo 조회 + 검증
  2. Issue 생성 (트랜잭션)
  3. Todo.iss_id 업데이트 (같은 트랜잭션)
  4. 응답: 생성된 Issue 반환
```

**장점**: 원자적 트랜잭션, 데이터 정합성 보장
**단점**: 새 API 엔드포인트 + DTO + 서비스 로직 필요

#### 방안 C: 하이브리드 (방안 A + 에러 처리 강화)
```
프론트엔드:
  1. IssueFormModal로 이슈 생성
  2. 성공 시 Todo 업데이트 (issue_id 세팅)
  3. Todo 업데이트 실패 시 → retry 또는 사용자 안내 (이슈는 이미 생성됨)
```

### 6.3 권장 방안: **방안 A (프론트엔드 2-Step)**

**근거**:
- 기존 `POST /issues` + `PATCH /todos/:id` API를 **그대로** 사용
- 백엔드 변경 **0건** (순수 프론트엔드 작업)
- IssueFormModal에 `initialData` prop만 추가하면 됨
- Todo 업데이트 실패는 극히 드물고, 실패해도 이슈는 정상 생성되어 치명적이지 않음
- 향후 방안 B로 전환 용이 (API 추가만 하면 됨)

## 10. 결론

### 구현 가능성: ✅ 높음

- 기존 데이터 모델에 `Todo.iss_id` FK가 이미 존재하여 추가 스키마 변경 불필요
- 기존 `POST /issues` (이슈 생성) + `PATCH /todos/:id` (이슈 연결) API 활용으로 백엔드 변경 0건
- IssueFormModal의 초기값 전달 패턴(`editingIssue`, `defaultProjectId`)이 이미 구현되어 확장 용이
- Issue 타입에 `TASK`가 이미 정의되어 있어 태스크 출처 이슈를 자연스럽게 분류 가능
- KMS WorkItem, 알림 시스템 등 사이드 이팩트가 모두 **의도된 방향으로 자동 동작**

### 주요 사이드 이팩트: 🟢 관리 가능

- KMS WorkItem 중복 등록 (태스크+이슈 각각): 허용 가능, 별개 추적 단위
- 프론트엔드 2-Step의 부분 실패: 이슈 생성 자체는 성공하므로 치명적이지 않음
- 역방향 링크 부재: Phase 2에서 대응 가능

### 권장 구현 순서

1. IssueFormModal에 `initialData` prop 추가
2. TodoItem에 "이슈로 전환" 버튼 추가
3. TodoPage에 전환 로직(이슈 생성 → Todo 업데이트) 구현
4. i18n 번역 키 추가
5. 테스트 및 배포
