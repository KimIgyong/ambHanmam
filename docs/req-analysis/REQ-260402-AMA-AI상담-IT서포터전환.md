## 1. 요구사항 요약

현재 우측 상단 AI Assistant를 AMA AI 상담 채널로 전환하고, 초기 화면의 Today's Tasks / My Issues 영역을 제거한다.

상담 에이전트는 IT 서포터 역할을 수행해야 하며 다음을 담당한다.
1. AMA 버그 신고 접수
2. 기능 개선 요청 접수
3. 신규 기능 요청 접수
4. AMA 기능 설명 및 사용 팁 제공

추가 요구사항으로 ADMIN_LEVEL 관리자는 Entity 유저와 AI Assistant의 대화 전체를 관제할 수 있어야 한다.
1. 모든 Entity 대화를 하나의 타임라인 창에서 통합 조회
2. 각 Entity별 대화창 개별 조회
3. 관리자 직접 대화 참여
4. 선택한 대화를 이슈 또는 메모(노트)로 등록
5. 이슈 등록 시 대상 Entity 및 해당 Entity의 프로젝트 선택
6. `admin@amoeba.group` (SUPER_ADMIN)은 HQ/VN01 관점에서 특히 VN01 Entity로 이슈 등록 가능해야 함

접수 후 처리 원칙:
1. 버그 신고: 빠른 대응 인사 + 신속 조치 안내 + HQ 계정 기준 시스템 버그 리포트 작성
2. 기능개선/신규기능: 감사 인사 + 개발팀 검토 및 일정 안내 문구 + HQ 계정 기준 이슈 등록
3. 기록: 요청자 Entity Code / Entity Name / Email 등 요청자 정보를 남겨 추적 가능해야 함

즉, 본 요구사항은 단순 UI 변경이 아니라 상담 응답 정책, 자동 접수, 관리자 관제, 수동 개입, 업무 전환까지 포함하는 기능개선이다.

## 2. AS-IS 현황 분석

### 2.1 프론트엔드 Assistant UI

현재 구조:
1. 우측 상단 버튼 클릭 시 Assistant 모달 오픈
2. Welcome 화면에서 `오늘의 할일`, `나의 이슈`, `에이전트 선택` 노출
3. 에이전트 선택 후 채팅 화면 진입

확인 결과:
1. 모달 타이틀은 locale `assistant.title` 기반이며 기본값은 AI Assistant
2. Welcome 화면에 Today Todo / My Issues 데이터 조회 및 목록 렌더링 로직이 존재
3. 해당 목록은 `/todos`, `/issues` 이동 버튼까지 제공

현 구조는 사용자 요청의 `Today's Tasks / My Issues 삭제` 요구와 충돌한다.

### 2.2 에이전트 아키텍처

현재 백엔드 채팅 흐름:
1. `/conversations` 생성 시 `unit_code` 저장
2. `/conversations/:id/messages` 호출 시 `ChatService`가 `AgentFactoryService.getAgent(unit)`로 에이전트 선택
3. 에이전트별 시스템 프롬프트 기반 답변 생성

현재 상태:
1. LEGAL / ACCOUNTING / TRANSLATION / PM / DEVELOPMENT에 더해 IT 에이전트 코드 경로가 추가됨
2. IT 에이전트는 `docs/plan`, `docs/implementation`, `docs/analysis`의 최근 문서를 읽어 프롬프트 컨텍스트에 반영함

즉, 기능 설명 기반의 IT 상담 에이전트 토대는 마련되었으나, 관리자 관제/개입 기능은 아직 별도 구현이 필요하다.

### 2.3 Agent Config 기능의 한계

`amb_agent_configs` 테이블과 설정 API는 존재한다.

현재 한계:
1. 관리자 설정은 프롬프트/활성화 관리 중심
2. 관리자 대화 관제, 관리자 참여, 전환 액션과는 직접 연결되지 않음

따라서 운영 설정은 가능하지만, 대화 처리 워크플로우까지는 미확장 상태다.

### 2.4 접수 데이터 등록(버그/기능요청)

현재 등록 가능 자산:
1. 일반 이슈 테이블 `amb_issues`
2. 이슈 코멘트 `amb_issue_comments`
3. 노트/미팅노트 모듈 존재
4. 고객 포털 이슈 등록 API(`client/issues`)는 별도 존재

현재 갭:
1. `시스템 버그 리포트` 전용 엔티티/모듈 없음
2. 요청자(Entity Code/Name/Email)를 별도 구조화 필드로 저장하는 스키마 없음
3. 상담 대화에서 자동으로 이슈를 생성하는 액션 파이프라인 없음

### 2.5 관리자 대화 조회 기능 (AS-IS)

현재 이미 관리자 대화 관리 화면이 존재한다.

확인 결과:
1. `/admin/conversations` 화면 존재
2. 관리자 API `/conversations/admin`, `/conversations/admin/:id` 존재
3. 현재 화면은 `대화 목록 테이블 -> 상세 모달` 구조
4. 상세 모달에서는 전체 메시지 조회 가능
5. 현재 가능한 후처리는 `convertToKnowledge` 한 가지뿐

현재 가능한 범위:
1. 관리자 목록 조회
2. 대화 상세 열람
3. 대화를 지식(work item)으로 변환

현재 불가능한 범위:
1. 모든 Entity 대화를 메시지 단위 타임라인으로 통합 조회
2. Entity별 대화창 구분/탭 구조
3. 관리자 메시지 참여
4. 선택 메시지/대화를 이슈 또는 노트로 직접 등록
5. 등록 시 Entity / Project 선택 UX

### 2.6 관리자 관점 Entity 지정 가능성

현재 프론트엔드는 `currentEntity`를 `X-Entity-Id` 헤더로 API에 전송한다.

백엔드 `EntityGuard` 동작:
1. `SUPER_ADMIN`, `ADMIN`은 EntityGuard를 바이패스
2. 단, 헤더에 `X-Entity-Id`가 있으면 `request.entityId`에 주입

의미:
1. ADMIN_LEVEL/SUPER_ADMIN 사용자는 원하는 Entity 컨텍스트로 API 호출 가능
2. 이슈 생성 시 현재 선택한 Entity 기준으로 `ent_id`를 지정할 수 있음

따라서 `admin@amoeba.group`가 VN01을 현재 Entity로 선택한 상태라면, 이슈를 VN01 Entity로 등록하는 구조는 현재 아키텍처상 구현 가능하다.

### 2.7 기능 설명/사용팁 제공 기반

현재 IT 에이전트는 프로젝트 산출 문서를 기반으로 컨텍스트를 구성하도록 구현되었다.

하지만 현재 한계:
1. 문서 헤더 요약 기반이며, 세부 섹션 검색/RAG는 아님
2. 최신 문서 중심이므로 누적 지식 검색 정밀도는 제한적

즉, 1차 기능 설명에는 활용 가능하지만 장기적으로는 문서 인덱싱 고도화가 필요하다.

## 5. 케이스 조사 (더 좋은 방식 검토)

### Case A. 최소 변경 방식

구성:
1. Assistant 명칭/화면 텍스트만 변경
2. 상담 응답만 개선
3. 버그/기능 요청은 사용자가 수동 등록

장점:
1. 구현 빠름
2. 리스크 낮음

단점:
1. 자동 접수/기록 미충족
2. 관리자 관제 요구 미충족

적합도: 낮음

### Case B. 상담 접수 강화 방식

구성:
1. IT 에이전트 상담 유지
2. BUG/IMPROVEMENT/NEW_FEATURE/HELP intent 분류
3. 분류 결과에 따라 이슈 자동 생성
4. 요청자 메타데이터를 이슈 description 구조화 블록으로 저장

장점:
1. 상담 요구사항 대부분 충족
2. 기존 `amb_issues` 재사용 가능

단점:
1. 관리자 관제/참여 요구는 별도 구현 필요

적합도: 높음

### Case C. 전용 접수 모듈 분리 방식

구성:
1. `amb_support_requests` 전용 테이블 신설
2. 요청유형/요청자 스냅샷/처리 SLA/연결 이슈를 구조화 저장
3. IT 서포터 대시보드 별도 구축

장점:
1. 장기 확장성 최고
2. 접수/운영 지표 체계화 가능

단점:
1. 개발 범위 큼
2. 배포/마이그레이션 비용 큼

적합도: 중장기

### Case D. 관리자 관제 콘솔 확장 방식 (권장)

구성:
1. 기존 `/admin/conversations`를 기반으로 확장
2. 좌측: Entity/사용자/대화방 목록
3. 중앙: 메시지 타임라인
4. 우측: 전환 액션(이슈 등록/노트 등록/관리자 메시지)

장점:
1. 기존 관리자 대화 화면/백엔드 재사용 가능
2. 통합 타임라인과 개별 대화방을 한 화면 안에서 함께 제공 가능
3. 운영자가 바로 사용할 수 있는 구조

단점:
1. UI 복잡도 증가

적합도: 높음

### 권장 결론

현 단계에서는 다음 2트랙 접근이 최적이다.
1. 상담 접수 기능은 Case B 방식으로 구현
2. 관리자 관제는 Case D 방식으로 `/admin/conversations` 확장

장기적으로 운영 데이터가 쌓이면 Case C(전용 support request)로 확장한다.

## 6. 사용자 플로우

1. 사용자가 우측 상단 AMA AI 상담 오픈
2. IT 서포터 상담 시작
3. 메시지 입력
4. 시스템이 요청 의도 분류
5. 의도별 처리
   - BUG: 공감/신속조치 응답 + 정책 기준 BUG 이슈 생성 + ref 전달
   - IMPROVEMENT/NEW_FEATURE: 감사/검토안내 응답 + 정책 기준 이슈 생성 + ref 전달
   - HELP: 기능 설명 + 사용팁 제공
6. 모든 접수 건에 요청자 Entity 정보/Email 스냅샷 포함
7. 관리자는 `/admin/conversations`에서 전체 타임라인 확인
8. 특정 Entity/사용자 대화방 진입
9. 필요 시 관리자 메시지 작성
10. 대화 일부를 이슈 또는 노트로 전환
11. 이슈 전환 시 Entity와 Project를 선택하고 저장

## 7. 기술 제약사항

1. 이슈 생성 API는 reporter를 호출 사용자로 기록하므로, HQ 기준 작성 정책의 구현 방식 결정 필요
2. 스키마 변경이 필요한 경우 스테이징/프로덕션은 수동 SQL 마이그레이션 필수
3. USER_LEVEL 데이터 격리(ent_id) 규칙 준수 필요
4. 현재 AdminConversation DTO에는 Entity Code/Name 정보가 포함되어 있지 않음
5. 현재 Message role은 사실상 `user`/`assistant` 중심 렌더링 구조이며 `admin` 역할 확장 필요

## 8. 적용 가능성 결론

적용 가능성: 가능 (조건부)

필수 선행 조건:
1. 상담 intent 분류 + 이슈 자동 등록 서비스 추가
2. 요청자 스냅샷 저장 방식(이슈 필드 확장 or description 구조화) 확정
3. HQ 계정 기반 등록 정책(대리 등록 전략) 설계
4. 관리자 메시지/타임라인/전환 API 설계
5. 관리자 화면에서 Entity/Project 선택 UX 설계

권장 구현 전략:
1. 1차: 기존 이슈 시스템 재사용 + 구조화 description 템플릿
