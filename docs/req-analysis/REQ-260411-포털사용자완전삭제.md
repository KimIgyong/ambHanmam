# REQ-260411 포털 사용자 완전삭제(Hard Delete) 기능

> 작성일: 2026-04-11

---

## 1. AS-IS 현황 분석

### 1.1 현재 삭제 메커니즘

포털 고객(Portal Customer) 삭제는 **소프트 삭제(Soft Delete)만 지원**한다.

| 항목 | 현재 동작 |
|------|---------|
| **삭제 방식** | `pct_deleted_at` 타임스탬프 설정 (TypeORM `@DeleteDateColumn`) |
| **매핑 처리** | 활성 매핑(`ACTIVE`) → `REVOKED` 상태 변경 + 취소자/시각 기록 |
| **결제 데이터** | 보존 (변경 없음) |
| **고객사 데이터** | 보존 (변경 없음) |
| **이메일 재사용** | ❌ 불가 — `pct_email` UNIQUE 제약조건이 soft-deleted 레코드에도 적용됨 |

### 1.2 관련 테이블 의존성

```
amb_svc_portal_customers (PK: pct_id)
├── amb_portal_user_mappings       (FK: pct_id → pct_id, 1:N)
├── amb_svc_portal_payments        (FK: ppm_customer_id → pct_id, 1:N)
└── amb_svc_clients                (FK: pct_cli_id ↔ cli_id, N:1, nullable)
    ├── amb_svc_subscriptions      (FK: cli_id → cli_id, 1:N)
    │   └── amb_svc_usage_records  (FK: sub_id → sub_id, 1:N)
    └── (기타 서비스 관련 테이블)
```

### 1.3 테이블별 삭제 특성

| 테이블 | Soft Delete | 비고 |
|--------|:-----------:|------|
| `amb_svc_portal_customers` | ✅ `pct_deleted_at` | 읽기전용 엔티티로 api에서 접근 |
| `amb_portal_user_mappings` | ❌ | `pum_status` REVOKED 상태 관리, 물리 삭제 가능 |
| `amb_svc_portal_payments` | ❌ | 결제 이력 감사 추적용, 물리 삭제 가능 |
| `amb_svc_clients` | ✅ `cli_deleted_at` | 고객사 마스터 데이터 |
| `amb_svc_subscriptions` | ✅ `sub_deleted_at` | 구독 정보, 빌링 연동 |
| `amb_svc_usage_records` | ❌ | 사용량 기록, 빌링 감사 추적용 |

### 1.4 현재 관련 코드

| 파일 | 역할 |
|------|------|
| `apps/api/.../portal-bridge.controller.ts` | `DELETE /portal-bridge/customers/:pctId` — @AdminOnly |
| `apps/api/.../portal-bridge.service.ts` | `deletePortalCustomer()` — 소프트 삭제 + 매핑 REVOKE |
| `apps/web/.../PortalCustomersTab.tsx` | 관리자 UI — Trash2 아이콘, `window.confirm()` 확인 |
| `apps/web/.../usePortalBridge.ts` | `useDeletePortalCustomer()` React Query mutation |
| `apps/web/.../portal-bridge.service.ts` | `deleteCustomer()` — API 호출 |

### 1.5 기존 Hard Delete 패턴 (참고)

`issue.service.ts`의 `permanentDeleteIssue()` 메서드에서 다음 패턴이 검증되어 있다:

```typescript
await this.issueRepository.manager.transaction(async (manager) => {
  // 1. 하위 관계 엔티티 순차 삭제 (역순)
  await manager.delete(IssueCommentReactionEntity, { iscId: In(commentIds) });
  await manager.delete(IssueCommentEntity, { issId: id });
  await manager.delete(IssueStatusLogEntity, { issId: id });
  await manager.delete(IssueParticipantEntity, { issId: id });
  await manager.delete(IssueRatingEntity, { issId: id });
  // 2. 본체 hard delete
  await manager.remove(entity);
});
```

## 2. TO-BE 요구사항

### 2.1 기능 목표

포털 고객(Portal Customer)을 **완전히 삭제(Hard Delete)** 하여:
1. DB에서 레코드를 물리적으로 제거
2. 해당 이메일로 재가입 가능하도록 UNIQUE 제약 해제
3. 관련 데이터를 안전하게 정리

### 2.2 삭제 레벨 정의

#### Level 1: 사용자 삭제 (기본)

고객 계정과 직접 연관 데이터만 삭제한다.

| 대상 테이블 | 삭제 방식 | 조건 |
|------------|---------|------|
| `amb_portal_user_mappings` | Hard Delete | `WHERE pct_id = :pctId` |
| `amb_svc_portal_payments` | Hard Delete | `WHERE ppm_customer_id = :pctId` |
| `amb_svc_portal_customers` | Hard Delete | `WHERE pct_id = :pctId` |

- `amb_svc_clients` 레코드는 **보존** (다른 시스템에서 참조 가능)
- `pct_cli_id` FK만 끊어짐 (고객 레코드 자체가 삭제되므로)

#### Level 2: 전체 데이터 삭제 (확장)

고객 계정 + 자동 프로비저닝으로 생성된 모든 데이터를 삭제한다.

| 대상 테이블 | 삭제 방식 | 조건 |
|------------|---------|------|
| `amb_svc_usage_records` | Hard Delete | `WHERE sub_id IN (해당 고객사 구독 ID)` |
| `amb_svc_subscriptions` | Hard Delete | `WHERE cli_id = :cliId` |
| `amb_svc_clients` | Hard Delete | `WHERE cli_id = :cliId` |
| `amb_portal_user_mappings` | Hard Delete | `WHERE pct_id = :pctId` |
| `amb_svc_portal_payments` | Hard Delete | `WHERE ppm_customer_id = :pctId` |
| `amb_svc_portal_customers` | Hard Delete | `WHERE pct_id = :pctId` |

> ⚠️ Level 2는 활성 구독(ACTIVE subscription)이 있는 경우 **차단**해야 한다.

### 2.3 안전장치

| 안전장치 | 설명 |
|---------|------|
| **권한** | `@AdminOnly()` — ADMIN_LEVEL만 실행 가능 |
| **삭제 미리보기** | 삭제 전 영향받는 데이터 수량 조회 API 제공 |
| **2단계 확인** | 미리보기 모달 → 이메일 입력 확인 (타이핑 일치 검증) |
| **트랜잭션** | 모든 삭제를 단일 트랜잭션으로 처리 (부분 삭제 방지) |
| **활성 구독 차단** | Level 2에서 ACTIVE 구독이 있으면 삭제 거부 |
| **감사 로그** | 삭제 실행 내역을 서버 로그에 기록 (누가, 언제, 무엇을) |

| 항목 | AS-IS | TO-BE | 갭 |
|------|-------|-------|-----|
| 삭제 방식 | Soft Delete만 지원 | Soft + Hard Delete 선택 | 신규 API + 서비스 메서드 필요 |

## 5. 기술 제약사항

### 5.1 데이터베이스

- **FK 제약조건**: TypeORM 엔티티에 DB-level FK constraint가 설정되어 있으므로, 삭제 순서가 중요 (자식 → 부모 순)
- **portal-api ↔ api 분리**: `amb_svc_portal_customers`와 `amb_svc_portal_payments`는 portal-api 소속이지만, `amb_portal_user_mappings`와 `amb_svc_clients`는 api 소속
  - Hard delete는 **api(portal-bridge)**에서 통합 처리해야 함 (portal-api의 테이블도 api에서 읽기전용 엔티티로 접근 가능)
- **UNIQUE 인덱스**: `pct_email`의 UNIQUE 제약은 soft-deleted 레코드에도 적용됨. Hard delete만이 이메일 재사용을 가능하게 함

### 5.2 보안

- ADMIN_LEVEL 전용 — `@AdminOnly()` 데코레이터 필수
- 이메일 확인 검증은 **서버 사이드**에서도 수행 (프론트엔드 검증만으로는 불충분)
- 삭제 로그는 최소 NestJS Logger를 통해 기록 (향후 감사 테이블 확장 가능)

### 5.3 성능

- 대량 데이터(수백 건 결제/사용기록)도 단일 트랜잭션 내에서 처리 가능 (PostgreSQL 자체 성능으로 충분)
- `IN` 절 사용 시 ID 수량 상한은 불필요 (현실적으로 수천 건 미만)

### 5.4 호환성

- 기존 소프트 삭제 API는 **그대로 유지** (하위 호환)
- 신규 완전 삭제는 별도 엔드포인트 (`DELETE /permanent/:pctId`)로 분리
- 프론트엔드 드롭다운 메뉴로 두 삭제 방식을 명확히 구분
