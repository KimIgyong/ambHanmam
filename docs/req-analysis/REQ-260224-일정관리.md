# Schedule Management — Requirements Analysis (일정관리 요구사항 분석서)

> **Module**: AMB Management > Work Schedule (일정관리)
> **Version**: 1.1.0 | **Status**: Review
> **Related Policies**: amb-access-control-policy.md (ACL-010~012), amb-kms-folksonomy-design.md

---

## 1. Overview (개요)

### 1.1 Purpose (목적)

AMB Management 내 임직원이 개인 일정, 팀 일정, 프로젝트 일정을 통합 관리하고, AI가 일정 생성·최적화·충돌 감지를 능동적으로 지원하는 **Born to be AI 기반 일정관리 모듈**을 구축한다.

**핵심 철학**: Human sets WHAT(목표/기한) → AI builds HOW(세부 일정/WBS) → Human confirms(확인/조정)

### 1.2 Background (배경)

| 현재 상황 | 개선 방향 |
|---------|---------|
| `amb_work_schedules`는 1인 1일 단순 출근 기록용으로만 사용 | 풍부한 일정 메타데이터(참여자, 반복, 카테고리, 공개범위) 지원 |
| AI 지원 없음 — 수동 생성/관리 | AI가 초안 생성, 충돌 감지, 주간 최적화 수행 |
| Google Calendar 연동 없음 | 내부 → Google 단방향 동기화 (MVP) |
| 알림 기능 없음 | 멀티채널 리마인더 (아메바톡 + 이메일) |

### 1.3 Scope (범위)

**In Scope**
- 일정 CRUD (개인/팀/프로젝트 구분)
- 반복 일정 (RFC 5545 기반 내부 enum)
- 반복 예외 처리 MVP (단건 취소 / 단건 시간 변경)
- 참여자 초대 및 응답 관리
- 알림/리마인더 (아메바톡 + 이메일)
- 캘린더 뷰 (월/주/일) + Gantt 뷰
- Google Calendar 단방향 동기화 (내부 → Google)
- AI 일정 생성·충돌 감지·주간 최적화
- 공개범위 관리 (ACL 정책 준수)
- KMS Todo 양방향 자동 동기화

**Out of Scope (Phase 2 이후 검토)**
- Google Calendar 양방향 동기화 (Google 변경사항 → 내부 반영)
- Google Calendar Webhook 기반 실시간 동기화
- 바쁨/한가 정보만 노출 옵션 (공개범위와 분리된 상세정보 제어)
- 타임존 계정별 설정 변경 (MVP: 서버 UTC 기준 브라우저 로컬타임 표시)
- 리소스(회의실/장비) 예약 시스템 (별도 프로젝트)
- 일정 시작/종료 트리거 기반 외부 워크플로 자동화 (KMS Todo 연동은 In Scope)
- 모바일 푸시 알림

### 4.2 Recurrence Management (반복 일정 관리)

| FR-SCH-014 | **Exception: change this and all following** — split strategy: end parent rule at current date, create new rule from next occurrence (이번 이후 모두 변경 — 스플릿 전략) | P1 | AC-SCH-2 |

> **Enum 정의**
>
> `scr_freq` (내부 enum, RFC 5545 FREQ와 별도 관리 — Google 연동 변환은 infra layer 처리):
> `DAILY` | `WEEKLY` | `MONTHLY` | `QUARTERLY` | `SEMIANNUAL` | `YEARLY`
>
> `sch_recurrence_type`:
> 반복 설정 유무를 나타내는 플래그 — `NONE` | `RECURRING`
> 반복 세부 설정은 `amb_schedule_recurrences` 테이블 참조

### 4.6 Google Calendar Integration (Google Calendar 연동)

> **MVP 정책: 내부 → Google 단방향 동기화만 지원. Google 변경사항 → 내부 반영은 Phase 2.**

| ID | Requirement (요구사항) | Priority | AC Ref |
|----|----------------------|----------|--------|
| FR-SCH-050 | User can connect their Google account via OAuth 2.0 to enable Google Calendar sync (Google 계정 연결) | P0 | AC-SCH-5 |
| FR-SCH-051 | On schedule create/update: push event to connected Google Calendar (google_event_id 저장) (내부 → Google 동기화) | P0 | AC-SCH-5 |
| FR-SCH-052 | On internal schedule delete: delete corresponding Google Calendar event (내부 삭제 → Google 이벤트 삭제) | P0 | AC-SCH-5 |
| FR-SCH-053 | Sync status tracking per schedule: `NOT_SYNCED` / `SYNCED` / `SYNC_FAILED` / `DISCONNECTED` (동기화 상태 관리) | P0 | AC-SCH-5 |
| FR-SCH-054 | Retry failed sync up to 3 times with exponential backoff; after 3rd failure set status to `SYNC_FAILED` and alert user (동기화 실패 재시도 정책) | P1 | AC-SCH-5 |

> **Phase 2 (Out of Scope)**: Google Webhook(Push Notification) 기반 Google → 내부 반영. Google 쪽 삭제 시 내부 일정 `sync_status = EXTERNAL_DELETED` 처리.

## 5. Data Model (데이터 모델)

```
amb_users ──< amb_schedules ──< amb_schedule_recurrences (1:1 optional)
```

| sch_updated_at | timestamptz | auto NOW() | 낙관적 락 기준 컬럼 |
| sch_created_at | timestamptz | auto NOW() | 생성일 |
| sch_deleted_at | timestamptz | nullable | Soft delete |

### 5.3 amb_schedule_recurrences (반복 규칙)

| Column | Type | Constraint | Description |
|--------|------|-----------|-------------|
| scr_id | uuid PK | gen_random_uuid() | 반복 규칙 ID |
| sch_id | uuid FK | UNIQUE → amb_schedules | 부모 일정 (1:1) |
| scr_freq | varchar(20) | NOT NULL | 내부 enum: `DAILY`\|`WEEKLY`\|`MONTHLY`\|`QUARTERLY`\|`SEMIANNUAL`\|`YEARLY` |
| scr_interval | int | default 1 | N주마다, N개월마다 |

| scr_month_day | int | nullable | 매월 N일 |

## 6. API Design (API 설계)

### 6.1 Endpoints

```
Base: /api/v1/schedules

[일정 기본 CRUD]
POST   /api/v1/schedules                          # 일정 생성
GET    /api/v1/schedules                          # 일정 목록 조회 (query params)
GET    /api/v1/schedules/:sch_id                  # 일정 상세
PATCH  /api/v1/schedules/:sch_id                  # 일정 수정 (낙관적 락)
DELETE /api/v1/schedules/:sch_id                  # 일정 삭제 (soft delete)

[반복 예외]
POST   /api/v1/schedules/:sch_id/exceptions       # 반복 예외 생성
GET    /api/v1/schedules/:sch_id/exceptions       # 반복 예외 목록

[참여자]
POST   /api/v1/schedules/:sch_id/participants     # 참여자 추가
PATCH  /api/v1/schedules/:sch_id/participants/me  # 내 응답 상태 변경
DELETE /api/v1/schedules/:sch_id/participants/:scp_id  # 참여자 제거

[Google Calendar 연동]
POST   /api/v1/schedules/google/connect           # Google 계정 연결
DELETE /api/v1/schedules/google/disconnect        # Google 계정 연결 해제
POST   /api/v1/schedules/:sch_id/google/sync      # 수동 동기화 트리거

[AI 기능]
POST   /api/v1/schedules/ai/generate-today        # AI 오늘 할 일 생성 (SSE)
POST   /api/v1/schedules/ai/generate-wbs          # AI WBS 초안 생성 (SSE)
POST   /api/v1/schedules/ai/optimize-week         # AI 주간 최적화 제안 (SSE)
```

## 8. Visibility Access Rules Summary (공개범위 접근 규칙 요약)

아래 표는 `amb-access-control-policy.md` ACL-010~012를 일정 도메인에 적용한 기준입니다.

| 공개범위 | 조회 가능 대상 | 통합검색 노출 | 매니저 접근 |
|---------|-------------|------------|-----------|
| PRIVATE | 소유자 + 명시 초대 참여자 | ❌ (참여자 외 제외) | ✅ (ACL-012 계층적 접근) |
| SHARED | 소유자 + 명시 공유 대상 | ❌ (공유 대상 외 제외) | ✅ |
| DEPARTMENT | 소유자 부서 전체 (하위 포함) | ✅ (부서 내) | ✅ |
| ENTITY | 동일 법인 전체 | ✅ (법인 내) | ✅ |

> **PRIVATE 일정의 제목/시간**: 비참여자에게는 조회·검색 모두 불가. "일정 있음" 여부조차 노출하지 않음.
> **바쁨/한가 정보만 노출 옵션**: Out of Scope (Phase 2 검토)

## 9. Sequence Diagrams (주요 시퀀스)

### 9.1 일정 생성 + 초대 알림 흐름

```
User → POST /schedules (participants 포함)
  → ScheduleService.create()
    → amb_schedules INSERT
    → amb_schedule_participants INSERT × N (status: NONE)
    → amb_schedule_notifications INSERT
    → NotificationService.sendInvitation(participants)
      → 아메바톡 내부 알림 발송 (즉시)
      → [optional] EmailService.sendInvitation() (SMTP)
    → [if google connected] GoogleCalendarService.pushEvent()
      → google_event_id 저장, sync_status = SYNCED
  ← 201 Created
```

### 9.2 반복 예외 — 이번 회차만 취소

```
User → POST /schedules/:sch_id/exceptions
  body: { sce_original_date: "2026-03-09", sce_exception_type: "CANCELLED" }
  → ScheduleExceptionService.create()
    → amb_schedule_exceptions INSERT
    → [if google connected] Google Calendar: delete specific occurrence
  ← 201 Created

Calendar View:
  → 반복 일정 렌더링 시 exceptions 목록 조회
  → sce_original_date 해당 회차는 렌더링에서 제외
```

### 9.3 낙관적 락 충돌 처리

```
User A: PATCH /schedules/:sch_id
  body: { ..., current_updated_at: "2026-03-01T10:00:00Z" }

User B (동시): PATCH /schedules/:sch_id
  body: { ..., current_updated_at: "2026-03-01T10:00:00Z" }
  → DB에서 sch_updated_at이 이미 변경됨 (User A가 먼저 저장)
  ← 409 Conflict { code: "E9001", message: "..." }
  → 프론트: "다른 사람이 이미 수정했습니다. 새로고침 후 다시 시도하세요." 토스트 표시
```

- `work-schedule` 모듈은 출퇴근/근무표 성격으로 존재하며, `usr_id + wks_date` 유니크 제약과 근태 정책 검증(평일/기간 제한/원격근무 제한)에 결합되어 있음.
- `settings` 권한 체계(메뉴/역할/그룹/사용자 권한) 재사용 가능.
- `search` + `amb_work_items` + `MODULE_DATA_EVENTS` 기반의 검색 연계 패턴 재사용 가능.
- `translation` 모듈의 `sourceType` 확장 패턴 재사용 가능.
- `@nestjs/schedule` + `Cron` 기반 배치/알림 실행 패턴 존재.

### C.2 Not Yet Implemented
- `schedule` 전용 도메인, 엔티티, API, 화면은 아직 없음.
- Google Calendar OAuth/Sync 구현 없음.
- 일정 알림/참여 응답/반복 예외 테이블 없음.

### C.3 MUST Separation Rule
- 신규 `schedule`은 기존 `work-schedule`과 **데이터/메뉴/API/권한**을 완전히 분리 구현해야 함.
- `work-schedule` 기존 동작(출퇴근 기록)은 변경하지 않음.
