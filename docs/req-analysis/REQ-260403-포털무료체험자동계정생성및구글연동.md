# REQ-포털무료체험자동계정생성및구글연동-20260403

## 1. 요구사항 요약 (Requirement Summary)

### 개선 1
- 대상: www.amoeba.site/register (portal-web)
- 현재 수동 승인 기반 프로세스를 자동화한다.
- 조건: 이름/이메일/비밀번호/회사/국가/전화번호를 입력한 무료체험 신청자.
- 목표: 어드민의 수동 Create Account 없이 AMA 계정 생성과 안내 메일 발송을 자동 처리.
- 결과: 사용자는 ama.amoeba.site/login으로 이동해 최초 1회 비밀번호 변경/등록 절차를 진행.

### 개선 2
- 대상: www.amoeba.site/register
- Google 연동 로그인/회원가입을 지원한다.
- 목표: Google 연동 완료 후 AMA 로그인 온보딩으로 연결.
- 후속 입력: 패스워드 등록, 회사명 등록, 사용국가 선택.

### 추가 정책
- register의 전화번호를 필수에서 옵션으로 변경.

## 3. TO-BE 요구사항 (Target State)

### 3.1 개선 1: 무료체험 신청 자동 계정 프로비저닝
1. Portal register 성공 시, 조건 충족 사용자에 대해 AMA 내부계정을 자동 생성한다.
2. 기존 수동 Create Account와 동일한 보안정책을 적용한다.
 - 랜덤 임시 비밀번호 또는 온보딩 토큰 발급
 - usrMustChangePw=true 유지
3. 사용자에게 ama 로그인 안내 메일을 자동 발송한다.
4. 기존 Admin 브릿지 화면은 예외 케이스(실패 재처리/재매핑) 용도로 유지한다.

### 3.2 개선 1-부가: 전화번호 옵션화
1. portal-web register 폼에서 phone을 optional로 변경한다.
2. portal-api register DTO에서도 phone optional로 변경한다.
3. 저장 시 null 허용 정책을 유지한다.

### 3.3 개선 2: Google 연동 가입/로그인
1. register 페이지에 Google 연동 진입 버튼을 추가한다.
2. Google OAuth 완료 후 Portal 고객 계정을 생성/매핑한다.
3. Google 연동 사용자에게 AMA 로그인 온보딩 경로를 제공한다.
 - 패스워드 등록
 - 회사명 등록
 - 사용국가 선택
4. 온보딩 완료 후 AMA 로그인으로 진입 가능해야 한다.

### 3.4 보안/정합성 요구
1. 동일 이메일 중복 생성 방지 (portal customer, AMA user, mapping 모두).
2. 프로비저닝 멱등성 보장(재시도 시 중복 계정 미생성).
3. 실패 시 보상 트랜잭션 또는 재처리 큐 전략 제공.

## 5. 사용자 플로우 (User Flow)

### 5.1 개선 1 자동 프로비저닝 플로우
1. 사용자가 www/register에서 필수 정보 입력 후 제출.
2. portal-api가 portal customer 생성.
3. 자동 프로비저닝 오케스트레이션이 AMA user 생성 및 mapping 생성.
4. 사용자에게 ama 로그인 안내 메일 발송.
5. 사용자가 ama/login에서 최초 1회 비밀번호 변경 후 서비스 이용.

### 5.2 개선 2 Google 연동 플로우
1. 사용자가 register에서 Continue with Google 선택.
2. Google 인증 완료 후 portal 신규/기존 사용자 판별.
3. 미완료 항목(회사명/국가/AMA 패스워드) 온보딩 화면 표시.
4. 온보딩 완료 시 AMA 계정 활성화 및 안내.
5. ama/login 접속 후 최초 비밀번호 정책 준수.

### 5.3 예외 플로우
1. 자동 프로비저닝 실패 시 상태를 FAILED로 기록하고 admin 재처리 가능.
2. 이메일 중복/법인 미지정/외부 API 실패 시 사용자 안내 메시지 표준화.

## 6. 기술 제약사항 (Technical Constraints)

1. 서비스 경계
- portal-api와 apps/api(AMA)는 분리 모듈로 운영되므로 동기 API 호출 또는 비동기 이벤트 설계가 필요하다.

2. 기존 수동 브릿지 호환
- [apps/api/src/domain/portal-bridge/service/portal-bridge.service.ts](apps/api/src/domain/portal-bridge/service/portal-bridge.service.ts#L129)의 수동 생성 로직과 충돌 없이 재사용/추상화해야 한다.

3. 비밀번호 강제 변경 정책
- 현재 usrMustChangePw 플래그 기반 정책이 이미 적용되어 있어 신규 자동 생성도 동일 플로우를 따라야 한다.

4. OAuth 보안
- state/nonce 검증, redirect URI 화이트리스트, 계정연동 탈취 방지(이메일 검증) 필요.

5. DB 변경 가능성
- Google 연동 메타데이터(예: provider, provider_user_id, onboarding_status) 추가가 필요할 수 있다.
- 스테이징/프로덕션은 synchronize 비활성 환경이므로 수동 SQL 마이그레이션이 필수다.

6. i18n
- portal-web UI 문구는 ko/en/vi 번역 리소스로 추가해야 하며 하드코딩 금지.
