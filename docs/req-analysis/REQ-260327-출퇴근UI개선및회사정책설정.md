## 2. AS-IS 현황 분석

### 2.1 근무 형태 (ATTENDANCE_TYPE)

현재 6종:

| 코드 | 한국어 | 영어 | 베트남어 | 시간 계산 |
|------|--------|------|----------|-----------|
| `WORK` | 출근 | Work | Đi làm | 출근시간 +9h |
| `REMOTE` | 재택 | Remote | Làm từ xa | 출근시간 +9h |
| `DAY_OFF` | 휴가 | Day Off | Nghỉ phép | null |
| `AM_HALF` | 오전 반차 | AM Half Day | Nghỉ nửa ngày sáng | 13:00~17:00 고정 |
| `PM_HALF` | 오후 반차 | PM Half Day | Nghỉ nửa ngày chiều | 출근시간~12:00 |
| `MENSTRUATION` | 생리 휴가 | Menstruation Rest | Nghỉ kinh nguyệt | null |

**문제점**:
- `WORK`가 "출근"으로 되어 있어 재택과의 구분이 모호 (사무실 출근 vs 일반 출근)
- 외근, 출장, 외부사이트 근무 형태가 없어 정확한 근무 위치 추적 불가

### 2.2 셀 렌더링 (수정 이력 표시)

현재 동작:
- 수정 이력이 있으면 **최신 수정본만** effectiveType으로 표시
- 오렌지 점(●)으로 수정 여부만 표시
- 원본 일정은 수정 이력 모달에서만 확인 가능

```tsx
// 현재 셀 렌더링 로직
const effectiveType = hasAmendments
  ? schedule!.amendments![schedule!.amendments!.length - 1].type
  : schedule?.type;
// → effectiveType만 표시, 원본 보이지 않음
```

**문제점**: 관리자가 한눈에 원본 → 수정 변경 내용을 파악하기 어려움

### 2.3 주말 근무 시간 선택

현재 동작:
- `validateWeekday()` 제거로 주말(토/일) 출퇴근 등록 가능
- 그러나 시간 선택 UI는 평일과 동일 (START_TIME_OPTIONS: 08:00~10:00)
- 주말 셀에 빈 상태일 때 연한 회색(`bg-gray-50/60`) 배경

**분석**: 주말 시간 선택은 이미 평일과 동일하게 동작하고 있으므로 추가 개발 불필요. 다만 주말 특화 시간 옵션이 필요한지 확인 필요.

### 2.4 재택근무 횟수 제한

현재 동작:
- `validateRemoteLimit()`: **월 1회** 하드코딩

```typescript
const count = await qb.getCount();
if (count + pendingRemoteCount >= 1) {  // ← 하드코딩 "1"
  throw new BadRequestException(ERROR_CODE.ATTENDANCE_REMOTE_LIMIT_EXCEEDED.message);
}
```

**문제점**:
- 회사별/법인별 정책 차이를 반영할 수 없음
- 정책 변경 시 코드 수정 + 재배포 필요
- 기본 제공 횟수 / 추가 사용 가능 횟수 개념 없음

### 2.5 회사 정책 설정 페이지

현재 상태:
- **존재하지 않음** — 출퇴근 관련 정책을 관리할 UI 없음
- entity-settings 하위에 16개 페이지 존재 (members, permissions, api-keys 등)
- HR 설정은 별도 `HrSettingsPage`에 보험료율, 세율, OT 등 관리
- 출퇴근 정책은 어디에도 설정 UI 없음

## 3. TO-BE 요구사항 상세

### 3.1 [R1] 수정된 일정 셀 — 원본 취소선 표시

**목표**: 수정 이력이 있는 셀에서 원본 일정을 취소선으로 표시하고, 하단에 수정본을 표시

**UI 설계**:

```
┌─────────────────┐
│ ~~09:00-18:00~~ │  ← 원본 (취소선, 연한 색)
│  재택 09:00     │  ← 최신 수정본 (정상 색)
│            ●    │  ← 오렌지 점 유지
└─────────────────┘
```

**세부 규칙**:
- 원본 타입/시간을 `line-through` + 연한 텍스트 색상으로 표시
- 최신 수정본(effectiveType)을 기존 방식대로 표시
- 수정이 여러 차례인 경우에도 **원본 → 최신** 만 표시 (중간 이력은 모달에서 확인)
- 원본과 수정본이 동일한 경우(시간만 변경 등) 둘 다 표시
- 셀 높이가 늘어나므로 행 높이 자동 조절 필요

**원본 표시 형식**:
| 타입 | 표시 형식 |
|------|----------|
| OFFICE/REMOTE + 시간 | `~~09:00-18:00~~` |
| DAY_OFF | `~~OFF~~` |
| AM_HALF | `~~오전반차~~` |
| PM_HALF | `~~오후반차~~` |
| MENSTRUATION | `~~생리휴가~~` |

### 3.2 [R2] 근무 형태 확장

**기존 → 변경 매핑**:

| 변경 | 기존 코드 | 신규 코드 | 한국어 | 영어 | 베트남어 | 시간 계산 |
|------|----------|----------|--------|------|----------|-----------|
| 이름 변경 | `WORK` | `OFFICE` | 사무실 | Office | Văn phòng | 출근시간 +9h |
| 유지 | `REMOTE` | `REMOTE` | 재택 | Remote | Làm từ xa | 출근시간 +9h |
| **신규** | - | `OUTSIDE_WORK` | 외근 | Outside Work | Làm ngoài | 출근시간 +9h |
| **신규** | - | `BUSINESS_TRIP` | 출장 | Business Trip | Công tác | 출근시간 +9h |
| **신규** | - | `EXTERNAL_SITE` | 외부사이트 | External Site | Công trường | 출근시간 +9h |
| 유지 | `DAY_OFF` | `DAY_OFF` | 휴가 | Day Off | Nghỉ phép | null |
| 유지 | `AM_HALF` | `AM_HALF` | 오전 반차 | AM Half Day | Nghỉ nửa ngày sáng | 13:00~17:00 |
| 유지 | `PM_HALF` | `PM_HALF` | 오후 반차 | PM Half Day | Nghỉ nửa ngày chiều | 출근~12:00 |
| 유지 | `MENSTRUATION` | `MENSTRUATION` | 생리 휴가 | Menstruation Rest | Nghỉ kinh nguyệt | null |

**WORK → OFFICE 마이그레이션**:
- DB의 기존 `WORK` 레코드를 `OFFICE`로 UPDATE 필요
- 코드에서 `WORK` 참조를 모두 `OFFICE`로 변경
- `amb_attendance_amendments` 테이블의 `aam_type`도 마이그레이션 대상

**새 타입 시간 계산 규칙**:
- OUTSIDE_WORK, BUSINESS_TRIP, EXTERNAL_SITE: `OFFICE`와 동일 (출근시간 + 9시간)
- 이 3개 타입은 **재택 횟수 제한과 무관** (제한은 REMOTE에만 적용)
- 승인 플로우: 당주는 PENDING, 다음주부터는 APPROVED (REMOTE와 동일)

**UI 색상 설계**:

| 타입 | 텍스트 색상 | 배지 필요 |
|------|-----------|----------|
| OFFICE | text-gray-700 | 없음 |
| REMOTE | text-green-700 | 없음 |
| OUTSIDE_WORK | text-blue-600 | 타입명 배지 |
| BUSINESS_TRIP | text-purple-600 | 타입명 배지 |
| EXTERNAL_SITE | text-teal-600 | 타입명 배지 |
| DAY_OFF | text-red-600 | 없음 (OFF 텍스트) |
| AM_HALF | text-amber-700 | 반차 배지 |
| PM_HALF | text-orange-700 | 반차 배지 |
| MENSTRUATION | text-pink-600 | 없음 |

### 3.3 [R3] 토요일/일요일 근무 시간 선택

**현재 상태**: 이미 동작 중 (validateWeekday 제거 완료)

**추가 개선 사항**:
- 주말 셀의 빈 상태 표시를 평일과 동일하게 변경 (현재 회색 배경 유지 가능)
- 시간 선택 옵션: 평일과 동일 (08:00, 08:30, 09:00, 09:30, 10:00)
- 주말에도 모든 근무 형태 선택 가능 (OFFICE, REMOTE, OUTSIDE_WORK 등)
- **별도 추가 개발 최소화** — 기존 로직으로 충분

### 3.4 [R4] 회사 규정 설정 페이지

**목표**: 회사(Entity)별 출퇴근 관련 정책을 관리하는 설정 페이지

#### 3.4.1 정책 항목

| 항목 | 필드명 | 타입 | 기본값 | 설명 |
|------|--------|------|--------|------|
| 월 재택근무 기본 제공 횟수 | `remote_default_count` | integer | 1 | 매월 기본 사용 가능한 재택 횟수 |
| 추가 사용 가능 횟수 | `remote_extra_count` | integer | 0 | 기본 외 추가로 사용 가능한 재택 횟수 |
| 초과 시 차단 여부 | `remote_block_on_exceed` | boolean | true | (기본+추가) 초과 시 재택 선택 차단 |
| 월차 자동 차감 | `leave_auto_deduct` | boolean | false | 휴가 등록 시 월차 자동 차감 여부 (향후) |
| 반차 자동 차감 | `half_leave_auto_deduct` | boolean | false | 반차 등록 시 0.5일 차감 여부 (향후) |

> **참고**: 월차/반차 처리(leave_auto_deduct, half_leave_auto_deduct)는 HR Leave 모듈과의 연계가 필요하므로 1차에서는 설정 UI만 준비하고, 실제 차감 로직은 향후 구현

#### 3.4.2 DB 테이블 설계

```sql
CREATE TABLE amb_attendance_policies (
  atp_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  ent_id UUID NOT NULL REFERENCES amb_hr_entities(ent_id),
  atp_remote_default_count INTEGER NOT NULL DEFAULT 1,
  atp_remote_extra_count INTEGER NOT NULL DEFAULT 0,
  atp_remote_block_on_exceed BOOLEAN NOT NULL DEFAULT true,
  atp_leave_auto_deduct BOOLEAN NOT NULL DEFAULT false,
  atp_half_leave_auto_deduct BOOLEAN NOT NULL DEFAULT false,
  atp_created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  atp_updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  CONSTRAINT uq_attendance_policy_entity UNIQUE (ent_id)
);
```

- Entity당 1개 정책 레코드 (UNIQUE ent_id)
- 정책이 없는 Entity는 기본값(월 1회 재택, 추가 0, 초과 차단) 적용

#### 3.4.3 validateRemoteLimit 변경

**AS-IS**:
```typescript
if (count + pendingRemoteCount >= 1) { // 하드코딩
```

**TO-BE**:
```typescript
const policy = await this.getAttendancePolicy(entityId);
const maxRemote = policy.remoteDefaultCount + policy.remoteExtraCount;
if (policy.remoteBlockOnExceed && count + pendingRemoteCount >= maxRemote) {
  throw new BadRequestException(ERROR_CODE.ATTENDANCE_REMOTE_LIMIT_EXCEEDED.message);
}
```

#### 3.4.4 설정 UI 위치

- **경로**: `/entity-settings/attendance-policy`
- **위치**: entity-settings 하위 메뉴에 "출퇴근 정책" 추가
- **권한**: MASTER 역할만 접근 가능 (EntitySettingsGuard)
- **구성**: 설정 폼 (숫자 입력 + 토글 스위치)

## 4. 갭 분석

### 4.1 변경 범위 요약

| 영역 | 현재 | 변경 | 영향도 |
|------|------|------|--------|
| `ATTENDANCE_TYPE` | 6종 | 9종 (+3종, WORK→OFFICE) | **높음** — FE/BE/types/i18n 전역 |
| 셀 렌더링 | effectiveType만 표시 | 원본(취소선) + effectiveType | **중간** — FE AttendancePage |
| `validateRemoteLimit()` | 하드코딩 1 | 정책 테이블 조회 | **높음** — BE + 신규 Entity/Service |
| `calculateTimes()` | WORK/REMOTE만 +9h | OFFICE/OUTSIDE_WORK/BUSINESS_TRIP/EXTERNAL_SITE도 +9h | **낮음** — case 추가만 |
| 정책 테이블 | 없음 | `amb_attendance_policies` 신규 | **높음** — DB + Entity + CRUD |
| 정책 설정 UI | 없음 | EntityAttendancePolicyPage 신규 | **높음** — FE 신규 페이지 |
| 색상 매핑 | 6종 | 9종 | **낮음** — 상수 추가 |
| i18n | 6종 타입명 | 9종 타입명 + 정책 키 | **중간** — 3개 언어 파일 |
| 라우터 | entity-settings 16개 | +1 (attendance-policy) | **낮음** |
| DB 마이그레이션 | - | WORK→OFFICE + 신규 테이블 | **높음** — 기존 데이터 변환 |

### 4.2 사이드 임팩트

1. **WORK → OFFICE 변경**: DB UPDATE + 코드 전역 치환 필요
   - `calculateTimes()`: WORK case → OFFICE
   - `TYPE_TEXT_COLOR`: WORK 키 → OFFICE
   - i18n 3개 언어 파일: type.WORK → type.OFFICE
   - AttendanceTypeBadge: WORK 참조 있으면 변경
   - 프론트엔드 formType 기본값이 WORK인 곳 → OFFICE

2. **새 타입 3종 추가**: 시스템 전반에 타입 추가
   - `ATTENDANCE_TYPE` const에 추가
   - `calculateTimes()` switch case 추가
   - 프론트 select 옵션 자동 반영 (ATTENDANCE_TYPE에서 순회하는 경우)
   - 색상/배지 매핑 추가

3. **정책 테이블**: 새 모듈 또는 기존 attendance 모듈에 추가
   - AttendanceModule에 PolicyEntity + PolicyService 추가
   - EntitySettingsGuard 라우트 추가

4. **승인 플로우**: 새 타입 3종도 REMOTE와 동일한 승인 로직 적용할지 결정 필요
   - 현재: REMOTE + isCurrentWeek → PENDING, 나머지 APPROVED
   - 제안: OFFICE/OUTSIDE_WORK/BUSINESS_TRIP/EXTERNAL_SITE는 APPROVED (REMOTE만 PENDING)

## 5. 사용자 플로우

### 5.1 일반 사용자 — 근무 등록 플로우

```
1. AttendancePage 접속
2. 빈 셀 클릭 → 유형 선택 드롭다운 (9종 표시)
3. OFFICE/REMOTE/OUTSIDE_WORK/BUSINESS_TRIP/EXTERNAL_SITE 선택 시:
   → 출근 시간 선택 (08:00~10:00)
   → 퇴근 시간 자동 계산 (+9h)
4. REMOTE 선택 시:
   → 정책 기반 월 한도 확인 (기본+추가)
   → 초과 시 에러 메시지 ("월 재택근무 한도를 초과했습니다")
   → 차단 해제 상태이면 경고만 표시하고 등록 허용
5. DAY_OFF/AM_HALF/PM_HALF/MENSTRUATION 선택 시:
   → 기존과 동일
6. 저장 → API 호출
```

### 5.2 관리자 — 수정된 셀 확인 플로우

```
1. AttendancePage에서 팀원 출퇴근 목록 확인
2. 수정된 셀:
   - 원본: "09:00-18:00" 취소선으로 표시 (연한 색)
   - 수정본: "재택 09:00-18:00" 정상 표시
   - 오렌지 점(●) 유지
3. 셀 클릭 → 수정 이력 모달에서 전체 변경 내역 확인 가능
4. 관리자가 다시 수정 시 → 기존 수정 모달 사용
```

### 5.3 MASTER — 정책 설정 플로우

```
1. 왼쪽 메뉴 → 회사 설정 → 출퇴근 정책
2. EntityAttendancePolicyPage 표시
3. 재택근무 정책:
   - 월 기본 제공 횟수 조절 (숫자 입력)
   - 추가 사용 가능 횟수 조절 (숫자 입력)
   - 초과 시 차단 ON/OFF 토글
4. 휴가 정책 (향후):
   - 월차 자동 차감 토글
   - 반차 자동 차감 토글
5. [저장] 클릭 → API 호출
6. 정책 없으면 자동 생성, 있으면 업데이트 (UPSERT)
```

## 6. 기술 제약사항 및 설계 결정

### 6.1 WORK → OFFICE 마이그레이션 전략

#### 옵션 A: 코드와 DB 동시 변경 (권장)
1. DB: `UPDATE amb_attendances SET att_type = 'OFFICE' WHERE att_type = 'WORK';`
2. DB: `UPDATE amb_attendance_amendments SET aam_type = 'OFFICE' WHERE aam_type = 'WORK';`
3. 코드: WORK를 모두 OFFICE로 변경
4. 배포 순서: DB 마이그레이션 → 코드 배포

#### 옵션 B: 하위 호환 유지 (WORK + OFFICE 공존)
- WORK를 레거시로 유지하되 UI에서는 OFFICE로 표시
- 새 등록은 OFFICE, 기존은 WORK 유지
- 코드 복잡도 증가

**결정**: **옵션 A** 권장 — 데이터가 적고 이중 관리 비용 회피

### 6.2 정책 테이블 위치

- `attendance` 모듈 내에 Entity/Service 추가
- 별도 모듈 불필요 — AttendanceModule에서 PolicyEntity를 import하여 사용

### 6.3 승인 플로우 결정

- REMOTE만 당주 PENDING 유지
- OFFICE, OUTSIDE_WORK, BUSINESS_TRIP, EXTERNAL_SITE: 항상 APPROVED
- 향후 요구사항에 따라 타입별 승인 정책 추가 가능

### 6.4 스테이징 DB 마이그레이션

- synchronize 비활성 → 수동 SQL 실행 필수
- 마이그레이션 SQL:
  1. `amb_attendance_policies` 테이블 생성
  2. `WORK` → `OFFICE` 데이터 변환
