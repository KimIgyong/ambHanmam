## 1. 요구사항 원문

> 모든 페이지에서 발생하는 에러로그를 어떤 사용자(모든 레벨: ADMIN_LEVEL, PARTNER_LEVEL, USER_LEVEL, CLIENT_LEVEL)의 어느 페이지에서 어떤 에러인지를 admin/site-error에 기록한다.

## 2. AS-IS 현황 분석

### 2.1 프론트엔드 에러 처리

| 항목 | 현황 | 문제점 |
|------|------|--------|
| ErrorBoundary | ❌ 없음 | 렌더링 에러 시 빈 화면(white screen) 발생 |
| 전역 에러 핸들러 | ❌ 없음 | `window.onerror`, `unhandledrejection` 미수집 |
| API 에러 인터셉터 | ✅ axios interceptor 존재 | 401 토큰 갱신만 처리, 에러 기록 없음 |
| 에러 표시 | ✅ toast.error 개별 처리 | 사용자에게만 알림, 서버 기록 없음 |
| 에러 로그 DB 저장 | ❌ 없음 | 어디서 어떤 에러가 발생했는지 추적 불가 |

### 2.2 백엔드 에러 처리

| 항목 | 현황 | 문제점 |
|------|------|--------|
| HttpExceptionFilter | ✅ 구현됨 | 응답만 포맷팅, DB 기록 없음 |
| BusinessExceptionFilter | ✅ 구현됨 | 응답만 포맷팅, DB 기록 없음 |
| 에러 코드 체계 | ✅ E1xxx~E22xxx 정의됨 | 잘 구축되어 있으나 발생 이력 미추적 |
| 에러 로그 테이블 | ❌ 없음 | 에러 발생 패턴 분석 불가 |

### 2.3 관리자 페이지

| 항목 | 현황 |
|------|------|
| admin/site-error 페이지 | ❌ 없음 |
| 에러 모니터링 기능 | ❌ 없음 |
| 에러 통계/대시보드 | ❌ 없음 |

### 2.4 사용자 레벨 체계 (현행)

| 레벨 코드 | 설명 | 앱 |
|-----------|------|-----|
| `ADMIN_LEVEL` | 시스템 관리자 | apps/web |
| `USER_LEVEL` | 일반 사용자 (사내) | apps/web |
| `PARTNER_LEVEL` | 파트너사 사용자 | apps/web |
| `CLIENT_LEVEL` | 고객 포탈 사용자 | apps/portal-web |

## 3. TO-BE 요구사항 정의

### 3.1 기능 요구사항

#### FR-01. 프론트엔드 에러 수집

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| FR-01-1 | **API 에러 자동 수집**: 모든 API 호출(4xx, 5xx)에서 발생하는 에러를 자동으로 서버에 전송한다 | 필수 |
| FR-01-2 | **JS 런타임 에러 수집**: `window.onerror`, `unhandledrejection` 이벤트를 캐치하여 서버에 전송한다 | 필수 |
| FR-01-3 | **React 렌더링 에러 수집**: ErrorBoundary를 통해 컴포넌트 렌더링 에러를 캐치하여 서버에 전송한다 | 필수 |
| FR-01-4 | **수집 정보**: 에러 메시지, 에러 코드, HTTP 상태코드, 발생 페이지 URL, 스택 트레이스, 브라우저 정보(User-Agent) | 필수 |
| FR-01-5 | **사용자 정보 포함**: 에러 발생 시 로그인된 사용자의 ID, 이메일, 레벨코드를 함께 전송한다 | 필수 |
| FR-01-6 | **비인증 에러 수집**: 로그인 전 발생하는 에러(로그인 페이지, 인증 실패 등)도 수집한다 (사용자 정보 null 허용) | 필수 |
| FR-01-7 | **중복 방지**: 동일 에러가 짧은 시간 내 반복 발생 시 최초 1건만 전송한다 (디바운싱) | 권장 |
| FR-01-8 | **앱 구분**: apps/web과 apps/portal-web의 에러를 구분하여 기록한다 (source 필드) | 필수 |

#### FR-02. 백엔드 에러 수집

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| FR-02-1 | **에러 로그 API**: `POST /api/v1/site-errors` 엔드포인트를 제공한다 | 필수 |
| FR-02-2 | **인증 선택적**: 비인증 사용자도 에러를 전송할 수 있도록 `@Public()` + Optional Auth를 적용한다 | 필수 |
| FR-02-3 | **백엔드 에러 기록**: ExceptionFilter에서 5xx 서버 에러 발생 시 자동으로 DB에 기록한다 | 필수 |
| FR-02-4 | **Rate Limiting**: 에러 제출 API에 IP당 분당 30건 제한을 적용한다 | 필수 |
| FR-02-5 | **데이터 정제**: 스택 트레이스에서 민감 정보(토큰, 비밀번호 등) 제거 후 저장한다 | 필수 |

#### FR-03. 에러 로그 저장

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| FR-03-1 | **에러 로그 테이블**: `amb_site_error_logs` 테이블을 생성한다 | 필수 |
| FR-03-2 | **저장 항목**: 에러원천(frontend/backend), 앱(web/portal-web/api/portal-api), 사용자ID, 사용자이메일, 사용자레벨, 페이지URL, API엔드포인트, HTTP메서드, HTTP상태코드, 에러코드, 에러메시지, 스택트레이스, User-Agent, IP주소, Entity ID, 발생시각 | 필수 |
| FR-03-3 | **자동 정리**: 90일 이상 된 에러 로그는 자동 삭제한다 (Cron) | 권장 |
| FR-03-4 | **Soft Delete**: 관리자가 해결 완료 표시한 에러는 숨김 처리 (삭제가 아닌 상태 변경) | 권장 |

#### FR-04. 관리자 에러 모니터링 페이지

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| FR-04-1 | **페이지 경로**: `/admin/site-errors`에 에러 로그 목록 페이지를 제공한다 | 필수 |
| FR-04-2 | **목록 표시**: 에러 발생시각, 에러원천, 사용자 정보, 페이지, 에러코드, 에러메시지를 테이블로 표시한다 | 필수 |
| FR-04-3 | **필터링**: 에러원천(frontend/backend), 앱(web/portal), 사용자레벨, HTTP상태코드, 에러코드, 날짜범위로 필터링한다 | 필수 |
| FR-04-4 | **검색**: 에러메시지, 사용자이메일, 페이지URL 기준 텍스트 검색을 지원한다 | 필수 |
| FR-04-5 | **상세 보기**: 에러 행 클릭 시 스택 트레이스, 요청 정보 등 상세 정보를 모달로 표시한다 | 필수 |
| FR-04-6 | **에러 통계 카드**: 페이지 상단에 오늘/이번주 에러 건수, 원천별/레벨별 분포를 요약 카드로 표시한다 | 권장 |
| FR-04-7 | **해결 완료 표시**: 에러 로그에 '해결완료' 상태를 마킹할 수 있다 | 권장 |
| FR-04-8 | **접근 권한**: ADMIN_LEVEL 사용자만 접근 가능하다 | 필수 |
| FR-04-9 | **페이지네이션**: 50건 단위로 페이지네이션한다 | 필수 |

### 3.2 비기능 요구사항

| ID | 요구사항 | 설명 |
|----|----------|------|
| NFR-01 | **성능** | 에러 수집이 사용자 경험에 영향을 주지 않아야 한다 (비동기, fire-and-forget) |
| NFR-02 | **안정성** | 에러 수집 자체가 실패해도 앱 동작에 영향을 주지 않아야 한다 (silent fail) |
| NFR-03 | **보안** | 스택 트레이스에 민감 정보(토큰, 비밀번호, API 키)가 포함되지 않아야 한다 |
| NFR-04 | **저장 용량** | 스택 트레이스 최대 5,000자로 제한하여 DB 비대화를 방지한다 |
| NFR-05 | **i18n** | 관리자 페이지 UI 텍스트는 i18n 번역 파일을 사용한다 (ko/en/vi) |

## 7. 사용자 플로우

### 7.1 에러 발생 → 자동 수집

```
사용자 (모든 레벨)
    │
    ├── [페이지 접속] → JS 런타임 에러 발생
    │       │
    │       └── window.onerror / unhandledrejection 캐치
    │               │
    │               └── ErrorReporter.report({ error, page, user })
    │                       │
    │                       └── POST /api/v1/site-errors (fire-and-forget)
    │
    ├── [API 호출] → 4xx/5xx 응답 수신
    │       │
    │       └── axios response interceptor 캐치
    │               │
    │               ├── toast.error(message) — 사용자에게 표시
    │               └── ErrorReporter.report({ error, endpoint, status })
    │                       │
    │                       └── POST /api/v1/site-errors (fire-and-forget)
    │
    └── [컴포넌트 렌더링] → React 에러 발생
            │
            └── ErrorBoundary 캐치
                    │
                    ├── Fallback UI 표시 ("오류가 발생했습니다")
                    └── ErrorReporter.report({ error, componentStack })
```

## 8. 에러 수집 범위

### 8.1 프론트엔드 수집 대상

| 유형 | 수집 조건 | 예시 |
|------|-----------|------|
| API 에러 (4xx) | 400, 401(보고만), 403, 404, 409, 422 | 유효성 검증 실패, 권한 없음 |
| API 에러 (5xx) | 500, 502, 503 | 서버 내부 에러 |
| 네트워크 에러 | `error.code === 'ERR_NETWORK'` | 서버 연결 불가 |
| JS 런타임 에러 | `window.onerror` | TypeError, ReferenceError |
| Promise 거부 | `unhandledrejection` | 미처리 비동기 에러 |
| React 렌더링 에러 | ErrorBoundary `componentDidCatch` | 컴포넌트 크래시 |

### 8.2 수집 제외 대상

| 유형 | 이유 |
|------|------|
| 401 토큰 갱신 성공 | 정상 흐름 (토큰 리프레시) |
| 네트워크 일시 단절 후 재연결 | 일시적 현상 |
| 에러 수집 API 자체의 실패 | 무한 루프 방지 |
| 정적 리소스 로딩 에러 (favicon 등) | 경미한 이슈 |

### 8.3 백엔드 수집 대상

| 유형 | 수집 조건 |
|------|-----------|
| 5xx 서버 에러 | HttpExceptionFilter에서 status >= 500 |
| 미처리 예외 | AllExceptionsFilter (새로 추가) |
| BusinessException | 선택적 (errorCode 기반 필터링 가능) |

### 8.4 백엔드 수집 제외 대상

| 유형 | 이유 |
|------|------|
| 400 유효성 검증 실패 | 빈도 높고 보안 이슈 아님 |
| 401 인증 실패 | 별도 로그인 실패 카운터 존재 |
| 404 Not Found | 일반적인 탐색 |

## 9. 기술 제약사항

| 항목 | 제약 | 대응 방안 |
|------|------|-----------|
| 프론트 에러 보고 시 인증 토큰 없을 수 있음 | 로그인 전 에러도 수집 필요 | `@Public()` + Optional JwtGuard |
| 에러 보고 API가 에러 나면 무한 루프 | 에러 보고 실패 시 추가 보고 금지 | `site-errors` 엔드포인트 URL 제외 필터 |
| 스택 트레이스가 minified 될 수 있음 | 프로덕션 빌드 시 난독화 | Source map 유무에 관계없이 원본 메시지 우선 저장 |
| 대량 에러 발생 시 DB 부하 | 동일 에러 반복 시 | 프론트: 디바운싱(10초), 백엔드: 분당 30건 제한 |
| TypeORM synchronize: false (스테이징/프로덕션) | 수동 마이그레이션 필요 | SQL 마이그레이션 스크립트 별도 작성 |

## 11. 참고 사항

### 11.1 기존 에러 코드 체계 활용

현재 `E1xxx~E22xxx`로 잘 구축된 에러 코드 체계가 있으므로, 에러 로그에 `sel_error_code`를 포함하여 코드별 에러 빈도 분석이 가능하도록 한다.

### 11.2 기존 감사 로그와의 차이

| 항목 | `amb_data_audit_log` (P17) | `amb_site_error_logs` (본 건) |
|------|---------------------------|-------------------------------|
| 목적 | 데이터 변경 추적 | 에러 발생 추적 |
| 기록 대상 | INSERT/UPDATE/DELETE | 4xx/5xx 에러, JS 에러 |
| 트리거 | TypeORM Subscriber | ExceptionFilter + 프론트 |
| 사용자 | 인증 사용자만 | 비인증 포함 |

### 11.3 유사 서비스 참고

- Sentry, Bugsnag 등 외부 에러 모니터링 SaaS의 핵심 기능을 내재화
- 자체 구현으로 외부 의존성 없이 에러 추적 가능
