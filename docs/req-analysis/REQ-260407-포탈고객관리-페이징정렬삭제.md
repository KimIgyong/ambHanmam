### 2.5 엔티티 접근 구조 특이점

`PortalCustomerReadonlyEntity`는 **읽기 전용 엔티티**로 설계됨:
- apps/api에서는 `SELECT` 전용으로 사용
- apps/portal-api에서 `PortalCustomerEntity`(읽기-쓰기)로 관리
- **삭제 기능 구현 시** apps/api에서 직접 삭제하거나, 별도 DELETE 엔티티/쿼리가 필요

## 3. TO-BE 요구사항

### 3.1 페이징 개선
- 페이지 크기 선택 (20 / 50 / 100)
- 현재 페이지 / 총 페이지 / 총 건수 표시
- 이전/다음 + 페이지 번호 네비게이션

### 3.2 정렬 필터
- 컬럼 헤더 클릭으로 정렬 (ASC/DESC 토글)
- 정렬 가능 컬럼: 이름, 이메일, 회사, 국가, 생성일
- 기본값: 생성일 DESC

### 3.3 국가별 필터
- 드롭다운 select로 국가 필터
- DB에 존재하는 국가 코드 목록을 동적으로 조회하여 표시
- "전체" 옵션 포함

### 3.4 상태 필터 (미사용 파라미터 활성화)
- 백엔드 DTO에 이미 `status` 파라미터 존재
- 프론트에서 ACTIVE / INACTIVE / SUSPENDED 필터 드롭다운 추가

### 3.5 삭제 기능
- 포탈 고객 소프트 삭제 (`pct_deleted_at` 설정)
- 삭제 확인 다이얼로그 (i18n)
- 삭제 시 연관 매핑 자동 REVOKE 처리
- ADMIN_LEVEL만 삭제 가능 (`@AdminOnly()`)

## 5. 삭제 시 사이드 임팩트 분석

### 5.1 `amb_svc_portal_customers` 테이블 참조 관계

| 참조 테이블 | FK 컬럼 | 관계 | 삭제 시 처리 |
|-------------|---------|------|-------------|
| `amb_portal_user_mappings` | `pct_id` | 1:N | **매핑 REVOKE 필수** (pumStatus → REVOKED) |
| `amb_svc_clients` | `pct_cli_id` (역방향) | N:1 | 영향 없음 (고객이 참조하는 방향) |

### 5.2 Soft Delete 시 영향 범위

| 영역 | 영향 | 처리 방안 |
|------|------|-----------|
| **포탈 로그인** | 삭제된 고객은 로그인 불가해야 함 | `pct_deleted_at IS NOT NULL` 검사 (portal-api 확인 필요) |
| **포탈-내부 매핑** | 활성 매핑이 남으면 데이터 불일치 | 삭제 전 ACTIVE 매핑 자동 REVOKE |
| **내부 사용자 계정** | 매핑 통해 생성된 amb_users 레코드 | **삭제하지 않음** (독립 엔티티) |
| **Stripe 연동** | pct_stripe_customer_id 참조 | Stripe 구독은 별도 관리 (삭제 차단 or 경고) |
| **목록 조회** | 이미 `pctDeletedAt IS NULL` 필터 사용 중 | 추가 조치 불필요 |
| **구독 관리** | portal-api의 구독 로직 | 활성 구독이 있으면 삭제 차단 권장 |

### 5.3 삭제 안전성 검증 포인트

1. **삭제 전 체크**: 활성 매핑 존재 시 → 자동 REVOKE 후 삭제
2. **복원 가능성**: Soft Delete이므로 pct_deleted_at을 NULL로 되돌리면 복원 가능
3. **portal-api 로그인**: `PortalCustomerEntity`의 `pctDeletedAt` 컬럼으로 이미 소프트 삭제 지원
4. **감사 로그**: 삭제 수행자(admin userId) 기록 권장

## 6. 사용자 플로우

### 6.1 목록 관리 플로우
```
관리자 → /admin/total-users → 포탈 고객 탭
  → 검색/필터 적용 (검색어, 국가, 상태, 매핑여부)
  → 컬럼 헤더 클릭 → 정렬 변경
  → 페이지 크기 변경 / 페이지 이동
  → 결과 확인
```

### 6.2 삭제 플로우
```
관리자 → 대상 고객 행의 삭제 버튼 클릭
  → 확인 다이얼로그 표시 (이름, 이메일 표시)
  → [활성 매핑 있음] → "매핑이 자동 해제됩니다" 경고 포함
  → 확인 → API 호출 (DELETE /portal-bridge/customers/:pctId)
  → 백엔드: 활성 매핑 REVOKE → pct_deleted_at 설정
  → 성공 토스트 → 목록 갱신
```

## 7. 기술 제약사항

| 제약 | 설명 |
|------|------|
| **읽기 전용 엔티티** | `PortalCustomerReadonlyEntity`는 SELECT 전용 → 삭제를 위해 TypeORM 직접 쿼리(`createQueryBuilder`) 또는 별도 writable 엔티티 필요 |
| **AdminOnly 권한** | 모든 portal-bridge 엔드포인트는 `@AdminOnly()` → ADMIN_LEVEL만 접근 |
| **i18n 3개 언어** | 새 UI 텍스트 추가 시 ko/en/vi 모두 작성 필수 |
| **토스트 피드백** | 삭제 성공/실패 시 반드시 toast 알림 표시 (sonner) |
| **Soft Delete** | 물리 삭제 금지, `pct_deleted_at` 타임스탬프 설정 방식 |
