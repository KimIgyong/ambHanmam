## 1. 시스템 개요

### 1.1 시스템 구성

AMB Management는 아메바 회사의 업무 관리 통합 플랫폼입니다.

| 구성 요소 | 설명 |
|-----------|------|
| **Frontend (Web)** | React 기반 SPA, 포트 5179 (개발) |
| **Backend (API)** | NestJS REST API, 포트 3009 (개발) |
| **Database** | PostgreSQL 15 |
| **AI 에이전트** | Anthropic Claude 기반 부서별 AI 채팅 |

### 1.2 권한 계층 구조 (5단계)

시스템은 다음 5가지 권한 계층으로 사용자 접근을 제어합니다:

```
┌─────────────────────────────────────────────┐
│  ① 시스템 역할 (System Role)                 │
│     SUPER_ADMIN > ADMIN > MANAGER > MEMBER > VIEWER │
├─────────────────────────────────────────────┤
│  ② 그룹 코드 (Group Code)                    │
│     ADMIN_GROUP (본사 관리자)                  │
│     USER_GROUP  (법인 사용자)                  │
├─────────────────────────────────────────────┤
│  ③ 법인 역할 (Entity Role)                   │
│     SYSTEM_ADMIN / ADMIN / MEMBER            │
├─────────────────────────────────────────────┤
│  ④ 부서 역할 (Department Role)               │
│     DEPARTMENT_HEAD > TEAM_LEAD > MEMBER     │
├─────────────────────────────────────────────┤
│  ⑤ 그룹별 메뉴 접근 (Group Permission)        │
│     법인 소속 그룹 → 메뉴별 접근 ON/OFF        │
└─────────────────────────────────────────────┘
```

## 2. 최초 세팅 및 기본 사용자

### 2.1 시스템 최초 구동 시 자동 생성되는 항목

서버가 처음 시작되면 아래 항목들이 자동으로 생성됩니다.

#### (1) 기본 법인 3개

| 법인 코드 | 법인명 | 국가 | 통화 | 레벨 | HQ 여부 |
|-----------|--------|------|------|------|---------|
| **HQ** | Amoeba HQ | GL (글로벌) | USD | ROOT | ✅ |
| **VN01** | AMOEBA CO., LTD | VN (베트남) | VND | SUBSIDIARY | ❌ |
| **KR01** | 아메바컴퍼니주식회사 | KR (한국) | KRW | SUBSIDIARY | ❌ |

#### (2) 최초 관리자 계정 (SUPER_ADMIN)

환경변수를 통해 자동 생성됩니다:

| 환경변수 | 설명 | 예시 |
|----------|------|------|
| `SEED_ADMIN_EMAIL` | 관리자 이메일 | `admin@amoeba.com` |
| `SEED_ADMIN_PASSWORD` | 초기 비밀번호 | (설정값) |
| `SEED_ADMIN_NAME` | 관리자 이름 | `System Admin` (기본값) |

**생성되는 계정 정보:**

| 항목 | 값 |
|------|-----|
| 역할 | `SUPER_ADMIN` |
| 그룹코드 | `ADMIN_GROUP` |
| 상태 | `ACTIVE` |
| 가입방법 | `SEED` |
| 소속 법인 | HQ |
| 비밀번호 변경 강제 | ✅ (첫 로그인 시 비밀번호 변경 필수) |
| 법인 역할 | HQ, VN01, KR01 모두 `SYSTEM_ADMIN` 자동 할당 |

#### (3) 메뉴 권한 기본값

39개 메뉴 × 5개 역할 = **195개 권한 레코드** 자동 생성  
(상세 기본 권한표는 [10장 메뉴 권한 설정](#10-메뉴-권한-설정) 참고)

### 2.2 최초 로그인 후 해야 할 일

1. **비밀번호 변경**: 첫 로그인 시 `/force-change-password` 화면으로 이동 → 새 비밀번호 설정
2. **법인 정보 보완**: `/settings/entities`에서 법인 상세 정보 입력 (주소, 대표자, 사업자등록번호 등)
3. **부서 구조 생성**: `/settings/departments`에서 각 법인의 부서 트리 생성
4. **그룹 생성**: `/settings/members` → 그룹 탭에서 법인별 사용자 그룹 생성
5. **메뉴 설정 확인**: `/settings/permissions` → 메뉴 설정 탭에서 사용할 메뉴 활성화/비활성화
6. **회원 초대**: `/settings/members` → 초대 탭에서 사용자 초대 시작

## 3. 회원 등급(역할) 체계

### 3.1 시스템 역할 (System Role)

| 역할 | 등급 | 그룹코드 | 설명 |
|------|------|----------|------|
| **SUPER_ADMIN** | 5 (최고) | ADMIN_GROUP | 시스템 최고 관리자. 모든 설정 및 데이터에 접근 |
| **ADMIN** | 4 | ADMIN_GROUP | 관리자. SUPER_ADMIN과 동일한 메뉴 접근 권한 |
| **MANAGER** | 3 | USER_GROUP | 매니저. 일부 설정 메뉴 접근 가능 |
| **MEMBER** | 2 | USER_GROUP | 일반 구성원. 업무 도구 중심 접근 |
| **VIEWER** | 1 (최저) | USER_GROUP | 열람자. 최소한의 읽기 접근 |

### 3.2 역할별 색상 배지

회원 목록에서 역할은 아래 색상으로 표시됩니다:

| 역할 | 배지 색상 |
|------|-----------|
| SUPER_ADMIN | 🟣 보라색 (`bg-purple-100 text-purple-700`) |
| ADMIN | 🔴 빨간색 (`bg-red-100 text-red-700`) |
| MANAGER | 🔵 파란색 (`bg-blue-100 text-blue-700`) |
| MEMBER | 🟢 초록색 (`bg-green-100 text-green-700`) |
| VIEWER | ⚪ 회색 (`bg-gray-100 text-gray-500`) |

### 3.3 역할별 기본 메뉴 접근 권한 요약

| 메뉴 카테고리 | SUPER_ADMIN / ADMIN | MANAGER | MEMBER | VIEWER |
|--------------|---------------------|---------|--------|--------|
| **업무 도구** (TODO, 회의록, 문서 등) | 전체 ✅ | 전체 ✅ | 대부분 ✅ | 제한적 ✅ |
| **모듈** (회계, HR, 결제, 메일) | 전체 ✅ | 전체 ✅ | 전체 ✅ | 메일만 ✅ |
| **설정** (회원, 권한, 법인 등) | 전체 ✅ | 일부 ✅ | 전체 ❌ | 전체 ❌ |
| **AI 채팅** | 전체 ✅ | 전체 ✅ | 전체 ✅ | 전체 ✅ |

### 3.4 역할 계층 (Hierarchy) 규칙

역할은 **계층적**으로 동작합니다:

- `@Roles('MEMBER')` 가드가 설정된 API → MEMBER(2) **이상** 모든 역할 접근 가능
- `@Roles('MANAGER')` → MANAGER(3), ADMIN(4), SUPER_ADMIN(5) 접근 가능
- `@Roles('ADMIN')` → ADMIN(4), SUPER_ADMIN(5)만 접근 가능

```
SUPER_ADMIN(5) ⊃ ADMIN(4) ⊃ MANAGER(3) ⊃ MEMBER(2) ⊃ VIEWER(1)
```

> ⚠️ **SUPER_ADMIN과 ADMIN의 차이**: 메뉴 접근 권한은 동일하지만, SUPER_ADMIN은 시스템에서 1명만 존재하는 것이 권장됩니다. 역할 변경 모달에서 SUPER_ADMIN 옵션은 노출되지 않으며, DB에서만 직접 설정 가능합니다.

## 4. 그룹 코드 체계

### 4.1 그룹 코드 (Group Code)란?

모든 사용자는 반드시 하나의 **그룹 코드**에 소속됩니다. 이것은 [8장 그룹 관리](#8-그룹group-관리)의 사용자 그룹(Group)과는 다른 개념입니다.

| 그룹 코드 | 대상 | 설명 |
|-----------|------|------|
| **ADMIN_GROUP** | 본사 관리자 | SUPER_ADMIN, ADMIN 역할의 HQ 소속 사용자 |
| **USER_GROUP** | 법인 사용자 | MANAGER, MEMBER, VIEWER 역할의 법인 소속 사용자 |

### 4.2 그룹 코드별 차이

| 항목 | ADMIN_GROUP | USER_GROUP |
|------|-------------|------------|
| 권한 설정 화면 | 전체 잠금 (수정 불가) | 역할별/그룹별 설정 가능 |
| 메뉴 접근 | 활성화된 모든 메뉴 접근 | 역할/그룹 기반 접근 |
| 설정 메뉴 진입 | 가능 | MANAGER의 일부만 가능 |
| 권한 설정 페이지 탭 | ADMIN_GROUP 탭 | USER_GROUP 탭 |

### 4.3 그룹 코드와 시스템 역할의 관계

```
ADMIN_GROUP ─── SUPER_ADMIN
            └── ADMIN

USER_GROUP  ─── MANAGER
            ├── MEMBER
            └── VIEWER
```

> 💡 그룹 코드는 회원 초대 시 역할에 따라 자동 설정되며, 수동 변경은 DB에서만 가능합니다.

## 5. 회원 추가 방법

회원을 추가하는 방법은 **3가지**입니다.

### 5.1 방법 1: 관리자 초대 (권장)

> **경로**: 설정 → 회원 관리 → 초대 탭 → `+ 초대`

**① 초대 폼 입력**

| 입력 항목 | 필수 | 설명 |
|-----------|------|------|
| 이메일 | ✅ | 초대할 사용자의 이메일 주소 |
| 역할 | ✅ | MEMBER (기본값) / MANAGER / VIEWER / ADMIN |
| 부서 | 선택 | 초기 소속 부서 선택 |
| 그룹 | 선택 | 소속 그룹 선택 |

**② 초대 프로세스**

```
[관리자] 초대 폼 작성 → 제출
    ↓
[시스템] 초대 레코드 생성 (토큰 발급, 유효기간 7일)
    ↓
[시스템] 초대 이메일 발송 (수락 링크 포함)
    ↓
[사용자] 이메일의 링크 클릭 → /invite/:token 접속
    ↓
[사용자] 이름, 비밀번호 입력 → 제출
    ↓
[시스템] 사용자 계정 생성
    ├── 자동 승인 ON  → 즉시 ACTIVE (바로 로그인 가능)
    └── 자동 승인 OFF → PENDING 상태 (관리자 승인 대기)
```

**③ 초대 상태 관리**

| 상태 | 설명 | 관리자 액션 |
|------|------|-------------|
| PENDING | 수락 대기 중 | 재전송 / 취소 가능 |
| ACCEPTED | 수락 완료 | - |
| CANCELLED | 관리자가 취소 | - |
| (만료) | 7일 경과 | 재전송으로 연장 |

> ⚠️ 초대 이메일을 찾지 못한 경우, 관리자가 초대 탭에서 **재전송** 버튼으로 이메일을 재발송할 수 있습니다.

### 5.2 방법 2: 자가 회원가입

> **경로**: 로그인 화면 → "회원가입" 링크 → `/register`

**① 가입 폼 입력**

| 입력 항목 | 필수 | 설명 |
|-----------|------|------|
| 이메일 | ✅ | 로그인용 이메일 (중복 불가) |
| 비밀번호 | ✅ | 8자 이상, 영문+숫자 조합 |
| 이름 | ✅ | 표시 이름 |
| 초대 코드 | 선택 | 초대 토큰이 있는 경우 입력 |

**② 가입 후 상태**

| 경우 | 결과 |
|------|------|
| 초대 코드 입력 (자동승인 ON) | 즉시 ACTIVE |
| 초대 코드 입력 (자동승인 OFF) | PENDING → 관리자 승인 필요 |
| 초대 코드 없음 | MEMBER / USER_GROUP / PENDING → 관리자 승인 필요 |

> 💡 초대 없이 가입한 사용자는 기본적으로 **MEMBER** 역할, **USER_GROUP** 코드로 생성되며, 관리자 승인을 받아야 시스템을 사용할 수 있습니다.

### 5.3 방법 3: 시드 계정 (시스템 자동 생성)

환경변수에 의해 시스템 시작 시 자동 생성되는 계정입니다.  
([2장 최초 세팅](#2-최초-세팅-및-기본-사용자) 참고)

### 5.4 비밀번호 규칙

| 규칙 | 내용 |
|------|------|
| 최소 길이 | 8자 이상 |
| 구성 요건 | 영문 + 숫자 조합 |
| 초기 비밀번호 | 시드 계정은 첫 로그인 시 변경 강제 |
| 재설정 | `/forgot-password`에서 이메일로 재설정 링크 발송 |

## 6. 회원 관리 화면

### 6.1 회원 관리 페이지 접근

> **경로**: `설정(⚙️)` → `회원 관리`  
> **URL**: `/settings/members`  
> **접근 권한**: ADMIN 이상 + SETTINGS_MEMBERS 메뉴 권한

### 6.2 화면 구성 (3개 탭)

```
┌────────────────────────────────────────┐
│         회원 관리                        │
├──────┬──────┬──────────────────────────┤
│ 회원  │ 그룹  │ 초대                     │
├──────┴──────┴──────────────────────────┤
│                                        │
│  (선택된 탭의 콘텐츠 영역)                │
│                                        │
└────────────────────────────────────────┘
```

### 6.3 회원 탭 (Member List)

**테이블 컬럼:**

| 컬럼 | 설명 |
|------|------|
| 이름 | 클릭 시 상세 페이지로 이동 |
| 이메일 | 로그인용 이메일 |
| 부서 | 소속 부서명 |
| 역할 | 색상 배지로 표시 (SUPER_ADMIN~VIEWER) |
| 그룹 | 소속 그룹명 |
| 상태 | ACTIVE / PENDING / INACTIVE / SUSPENDED / WITHDRAWN |

**관리자 기능:**

| 상태 | 가능한 액션 |
|------|-------------|
| PENDING | ✅ 승인 / ❌ 거부 버튼 |
| ACTIVE | 🔄 역할 변경 버튼 |
| INACTIVE | (표시만) |

### 6.4 회원 상세 페이지

> **경로**: `/settings/members/:userId`

**화면 구성:**

```
┌─────────────────────────────────────────────┐
│  ← 뒤로   회원 상세 정보                      │
├─────────────────────────────────────────────┤
│                                             │
│  📧 이메일: user@example.com                 │
│  👤 이름: [홍길동        ] (수정 가능)         │
│  📬 회사 이메일: [gildong@amoeba.co] (수정)    │
│  💼 직책: [매니저        ] (수정 가능)          │
│  🏷️ 역할: MEMBER (배지)                      │
│  📊 상태: ACTIVE (배지)                       │
│                                             │
│  ─── 소속 법인 ────────────── [+ 법인 추가]   │
│  │ HQ (ROOT) - MEMBER          [제거]       │
│  │ VN01 (SUBSIDIARY) - ADMIN   [제거]       │
│                                             │
│  ─── 소속 부서 ────────────── [+ 부서 추가]   │
│  │ 경영관리부 - DEPARTMENT_HEAD  [제거]       │
│  │ 회계팀 - MEMBER               [제거]      │
│                                             │
│  ─── 소속 그룹 ────────────── [+ 그룹 추가]   │
│  │ VN01 일반그룹                  [제거]      │
│                                             │
│  ─── HR 직원 연결 ─────────── [+ 직원 연결]   │
│  │ 홍길동 (VN01)                 [해제]       │
│                                             │
└─────────────────────────────────────────────┘
```

**주요 관리 기능:**

| 기능 | 설명 |
|------|------|
| **법인 추가/제거** | `+ 법인 추가` → 법인 목록에서 선택 + 역할(MEMBER/ADMIN/SYSTEM_ADMIN) 지정 |
| **부서 추가/제거** | `+ 부서 추가` → 법인 선택 → 부서 선택 + 역할(MEMBER/TEAM_LEAD/DEPARTMENT_HEAD) 지정 |
| **그룹 추가/제거** | `+ 그룹 추가` → 그룹 목록에서 선택 후 추가 |
| **HR 직원 연결** | `+ 직원 연결` → HR 모듈의 직원 레코드와 매핑 |
| **PENDING 승인** | 상태가 PENDING인 경우 승인/거절 버튼 표시 |

## 7. 법인(Entity) 관리

### 7.1 법인 관리 페이지 접근

> **경로**: `설정(⚙️)` → `법인 관리`  
> **URL**: `/settings/entities`  
> **접근 권한**: ADMIN 이상 + SETTINGS_ENTITIES 메뉴 권한

### 7.2 법인 계층 구조

```
HQ (ROOT / 본사)
├── VN01 (SUBSIDIARY / 법인)
└── KR01 (SUBSIDIARY / 법인)
    └── (추가 법인 생성 가능)
```

| 레벨 | 설명 |
|------|------|
| **ROOT** | 본사 (HQ). 시스템에 1개만 존재. isHq=true |
| **SUBSIDIARY** | 자회사/법인. 여러 개 생성 가능 |

### 7.3 법인 정보 필드

| 필드 | 필수 | 설명 |
|------|------|------|
| 법인 코드 | ✅ | 고유 코드 (예: VN01, KR01). 변경 불가 |
| 법인명 | ✅ | 현지 법인명 |
| 영문 법인명 | 선택 | 영문 법인명 |
| 국가 | ✅ | 2자리 국가 코드 (VN, KR, JP 등) |
| 통화 | ✅ | 3자리 통화 코드 (VND, KRW, JPY 등) |
| 사업자등록번호 | 선택 | |
| 주소 | 선택 | 법인 주소 |
| 대표자 | 선택 | 대표이사명 |
| 전화번호 | 선택 | |
| 이메일 | 선택 | 법인 대표 이메일 |
| 급여일 | 선택 | 월 급여 지급일 (기본: 25일) |
| 상태 | ✅ | ACTIVE / INACTIVE |
| 직인 이미지 | 선택 | 법인 직인 (전자 문서용) |

### 7.4 법인 역할 (Entity Role)

사용자를 법인에 할당할 때 법인 내 역할을 지정합니다:

| 법인 역할 | 설명 |
|-----------|------|
| **SYSTEM_ADMIN** | 해당 법인의 시스템 관리자 (법인 설정 전체 관리) |
| **ADMIN** | 법인 관리자 (법인 내 데이터 관리) |
| **MEMBER** | 법인 일반 구성원 (데이터 조회/입력) |

### 7.5 법인 컨텍스트 전환

화면 상단에 **법인 선택 드롭다운**이 있습니다. 법인을 전환하면:

- API 요청에 `X-Entity-Id` 헤더가 자동 추가됨
- 해당 법인의 데이터만 조회/관리 가능
- HR, 회계, 결제 등 법인별 기능이 해당 법인으로 전환

> 💡 사용자는 자신이 할당된 법인만 드롭다운에 표시됩니다.

### 7.6 법인 추가 순서

1. `/settings/entities` 접속
2. `+ 법인 추가` 버튼 클릭
3. 법인 코드, 법인명, 국가, 통화 등 입력
4. 저장
5. 사용자를 법인에 할당 (회원 상세 페이지에서 `+ 법인 추가`)

## 8. 그룹(Group) 관리

### 8.1 그룹이란?

그룹은 **법인 내에서 사용자를 묶어 메뉴 접근 권한을 추가로 관리**하는 단위입니다.

> ⚠️ **그룹 코드 vs 그룹**: 
> - **그룹 코드** (ADMIN_GROUP / USER_GROUP): 시스템 역할과 연동되는 고정 분류
> - **그룹** (Group): 법인별로 자유롭게 생성/관리하는 사용자 묶음

### 8.2 그룹 관리 페이지 접근

> **경로**: `설정(⚙️)` → `회원 관리` → `그룹 탭`  
> **URL**: `/settings/members` (그룹 탭 선택)  
> **접근 권한**: ADMIN / SUPER_ADMIN

### 8.3 그룹 목록 화면

카드 형태로 표시됩니다:

```
┌─────────────────────┐  ┌─────────────────────┐
│  [VN01] VN 영업팀     │  │  [KR01] KR 관리팀     │
│                     │  │                     │
│  영업부서 사용자 그룹  │  │  관리부서 사용자 그룹  │
│                     │  │                     │
│  👤 3명              │  │  👤 5명              │
│  [✏️] [🗑️] [👥]    │  │  [✏️] [🗑️] [👥]    │
└─────────────────────┘  └─────────────────────┘
```

각 카드에는:
- 그룹명 + 법인 코드 배지
- 설명
- 멤버 미리보기 (최대 5명 아바타)
- 관리 버튼: 수정(✏️), 삭제(🗑️), 멤버 관리(👥)

### 8.4 그룹 생성

1. `+ 그룹 추가` 버튼 클릭
2. 그룹 생성/수정 모달 입력:

| 입력 항목 | 필수 | 설명 |
|-----------|------|------|
| 그룹명 | ✅ | 고유한 그룹 이름 (중복 불가) |
| 설명 | 선택 | 그룹 설명 |
| 소속 법인 | 선택 | 드롭다운에서 법인 선택 |

3. 저장

### 8.5 그룹 멤버 관리

그룹 카드의 **👥 멤버 관리** 버튼 클릭 → 모달 표시:

- **현재 멤버 목록**: 이름, 이메일 표시. 각 멤버 옆에 [제거] 버튼
- **멤버 추가**: 전체 사용자 목록에서 검색/선택하여 추가

### 8.6 그룹별 메뉴 접근 권한

그룹에 대한 메뉴 접근 권한은 **권한 설정 페이지 (`/settings/permissions`)** 에서 설정합니다.  
(자세한 방법은 [10장 메뉴 권한 설정](#10-메뉴-권한-설정) 참고)

## 9. 부서(Department) 관리

### 9.1 부서 관리 페이지 접근

> **경로**: `설정(⚙️)` → `부서 관리`  
> **URL**: `/settings/departments`  
> **접근 권한**: ADMIN 이상 + DEPARTMENTS 메뉴 권한

### 9.2 부서 트리 구조

부서는 **법인별 2계층 트리** 구조입니다:

```
VN01 (법인)
├── 📁 경영관리부 (Level 1: Department)
│   ├── 📄 경영기획팀 (Level 2: Team)
│   └── 📄 인사총무팀 (Level 2: Team)
├── 📁 개발부 (Level 1: Department)
│   ├── 📄 프론트엔드팀 (Level 2: Team)
│   └── 📄 백엔드팀 (Level 2: Team)
└── 📁 영업부 (Level 1: Department)
    └── 📄 국내영업팀 (Level 2: Team)
```

| 레벨 | 타입 | 설명 |
|------|------|------|
| Level 1 | **Department** (부서) | 최상위 조직 단위 |
| Level 2 | **Team** (팀) | 부서 하위의 실무 단위 |

### 9.3 부서 관리 화면 구성

```
┌──────────────┬────────────────────────────────┐
│   부서 트리     │   부서 상세/편집                  │
│              │                                │
│ ▶ 경영관리부   │  부서명: [경영관리부      ]        │
│   └ 경영기획팀 │  로컬명: [Management Dept]        │
│   └ 인사총무팀 │  활성화: [✅]                     │
│ ▶ 개발부      │  정렬순서: [100]                  │
│   └ FE팀     │                                │
│   └ BE팀     │  ─── 소속 멤버 ── [+ 멤버 추가]    │
│              │  │ 김철수 - DEPARTMENT_HEAD       │
│ [+ 부서 추가]  │  │ 이영희 - TEAM_LEAD            │
│              │  │ 박민수 - MEMBER                │
└──────────────┴────────────────────────────────┘
```

### 9.4 부서 역할 (Department Role)

| 부서 역할 | 설명 |
|-----------|------|
| **DEPARTMENT_HEAD** | 부서장. 부서 전체 데이터 접근/관리 |
| **TEAM_LEAD** | 팀장. 팀 데이터 관리 |
| **MEMBER** | 부서/팀 일반 구성원 |

### 9.5 부서에 멤버 배정

1. 부서 트리에서 대상 부서 클릭
2. 우측 상세 영역에서 `+ 멤버 추가` 클릭
3. 사용자 선택 + 역할(MEMBER / TEAM_LEAD / DEPARTMENT_HEAD) 지정
4. 저장

> 💡 **주 소속 설정**: 한 사용자가 여러 부서에 소속될 수 있으며, `주 소속(Primary)` 플래그로 대표 소속을 지정합니다.

### 9.6 부서 생성/수정

| 입력 항목 | 필수 | 설명 |
|-----------|------|------|
| 부서명 | ✅ | 부서/팀 이름 |
| 로컬명 | 선택 | 현지 언어 부서명 |
| 상위 부서 | 선택 | 팀 생성 시 상위 부서 선택 (자동으로 Level 2) |
| 활성화 | ✅ | ON/OFF |
| 정렬 순서 | 선택 | 트리 내 표시 순서 |

> ⚠️ **삭제 제약**: 하위 부서(팀)가 있는 부서는 삭제할 수 없습니다. 하위를 먼저 삭제해야 합니다.

4. **그룹별 권한 체크**: 그룹 열에서 메뉴별 체크박스 ON/OFF
5. **저장** 버튼 클릭

> 💡 **비활성화된 메뉴**: 메뉴 설정 탭에서 비활성화된 메뉴는 회색으로 표시되며, 역할/그룹 권한과 관계없이 접근 불가합니다.

### 10.5 탭 3: 사용자별 권한 (UserPermissionTab)

- 메뉴별 ON/OFF 토글
- 저장 시 해당 사용자만 적용 (다른 같은 역할 사용자에 영향 없음)

> 💡 사용자별 설정은 역할 기본값보다 **우선순위가 높습니다**.

## 11. 권한 판정 로직

### 11.1 메뉴 접근 판정 플로우차트

사용자가 특정 메뉴에 접근할 때 시스템은 다음 순서로 권한을 판정합니다:

```
사용자가 메뉴 접근 시도
        │
        ▼
┌─ 메뉴가 활성화(enabled)되어 있는가? ─┐
│                                     │
│  NO → 접근 차단 (숨김)               │
│                                     │
│  YES                                │
│   ▼                                 │
│  사용자가 ADMIN_GROUP인가?            │
│  (SUPER_ADMIN 또는 ADMIN)            │
│                                     │
│  YES → ✅ 접근 허용                  │
│                                     │
│  NO                                 │
│   ▼                                 │
│  사용자별 개별 설정이 있는가?           │
│                                     │
│  YES → 개별 설정값 적용              │
│        (true→허용, false→차단)       │
│                                     │
│  NO                                 │
│   ▼                                 │
│  역할별 기본 설정 적용                │
│  (DEFAULT_PERMISSIONS 기반)          │
│                                     │
│   ▼                                 │
│  그룹별 권한 필터가 있는가?            │
│                                     │
│  YES → 소속 그룹 중 하나라도          │
│        accessible이면 → 허용         │
│        모두 차단이면 → 차단           │
│                                     │
│  NO → 역할 기본값 그대로 적용         │
└─────────────────────────────────────┘
```

### 11.2 우선순위 요약

| 순위 | 판정 기준 | 설명 |
|------|-----------|------|
| 1 (최고) | 메뉴 활성화 | 비활성 메뉴는 누구도 접근 불가 |
| 2 | ADMIN 역할 | ADMIN_GROUP은 무조건 접근 |
| 3 | 사용자별 설정 | 개별 오버라이드 (가장 세밀한 제어) |
| 4 | 역할별 설정 | MANAGER/MEMBER/VIEWER 기본값 |
| 5 (최저) | 그룹별 필터 | 추가적인 접근 제한/허용 |

### 11.3 API 가드 체계

| 가드 | 설명 | 적용 위치 |
|------|------|-----------|
| **AuthGuard** | JWT 인증 확인 | 모든 보호 API |
| **RolesGuard** | 역할 계층 확인 (예: MANAGER 이상만) | API 컨트롤러별 |
| **AdminGuard** | ADMIN/SUPER_ADMIN만 허용 | 설정 관련 API |
| **EntityGuard** | 법인 컨텍스트 설정 | 법인별 데이터 API |
| **MenuGuard** | 메뉴 접근 권한 확인 | 메뉴별 페이지 |

## 12. 회원 상태 관리

### 12.1 회원 상태 종류

| 상태 | 설명 | 로그인 가능 |
|------|------|-------------|
| **PENDING** | 가입 후 관리자 승인 대기 | ⚠️ 로그인 가능 (대기 화면 표시) |
| **ACTIVE** | 정상 활성 상태 | ✅ |
| **INACTIVE** | 비활성 (장기 미접속 등) | ❌ |
| **SUSPENDED** | 정지 (관리자 조치) | ❌ |
| **WITHDRAWN** | 탈퇴 | ❌ |

### 12.2 상태 전환 흐름

```
┌──────────┐     승인      ┌──────────┐
│ PENDING  │ ────────────→ │  ACTIVE  │
│ (대기중)  │               │ (활성)    │
└──────────┘               └──────────┘
     │                          │
     │ 거부                      │ 비활성화/정지
     ▼                          ▼
┌──────────┐               ┌──────────┐
│WITHDRAWN │               │INACTIVE /│
│ (탈퇴)    │               │SUSPENDED │
└──────────┘               └──────────┘
```

### 12.3 로그인 시 상태별 동작

| 상태 | 동작 |
|------|------|
| ACTIVE | 정상 대시보드로 이동 |
| ACTIVE + mustChangePw | `/force-change-password`로 리다이렉트 |
| PENDING | `/pending` (승인 대기 안내 화면) |
| INACTIVE / SUSPENDED | `/inactive` (계정 비활성 안내 화면) |
| WITHDRAWN | 로그인 차단 (에러 메시지) |

## 13. 운영 시나리오별 가이드

### 시나리오 1: 새 법인 추가 + 직원 초대

**상황**: 일본 법인(JP01)을 신설하고, 법인장을 초대해야 합니다.

**절차:**

1. **법인 생성**
   - `/settings/entities` → `+ 법인 추가`
   - 코드: `JP01`, 이름: `アメーバジャパン`, 국가: `JP`, 통화: `JPY`
   - 저장

2. **회원 초대**
   - `/settings/members` → 초대 탭 → `+ 초대`
   - 이메일: `tanaka@example.com`
   - 역할: `MANAGER`
   - 저장 → 이메일 발송

3. **초대 수락 후 설정**
   - 사용자가 초대 수락 → ACTIVE 상태
   - `/settings/members` → 회원 탭 → 해당 사용자 클릭
   - 상세 페이지에서:
     - `+ 법인 추가` → JP01 선택, 역할: ADMIN
     - `+ 부서 추가` → JP01의 부서 선택, 역할: DEPARTMENT_HEAD

4. **그룹 생성 (필요시)**
   - 회원 관리 → 그룹 탭 → `+ 그룹 추가`
   - 그룹명: `JP 관리팀`, 소속 법인: JP01
   - 멤버 관리에서 해당 사용자 추가

### 시나리오 2: 역할 변경 (MEMBER → MANAGER)

**상황**: 일반 멤버를 매니저로 승격합니다.

1. `/settings/members` → 회원 탭
2. 대상 사용자 행에서 **역할 변경** 버튼 클릭
3. 역할 선택 드롭다운에서 `MANAGER` 선택
4. 확인 → 즉시 반영

> 💡 역할 변경 후 사용자는 다음 페이지 로드 시부터 새 역할의 메뉴 접근 권한이 적용됩니다.

### 시나리오 3: 특정 그룹에만 메뉴 접근 허용

**상황**: VN01의 "영업팀" 그룹에만 프로젝트 관리 메뉴를 열어주고 싶습니다.

1. **그룹 생성** (이미 있다면 건너뜀)
   - `/settings/members` → 그룹 탭 → `+ 그룹 추가`
   - 그룹명: `VN 영업팀`, 소속 법인: VN01
   - 멤버 추가

2. **그룹 권한 설정**
   - `/settings/permissions` → 역할별 권한 탭
   - USER_GROUP 탭 선택
   - 법인 선택: VN01
   - `프로젝트` 메뉴 행에서 `VN 영업팀` 열의 체크박스 ✅ 체크
   - 저장

3. **결과**: VN 영업팀 그룹에 속한 MEMBER 역할 사용자도 프로젝트 메뉴에 접근 가능

### 시나리오 4: 신규 사용자 자가 가입 후 승인

**상황**: 초대 없이 회원가입한 사용자를 승인합니다.

1. 사용자가 `/register`에서 가입 → PENDING 상태
2. 사용자가 로그인하면 `/pending` (승인 대기) 화면 표시
3. 관리자: `/settings/members` → 회원 탭
4. PENDING 상태 사용자의 **승인** 버튼 클릭
5. 승인 완료 → ACTIVE 전환 → 사용자는 정상 로그인 가능

### 시나리오 5: 특정 사용자에게만 추가 메뉴 접근 허용

**상황**: MEMBER 역할이지만 특별히 회계 설정 메뉴에 접근이 필요합니다.

1. `/settings/permissions` → 사용자별 권한 탭
2. 사용자 검색 → 선택
3. 필요한 메뉴의 접근 토글 ON
4. 저장

> 💡 사용자별 설정은 역할 기본 권한보다 우선합니다.

### 시나리오 6: 부서 구조 개편

**상황**: 기존 영업부 아래에 새로운 해외영업팀을 추가합니다.

1. `/settings/departments` → 법인 선택
2. 좌측 트리에서 `영업부` 선택
3. `+ 하위 부서 추가` 클릭
4. 부서명: `해외영업팀`, 활성화: ON
5. 저장
6. 멤버 배정: `+ 멤버 추가` → 사용자 선택 + 역할 지정

## 14. 자주 묻는 질문(FAQ)

### Q1. SUPER_ADMIN과 ADMIN은 어떤 차이가 있나요?

**메뉴 접근 권한은 동일**합니다. 차이점:
- SUPER_ADMIN은 시스템 최고 관리자로, 역할 변경 모달에서 선택할 수 없습니다 (DB 직접 변경만 가능)
- 시스템에 1명만 존재하는 것이 권장됩니다
- 시드 계정이 SUPER_ADMIN으로 생성됩니다

### Q2. 초대 이메일이 오지 않았다면?

관리자가 `/settings/members` → 초대 탭에서 해당 초대의 **재전송** 버튼을 클릭하면 됩니다.

### Q3. 초대 링크가 만료되었다면?

초대 유효기간은 7일입니다. 만료 시 관리자가 **재전송**하면 새 토큰이 발급되어 기간이 연장됩니다.

### Q4. 그룹 코드(ADMIN_GROUP/USER_GROUP)를 변경할 수 있나요?

UI에서는 변경할 수 없습니다. DB에서 직접 `usr_group_code` 필드를 수정해야 합니다. 일반적으로 변경할 필요가 없습니다.

### Q5. 한 사용자가 여러 법인에 소속될 수 있나요?

**네.** 회원 상세 페이지에서 `+ 법인 추가`를 통해 여러 법인에 할당 가능합니다. 각 법인별로 별도의 법인 역할(MEMBER/ADMIN/SYSTEM_ADMIN)을 지정합니다.

### Q6. 한 사용자가 여러 부서에 소속될 수 있나요?

**네.** 여러 부서에 소속될 수 있으며, **주 소속(Primary)** 플래그로 대표 소속을 지정합니다.

### Q7. 한 사용자가 여러 그룹에 소속될 수 있나요?

**네.** 그룹별 메뉴 권한은 **OR 로직**으로 판정됩니다. 소속 그룹 중 하나라도 해당 메뉴에 접근 가능하면 접근이 허용됩니다.

### Q8. 메뉴를 비활성화하면 이미 로그인한 사용자에게도 적용되나요?

비활성화된 메뉴는 다음 API 호출 시(페이지 새로고침 또는 메뉴 목록 재요청 시)부터 반영됩니다.

### Q9. VIEWER 역할은 어떤 경우에 사용하나요?

- 외부 감사인, 투자자 등 **제한적 열람만 필요한 사용자**
- 인턴 등 **최소 권한이 필요한 임시 구성원**
- 퇴직 예정자의 **권한 축소** 단계

### Q10. 비밀번호를 잊었을 때는?

로그인 화면에서 **"비밀번호를 잊으셨나요?"** 링크 클릭 → 이메일 입력 → 비밀번호 재설정 링크 발송 → 새 비밀번호 설정

### Q11. 회사 이메일은 어떻게 생성되나요?

회원가입 또는 초대 수락 시 **자동으로** 회사 이메일이 생성됩니다 (Postal API 기반). 생성된 이메일은 회원 상세 페이지에서 확인/수정할 수 있습니다.

### Q12. 법인 컨텍스트를 전환하면 어떤 데이터가 바뀌나요?

법인 전환 시 아래 영역의 데이터가 해당 법인으로 전환됩니다:
- HR (직원, 급여, 근태)
- 회계 (전표, 재무제표)
- 결제 (전자세금계산서)
- 부서 정보
- 그룹 목록 (해당 법인의 그룹만 표시)
