# Upload Files Migration Guide (Staging -> Production)

## 1) Objective
- Migrate uploaded files and image assets stored in Docker volume from staging to production.
- Target scope: files under `api_uploads` volume mounted to `/app/apps/api/uploads` in API container.

## 2) Environment Analysis Summary

### Staging
- Docker volume: `staging_api_uploads`
- Mount path (host): `/home/docker/volumes/staging_api_uploads/_data`
- Container mount path: `/app/apps/api/uploads`
- Current data size: `313.9M`
- Current file count: `533`
- Main directory: `/data/mail-attachments` plus root-level upload files

### Production
- Docker volume: `production_api_uploads`
- Mount path (host): `/var/lib/docker/volumes/production_api_uploads/_data`
- Container mount path: `/app/apps/api/uploads`
- Current data size: `812.0K`
- Current file count: `3`
- Main directory: `/data/mail-attachments`

## 3) Migration Strategy
- Recommended: full overwrite migration with maintenance window.
- Reason:
  - Staging data is significantly larger and appears to be source-of-truth in this request.
  - Upload files should remain consistent with already migrated DB references.

## 4) Pre-checklist
- Confirm maintenance window and freeze file upload operations.
- Confirm SSH access:
  - `ssh amb-staging`
  - `ssh amb-production`
- Confirm API container names:
  - staging: `staging-amb-api`
  - production: `amb-api`
- Confirm volume names exist:
  - staging: `docker volume inspect staging_api_uploads`
  - production: `docker volume inspect production_api_uploads`

## 5) Execution Steps (Recommended)

### Step A. Backup production uploads first (mandatory)
Run on production server:

```bash
ssh amb-production '
  TS=$(date +%Y%m%d_%H%M%S)
  docker run --rm -v production_api_uploads:/data -v /tmp:/backup alpine \
    sh -c "cd /data && tar czf /backup/production_api_uploads_backup_${TS}.tgz ."
  ls -lh /tmp/production_api_uploads_backup_${TS}.tgz
'
```

### Step B. Stop production API to avoid write race
Run on production server:

```bash
ssh amb-production 'cd ~/ambManagement && docker compose -f docker/production/docker-compose.production.yml stop amb-api'
```

### Step C. Stream-copy staging uploads to production volume
Run from local machine (or any host that can access both SSH targets):

```bash
ssh amb-staging "docker run --rm -v staging_api_uploads:/from alpine sh -c 'cd /from && tar czf - .'" \
| ssh amb-production "docker run --rm -i -v production_api_uploads:/to alpine sh -c 'cd /to && tar xzf -'"
```

Notes:
- This command appends/extracts files into production volume.
- If strict overwrite is required, clear destination first (optional):

```bash
ssh amb-production "docker run --rm -v production_api_uploads:/to alpine sh -c 'rm -rf /to/* /to/.[!.]* /to/..?* 2>/dev/null || true'"
```

### Step D. Fix ownership/permissions (if needed)
Run on production server:

```bash
ssh amb-production "docker run --rm -v production_api_uploads:/to alpine sh -c 'chown -R 1000:1000 /to && find /to -type d -exec chmod 755 {} \; && find /to -type f -exec chmod 644 {} \;'"
```

### Step E. Restart production API
Run on production server:

```bash
ssh amb-production 'cd ~/ambManagement && docker compose -f docker/production/docker-compose.production.yml start amb-api'
```

## 6) Verification

### Volume-level verification
```bash
ssh amb-production "docker run --rm -v production_api_uploads:/data alpine sh -c 'du -sh /data; find /data -type f | wc -l; ls -lt /data/mail-attachments | head -n 10'"
```

### App-level verification
- Open production UI and check several records with attachments/images.
- Verify file URL responses are `200` and media renders correctly.
- Verify recent staging sample files are accessible in production.

## 7) Rollback
If issue occurs, restore backup archive from Step A:

```bash
ssh amb-production '
  BACKUP=/tmp/production_api_uploads_backup_YYYYMMDD_HHMMSS.tgz
  docker run --rm -v production_api_uploads:/to -v /tmp:/backup alpine \
    sh -c "rm -rf /to/* /to/.[!.]* /to/..?* 2>/dev/null || true; cd /to && tar xzf ${BACKUP}"
'
```

Then restart API and re-verify.

## 8) Risks and Controls
- Risk: upload writes during copy cause partial inconsistency.
  - Control: stop API before copy (Step B).
- Risk: wrong destination volume target.
  - Control: verify volume names and mountpoints before copy.
- Risk: permission mismatch after extract.
  - Control: enforce ownership/permission step (Step D).

